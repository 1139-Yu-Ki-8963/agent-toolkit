§1Subagent の定義

subagent とは、Agent ツールの subagent_type パラメータで指定して起動する 独立コンテキストの専門実行者。 メイン会話の文脈から切り離されたウィンドウで動作し、最終結果だけを返す。
  • 定義ファイルは ~/.claude/agents/<name>/<name>.md に置く。
  • frontmatter でツール・モデル・起動条件を宣言し、本文がシステムプロンプトになる。
  • skill が「いつ何をするか」の手順を持つのに対し、subagent は「どんな専門性で実行するか」の人格を持つ。

Skill との違い

観点SkillSubagent
コンテキスト メイン会話を共有 独立ウィンドウで実行
起動方法 Skill ツール / 自動マッチ Agent(subagent_type: "name")
提供するもの 手順・知識・強制 専門性・独立判断
配置先 ~/.claude/skills/ ~/.claude/agents/

§24 役割アーキテクチャ

現在の subagent 体系は 4 役割で構成される。新規追加は、既存の役割で不足する専門性がある場合にのみ行う。

役割モデル責務ツール
brain opus タスク分解・作業指示の組み立て・実行結果の検証。Planning モードと Reviewing モードの 2 面を持つ Read, Grep, Glob, Bash
worker-sonnet sonnet 文脈を読んで判断しながらローカルファイルの調査・修正を行う。影響範囲分析・規約チェック・リファクタリング Read, Write, Edit, Bash, Grep, Glob
worker-haiku haiku 判断不要の機械的タスクを高速・低コストで実行。一括リネーム・テンプレ展開・テスト実行・grep 収集 Read, Write, Edit, Bash, Grep, Glob
researcher sonnet MCP ツールで外部情報を収集して構造化レポートを返す。Web 検索・ライブラリ仕様・API ドキュメント参照 MCP 各種, WebSearch, WebFetch, Read, Grep, Glob
各役割は 1 つの専門性に集中する。brain が直接コードを書いたり、worker-haiku が判断を下すことはない。 この境界を侵す定義は設計不良のサイン。

§3役割選択の判定フロー

Q1. タスクの分解・計画・結果検証が必要か？ YES → brain（Planning or Reviewing モード） NO → Q2 へ Q2. 外部情報（Web・API ドキュメント・ライブラリ仕様）が必要か？ YES → researcher NO → Q3 へ Q3. 作業に判断が必要か？ （文脈を読んで適切な変更を決める・影響範囲を分析する 等） YES → worker-sonnet NO（指示が明確で機械的に実行するだけ）→ worker-haiku
判定の注意点
  • 「判断が必要か」で迷ったら worker-sonnet に倒す。haiku に判断タスクを渡すと品質が崩壊する
  • brain は 自分では実装しない。計画を立てて worker に渡すか、worker の結果を検証するだけ
  • researcher は ローカルファイルの修正を行わない。情報収集に特化する

§4frontmatter 仕様

必須フィールド

フィールド型制約説明
name string kebab-case, 64 字以内 一意識別子。ディレクトリ名・ファイル名と一致すること
description multiline string TRIGGER when + SKIP 必須 1 行目に説明（50 字以内）、TRIGGER when / SKIP を含む
tools comma-separated 使用するツールのみ subagent が利用可能なツール一覧
model string opus / sonnet / haiku 使用する Claude モデル

frontmatter テンプレート

テンプレート
--- name: <kebab-case-name> description: | <50 字以内の説明> TRIGGER when: <具体的な起動条件・キーワード> SKIP: <起動しない条件。代替先を明示> tools: <comma-separated tool list> model: <opus | sonnet | haiku> ---
frontmatter の注意点
  • name はディレクトリ名・ファイル名と 完全一致 させる（例: brain → brain/brain.md）
  • TRIGGER when: / SKIP: は固定英語キーワード。Use when: や 使用時: はシステムが認識しない
  • tools に不要なツールを含めない。最小権限原則に従う

§5命名規約

名前だけで「どんな専門性を持つか」が一意に伝わること。
  • 語形: 役割名（名詞）で確定。brain / worker-sonnet / researcher など
  • 新規追加時はモデル名やドメインを含めて差別化する（例: worker-haiku, researcher）
  • kebab-case、64 字以内
  • 予約語 anthropic / claude を含めない
  • 既存の 4 役割名と衝突しないこと

既存の 4 名

name由来
brain計画・判断の中枢
worker-sonnet中判断の作業者 + 使用モデル
worker-haiku機械的作業者 + 使用モデル
researcher外部情報の調査者

§6description 規律

  • 1 行目: 50 字以内の説明。subagent の責務を能動文で 1 文にまとめる
  • 2 行目: TRIGGER when: で起動条件を具体的キーワードで列挙
  • 3 行目: SKIP: で非対象条件を明示し、代替 subagent を案内

良い例 / 悪い例

判定例理由
✓ 機械的な作業を高速・低コストで実行するエージェント。判断しない。 責務と制約が 1 文で明確
✗ いろいろな作業を手伝うエージェント。 曖昧。何に特化しているかが伝わらない
✓ TRIGGER when: 一括リネーム、テンプレ展開、テスト実行、ビルド確認 具体的なキーワード列挙
✗ TRIGGER when: 必要な時に使う 条件が不定。自動マッチの精度が下がる

§7ツール選択

subagent に付与するツールは最小権限原則に従う。 使わないツールを与えると、想定外の操作リスクが増える。
役割パターン推奨ツールセット禁止ツール
計画・検証系 Read, Grep, Glob, Bash Write, Edit（計画者は書かない）
調査・修正系 Read, Write, Edit, Bash, Grep, Glob Agent（再帰呼び出し禁止）
機械的実行系 Read, Write, Edit, Bash, Grep, Glob Agent（再帰呼び出し禁止）
外部調査系 MCP 各種, WebSearch, WebFetch, Read, Grep, Glob Write, Edit（調査者は書かない）
ツール選択の注意点
  • Agent ツールを subagent に付与すると再帰呼び出しが発生する。原則禁止
  • MCP ツールは接続状態に依存する。headless 環境では使えない場合がある
  • 計画者（brain）に Write/Edit を与えない理由: 計画と実装を同じ agent がやると品質管理ができない

§8フォルダ構造

~/.claude/agents/ ├── brain/ │ ├── brain.md ← 定義本体（frontmatter + 本文） │ └── references/ │ ├── planning.md ← Planning モード参照資料 │ └── reviewing.md ← Reviewing モード参照資料 ├── worker-sonnet/ │ ├── worker-sonnet.md │ └── references/ │ └── patterns.md ← 調査・修正パターン集 ├── worker-haiku/ │ └── worker-haiku.md ← references 不要（判断しない） └── researcher/ └── researcher.md ← MCP ツール選択表を本文に内包

構造ルール

  • 1 サブエージェント = 1 ディレクトリ。ディレクトリ名 = name フィールド
  • 定義ファイル名も <name>.md で一致させる
  • references/ は参照資料が必要な場合のみ作成。判断不要の worker-haiku のように不要なら省く
  • 可変データ（ログ・状態）は ~/.claude/agents/ 配下に置かない。skill と同様、外部に出す

§9本文ルール

構成テンプレート

# <Name>: <役割の要約> <1-2 文の責務説明> ## <モード or 得意な作業> <具体的な作業項目のリスト> ## 出力フォーマット <返却する結果の構造>

制約

制約上限根拠
本文行数 100 行以内 subagent はシステムプロンプトとして全文ロードされるため、簡潔に保つ
セクション数 2〜4 セクション モード説明 + 出力フォーマットが基本
見出し言語 日本語統一 CLAUDE.md の言語方針に従う
絶対パス 禁止 環境依存を排除。~/.claude/ 形式を使う
subagent の本文は「このエージェントはこう振る舞え」というシステムプロンプト。 具体的な手順書ではなく、人格と専門性を定義する。 詳細な手順は references/ に分離する。

§10references の設計

  • 必要な場合のみ作成する。全 subagent に references が必要なわけではない
  • 本文が 50 行を超えそうな詳細は references/ に逃がす
  • references は subagent 固有の知識。汎用パターンは skill や rules に置く

現在の references 構成

subagentファイル内容
brain planning.md worker 種別ごとの指示テンプレート、タスク分解チェックリスト
brain reviewing.md 5 観点検証フレームワーク、判定基準表、プロンプト品質 8 観点
worker-sonnet patterns.md 呼び出し元追跡・影響範囲分析・規約チェック・PR 分析のパターン集
worker-haiku （なし） 判断不要のため、本文の得意な作業リストで十分
researcher （なし） MCP ツール選択表を本文に内包。外部ツール依存のため静的 reference は不適

§11Skill との連携

Skill が「いつ・何をするか」を定め、Subagent が「どんな専門性で実行するか」を定める。 Skill は Subagent を呼ぶ。Subagent は Skill を呼ばない。

連携パターン

パターン説明例
調査委任 Skill が worker-sonnet に影響範囲分析を委任 Agent(subagent_type: "worker-sonnet", prompt: "影響範囲を分析して")
並列実行 Skill が複数の worker-haiku を並列起動して一括処理 複数ファイルの同時リネーム
外部調査 Skill が researcher にライブラリ仕様の確認を依頼 Agent(subagent_type: "researcher", prompt: "最新の API 仕様を調べて")
計画→実行 brain がタスクを分解し、各 worker に配布 大規模リファクタリングの多段実行
連携の注意点
  • Skill の allowed-tools に Agent が含まれていないと subagent を呼べない
  • subagent 間の直接通信はない。brain → worker の指示は呼び出し元の Skill が中継する
  • config-placement.md の Q5「独立コンテキストで完結させたい」に YES → Subagent が正解

テスト方法

  1. 直接起動テスト: Agent(subagent_type: "<name>") で実際に呼び出し、期待する専門性で応答するかを確認
  2. 境界テスト: 本来 SKIP すべきタスクを渡し、拒否するか代替を案内するかを確認
  3. 出力形式テスト: 定義した出力フォーマットに沿った結果が返るかを確認

§13調査チェックリストパイプライン（2026-06-27 導入）

背景と課題

2026-06-27 のプロジェクト構成レビューで 5 件の事実誤認が発生した。

#誤り実際
1Playwright 設定がルート直下frontend/ 配下だった
2logs/ は .gitkeep だけ554 ファイル / 7.6MB が Git 追跡
3render.yaml はルート固定Dashboard で任意パス指定可能
4vercel.json はルート固定Root Directory 設定で変更可能
5.vercelignore を ignoreCommand で代替可能ignoreCommand はビルドスキップ判定で別機能
共通原因: 確認せずに推測を事実として断言した。

パイプライン構造

調査・分析・レビューのサブエージェント委任は、以下の 3 ステップで実行する。

Step 1: チェックリスト作成 Skill(subagent-investigation-checklist) で調査チェックリストを作成。 タスク種別（構成レビュー / コード調査 / 仕様調査 / ログ分析）ごとの 必須項目 + タスク固有項目を生成し、worktree 内に配置。 Step 2: 調査実行 Agent(model: opus) にチェックリスト全文を埋め込んで委任。 調査エージェントはチェックリスト項目を 1 つずつ実行し、 各 finding に証拠（コマンド出力）を添付して報告。 Step 3: レビュー Agent(subagent_type: "reviewer") にチェックリスト + 調査結果を渡す。 レビュアーが 4 観点（完了性・証拠・事実性・再現性）で検証し PASS / FAIL を返す。 FAIL なら Step 2 に戻る（最大 2 回）。

チェックリストの配置と破棄

項目内容
配置先 worktree 内の .investigation-checklist.md
破棄 worktree 削除時に自動消滅。手動 rm 不要

機械強制

PreToolUse(Agent) hook: check-investigation-checklist.sh
調査・レビュー系の Agent 委任で prompt 内に ## 調査チェックリスト がなければ [CHECKLIST-MISSING] で exit 2 block。
例外: worker-haiku / routine-worker / [CHECKLIST-EXEMPT] 明示。
