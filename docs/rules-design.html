§1定義

rules とは、カテゴリ別にトピック分割した規約集で、状況に応じて context に注入される。 CLAUDE.md がモノリシックなのに対し、rules はカテゴリ別に育てる。 paths frontmatter の有無で 2 つの注入モード（eager / lazy）を使い分けられる、公式機能。
公式: 「Rules without `paths` frontmatter are loaded at launch with the same priority as `.claude/CLAUDE.md`. Path-scoped rules trigger when Claude reads files matching the pattern, not on every tool use.」
— memory#organize-rules

3 つの責務を 1 つの場所に集約

  • 規約の正本（人間とエージェントの両方が読む長文ルール）
  • 機械強制の手順書（hook が注入するタグに対応する Claude の修正手順）
  • 関連スクリプト（規約を強制する hook script・検査スクリプト本体）
「rule + hook + script」を 1 フォルダに同居させることで、規約とその機械強制を物理的に結びつける。 別フォルダに分散させると、片方の更新がもう片方に反映されずに腐る。

§2ロードタイミング（公式仕様）

2 モードの違い

モード初回ロード再注入トリガーsession 中の総ロード回数
eager
paths 無し		 session 開始時 compaction（手動 /compact + 自動 compact）のたび 1 + compact 回数（CLAUDE.md と同列）
lazy
paths 有り		 マッチするファイル Read 時 compaction で消失 → 次のマッチ Read で再ロード マッチファイル Read 回数（compact 区間ごとにリセット）
公式: 「Path-scoped rules and nested CLAUDE.md files load into message history when their trigger file is read, so compaction summarizes them away with everything else. They reload the next time Claude reads a matching file.」
— context-window#what-survives-compaction

「定期的に注入される」の正体

session start ↓ eager rules（paths 無し）+ CLAUDE.md がディスクから注入 ↓ Read src/api/foo.ts → lazy rule（paths: src/api/**）が初注入 ↓ ... 作業継続 ... ↓ 自動 compaction（context 80% 圧迫） ↓ ★ eager rules は disk から再注入される ↓ lazy rule は一旦消失 ↓ Read src/api/bar.ts → lazy rule が再注入される ↓ ... 作業継続 ... ↓ 自動 compaction ↓ ★ eager rules は disk から再注入される ↓ ...
長尺セッションでは eager rule が 5〜10 回再注入されることがある。「session 開始時に 1 回だけ」は誤り。 これが「カテゴリ別に状況に応じて育てる」設計を支える基盤。

§3記載方針

原則：カテゴリ別に状況に応じて注入される「育てる規約集」として扱う。 CLAUDE.md がユーザー指定の固定内容を保持するのに対し、rules はトピックを増やし・統合し・paths を絞り、エージェントの挙動を観察しながら継続的にチューニングする。

rules に書くべきもの

項目例
機械強制と pair の規約「ルート直下マーカー禁止」「.sh 作成時 ADR 必須」
hook が注入する [] への対応手順[NO-ROOT-MARKER-BLOCK] → 書き出し先を ${TMPDIR}/claude-hooks/${session}/ に変更
例外ホワイトリスト「/tmp/claude-hooks/** は block しない」
違反の背景・なぜ禁止か「過去 821 件のドットファイル蓄積を引き起こした」
カテゴリ別の長文ルールsecurity / hooks-architecture / no-root-marker / naming など
該当タスク中だけ効かせたい規約paths frontmatter 付きで **/*.sh など

rules に書かないもの

  • 全タスク共通の根本姿勢（言語・文体・対話原則）→ CLAUDE.md
  • トリガー駆動の手順（PR レビュー手順・worktree 作成など）→ skills/
  • 1 度しか使わない使い捨て規約 → どこにも書かない

§4カテゴリ設計

1 カテゴリ = 1 フォルダまたは 1 ファイル。カテゴリ名 = 規約の主題。 カテゴリは「禁止事項の領域」で切る（言語・操作対象・違反イベント）。

既存のカテゴリ例

カテゴリ領域形式
security削除操作・スクリプト作成・worktree 畳み単一ファイル security.md
no-root-markerルート直下マーカー禁止単一ファイル no-root-marker.md
hooks-architecturehook script の配置規約フォルダ hooks-architecture-rules/

単一ファイル vs フォルダの選択

形式選択条件
.md
単一ファイル		 規約本体のみ。関連スクリプトを持たない or 持っても 1 本だけで他にメタファイルがない
-rules/
フォルダ		 規約本体 + 複数の hook script・検査スクリプト・ADR を同居させる必要がある

カテゴリ分割の判定

  • 2 つの規約が同じイベント（同じ hook タイミング）で発火するなら 1 カテゴリ
  • 違反検知の [] が異なるなら別カテゴリ
  • 規約本体が 200 行を超え、3 つ以上のサブトピックを含むなら分割を検討

§5paths 戦略

paths frontmatter は「いつ attention に乗せたいか」を制御する道具。 常時必要なら paths 無し、特定タスク中だけ強く効かせたいなら paths 指定。

判定フロー

Q1. この規約は全タスクで違反しうるか？ YES → paths 無し（eager）。CLAUDE.md と同列で常時注入 NO → Q2 へ Q2. 特定の path・拡張子の作業中だけ違反しうるか？ YES → paths 指定（lazy）。該当 Read 時のみ高 attention で注入 NO → Q3 へ Q3. 特定キーワードを含む prompt 時だけ違反しうるか？ YES → rules に置かず、UserPromptSubmit hook の additionalContext で rule.md の該当箇所を抜き出して注入 NO → skill 化を検討

paths frontmatter のサンプル

--- paths: - "**/*.sh" - "**/Makefile" - "**/package.json" --- # シェルスクリプト作成時の規約 - `.sh` の新規作成は ADR 必須 - ...

トレードオフ

方針メリットデメリット
paths 無し（eager） 常に context にある → 違反防止が確実 全タスクで context を消費。compact のたび再注入されるため複数回トークン消費
paths 有り（lazy） 該当タスク中だけ高 attention で乗る。不要タスクでは 0 トークン マッチしないと注入されないため、トリガー漏れがあると違反する
Gotchas
  • compaction で path-scoped rule は消える。再注入には次のマッチ Read が必要
  • 「絶対に守らせたい」規約は paths 無しにする（lazy だと Read 漏れで違反する）
  • paths 指定の glob は **/*.sh のように広めに取ると Read 漏れを防げる

§6配置とフォルダ構造

基本配置

~/.claude/rules/ .md … 単一ファイル規約 -rules/ … 複数ファイル規約 rule.md … 規約本体 .sh … 対になる hook script .sh … 検査スクリプト README.md … 必要に応じて ...

関連スクリプトの同居原則

rules 関連の hook script・検査スクリプトは rules/-rules/ 配下に同居させる。 別フォルダ（~/.claude/hooks/ や <project>/tools/hooks/）に置くと、 規約と機械強制の対応関係が見えなくなり、片方の更新がもう片方に反映されずに腐る。

配置の 4 象限

ownership × scope置き場
独立規約 × global~/.claude/rules/-rules/
独立規約 × project/.claude/rules/-rules/
skill 延長 × global~/.claude/skills//scripts/
skill 延長 × project/.claude/skills//scripts/
判定: 特定 skill の前提を強制するなら skill 延長。単一 skill に紐付かないシステムメタ規約なら独立規約。詳細は Hooks 設計ガイド を参照。

禁止される配置

  • ~/.claude/hooks/ 直下に rules 連動の hook script を新規作成（flat な hooks/ バケット禁止）
  • <project>/tools/hooks/ 直下に新規作成（flat bucket 禁止）
  • rule.md と hook script を別フォルダに分離（同居強制）

§7hook 連携

rules + hook は pair で動作する。rules が「規約と修正手順の正本」を、hook が「違反検知と additionalContext 注入」を担う分業。

標準パターン

[1] hook script が違反コマンドを検知（PreToolUse など） ↓ [2] additionalContext に [] と修正手順の short text を出力 ↓ [3] Claude が を受信し attention 上位に置く ↓ [4] Claude が rule.md 本体（既に context にロード済み）を参照し詳細手順を実行

タグの規約

項目規約
タグ名大文字ハイフン区切り（例: [NO-ROOT-MARKER-BLOCK]）
タグと rule.md の対応1:1。[FOO-BLOCK] なら rules/foo*.md に対応手順がある
additionalContext の長さ短文（300 字以内）。詳細手順は書かず、タグ＋短い指示＋rule.md への path
修正手順の所在rule.md 本体に書く（既に launch 時にロード済みなので Claude が即参照できる）

hook 側に書く内容

例: rules/no-root-marker-rules/no-root-marker-check.sh
#!/bin/sh cmd=$(jq -r '.tool_input.command') if echo "$cmd" | grep -qE 'touch \"?\$HOME/\.claude/\.[a-z-]+'; then cat < に変更。詳細: ~/.claude/rules/no-root-marker.md" }, "permissionDecision": "deny" } EOF exit 0 fi
Gotchas
  • additionalContext に rule.md 全文をコピペしない（毎発火で token を浪費）
  • タグ名と rule.md ファイル名の対応を崩さない（Claude が rule.md を探せなくなる）
  • hook script は decision 系（block / deny）と observability 系（log のみ）を 1 ファイルに混ぜない

§8ルールの育て方

rules はカテゴリ別に「状況に応じて注入されるルール集」として継続的に育てる。 CLAUDE.md がユーザー指定の固定内容を保つのに対し、rules はエージェントの違反パターンを観察しながら成長させる。

追加プロセス

  1. 違反検知: Claude が規約違反を起こすパターンを発見
  2. カテゴリ判定: 既存カテゴリに統合するか、新規カテゴリを作るか
  3. paths 判定: 常時必要か特定 path のみか
  4. rule.md 起草: 違反背景・規約・例外・修正手順を記載
  5. hook 設計: 必要なら hook script を同居フォルダに作成
  6. ADR 作成: 規約追加の必要性・代替案不採用理由・廃棄条件を残す
  7. 実測: 5〜10 セッション運用して違反 0 を確認

削除プロセス

  1. 四半期に 1 回、hook 発火ログを集計
  2. 発火 0 件のカテゴリは「規約が陳腐化」か「Claude が学習した」のいずれか
  3. rule.md・hook script・ADR を同時に削除（孤児を残さない）

カテゴリ統合の判定

  • 2 つのカテゴリの違反が同じ場面で連続発生するようになったら統合検討
  • 統合後の rule.md が 200 行を超えるなら分割を維持
  • 統合後のタグ名は 1 つに集約し、旧タグは hook の中で alias 化

命名規約

対象命名規則例

注入タグUPPER-HYPHEN[NO-ROOT-MARKER-BLOCK]

設計判断の記載先

  • カテゴリ追加時の設計判断は対象の rule.md 内に ## 設計判断 セクションとして記載
  • 必須項目: 必要性 / 代替案不採用理由 / 保守責任者 / 廃棄条件
  • カテゴリ削除時は設計判断セクションも同時削除

§10テンプレート

rule.md の構造

--- paths: # 任意。常時必要なら省略 - "**/*.sh" --- # <カテゴリ名>（） <禁止 / 規約の宣言文（1 段落）> ## 背景 <なぜこの規約が必要か・過去の事故事例> ## 機械強制 | timing | hook | 注入タグ | 挙動 | |---|---|---|---| | PreToolUse | | [] | <検知内容と block 条件> | ## 例外（block されないパス / 条件） <ホワイトリスト> ## 違反検知時の手順 <タグ注入時の Claude の対応手順。具体的な edit が打てる粒度で> ## 関連 <関連 skill / 他 rules / hook script の path>

フォルダ構造の例

~/.claude/rules/ security.md … 単一ファイル形式 no-root-marker.md … 単一ファイル形式 hooks-architecture-rules/ … フォルダ形式 rule.md … 規約本体 hooks-architecture-check.sh … PreToolUse hook README.md … 任意 no-root-marker-rules/ … フォルダ形式（hook 同居版） rule.md no-root-marker-check.sh cleanup-session-markers.sh ※ 設計判断は各 rule.md 内に ## 設計判断 セクションとして記載（独立 ADR ファイルは廃止）

§11アンチパターン

パターン何が悪いか正解
hook script を ~/.claude/hooks/ 直下に置く rule.md との対応が見えなくなり片方が腐る rules/-rules/.sh に同居
1 つの rule.md に 5 カテゴリ詰め込む 200 行超で attention が落ち、違反検知の対応手順が見つからない カテゴリ分割
全 rule に paths を付ける Read 漏れで違反が出る 常時必要な規約は paths 無しで eager
全 rule を paths 無しにする 全タスクで context を圧迫し compact 連動で再注入のたび token 消費 該当 path 限定の規約は paths 指定で lazy 化
additionalContext に rule.md 全文をコピペ 毎発火で token を浪費。rule.md と二重管理で腐る タグ + 短い指示 + rule.md の path だけ
違反検知の hook が決定（deny）と log を 1 ファイルで担当 役割が混在し、変更時の影響範囲が読めない decision 系と observability 系を別 script に分離
ADR なしで .sh や規約を追加 必要性を説明できない規約が残り、削除判断ができなくなる 追加時に ADR を起こす（4 項目）
