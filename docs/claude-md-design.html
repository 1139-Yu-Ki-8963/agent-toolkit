§1定義

CLAUDE.md とは、全タスクに通底する根本姿勢を user message として常時注入する場所。 機械強制を伴わず、Claude が「読む → 解釈する → 従う」というソフトな経路で効く。 ユーザーが明示的に指定した内容だけを記載する、モノリシックな単一ファイル。
公式: 「CLAUDE.md content is delivered as a user message after the system prompt, not as part of the system prompt itself. Claude reads it and tries to follow it, but there's no guarantee of strict compliance.」
— memory#troubleshoot

§2ロードタイミング（公式仕様）

イベント挙動
session 開始ディスクから読まれ、user message として context に注入
手動 /compactproject-root CLAUDE.md は ディスクから再注入
自動 compaction同上（context 圧迫時にハーネスが自動で走らせる）
nested CLAUDE.md（サブディレクトリ）該当ディレクトリ Read 時にのみ遅延ロード。compaction で消失し、再 Read で復活
subagent 起動subagent は自分の独立 context で CLAUDE.md を別途ロード（親の context は使わない）

session 中の総ロード回数

root CLAUDE.md の総ロード回数 ＝ 1 + compaction 発生回数。8 時間の長尺セッションでは 4〜5 回再ロードされることがある。

「session 開始時に 1 回だけロード」は誤り。長尺セッションでは compaction 連動で繰り返し disk から再注入される。

§3記載方針

原則：ユーザーが明示的に「ここに書け」と指定した内容だけを書く。 Claude が自律的に CLAUDE.md を肥大化させない。「念のため書いておく」「将来必要かもしれない」は禁止。

CLAUDE.md に書くべきもの

項目例
言語規約「思考は英語で、ユーザーとのやり取りは日本語で行う」
文体の根本姿勢「受動態を能動表現に変える」「推測を断言しない」
対話の根本姿勢「指示に不足がある場合は 1 問ずつ確認する」
コーディング原則YAGNI / KISS / DRY / 根本原因主義
正本ポインタ「禁止事項は ~/.claude/rules/*.md を参照」

共通する性質

  • 機械強制が不可能（言語の選択や文体の柔らかさは hook で弾けない）
  • 全タスクに通底する（トリガー駆動ではない）
  • Claude のデフォルトと異なる（明示しないとデフォルト挙動に流れる）
  • ユーザーが「これは書く」と明示的に指定したもの

§4書く / 書かない判定

判定フロー

Q1. ユーザーが「CLAUDE.md に書け」と明示したか？ NO → 書かない（自律追加禁止） YES → Q2 へ Q2. それは機械強制（hook / permissions / CI / lint）で技術的に強制できるか？ YES → rules/ + hook へ誘導（CLAUDE.md には書かない） NO → Q3 へ Q3. それは特定トリガー時だけ必要か？ YES → skill にする（CLAUDE.md には書かない） NO（全タスク常時必要） → Q4 へ Q4. それは Claude がデフォルトで既に知っているか？ YES → 書かない（dead code になる） NO → ✅ CLAUDE.md に書く

除外則（当たったら即・非 CLAUDE.md）

  • 機械強制可能なもの → rules/ + hook
  • トリガー駆動のもの → skills/
  • Claude のデフォルトと同じもの → 書かない
  • ユーザーが明示指定していないもの → 書かない

遵守率の実測ルール

新規の行動規約は実測してから残す。 「気をつける」「丁寧に」のような検証不能な動詞、attention が落ちる場面で違反する規約は、CLAUDE.md には残さず rules/ + hook へ作り替える。

§5サイズと構造

項目制約
推奨上限200 行（公式推奨：「Files over 200 lines consume more context and may reduce adherence」）
構造markdown ヘッダで明示的にグループ化
具体性「2 スペース字下げ」のような検証可能な動詞で書く
一貫性用語は 1 概念 1 語で統一

長くなったときの対応

  1. 機械強制可能な項目を rules/ へ切り出す
  2. トピック分割できる項目を rules/<カテゴリ>.md へ移す（同じ launch ロード扱いになる）
  3. 条件付き発火項目は path-scoped rule または skill 化
Gotchas
  • 「常に context にロードされる」≠「常に遵守される」: 200 行を超えたあたりから個別ルールの attention が下がる
  • 長くするほど、後半の指示は context 中盤に埋もれて効かなくなる
  • 「定期的に削除する」を明文化しないと broken windows 効果で全体の遵守率が落ちる

§6正本の決定と重複の禁止

同じ規約を CLAUDE.md と rules/ の両方に書くと、片方が腐った瞬間に矛盾する。正本は 1 つに決める。
規約の種類正本
機械強制を伴う禁止 / 規約rules/.md
トリガー駆動の手順・知識skills//SKILL.md
hook の挙動・注入タグhook script のヘッダコメント
上記いずれでもない、全タスク共通の根本姿勢CLAUDE.md

CLAUDE.md からは正本へのポインタだけ残す

禁止事項・ふるまい規約は `~/.claude/rules/*.md` を正本とする。 hook 注入タグへの対応手順は、注入されたメッセージと参照先 rules/skills を読む。
手順を CLAUDE.md に重複展開しない。発見次第 rules/ または skills/ に移し、CLAUDE.md からは削除する。

§7メンテナンス

頻度アクション
毎週CLAUDE.md の行数チェック。200 行超なら切り出し検討
規約追加時ユーザー指示があったかを確認。なければ書かない
規約削除時陳腐化した指示は即削除（残すと broken windows 効果）
四半期遵守率が低い規約を抽出し、rules/ + hook へ作り替えるか削除

追加前チェックリスト

  1. ユーザーが「CLAUDE.md に書け」と明示したか？
  2. 機械強制できないか？できるなら rules/ + hook へ
  3. skill 化できないか？できるなら skill へ
  4. デフォルトと同じか？同じなら書かない
  5. 検証可能な動詞で書けているか？

§8アンチパターン

パターン何が悪いか正解
CLAUDE.md に rm -rf 禁止を書く hook で block できる。書いても context 中盤で attention が落ちて違反する rules/security.md + permissions.deny
CLAUDE.md にスキル一覧を書く skill description で自動 expose される。重複は腐る 書かない
「丁寧に対応する」のような検証不能な規約 遵守率が測れず dead code 化 検証可能な動詞に書き換える or 削除
同じ禁止事項を CLAUDE.md と rules/ に両方書く 片方が腐ったとき矛盾する rules/ を正本とし CLAUDE.md からはポインタのみ
CLAUDE.md が 300 行を超える attention 低下で個別ルールが効かなくなる 機械強制可能なものを rules/ へ切り出す
Claude が自律的に「念のため」追加 ユーザー指定主義に反する。肥大化の原因 追加前にユーザー確認、または rules/ への提案にとどめる
