§1工程フロー図（成果物の積み上がり）

指揮役（orchestrating-reverse-docs-flow）が左列の工程を上から順に進める。各工程が完了するたびに、右列の成果物が docs_root に増える。緑枠の「ユニット反復」は画面 1 つごとに繰り返す。

本文書で使う「facts」は、原本コードから抽出した宣言的契約の事実表（項目・分岐・呼び出しの数まで数え上げた一次資料）を指す。「基準タグ」は、2 環境の完全一致を証明できた時点のコミットに打つ git タグ reverse-baseline/<scope> を指す。

Phase 0（ヒアリング）で対象パス・出力先・画面スコープを確定後、Phase 1 で状態判定を行い全工程をタスク一覧として一括登録する（マニフェスト先出し方式）。「一覧未生成」状態の解決は Phase 1.5 で、実在する種別（unit_kinds_present）の一覧生成スキルのみを Agent ツールで並列起動する。基本設計と詳細設計も同じ並列パターンで実行する。画面一覧の確立後、Phase 1.7 で機能一覧（派生一覧）を生成する。

§5責務境界の確定事項

工程をまたいで迷いやすい「どちらのスキルが持つか」の確定事項。ここに挙げた 7 点は仕様として確定しており、変更する場合は contract.md と各スキルガイドを同時に改訂する。

本HTMLの保守規約
正本はこのファイル自身（reverse-docs-overview.html）。更新契機はヘッダの meta-table に記載のとおりスキル構成・契約（contract.md）変更時。§3 の実装状況タグ（済／一覧のみ／段階計画）と §4 の対象外タグ（対象外）は、各スキル SKILL.md・contract.md の実装状況と一致させる。スキルの新設・削除・状態遷移の変更時は、本ファイルの該当タグを同一 commit 内で追従させ、陳腐化した記載を放置しない。
scaffold の正本
画面の骨組み生成スクリプトの正本は shared/scripts/scaffold-screen.sh の 1 本のみ。利用側スキルは引数渡しで呼び出し、複製・改変版を持たない。
単体テスト正本の一本化
単体テストコードの正本は、ファイル単位検証（rebuilding-screen-unit-from-docs）が保存した saved_test_paths（設計書リポジトリの テスト項目書/テストコード/単体/ 配下）の 1 系統のみ。画面単位検証（rebuilding-code-from-docs）はこれを対象コードリポジトリへ配置して実行するだけで、単体テストコードの新規作成は行わない。
E2E の責務分離
E2E（画面を実際に操作して端から端まで通す試験）は、作成が rebuilding-code-from-docs（mode=implement）、実行が syncing-reverse-env の動的比較。作る側と走らせる側を分けることで、試験内容が比較判定に引きずられない。
status の解釈権
syncing-reverse-env の返す PASS / FAIL は計測事実（一致したか）であり、意味の解釈（設計書の欠落か・許容差分か）は rebuilding-code-from-docs（mode=judge）が持つ。計測と解釈を混ぜない。
unlocking の単独完走
unlocking-reverse-target-screens が画面レジストリ記帳から syncing-reverse-env（mode=registry） 起動・基準タグ確立まで単独で完走するのは意図した正式仕様。開通の事実を知るのは本スキルだけであり、指揮役が能動的に検知できないため。
画面レジストリの正本
開通済み画面の記帳台帳の正本は <docs_root>/一覧/reverse-screen-registry.yml。設計書リポジトリ側に置くため、スキル同期・上書きコピーの影響を受けない。unlocking 以外の子スキルは直接触れない。
