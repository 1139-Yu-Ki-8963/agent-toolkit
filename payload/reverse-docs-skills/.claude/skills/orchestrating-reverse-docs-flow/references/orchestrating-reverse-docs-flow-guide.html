§1概要

orchestrating-reverse-docs-flow はリバース設計書往復検証フローの進行係（管理者）。 自分では検証・比較・実装を一切行わず、状態判定 → 子スキルを args 全量指定で Skill 起動 → 返却ブロックの status で検収 → 次工程決定、というループで工程全体を統括する。

子スキル4つ（generating-screen-list-for-reverse-docs / syncing-reverse-env / rebuilding-screen-unit-from-docs / rebuilding-code-from-docs）は互いを知らず、 工程間の受け渡しはすべて本スキルが仲介する（完全仲介方式）。契約の正本は references/contract.md。

使用タイミング / スキップ条件
使用する: リバース検証を工程統括したいとき（画面一覧生成から基準タグ確立までの一連の流れ）。
スキップする: 個別工程だけを動かしたい場合。各子スキルは同じ args を手渡せば単独でも動く契約のため、単体実行なら子スキルを直接起動する。

§2状態判定表と決定木（S0/S0u/S1a/S1b/S2/S3/S4）

成果物の実在から現在の状態（S0/S0u/S1a/S1b/S2/S3/S4）を判定し、次に起動する子スキルを機械的に決定する。 判定は「画面一覧HTMLの実在 → 画面開通有無 → 設計書/対象ファイル/著者スキルの完全性ゲート成果物の実在 → 検証記録の再現一致有無 → ②setup返却の baseline_tag → ④judge の status」の順に降りる決定木で、7状態を漏れなく被覆する。 成果物の実在から毎回評価するので中断後も再開できる。

状態キー判定の要点次に起動する子スキル
S0 画面未列挙 画面一覧HTMLが不在 generating-screen-list-for-reverse-docs
S0u 画面未開通 画面一覧HTML有・画面が未開通（設計書も基準タグも無い新規画面） unlocking-reverse-target-screens → syncing-reverse-env（mode=registry）
S1a 設計書未著述 画面開通済み・画面ディレクトリ不在 or §15.1に対象ファイル行なし or 著者スキルの完全性ゲート成果物不在（任意工程） authoring-screen-docs-from-code
S1b ファイル単位未検証 著述済み（S1aの成果物実在）かつ検証記録に再現一致なし（任意工程） rebuilding-screen-unit-from-docs
S2 基準未確立 設計書有・baseline_tag 未確立 syncing-reverse-env（mode=setup → sync）
S3 往復未検証 baseline_tag 有・reverse未実装 or 未突合 rebuilding-code-from-docs（implement）→ syncing-reverse-env（sync,dry-run）→ rebuilding-code-from-docs（judge）
S4 検証完了 judge の status=PASS syncing-reverse-env（mode=sync 本番 / 依頼時 teardown）

S1b が status=差し戻し（内容起因: 設計書に対象契約なし等）を返した場合は S1a へ戻す。target_repo_path 等の必須引数欠落による起動不可は status=差し戻し を返さない Skill 呼び出し失敗として扱われ、このループの対象外である。S1a/S1b は任意工程。設計書が揃い検証記録に再現一致がある画面はファイル単位工程をスキップし S2/S3 から開始してよい。

§39 Phase のワークフロー

Phase内容実行条件
Phase 1状態判定。実在判定を決定木の順に確認し S0/S0u/S1a/S1b/S2/S3/S4 を確定。必要な工程だけを 1 つずつ TaskCreate（一括登録禁止）常時
Phase 2①画面一覧生成。generating-screen-list-for-reverse-docs を source_dir・output_dir 指定で起動S0 時のみ
Phase 3②setup（環境ブロック取得）。syncing-reverse-env を design-doc・mode=setup で起動し env_block を抽出常時
Phase 4③ファイル単位検証。(a)S0u画面未開通なら unlocking-reverse-target-screens→syncing-reverse-env(registry)。(b)S1a設計書未著述なら authoring-screen-docs-from-code を起動。(c)S1bファイル単位未検証なら rebuilding-screen-unit-from-docs を対象ファイル1件ごとに繰り返し起動。白紙化のため事前にユーザー承認を取得。status=差し戻し なら(b)へ戻すS1a/S1b（任意工程）
Phase 5②sync（基準確立）。syncing-reverse-env を design-doc・mode=sync で起動し baseline_tag を確立S2 時のみ
Phase 6④implement。rebuilding-code-from-docs を mode=implement で起動し compare_request を取得S3 時のみ
Phase 7②sync dry-run（比較）。syncing-reverse-env を design-doc・mode=sync・dry-run で起動し比較結果ブロックを取得常時（Phase 6 の後）
Phase 8④judge。rebuilding-code-from-docs を mode=judge・compare_result・reverse_worktree・freeze_commit で起動し PASS/FAIL を確定常時（Phase 7 の後）
Phase 9②sync本番/teardown。syncing-reverse-env を mode=sync・user-approved で起動し基準タグを本番更新。検証終了依頼時は mode=teardownPhase 8 が PASS のとき（S4）
Phase 6〜8 の関係
④（rebuilding-code-from-docs）は mode で2分割される（implement=比較要求を返して停止 / judge=比較結果を受け取り判定）。 管理者がこの2回を別々に起動し、間に②sync dry-run（Phase 7）を挟む。

重要な注意事項
白紙化などの破壊的操作のユーザー承認は管理者が事前に取り、user-approved として args で渡す（子はユーザーに直接聞かない）。
docs_root が null のときの展開先確認も管理者が担う。
各子スキルは単独起動可能（ユーザーが同じ args を手渡せば動く）。工程順序を知るのは管理者だけ。

§5返却ブロック契約

正本は references/contract.md。管理者は各子スキルの返却ブロックを共通サブセットで検収し、状態判定表に従って次工程を機械的に決定する。 これにより管理者と子スキルの間には契約書という単一の仲介点だけが存在し、子スキル同士が互いの内部仕様を知る必要がない完全仲介方式が成立する。

返却ブロック共通サブセット

キー型意味
statusenum各スキル固有の enum。管理者はこれで go / 差し戻しを判定する
scopestring<system>-<画面ID>。工程を跨いだ同一性キー
artifactspath[]生成物のパス（画面一覧HTML / 修正指示書.md / 最終報告.md / 証跡ディレクトリ 等）
hintstring次工程への申し送り（差し戻し理由・不足引数・後続工程への申し送り 等）

注記: next_request という語は使わない。syncing-reverse-env の実返却ブロックに存在しないフィールドのため。次工程への依頼内容は hint または各スキル固有の拡張フィールドで表す。

各スキル固有の status enum と拡張フィールド

子スキルstatus enum拡張フィールド
generating-screen-list-for-reverse-docs DONE | ERROR screen_list_html（= artifacts[0]）、embedded_json_ref（HTML内埋め込みマニフェストJSONへの参照）
syncing-reverse-env PASS | FAIL | ERROR | INCOMPLETE 既存の15フィールド返却ブロック（status / mode / scope / screen_id / slot / ports / original_code / reverse_code / baseline_tag / static_diff / dynamic / env_check / artifacts / docs_root / hint）はそのまま正で改変しない。 管理者が読むのは status / scope / ports / baseline_tag / docs_root / artifacts / hint。 mode=setup の返却からは env_block として docs_root / scope / ports / slot / baseline_tag / original_code / reverse_code を抽出する。 mode=sync,dry-run の返却が「比較結果ブロック」= static_diff / dynamic / env_check / status / hint を含む15フィールドそのもの。
authoring-screen-docs-from-code（設計書著述） AUTHORED | BLOCKED fact_table_path（事実表 fact-table.md のパス）、measurement_pending（実測委譲項目一覧）
rebuilding-screen-unit-from-docs（ファイル単位検証） 差し戻し | 再現一致 | 再現不一致 instruction_doc（修正指示書.md のパス）、handoff_files（画面単位検証で要注意なファイル一覧）、saved_test_paths（本スキルが保存した最終テストコードのパス一覧）。status=差し戻し（内容起因のみ）の場合、hint に「authoring-screen-docs-from-code を実行してから再起動せよ」等の理由を記録する。必須引数欠落による起動不可は status=差し戻し を返さない
rebuilding-code-from-docs（画面単位検証・mode分割） mode=implement: NEED-COMPARE | INTERNAL-CONTRADICTION | ERROR | BLOCKED
mode=judge: PASS | FAIL | DESIGN-INCOMPLETE | DYNAMIC-UNVERIFIED 		mode=implement: compare_request { scope, design_doc（パス）, freeze_commit（凍結コミットのハッシュ）, scenarios_ready（真偽値）}。これは「比較してほしい」を表す専用フィールドで、管理者はこれを見て syncing-reverse-env を sync,dry-run で起動する。
mode=judge: 入力 args として compare_result（= syncing-reverse-env の sync,dry-run 返却ブロック全文）を受け取る。拡張: instruction_doc（修正指示書.md）、final_report（最終報告.md）、loop_verdict（発散 / 収束の判定）

§7ループ設計

要素内容
反復条件Phase 8 ④judge が FAIL → Phase 6 ④implement へ戻す
上限回数max_loop（既定3。②の max_loop とは別軸の工程ループ）
停止条件 ① 収束停止: 全対象画面が PASS（2連続で確定）
② リソース上限: max_loop 到達で FAIL 確定
③ 発散検知: ④judge が2連続同一差分（compare_result の static_diff 署名一致）で上限前に打切り
検証役の分離各工程の判定は子スキルの返却ブロック（status）のみで行い、管理者は自然文で判定しない

この外側ループ（発散判定2連続・上限）は、元々④（rebuilding-code-from-docs）が持っていた責務を管理者へ移管したものである。

Phase 4 内: S1a⇄S1b ループ

要素内容
反復条件rebuilding-screen-unit-from-docs（S1b）が status=差し戻し を返したら authoring-screen-docs-from-code（S1a）へ戻し、再著述後に S1b を再実行する
上限回数5回目安（rebuilding-screen-unit-from-docs 自身の内側ループ上限と揃える）
停止条件① 収束停止: rebuilding-screen-unit-from-docs が status=再現一致 を返す
② リソース上限: 5回到達しても差し戻しが続く場合はユーザーに報告する
検証役の分離S1a（著述）と S1b（盲検検証）は別スキル・別セッションで実行され、S1b の判定は自身の完全性ゲート・6計測の決定的出力のみで行う
