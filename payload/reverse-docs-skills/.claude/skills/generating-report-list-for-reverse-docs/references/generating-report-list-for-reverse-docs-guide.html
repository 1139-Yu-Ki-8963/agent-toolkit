帳票一覧生成スキル
generating-report-list-for-reverse-docs

既存コードベースを調査して帳票の単位を洗い出し、整合検証を通過したマニフェストだけから 帳票一覧.html を生成する帳票専用スキルの確定仕様。

対応 OSWindows WSL2 環境
検証状況共有エンジン（validate-manifest.sh・build-unit-list.sh）は分割元スキルの画面一覧生成で動作確認済み。帳票専用スキルとしての単独実機検証は分割直後のため未実施
依存jq ／ 共有スクリプト shared/scripts/unit-list/validate-manifest.sh・build-unit-list.sh

§1概要

リバース設計の出発点は「このシステムには帳票が何枚あり、それぞれどのファイルでできているか」を正確に確定させることにある。本スキルは既存コードベースを スタック調査 → 検出戦略の宣言 → 戦略に基づく抽出 → 整合検証 の順で調べ、帳票の単位にファイルをグルーピングした帳票一覧.html を作成する。

帳票はプロジェクトごとに実現方式が大きく異なる（テンプレートファイル主体・生成コード主体・両者の複合）。そのため画面一覧のような機械的な組み込み検出器は持たず、対象プロジェクト固有の帳票規約を先に調査・宣言してから、その宣言に沿って Claude 自身が抽出するカスタム抽出パスだけを使う。抽出結果は共有の検証スクリプトが機械検査し、検証を通らないデータからは一覧を生成できない。

本スキルがやること
帳票の洗い出し・マニフェスト作成・整合検証・帳票一覧.html の生成
本スキルがやらないこと
帳票設計書の雛形展開・生成・記入（後続工程の担当）／ 他種別（画面・API 等）の一覧生成（各種別別一覧スキルの担当）

起動引数

引数内容
source_dir探索対象のソースコードディレクトリ。モノレポの場合は対象アプリのディレクトリのみを渡す
output_dir帳票一覧.html の書き出し先ディレクトリ

種別を切り替える引数は無い。本スキルは帳票（unit_kind=report）固定で動作する。

§2出力

このスキルを実行すると、帳票 1 種別につき以下の 2 点が生まれる。

帳票一覧.html（<output_dir>/帳票一覧/帳票一覧.html）
システム内の全帳票を 1 行 1 帳票で並べた一覧表。帳票ごとの推定名・主ファイル・構成ファイル数・検出手法・確度が読め、「帳票設計書を何冊、どの単位で書くべきか」を決める正本として機能する。未解決（主ファイルを特定できなかった帳票）や診断警告も隠さず可視化される。
検証済みマニフェスト（HTML に埋め込み）
一覧の元データである JSON。整合検証スクリプトの全項目を通過したものだけが HTML に埋め込まれるため、一覧の各行は「機械検査済みの事実」として後続工程（設計書執筆・往復検証）から参照できる。

§3用語

帳票（unit_kind=report）
システムが出力する定型文書（PDF・Excel・CSV・HTML 等）の 1 単位。テンプレートファイル 1 枚が 1 帳票のこともあれば、生成関数 1 つが 1 帳票のこともあり、その数え方自体を Phase 1 で確定させる。
検出戦略宣言
Phase 1 の調査結果を固めた JSON（unitKind／extractionMethod／unitIdRegex／excludePatterns／approvedByUser／notes）。ユーザー承認（approvedByUser: true）が無いと後段の検証を通らない。
カスタム抽出パス
戦略宣言に沿って Claude 自身が抽出手順を設計・実行し、マニフェスト JSON を書き出す経路。帳票にはこの経路しか無い（組み込み検出器は画面専用）。
マニフェスト
抽出結果の JSON。配列キーは units で、各要素が 1 帳票（unitKey・sourceFile・kind・files 等）。スキーマの全文は references/report-detection.md を参照。
kind 区分値
帳票の実現方式の区分。template（テンプレートファイル主体）／generator（生成コード主体）／unresolved（主ファイル未解決の隔離用）の 3 値。
共有エンジン
整合検証 validate-manifest.sh と HTML 生成 build-unit-list.sh。種別別一覧スキル群が共用する決定的スクリプトで、スキルフォルダから ../../../shared/scripts/unit-list/ の相対パスで参照する（正本リポジトリと公開先はレイアウト同一のため両環境で成立する）。

§4動作フロー（Phase 1〜4）

── 帳票一覧ができるまで ─────────────────────────────

  Phase 1  スタック・帳票規約の特定
           帳票生成ライブラリ／テンプレートの所在／ID 命名／除外パターン
           └─▶ 検出戦略宣言（JSON）をユーザー承認
                    │
  Phase 2  戦略に基づく抽出（カスタム抽出パスのみ）
           宣言どおりに走査し、マニフェスト JSON を書き出す
           0 件検出ならここでハード停止（帳票を捏造しない）
                    │
  Phase 3  整合検証（機械実行）
           validate-manifest.sh --unit-kind report
           FAIL ─▶ 修正して再検証（3 回失敗で Phase 2 へ差し戻し）
                    │ PASS
  Phase 4  帳票一覧.html 生成
           build-unit-list.sh（内部で validate を再実行）
           └─▶ <output_dir>/帳票一覧/帳票一覧.html
Phaseやること完了条件
Phase 1
調査と戦略宣言		 package.json・lock ファイルから帳票生成ライブラリを確定し、帳票定義の所在（テンプレート／生成コード／定義設定）・ID 命名パターン・除外パターンを調査する。調査項目の正本は references/report-detection.md 検出戦略宣言 JSON がユーザー承認済み（approvedByUser: true）
Phase 2
抽出		 宣言した手順を Bash/Grep/Read で実行し、スキーマ準拠のマニフェスト JSON を一時ディレクトリに書き出す マニフェストが 1 件以上確定、または 0 件検出を報告して停止している
Phase 3
整合検証		 validate-manifest.sh <manifest.json> --unit-kind report を実行し、FAIL は指摘に応じて修正する 全項目 PASS（修正ループは 3 回以内）
Phase 4
HTML 生成		 build-unit-list.sh <manifest.json> <output_dir>/帳票一覧/帳票一覧.html --unit-kind report を実行する 帳票一覧.html が生成され、埋め込み JSON がマニフェストと一致している

進捗は TaskCreate/TaskUpdate で Phase 1〜4 を登録・更新しながら進める。完了時は orchestrating-reverse-docs-flow の契約に沿って status（DONE ／ ERROR）・artifacts・unit_list_html・embedded_json_ref・unit_kind=report を返す。

§5守り（禁止事項と機械検証）

手作業でのプレースホルダ置換は禁止。 帳票一覧.html は必ず build-unit-list.sh 経由の決定的処理で生成する。過去に手作業置換で entryFile=None という不正データが 10 件混入した実害があり、build 側が内部で validate を再実行することで「検証を経ないデータからは一覧を作れない」構造にしている。
0 件検出はハード停止。 帳票が 1 件も見つからない場合、AskUserQuestion で手動リストを聞き出して即興の境界を確定させることはしない。ユーザーに報告して停止し、戦略の見直しから再開する。
設計書には触れない。 帳票設計書の雛形展開・生成・記入はスコープ外。本スキルの成果物は帳票一覧.html のみ。

そのほかの注意

注意理由
共有エンジンは jq に依存する未インストール環境では事前導入が必要
コメントアウトされた帳票定義は抽出前に除去するコメント内の定義を実在として誤検出した実害がある
汎用の unitIdRegex を無調査で当てない帳票の命名規約・生成方式はプロジェクトごとに異なる
マニフェストの配列キーは unitsscreens は画面一覧専用の後方互換キー
