機能一覧生成スキル
generating-feature-list-for-reverse-docs

既存の種別別一覧（画面一覧は必須、API・テーブル等の一覧は任意）とソースコード構造を突合し、業務機能の単位（大分類 + 機能の2階層）に画面・API・テーブルをグルーピングした「機能一覧.html」を、機械検証を通したマニフェストだけから生成するスキルの確定仕様。他の種別別一覧を直接検出せず、既存一覧から派生させる点が本スキル固有の性質である。

§1概要

本スキルは「機能一覧」という 派生一覧 を生成する。他の種別別一覧（画面・API・テーブル等）がソースコードから直接ユニットを検出するのに対し、機能一覧は既存の一覧マニフェスト（画面一覧は必須入力、API・テーブル等の一覧は任意入力）とコード構造（ルート定義・ナビメニュー・APIプレフィックス・ディレクトリ構成）を突合し、業務機能の単位でグルーピングし直す。

機能は 2 階層で構成する。第 1 階層は 大分類（ルートprefix第1セグメント単位の業務領域）、第 2 階層は 機能（同一業務対象への操作一式）である。手順は「入力収集 → 大分類候補の導出 → 機能の導出と関連付け → ユーザー承認 → 整合検証とHTML生成」の順で進む。

派生一覧であるため、アーキテクチャ調査の存在判定（unit_kinds_present）の対象にならない（機能は常に存在するため判定不要）。excluded-kinds.json の allKinds にも含めない。

責務境界

担当責務
本スキル
generating-feature-list-for-reverse-docs		 機能一覧.html の生成のみ。既存一覧マニフェストの読み込み、大分類・機能のグルーピング、related* の突合、整合検証とHTML生成を担う
担当外 設計書の雛形展開・生成・記入（後続工程が担う）。画面・API・テーブル等 6 種の一覧生成（それぞれの種別別一覧スキルが担う）。往復検証・実装への反映は一切行わない

§2出力

このスキルを実行すると、以下の 2 点が生まれる。

機能一覧.html（<output_dir>/機能一覧/機能一覧.html）
大分類ごとに機能と関連画面・関連API・関連テーブルの対応が読める横断目録。読んだ人は「このシステムが業務として何ができるか」を、画面や API の技術的な単位ではなく業務機能の単位で一覧として把握できる。割当根拠が得られなかった unresolved 行も隠さず可視化され、後続工程の要手動確認事項として引き継がれる。
検証済みマニフェスト（HTML 内に埋め込み）
一覧の元データである JSON。機械可読な機能ごとのグルーピング結果を持ち、整合検証スクリプトの全項目 PASS と related* 参照実在検査を通過したものだけが HTML に埋め込まれる。後続工程やスクリプトがそのまま読み取れる。

§3用語

機能
同一業務対象への操作一式（CRUD集約）。本スキルが数え上げる対象の最小単位。画面 1〜5 枚につき 1 件が目安。
大分類
ルートprefix第1セグメント単位で境界を引く業務領域。機能を束ねる第 1 階層。期待数は 5〜10。
派生一覧
ソースコードから直接検出するのではなく、既存の種別別一覧マニフェストをグルーピングし直して作る一覧。機能一覧はこの唯一の実例。
unresolved（未解決）
大分類または機能への割当根拠がゼロだった画面・APIの行。捏造して割り当てず、要手動確認として可視化する。
related*
relatedScreens・relatedApis・relatedTables。機能マニフェストが持つ、他一覧マニフェストのキー（screenKey / unitKey）への参照。解決できないものは空のままとする。

Phase内容完了条件
Phase 1
入力収集		 既存一覧HTMLを列挙し埋め込みマニフェストを抽出（画面一覧は必須）。source_dir からルート定義・ナビ・APIプレフィックス・ディレクトリ構造を特定する 画面一覧マニフェスト抽出済み・手がかり①〜④の抽出元が列挙済み
Phase 2
大分類候補の導出		 ルートprefix第1セグメントで境界を引き、APIプレフィックス・ディレクトリ構造で裏取りする。ナビメニューは命名のみに使う 全画面が大分類候補または unresolved 候補に分類済み
Phase 3
機能の導出と関連付け		 大分類内で画面群を機能単位に分割し、related* を突合してマニフェスト JSON を出力する 全画面が relatedScreens または unresolved に載り、マニフェスト JSON が生成済み
Phase 4
戦略・構成のユーザー承認		 大分類と機能の構成案（unresolved 行含む）を AskUserQuestion で提示し承認を得る。修正指示があれば Phase 2〜3 を再実行 approvedByUser: true が記録済み
Phase 5
検証とHTML生成（機械実行）		 validate-manifest.sh --unit-kind feature を実行し、related* 参照実在の jq 自前検査を行い、build-unit-list.sh --unit-kind feature で機能一覧.html を生成する 全項目 PASS・related* 参照実在検査 PASS・機能一覧.html 生成済み

§5検証内容

グルーピングが Claude の手作業であっても品質を機械保証するため、HTML になる前に必ず 3 段の検証を通す。

段実行主体検査内容
整合検証 validate-manifest.sh --unit-kind feature マニフェストのスキーマ準拠・重複キー・未解決行の隔離・承認済みフラグ（approvedByUser: true）の存在を、7 項目にわたり検査する
related* 参照実在検査 jq 自前検査（Phase 5 Step 2） relatedScreens/relatedApis/relatedTables が参照先一覧の screenKey/unitKey に実在するかを comm -23 で検査する。validate-manifest.sh の参照整合検査は screen 専用のため、本スキルが自前で担う
完全性検査 Phase 3 完了条件 全 screenKey がいずれかの機能の relatedScreens または unresolved に載っていること（取りこぼしゼロ）を確認する
エンジンスクリプトの参照方法: エンジンスクリプトはスキルフォルダからの相対パス ../../../shared/scripts/unit-list/ で参照する。正本リポジトリと公開先はディレクトリレイアウトが同一のため、この相対参照は両環境でそのまま成立する。jq が必要（未導入環境では事前に導入する）。

§6禁止事項と注意点

項目内容
手作業HTML組み立ての禁止HTML はスクリプト経由の決定的処理でのみ生成する。build-unit-list.sh を必ず経由する
機能・大分類の捏造禁止すべての割当に手がかり①〜④いずれかの根拠を持たせる。根拠ゼロは unresolved とする
related* の推測埋め禁止突合で解決できないものは空のままとする。推測で埋めない
ナビを境界決定に使わない大分類の境界はルートprefix（手がかり①）のみで引く。ナビメニューの表示文言は名前付けのみに使い、境界の決定根拠にはしない（ナビは全機能を網羅しないため）
unresolved 残置は完了扱いunresolved が残った状態も status=DONE で完了とする（既存6種の「要手動確認」と同じ扱い。ERROR ではない）
設計書の雛形展開はスコープ外本スキルの成果物は機能一覧.html のみ。設計書の生成・記入は後続工程が担う
画面一覧が空の場合のハード停止画面一覧が空（screens が 0 件）の場合はハード停止しユーザーに報告する。手動リストを聞き出さない
