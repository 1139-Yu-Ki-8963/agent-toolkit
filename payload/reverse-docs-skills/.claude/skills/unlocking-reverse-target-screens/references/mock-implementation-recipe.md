# モック実装レシピ（Phase 3 の詳細手順）

`SKILL.md` の Phase 3（画面別モック実装）が参照する具体的な作業レシピ。Phase 1〜3
本文で述べた完了条件（登録名一覧×実名一覧の突合差分ゼロ、サーバー再起動後の反映確認）
はここでは繰り返さない。ここでは「どう実装するか」「起きやすい失敗をどう避けるか」に
絞る。

## 返却値は都度、正本から書き写す

APIの返却値は、既存の別モックの形状を流用してはならない。似た名前・似た用途のモックが
既にあっても、その形状をコピーして書き換える方法は取らない。都度 `api_source_of_truth.
definition_root`（Phase 1で確認した正本）を開き、そこに書かれているフィールド・型・
ネスト構造をそのまま書き写す。流用は「たまたま今回は当たっている偶然の一致」を「検証
済みの一致」と誤認させ、後続画面で必ず破綻する。

## 消費側から逆引きして修正対象を確定する

期待したフィールドがセッションやストアに反映されない場合、直接そのモックを疑って値を
書き換えることを最初にしてはならない。まず、そのフィールドを最終的に書き込んでいる処理
（store の setter、reducer、セッション書き込み関数等）をソースから特定し、そこから逆に
辿って実際にどのAPI呼び出し・どのモックハンドラが消費されているのかを確定する。手順:

1. 反映されるべきフィールド名で、それを書き込んでいる箇所（store/session）をGrepする
2. その書き込み処理を呼び出しているAPIレスポンスの受信箇所を特定する
3. そのAPI呼び出しが実際にどのモックハンドラにヒットしているかを確認する
4. 3で確定したハンドラだけを修正する

**この手順を飛ばすと、名前が似た別の未使用モック（呼ばれていないモック）を誤って修正
し、何度直しても反映されないという失敗が起きやすい。** 修正前に必ず「そのモックが実際
に呼ばれているか」をブラウザ自動化のネットワーク記録で確認する。

## 既存共通モックの登録名も実名と突合する

登録名×実名の突合は、今回新規に実装したモックだけを対象にしてはならない。複数画面で
使い回される既存の共通モックファイルには、登録済みのオペレーション名が実際にフックが
発行する名前と食い違ったまま放置され、登録されているのに一度もマッチしない「死んだ
モック」が潜在することがある。対象画面が依存する共通フックについては、既存共通モック
側の登録名も都度、実行時のネットワーク記録で実際に発行された名前と突合する。「共通
ファイルは他画面での実績があるから正しい」という前提を置かない。

前節「消費側から逆引きして修正対象を確定する」が「自分の修正が反映されない（似た別の
モックを直している）」場面の切り分けを担うのに対し、本節は「そもそも一度もマッチして
いない既存登録」を発見する検査を担う。

## 共通モックヘルパーに固定値を持たせない（ドメイン値は必須引数）

複数画面から呼ばれる汎用ヘルパー（モックデータ生成関数等）に、特定ドメインの固定値
（分類コード・年度・組織種別等）をハードコードしたまま再利用してはならない。無関係な
ドメインの画面から呼ばれたとき、クラッシュせずに静かに誤った値を返すため、発見が最も
遅れる失敗になる。共通ヘルパーはドメイン依存の値を内部に持たず、呼び出し側の画面が
明示的に渡す必須引数として受け取る設計にする（省略時の既定値も持たせない。省略できる
時点で誤用が静かに成立する）。

後述「同じ名前のAPIが呼び出し箇所ごとに異なる内容を要求する場合の判断基準」が単一
エンドポイントのハンドラ内分岐を扱うのに対し、本節は複数画面で共有するヘルパー関数の
引数設計を扱う。

## 権限分類コードは消費系統を横断的に洗い出してから一括登録する

権限判定に使う分類コードをモックデータへ登録する際は、同一の分類キーを消費するクエリ・
データ構造をキー名の機械的検索で全系統洗い出し、全系統へ一括で登録する（洗い出しと
早期判断禁止の規律は `SKILL.md` Phase 2 を参照）。1系統だけ登録して表示が通っても、
別系統（画面表示用・過去年度権限用等）が未登録のままなら別の操作・別の表示で再クラッ
シュする。

## 型は消費側コードの比較方法に合わせる

返却値の型は、APIスキーマ上の宣言（例: OpenAPIの `string` 指定）をそのまま鵜呑みにして
決めてはならない。実際にその値を消費している側のコードが、どういう比較・変換をして
いるか（`===` の右辺の型、`parseInt`/`Number()` の有無、文字列の前方一致判定の有無等）
を確認し、それに一致する型・形式で返す。スキーマ上は文字列指定でも、消費側が数値
比較をしていれば数値で返さない限り判定は成立しない。

## 同じ名前のAPIが呼び出し箇所ごとに異なる内容を要求する場合の判断基準

同一APIエンドポイントが複数の呼び出し箇所（画面、共有ゲート部品等）から呼ばれ、
それぞれ期待する返却内容が食い違う場合の切り分け:

| 食い違いの種類 | 対処 |
|---|---|
| フィールドの過不足程度の差異（呼び出し箇所によって一部フィールドを見ないだけ） | 単一のモックハンドラに統合し、両方の呼び出し箇所が必要とするフィールドを全て含めた形状を1つ返す |
| レコード集合自体が異なる（呼び出し箇所によって本質的に別のデータセットを期待している） | ハンドラ内でリクエスト変数（クエリパラメータ・リクエストボディの識別子等）を見て、集合を分岐して返す。単一の固定レスポンスに統合しない |

分岐が必要か迷った場合は、両呼び出し箇所の期待値を並べて「単に足りないだけか」「そもそ
も別物か」を先に確定してから実装する。
