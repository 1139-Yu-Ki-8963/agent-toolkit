# 詰まりパターン集（症状・原因・対処）

プロジェクト固有の詰まり（具体的なAPI名・セッションキー名等）はこのファイルには書か
ない。それは `manifest.yml` の `projects.<system>.known_gotchas` に書く。このファイルは
プロジェクト非依存の一般的な症状パターンのみを扱う。

| 症状 | 原因 | 対処 |
|---|---|---|
| 認証チェックが永久に完了しない（画面がローディングのまま進まない） | セッションキーが空でクエリ自体が送信されていない | Phase 2の実測手順に戻り、セッションキーの充足タイミング（sync-initial/async）を確認する。非同期であれば投入完了を待たずに次の処理へ進んでいないか確認する |
| 画面固有のAPIが1件も発火しない | 画面が取り込む共有ゲート部品（年度・組織セレクタ等）側の依存が未対応のまま | Phase 1のshared_gate_components追跡に戻り、共有部品が要求するAPI・セッション値もモック対象に含める |
| モック操作（モーダル・選択肢）を突破してもメインのデータが空、または想定外の画面に遷移する | 操作で埋まるパラメータと、メインのデータ取得条件が要求するパラメータが食い違っている | 操作完了後に実際に発火するクエリのパラメータを記録し、Phase 1で列挙した画面本体の要求値（URLパラメータ/ストア値）と突合する |
| データ自体は返るのに画面に何も表示されない、または選択肢が埋まらない | 権限の連鎖（route guard/APIミドルウェア/UI表示条件等）のどこかの層に対象画面の登録が抜けている | Phase 2のpermission_chain_layers各層を1つずつ確認し、対象画面のエントリが未登録の層を特定する |
| 識別子やコード値が期待通りに解釈されない | 文字列か数値か、接頭辞の有無などの形式が食い違っている | 消費側コードが実際に比較に使っている型・形式に合わせてモック値を書き直す（`references/mock-implementation-recipe.md` の型決定基準を参照） |
| ブラウザ自動化で操作対象の要素が見つからない | 表示名を省略・推測して指定している | 実際の画面に表示されているテキストを完全な形でそのまま指定する（省略・意訳・部分一致に頼らない） |
| 開発サーバーのポートが他の検証環境と衝突する | 複数エージェント・複数環境が同一ポートを使用している | 自分が起動したサーバーのポートのみを使う。稼働中の他エージェントの環境に接続・干渉しない（前提ゲートの原則） |
| モックの返却値や型を直しても効果が出ない | 名前が似た別の未使用モックを誤って修正している | 期待フィールドを最終的に書き込んでいる処理をソースから逆引きし、実際に消費されているAPI・モックを確定してから修正する（`references/mock-implementation-recipe.md` 参照） |
| スキーマ上は正しい型で返しているのに判定が常に成立しない | APIスキーマ上の宣言型と、消費側コードが比較に使う型が食い違っている | 消費側コードの比較方法（`===`の右辺の型・`parseInt`の有無等）に合わせて返却値の型を決め直す（`references/mock-implementation-recipe.md` 参照） |
| 画面全体のクラッシュ（エラー境界への遷移） | エラーバウンダリに捕捉され、クエリ記録上は正常に見えてしまう | クエリ記録では検知できない。ページの `pageerror` イベントから診断する |
| 同一条件で成功と失敗が混在する | レース条件と即断しがちだが、実際はモックデータ欠落によるタイミング依存の見かけ成功であることが多い | `SKILL.md` の「同一条件で成功と失敗が混在する場合の対処」の手順に従う（即断せず、まずPhase 1/3の突合でデータ欠落を排除する） |
