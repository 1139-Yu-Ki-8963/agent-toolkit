# 設計書なし画面の開通スキル

このスキルは開通からリバース基準タグ確立までを単独で完走する自己完結型スキルである。開通の事実（どのURLなら検証できるか）を知るのは本スキルだけであり、下流の管理者（orchestrating-reverse-docs-flow）はこの事実を能動的に検知できない。そのため本スキルは、画面レジストリへの直接読み書きと `syncing-reverse-env` の直接起動を自ら行う。これは完全仲介方式の例外ではなく、基準タグ確立まで単独完走するという設計要件に基づく意図した正式仕様である（他の子スキルは完全仲介方式に従い、この2点に触れない）。プロジェクト固有の値（パス・コマンド・API名・ポート・画面ID等）は本文に一切書かず、すべて同ディレクトリの `manifest.yml` の `projects.<system>` から取得する。起動経路は二重である。単独起動時はユーザー自身が起点となり、管理者経由時は `orchestrating-reverse-docs-flow` による画面未開通状態の判定が起点となる。これは意図した二重運用であり、単独起動条件・返却ブロック契約への参照は削除すべき旧設計の名残ではない。

## 使用タイミング

- 対象画面にまだ設計書が無く、既存コードがログインを要求するなどの理由でそのままでは動作確認できないとき
- 起動引数は system・screen_id・reverse_worktree・ports・docs_root・user-approved の全量（管理者から渡される。単独起動時はユーザーから直接取得してよい。`user-approved` は Phase 5 で `syncing-reverse-env(mode=registry)` へ転送するため必須）
- 対象プロジェクトの `manifest.yml` に `projects.<system>` エントリが無い、または未確定キー（`<FILL:...>`）が残っている場合は `assets/manifest-template.yml` を複製してまず埋める（前提ゲートで検出・差し戻し）
- 設計書が既にある画面はこのスキルの対象外（`rebuilding-screen-unit-from-docs` / `rebuilding-code-from-docs` を使う）

## 埋め込みビュー画面の開通（ルートを持たない画面）

対象画面が独自のルーティングエントリを持たず、常に親画面の内部（タブ・モーダル・埋め込みパネル等）としてのみマウントされる画面を指す。判定基準は次の2つで、いずれかに該当すれば本節の手順を適用する。(1) Phase 1でのトップレベルルート検索（画面ID・画面コンポーネント名によるルーティング定義の検索）が空振りする。(2) ルート定義は存在するが、直接アクセスでは初期表示データの取得が発火・充足されず、実質的に親画面の操作を経由しないと機能しない（ルートの有無だけで判定を打ち切らず、直接アクセスでの初期表示データ発火まで確認して判定する。この場合も埋め込みビュー相当として扱う）。

この場合、対象画面コンポーネントのimport元を逆引きし、どの親画面がこの画面を取り込んでいるかを特定する。親画面が複数存在する場合は、最も単純な操作手順で到達できる1つを選ぶ。

操作手順を書き下ろす前に、選定した親画面の既存の基準確立記録（画面レジストリのエントリ内の `verification_url`・`status` 等）を確認し、親画面自身が抱える共有ゲート部品（多段階の選択ウィザード等）に同じ残留・未完了状態が記録されていないか確認する。ここでいう「同じ状態」とは、当該共有ゲート部品の識別名が一致することを指す。記録されている場合、それは親画面自身の合格基準を満たす到達点であり、埋め込み対象画面への到達点として十分であるとは限らない。この場合は親画面のゲート未完了部分について、埋め込み対象向けの追加調査が必要と判定する。

検証URLは対象画面単体のURLとしては構築できないため、「親画面のURL＋そこから対象画面を表示させる操作手順」の組で表現する。操作手順は、クリック対象・入力値・遷移待機条件など、再現に必要な情報を過不足なく記述する。

この読み替えは実行手順全体に及ぶ。前提ゲートの対象URLマウント確認、Phase 4の実データ表示確認、Phase 5の `verification_url` 記録は、いずれも「親画面URL＋操作手順」による確認・記録に読み替える。

## 実行手順

### 前提ゲート

`manifest.yml` の `projects.<system>` を読み込む。エントリ不在、または値に未確定トークン（`<FILL:...>`）が1つでも残っていれば `status=ERROR` とし、hint に不足キー一覧と `assets/manifest-template.yml` への案内を記して停止する。値の穴埋めは実行者自身が推測して行わない。人間または呼び出し元が `assets/manifest-template.yml` を複製して実値を埋めてから前提ゲートを再実行する（本スキルの自己完結性は「開通〜基準タグ確立」の作業範囲に限られ、manifestのブートストラップ自体は対象外）。`manifest.<system>.repo_and_launch` に従い、対象リポジトリで dev サーバーを自分で起動する（稼働中の他エージェントのサーバーには接続・干渉しない）。画面レジストリ（`manifest.<system>.handoff.screen_registry_path`）を `<system>-` プレフィックスで検索し、該当エントリが1件も無い（今回が最初の1枚目である）ことを確認できた場合は、次の健全性確認を省略し、devサーバー起動確認のみで前提ゲートを通過してよい。それ以外の場合は `manifest.<system>.reference_screen`（開通済みの健全性確認用画面）を、自分が起動したサーバー上でブラウザ自動化により表示確認する。合格基準は実データが最低1件表示されること、または空表示が仕様上正当である根拠を確認できることのいずれかとする。失敗したら環境自体の不備として `status=ERROR` で停止する（この画面自体の開通作業には進まない）。

この健全性確認は対象画面と同一の業務ドメイン内でのみ有効とみなす。対象画面が異なる業務ドメインに属する場合、過去に蓄積した知見は「仮説」として扱い、実測をやり直す（`reference_screen.domain_tag`（任意キー）で補助判定してよい）。

前提ゲート通過後・Phase 1着手前に、対象URL（画面が埋め込みビュー型の場合は「埋め込みビュー画面の開通（ルートを持たない画面）」節に従い親経由URL）を1回開き、対象コンポーネントがマウントされることを確認する。マウントが確認できなければAPI調査には進まず `status=BLOCKED` とする。

TaskCreate で本前提ゲートを含む全Phase分のタスクを1つずつ登録し、各Phase開始時にTaskUpdateでin_progressに更新する。

完了条件: (a) manifest必須キー全確定 (b) devサーバー起動確認 (c) リファレンス画面の健全性確認PASS (d) 対象URL（埋め込みビュー画面の場合は親経由URL）のマウント確認PASS の4点がすべて満たされている（開通済み画面が1件も無い最初の1枚目の場合は(c)を省略可）

### Phase 1: API依存の特定

対象画面のルーティング定義を画面IDで検索する。ルートが1件も見つからない場合は、その画面が「埋め込みビュー画面の開通（ルートを持たない画面）」節が扱うルートを持たない画面である可能性を疑い、同節の手順に従う。

画面本体のソースコード一式（ルーティング定義・画面コンポーネント・APIクライアント）を Grep/Glob で特定する。画面IDでのファイル検索が空振りする場合は、ルーティング定義側から画面IDで逆引きする。複数の画面IDが同一コンポーネントを共有している場合は、ID固有の分岐処理をすべて洗い出すことを完了条件に含める。ストーリーファイル・テストファイルも一次情報源として探索対象に含める（実装コードだけでなく、これらが期待する props・API応答の形からも依存関係を特定できる）。

画面が取り込む共有ゲート部品（年度・組織セレクタ等、`manifest.<system>.api_dependency_entrypoints.shared_gate_components` が示す箇所）を、実際の import 等の取り込み記述から辿り、これらが呼ぶAPIも対象に含める（推測で決めつけない）。画面が複合画面（複数の子画面・タブ・モーダルを内包する画面）の場合、取り込み範囲を「初期表示で無条件にマウントされる部分＝今回の開通対象」「ユーザー操作後にのみマウントされる部分＝後続タスクとして `handoff.unlock_record_dir` に記録し今回は対象外」に振り分ける。

同一の呼び出しシンボル（コンポーネント等）が複数箇所から呼ばれている場合、シンボル単位で一括判定せず、個々の呼び出し箇所ごとにマウント条件（無条件でマウントされるか、特定の状態変化後にのみマウントされるか）を確認したうえで振り分ける。

依存追跡の原則: 共有部品への依存追跡は、画面本体からの直接の取り込みだけで打ち切らず、共有ゲート部品からの遷移先や、無条件でマウントされる子コンポーネントの取り込みまで再帰的に辿る。追跡の対象範囲は、配置ディレクトリ名やファイル名の見た目（共通部品らしい名前かどうか等）で判断せず、import 文の実態のみを根拠に機械的に確認する。直前の「呼び出し箇所ごとの振り分け」が個々の呼び出し箇所のマウント条件の判定を担うのに対し、本原則は追跡をどこまで辿るか（再帰）と何を根拠に辿るか（import 実態）を定める。

各APIについて `manifest.<system>.api_source_of_truth.definition_root`（型定義・OpenAPI等の正本）でワイヤ上の実名を確認する（コード生成名と実名は大小文字が食い違うことがある点に注意）。URLパラメータ・ストア値・セッションキーを列挙する。ストア値は充足元で「ロード時に自動充足されるもの」「前画面での操作を経由してのみ充填されるもの」の二分類に整理する。後者は前画面での操作手順を再現するか、それが困難な場合は起動時初期投入での代替充足を検討する。セッションキーは「いつ埋まるか」（同期的初期投入 / 非同期処理）まで確認する。

対象画面が保持するオペレーション（API呼び出し・状態遷移）数を数え、20件を超える大型複合画面と判明した場合は、初期表示に必要な最小集合のみを今回の開通対象とし、残りは未開通一覧として記録する。この場合の合格基準は最小集合の実データ表示確認とする（20件は本スキル本文で定めた既定のしきい値であり、manifestへの閾値キー追加は行わない）。

大型複合画面、またはモック前例の無い新規の共有業務ロジックフックに依存する画面では、オペレーション数の列挙だけでは初期表示の完了判定（ローディング解除・描画開始を決めるフラグ群）の依存関係を捉えきれない。この場合は完了判定フラグそのものを起点に、そのフラグを真にする条件（どのオペレーションの成功・どのストア値の充足に依存するか）を逆に辿って依存関係を洗い出す。併せて、当該共有フックに対する既存モックの前例の有無を先に確認し、前例が無ければ追加の調査時間を計画に織り込む（前例ありの想定工数で見積もらない）。

完了条件: API一覧（画面本体分＋共有ゲート部品分・オペレーション数20超の場合は初期表示最小集合分）・実名対応表・URLパラメータ/ストア値（充足元の二分類付き）/セッションキー（充足タイミング付き）の一覧が記録済み。複数画面IDを共有する場合はID固有分岐の洗い出しも完了済み。同一シンボルを複数箇所から呼び出す場合は、呼び出し箇所単位のマウント条件記録も完了済み。大型複合画面・新規の共有業務ロジックフックに依存する場合は、完了判定フラグの依存関係の洗い出しと既存モック前例有無の確認も完了済み

### Phase 2: 認証・権限基盤の登録

着手前に、対象画面がそもそも権限チェック関数（権限判定API・権限チェック用フック等）を呼び出しているかを、取り込み記述の実態から確認する。呼び出していない画面であれば、権限チェーンへの登録作業（本Phaseの各層登録）はスキップし、その判定根拠を記録して次のPhaseへ進む（存在しない権限チェックへの登録作業に時間を使わない。セッションキーの実測など権限以外の確認は通常どおり行う）。

セッションキーの存在確認はブラウザ自動化での実測とする（静的読解だけで済ませない）。`manifest.<system>.auth_session_permission.identifier_field_check` が指す、認証系APIが読む識別子フィールドについて、キーの存在だけでなく値が空でないことまで確認する。

`manifest.<system>.auth_session_permission.permission_chain_layers`（route guard / APIミドルウェア / UI表示条件等）の各層に対象画面のエントリを登録する前に、消費側の比較ロジックを読み、権限チェーンが単層か二層構造かを確認する（分類キー値の突合を先に行う）。二層構造で分類キーが一致しない場合は、既存の分類グループへの追加では解消しない。最上位グループ要素の新設が必要と判断し、その旨を登録内容に反映する。

権限判定に使う分類コードは、1つの権限チェーンの中だけでなく、複数の独立したクエリ・データ構造（画面表示用の分類取得・過去年度権限用の分類取得等）に分散して消費されていることがある。登録前に、同一の分類キーを消費する全系統を分類キー名の機械的検索で横断的に洗い出し、洗い出した全系統へ一括で登録する。1系統に追記して画面が表示されたことをもって「直った」と早期判断してはならない。別系統が同じ分類キーを未登録のまま参照していれば、別の操作・別の表示で再クラッシュする。前段の単層/二層の確認が1つの権限チェーン内部の構造（粗分類キー→許可コード配列）を扱うのに対し、本項は同じ分類キーを消費する系統が複数並存する横断方向の広がりを扱う。

静的読解だけで「このデータはこのAPIから来ている」と消費先・供給元を推定した場合、その推定は実行時ログ（実際に発火したオペレーション名と、その応答を受け取った後続処理）で必ず裏取りしてから、登録・モック実装の根拠にする。裏取りの無い推定を完了条件の記録に含めてはならない。

完了条件: 権限チェック呼び出し有無の事前確認済み（呼び出さない場合はスキップ判定の根拠を記録済み）、セッションキー実測結果（充足済み・識別子フィールド非空を確認済み）、権限チェーン構造（単層/二層）の分類キー突合確認、同一分類キーを消費する全系統の横断洗い出しと一括登録完了、消費先推定の実行時ログ裏取り完了、各層への登録完了

### Phase 3: 画面別モック実装

`manifest.<system>.mock_conventions`（有効化方法・配置規約・登録方法）に従い、Phase 1 で特定したAPI一覧にモックを実装する。実装後、登録した名前の一覧と `api_source_of_truth` の実名一覧を機械的に突合し、差分ゼロを完了条件とする（自然文の目視確認で済ませない）。モック変更はホットリロードで反映されないことがあるため、反映確認は必ずサーバー再起動後に行う。具体的な実装レシピ（正本からの書き写し方・消費側からの逆引き特定・型決定基準・同名API分岐の判断基準・既存共通モックの実名突合・共通ヘルパーの必須引数化・分類コードの全系統一括登録）は `references/mock-implementation-recipe.md` を参照する。実名突合は今回の新規実装分だけでなく、対象画面が依存する既存の共通モックファイルの登録名にも広げる（詳細は同レシピを参照）。

完了条件: 登録名一覧×実名一覧の突合差分ゼロ（既存共通モックのうち対象画面が依存する分を含む）、サーバー再起動後の反映確認済み

### Phase 4: 検証ループ

検証は直接URL・パラメータでの到達を優先する。UI操作（クリック・選択等）を経由した到達は、直接URL・パラメータでは対象画面に到達できない場合の最終手段としてのみ試みる。直接URL経由とUI操作経由では到達するコード経路自体が異なりうるため、この優先順位を崩すと無関係な分岐の検証結果を対象画面の検証結果と誤認しかねない。

ブラウザ自動化で実データ表示を確認する。失敗症状は「クエリ未発火」「権限エラー」に加え「画面全体のクラッシュ（エラー境界への遷移）」を明示する。「クエリ未発火」の亜種として、権限判定ゲートを構成する複数フラグの論理積が未解決のまま主要クエリが発火しない症状がある（詳細は `references/troubleshooting-patterns.md` を参照）。クラッシュはクエリ記録では検知できないため、ページの `pageerror` イベントから診断する。

未登録オペレーションに汎用の空応答を返すフォールバック機構がある構成では、上記3症状のいずれとも異なる第四の失敗モードがある。クエリは発火し権限も通っているのに、フォールバック応答の汎用形状に無いフィールドを消費側コードが直読みしてクラッシュする「フォールバック応答の形状不一致」である。クラッシュを診断する際は、「フォールバック応答の形状」と「消費側コードが直読みしているフィールド名」の突合を明示的な診断ステップとして必ず実施する（この突合により、モック未登録が原因か、フォールバック形状が原因かを切り分けられる）。

ブラウザ自動化がナビゲーションタイムアウトし未描画と判定された場合、直ちに失敗と結論しない。`syncing-reverse-env/config.yml` の `projects.<system>` 上書きでナビゲーションタイムアウトを延長し、再検証してから判定する。

差分が見つかれば原因のPhase（1/2/3）へ戻る。スクリーンショットまたはログ抜粋を伴わない合格判定は禁止。未検証項目は「未検証」と明記する。検証ループの反復規律は下記「ループ設計」に従う。

完了条件: 収束条件を満たすか、発散/上限到達により `status=BLOCKED` が確定している

### Phase 5: 基準確立への引き渡し

開通状態をコミットする（コミットメッセージ:「【機能追加】<画面名> をモックAPIで開通」）。開通確認時点のコミットハッシュを `source_ref` とする。このオリジナル参照点となる元リポジトリは `manifest.<system>.repo_and_launch.original_repo_path`（任意キー。無指定なら `work_repo_path` と同一視する）を正本とする。存在確認は前提ゲートの必須キー確認で完了済みのため、Phase 5独自のエラー分岐は設けない。画面を表示確認できたURLを `verification_url` とする（画面が埋め込みビュー画面の場合は「埋め込みビュー画面の開通（ルートを持たない画面）」節に従い親画面URL＋操作手順の組で記録する。ローカル起動できず確認手段が無ければ「未実施」と明記する）。`docs_root` 起点で `<system>-<screen_id>` 相当のパスを組み立てて `design_doc_path`（今後の設計書の想定配置パス）とする。画面レジストリ（`manifest.<system>.handoff.screen_registry_path`。既定は `<docs_root>/一覧/reverse-screen-registry.yml`。docs_root はスキルフォルダ外のため、スキル同期・上書きコピーの影響を受けない。contract.mdの正本と一致させる）へ `source_ref`・`verification_url`・`design_doc_path`・`status=unlocked` を記帳する。

`syncing-reverse-env` へ渡す `reverse_worktree` は、呼び出し前に命名規則（`reverse-code-<scope>` / `feature/reverse-code-<scope>`）に従い自ら用意する。既存のworktreeがあれば再利用し、新規作成はしない。並列実行時は「並列実行時の運用規約」節で定義する作業コピーとは別物であることに注意する。起動引数 `ports` は前提ゲートで自ら起動したdevサーバー専用の値であり、`syncing-reverse-env` が管理する環境のポートは同スキルの `config.yml` 計算式が常に正となる。両者に食い違いが生じても異常ではなく、完了報告にその旨を記す。`Skill` で `syncing-reverse-env` を `mode=registry`・`system`・`screen_id`・`reverse_worktree`・`ports`・`user-approved` で起動する。

返却 `status=PASS` なら、画面レジストリの該当エントリを `status=baseline-established` に更新し、`git tag -l "reverse-baseline/<scope>"` 等の決定的コマンド出力でタグ確立を確認する（自然文の自己申告で完了と判定しない）。

開通完了後、Playwright で対象画面のスクリーンショットを撮影し `<docs_root>/画面/screen-<screen_id>/詳細設計/original.png` として保存する。これは基本設計書・詳細設計書の画面キャプチャとして参照される。撮影には syncing-reverse-env の既存 Playwright 設定（viewport・loading 待機条件）を流用する。

確認できたら `status=BASELINE-ESTABLISHED` で返却する（返却フィールドに `baseline_tag` を追加し、`syncing-reverse-env` の返却値をそのまま転記する）。返却が `PASS` 以外（`FAIL`/`ERROR`/`INCOMPLETE`）の場合は、画面レジストリを `status=unlocked` のまま残し、`status=UNLOCKED`（部分完了）で hint に理由を記して差し戻す。返却が `status=ERROR` かつポートスロット上限（`max_slots`）到達が理由の場合は `status=BLOCKED` とし、hint に「(a) 並行検証の要否再確認」「(b) baseline-established済み環境のteardown可否」「(c) 上限拡張要否のユーザー確認」の3点を記す。

基準確立が完了した時点で、並行して `status=baseline-established` にある環境数が `syncing-reverse-env` の容量上限（`max_slots` 等）に近づいていないか確認する。近づいている場合は、次の画面の開通に着手する前に、拡張または整理（teardown可能な環境の有無）の要否をユーザーに確認する。この確認は、上限到達により `syncing-reverse-env` が `status=ERROR` を返してから対処する前段落のhint（(a)〜(c)）とは異なり、ERROR発生そのものを未然に防ぐための予防的な位置付けである。ここで拡張・整理を済ませておけば、後続画面の着手時に前段落のhintで同じ判断をやり直す必要はない。

完了条件: レジストリ登録済み、`reverse_worktree` の命名規則準拠を確認済み、かつ（成功時のみ）`git tag -l` 等の決定的出力でタグ確立を確認済み

## 「同一条件で成功と失敗が混在する」場合の対処

1. 直ちにレース条件と断定しない
2. まず「チェックが実行されれば確実に失敗するデータ欠落が、チェック自体のスキップというタイミングの穴で時々成功して見える」可能性を排除する: Phase 1 で列挙した画面要求値（URLパラメータ・ストア値・セッションキー）の一覧と、Phase 3 で実装したモックデータの一覧を機械的に突合する
3. 突合で不一致（画面が要求する値にモック側の対応が無い等）が見つかれば「データ欠落」と確定し、発散カウントには含めずPhase 3へ差し戻す（原因が特定できたやり直しのため）
4. 突合が完全一致してもなお混在する場合のみ、次の対処に進む: 同一シナリオを3回連続実行し多数決を取る。1回でも失敗が出たら「モック側の非同期タイミング（データ投入とクエリ発火の順序保証が無い）」を疑い、モックを固定遅延無し・即時応答型に修正して再検証する。それでも混在が解消しない場合は `status=BLOCKED` とし、hint に「モック実装ではなくアプリ側の真のレースコンディションの疑いあり」と明記してユーザーに報告する

## 「同一条件で常に同じ失敗（決定的クラッシュ）が再現する」場合の対処

混在（断続的に成功・失敗が入れ替わる）か決定的（同一条件で毎回同一の失敗が再現する）かをまず判定する。混在なら前節（「同一条件で成功と失敗が混在する」場合の対処）に進み、決定的なら本節に進む。

Phase 1で列挙した画面要求値とPhase 3で実装したモックデータの突合が完全一致していて、なお毎回100%同一のクラッシュが再現する場合であっても、直ちにアプリ本体側の問題と結論しない。まず、非同期（async）処理で埋まるはずのセッションキーを、起動時の初期投入に一時的に付け替えて再検証する。「キーが丸ごと欠落している」場合だけでなく「キー自体は存在するが必要な行・ノードが不足している」場合も、同じ付け替えで解消することがある。

ただし、対象自身の初期化処理が外部から投入した値を上書き・再構築する実装の場合、この付け替えは効かない。その場合は初期化処理自体の実行タイミングやトリガー条件を確認し、別の充足経路を検討する。

それでもクラッシュが解消しない場合は、「重要な注意事項」に記す事前バグ対応方針（修正せず記録してユーザー判断に委ねる）に従う。

## 予想を裏切る挙動

一般的な症状別の切り分け（詰まりパターン）は `references/troubleshooting-patterns.md` を参照する。以下は同ファイルに寄せていない個別の注意点。

- 設計書がある画面は対象外（`rebuilding-screen-unit-from-docs` / `rebuilding-code-from-docs` を使う）
- モック変更はホットリロードで反映されないことがある。反映確認は必ずサーバー再起動後に行う
- コード生成名とワイヤ上の実名は大小文字が食い違うことがある。実名は `api_source_of_truth.definition_root` で確認する
- manifest に未確定キー（`<FILL:...>`）が残ったまま作業を開始してはならない
- manifestの穴埋めは実行者ではなく人間・呼び出し元の責任。実行者が値を推測して埋めてはならない
- 対象プロジェクトの最初の1枚目を開通する場合、`reference_screen` による健全性確認は省略してよい（開通済み画面が存在しないため）
- 権限チェーンは単層とは限らない。二層構造の場合は登録前に分類キー値の突合を先に行う
- 権限分類コードは複数の独立した系統（画面表示用・過去年度権限用等）に分散して消費されていることがある。1系統に登録して直っても早期判断せず、同一分類キーを消費する全系統へ一括登録する
- 未登録オペレーションに汎用の空応答を返すフォールバック機構がある構成では、第四の失敗モード（フォールバック応答の形状不一致によるフィールド直読みクラッシュ）を疑い、フォールバック形状×消費側直読みフィールドの突合を行う
- 複数画面から呼ばれる共通モックヘルパーに単一ドメインの固定値をハードコードしない。ドメイン値は呼び出し画面が明示的に渡す必須引数とする（クラッシュせず静かに誤るため発見が遅れる）
- 同一条件で毎回同一のクラッシュが再現する場合は、非同期セッションキーの初期投入への一時的な付け替えを試してから本体側の問題と結論する
- 対象画面自身の既存バグは修正しない。事前バグの可能性として記録しユーザー判断に委ねる
- 並列実行時は作業コピー（worktree・ブランチ・ポート）を画面ごとに分離し、画面レジストリへの記帳のみを直列化する
