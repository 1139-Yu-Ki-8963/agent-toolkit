§1概要

リバースエンジニアリング設計の中心課題は「設計書が完全であることの証明」にある。本スキルはこれを、設計書だけを見て再構築したリバースコード環境が、オリジナルコード環境とポート番号以外で完全に同じ動作をすることで証明する。

本ガイドで使う「L1〜L5」は、起動確認から画面操作結果の突合までを 5 段階に分けた動的検証レベルを指す（内訳は §8 検証内容）。「drvfs」は、WSL2 環境で Windows 側のドライブがマウントされる特殊なファイル領域（/mnt/*）を指す（詳細は §6）。

§2出力

環境同期が成功すると、以下の状態が確立される。

基準コミット
「設計書だけを見て再構築したコード」と「オリジナルのコード」の 2 環境を実際に起動し、画面表示・API 応答・DOM 構造まで一致することを証明済みのコミット。このコミットが存在する限り「設計書はコードを完全に再現できる品質である」と主張できる。
検証報告
ファイル差分の静的比較、環境設定の同一性チェック、ブラウザを使った画面描画の動的比較（5 段階）の結果を、合否とともにまとめた報告文。不一致がある場合は具体的な差分箇所を指摘する。

§3リバース以外の使い方

本スキルはリバース検証のために設計する。ただし「ポート以外同一の 2 環境を永続 worktree で維持し、静的 diff ＋ Playwright 5 層で突合する」という仕組みの性質上、リバース以外にも次のような使い方ができると考えられる。

応用オリジナルコード環境（正）リバースコード環境（対象）使い方
PR レビュー実機確認main ブランチPR ブランチ 並行起動して挙動差分・ビジュアル差分を実機で突合する。「意図した変更以外に差分がないか」を機械判定できる
リファクタ安全性証明リファクタ前リファクタ後 「挙動が変わらないこと」を L1〜L5（該当画面）の全層一致で証明する。この場合は実差分ゼロではなく動的比較の全層一致が合格条件になる
UI ビジュアルリグレッション変更前変更後 L3（スクリーンショット画素比較）を単独利用し、意図しない見た目の変化を検出する

応用時は「何を一致とみなすか」（静的一致まで求めるか、動的一致のみか）が用途ごとに変わる。合格条件のプリセットを mode とは別の軸で持たせることが、スキル本体実装時の検討事項になる。

§4用語

<system>
設計書 frontmatter の source_repo 末尾のリポジトリ名（.git 除去）から機械的に導出する識別子。
<画面ID>
設計書 frontmatter doc_id: screen-<画面ID> から screen- 接頭辞を除去して導出する識別子。
<scope>
<system>-<画面ID>。worktree・ブランチ・基準タグ・ポートスロットの管理単位。共有アプリを画面単位で最大 5 画面まで並行検証するために、この粒度で分離する。
オリジナルコード環境
リバース対象のオリジナルコードを保持する git worktree（worktree 名 original-code-<scope>）。設計書 frontmatter の source_repo / source_ref から本スキルが用意する読み取り専用コピーで、常に「正」。内容を変更することはない。
リバースコード環境
設計書だけを見て再構築したコードの git worktree（worktree 名 reverse-code-<scope>、ブランチ feature/reverse-code-<scope>）。容器（worktree・ブランチ・ポート）は本スキルが用意し、中身のコードは呼び出し元（orchestrating-reverse-docs-flow が仲介）が書く。
worktree のライフサイクル（永続・使い回し）
両 worktree は初回実行時に一度だけ作成し、以後の検証ではすべて同じものを使い回す。削除はユーザーが明示的に依頼した時（teardown）のみ。
基準コミット / 基準タグ
ポート以外の完全一致を証明できた時点のリバースコード環境側コミットが基準コミット。annotated git tag reverse-baseline/<scope>（基準タグ）がその唯一の記録で、タグメッセージに検証日と判定サマリを持つ。コミットとタグはリポジトリ本体に保存されるため、worktree を破棄しても消えない。リバースコード環境はいつでも git reset --hard reverse-baseline/<scope> で基準状態へ復帰できる。
ポートモデル
両環境の稼働ポートは <scope> ごとに動的割当てするスロット番号（1〜5）から計算式で決まる（§6）。両環境は必ず異なる帯域を持ち、これが「唯一許容される差分＝ポート番号」の根拠になる。
config.yml（スキルフォルダ直下）
プロジェクトによって変えたくなる可能性がある値（ポート計算式・サービス一覧・起動手順・diff 除外リスト・L3 閾値・max-loop 既定値）の置き場。defaults ＋ projects 別上書きの 2 層構造。
L1〜L5
動的検証を、起動確認から画面操作結果の突合までの 5 段階に分けた検証レベル（L1 起動確認・L2 構造スナップショット・L3 画素比較・L4 コンソールエラー・L5 操作シーケンス突合。詳細は §8）。
drvfs
WSL2 環境で Windows 側のドライブ（C:\ 等）が /mnt/* としてマウントされる特殊なファイル領域。ネイティブ Linux ファイルシステムより低速なため、worktree の配置先として非推奨（詳細は §6）。

§5使い方（mode とユースケース）

mode は運用の 3 局面 —「初回の環境整備」「それ以降の検証サイクル」「検証終了時の片付け」— に対応する 3 つ。

mode局面動作
setup初回両 worktree を作成し、名前・固定ポートを割り当てて返す。比較は実行しない
sync（既定）2 回目以降検証（静的＋環境＋動的）→ 差分があれば整列 → 一致で基準タグを更新
teardown検証終了時固定ポート帯の一括 kill ＋ 両 worktree の削除。ユーザーの明示依頼がある時だけ受け付ける

sync のオプション

オプション動作
dry-run整列・基準タグ更新を行わず、差分レポートだけを返す。無副作用の対象は「リバースコード環境のgit 管理内容」と「基準タグ」の 2 つ（Phase 5 の起動・観測とオリジナル側の source_ref 復元は実行する）。node_modules・バンドラキャッシュは git 管理外の生成物であり、その修復（npm ci・キャッシュ削除）は診断を最大化するため dry-run でも実行を許容する
reset-first開始前にリバースコード環境を基準タグへ git reset --hard で復帰させてから検証サイクルに入る

本スキルは macOS と Windows（WSL2・Ubuntu）の両方に対応する。起動時に OS を判定し、以下のとおりコマンドを解決する。

命名規則と環境確保

worktree の用意・命名・再発見はすべて本スキルが担う。作成は初回実行時の一度だけで、以後は同じ worktree を再発見して使い回す。どの mode でも Phase 2 で必ず実行し、冪等に振る舞う。

対象規則

共有オリジナル環境（既定・by-source-ref）

original_sharing（config.yml、既定 by-source-ref）が by-source-ref の場合、複数画面の検証で 1 つの読み取り専用オリジナル worktree を使い回す。命名は original-code-<system>@<sha8>（sha8 = 解決済み source_ref コミットの先頭 8 文字）。original_sharing: per-scope を選択した場合、または下記の共有前提条件を 1 つでも満たさない画面は、従来どおり画面ごとの original-code-<scope> に自動で degrade する。

共有前提条件内容
1. 同一 <system>設計書 frontmatter の source_repo から導出した <system> が一致すること
2. 同一の解決済み source_refセッション開始時（最初の setup/sync）に git rev-parse <source_ref> で SHA（コミットのハッシュ値）へ解決して pin する。以後そのセッションは全画面が同一 SHA を使う（ブランチが途中で動いても pin した SHA を使い続ける）
3. 同一の起動プロファイルlaunch コマンド・関連 env が一致すること
4. オリジナルがリクエスト独立サーバー側の可変状態を持たず、同一リクエストが状態非依存に同一応答を返すこと。満たさなければ共有時に N 画面の並行キャプチャがサーバー状態を汚し合う
不変性: 共有オリジナルは初回に一度だけ git worktree add --detach <sha> で作成し読み取り専用。使用中（参照カウント>0）は再チェックアウト・source_ref 復元をしない。dirty 検出時は自動復元せず ERROR（読み取り専用契約の違反）。復元は参照カウント=0 の時だけ行う。

参照カウント方式のライフサイクル: 各画面（reverse-code-<scope>）が共有オリジナルの使用を開始する時、共有オリジナル root の .shared-original-refcount.d/<session>-<pid> を mkdir で登録する（原子ロック .shared-original-lock.d を mkdir で取得 → 登録 → 解放）。使用終了・teardown で該当エントリを削除する。参照カウント（生存セッションのエントリ数）が 0 の時だけ、明示 teardown で共有オリジナルの dev サーバー kill + worktree 削除が許される。Phase 2 開始時と teardown 時に、死んだセッション（pid 不在）のエントリを prune する。

基準タグ reverse-baseline/<scope> のメッセージには検証時の解決 source_ref SHA を記録する。sync 時、設計書 source_ref の解決 SHA が基準タグ記録の SHA と異なれば「別オリジナルに対する古い基準」＝再検証必要と判定する（§7 の基準タグ乖離検出に source_ref SHA 軸を追加）。

障害ドメイン: 共有オリジナルの dev サーバーが落ちると、それを使う全画面が同時失敗する。§8-3 の L1 段階診断のプロセス再起動は system 単位（共有オリジナルを 1 回再起動）で行い、影響画面をまとめて再走行する。per-scope は画面ごとに障害隔離できる。

ポートモデル（計算式）

worktree は初回に一度だけ作成し、teardown まで使い回す。既定の共有オリジナル方式（original_sharing: by-source-ref）では、reverse ポートは <scope> ごとの reverse スロット、original ポートは <system>@<sha8> ごとの共有オリジナルスロット（oslot）から、それぞれ別の帯域で決まる。

── ポート計算式 ─────────────────────────────────────────────

  reverse スロット k = 1..max_slots（既定 5・<scope> ごとに動的割当）
  reverse = band_start + slot_stride×(k-1) + reverse_offset + サービス index
  既定: band_start=9100, slot_stride=20, reverse_offset=10
  → slot k の reverse 占有帯 = [9110+20(k-1), 9119+20(k-1)]

  共有オリジナルスロット o = 1..max_shared_originals（既定 5・<system>@<sha8> ごとに動的割当）
  original = original_shared_band_start + original_shared_slot_stride×(o-1) + サービス index
  既定: original_shared_band_start=9000, original_shared_slot_stride=10
  → oslot o の占有帯 = [9000+10(o-1), 9009+10(o-1)]

  per-scope フォールバック（original_sharing: per-scope 選択時、または共有前提条件を
  満たさず degrade した画面）:
  slot k = 1..max_slots（<scope> ごとに動的割当）
  original = band_start + slot_stride×(k-1) + サービス index
  reverse  = original + reverse_offset
  → slot k の占有帯 = [9100+20(k-1), 9119+20(k-1)]（連続 20 ポート、original/reverse 同一帯）

サービス index は config.yml の services 配列の並び順（既定 [frontend] → index 0）。共有オリジナル方式の既定 oslot 1・frontend では original=9000 になる。per-scope フォールバックの既定 slot 1・frontend では original=9100 / reverse=9110 になる。

reverse スロット割当手順: 兄弟 worktree（reverse-code-*）を走査し、各ルート直下の .reverse-port-slot（番号 1 行）を読んで使用中スロットを把握 → 空いている最小番号を reverse worktree のルートに書き込む。開発用 worktree のスロットファイル .port-slot とは誤カウント防止のため別名にしている。max_slots（既定 5）を超えて空きがない場合は ERROR。

共有オリジナルスロット割当手順: 兄弟 worktree（original-code-<system>@*）を走査し、共有オリジナル worktree ルート直下の .shared-original-slot（番号 1 行）を読んで使用中 oslot を把握 → 空いている最小番号を共有オリジナル worktree のルートに書き込む（割当は reverse スロットと同じ .reverse-slot-lock.d の mkdir 原子ロックで排他する）。max_shared_originals（既定 5）を超えて空きがない場合は ERROR。

互換注記: 旧固定表（frontend = オリジナル 9101 / リバース 9111）は廃止した。per-scope フォールバックの既定 slot 1 では frontend = オリジナル 9100 / リバース 9110 になる（稼働中環境が存在しないため移行措置は設けない）。

環境差の自動解決（ゼロコンフィグ）

node_modules_strategy / allow_mnt_fs / playwright_exec.mode / playwright_exec.node_path の 4 値は既定がすべて auto（node_path のみ null）で、Phase 2 プリフライトが source_repo のパス・fs 種別・Playwright MCP 可用性を実行時検出して具体値へ自動解決する。ただし node_modules_strategy=auto は fs 種別の検出結果に関わらず常に npm_ci に解決する。独立性を犠牲にする symlink_main の自動選択は行わず、drvfs 検出時は高速化の選択肢を WARN で提示するに留める（人間が projects で明示指定した時だけ symlink_main になる）。人間が projects を手書きする必要はなく、アプリ毎に人手で書くのは起動コマンド（services / launch）だけになる。明示値（true/false・npm_ci/symlink_main・mcp/node-script・明示 node_path）は auto の自動解決を上書きする強制指定として働く。解決結果は §9 の報告に含める。

§8検証内容

8-1. 静的比較

「パス除外 → 残った差分行のポート正規化判定」の 2 段構えで、全差分を 3 分類に落とす。除外パスは config.yml の diff_exclude ＋ 設計書 frontmatter での画面固有の追加指定。

ポートが計算式で決定的であることを利用し、差分行ごとに「オリジナル側の値をサービス index へ逆引きし、リバース側の同位置の値が reverse = original + reverse_offset の期待値と一致する」場合のみ許容する。行内のポート以外の部分は完全一致を要求し、「数字が違うだけなら許容」にはしない。

分類意味扱い
一致差分なし—
ポート差のみ計算式どおりの期待値どおりの番号相違許容（件数をレポートに記録）
実差分上記以外のすべての相違FAIL 対象。ファイル別・追加/削除/変更別に集計

8-2. 環境同一性チェック（全項目）

実行環境そのものが「ポート以外は同一」かつ「互いに独立」であることを機械的に確認する。起動前（listen-actual・process-independence を除く全項目）と起動後（listen-actual・process-independence）に実行する。

#キー検証項目確認方法NG 時の扱い
1port-ownerポート占有の帰属os-toolchain の解決結果に従い確認する（macOS: lsof -i の cwd 突合 / WSL: ss -ltnp ＋ /proc/<pid>/cwd の cwd 突合）で当該 <scope> のスロット帯を走査し、LISTEN プロセスの cwd が自環境の worktree かを突合。共有オリジナル方式では 1 つの待受（oslot 帯）を複数 reverse が参照するのが正常で、reverse 側は各自の reverse スロット帯の帰属のみ確認するstale は kill して続行。無関係な稼働プロセスは ERROR
2port-config-crossポート設定の混入（転写後）リバース側をオリジナル側帯域で、オリジナル側をリバース側帯域で相互に grep相手帯域の番号が 1 件でもあれば転写ミス。ポート再生成をやり直す
3listen-actual実際の待受ポート起動後の LISTEN 一覧を §6 の計算式による期待値と突合。共有オリジナル方式では oslot 帯の待受を突合し、reverse 帯は各自の reverse スロットと突合する（1 つの共有オリジナル待受を複数 reverse が参照する構成は正常）期待外の待受・不応答は FAIL
4symlink-independenceシンボリックリンクの実態find -type l ＋ realpath で実体を解決し「自 worktree 内 / 共有 store / worktree 外」に分類相手側 worktree・元リポジトリを指すものは FAIL。共有 store は同一バージョンの実体を指すことを確認して許容。ただし node_modules_strategy: symlink_main（人間の明示指定時のみ発生）のときは node_modules（およびその配下）が source_repo を指す symlink であることは想定内であり、独立性の証明対象から除外する（FAIL としない）
5dep-tree-parity依存ツリーの同一性npm ls --all 相当を正規化し、名前＋バージョン集合を比較依存再構築（npm ci 再実行）を 1 回リトライ → なお不一致は FAIL
6node-modules-healthnode_modules の健全性node_modules の存在・.bin/vite 等の実行可能性（--version 実行）・独立性（相手側/元リポジトリへの symlink でないこと）を確認。ただし node_modules_strategy: symlink_main（人間の明示指定時のみ発生）のときは source_repo への symlink が正常状態であり、存在と .bin 実行可能性のみ確認する。なお symlink_main 運用時、片方だけ描画が異常に遅い・失敗する場合は共有 .vite のレースが原因の可能性が高い、という hint を出すNG は npm ci で修復してから再検証
7runtime-parityランタイムの同一性node -v 等を両環境で実測して突合不一致は ERROR（実行基盤の問題）
8env-runtime-parity環境変数の同一性起動時に注入される env をダンプし、ポート正規化後に比較ポート以外の差分は FAIL
9env-file-parityenv ファイルの同一性両環境の .env* ファイル群（config.yml env_file_globs）をポート正規化付きで直接突合し、バンドラのみが読む変数の差分も捕捉するポート以外の差分は FAIL
10file-mode-parityファイルモード・種別git ls-files --stage のインデックス値のみで実行ビット・種別を比較する（実 FS の stat は使わない。/mnt 配下の 777 化対策）実差分として扱い、整列で揃える
11bundler-cache-freshバンドラキャッシュの新鮮さ整列が走った周回では config.yml の bundler_cache_dirs が削除済みであることを確認。node_modules_strategy: symlink_main（人間の明示指定時のみ発生）のときは .vite が source_repo と共有されるため、整列後の削除ではなく初回 navigate 時の dependency re-optimize を許容し、巻き込まれ reload は 1 回リトライで吸収する。ただし symlink_main では .vite が共有されるため、片方の環境で持続的な描画失敗（非対称）が出たら共有キャッシュのレースを疑い、該当環境の node_modules を独立 npm ci に切り替える検証不能なら起動時の強制再構築を案内
12process-independenceプロセス独立性共有オリジナル方式では reverse 側を停止してオリジナル側が応答し続けることを確認する（共有オリジナルは他画面を巻き添えにしないため kill しない）。per-scope 方式では従来どおり片方を停止してももう片方が応答し続けることを確認して再起動する連動停止は FAIL（プロセス・DB の共有）
13storage-isolationストレージ分離DB・キャッシュ等のパス設定が worktree 内に閉じているか走査同一の外部パスを読み書きしていれば FAIL

port-owner・symlink-independence・process-independence・storage-isolation は「同一」ではなく「独立」の検証。実体を共有した 2 環境は一致して見えて当然であり、完全一致の証明は「独立した 2 環境が同じに振る舞う」ことで初めて成立する。共有オリジナル方式では独立性は「共有オリジナル 1 つ」と「各 reverse」のペアで成立し、共有オリジナル同士の独立性は問わない。

8-3. 動的比較（render-ready ＋ Playwright 5 層）

両環境をそれぞれの固定ポートで起動し、scenarios の各画面シナリオを render-ready 到達確認・内容一致・5 層で突合する。総合判定は render-ready 到達 ∧ L2' 内容一致 ∧ L2（ARIA）∧ L4 を必須とし、operations を持つシナリオが 1 つでもある画面はさらに L5 一致 も必須とする（operations を持つシナリオが無い画面は L5 を評価対象外とし従来どおりの条件のみで判定）。L3 は閾値超過で FAIL（画像ペアを添付して人間が最終裁定できる）。静的一致・L2（ARIA）一致だけでは PASS にしない。両環境が同一のスピナーで止まっていても render-ready 未到達であれば PASS ではない。

層内容判定基準
L1 起動両環境のヘルスチェック / トップ応答 かつ render-ready 到達（scenario の ready.selector が可視 ∧ loading_selectors（既定 ∪ ready.not）が不可視）。playwright_exec.mode で実行手段が分岐する: mode=mcp は MCP ツールの navigate/wait_for/evaluate、mode=node-script は config.yml の command_template を Bash で直接実行する。navigate 先 URL は 127.0.0.1 固定・waitUntil="load" 固定。navigate / readiness 待ち / 描画安定待ちのタイムアウトは config.yml の nav_timeout_ms / ready_timeout_ms / settle_wait_ms（projects で上書き可）両方の成功 かつ render-ready 到達が前提。片方でも失敗・未到達なら NOT_RENDERED とし以降をスキップして FAIL
L2 構造アクセシビリティスナップショット（DOM 構造）をポート・URL 正規化後に比較。node-script モードでは body の ARIA snapshot 文字列（ariaSnapshot()）を突合対象とする一次スクリーニング。完全一致を要求するが、単独では PASS 条件にしない（L2' 内容一致・render-ready 到達と併せて判定する）
L2' 内容一致scenario の assert.tables / assert.texts 各セレクタの innerText を正規化し、両環境の内容 JSON（rendered/reason/tables/texts）を突合する。mcp・node-script のいずれも同一形の内容 JSON を出力する（node-script は CAP_CONTENT）tables/texts が完全一致。不一致はデータ差分として FAIL
L3 視覚同一ビューポートのスクリーンショット画素比較（動的領域は mask でマスク）差分画素率が閾値以下（config.yml、既定 0.1%）。補助判定・証跡
L4 コンソールconsole error / warning の集合比較（ポートは正規化）リバース側にのみ存在するエラーで FAIL。両方に出る既存エラーは許容
L5 操作シーケンス突合設計書 frontmatter の operation_test_spec が指す「操作シナリオ仕様書.md」から scenarios[].name と同名のシナリオの operations（{action, selector, value?, key?} の配列。action は click/fill/selectOption/press）を解決し、初回内容抽出後に両環境で同順序に実行、実行後に同じセレクタで再抽出した postContent（tables/texts）を突合する。node-script は config.yml command_template の extract() 再実行結果、mcp は browser_click/browser_fill_form/browser_select_option/browser_press_key 実行後の browser_evaluate 再抽出結果を用いる（mcp 側の詳細実装は本改修のスコープ外・概念記述のみ）operations を持つシナリオが 1 つでもある画面は必須。postContent の tables/texts が完全一致。不一致は操作後の挙動差として FAIL。operations を持つシナリオが無い画面は評価対象外

L1 が失敗した場合の段階診断手順: ① dev サーバーの stdout/stderr を artifacts へ収集 → ② 待受ポートを再確認 → ③ navigate を 1 回再試行 → ④ なお失敗するなら dev サーバープロセス自体を kill → 再起動して再試行する（navigate の再試行だけでは解消しない dependency re-optimize 起因のケースが実測されているため。symlink_main で .vite を共有している時に起きやすい）→ ⑤ symlink_main 運用中で片方だけが持続的に失敗するなら共有 .vite のレースが原因なので、該当環境の node_modules を独立 npm ci に切り替えて再試行 → ⑥ それでも失敗すれば ERROR とし、収集した log のパスを添付する。④の再起動対象が共有オリジナルの場合は system 単位（共有オリジナルを 1 回再起動）で行い、同一 system@sha を使う影響画面をまとめて再走行する。per-scope は画面ごとに障害隔離できる。

DB 内容・乱数・日時などの動的データによる誤検出には、マスク対象セレクタ（mask）を設計書 frontmatter で宣言して対応する。

mcp / node-script の等価性: 両手段は同一の内容 JSON 契約（rendered/reason/tables/texts。operations 指定時はさらに postContent）を出力する。node-script は scenario 引数（url・ready・assert・loading・operations）を CAP_SCENARIO の JSON で受け取り、内容抽出結果を CAP_CONTENT に書き出す。operations がある場合は初回抽出後に page.click/page.fill/page.selectOption/page.keyboard.press を順に実行し、同じ extract() を再実行して postContent を得る。mcp は browser_navigate → browser_wait_for（ready）→ browser_evaluate（loading 不可視確認・innerText 抽出）→ browser_take_screenshot → browser_snapshot → browser_console_messages で同形の結果を得る。operations 指定時は上記の後に browser_click/browser_fill_form/browser_select_option/browser_press_key を同順序で実行し、同一の browser_evaluate 抽出を再実行して postContent を得る（概念記述。詳細実装は本改修のスコープ外）。

結果分類: 両環境が render-ready 到達 ∧ 内容一致で PASS 候補。reverse 側だけ NOT_RENDERED は再構築未完のため FAIL。両環境が同様に NOT_RENDERED（引数不足でスピナー等）は DESIGN-INCOMPLETE として設計書 scenarios への query/path_params/ready 追加を促す（§9）。

8-4. 整列

sync で実差分が見つかった場合（dry-run 指定がない時）、リバースコード環境をオリジナルコード環境に揃える。整列は git 操作（checkout / merge）ではなくファイル転写で行い、両ブランチの歴史を混ぜない。

── 整列の手順 ───────────────────────────────────────────────

  1. 保全    リバースコード環境のポート設定ファイル群を退避
  2. 転写    オリジナルコード環境 → リバースコード環境へミラーコピー
             （削除同期あり。除外は 8-1 の diff_exclude から
              *.lock を除いたもの。lock ファイルは転写対象に含める）
  3. 再生成  当該 <scope> のスロットから算出したリバース側ポートで
             リバースコード環境側のポート設定を再生成
  4. 再構築  node_modules_strategy で分岐する（auto は常に npm_ci に
             解決済み。symlink_main は人間の明示指定時のみ）:
             npm_ci（既定）: install_command で各環境に独立インストール。
                            lock 不整合による失敗は ERROR
             symlink_main（明示指定時のみ）:
                            source_repo の node_modules へ symlink を張る
  5. 掃除    node_modules_strategy=npm_ci のみ実施。config.yml の
             bundler_cache_dirs を削除。symlink_main（明示指定時のみ）
             では .vite が source_repo と共有されるためスキップし、
             §8-2 の bundler-cache-fresh 但し書きに従う
  6. 再検証  Phase 3（静的）→ Phase 5（環境＋動的）へ合流し、
             収束ループを継続
