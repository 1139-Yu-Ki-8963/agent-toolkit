§1概要

工程全体は orchestrating-reverse-docs-flow が案内する。本スキルはポータルの将来ページ受け口のうち状態遷移図（T7）のみを担い、単独起動できる（起動引数を渡せば動く）。

データ設計.md の §6 状態遷移表（生成済み。generating-reverse-common-docs が採録する）をエンティティの状態遷移の正とする。表の各行を機械的に nodes/edges へ変換し、状態遷移図.html として書き出す。本スキルは判定・評価を一切行わない。状態遷移表に記載された遷移のみを転記し、記載がない状態遷移を推測・補完しない。状態遷移表が未採録、または該当章に行が 1 件もない場合は、生成せず事実として停止報告する。

設計原則内容
抽出元は単一データ設計.md の §6 状態遷移表 1 箇所のみを一次情報とする。対象リポジトリへの新規走査は行わない（根拠パスの実在検証のみ --target-repo で行う）
表外の遷移は捏造しない状態遷移表に記載がない状態・遷移をコード調査や推測で補わない
行 0 件は停止§6 状態遷移表に行が 1 件もない場合、page-data を生成せずユーザーへ報告して停止する
固定と可変の分離整合検証（validate-page-data.sh）と HTML 生成（build-detail-page.sh）は決定的スクリプトに固定する。抽出（表の行→nodes/edges への変換）は Claude 自身が Bash/Read で行う

§2出力

本スキルを実行すると、以下の 1 点が生まれる。

状態遷移図.html
<output_dir> 直下に固定生成される、エンティティごとの状態遷移図ページ。データ設計.md §6 状態遷移表の各行を nodes（状態）・edges（遷移。契機と根拠パス付き）に機械変換し、検証済みの page-data から build-detail-page.sh が決定的に組み立てる。エンティティ絞り込みセレクタで画面を切り替えながら、どの状態からどの契機でどの状態へ遷移するかを、根拠パス付きで確認できる。

§3使用タイミング

区分内容
前提データ設計.md の §6 状態遷移表が確定済みで、ポータルに状態遷移図カードを追加したいとき
起動引数target_repo_path（対象リポジトリの絶対パス。根拠パスの実在検証に使用）・output_dir（データ設計.md 所在 / 状態遷移図.html 出力先）・portal_output_dir（任意）
出力先<output_dir>/状態遷移図.html に固定（build-detail-page.sh の get_page_filename と同値）
前提ファイル既定パス<output_dir>/プロジェクト共通/データ設計.md

portal_output_dir を指定した場合、生成後に build-portal.sh を再実行する。ただし状態遷移図（entity-state）のポータルカード受け口は本スキル作成時点で build-portal.sh に未配線であり、再実行してもカードが増えないことがある（別途の配線作業を要する。本スキルの責務外）。

§4用語

状態遷移表
データ設計.md §6 の表。列はエンティティ／状態／遷移前／契機／遷移後／根拠パス。本スキルが参照する唯一の一次情報。
nodes
(エンティティ, 状態) の組を集めた一覧。key は <エンティティ>.<状態>（例: 注文.下書き）、label は状態名そのもの、entity はエンティティ名。
edges
表の各行を 1 edge に変換したもの。from/to は nodes[].key、trigger は契機列の値、sourceRef は根拠パス列の値。
孤児参照
edges[].from/to が nodes[].key に存在しない状態。validate-page-data.sh が機械検査する。

§5Phase 手順

スキル開始時に TaskCreate で Phase 1〜4 のタスクを登録する。各 Phase 開始時に該当タスクを in_progress に、完了時に completed へ TaskUpdate で更新する。

Phase内容完了条件
1. 前提確認データ設計.md §6 状態遷移表の実在確認。データ行数の確認（プレースホルダ行のみは 0 件扱い）表の実在・データ行 1 件以上を確認済み（または不在・0 件を報告して停止）
2. 抽出Step 1: nodes 組み立て（状態列 + 遷移前/遷移後からの補完）。Step 2: edges 組み立て。Step 3: page-data.json 組み立て（pageKind: "entity-state"、legend/nodes/edges/unresolved）表の全行から nodes[]・edges[] へ変換済み
3. 整合検証validate-page-data.sh <page-data.json> --target-repo <target_repo_path> を実行。孤児参照検査を含むvalidate-page-data.sh --target-repo が全項目 PASS
4. HTML 生成build-detail-page.sh <page-data.json> <output_dir> --page entity-state を実行。portal_output_dir 指定時は build-portal.sh を再実行<output_dir>/状態遷移図.html が生成済み

page-data.json の保存先は $CLAUDE_JOB_DIR/tmp/entity-state-page-data.json（未設定時は ${TMPDIR:-/tmp}/claude-job-${session}/tmp/ 配下）。手作業でのプレースホルダ置換は禁止し、HTML 生成は必ず build-detail-page.sh 経由の決定的処理で行う。

§6ループ設計

要素内容
反復対象Phase 3 Step 1 が FAIL → Step 2 の分岐で修正して再実行
上限回数3 回
収束停止validate-page-data.sh が exit 0（孤児参照検査含め全項目 PASS）
発散検知同一検査項目の同一 FAIL が 2 回連続で再発した場合は即座に Phase 2 Step 1（nodes 組み立て）へ差し戻す
リソース上限3 回失敗で Phase 2 Step 1 へ差し戻す

§7重要な注意事項・予想を裏切る挙動

判定・評価はしない。状態設計の良否・遷移の妥当性には一切踏み込まず、状態遷移表に記載された遷移の事実のみを転記する。状態遷移表に記載がない遷移を AskUserQuestion で聞き出さない。対象リポジトリへの書き込み・変更は一切行わない。shared/scripts/build-portal.sh は本スキルの責務外であり編集しない。
予想を裏切る挙動内容
出力先の階層<output_dir> 直下（「状態遷移図」のような種別専用フォルダは作らない）。build-detail-page.sh の --page entity-state 固定出力名仕様に従う
nodes.key の複合形式<エンティティ>.<状態> 形式に固定する（ER図の entities[].key のような単一識別子ではなく、複数エンティティを横断する状態名の衝突を避けるための複合キー）
自己遷移の扱いfrom/to が同一状態を指す行があってもそのまま転記する。孤児参照には該当しない
portal_output_dir 未指定時build-portal.sh を実行しない。生成済み状態遷移図.html はそのまま残り、entity-state のカード受け口が配線された時点で次回ポータル生成時にカード化される
