API一覧生成スキル
generating-api-list-for-reverse-docs

既存コードベースを調査し、API エンドポイントの単位にファイルをグルーピングした「API一覧.html」を、機械検証を通したマニフェストだけから生成するスキルの確定仕様。リバース設計で API 単位の設計書をどう分けるかを決めるための正本を作る。

対応 OSWindows WSL2 環境
検証状況分割元スキル generating-unit-list-for-reverse-docs（unit_kind=api）の枠組みを踏襲。エンジンスクリプト（validate-manifest.sh・build-unit-list.sh）はサンプルプロジェクトで動作確認済み
依存jq ／ shared/scripts/unit-list/validate-manifest.sh ／ shared/scripts/unit-list/build-unit-list.sh

§1概要

リバース設計では、設計書を書き始める前に「何を 1 冊の設計書の単位とするか」を確定させる必要がある。本スキルはその API 版を担い、既存コードベースから API エンドポイントの一覧 を調査・抽出し、機械検証済みの一覧 HTML として確定させる。

手順は「スタック調査 → 検出戦略の宣言（ユーザー承認）→ 戦略に基づく抽出 → 整合検証 → HTML 生成」の順で進む。抽出は Claude が対象プロジェクト固有の API 規約を調べてから行い、その結果は必ず検証スクリプトを通してから HTML になる。汎用の正規表現を無条件に当てて境界を取り違えることを防ぐ設計である。

本スキルの守備範囲は API一覧.html の作成まで。設計書（API 詳細設計書等）の雛形展開・生成・記入は一切行わない。画面・テーブル等の他種別は、それぞれの種別別一覧スキルが担当する。

§2出力

このスキルを実行すると、以下の 2 点が生まれる。

API一覧.html（<output_dir>/API一覧/API一覧.html）
対象コードベースの API エンドポイントを 1 行 1 件で一覧化した HTML。各行にエンドポイントの識別子（HTTP メソッド + パス）・推定名・主ファイル・確からしさ・検出手法が書かれており、読んだ人は「このシステムにはどんな API が何本あり、どのファイルが実体か」を一目で把握できる。未解決（境界を確定できなかったもの）と診断警告も隠さず可視化される。後続のリバース設計工程は、この一覧を設計書単位の判断材料として使う。
検証済みマニフェスト（HTML 内に埋め込み）
一覧の元データである JSON。検出戦略（ユーザーが承認した抽出方針）と全エンドポイントの構造化データを持ち、整合検証スクリプトの全項目 PASS を通過したものだけが HTML に埋め込まれる。機械可読なので、後続工程やスクリプトがそのまま読み取れる。

§3用語

エンドポイント
HTTP メソッドとパスの組で識別される API の 1 単位（例: GET /users/:id）。本スキルが数え上げる対象。
マニフェスト
抽出結果を構造化した JSON。配列キーは units。各行は kind（endpoint / middleware / unresolved）・識別子・主ファイル・確からしさを持つ。スキーマの全文は references/api-detection.md を参照。
検出戦略宣言
Phase 1 の調査結果をまとめた抽出方針の JSON（unitKind: "api"・extractionMethod: "custom"・ID 命名パターン・除外パターン）。ユーザー承認（approvedByUser: true）を得てから抽出に進む。
カスタム抽出パス
Claude 自身が戦略宣言に沿ってプロジェクト専用の抽出手順を設計・実行する経路。API 種別には画面種別のような組み込み検出器が無いため、常にこの経路を使う。
整合検証（validate-manifest.sh）
マニフェストのスキーマ・重複キー・未解決の隔離などを、抽出者に依存しない同一基準で機械検査するスクリプト。全項目 PASS しない限り HTML 生成には進めない。
unresolved（未解決）
定義の存在は確認できたが、メソッド・パス・実装ファイルのいずれかを確定できなかった行。捏造して endpoint にせず、未解決のまま可視化する。

§4動作フロー（Phase 1〜4）

── 全体の流れ ───────────────────────────────────────────

 Phase 1                Phase 2              Phase 3           Phase 4
┌──────────────┐   ┌──────────────┐   ┌─────────────┐   ┌─────────────┐
│ スタック調査    │   │ カスタム抽出    │   │ 整合検証      │   │ HTML 生成     │
│ API 規約特定   │──▶│ マニフェスト    │──▶│ validate-    │──▶│ build-unit-  │
│ 戦略宣言       │   │ JSON 出力     │   │ manifest.sh  │   │ list.sh      │
│ （ユーザー承認）│   │ （0件は停止）   │   │ 全項目 PASS   │   │ API一覧.html  │
└──────────────┘   └──────────────┘   └──────┬──────┘   └─────────────┘
                          ▲                  │ FAIL（3回まで修正）
                          └──────────────────┘
Phase内容完了条件
Phase 1
スタック・API規約の特定		 フレームワークの確定 → エンドポイント定義の所在特定 → ID 命名パターン・ミドルウェア方式の調査 → 除外パターンの確定 → 検出戦略宣言の作成と承認取得。調査項目の詳細は references/api-detection.md 戦略 JSON がユーザー承認済みで保存されている
Phase 2
戦略に基づく抽出		 宣言した手順（ルート定義の走査・デコレータ収集・OpenAPI 定義の解析等）を実行し、スキーマ準拠のマニフェスト JSON を出力する。0 件検出ならハード停止して報告する マニフェストが 1 件以上確定、または 0 件を報告して停止
Phase 3
整合検証（機械実行）		 validate-manifest.sh <manifest.json> --unit-kind api を実行。FAIL は指摘に応じて修正し再実行。3 回失敗で抽出手順の再検討へ差し戻す 全項目 PASS（exit 0）
Phase 4
API一覧.html 生成		 build-unit-list.sh <manifest.json> <output_dir>/API一覧/API一覧.html --unit-kind api を実行。build 側が内部で validate を再実行する HTML が生成され、埋め込み JSON がマニフェストと一致

入力（起動契約）

引数内容

output_dir出力先ディレクトリ。この直下に API一覧/ フォルダが作られる

種別を指定する引数は無い。本スキルは api 専用であり、種別は固定されている。

§5検証内容

抽出が Claude の手作業であっても品質を機械保証するため、HTML になる前に必ず 2 段の検証を通す。

段実行主体検査内容
整合検証 validate-manifest.sh --unit-kind api マニフェストのスキーマ準拠・重複キー・未解決行の隔離・戦略宣言の承認済みフラグ（approvedByUser: true）の存在を、抽出者非依存の同一基準で検査する
生成時再検証 build-unit-list.sh（内部） HTML 生成の直前に validate を再実行する。検証を経ないマニフェストからは物理的に HTML を生成できない
エンジンスクリプトの参照方法: 両スクリプトはスキルフォルダからの相対パス ../../../shared/scripts/unit-list/ で参照する。正本リポジトリと公開先はディレクトリレイアウトが同一のため、この相対参照は両環境でそのまま成立する。jq が必要（未導入環境では事前に導入する）。

§6禁止事項と注意点

項目内容
手作業プレースホルダ置換の禁止HTML はスクリプト経由の決定的処理でのみ生成する。手作業の置換で entryFile=None が混入した実害があるため
0 件検出時のハード停止エンドポイントが 1 件も見つからなければ停止して報告する。手動リストを聞き出して誤った境界を即興確定させない。エンドポイントを捏造しない
設計書の雛形展開はスコープ外本スキルの成果物は API一覧.html のみ。設計書の生成・記入は後続工程が担う
汎用正規表現の禁止Phase 1 の調査を省略して汎用の ID 命名パターンを当てない。命名規約・ルーティング方式はプロジェクトごとに異なる
コメント内定義の除去コメントアウトされたルート定義・import 文を除去してから抽出する（コメント内の定義を実在として誤検出した実害がある）
動的ルートの扱い変数結合・ループ登録で組み立てられるパスは静的走査で取りこぼしやすい。展開ロジックを追跡して確定できなければ unresolved に降格する
ミドルウェアの数え方認証・バリデーション等の共通処理はエンドポイントとして数えず、kind: middleware の別区分で記録する
