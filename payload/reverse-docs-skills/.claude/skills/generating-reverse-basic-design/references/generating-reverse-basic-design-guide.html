封印済みfactsからの基本設計書執筆スキル
generating-reverse-basic-design

封印済み facts（facts.yml）とプロジェクト共通文書だけを情報源に、業務語彙のみで書かれた画面基本設計書を著述する執筆専任スキル。対象リポジトリの原本コードは一切 Read しない。

§1概要

本スキルは generating-reverse-detailed-design と同じく「原本を読まずに書く」執筆役である。原本を読むのは上流の extracting-unit-facts-from-code だけであり、本スキルはその成果物（封印済み facts.yml）とプロジェクト共通文書だけを情報源に、業務語彙のみで書かれた画面基本設計書を著述する。

執筆専任・業務語彙限定: facts抽出（→ extracting-unit-facts-from-code）・詳細設計執筆（→ generating-reverse-detailed-design）は本スキルの範囲外。基本設計書はコード識別子・フレームワーク用語・型構文・ファイルパス・ライブラリ名を一切含めない、業務担当者が読める記述に限定する。
原本 Read 全面禁止・詳細設計非依存: 本スキル実行中に対象リポジトリの原本コードを Read することは全面禁止する。また詳細設計書を参照・要約しない。基本設計書は facts.yml から独立して業務語彙で直接書く成果物であり、詳細設計書の劣化版・要約版ではない。

§2出力

画面 1 つにつき、以下の設計書が生まれる。

画面基本設計書
facts.yml に記録された画面の挙動（機能・業務ルール・入出力・画面遷移）を、コード識別子や型構文を含まない業務語彙だけで再構成したドキュメント。業務担当者がコードを読まなくても、その画面が業務上「何のために存在し」「何ができ」「何を扱うか」を把握できる。

§3用語

facts_ref
extracting-unit-facts-from-code が確定した、封印済み facts ディレクトリの絶対パス（<verification_dir>/screen-<画面ID>/facts/<run_id>/）。facts.yml・facts.lock・recount-report.txt を含む。
common_docs_root
generating-reverse-common-docs が採録したプロジェクト共通文書のルート（プロジェクト共通/）。§5 画面遷移の業務文脈で共通設計書を参照引用する際に使う。
封印検証
shared/scripts/seal-facts.sh verify <facts_ref>による、facts.yml が封印時から改変されていないことの確認。Phase 1の必須ゲート。
unit_kind
screen / batch / report / external を区別する契約上のパラメータ。現時点で実装済みなのは screen のみ。

§4使い方（起動引数）

全量指定・対話ゼロ。単独起動時は下表の args をユーザーから直接取得する。

引数必須内容
screen_dir必須画面ディレクトリ絶対パス
output_dir必須設計書展開先ルート
template_root必須テンプレート原本ルート
scaffold_script_path必須スキャフォールディングスクリプト（実体: shared/scripts/scaffold-screen.sh。正本はこの1本のみ）
facts_ref必須封印済み facts ディレクトリの絶対パス
common_docs_root必須プロジェクト共通文書ルートの絶対パス
unit_kind任意（既定 screen）screen / batch / report / external。screen 以外は未実装のため status=基本設計著述失敗
chapter_map_path・audit_script_path は受け取らない。基本設計書は5章固定のテンプレートであり、章役割キーの解決も15章監査も不要なため。

§5設計原則

原則内容
正本一元化facts.yml は封印済みの確定情報であり第二の正本ではない。正は基本設計書。食い違ったら基本設計書を直す
「該当なし」に根拠必須根拠なしの裸の「未確認」は完了条件違反
プロジェクト非依存固有値は起動引数・設計書側に置き、SKILL.md本文には書かない
原本Read禁止本スキル実行中に対象リポジトリの原本コードをReadすることを全面禁止する。情報源はfacts_refとcommon_docs_rootに限定する
業務語彙限定コード識別子・フレームワーク用語・型構文・ファイルパス・ライブラリ名を一切含めない
詳細設計非依存詳細設計書を参照・要約しない。facts.ymlから独立して業務語彙で直接書く

§6動作フロー（Phase 1〜3）

Phase 1: テンプレート展開と facts 読込
  必須引数を検収する(screen_dir/output_dir/template_root/scaffold_script_path/
   facts_ref/common_docs_root/unit_kind)。unit_kindがscreen以外ならBLOCKED相当。
  画面ディレクトリ未存在ならscaffold_script_path(実体: shared/scripts/scaffold-screen.sh)で展開、
   既存なら--verifyで健全性確認
  seal-facts.sh verify  を実行、exit 0を確認(必須ゲート)
  exit 1なら status=基本設計著述失敗(hint: extracting-unit-facts-from-codeで再封印せよ)
  exit 0ならfacts.ymlとcommon_docs_root配下の共通文書を読み込む
  対象リポジトリの原本コードは一切Readしない
  完了条件: 画面ディレクトリの構造健全性確認済み・seal-facts.sh verifyがexit 0・facts.ymlと共通文書の読込完了

Phase 2: facts → 業務語彙への転記（章ごとに実施）
  facts.ymlの各セクション(state/handler/jsx/api/meta.route)を転記マップ(§7)に従って各章へ転記する
  import/export_type/const/style/measurement_pendingは転記しない(詳細設計書の担当)
  facts.ymlに該当分類が無い場合は「該当なし」+根拠を記す
  完了条件: テンプレートの5章(§1〜§5)すべてに記述がある

Phase 3: 実装用語混入検査
  画面基本設計書.md全文に対しgrepでコード識別子・フレームワーク用語・型構文・
   ファイルパス・ライブラリ名の混入を検査する
  検出0件で完了。1件でも検出されたら該当箇所を書き直してPhase 3を再実行(上限3回)
  完了条件: 実装用語検出0件

§7facts.yml → 章の転記マップ

facts.yml のスキーマ正本は shared/references/facts-schema.md（9分類）。基本設計書が使うのはこのうち5分類のみ。

facts.yml セクション基本設計書の章変換規則
handler§2 機能仕様業務動作に翻訳する（例: onClick → 「ボタン押下時の処理」）
jsx§1 画面の目的業務目的に翻訳する（画面が「何を見せるか」）
api§4 入出力の業務的意味業務目的に翻訳する（「何のデータをやり取りするか」）
state§3 業務ルール業務的な制約・条件に翻訳する
meta.route（+共通設計書）§5 画面遷移の業務文脈共通設計書の遷移情報を参照引用する

import / export_type / const / style / measurement_pending（実装寄り・実測系の5分類）は基本設計書に転記しない（詳細設計書の担当）。各章のキーは facts.yml のキーをそのまま流用せず、業務語彙で新規に付け直す。

§8実装用語混入検査

合否は grep の検出件数のみで判定する（自然文の自己申告での合格判定は行わない）。

項目内容
入力生成した画面基本設計書.md 全文
検査パターンuseState|useEffect|useReducer|Props|styled-components|React|Vue|Angular|interface [A-Z]|: (string|number|boolean)|ファイル拡張子付きパス
判定検出0件で完了。1件でも検出されたら該当箇所を書き直して再検査（上限3回）

§9返却ブロック

orchestrating-reverse-docs-flow の契約（返却ブロック共通サブセット: status/scope/artifacts/hint）に準拠する。

キー値
status基本設計著述完了 または 基本設計著述失敗（facts未封印・unit_kind未実装・実装用語混入が上限内で解消しない等）
scope<system>-<画面ID>（工程を跨いだ同一性キー）
artifacts画面基本設計書のパス
facts_ref（拡張）入力で受け取ったfactsディレクトリの絶対パスをそのまま転記（下流工程への追跡用）
hint次工程（generating-reverse-detailed-design 起動）への申し送り・差し戻し理由

§10ループ設計

要素内容
反復条件Phase 3のgrep検査で実装用語が1件でも検出されたら、該当箇所を書き直してPhase 3を再実行する
上限回数3回
停止条件収束停止: grep検出0件 ／ リソース上限: 3回到達（未収束なら基本設計著述失敗）
Phase 1ゲートとの違いPhase 1の封印検証失敗（exit 1）はこのループの対象外。即status=基本設計著述失敗の終端条件
サブエージェント委任facts読込・執筆(Phase 1〜2)はサブエージェントへ委任しない。単一のメインエージェントが業務語彙の一貫性を保って通しで担う

§11予想を裏切る挙動

  • 原本コードのReadは全面禁止。情報源はfacts_ref配下のfacts.ymlとcommon_docs_root配下の共通文書のみ
  • 詳細設計書は参照しない。基本設計書はfacts.ymlから独立して業務語彙で直接書く（詳細設計の劣化版・要約版ではない）
  • chapter_map_path・audit_script_pathは受け取らない。基本設計書は5章固定のテンプレートであり、章役割キーの解決も15章監査（audit-consistency.sh）も不要。本スキル独自のPhase 3 grep検査で完結する
  • unit_kindは受け取るがscreen以外は未実装。batch/report/externalの基本設計書テンプレートが存在しないため、無理にscreen用テンプレートを流用しない
  • facts.ymlのimport/export_type/const/style/measurement_pending（実装寄り・実測系の5分類）は基本設計書に一切転記しない
  • 本SKILL.md本文にリバース対象の固有値（対象リポジトリパス・画面ID・業務名）を書かない
  • 進捗はStep単位でTaskCreate/TaskUpdateする（一括登録しない）
