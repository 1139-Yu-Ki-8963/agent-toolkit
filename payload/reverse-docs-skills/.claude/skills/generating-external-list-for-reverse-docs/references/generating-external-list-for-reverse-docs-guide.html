外部連携一覧生成スキル
generating-external-list-for-reverse-docs

既存コードベースを調査し、外部連携（API クライアント・webhook ハンドラ・メッセージキュー連携等）の単位にファイルをグルーピングして「外部連携一覧.html」を作成するスキルの確定仕様。リバース設計の工程で、設計書をどの単位で書くかを決める判断材料を作る。

対応 OSWindows WSL2 環境

§1概要

リバース設計では、設計書を書き始める前に「対象システムに外部連携がいくつあり、それぞれどのファイル群で実装されているか」の正本が要る。本スキルはこの正本となる外部連携一覧.html を、スタック調査 → 検出戦略の宣言 → 戦略に基づく抽出 → 整合検証、の 4 段階で作成する。

ユニット種別（画面・API・テーブル・バッチ・帳票・外部連携）を 1 つの引数で切り替えていた旧スキル（generating-unit-list-for-reverse-docs）から、外部連携専用に分割したものが本スキルである。種別引数は持たず、unit_kind は常に external として動く。

外部連携には機械的な組み込み検出器が存在しない。抽出は Claude 自身がプロジェクトごとに設計するカスタム抽出パスのみだが、抽出結果のマニフェスト JSON は決定的スクリプト validate-manifest.sh が抽出者非依存で機械検証し、HTML 生成も決定的スクリプト build-unit-list.sh に固定する。手作業の混入余地を検証と生成の両側で塞ぐ設計である。

本スキルの仕事は外部連携一覧.html の作成のみ。設計書の雛形展開・生成・記入は一切行わない（後続工程の担当）。

§2出力

このスキルを実行すると、以下の 2 点が生まれる。

外部連携一覧.html（<output_dir>/外部連携一覧/ 配下）
対象システムの外部連携を 1 行 1 ユニットで一覧化した HTML。連携先の識別子・送信/受信の区分（client/webhook）・主ファイルと構成ファイル群・検出の確からしさ（confidence）が読み取れる。設計書を「どの連携単位で何冊書くか」を決める正本になる。
検証済みマニフェスト JSON（HTML に埋め込み）
一覧の元データ。ユーザー承認済みの検出戦略宣言と、validate-manifest.sh の全項目 PASS を経た内容だけが埋め込まれる。後続工程はこの埋め込み JSON を機械的に参照できる。

§3用語

外部連携（external ユニット）
プロセス外・組織外のシステムとの通信を担う実装単位。API クライアント・SDK 統合・キュー発行（送信側）と、webhook ハンドラ・キューコンシューマ（受信側）を含む。自プロジェクト内の別モジュール呼び出しは含めない。
検出戦略宣言
Phase 1 の調査結果をまとめ、ユーザー承認（approvedByUser: true）を取った JSON。連携の識別パターン・除外パターン・抽出手順の方針を固定し、以後の抽出はこの宣言に沿ってのみ行う。
マニフェスト
抽出結果の JSON。配列キーは units（screens は画面種別専用の後方互換キーで、本スキルでは使わない）。スキーマは references/external-detection.md が正本。
カスタム抽出パス
組み込み検出器を使わず、Claude 自身が戦略宣言に沿って Bash/Grep/Read で連携単位を収集し、スキーマ準拠のマニフェストを書き出す抽出経路。外部連携はこの経路のみを使う。
整合検証
validate-manifest.sh --unit-kind external による機械検証。スキーマ・重複キー・unresolved 隔離などを抽出者非依存で検査し、通過しないマニフェストからは HTML を生成できない。

§4使い方（起動引数と返却）

起動引数（2 つのみ）

引数内容
source_dir探索対象のソースコードディレクトリ。対象プロジェクトの実コードルート（例: backend/src）を指定する。モノレポでは対象アプリのディレクトリのみを渡す（自動判別なし）
output_dir外部連携一覧.html の書き出し先ディレクトリ。この配下に 外部連携一覧/ フォルダが作られる

旧スキルにあった unit_kind 引数は持たない（external 固定）。他種別の一覧が必要な場合は対応する種別別一覧スキルを使う。

返却（orchestrating-reverse-docs-flow の契約準拠）

キー内容
statusDONE / ERROR
artifacts生成した外部連携一覧.html のパス
unit_list_htmlartifacts[0] の汎用名
embedded_json_refHTML 内に埋め込んだマニフェスト JSON への参照

§5動作フロー

Phase 1〜4 で進む。開始時に TaskCreate で 4 つの Phase タスクを登録し、進行に合わせて TaskUpdate で更新する。

── 動作フロー（Phase 1〜4）─────────────────────────────

Phase 1  スタック・連携規約の特定
         依存ライブラリ → 連携定義の所在 → 識別パターン → 除外パターン
         └─▶ 検出戦略宣言（unitKind=external）をユーザー承認で確定
                     │
Phase 2  戦略に基づく抽出（カスタム抽出パスのみ）
         宣言どおりに連携単位を収集し、マニフェストJSONを書き出す
         0件なら報告してハード停止（連携を捏造しない）
                     │
Phase 3  整合検証（機械実行）
         validate-manifest.sh --unit-kind external ─ 全項目PASSまで
         3回FAILしたら Phase 2 の抽出手順へ差し戻し
                     │
Phase 4  外部連携一覧.html 生成
         build-unit-list.sh（内部でvalidate再実行）
         └─▶ <output_dir>/外部連携一覧/外部連携一覧.html
Phase完了条件
Phase 1調査 4 項目（スタック・所在・識別パターン・除外パターン）が完了し、検出戦略宣言がユーザー承認済み
Phase 2スキーマ準拠のマニフェスト（配列キー units）が 1 件以上確定、または 0 件検出を報告して停止している
Phase 3validate-manifest.sh --unit-kind external が全項目 PASS（FAIL 時の修正ループは 3 回以内）
Phase 4外部連携一覧.html が生成され、埋め込み JSON がマニフェストと一致している

§6検出対象と境界の判断基準

調査対象検出手法
外部連携定義の所在API クライアントラッパー、SDK 統合コード、webhook ハンドラ、メッセージキューコンシューマ
プロトコルREST/GraphQL/gRPC/SOAP/WebSocket/メッセージキュー
認証方式API キー/OAuth/証明書
除外パターンモック・スタブ

境界の判断基準

  • 数えるのは「プロセス外・組織外のシステムとの通信」を担う単位のみ。自プロジェクト内の別モジュール呼び出しは含めない
  • 同一連携先に対する複数ファイル（クライアント本体・型定義・設定）は 1 ユニットにまとめ、files に列挙する
  • 送信（kind: client）と受信（kind: webhook）は区別する。確定できないものは unresolved に隔離する

調査項目の Step 別詳細とマニフェストスキーマの正本は references/external-detection.md を参照。

§7制約・禁止事項

制約理由
HTML の手作業組み立て・プレースホルダの手動置換の禁止過去に entryFile=None の混入という実害が発生した。生成は必ず build-unit-list.sh 経由の決定的処理で行う
0 件検出時のハード停止AskUserQuestion で手動リストを聞き出して誤った境界を即興確定させない。連携を捏造しない
設計書の雛形展開・生成・記入はスコープ外本スキルの成果物は外部連携一覧.html のみ。設計書は後続工程が担う
Phase 1 の調査省略の禁止汎用の識別パターンを無条件に当てると境界を取り違える。プロジェクト固有の連携規約を先に確認する
モック・スタブの除外テスト用の偽クライアントを実連携として数えない。Phase 1 の除外パターンで先に隔離する

エンジンスクリプト（validate-manifest.sh / build-unit-list.sh）は jq に依存する。未インストール環境では事前に導入する。スクリプトはスキルフォルダ相対 ../../../shared/scripts/unit-list/ で参照する（正本リポジトリと公開先はレイアウト同一のため両環境で成立する）。
