バッチ一覧生成スキル
generating-batch-list-for-reverse-docs

既存コードベースからバッチ（定期実行ジョブ・トリガー起動ジョブ）の単位を洗い出し、検証済みマニフェストだけを材料に「バッチ一覧.html」を生成するスキルの確定仕様。種別はバッチ固定で、他種別（画面・API 等）は対応する種別別スキルが担う。

対応 OSWindows WSL2 環境
検証状況整合検証・HTML生成エンジン（validate-manifest.sh・build-unit-list.sh）は画面一覧生成で実プロジェクト動作確認済み。バッチ種別のカスタム抽出はこの仕様に基づく設計段階
依存jq / shared/scripts/unit-list/validate-manifest.sh / shared/scripts/unit-list/build-unit-list.sh

§1概要

リバース設計では、設計書を書き始める前に「このシステムにはどんなバッチが何本あるのか」という単位の正本が要る。本スキルは既存コードベースを スタック調査 → 検出戦略の宣言 → 戦略に基づく抽出 → 整合検証 の順に調査し、バッチの単位にファイルをグルーピングした「バッチ一覧.html」を作成する。

仕事は一覧の作成のみで、設計書の雛形展開・生成・記入は一切行わない。バッチ種別に組み込み検出器は無く、抽出は常にカスタム抽出パス（Claude 自身がプロジェクト専用の抽出手順を設計・実行する経路）を取るが、抽出結果は決定的スクリプト validate-manifest.sh が抽出者非依存で機械検証するため、手書き JSON でも品質が保証される。

設計原則: 固定と可変の分離。 マニフェストスキーマ・整合検証・HTML 生成は決定的スクリプトに固定し、プロジェクトごとに異なる「バッチ境界の検出」だけを可変にする。汎用の正規表現を無条件に当てるのではなく、対象プロジェクト固有のバッチ規約（スケジューラ・ジョブ命名）を先に確認してから検出する。

§2出力

スキルを完走すると、以下の 2 点が生まれる。

バッチ一覧.html（<output_dir>/バッチ一覧/バッチ一覧.html）
そのシステムに存在するバッチの全量を 1 冊にまとめた一覧。ジョブごとに推定名・起動方式（定期実行かトリガー起動か）・主識別子（ジョブ名や cron エントリ名）・実体ファイル・検出の確からしさが読める。設計書を「バッチ 1 本につき 1 冊」で書き分けるための単位の正本として機能し、起動方式を確定できなかったものは未解決として区別されているため、調査の残りも一目で分かる。
埋め込みマニフェスト JSON
一覧 HTML の中に埋め込まれた機械可読データ。整合検証（validate-manifest.sh）を通過した内容だけが埋め込まれるため、後続工程（facts 抽出・設計書執筆）がこの JSON を読み直せば、一覧と食い違わない単位情報を機械的に取り出せる。

§3用語

バッチ（ユニット種別 batch）
画面遷移を持たない自動処理の総称。cron 等で時刻起動される定期実行ジョブと、イベント・キュー投入・CLI コマンドで起動されるトリガー起動ジョブの両方を含む。
マニフェスト
検出したバッチ単位を列挙した JSON。配列キーは units。各行は unitKey（意味キー）・kind（scheduled / triggered / unresolved）・identifier・sourceFile・confidence を持つ（スキーマ全文は references/batch-detection.md）。
検出戦略宣言
Phase 1 の調査結果を固めた JSON（unitKind / extractionMethod: "custom" / unitIdRegex / excludePatterns / approvedByUser / notes）。ユーザー承認（approvedByUser: true）が抽出開始の前提条件で、整合検証がその存在を機械確認する。
カスタム抽出パス
Claude 自身が戦略宣言に沿って Bash/Grep/Read で抽出手順を実行し、スキーマ準拠のマニフェストを書き出す経路。バッチ種別ではこの経路のみを使う（組み込み検出器なし）。
整合検証
validate-manifest.sh によるマニフェストの機械検査。スキーマ・重複キー・未解決の隔離を、抽出者に依らない同一基準で判定する。

§4使い方（起動引数）

引数内容
source_dir探索対象のソースコードディレクトリ。対象プロジェクトの実コードルート（例: backend/src）を指定する。モノレポでは対象アプリのディレクトリのみを渡す（自動判別なし）
output_dir出力先ディレクトリ。この直下に バッチ一覧/バッチ一覧.html が作られる

種別は batch 固定のため、種別を選ぶ引数は無い。工程全体は orchestrating-reverse-docs-flow が案内するが、本スキルは上記 2 引数だけで単独起動できる。

§5動作フロー（Phase 1〜4）

── バッチ一覧が生まれるまで ───────────────────────────────

 Phase 1  スタック・バッチ規約の特定
          依存定義からスケジューラ特定 → バッチ定義の所在列挙
          → 識別要素（ジョブ名・cron 式・入出力）調査 → 除外確定
          → 検出戦略宣言（ユーザー承認）
              │
              ▼
 Phase 2  戦略に基づく抽出（カスタム抽出パスのみ）
          宣言した手順で走査し、マニフェスト JSON を書き出す
          0 件検出はハード停止（バッチを捏造しない）
              │
              ▼
 Phase 3  整合検証（機械実行）                 ┌─ FAIL: 修正して再検証
          validate-manifest.sh --unit-kind batch ┤   （3 回失敗で Phase 2 へ差し戻し）
              │ 全項目 PASS                    └───────────────
              ▼
 Phase 4  バッチ一覧.html 生成
          build-unit-list.sh（内部で validate を再実行）
          → <output_dir>/バッチ一覧/バッチ一覧.html
Phase完了条件
Phase 1スケジューラ・バッチ定義所在・識別要素・除外パターンの調査完了。検出戦略宣言がユーザー承認済み
Phase 2スキーマ準拠のマニフェストが 1 件以上確定、または 0 件検出を報告して停止している
Phase 3validate-manifest.sh --unit-kind batch が全項目 PASS。FAIL 時の修正ループは 3 回以内
Phase 4バッチ一覧.html が生成され、埋め込み JSON がマニフェストと一致している

エンジンスクリプトはスキルフォルダ相対の ../../../shared/scripts/unit-list/ 配下を参照する。正本リポジトリと公開先はディレクトリレイアウトが同一のため、この相対参照は両環境で成立する。進捗は Phase 1〜4 のタスク登録（TaskCreate/TaskUpdate）で管理する。

§6禁止事項と注意点

項目内容
手作業プレースホルダ置換の禁止HTML は必ず build-unit-list.sh 経由の決定的処理で生成する。手動置換による誤データ混入（画面種別で entryFile=None が 10 件混入した実害）を根絶するための固定ルール
0 件検出時のハード停止1 件も検出できなかったら停止して報告する。その場で手動リストを聞き出して即興確定させない（誤った境界の混入防止）
設計書はスコープ外設計書の雛形展開・生成・記入は行わない。成果物はバッチ一覧.html のみ
コメント内定義の除去ソース解析時はコメントアウトされたジョブ定義・import 文を除去してから抽出する（コメント内の定義を実在として誤検出した実害あり）
動的なジョブ名・cron 式変数結合等で動的に組み立てられるものは静的走査で確定できない。confidence: low または unresolved として可視化し、断定しない
jq 依存validate-manifest.sh・build-unit-list.sh は jq に依存する。未導入環境では事前に導入する

§7関連スキル
