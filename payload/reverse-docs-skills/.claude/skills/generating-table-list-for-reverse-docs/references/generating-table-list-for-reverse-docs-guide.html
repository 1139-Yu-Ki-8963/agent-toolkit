テーブル一覧生成スキル
generating-table-list-for-reverse-docs

既存コードベースからテーブル定義（マイグレーション・ORM モデル・SQL DDL）を調査し、テーブル単位の一覧である「テーブル一覧.html」を機械検証付きで生成するスキルの確定仕様。

こんな人向け
リバースエンジニアリング設計で DB スキーマの全体像を先に確定したい人 ／ 設計書を書き始める前にテーブルの総数と境界を正確に把握したい人
こんな場面で使う
リバース設計の初期工程でテーブル一覧を確立する時 ／ マイグレーションが積み重なったプロジェクトで「実テーブルは何個あるか」を機械的に数えたい時
対応 OSWindows WSL2 環境
検証状況旧統合一覧スキル（unit_kind=table。現在は種別別6スキルへ分割済み）から分割。エンジンスクリプト（validate-manifest.sh / build-unit-list.sh）は分割元でサンプルプロジェクトの動作確認済み。分割後スキル単体の実機検証は未実施
依存jq ／ shared/scripts/unit-list/validate-manifest.sh・build-unit-list.sh（スキルフォルダ相対 ../../../shared/scripts/unit-list/）

§1概要

リバースエンジニアリング設計では、設計書を書き始める前に「対象は全部で何個あるか」の正本が要る。本スキルはテーブル種別（DB テーブル・ビュー）専任の一覧生成係として、既存コードベースを調査し、テーブル一覧.html を機械検証付きで作成する。

調査は場当たりに行わない。スタック調査（どの ORM・マイグレーションツールか）→ 検出戦略の宣言（何をどう数えるかのユーザー承認）→ 戦略に基づく抽出 → 整合検証、の順で進み、検証を通過したマニフェストだけが一覧 HTML になる。

── 処理の全体像 ──────────────────────────────

  対象コードベース
       │  Phase 1  スタック・テーブル規約の特定（ORM / マイグレーション / SQL DDL）
       ▼
  検出戦略宣言 ──── AskUserQuestion でユーザー承認
       │  Phase 2  戦略に沿ったカスタム抽出（Claude 自身が実行）
       ▼
  マニフェスト JSON（配列キー units）
       │  Phase 3  validate-manifest.sh --unit-kind table で機械検証
       ▼
  検証済みマニフェスト
       │  Phase 4  build-unit-list.sh（内部で validate 再実行）
       ▼
  <output_dir>/テーブル一覧/テーブル一覧.html

画面種別と違い、テーブル種別に組み込み検出器はない。抽出は常にカスタム抽出パス（Claude 自身が戦略宣言に沿って実行）だが、整合検証と HTML 生成は決定的スクリプトに固定されるため、成果物の品質は抽出者に依存しない。

§2出力

スキルを実行すると、以下の 2 点が生まれる。

テーブル一覧.html（<output_dir>/テーブル一覧/ 配下）
対象コードベースに存在するテーブル・ビューの全量を 1 行 1 ユニットで示す一覧。各行にはテーブル名（主識別子）・定義ファイルの所在・関連マイグレーション・検出の確からしさが記載され、後続工程が「どの単位で設計書を 1 冊ずつ書くか」を判断する正本になる。検証を通過したマニフェストだけから生成されるため、根拠のない行は混入しない。
埋め込みマニフェスト JSON
一覧 HTML の中に、生成の元になったマニフェスト JSON がそのまま埋め込まれる。読んだ人は「この一覧がどの戦略（走査対象・除外パターン）で、いつ、何件検出されたか」を HTML 単体で追跡でき、機械処理（後続スキルからの参照）にも使える。

設計書そのもの（テーブル定義書等）の雛形展開・生成・記入は行わない。本スキルの成果物は一覧のみ。

§3用語

ユニット（テーブル種別）
一覧の 1 行に対応する管理単位。実テーブル（kind: table）・ビュー（kind: view）・集約できない独立マイグレーション（kind: migration）・実体を特定できないもの（kind: unresolved）の 4 区分を持つ。
マニフェスト
抽出結果を保持する JSON。配列キーは units。スキーマの全量は references/table-detection.md を参照。
検出戦略宣言
Phase 1 の調査結果をまとめた JSON（unitKind: "table"・extractionMethod: "custom"・unitIdRegex・excludePatterns 等）。AskUserQuestion でユーザー承認（approvedByUser: true）を得てから抽出に進む。
カスタム抽出パス
Claude 自身が戦略宣言に沿ってプロジェクト専用の抽出手順を設計・実行する経路。テーブル種別はこの経路のみ（組み込み検出器なし）。
整合検証
validate-manifest.sh によるマニフェストの機械検査。抽出者が誰か（どの経路か）に依存せず同一基準で検査し、通過しないマニフェストは一覧 HTML にならない。

§4動作フロー

起動引数は source_dir（探索対象）と output_dir（書き出し先）の 2 つ。種別は table に固定されているため、種別を選ぶ引数はない。開始時に TaskCreate で Phase 1〜4 のタスクを登録し、進捗を TaskUpdate で更新する。

Phase内容完了条件
Phase 1
スタック・テーブル規約の特定		 ORM・マイグレーションツール（Prisma/TypeORM/Knex/Alembic 等）の確定、テーブル定義の正本（マイグレーション／モデル／SQL DDL）の特定、命名パターン・除外パターンの調査、検出戦略宣言の作成 戦略宣言 JSON がユーザー承認済み
Phase 2
戦略に基づく抽出		 宣言した手順（マイグレーション走査・モデル解析・CREATE TABLE/CREATE VIEW 抽出等）を Claude 自身が実行し、マニフェスト JSON を出力 1 件以上の検出を確認済み。0 件なら報告して停止
Phase 3
整合検証（機械実行）		 validate-manifest.sh <manifest> --unit-kind table を実行。FAIL は修正して再実行（3 回失敗で Phase 2 へ差し戻し） 全項目 PASS（exit 0）
Phase 4
一覧 HTML 生成		 build-unit-list.sh <manifest> <output_dir>/テーブル一覧/テーブル一覧.html --unit-kind table を実行。build 側が内部で validate を再実行 HTML 生成済み・埋め込み JSON がマニフェストと一致

エンジンスクリプトはスキルフォルダ相対 ../../../shared/scripts/unit-list/ で参照する。正本リポジトリと公開先はディレクトリレイアウトが同一のため、この相対参照は両環境でそのまま成立する。

§5制約と禁止事項

手作業でのプレースホルダ置換は禁止。 一覧 HTML は必ず build-unit-list.sh 経由の決定的処理で生成する。手動置換による entryFile=None 等のデータ混入が実害として発生した経緯があるため。
0 件検出はハード停止。 テーブルが 1 件も検出できなかった場合はユーザーに報告して停止する。AskUserQuestion で手動リストを聞き出してテーブルを捏造しない。
設計書の雛形展開はスコープ外。 本スキルの成果物はテーブル一覧.html のみ。テーブル定義書等の雛形展開・生成・記入は一切行わない。

抽出時の落とし穴

落とし穴対処
マイグレーションと ORM モデルの二重計上どちらを正本とするかを Phase 1 で確定してから抽出する
create → alter の積み重ねを別テーブルと誤認テーブル単位に集約し、files に関連マイグレーションを列挙する
コメントアウトされた DDL の誤検出コメント（-- CREATE TABLE ... 等）を除去してから抽出する
jq 未インストール環境での検証・生成失敗validate/build 両スクリプトが jq に依存するため事前に導入する
