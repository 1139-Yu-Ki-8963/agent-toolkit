# 用語辞書の採録戦略ガイダンス（層定義・識別子復元規則・除外既定）

`SKILL.md` の Phase 1（採録方針の承認）・Phase 2（採録）が参照する詳細手順。ここでは「採録源をどう層に分けるか」「コード識別子からどう用語候補を復元するか」「何を既定で除外するか」に絞る。

## 層の定義（文書種別 × コード層の直積）

採録源は 2 系統からなり、それぞれ独立に層化する。層内の選定は `generating-reverse-common-docs` の `sampling-rules.md` と同じ決定的手順（`find`/`sort`/`head`。乱数・目視選定を禁止）に従う。

### 文書種別層

`<output_dir>/プロジェクト共通/` 配下の共通文書 10 種＋アーキテクチャ調査書を、以下の層に分ける。層内は「文書 1 件 = 対象 1 件」であり、追加のサンプリングは行わない（文書はそのまま全読する）。

| 層 | 対象文書 | 主な採録候補 |
|---|---|---|
| 規約層 | コーディング規約.md / 命名規約.md / ディレクトリ構成規約.md / コンポーネント設計規約.md | 略語・命名パターン用語 |
| 設計層 | 共通設計書.md / DESIGN.md / 基盤設計.md / UI共通設計.md / データ設計.md | 技術用語 |
| メッセージ層 | メッセージ定義書.md | 業務用語（ユーザー向け文言に現れる語） |
| 調査層 | アーキテクチャ調査書.md | 業務用語・技術用語の双方（§1〜§4 全体） |

### コード層

`target_repo_path` のコード識別子は、アーキテクチャ調査書の「ディレクトリ責務マップ」節の行をそのまま層として使う。これは `sampling-rules.md`「層の定義」節と同じ考え方である。調査書が既に対象リポジトリを走査済みのため、本スキルが独自に層を再定義することはしない。ディレクトリ責務マップに無いディレクトリは層に含めない。

各層内のファイル選定は `sampling-rules.md`「層あたり k 値の決め方」「決定的選択手順」をそのまま流用する（層内ファイル数 N の平方根切り上げ、3 以上 10 以下に丸める）。

### 直積の扱い

「文書種別層 × コード層」は独立集合として扱い、掛け合わせて選定件数を増やすことはしない。文書種別層は全読、コード層は層化サンプリングというように、採録方法自体が層系統ごとに異なる。そのため 2 系統の結果（文書由来候補・コード由来候補）は、Phase 2 Step 3 で統合する際に初めて突き合わせる。

## 識別子からの用語復元規則

コード層で選定したファイル内の識別子（変数名・型名・関数名・定数名）から用語候補を復元する手順。

1. **分解** — 識別子を構成語に分解する。
   - `camelCase` / `PascalCase` は大文字境界で分割する（例: `orderTotalAmount` → `order` / `total` / `amount`）
   - `snake_case` / `SCREAMING_SNAKE_CASE` はアンダースコアで分割する（例: `RETRY_LIMIT_COUNT` → `retry` / `limit` / `count`）
   - 数字は直前の語と結合せず独立トークンとして扱う（例: `user2Id` → `user` / `2` / `id`）
2. **複合語の再結合** — 分解した構成語のうち、ドメイン概念を表す隣接 2〜3 語（例: `order` + `total` + `amount` → 「注文合計金額」相当）を対象にする。対応する文書種別層の記述と突き合わせが取れたものだけを 1 用語として再結合する。文書側に対応する記述が無い場合は再結合せず、個別トークンのまま候補にするか、除外既定に該当すれば候補から外す。
3. **定義の復元** — 復元した語の定義は、以下の優先順位で確定する。文書側の記述が最優先であり、コード単独からの推測定義は作らない。
   - 優先度 1: 文書種別層に同一語（表記揺れ含む）の記述がある → その記述を定義として採用し、`sourceRef` は文書参照形式にする
   - 優先度 2: 文書側に記述が無いが、識別子の使用文脈（型定義・コメント・関数シグネチャ）から意味が一義的に読み取れる → その使用文脈を根拠に定義を記述し、`sourceRef` はコード実パスにする
   - 優先度 3: いずれにも該当しない → `terms[]` に含めず `unresolved[]` へ退避する（`reason` に「文書記述なし・コード文脈から意味を一義的に確定できない」旨を記録する）

## 除外既定

以下は既定で採録候補から除外する。Phase 1 Step 3 でプロジェクト固有の除外語を追加できる。

| 除外区分 | 内容 | 例 |
|---|---|---|
| 一般英単語 | ドメイン概念を持たない汎用語（型・データの入れ物・処理単位を指すだけの語） | `data` / `value` / `item` / `list` / `result` / `type` / `name` / `id`（単独） |
| フレームワーク API 名 | 使用フレームワーク・言語標準ライブラリが定義する識別子（プロジェクト固有の意味を持たない） | `useState` / `useEffect` / `Promise` / `Array` / `Object` / ORM の組み込みメソッド名（`save`/`findOne` 等） |
| 構造的トークン | 識別子分解で生じる助詞・接続語相当のトークン | `is` / `has` / `get` / `set`（プレフィックスとして機械的に付与される場合） |

除外判定は候補語の完全一致（大文字小文字を無視）で行う。複合語の構成要素として除外語が含まれていても、複合語全体が除外既定に一致しない限り複合語自体は除外しない。例えば `getOrderStatus` は `get` を含むが、複合語として再結合した「注文状態」相当の語は除外対象ではない。

## 記録様式

Phase 2 で実行した層ごとの選定コマンドと選定結果は、`sampling-rules.md`「サンプル記録.md の記載様式」と同じ形式で記録する。記録先は一時ファイル `$CLAUDE_JOB_DIR/tmp/glossary-sampling-record.md` とする。再現可能性の担保のため、同一入力に対して常に同一の選定結果になることを確認してから Phase 2 Step 3 へ進む。
