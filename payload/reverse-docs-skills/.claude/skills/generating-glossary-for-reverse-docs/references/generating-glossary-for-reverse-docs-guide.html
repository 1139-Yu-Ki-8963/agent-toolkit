用語辞書ページ生成スキル
generating-glossary-for-reverse-docs

プロジェクト共通文書・アーキテクチャ調査書・対象リポジトリのコード識別子を層化サンプリングで採録し、二段承認と機械検証を経て「用語辞書.html」を生成するスキルの確定仕様。

こんな人向け
ポータルの将来ページ受け口に用語辞書カードを反映したい人 ／ 設計文書・コードに散在する業務用語・技術用語・略語を一箇所に集約したい人
こんな場面で使う
プロジェクト共通文書とアーキテクチャ調査書が確定済みで、次にポータル基盤情報ページを揃えたい時 ／ 新規参加者向けに用語の出所（文書・コード）を追跡可能な形で残したい時
対応 OSWindows WSL2 環境
検証状況エンジンスクリプト（validate-page-data.sh / build-detail-page.sh）はフィクスチャでの動作確認済み。スキル単体の実機検証は未実施
依存jq ／ shared/scripts/detail-pages/validate-page-data.sh・build-detail-page.sh（スキルフォルダ相対 ../../../shared/scripts/detail-pages/）／ shared/scripts/build-portal.sh（任意・カード反映時）

§1概要

リバースエンジニアリング設計のポータルには、将来ページの受け口（docs_root 直下に該当 HTML が実在するときだけカード化される仕組み）がある。本スキルはそのうち用語辞書（T2）専任の生成係として、プロジェクト共通文書・アーキテクチャ調査書・対象リポジトリのコード識別子を層化サンプリングで調べ、用語辞書.html を二段承認と機械検証付きで作成する。

他の detail-pages 系スキルと異なり、用語辞書には構造化された一次情報（manifest・調査書表）が存在しない。本スキルは採録型を取り、採録源に実在する記述・識別子から用語候補を構築する。語の意味を創作せず、根拠の無い語は生成せず unresolved[] へ退避する。

── 処理の全体像 ──────────────────────────────

  プロジェクト共通文書 + アーキテクチャ調査書 + コード識別子
       │  Phase 1  採録方針（分類軸・採録源・除外パターン）の承認〈二段承認 1段目〉
       ▼
  文書由来候補 + コード由来候補（層化サンプリング）
       │  Phase 2  抽出・統合（根拠なき語は unresolved へ）
       ▼
  用語候補一覧
       │  Phase 3  候補一覧の取捨〈二段承認 2段目〉+ page-data.json 組み立て
       ▼
  page-data.json（pageKind: glossary）
       │  Phase 4  validate-page-data.sh --target-repo で機械検証 + build-detail-page.sh
       ▼
  <docs_root>/用語辞書.html（+ 任意で build-portal.sh 再実行）

承認は 2 箇所（採録方針・候補一覧）にあり、無人実行時はいずれも既定値での自動承認として扱う（詳細は §4「無人実行時の扱い」）。

§2出力

スキルを実行すると、以下の 2 点が生まれる。

用語辞書.html（<docs_root> 直下）
分類チップ（業務用語／技術用語／略語が既定）と検索ボックスで絞り込める用語一覧ページ。各行には定義・コード参照（codeRefs[]）・出所（sourceRef）が付き、読んだ人は語がどの文書またはどのコード識別子から採録されたかをその場で追跡できる。根拠の無い語があれば「未解決」節に理由付きで表示される。
埋め込み page-data JSON
一覧 HTML の中に、生成の元になった page-data.json がそのまま埋め込まれる。読んだ人は「いつ・どの採録源から生成されたか」を HTML 単体で追跡でき、機械処理（後続スキルからの参照）にも使える。

Phase 1（採録方針）・Phase 3（候補一覧）のいずれかで承認が得られない場合、本スキルは生成を行わず承認状況を報告して停止する。

§3用語

page-data.json
detail-pages 系 5 スキルが共有する入力 JSON。pageKind ごとに型別スロットを持つ。本スキルは pageKind: "glossary" を固定で使う。スキーマの全量は page-data-schema.md を参照。
categories / terms
glossary 型のスロット。categories[] は分類軸（既定: 業務用語／技術用語／略語）、terms[] は {term, definition, codeRefs[], category, sourceRef} 形式の用語 1 件 = 1 要素。
層化サンプリング
コード識別子の採録源を決定的な手順（find/sort/head）で選定する Phase 2 の処理。層定義・識別子の分解規則・除外既定は glossary-extraction.md を参照。
二段承認
Phase 1（採録方針）と Phase 3（候補一覧）の 2 箇所で行う承認。転記のみの他種別（techstack 等）と異なり、用語辞書は「何を採るか」と「採った結果をどう間引くか」の 2 つの意思決定を要するため。
unresolved
採録源に定義の根拠が無い語の退避先。{label, reason, sourceRef?} 形式。捏造せず「未解決」として可視化する。

§4動作フロー

起動引数は target_repo_path・docs_root（必須）と portal_output_dir（任意）の 3 つ。種別は glossary に固定されているため、種別を選ぶ引数はない。

Phase内容完了条件
Phase 1
採録方針の承認〈1段目〉		 分類軸・採録源（3 系統の実在確認）・除外パターンを確定し AskUserQuestion で承認を取る 採録方針が承認済み、または採録源不在を報告して停止
Phase 2
採録		 文書由来候補（全読）とコード由来候補（層化サンプリング + 識別子分解）を抽出し統合。根拠なき語は unresolved へ terms[] と（該当あれば）unresolved[] が確定済み
Phase 3
候補一覧の承認〈2段目〉		 HTML 化前に用語候補一覧を提示し AskUserQuestion で取捨（削除・言い換え）を受け page-data.json を組み立て 取捨結果を反映した page-data.json 保存済み
Phase 4
整合検証・用語辞書.html 生成		 validate-page-data.sh --target-repo で検証後 build-detail-page.sh --page glossary を実行。portal_output_dir 指定時は build-portal.sh を再実行 全項目 PASS。HTML 生成済み。指定時は build-portal.sh 再実行済み

エンジンスクリプトはスキルフォルダ相対 ../../../shared/scripts/detail-pages/ で参照する。

無人実行時の扱い

承認対象既定値
Phase 1（採録方針）分類軸=業務用語/技術用語/略語の3軸、採録源=実在確認済みの3系統すべて、除外パターン=glossary-extraction.md「除外既定」節の値のみ
Phase 3（候補一覧）Phase 2で確定した terms[] をそのまま採用（削除・言い換えなし）

両方とも自動承認した旨と適用した既定値を返却ブロックの hint に記録する。呼び出し元の契約（contract.md）への置換規則の反映は本スキルのスコープ外。

返却（呼び出し元との契約）

キー内容

§5制約と禁止事項

手作業でのプレースホルダ置換は禁止。 用語辞書.html は必ず build-detail-page.sh 経由の決定的処理で生成する。
語の意味の創作を禁止。 採録源（文書・コード識別子の使用文脈）に根拠の無い定義は書かない。根拠が無い語は unresolved へ退避する。
層化サンプリングは決定的コマンドに固定。 find/sort/head による決定的選定のみを使う。乱数・目視選定を禁止する。

抽出時の落とし穴

落とし穴対処
Phase 3 の言い換え指示で採録源に無い新規事実を追加してしまう既存記述の言い回し変更に限る。新規事実の追加指示は反映せず hint に記録する
フレームワーク API 名（useState 等）を業務用語として誤採録するglossary-extraction.md「除外既定」節の除外パターンを Phase 1 で確認する
jq 未インストール環境での検証・生成失敗validate/build 両スクリプトが jq に依存するため事前に導入する
