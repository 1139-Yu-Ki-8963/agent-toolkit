# チャット最終報告の書式

Phase 9 完了後、チャットでユーザーに直接返す最終メッセージの書式の正本。`references/report-format.md`（設計書リポジトリに保存する「最終報告.md」「修正指示書.md」の書式）とは別レイヤーを担う。前者は検証記録として残す文書、本書式は対話画面でユーザーが今すぐ判断できるようにするための要約。両方を作成する。

## 設計意図

- **結論を最初の1行に**: PASS / FAIL とその理由を専門用語なしで書く。「L3未実施」ではなく「画素単位の見た目比較だけ、道具がなくて実施できていない」のように、初見の読み手が意味を取れる言葉に置き換える
- **人間の目視確認を最優先に配置**: 機械比較の詳細より前に、オリジナルと復元後の URL を並べて示し、ユーザーが自分の目で見比べられるようにする
- **見比べるポイントは絞る**: 今回の実行で差異が出た箇所・直した箇所に限定して 2〜4 点。全項目を並べると要点が埋もれる
- **設計書の欠落こそが本来の成果**: 機械比較の結果とは別の表として独立させ、埋もれさせない
- **判断事項は分岐後の結果まで明示**: 「はい」「いいえ」それぞれを選んだ場合に何が起きるかまで書き、ユーザーが即答できる形にする

## テンプレート

````markdown
# 往復検証 結果報告：<画面ID>

## 結論
**<PASS / FAIL>** - <なぜその判定かを1文で。専門用語なし>

## 目でご確認ください（必須）
| 環境 | URL |
|---|---|
| オリジナル（正解） | http://localhost:<PORT1>/<path>?<query> |
| 設計書から復元（今回の成果） | http://localhost:<PORT3>/<path>?<query> |

**見比べるポイント**（この run で差異が出た/直した箇所に限定して2〜4点）：
- <例：下段テーブル「担当」列に氏名が出ているか>
- <例：2つの見出しが開いた状態で表示されるか>

## 機械比較の結果（要約）
| 何を比べたか | 結果 | 一言で |
|---|---|---|
| 画面が表示されるか | ✅/❌ | ... |
| 表示内容（文言・テーブル）の一致 | ✅/❌ | ... |
| 画面構造（アクセシビリティツリー）の一致 | ✅/❌ | ... |
| エラーの有無 | ✅/❌ | ... |
| 画素単位の見た目一致 | ✅/❌/未実施 | ... |

## 発見した設計書の欠落（このスキルの本来の成果）
| # | 設計書のどこ | 何が書かれていなかったか | 実装で何が起きたか |
|---|---|---|---|

## ご判断ください
1. **<判断事項>** - はい→<何が起きるか> / いいえ→<何が起きるか>
2. ...

## 詳細資料
- 最終報告：<path>
- 修正指示書：<path>
- スクリーンショット証跡：<path>
````

## 記入時の注意

- `<画面ID>` は設計書側の画面識別子をそのまま使う。連番を発明しない
- 「発見した設計書の欠落」表の `#` 列は行番号であり管理 ID ではない。行を増減しても振り直すだけでよい（意味キー規約の対象外。表自体が検出順の列挙であり、恒久的な相互参照キーではないため）
- NG が 0 件でも表の見出しは残し、本文に「欠落なし」と明記する
- `PORT1` / `PORT3` は起動引数として渡された実際のポート番号に置き換える。プレースホルダーのまま出力しない
- `L3 未実施` のように機械比較の内部呼称をそのまま書かず、「一言で」欄は必ず平易な日本語に言い換える
- 「見比べるポイント」に今回差異が出た/直した箇所が1件もない場合は、箇条書きを空にせず「差異なし。上記2つのURLを開き、画面全体を軽く見比べていただければ十分です」のように一言で明記する
- 「ご判断ください」に判断依頼事項が1件もない場合も、セクション自体は省略せず「特にご判断いただく事項はありません。このまま次に進んでよいか一言確認をお願いします」のように軽い確認文に置き換える
- 「差異なし」の定型文も含め、「見比べるポイント」は常に箇条書き（`- `）の体裁を保つ。見出し自体を書き換えたり通常の段落文に崩したりしない
- URL の `?<query>` は、対象画面にクエリパラメータが無い場合 `?` ごと省略する（`http://localhost:<PORT>/<path>` のみを記載する）
