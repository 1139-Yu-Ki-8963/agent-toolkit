# Phase 詳細（P2 突合式・P4 実装規律・P5 検証コマンド）

`SKILL.md` の Phase 2・4・5 で参照する詳細手順。SKILL.md 本体には要約のみを置き、具体的なチェック方法はここに集約する。

## スクリプト入力契約

### 起動引数 `audit_script_path <画面ディレクトリ>`（管理者から渡されるスクリプト）

- **設計書の特定**: `<画面ディレクトリ>/画面詳細設計書.md` を第一候補とする。存在しなければ frontmatter に `type: screen-detail-design` を持つ `.md` を探す（観点表にも `doc_id` があるため、`doc_id` の有無だけでは設計書を一意に特定できない）。どちらも見つからなければ「設計書を特定できません」エラーで `exit 1`
- **終了コード**: 違反（機能一覧章の機能一覧表〔既定 §2〕×観点表キー不一致・未記入プレースホルダ）が 1 件以上で `exit 1`。違反 0 件なら WARN（連番キー疑い・観点表未検出）があっても `exit 0`
- **キー突合ルール**: 機能一覧章の機能一覧表（既定 §2）の各機能キーは単体/結合いずれかの観点表に出現すればよい。観点表側にしか無いキーは違反（機能一覧の記載漏れ）
- **検査 a の抽出範囲**: 観点表側の機能キー抽出は、観点表ファイル内の「## 観点表」見出し配下（次の `## ` 見出しの手前まで）にあるテーブルのみを対象にする。テストサイズ対応表・本書に書かないもの・観点の導出元マップ等のガイドテーブルは対象外（これらの先頭列まで機能キーとして拾うと誤検出するため）。（手動観点表・テストサイズ対応表・乖離分類の定義などの隣接テーブルは対象外）
- **プレースホルダ検出（検査 b）の対象範囲**: frontmatter を含むファイル全体。ただし HTML コメント（`<!-- ... -->`）内は除外する
- **突合方式**: 機能一覧章の機能一覧表（既定 §2）機能キー集合と観点表機能キー集合の**両方向一致**（キー集合突合）で判定する。件数比較（行数の合計と項目数を数だけ比べる）ではない
- **`--list-contract-files <画面ディレクトリ>`**: 実装契約章のファイル分割表（既定 §15.1）に列挙された対象ファイルパスを 1 行 1 パスで stdout に出力する。役割キー行なし・表なし・有効行 0・全行プレースホルダのいずれかに該当する場合は「実装契約章の不備」を stderr に出力して `exit 1`。Phase 3 白紙化ゲートの対象ファイル抽出専用モード

### `scripts/check-freeze.sh <reverse-code worktree パス> <凍結コミットハッシュ>`

- **検査内容**: (1) worktree の `HEAD` が指定コミットハッシュと一致するか、(2) `git status --porcelain` が空（作業ツリークリーン）か、の 2 点
- **終了コード**: 両方満たせば `exit 0`（PASS）。いずれか不一致なら違反内容を stderr に出力して `exit 1`（FAIL）

## 章の役割キー → 節キーワード対応表（正本）

章の特定は設計書の章マップ表（役割キー列）で行う。起動引数 `audit_script_path` のスクリプトは章マップ表から役割キー→§番号を解決し、以下の節キーワードで二次解決する。本表が節キーワード対応の正本であり、SKILL.md・ng-classification.md・report-format.md・guide.html はここへのポインタとする。

| 役割キー | 節キーワード（既定 §） |
|---|---|
| 機能一覧 | 機能一覧表（既定 §2） |
| 画面構造 | スタイル適用パターン（既定 §3.6） |
| 状態管理 | メイン画面の状態変数（既定 §5.3）・サブコンポーネントの状態変数（既定 §5.4） |
| データフロー | 空状態（既定 §6.6） |
| API通信 | 各 API のリクエスト・レスポンス型（既定 §7.2） |
| イベント処理 | イベント処理挙動（既定 §8） |
| 領域別仕様 | 項目定義・イベント挙動（既定 §9） |
| 定数設定値 | 定数値（既定 §10） |
| エラーハンドリング | メッセージ定義・空状態エラー状態（既定 §11.2/§11） |
| 画面遷移 | 遷移方式・パラメータ（既定 §12） |
| 実装契約 | ファイル分割（既定 §15.1）・型定義（既定 §15.2）・依存（既定 §15.3）・スコープ分類（既定 §15.5） |

## Phase 2: 設計書内部整合性監査の突合式

起動引数 `audit_script_path` のスクリプトが機械チェックする 3 観点に加え、目視で以下を突合する。

### 突合 1: 機能一覧章の機能一覧表 × テスト観点表

- 機能一覧章の機能一覧表（既定 §2.1）の「キー」列（意味キー）の集合と、frontmatter の `unit_test_sheet` / `integration_test_sheet` が指す観点表に出現する機能キーの集合を、両方向一致（キー集合突合）で突合する
- 機能一覧表にあって観点表にないキー → 「観点未整備」として記録する
- 観点表にあって機能一覧表にないキー → 「機能一覧の記載漏れ」として記録する

### 突合 2: API通信章の API 型 × 実装契約章の型定義

- API通信章の各 API のリクエスト/レスポンス型（既定 §7.2）に登場するフィールド名と、実装契約章の型定義（既定 §15.2）に登場するフィールド名を突合する
- フィールド名・型の両方が完全一致することを確認する。片方向のみの一致は不一致として扱う

### 突合 3: 状態管理章の状態変数 × 領域別仕様章/実装契約章の経路

- 状態管理章のメイン画面の状態変数（既定 §5.3）・サブコンポーネントの状態変数（既定 §5.4）に列挙された変数名が、領域別仕様章の項目定義・イベント挙動（既定 §9）、または実装契約章のファイル分割（既定 §15.1）・スコープ分類（既定 §15.5）のいずれかで参照されていることを確認する
- どこからも参照されない状態変数は「デッド変数疑い、または領域別仕様章/実装契約章（既定 §9/§15）の記載漏れ」として記録する

### 突合 4: 意味キー連番検出

- `~/.claude/rules/always/naming/semantic-key/rule.md` 準拠。`grep -E '\b[A-Z]{1,4}-[0-9]+\b'` と `grep -E '^\| *ID *\|'` で連番 ID・「ID」列を検出する
- `UTF-8` / `SHA-256` 等の正当な表記は誤検知として除外する（WARN 扱い、exit 1 にはしない）

### 突合 5: 未記入プレースホルダ検出

- `<...>` 形式のプレースホルダが本文（HTML コメント外）に残っていないか検出する
- テンプレートの `<!-- ... -->` コメント内の説明文は対象外とする

### 突合6: §16要確認事項一覧の未解消チェック

- §16（要確認事項一覧、既定）の状態列（6列テンプレのみ）を確認し、状態≠解消済みの行数を記録する
- 既定WARN。AUDIT_STRICT_P16=1環境変数を設定してaudit_script_pathを実行した場合のみ違反（exit 1）に昇格する
- 旧5列テンプレ（状態列なし）はSTRICT指定時も違反に昇格しない

## Phase 4: 実装規律

### Step 1: READ（設計書・規約の読み込み）

Phase 1 でロード済みの設計資産を、以下の順序で参照する。

1. `プロジェクト共通/規約/` 配下 — コードを書く前にスタイル・命名・配置・設計パターンを把握する
2. `プロジェクト共通/共通設計書.md` — 全画面共通の挙動を把握する
3. `プロジェクト共通/DESIGN.md` — 共通デザイン値を把握する
4. `画面/詳細設計/画面詳細設計書.md` — 該当章のみ。章マップの役割キーで節を特定する
5. `画面/詳細設計/DESIGN.md` — 画面固有のスタイル数値

読んではいけないもの:
- オリジナルコード環境のソースコード（Phase 3〜7 の間はカンニング禁止）
- 他画面の設計書（対象画面の設計書のみ）

### Step 2: SPEC（テストケース作成 = TDD の Red）

参照優先順位:
1. `画面/テスト項目書/単体テスト仕様書.md`・`結合テスト仕様書.md` が存在すれば、これを第一ソースとする（具体的な入力値・期待結果が記載されている）
2. テスト仕様書が不在の場合は、`画面/詳細設計/単体テスト観点表.md`・`結合テスト観点表.md` の各観点キーから具体的なテストケースを導出する

テストコードの書き方:
- ファイル名は `プロジェクト共通/規約/命名規約.md` のテストファイル命名パターンに従う
- 配置先は `プロジェクト共通/規約/ディレクトリ構成規約.md` のディレクトリ構造に従う
- テストが全て Red（失敗）であることを確認してから Step 3 へ進む

### Step 3: WRITE（最小実装 = TDD の Green）

テストを通す最小実装を書く。

規約準拠チェックリスト:
- [ ] `コーディング規約.md` のインデント・フォーマット・import 順序に従っているか
- [ ] `命名規約.md` のファイル名・変数名・コンポーネント名パターンに従っているか
- [ ] `ディレクトリ構成規約.md` の配置ルールに従っているか
- [ ] `コンポーネント設計規約.md` の Props 設計・状態管理・エラーハンドリングパターンに従っているか

設計書準拠チェックリスト:
- [ ] import/export/Props 名が設計書の記載と文字列完全一致しているか
- [ ] 設計書のキー集合（型フィールド・状態変数・定数名等）と実装のキー集合が EXTRA/MISSING 両方向で一致しているか
- [ ] 絶対配置+座標で表示制御する要素が DOM 常時マウントになっているか（条件付きレンダリング禁止）

書き込み範囲の規律:
- Phase 4 で書き込み可能なのは Phase 3 の白紙化リスト（`--list-contract-files` の出力）内のファイルのみ
- リスト外のファイルへの新規書き込み・変更は禁止する。違反があった場合は Phase 6 の凍結コミット差分（白紙化リストとの過不足）で機械検出する

### Step 4: VERIFY（実装契約との過不足チェック）

実装契約章のファイル分割表（既定 §15.1）と実装ファイルの過不足を機械チェックする:
- 設計書にあるが実装にないファイル → MISSING として記録
- 実装にあるが設計書にないファイル → EXTRA として記録
- MISSING または EXTRA が 1 件でもあれば Step 3 へ戻る（内側ループ、上限 3 回）

### import/export/Props 名の完全一致

- 実装契約章のファイル分割表の export 名（既定 §15.1）、依存の import 名（既定 §15.3）、コンポーネント Props 名は設計書の記載と実装で **文字列完全一致** させる。省略形・別名は禁止
- 一致しない場合は Phase 8 の失敗クラス「export-import-型不一致」として記録する

### キー集合 EXTRA/MISSING 両方向一致

- 設計書に記載されたキー集合（型フィールド・状態変数・定数名等）と実装のキー集合を両方向で突合する
- **EXTRA**（実装にあるが設計書にない）と **MISSING**（設計書にあるが実装にない）の両方を検出する。片方向だけの確認は不十分

### 絶対配置+座標で表示制御する要素は常時マウント（条件付きレンダリング禁止）

- 画面構造章のスタイル適用パターン（既定 §3.6）や領域別仕様章の領域仕様（既定 §9）で、絶対配置（`position: absolute` 等）+ 座標により表示/非表示を制御する要素は、DOM 上は常時マウントし CSS で表示制御する
- `{condition && <Component />}` のような条件付きレンダリングで対応する要素は、DOM 構造がオリジナルと一致せず Phase 7 の L2（構造スナップショット）で実差分となる。設計書に明記されたパターンに厳密に従う

## Phase 5: 検証コマンド詳細

- **型チェック**: `reverse-code-<scope>` worktree 内で実行する。solution-style tsconfig（ルートの `tsconfig.json` が `"files": []` または `references` のみを持ち、実体の型定義を含まない構成）を検出した場合は、`references` 先の実体 config（例: `tsconfig.app.json`・`tsconfig.src.json` 等）を特定し、そちらを `-p` に指定する: `NODE_OPTIONS=--max-old-space-size=8192 tsc --noEmit -p <実体config>`。solution-style でない場合（ルートの `tsconfig.json` 自体が実体 config の場合）は `tsc --noEmit` をそのまま実行する。偽通過検知: 検査対象に白紙化リストのファイルが含まれることを `tsc` 出力から確認してから合否判定する。合格条件は「リポジトリ全体 0 エラー」ではなく「白紙化リスト内ファイルのエラー 0 件」とする（既存コードに大量のエラーがある場合、全体基準は運用不能なため）
- **単体/結合テスト**: 単体テスト観点表・結合テスト観点表に記載されたケースを実装したテストファイル（Step 2 で配置した上流提供の単体テスト＋自作の結合テスト）を全ファイル・全件実行する。部分実行は禁止。テストランナーが vitest の場合は必ず `vitest run --ui=false` で実行する（対象リポジトリの vitest.config が `ui:true` でも watch/UI モードに入らせない。config は変更禁止）。E2E 系スクリプト（`RT-` / `SM-` / `IT-` / `CMP-` 接頭辞）が納品されていれば実行し、未納品なら「未実施」と記録して PASS 判定の根拠に数えない。テスト失敗時に修正してよいのは実装（白紙化リスト内ファイル）のみで、テストコード側は変更しない
- **Playwright スモーク**: **reverse 側 9110 番台のみ**を対象とする。オリジナル側 9100 番台は Phase 7 の答え合わせでのみ起動する（Phase 5 時点でオリジナル側を見るとカンニングになる）

### 発散検知の記録

- 同一エラーシグネチャ（エラーメッセージの型・スタックの先頭フレーム等で識別）が 3 連続で再発した場合、発散確定とする
- 発散確定したエラーは「設計書に記載が足りず実装が収束しない箇所」の候補として、エラー内容そのものを Phase 8 へ引き継ぐ
