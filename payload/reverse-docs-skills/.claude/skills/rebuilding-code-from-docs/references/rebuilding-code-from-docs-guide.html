設計書精度検証のためのコード再構築
rebuilding-code-from-docs

設計書だけからコードを再生成し、元コードと機械突合して設計書の欠落を発見する往復検証スキル。

こんな人向け
リバースエンジニアリングで作成した画面詳細設計書の品質を、実際にコードが再現できるかどうかで検証したい人。
こんな場面で使う
リバース済み設計書の往復検証・設計書からのコード再生成・設計書の粒度検証や品質検証を行うとき。環境の整備・同期・基準タグ操作、設計書の直接修正や通常実装は本スキルの対象外。

§1概要・検証範囲

リバースエンジニアリングで作成した画面詳細設計書（assets/リバース検証/画面/詳細設計/ 準拠）は納品物であり、「設計書だけを読んで忠実にコーディングできる」詳細度が求められる。設計書は 16 種類の章の役割キー（各章に付けられた意味分類ラベル。例: 機能一覧・状態管理・API通信）で構成される。以下で頻出する「既定 §n」という表記は、各役割キーがテンプレート上で通常割り当てられる章番号を指す注記である（正の識別子は章番号ではなく役割キーであり、個別の設計書では番号がずれる場合がある）。本スキルはその品質を検証するため、設計書だけからコードを再生成し、元コードと機械突合して設計書の欠落を発見する往復検証を行う。

検証範囲: 本スキルは単体テスト・結合テストに加え、Playwright による動的検証まで行う。動的検証は、起動確認から画面操作結果の突合までを 5 段階に分けた検証レベル（L1〜L5。内訳は後述の Phase 7 の表を参照）で構成される。最終段階の L5 は複数の操作をまたぐ操作シーケンス比較であり、e2e（end-to-end。実際の利用者の操作の流れを再現して検証する手法）に相当する。ファイル単位の往復検証は前段工程（rebuilding-screen-unit-from-docs）が担い、本スキルは画面単位の結合・動的検証を担う。
項目前段工程（rebuilding-screen-unit-from-docs）rebuilding-code-from-docs（本スキル）
検証単位1 ファイル画面単位（結合・e2e を含む）
用途from-zero 品質ブートストラップリバース済み設計書の往復品質検証
対象外画面単位結合・e2e 検証（→本スキル）—

Phase 2: 設計書内部整合性監査の具体的な突合内容

起動引数 audit_script_path のスクリプトによる機械チェック（a・d・e）と、目視突合（b・c）を組み合わせる。突合式の全文は references/phase-details.md「Phase 2: 設計書内部整合性監査の突合式」を参照。

突合方式内容
a. 機能一覧表 × 観点表機械機能一覧章の機能一覧表（既定 §2.1）のキー集合と観点表の機能キー集合を両方向一致で突合
b. API 型 × 実装契約型目視API通信章のリクエスト/レスポンス型（既定 §7.2）と実装契約章の型定義（既定 §15.2）のフィールド名・型を完全一致で突合
c. 状態変数 × 参照経路目視状態管理章の状態変数（既定 §5.3/5.4）が領域別仕様章（既定 §9）または実装契約章（既定 §15.1/15.5）のいずれかから参照されているかを確認
d. 意味キー連番検出機械grep -E '\b[A-Z]{1,4}-[0-9]+\b' 等で連番 ID・「ID」列を検出（WARN）
e. 未記入プレースホルダ検出機械<...> 形式の未記入箇所を本文（HTML コメント外）から検出

Phase 7: 答え合わせ（mode=judge・Playwright L1〜L5）の具体的な判定内容

起動引数 compare_result（管理者が比較〔dry-run〕を実行して取得した結果ブロック）が持つ静的 3 分類・env_check・Playwright 動的検証（L1〜L5）を判定根拠とする。比較の自前実装は禁止。起動引数 reverse_worktree・freeze_commit も併せて受け取り、Phase 9 の凍結検証に用いる。L1〜L5 はそれぞれ次を確認する検証レベルである。

レベル確認内容
L1 起動確認アプリが起動し、render-ready（初期表示が完了し読み込み中表示が消えた状態）に到達しているか。失敗時は L2 以降を評価せず、後述の DESIGN-INCOMPLETE 等として扱う（単独の失敗クラスとしては計上しない）
L2 画面構造スナップショットPlaywright の構造スナップショット（ARIA。画面上の要素構成を表現した情報）が両環境で完全一致するか
L3 画素比較画面のスクリーンショットを画素単位で比較し、差分が config.yml の閾値を超えないか
L4 コンソールエラーブラウザのコンソールに出力されたエラー集合が両環境で一致するか
L5 操作シーケンス比較クリック・入力等の一連の操作（operations）を実行した後の画面内容（postContent の tables/texts）が両環境で一致するか。operations を持つ画面のみ評価対象

上記とは別軸で、静的比較（ファイル diff の「実差分」判定）も同時に受領する。

status が DESIGN-INCOMPLETE（両環境とも render-ready 未到達）の場合は scenarios への query/path_params/ready 追加を指示する。DYNAMIC-UNVERIFIED（MCP・Playwright とも不在）の場合は PASS 扱いにせず往復検証未完了として報告する。

Phase 9: 凍結検証と最終報告の定量欄

起動引数 reverse_worktree・freeze_commit を用いて scripts/check-freeze.sh <reverse_worktree> <freeze_commit> を実行し、現状（HEAD・作業ツリー）と突合する。PASS = HEAD 一致かつ作業ツリークリーン、FAIL = 不一致（結果全体を無効宣言）。最終報告には DESIGN.md トークン適用率（n/m）・対象ファイル型エラー数・L3 画素差分値（証跡パス付き）を必須記載する。書式全体は §8 報告フォーマット と references/report-format.md を参照。

目的の分離: コードを直すことは目的ではない。フィードバック先は (a) 当該画面設計書の該当章（役割キー）、(b) 欠落が汎用的ならテンプレート自体の記載粒度改善の 2 つ。

環境管理・比較エンジンは管理者が仲介する。本スキルは値を保持せず、起動引数 mode（implement | judge）と mode ごとの args から受け取る。

§2出力

画面単位の往復検証を行い、設計書の完全性を判定する。

往復検証の合否判定
設計書だけを読んで画面のコードを丸ごと再構築し、オリジナルと突合した結果。「設計書の記述だけでコードを再現できたか」を判定し、再現できなかった箇所を設計書の欠落として具体的に報告する。合格すれば「この設計書は画面を完全に記述している」という証明になる。

§4使い方

起動引数は mode（implement | judge）と mode ごとの args 全量。設計書・DESIGN.md・共通設計書・メッセージ定義書・テスト観点表は設計書 frontmatter を経由してロードする。

前提条件: 基準タグ reverse-baseline/<scope> が確立済みであること。未確立の場合、mode=implement は Phase 1 で BLOCKED を返して管理者へ差し戻す。
項目内容
mode=implement の起動引数screen_dir・scope・reverse_worktree・ports・baseline_tag_status・docs_root・template_root・audit_script_path・chapter_map_path・user-approved
mode=judge の起動引数screen_dir・compare_result・reverse_worktree・freeze_commit
環境値の取得元管理者が事前解決した args（env_block）。本スキルは自ら取得・導出しない
比較エンジン自前実装しない。mode=judge は起動引数 compare_result のみを判定根拠とする
1 起動の単位1 起動 = 1 往復（mode どちらか一方のみ）。設計書修正後の再往復は管理者による再起動として扱う

§6設計原則

  1. 正本一元化: 環境固有値の正は起動引数の env_block（scope/reverse_worktree/ports/baseline_tag_status/docs_root）。画面固有値の正は設計書 frontmatter。本スキル専用の config.yml は新設しない
  2. 比較エンジンの非自前実装: 判定は mode=judge が受け取る起動引数 compare_result のみ
  3. 1 起動 = 1 往復: 設計書修正後の再往復は管理者による本スキルの再起動として扱う
  4. 検証役と生成役の分離: mode=implement 実装時点の自己申告では判定しない。判定は mode=judge が受け取る compare_result の決定的出力のみ

§9重要な注意事項・予想を裏切る挙動

コード記述はフェーズ限定: Phase 4 以外でコードを書いてはならない。Phase 3〜Phase 7 の間、オリジナルコード環境への Read は禁止（カンニング防止）。

重要な注意事項

  • Phase 8 ではコード・設計書とも修正禁止。修正指示書の作成と test-item-patterns.md への台帳追記のみ許可
  • push は一切禁止。凍結コミットはローカルブランチに留める
  • Phase 管理: 次の Phase に入る前に、その Phase 内の全 Step を TaskCreate で登録する。各 Step 完了時に TaskUpdate で completed にする。Phase の全 Step が completed になるまで次の Phase に進まない。一括登録（全 Phase 分を最初に登録）は禁止し、1 Phase ずつ登録する
  • 基準タグ更新は本スキルでは実行しない。管理者が比較エンジンの本番実行（非 dry-run）で行う

予想を裏切る挙動

  • Phase 2 で内部矛盾を検出した場合、Phase 4 の実装は行わない。欠陥設計書からの実装検証は往復検証として無意味なため、直ちに Phase 8 の指示書作成に移る
  • Phase 5 の発散確定は失敗ではなく手がかり。発散したエラー自体を設計書欠落候補として Phase 8 に渡す
  • 検証成果物は <verification_dir>/screen-<画面ID>/<timestamp>/ に置く。worktree 内には置かない（静的比較を汚すため）
  • 生証跡（スクリーンショット・テストログ・返却ブロック全文）は <verification_dir>/screen-<画面ID>/<timestamp>/ 配下に同梱し、指示書からは同ディレクトリ内の相対パスで参照する
