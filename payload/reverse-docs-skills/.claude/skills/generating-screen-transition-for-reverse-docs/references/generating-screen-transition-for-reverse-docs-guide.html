画面遷移図生成スキル
generating-screen-transition-for-reverse-docs

画面一覧.html のマニフェストと対象リポジトリのルーティング定義を突合し、「画面遷移図.html」を機械検証付きで生成するスキルの確定仕様。

こんな人向け
ポータルの将来ページ受け口に画面遷移図カードを反映したい人 ／ コードの実測から解決できた遷移だけを裏取りしたい人

§1概要

リバースエンジニアリング設計のポータルには、将来ページの受け口（docs_root 直下に該当 HTML が実在するときだけカード化される仕組み）がある。本スキルはそのうち画面遷移図（T4）専任の生成係として、画面一覧.html に埋め込まれたマニフェストと対象リポジトリのルーティング定義を突合し、画面遷移図.html を機械検証付きで作成する。

本スキルは判定・評価をしない。画面設計の良否は問わず、コードの実測から解決できた遷移のみを転記する。宛先を解決できない遷移や route 情報を持たない画面（旧形式マニフェスト）は捏造せず、未解決として可視化する。

── 処理の全体像 ──────────────────────────────

  画面一覧.html（script#screen-manifest 埋め込み）
       │  Phase 1  実在確認 + Router 種別判別 + 検出戦略宣言（承認）
       ▼
  対象リポジトリのルーティング定義（Router 定義・navigate・Link・redirect）
       │  Phase 2  nodes 転記 + edges 抽出 + 宛先解決（未解決は分離）
       ▼
  page-data.json（pageKind: transition）
       │  Phase 3  validate-page-data.sh --target-repo + 孤児 edge 検査
       ▼
  検証済み page-data
       │  Phase 4  build-detail-page.sh --page transition
       ▼
  <docs_root>/画面遷移図.html（+ 任意で build-portal.sh 再実行）

テーブル一覧のような組み込み検出器はなく、Router 種別の判別からカスタム抽出パスで進める点はテーブル一覧生成スキルと同型である。

§2出力

スキルを実行すると、以下の 2 点が生まれる。

画面遷移図.html（<docs_root> 直下）
画面をノード、遷移をエッジとした図を client-side SVG で描画するページ。図の下には各遷移の発生元・宛先・契機・出所（sourceRef）・confidence を補足表として一覧表示する。route 情報を持たない旧形式画面や宛先未解決の遷移は「未解決」節にまとめて表示され、読んだ人はどこまでが確定情報かをその場で判別できる。
埋め込み page-data JSON
画面遷移図.html の中に、生成の元になった page-data.json がそのまま埋め込まれる。読んだ人は「いつ・どの画面一覧・どのコード実測から生成されたか」を HTML 単体で追跡でき、機械処理（後続スキルからの参照）にも使える。

画面一覧.html が不在の場合、本スキルは生成を行わず先行実行の案内のみで終わる。この場合、画面遷移図.html は生まれない。

§3用語

page-data.json
detail-pages 系 5 スキルが共有する入力 JSON。pageKind ごとに型別スロットを持つ。本スキルは pageKind: "transition" を固定で使う。スキーマの全量は page-data-schema.md を参照。
nodes / edges
transition 型のスロット。nodes[] は {unitKey, label} 形式の画面 1 件 = 1 要素、edges[] は {from, to, trigger, sourceRef, confidence} 形式の遷移 1 経路 = 1 要素。from/to は nodes[].unitKey を参照する。
孤児 edge
from または to が nodes[].unitKey のいずれとも一致しない edges[] 要素。Phase 3 で 0 件であることを機械検証する。
旧形式マニフェスト
route フィールドを持たない画面一覧マニフェスト（過去バージョンの生成物）。該当画面は nodes[] から除外し unresolved[] へ登録する。
整合検証
validate-page-data.sh --target-repo による page-data.json の機械検査。sourceRef が対象リポジトリ上に実在するかを検査する。

§4動作フロー

起動引数は target_repo_path・docs_root（必須）と portal_output_dir（任意）の 3 つ。種別は transition に固定されているため、種別を選ぶ引数はない。

Phase内容完了条件
Phase 1
前提確認・検出戦略の宣言		 画面一覧.html の実在確認（不在ならハード停止し generating-screen-list-for-reverse-docs の先行実行を案内）、埋め込みマニフェストの確認、Router 種別の判別、検出戦略の AskUserQuestion 承認 実在確認済み（または停止）。Router 種別と検出戦略がユーザー承認済み
Phase 2
抽出		 nodes[] の転記、Router 定義・navigate・Link・redirect の走査による遷移候補の抽出、宛先の unitKey 解決（未解決は分離） nodes[]/edges[] 確定・page-data.json 保存済み
Phase 3
整合検証（機械実行）		 validate-page-data.sh <page-data.json> --target-repo <target_repo_path> の実行、孤児 edge の検査。FAIL は修正して再実行（3 回失敗で Phase 2 へ差し戻し） 全項目 PASS（exit 0）かつ孤児 edge 0 件
Phase 4
画面遷移図.html 生成		 build-detail-page.sh <page-data.json> <docs_root> --page transition を実行。portal_output_dir 指定時は build-portal.sh を再実行しカードへ反映 HTML 生成済み。指定時は build-portal.sh 再実行済み

エンジンスクリプトはスキルフォルダ相対 ../../../shared/scripts/detail-pages/ で参照する。

返却（呼び出し元との契約）

キー内容
statusDONE | STOPPED（画面一覧.html 不在）| ERROR
artifacts生成した画面遷移図.html のパス（STOPPED/ERROR 時は空）
page_kindtransition（固定値）
portal_rebuiltbuild-portal.sh 再実行の有無
hint停止理由、旧形式画面・宛先未解決件数、または次工程への申し送り

§5制約と禁止事項

手作業でのプレースホルダ置換は禁止。 画面遷移図.html は必ず build-detail-page.sh 経由の決定的処理で生成する。
宛先未解決はハード停止せず未解決として可視化。 解決できない遷移・route 情報を持たない画面は unresolved[] へ隔離し、AskUserQuestion で手動確定しない。
判定・評価は対象外。 画面設計の良否・遷移の妥当性には踏み込まない。本スキルの仕事はコード実測からの転記のみ。

抽出時の落とし穴

落とし穴対処
route を持たない旧形式画面を nodes に含めてしまうnodes[] から除外し unresolved[] へ登録する
変数のみで静的解決できない遷移先に無理に confidence を付ける解決不能として unresolved[] へ回す（references/transition-detection.md 参照）
page-data の nodes[] に route を持ち込んでしまうroute は Phase 2 の解決処理でのみ使う一時情報とし、page-data には持ち込まない
jq 未インストール環境での検証・生成失敗validate/build 両スクリプトが jq に依存するため事前に導入する
