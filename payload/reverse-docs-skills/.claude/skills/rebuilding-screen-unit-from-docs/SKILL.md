# 設計書からの単体ファイル再構築スキル

リバース設計書（`assets/リバース検証/画面/詳細設計/` 準拠、章の役割キー 16 種で構成）の往復検証を、**「1 起動 = 1 ファイル・単体観点・検証専任」** で行うスキル。画面単位・結合観点・e2e の検証は行わず、後続の画面単位検証工程へ引き継ぐ（引き継ぎは orchestrating-reverse-docs-flow が案内する）。本スキルは著述済みの設計書だけを入力に対象ファイルを盲目再生成し原本と突合して記載漏れを検出する検証専任スキルであり、設計書の著述・スキャフォールディングは行わない（それらは `generating-reverse-detailed-design` の責務）。設計書に対象ファイルの契約が無い場合は fail-closed で著者スキルへ差し戻す。

コードを直すことは目的ではない。P5 の差分は「コードの誤り」ではなく「設計書の記載不足」として扱い、フィードバック先は当該設計書の該当章（役割キー）。この「コードを直さない」は**修正先の指定**（コードでなく設計書を直す）であり、対象ファイルを元の内容へ復元する処理を意味しない。P4 で生成したコードは検証のための使い捨て素材であり、スキル終了後もそのまま target_repo_path に残ってよい（原本への復元は行わない）。設計書に対象ファイルの記載自体が無い場合は、新規作成せず `status=差し戻し` として `generating-reverse-detailed-design` を先に実行するよう返す。

比較エンジンは自前実装する。重い環境同期・基準タグ運用（後続工程の責務）には委譲しない（本スキルは 1 ファイル規模の軽量往復であり、画面単位の環境同期は過剰）。

## 使用タイミング

- リバース済み設計書のファイル単位往復検証を行いたいとき（設計書の著述・品質ブートストラップは `generating-reverse-detailed-design` が担当）
- 起動引数は **画面ディレクトリ + 対象ファイルパス**（実装契約章のファイル分割表〔既定 §15.1〕の 1 エントリ）
- 前提: 対象ファイルが §15.1 のファイル分割表に列挙されていること。列挙が無い場合は `status=差し戻し` として P1 で終了し、`generating-reverse-detailed-design` の実行を hint に記録する
- 画面単位の結合・e2e 検証（Playwright スモーク・複数ファイル横断の突合）が必要なときは画面単位検証（後続工程）を使う（起動判断・引き継ぎは管理者 orchestrating-reverse-docs-flow が行う）。本スキルは単体（small）観点のみを扱う

## 設計原則

1. **正本一元化**: 環境固有値（ポート・worktree パス）には触れない。画面固有値の正は設計書 frontmatter。本スキル専用の設定ファイルは新設しない
2. **比較は自前の 8 計測で行う**。単体テスト仕様の検査 + `scripts/measure-file-diff.sh` の決定的出力（import diff・style diff・契約突合。全体 diff・実質 diff・スタイル正規化 diff は参考値）に加え、原本での P3 テスト実行結果（元コード全緑証明。詳細は P5 参照）を判定根拠とし、環境同期・基準タグ運用（後続工程の責務）には委譲しない
3. **1 起動 = 1 ファイルの往復**。他のファイルへは進まない。画面内の別ファイルを検証する場合は本スキルの再起動として扱う
4. **カンニング防止の二層**（詳細は次節）

## カンニング防止（二層）

生成物が「正解を見て書いた」ものにならないよう、2 つの独立した層で防止する。

### 層 1: ファイル内の答え

対象ファイルを P2 で `git rm` により白紙化する。削除直前の HEAD コミットを「原本参照コミット」として記録し、以降 P5 まで対象ファイルの内容は作業ツリー上に存在しない。原本の内容は P5 でのみ、原本参照コミットから `git show <commit>:<path>` で一時ファイルへ抽出する。

### 層 2: セッション文脈の汚染

P4（コード生成）は必ず `worker-sonnet` へ隔離委任する。prompt には「対象設計書の該当章 + 共通部品（DESIGN.md・共通設計書・メッセージ定義書の関連部分）+ 単体テスト観点表の該当行 + P3 で書いたテスト」のみを渡し、**元コードのパス・内容を一切含めない**。メインエージェント自身も元コードを P5（差分比較）でのみ Read する。ループの各周でサブエージェントを新規起動し、前周の生成文脈を持ち越さない。これにより、メインが P5 で原本を見て汚染された状態になっても、次周の生成コンテキストには漏れない（新規サブエージェントは前周の会話を一切継承しない）。

## 基本ワークフロー（P1〜P7）

### P1: 入力ロード + preflight

**作業対象リポジトリ・ブランチの確定**: 起動引数 target_repo_path（対象コードの実体を持つリポジトリの絶対パス）・target_branch（作業対象ブランチ）を受け取る。いずれかが渡されない場合は**起動不可**として管理者へ差し戻す（`status=差し戻し` は返さず、Skill 呼び出し自体が失敗した旨を管理者へ即時報告する。詳細は「重要な注意事項」の起動不可の扱いを参照）。以降 P2 の白紙化（`git rm`）・P4 の生成・P5 の原本取得（`git show`）は、すべてこの target_repo_path・target_branch を対象に行う。他のリポジトリ・ブランチには一切影響を与えない。target_repo_path が現在 target_branch をチェックアウト済みであることを確認し、異なる場合は `git -C <target_repo_path> checkout <target_branch>` を実行してから続行する（未コミット変更との衝突等でチェックアウトに失敗した場合は同様に起動不可とする）。

#### 画面ディレクトリの健全性ゲート

起動引数の画面ディレクトリが存在しない場合、著者スキル `generating-reverse-detailed-design` が未実行の状態とみなし、テンプレート展開は行わず `status=差し戻し` として本スキルを終了する（hint に「画面ディレクトリ・設計書が未著述のため `generating-reverse-detailed-design` を先に実行せよ」を記録する）。

画面ディレクトリが既に存在する場合は `bash <scaffold_script_path> --verify <docs_root> <画面ID>`（scaffold_script_path は管理者が解決して渡すスキャフォールディングスクリプトのパス。audit_script_path と同型。実体: `shared/scripts/scaffold-screen.sh` の1本が正本）を実行し、既存構造の健全性（詳細設計/テスト項目書の必須ファイルの有無、検証記録/ 配下の `<timestamp>` 等未展開プレースホルダの残留有無）を確認する。exit 0 なら従来の P1 ロジック（下記）へ直接進む。exit 1 の場合は起動引数 template_root 起点のテンプレート原本（`<template_root>/画面/詳細設計/` 等、欠落ファイルの種類に応じたディレクトリ）から欠落ファイルのみをコピーして復元し `--verify` を再実行する。再実行しても exit 0 にならない場合は `status=差し戻し` として `generating-reverse-detailed-design` へ差し戻す（fail-closed）。壊れた構造のまま P1 の残りへ進んではならない。

**進行前ガード（未記入テンプレート検知）**: 対象設計書が scaffold 直後の未記入テンプレートのまま（audit-consistency.sh 通常モードの検査 b が未記入プレースホルダ `<...>` を多数報告する状態）であれば、P2 の白紙化へ進んではならない。この場合は「設計書が未記入のため往復検証を開始できない。`generating-reverse-detailed-design` で設計書を記入してから対象ファイルを指定して再起動せよ」を返却ブロックの hint に記録し、`status=差し戻し` として本スキルを終了する（fail-closed）。記入済みの画面ディレクトリ（検査 b の未記入プレースホルダが実質ゼロ）では本ガードは発火しない。

起動引数（画面ディレクトリ + 対象ファイルパス）から設計書・`DESIGN.md`・共通設計書・メッセージ定義書・単体テスト観点表を frontmatter 経由でロードする。設計書は必須。他は frontmatter にキーがあれば必須（キーがあるのに実体が無ければ preflight エラー）、キー自体が無ければ「不在」と記録して続行する。起動引数 audit_script_path を Bash 実行する（`bash <audit_script_path> --list-contract-files <画面ディレクトリ>`）で実装契約章のファイル分割表（既定 §15.1）の対象ファイル一覧を取得し、起動引数の対象ファイルパスがその一覧に含まれるか確認する。含まれない場合は、設計書に対象ファイルの契約が無い状態として `status=差し戻し` を返し、hint に「`generating-reverse-detailed-design` で §15.1 表・関連章へ対象ファイルの契約を追記してから再起動せよ」を記録して本スキルを終了する（fail-closed）。

元コード Read 可否: **可（存在確認のみ・字面読み禁止）**。target_repo_path 上で `git ls-files <path>` 等でパスの実在を確認するに留め、内容を Read して転記してはならない。

完了条件: 画面ディレクトリの健全性ゲート通過（`<scaffold_script_path> --verify` exit 0）・対象ファイルが §15.1 リストに含まれることを確認済み（含まれなければ P1 時点で `status=差し戻し`）・設計書一式がロード済み

### P2: 白紙化ゲート

白紙化の前に起動引数 audit_script_path を通常モードで Bash 実行する（`bash <audit_script_path> <画面ディレクトリ>`）し、対象ファイルに関連する章の内部整合性を確認する。監査で矛盾が出た場合は `status=差し戻し` として P7 へ離脱する（設計書側の記載補いは `generating-reverse-detailed-design` の責務であり、本スキルは修正しない）。違反（機能一覧×観点表のキー不一致〔両方向一致が要件: 機能一覧にあって観点表に無い＝観点未整備、観点表にあって機能一覧に無い＝機能一覧の記載漏れ、いずれも違反〕・未記入プレースホルダ等）があれば白紙化・実装を経ず、直接 P7（NG 分類）へ直行する（欠陥のある設計書からの生成は往復検証として無意味なため）。この離脱は P7 で「設計書内部矛盾」クラス・`status=差し戻し`（P2内部矛盾）として扱う。

内部整合性に問題が無ければ、対象ファイル **1 件のみ** を白紙化する。白紙化の承認は起動引数 user-approved で受け取る（管理者が事前にユーザー承認を取得済み）。user-approved が無ければ**起動不可**として管理者へ差し戻す（fail-closed。`status=差し戻し` は返さない。詳細は「重要な注意事項」を参照）。

**全消しフォールバックは禁止**（対象ファイルの特定に失敗しても、無関係なファイルや画面ディレクトリ全体を削除してはならない、fail-closed）。user-approved 確認後、削除直前の HEAD コミットハッシュを「原本参照コミット」として記録し、target_repo_path 上で対象ファイルを `git rm` で削除してコミットする（コミットメッセージ: `【リファクタ】<対象ファイル> を白紙化（単体往復検証）`）。以降 P5 まで **原本参照コミットの内容の Read を禁止**する（カンニング防止層 1）。

元コード Read 可否: **可（対象特定のため。字面読み禁止は維持）**

完了条件: 内部整合性の監査結果が記録済み（矛盾があれば `status=差し戻し` として P7 直行）・ユーザー承認取得済み・原本参照コミットハッシュ記録済み・白紙化コミット完了

### P3: 単体テスト仕様作成

単体テスト観点表から、対象ファイルが担う観点行を全て拾い、テストケースを先に書き出す（TDD の Red、テストが先）。対象ファイルに対応する観点が 1 件も無い場合は、観点表に契約が無い状態として `status=差し戻し` を返し、hint に「`generating-reverse-detailed-design` で観点表へ対象ファイルの観点を追記してから再起動せよ」を記録して本スキルを終了する（fail-closed）。

原本参照禁止のため P3 テストの妥当性が P5 まで判明しない。テストは「設計書に記載された宣言的契約（型・API 契約・ロジック構造・定数値〔静的に決まるもの〕）」に限定して書き、原本の挙動を推測して書かない。P5 の差分に現れた宣言的契約は `p5-diff-facts.md` への記録を経て、観点表・設計書への追補材料として P6 で扱う。

**実測が要る断定は本スキルでは確定しない**。初期表示値・DOM 配置順序・要素位置・レイアウトなど、実行時の観測（Playwright 実測等）でしか正しく決められない事実は、目視転記も推測もせず、設計書・観点表に `[画面単位検証で実測]` プレースホルダとして残す。本スキルが確定させるのは、静的に決定可能な宣言的契約（型・import・export・Props 名・API 契約形・ロジック構造・定数値）に限定する。理由: 本スキルは Playwright を持たない軽量往復であり、実測できない構造断定を字面読み（カンニング違反）や推測で埋めると往復検証の土台が崩れるため。実測が要る断定は Playwright を持つ画面単位検証の担当とする。

元コード Read 可否: **禁止**。白紙化済みで作業ツリー上に実体が無いことに加え、`git show` 等の間接参照も本フェーズでは禁止する。

本フェーズで書いたテストコードは、P6 のループで再現一致するまで随時更新され、再現一致後に P7 で最終版として設計書リポジトリへ正本保存する（納品物化。詳細は P7 参照）。target_repo_path への配置は P4〜P5 実行時の一時コピーに限り、正本は設計書リポジトリ側に置く。

完了条件: 対象ファイルの観点に対応するテストケース一式が書き出し済み（RED 状態）

### P4: サブエージェント生成

**唯一コードを書けるフェーズ**。`worker-sonnet` へ隔離委任する（1 エージェント = 1 ファイル）。prompt 骨格・禁止事項の詳細は `references/phase-details.md` を参照。

worker-sonnet の起動が環境上ブロックされる場合（PreToolUse hook 等）、P1 へは差し戻さず、P4 時点で起動不可として管理者へ返却する（fail-closed）。P1 での模擬起動確認は行わない（実際に1エージェント起動してみるまで判定できず、模擬確認はコスト対効果が見合わないため）。

元コード Read 可否: **禁止**（メイン・サブエージェントとも）

完了条件: 対象ファイルが生成され、P3 のテストがローカルで実行可能な状態になっている

### P5: 差分比較（8 計測）

ここで初めて原本参照コミットの内容の Read を許可する。target_repo_path 上で `git show <原本参照コミット>:<対象ファイルパス>` でスクラッチ領域へ一時ファイルとして抽出する（worktree・設計書リポジトリを汚染しないよう `$CLAUDE_JOB_DIR/tmp/` 配下に置く）。

1. **単体テスト仕様の検査**: P3 で書いたテストを P4 の生成物に対して実行する（禁止パターン・import・コンポーネント/API 利用の確認）。`vitest run --ui=false` で実行する。全項目合格が必須
2. **契約突合を含む計測**: `scripts/measure-file-diff.sh <generated-file> <original-tmp-file>` を実行し、`import_diff_lines / style_diff_lines / export_diff_lines / const_diff_lines / handler_diff_lines / type_diff_lines / state_diff_lines / apicall_diff_lines / contract_match=YES|NO / total_diff_lines（参考）/ substantive_diff_lines（参考）/ style_normalized_diff_lines（参考。第8計測。スタイル差〔クォート種別・末尾カンマ・空白等〕を正規化してから取る diff）/ verdict=PASS|FAIL` を取得する。契約突合（`contract_match`）は、生成物と原本から export名集合・定数値集合・ハンドラ名集合・型名集合・状態変数名集合・API呼出先集合の 6 カテゴリを機械抽出して集合突合した結果である
3. **元コード全緑証明（原本での P3 テスト実行）**: 原本参照コミットから原本を一時抽出できた時点で、P3 テストを生成物だけでなく **原本にも** 実行する。手順は以下の順を厳守する。
   1. 原本の内容を target_repo_path の対象ファイルパスへ一時配置する
   2. `vitest run --ui=false` を実行する
   3. 実行結果（成功・失敗・中断のいずれであっても）を記録した直後に、対象ファイルパスの内容を P4 の生成物へ **必ず復元する**（vitest 実行が失敗・中断した場合でも finally 相当として復元処理を実行する。復元漏れは重大違反）
   4. 実行結果（PASS/FAIL の件数）のみを判定に利用し、原本の字面読み禁止は本フェーズでも維持する
4. **差分事実の記録**: 原本を Read できる本フェーズの機会に、差分に現れた宣言的契約（import/export・型/定数/状態変数/イベント/JSX構造/スタイル値/API呼出）を `<verification_dir>/screen-<画面ID>/単体-<対象ファイルbasename>/<timestamp>/p5-diff-facts.md` へ記録する。P6 はこの記録のみを根拠に設計書を修正する（原本の再Read・字面転記は P6 で禁止）

差分は「コードの誤り」ではなく「設計書の記載不足」として解釈する。この段階でのコードの手修正は禁止する。

`measure-file-diff.sh` の verdict は生シグナルとして扱う。差分行のうち `[画面単位検証で実測]` プレースホルダに該当する実測事実（初期表示値・DOM 配置順・要素位置・レイアウト起因の style/JSX 差分）は「画面単位検証への委譲」として分類し、**再現一致判定の対象から除外**する。判定（PASS/FAIL）は import diff・style diff・契約突合（`contract_match`）だけで行い、参考値（全体 diff・実質 diff）は判定に用いない。除外した実測差分は P7 の画面単位検証への申し送りに記録する。

**元コード全緑証明の判定への組み込み**:
- 生成物 green + 原本 green → P3 のテストは正しい契約を表現している。再現一致判定へ進む
- 生成物 green + 原本 red → P3 のテストが誤った推測を含む。P6 で観点表・設計書側を修正し P2 へ再投入する（原本 red を黙って無視して再現一致扱いにする silent accept を禁止する）
- 原本 red の内容が「原本自体のバグ」の疑いである場合は、P6 での機械的な再投入を行わずユーザーへ判断を仰ぐ（実例: ある画面の既存テストは元コードで一部失敗していた）

元コード Read 可否: **可**（本フェーズのみ）

完了条件: 単体テスト仕様の検査結果 + 8 計測の結果（省略なし）+ `p5-diff-facts.md` の記録 + 差分の画面単位検証への委譲/静的契約の分類結果が記録済み

### P6: 設計書修正

P5 の差分を 4 分類に振り分け、修正先の章を決定する。

| 差分種別 | 修正先 |
|---|---|
| import | 実装契約章の依存（既定 §15.3） |
| style | `DESIGN.md`（画面構造章のスタイル適用パターン経由、既定 §3.6） |
| logic | データフロー（既定 §6）・API通信（既定 §7） |
| JSX | 画面構造（既定 §3）・領域別仕様（既定 §9） |

**実測断定に該当する差分（初期表示値・DOM 順・位置・レイアウト）は確定修正せず `[画面単位検証で実測]` を維持する**。これらは設計書へ確定値を書き込まず、再現一致判定の分母からも除外し、画面単位検証での実測での確定に委ねる。P6 で確定修正するのは静的な宣言的契約由来の差分のみ。

元コード Read 可否: **禁止**。設計書修正は `p5-diff-facts.md` のみを根拠に行い、原本コードを再度 Read・字面転記しない。P6 で補修できるのは増分（書き漏らし）のみで、丸ごとの著述不足は P1/P2/P3 の `status=差し戻し`（契約不在）対象であり P6 では扱わない。

完了条件: 差分 4 分類・修正内容（または P2 再投入の判断）が記録済み

### P7: NG 分類 + 報告

`status`（再現一致 / 再現不一致）を判定する。再現一致 = 8 計測の主判定（import diff 0・style diff 0・契約突合一致・かつ実質 diff 20行以下またはスタイル正規化 diff 20行以下のいずれか）かつ単体テスト仕様の検査 全項目合格。再現不一致 = 上限 5 回到達、または発散検知（同一差分集合 2 連続再発）。再現一致時は `references/ng-classification.md` の該当有無を確認し「NG なし」または軽微な修正指示書を出力する。再現不一致時は検出シグナルを `references/ng-classification.md` の表で失敗クラスに分類し、修正指示書を作成する。再現不一致の内容は「画面単位検証時の要注意ファイル」として検証記録に記録し、画面単位検証の実行時に参照できるようにする（引き継ぎは orchestrating-reverse-docs-flow が案内する）。

P2 の内部整合違反による早期離脱の場合は、白紙化・生成を経ていないため「設計書内部矛盾」クラスの修正指示書を作成し、`status=差し戻し`（再現一致/再現不一致の判定対象外・実装検証未実施）として明示する。

**テストコードの納品物化**: 再現一致確定時、P3〜P6 で再現一致した最終テストコードを設計書リポジトリの `<画面ディレクトリ>/テスト項目書/テストコード/単体/<対象ファイル basename>/` に正本として保存する（検証記録とは別枠。設計書・観点表と同格の納品物として扱う）。target_repo_path 内へのテスト配置は P4〜P5 実行時の一時コピーに限定し、本フェーズ終了時に撤去する。target_repo_path に同名の既存テストが存在する場合は上書きを禁止し、既存テストの扱い（共存させるか置換するか）を返却ブロックの hint に記録して管理者へ申し送る。テンプレートには `テストコード/` ディレクトリを含めない（本フェーズで対象ファイルごとに動的作成する）。

**観点網羅ゲート**: `scripts/check-viewpoint-coverage.sh <単体テスト観点表> <テストコード>` を実行し、exit 0（観点網羅ゲート通過）であることを完了条件に含める。観点表のキーが 1 件でもテストコードに未言及であれば再現一致扱いにしない。ただし「言及＝実装」を意味するものではない点に注意し、P5 の単体テスト仕様の検査（テスト全件 PASS）との組み合わせで実装の妥当性を担保する。

**観点表→仕様書の機械突合**: テスト仕様書の記入完了条件に「観点表の全観点キーがテスト仕様書のケースまたは保存済みテストコードに対応していること」の機械突合を追加する。`check-viewpoint-coverage.sh` の既存機構（観点キー→テストコード突合）を拡張し、テスト仕様書（単体テスト仕様書.md）も突合対象に含める。対応しない観点が残る場合は観点表の自動化宣言との矛盾として完了扱いにしない。

**返却ブロックの拡張**: 返却拡張フィールドに `saved_test_paths`（本フェーズで保存した最終テストコードのパス一覧）と `status`（差し戻し / 再現一致 / 再現不一致）を追加する。フィールド名は必ず `saved_test_paths` の表記のまま用いる（管理者 orchestrating-reverse-docs-flow の契約正本 `contract.md` 側と一致させるため）。なお E2E（RT-/SM-/IT-/CMP- 系）テストは、作成と元コードでのベースライン実測を rebuilding-code-from-docs（mode=implement）が担い、実行を管理者が dynamic 検査として担う（確定責務。compare_request.scenarios_ready / syncing の scenarios 引数に整合する）。本スキルは Playwright 非保持のため E2E には関与せず、hint への申し送りも不要である。

元コード Read 可否: **禁止**（報告作成は P5 の結果のみを参照する）

## 8 計測の合格判定表

| 計測 | 合格条件 |
|---|---|
| 単体テスト仕様の検査（禁止パターン・import・コンポーネント/API 利用） | 全項目合格 |
| import の diff | 0 行 |
| スタイル定数の diff | 0 行 |
| 契約突合（export名/定数値/ハンドラ名/型名/状態変数名/API呼出先の集合突合） | `contract_match=YES` |
| 元コード全緑証明（原本での P3 テスト実行） | 原本 green、または原本 red がユーザー承認済みの原本バグ |
| ファイル全体の diff | 行数を出力（参考値・判定には使わない） |
| 実質 diff（コメント・空行除外） | 20 行以下、またはスタイル正規化 diff が 20 行以下なら免除 |
| スタイル正規化 diff（クォート種別・末尾カンマ・空白等のスタイル差を正規化してから取る diff。第8計測） | 20 行以下（実質 diff が 20 行以下ならこちらは判定に用いない） |

**import diff だけでの合格判定は重大違反**とする。8 計測はすべて出力し、いずれか 1 つでも省略してはならない。

**契約突合（`contract_match`）は export名・定数値・ハンドラ名・型名・状態変数名・API呼出先という宣言レベルの欠落しか検出できず、関数本体のロジック（条件分岐・算術式・JSX子要素の並び等）は対象外である**。このロジック差分を見逃さないための安全網として、実質 diff 20行以下のしきい値を維持する。ただし実質 diff は行文字列比較のためスタイルの揺れ（クォート種別・末尾カンマ・空白等）だけで20行を超えることがあり、この種の非本質差分を実質 diff から分離するためスタイル正規化 diff（第8計測）を判定式に加え、二段化した（`import_diff_lines==0 ∧ style_diff_lines==0 ∧ contract_match==YES ∧ (substantive_diff_lines<=20 ∨ style_normalized_diff_lines<=20)` が再現一致の判定式）。

**実質 diff の行数は設計書の記載完全性と厳密には比例しない場合がある**。`measure-file-diff.sh` は行文字列の比較であり、設計書がそもそも規定すべきでない実装スタイル（コンポーネント宣言方式・内部変数名・条件分岐の書き方等）の揺れは、独立した2回の生成で一致する保証が無い。P6 で設計書側の記載を正しく補っても、この種の実装スタイル差が残る限り実質 diff が 20 行に収まらないことがある。トップレベル宣言ブロックの順序入替も同様に、内容が同一でも実質 diff に全行乗る（measure-file-diff.sh はブロック単位の正規化ソートを行わない。これは bash での実装が過剰・脆弱なため意図的に見送った仕様）。順序差だけで実質 diff が20行を超えた場合は「設計書の記載不足」と誤認せず、P7 で「実装スタイル差」クラスとして扱う。P7 では、こうした差分全般を `references/ng-classification.md` の「実装スタイル差」クラス（章に帰着させない）として扱う。

## 設計判断

### measure-file-diff.sh

**必要性**: P5 の 8 計測のうち import diff・style diff・全体 diff・実質 diff の 4 項目は、複数のヒューリスティック抽出（import 行・スタイル定数行・コメント/空行除外）を組み合わせた決定的検査であり、毎回手書きの diff コマンド列で実行すると抽出条件がぶれ再現性が失われる。「import diff だけでの合格判定は重大違反」とする規約上、これらを同一スクリプトで一括算出し、省略を構造的に防ぐ必要がある。

**代替案を採用しなかった理由**:
- Bash 直叩き: 複数種の diff 抽出条件（import 行パターン・スタイル定数パターン・コメント/空行除外パターン）を都度手書きすると、一部の計測を省略しがちになり「import diff だけで合格判定」という重大違反を誘発する
- 既存 Makefile ターゲット拡張: 本スキルの配置先リポジトリには Makefile がない
- package.json scripts 追加: グローバルスキル用途でありプロジェクトの package.json に属さない

**保守責任者**: 人手（ユーザー）。スタイル定数抽出パターン変更時に更新する。

**廃棄条件**: 本スキル廃止時、または画面単位検証側の比較エンジンにファイル単位検証相当の軽量 diff モードが統合された時

**追記（前後空白除去）**: 架空プロジェクトでの E2E 検証で、設計書の記載を正しく補った後の再生成でも、条件分岐の書き方（早期return vs 三項演算子）の違いでインデント幅がずれ、内容が同一の行が style diff / 実質 diff で誤って差分と判定される事象を確認した。抽出した行の前後空白を除去してからソート・diff する処理を追加し、インデント差のみに起因する誤検出を解消した。本追記は表示上同一のDOM構造に対するインデント差のみを対象とし、分岐間で外殻要素そのものが異なる/欠落するケースは対象外である。後者は「表示制御方式差」として扱う（references/ng-classification.md 参照）。

**追記（契約突合の追加）**: import diff・style diff の 0 判定だけでは、export名・定数値・ハンドラ名・型名・状態変数名・API呼出先といった宣言的契約の欠落を検出できない（実質 diff 20行以下という緩いしきい値の内側で、契約の抜け漏れが verdict=PASS をすり抜ける事例があった）。生成物と原本から 6 カテゴリの識別子集合を抽出し完全一致を要求する `contract_match` を追加し、verdict の判定式を `import_diff_lines==0 ∧ style_diff_lines==0 ∧ contract_match==YES ∧ substantive_diff_lines<=20` に変更した。**contract_match は宣言レベルの欠落しか検出できず関数本体のロジック差分は対象外のため、実質 diff 20行以下のしきい値はロジック差分の安全網として維持する**（全体 diff のみ参考値のまま）。当初 contract_match 追加時に実質 diff のしきい値を撤去したところ、識別子は一致するがロジックが異なる生成物が verdict=PASS をすり抜ける事例をコードレビューで指摘され、しきい値を復元した。

**追記（スタイル正規化 diff の追加・判定式の二段化）**: 実質 diff は行文字列の比較であり、クォート種別・末尾カンマ・空白等のスタイルのみの揺れでも 20 行を超えることがある。設計書が規定すべきでない実装スタイル差が再現一致判定を妨げる事例を踏まえ、スタイル差を正規化してから取る diff（`style_normalized_diff_lines`。第8計測・参考値）を追加し、verdict の判定式を `import_diff_lines==0 ∧ style_diff_lines==0 ∧ contract_match==YES ∧ (substantive_diff_lines<=20 ∨ style_normalized_diff_lines<=20)` の二段化した形へ変更した。import diff・style diff・contract_match の 3 条件は従来どおり AND 必須のまま維持し、しきい値判定（実質 diff 20行以下）だけをスタイル正規化 diff とのいずれか一方で満たせばよい OR 条件に緩和する。

### measure-file-diff.test.sh

**必要性**: `measure-file-diff.sh` は import/style/契約突合の抽出パターン・verdict 判定式という複数の分岐を持ち、8 計測すべての key=value 出力を都度手動確認すると回帰の見落としが生じる。契約突合の 6 カテゴリ抽出・verdict 判定式の変更（`contract_match` 追加）を機械的に固定するための単体テストが必要。

**代替案を採用しなかった理由**:
- Bash 直叩きでの都度検証: 6 カテゴリ抽出 × verdict 判定の組合せを毎回手書きすると再現性が無く回帰テストとして機能しない
- 既存 Makefile ターゲット拡張: 本スキルの配置先リポジトリには Makefile がない
- package.json scripts 追加: グローバルスキル用途でありプロジェクトの package.json に属さない

**保守責任者**: 人手（ユーザー）。`measure-file-diff.sh` の抽出パターン・verdict 判定式変更時に同時更新する。

**廃棄条件**: `measure-file-diff.sh` 廃止時。

### check-viewpoint-coverage.sh

**必要性**: 単体テスト観点表の各観点が最終的にテストコードへ落ちたかどうかの網羅を機械ゲート化する必要がある。P5 の 8 計測は「書かれたテストが全件 PASS するか」を検査するが、「観点表の全観点がテストとして書かれているか」は別軸の検査であり、これが欠けると観点の実装漏れが再現一致判定をすり抜ける。観点表のキー集合とテストコードの言及を突合し、未言及が 1 件でもあれば exit 1 とすることで、P7 の再現一致判定に組み込む。

**代替案を採用しなかった理由**:
- AST・カバレッジ計測: 対象プロジェクトの言語・テストランナーに依存し、観点表の観点粒度（意味キー単位）とカバレッジのメトリクス粒度（行・分岐単位）が一致しない
- 観点キー→テスト名の完全一致要求: 観点キー（意味キー規約準拠の要約語）をそのままテスト名に強制すると、テストコードの命名が観点表の語彙に縛られ硬直化する

**保守責任者**: 人手（ユーザー）。観点表のキー抽出パターン・テストコードの言及判定パターン変更時に更新する。

**廃棄条件**: 観点対応づけがカバレッジ計測由来で自動化された時、または観点表の「コメント参照」規約が廃止された時
