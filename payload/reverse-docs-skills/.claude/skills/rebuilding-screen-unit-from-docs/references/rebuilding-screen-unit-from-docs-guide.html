設計書精度検証のための単体ファイル再構築
rebuilding-screen-unit-from-docs

設計書だけから1ファイルを再生成し、元ファイルと軽量突合して設計書の欠落を発見するスキル（単体観点・from-zero）。

こんな人向け
from-zero でリバース設計書の記載粒度をファイル単位でブートストラップしたい人。既存の画面詳細設計書.md が「1 ファイル分だけ」正しく書けているかを軽量に検証したい人。
こんな場面で使う
ファイル単位の往復検証・from-zero での設計書品質ブートストラップを行うとき。画面単位の結合・e2e 検証（Playwright スモーク・複数ファイル横断の突合）は後続の画面単位検証、環境同期・基準タグ運用は後続工程の対象であり、本スキルの対象外。
対応 OSWindows WSL2 環境
検証状況実機検証済み
依存管理者: orchestrating-reverse-docs-flow（args 配布・合格後の画面単位検証への引き継ぎ判断）
関連資料references/phase-details.md, references/ng-classification.md, orchestrating-reverse-docs-flow の契約（references/contract.md）

§1概要

リバース設計書（assets/リバース検証/画面/詳細設計/ 準拠）の往復検証を、「1 起動 = 1 ファイル・単体観点・from-zero」で行うスキル。設計書は 16 種類の章の役割キー（各章に付けられた意味分類ラベル）で構成され、本文で使う「既定 §n」という表記は、各役割キーがテンプレート上で通常割り当てられる章番号を指す注記である（正の識別子は章番号ではなく役割キーであり、個別の設計書では番号がずれる場合がある）。画面単位・結合観点・e2e の検証は行わず、後続の画面単位検証工程へ引き継ぐ（引き継ぎは orchestrating-reverse-docs-flow が案内する）。本スキルで設計書の記載粒度をファイル単位でブートストラップする。

目的の分離: コードを直すことは目的ではない。P5 の差分は「コードの誤り」ではなく「設計書の記載不足」として扱い、フィードバック先は当該設計書の該当章（役割キー）。設計書に対象ファイルの記載自体が無い場合は、ひな形から新規作成してよい（from-zero）。

比較エンジンは自前実装する。重い環境同期・基準タグ運用（後続工程の責務）には委譲しない（本スキルは 1 ファイル規模の軽量往復であり、画面単位の環境同期は過剰）。

設計原則

  1. 正本一元化: 環境固有値（ポート・worktree パス）には触れない。画面固有値の正は設計書 frontmatter。本スキル専用の config.yml は新設しない
  2. 比較は自前の 6 計測で行う。scripts/measure-file-diff.sh の決定的出力のみを判定根拠とし、環境同期・基準タグ運用（後続工程の責務）には委譲しない
  3. 1 起動 = 1 ファイルの往復。他のファイルへは進まない。画面内の別ファイルを検証する場合は本スキルの再起動として扱う
  4. カンニング防止の二層（詳細は §4）

§2位置づけ（ファイル単位検証 / 画面単位検証 パイプライン）

本スキルは、リバースドキュメント生成からコード再構築に至る一連のパイプラインのうち、設計書のファイル単位ブートストラップを担う工程に位置する。

本スキルが直接読み書きするのは起動引数で受け取る資産（章役割キー対応表・監査スクリプト・テンプレート）のみ。環境同期・比較エンジンは後続工程の担当であり、本スキルからは呼び出さない。

§3使い方

前提条件: 対象ファイルが実装契約章のファイル分割表（既定 §15.1）に列挙されていること。列挙が無い場合は from-zero として扱い、P1 で新規追加を宣言してから進める。
項目内容
起動引数画面ディレクトリ + 対象ファイルパス（実装契約章のファイル分割表〔既定 §15.1〕の 1 エントリ）
比較エンジン自前実装（scripts/measure-file-diff.sh）。後続工程の環境同期には委譲しない
1 起動の単位1 起動 = 1 ファイルの往復。画面内の別ファイルは本スキルの再起動として扱う
合格後の遷移合格した画面から後続の画面単位検証（結合検証）へ引き継ぐ（引き継ぎ判断は orchestrating-reverse-docs-flow が行う）

§4カンニング防止（二層）

生成物が「正解を見て書いた」ものにならないよう、2 つの独立した層で防止する。

層対象方式
層 1: ファイル内の答え対象ファイルの内容P2 で git rm により白紙化。削除直前の HEAD を「原本参照コミット」として記録し、以降 P5 まで作業ツリー上に実体が存在しない。原本は P5 でのみ git show <commit>:<path> で一時ファイルへ抽出する
層 2: セッション文脈の汚染P4 生成エージェントの会話文脈P4 は必ず worker-sonnet へ隔離委任。prompt には設計書の該当章・共通部品・単体テスト観点表の該当行・P3 のテストのみを渡し、元コードのパス・内容を一切含めない。ループの各周で新規サブエージェントを起動し前周の生成文脈を持ち越さない

§5動作フロー（P1〜P7）

P1 入力ロード + preflight ── §15.1 対象確認 or from-zero 宣言
P2 白紙化ゲート ── 内部整合監査 → ユーザー承認 → 対象ファイル1件のみ git rm
P3 単体テスト仕様作成 ── 観点表からRED（TDD）
P4 サブエージェント生成 ── worker-sonnetへ隔離委任・唯一コードを書けるフェーズ
P5 差分比較（6計測） ── 原本Read解禁・measure-file-diff.sh
P6 設計書修正 ── 差分4分類・不合格ならP2へ再投入（上限5回）
P7 NG分類 + 報告 ── 収束/未収束判定・修正指示書

各 Phase における元コード（原本参照コミットの内容）Read 可否を色分けして示す。

Phase内容元コード Read 可否完了条件
P1 設計書・DESIGN.md・共通設計書・メッセージ定義書・単体テスト観点表を frontmatter 経由でロード。audit-consistency.sh --list-contract-files で §15.1 対象一覧を取得し、対象ファイルの列挙有無を確認 可（存在確認のみ） 対象ファイルが §15.1 リストに含まれることを確認済み（または from-zero 宣言済み）・設計書一式ロード済み
P2 audit-consistency.sh 通常モードで内部整合性を確認。違反があれば P7 へ直行。問題無ければ AskUserQuestion で承認後、対象ファイル1件のみ git rm 可（対象特定のため。字面読み禁止） 内部整合監査結果記録済み・ユーザー承認取得済み・原本参照コミット記録済み・白紙化コミット完了
P3 単体テスト観点表から対象ファイルの担当観点を抽出しテストケースを先に書く（TDD の Red） 禁止（git show 等の間接参照も禁止） 対象ファイルの観点に対応するテストケース一式が RED で書き出し済み
P4 唯一コードを書けるフェーズ。worker-sonnet へ隔離委任（1 エージェント = 1 ファイル） 禁止（メイン・サブエージェントとも） 対象ファイルが生成され、P3 のテストがローカルで実行可能
P5 原本参照コミットから一時ファイルへ抽出。単体テスト仕様の検査 + 6 計測（measure-file-diff.sh）を実施 可（本フェーズのみ） 単体テスト仕様の検査結果 + 6 計測の結果が省略なく記録済み
P6 P5 の差分を 4 分類（import / style / logic / JSX）に振り分け、設計書修正先を決定。不合格なら P2 へ再投入 禁止（P5 の数値・パターンのみを根拠にする） 差分 4 分類・修正内容（または P2 再投入判断）が記録済み
P7 収束（6 計測 PASS かつ単体テスト仕様の検査 全項目合格）/ 未収束を判定し、修正指示書または「NG なし」を出力 禁止（P5 の結果のみ参照） 修正指示書または「NG なし」明示・収束/未収束判定完了
from-zero: §15.1 に対象ファイルが存在せず設計書側にも記載が無い場合、P1 で新規追加を宣言してから P2 以降に進む。ひな形は assets/リバース検証/画面/詳細設計/画面詳細設計書.md のテンプレ構造に従う。単体テスト観点表に対応する観点が無い場合も同様に、意味キー規約（連番禁止・内容要約キー）に沿って最低限の行を追記してから進める。
実測が要る断定の scope 境界（P3）: 初期表示値・DOM 配置順序・要素位置・レイアウトなど、実行時の観測（Playwright 実測等）でしか正しく決められない断定は、本スキルの from-zero では [画面単位検証で実測] プレースホルダに留める。本スキルが確定させるのは静的に決定可能な宣言的契約（型・import・export・Props 名・API 契約形・ロジック構造・定数値）に限定し、実測は Playwright を持つ画面単位検証に委ねる。

§66 計測（合格判定表 + measure-file-diff.sh 入出力）

計測合格条件
単体テスト仕様の検査（禁止パターン・import・コンポーネント/API 利用）全項目合格
import の diff0 行
スタイル定数の diff0 行
ファイル全体の diff行数を出力（参考値・合格判定には使わない）
実質 diff（コメント・空行除外）20 行以下
重大違反: import diff だけでの合格判定は重大違反とする。6 計測はすべて出力し、いずれか 1 つでも省略してはならない。

measure-file-diff.sh の verdict は生シグナルである。差分のうち [画面単位検証で実測] に該当する実測委譲差分（初期表示値・DOM 順・位置・レイアウト起因）は収束判定の分母から除外し、静的な宣言的契約由来の差分のみで PASS/FAIL を判定する。

measure-file-diff.sh の入出力

項目内容
呼び出しscripts/measure-file-diff.sh <generated-file> <original-file>
標準出力import_diff_lines=<整数> / style_diff_lines=<整数> / total_diff_lines=<整数> / substantive_diff_lines=<整数> / verdict=PASS|FAIL（key=value を1行ずつ・機械可読）
verdict 判定式import_diff_lines == 0 && style_diff_lines == 0 && substantive_diff_lines <= 20 のとき PASS、それ以外 FAIL
対象外「単体テスト仕様の検査」（禁止パターン・import・コンポーネント/API 利用）は本スクリプトの対象外。P5 でメインが別途実施する
異常系引数不足・ファイル不在は stderr にメッセージを出して exit 1。正常時は exit 0

4 計測の算出方法（heuristic）

計測算出方法
import diff両ファイルから ^\s*import 行を抽出→ソート→diff の差分行数（^[<>] 行の数）
style diffclassName / style= 属性・const .*Style 系変数・16 進カラーコード（#[0-9a-fA-F]{3,8}）・px/rem/em/vh/vw を含む数値行を抽出→ソート→diff の差分行数
全体 diffdiff <original> <generated> の差分行数（参考値。合格判定には使わない）
実質 diffコメント行（^\s*//・^\s*/\*・^\s*\*）と空行を除外した上での diff 差分行数

P5 の抽出手順: git show <原本参照コミット>:<対象ファイルパス> > $CLAUDE_JOB_DIR/tmp/<basename>.original（$CLAUDE_JOB_DIR 未設定時は ${TMPDIR:-/tmp}/claude-job-<session>/ にフォールバック）。worktree・設計書リポジトリを汚染しないよう常にスクラッチ領域へ抽出する。

§7ループ設計

要素内容
反復条件（内側）P5 の判定（単体テスト仕様の検査 または verdict）が FAIL の場合、P6 で設計書を修正した上で P2（白紙化）へ再投入し、P3→P4→P5 をやり直す
上限回数（内側）5 回
停止条件（内側）収束停止: 単体テスト仕様の検査 全項目合格 かつ verdict=PASS / リソース上限: 5 回到達 / 発散検知: 同一差分集合（import diff・style diff・実質 diff の内容シグネチャ）が 2 連続で再発
反復条件（外側）なし（1 起動 = 1 ファイルの往復。他のファイルへは進まない）
上限回数（外側）1
検証役の分離P5 の判定は scripts/measure-file-diff.sh の決定的出力のみで行う。P4 生成エージェントの自己申告では判定しない

P6 再投入時、原本参照コミットは変わらないため再取得不要（対象ファイルを再度白紙化し P3〜P5 をやり直すのみ）。発散確定（同一差分集合 2 連続）は失敗ではなく手がかりであり、差分内容自体を P7 の NG 分類材料として使う。

§8サブエージェント委任仕様

呼び出し箇所subagent_typeprompt 骨格期待返却値
P4（生成） worker-sonnet 対象設計書の該当章の抜粋 + 共通部品（DESIGN.md・共通設計書・メッセージ定義書の関連部分）+ 単体テスト観点表の該当行 + P3 で書いたテスト + 「元コードのパス・内容を一切含めない」を明記 実装ファイル1件・変更内容・P3 テストのローカル実行可否
P5（テスト実行・6 計測スクリプト実行） worker-haiku 実行するテストコマンド・scripts/measure-file-diff.sh 呼び出しコマンドをベタ書き・判断不要と明記 各コマンドの終了コード・出力全文（6 計測の key=value を省略なく）
新規起動の徹底: ループの各周（P6→P2 再投入後の P4 再実行時）で必ず新規に worker-sonnet を起動する。前周の会話・生成物の文脈を prompt に含めない（同一エージェントへの追加指示は禁止）。
