封印済みfactsからのリバース設計書執筆スキル
generating-reverse-detailed-design

封印済み facts（facts.yml）とプロジェクト共通文書だけを情報源に、リバース設計書（画面詳細設計書・DESIGN.md・単体テスト観点表）を著述する執筆専任スキル。対象リポジトリの原本コードは一切 Read しない。

こんな人向け
facts が封印済みで、リバース設計書を新規著述・再著述したい人。
こんな場面で使う
extracting-unit-facts-from-code が status=封印済みでfacts_refを返した後。検証スキルが「差し戻し」で戻ってきたときの再著述。

§1概要

本スキルは体系内で唯一「原本を読まずに書く」ことが仕事の執筆役である。原本を読むのは上流の extracting-unit-facts-from-code だけであり、本スキルはその成果物（封印済み facts.yml）とプロジェクト共通文書だけを情報源に、宣言的契約を各章のテンプレート様式へ転記・整形する。

執筆専任: facts抽出（→ extracting-unit-facts-from-code）・共通文書採録（→ generating-reverse-common-docs）は本スキルの範囲外。本スキルは「封印済みfactsが既にある」ことを前提に動く。
原本 Read 全面禁止: 本スキル実行中に対象リポジトリの原本コードを Read することは、検証の盲検性を壊す契約違反として全面禁止する。facts.yml の欠落・矛盾に気づいても自ら原本を確認せず、extracting-unit-facts-from-code への差し戻し事由として記録する。

§2出力

画面 1 つにつき、以下の設計書が生まれる。

画面詳細設計書
事実一覧表に記録された画面の API 呼び出し・状態管理・バリデーション・UI コンポーネント構成などの事実を、章構成に沿って人間が読める設計書として再構成したドキュメント。コードを読まなくても、その画面が「何を表示し」「どんな操作を受け付け」「裏でどんな処理が走るか」を把握できる。

§3用語

facts_ref
extracting-unit-facts-from-code が確定した、封印済み facts ディレクトリの絶対パス（<screen_dir>/検証記録/facts/<run_id>/）。facts.yml・facts.lock・recount-report.txt を含む。
common_docs_root
generating-reverse-common-docs が採録したプロジェクト共通文書のルート（プロジェクト共通/）。本スキルの情報源のもう一方。
封印検証
shared/scripts/seal-facts.sh verify <facts_ref>による、facts.yml が封印時から改変されていないことの確認。Phase 2の必須ゲート。
実測委譲
facts.yml の measurement_pending（⑨）に由来する事実を確定値として転記せず、実測委譲（画面単位検証で確定）のプレースホルダを残す規律。画面単位検証（後続スキル）が実測で確定する。

§4使い方（起動引数）

全量指定・対話ゼロ。単独起動時は下表の args をユーザーから直接取得する。

引数必須内容
screen_dir必須画面ディレクトリ絶対パス
docs_root必須設計書展開先ルート
template_root必須テンプレート原本ルート
chapter_map_path必須章役割キー対応表
audit_script_path必須内部整合性監査スクリプト
scaffold_script_path必須スキャフォールディングスクリプト（実体: shared/scripts/scaffold-screen.sh。正本はこの1本のみ）
facts_ref必須封印済み facts ディレクトリの絶対パス
common_docs_root必須プロジェクト共通文書ルートの絶対パス
mode必須file または screen
target_file_pathmode=file時必須対象ファイルのパス
screenshot_dir / registry値任意補助情報源
target_repo_path・target_branch・source_ref は受け取らない。原本アクセスの入口を残さないため args から削除済み。

§5設計原則

原則内容
正本一元化facts.yml は封印済みの確定情報であり第二の正本ではない。正は設計書。食い違ったら設計書を直す
「該当なし」に根拠必須根拠なしの裸の「未確認」は完了条件違反
プロジェクト非依存固有値は起動引数・設計書側に置き、SKILL.md本文には書かない
原本Read禁止本スキル実行中に対象リポジトリの原本コードをReadすることを全面禁止する。情報源はfacts_refとcommon_docs_rootに限定する
検証スキルとの関係AUTHORED後に検証スキル(rebuilding-screen-unit-from-docs)を起動し、検証「差し戻し」の差し戻し先は本スキル

§6動作フロー（Phase 1〜6）

Phase 1: preflight（起動引数検収・スキャフォールディング）
  必須引数を検収する(screen_dir/docs_root/template_root/chapter_map_path/
   audit_script_path/scaffold_script_path/facts_ref/common_docs_root/mode/target_file_path)
  画面ディレクトリ未存在ならscaffold_script_path(実体: shared/scripts/scaffold-screen.sh)で展開、
   既存なら--verifyで健全性確認
  完了条件: 必須引数が揃い、画面ディレクトリの構造健全性を確認済み

Phase 2: 封印検証と facts 読込
  seal-facts.sh verify  を実行、exit 0を確認(必須ゲート)
  exit 1ならstatus=BLOCKED(hint: extracting-unit-facts-from-codeで再封印せよ)
  exit 0ならfacts.ymlとcommon_docs_root配下の共通文書を読み込む
  対象リポジトリの原本コードは一切Readしない
  完了条件: seal-facts.sh verifyがexit 0、facts.ymlと共通文書の読込完了

Phase 3: 観点表追記
  facts.ymlから単体テスト観点表へ観点行を追記する(意味キー規約準拠)
  measurement_pending由来の観点は「実測委譲(画面単位検証で確定)」として留保
  完了条件: facts.yml由来の観点行が観点表に追記済み・意味キー規約準拠

Phase 4: 設計書転記
  facts.ymlの各セクションをマップ(§6)に従って各章へ転記する
  measurement_pendingは転記せず「実測委譲(画面単位検証で確定)」を残す
  完了条件: 転記完了・measurement_pendingが留保済み

Phase 5: 完全性ゲート
  check-fact-coverage.sh /facts.yml <設計書.md> []
  audit-consistency.shを通常モードで実行、違反0件を確認
  完了条件: check-fact-coverage.shがexit 0かつaudit-consistency.sh違反0件

Phase 6: 返却
  返却ブロックを検証記録に保存する
  完了条件: status=AUTHOREDの返却ブロックが検証記録に保存済み

§7facts.yml → 章の転記マップ

スキーマの正本は shared/references/facts-schema.md。転記先の判定規律の正本は references/writing-rules.md。

facts.yml セクション転記先
import§15.3（依存）
export_type§15.1（ファイル分割）/ §15.2（型定義）
const§10（定数・設定値）
state§5（状態管理）
handler§8（イベント処理）
jsx§3（画面構造）/ §9（領域別仕様）
styleDESIGN.md + §3.6/§15.6のキー参照
api§7（API通信仕様）
measurement_pending転記せず「実測委譲（画面単位検証で確定）」+ measurement_pending

章の役割キー→§番号の解決は起動引数 chapter_map_path を正本とする。§番号は既定値であり、設計書の章マップ表で解決する。

§8執筆規律（writing-rules.md）

references/writing-rules.md が転記の判定規律・字面転記と要約の境界・実測委譲の書式・禁止事項の正本。

対象許可（正規化）禁止（字面転記・創作）
型定義export_type の全フィールドを§15.2のtypescriptブロックで表現facts.ymlのvalueをそのままYAML引用符付きでコピー
ロジックhandlerのvalue（処理1行要約）を挙動表の文へ整形facts.ymlに無い条件分岐・処理詳細を推測で補う
JSXjsxのvalue（ネスト記述）をそのまま構造記述として使うfacts.ymlに無い属性値・実測レイアウトを創作する
スタイルstyleのvalueをDESIGN.md側の記載とし設計書側はキー参照に留めるfacts.ymlに無い数値・色を推測で埋める
APIapiのvalue（req/res形）を§7.2の表形式へ整形facts.ymlに無いエンドポイント・パラメータを創作する

禁止事項: facts に無い事実の創作／原本参照／facts.yml の改変／意味キーの変更（キー文字列は check-fact-coverage.sh の転記突合で使うため言い換え禁止）。

§9完全性ゲート（check-fact-coverage.sh）

合否は exit code のみで判定する（自然文の自己申告での合格判定は行わない）。

項目内容
入力<facts.yml> <画面詳細設計書.md> [<DESIGN.md> ...]
突合方法facts.yml の sections 配下 items[].key を抽出し、設計書いずれかに固定文字列で言及されているかを確認する
measurement_pending の特例設計書に「実測委譲」の表記が1箇所でもあれば、個別キー一致なしで転記済み扱い。表記が無ければ他セクション同様に個別キー一致を要求する
判定未転記が1件でもあれば exit 1（fail-closed）。全件転記済みで exit 0
自己テスト--self-test: 陽性（転記済み+実測委譲表記あり）／ 陰性1（通常キー未転記）／ 陰性2（実測委譲表記なし・個別キー未転記）の3ケースを検証する

§10返却ブロック

orchestrating-reverse-docs-flow の契約（返却ブロック共通サブセット: status/scope/artifacts/hint）に準拠する。

キー値
statusAUTHORED（著述完了）または BLOCKED（facts未封印・引数不足等）
scope<system>-<画面ID>（工程を跨いだ同一性キー）
artifacts画面詳細設計書・DESIGN.md・単体テスト観点表のパス
facts_ref（拡張）入力で受け取ったfactsディレクトリの絶対パスをそのまま転記（下流工程への追跡用）
measurement_pending（拡張）実測系として設計書に確定せず画面単位検証へ委譲した項目の一覧
hint次工程（検証スキル起動）への申し送り・差し戻し理由

§11ループ設計

要素内容
反復条件check-fact-coverage.shが未転記キーを検出（exit 1）したら、Phase 4で転記を補いPhase 5を再実行する
上限回数5回
停止条件収束停止: check-fact-coverage.sh exit 0 かつ audit-consistency.sh違反0件 ／ リソース上限: 5回到達（未収束ならBLOCKED）
Phase 2ゲートとの違いPhase 2の封印検証失敗（exit 1）はこのループの対象外。即status=BLOCKEDの終端条件
サブエージェント委任facts読込・執筆(Phase 2〜4)はサブエージェントへ委任しない。単一のメインエージェントが章マップ・執筆規律の一貫性を保って通しで担う

§12予想を裏切る挙動

  • 原本コードのReadは全面禁止。情報源はfacts_ref配下のfacts.ymlとcommon_docs_root配下の共通文書のみ
  • facts.ymlの字面（value列）をそのまま書き写すだけでなく、章の文脈に沿って正規化して書く。ただしfactsに無い事実を創作しない
  • measurement_pending（⑨実測系）を目視転記・推測で確定しない。「実測委譲（画面単位検証で確定）」に留めmeasurement_pendingへ回す
  • 「該当なし」は必ず根拠を添える。裸の「未確認」は完了条件違反
  • facts.yml自体の誤り・欠落に気づいても本スキルは書き換えない。extracting-unit-facts-from-codeへの差し戻しとしてhintに記録する
  • 本SKILL.md本文にリバース対象の固有値（対象リポジトリパス・画面ID・BL名）を書かない
  • 進捗はStep単位でTaskCreate/TaskUpdateする（一括登録しない）
