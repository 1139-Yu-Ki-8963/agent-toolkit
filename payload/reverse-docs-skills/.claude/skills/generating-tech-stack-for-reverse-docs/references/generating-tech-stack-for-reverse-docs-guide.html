技術スタックページ生成スキル
generating-tech-stack-for-reverse-docs

アーキテクチャ調査書と対象リポジトリの定義ファイルを突合し、一致を確認できた項目だけから「技術スタック.html」を機械検証付きで生成するスキルの確定仕様。

こんな人向け
ポータルの将来ページ受け口に技術スタックカードを反映したい人 ／ 調査書に書かれた技術選定が実コードの定義ファイルと一致しているかを機械的に裏取りしたい人
こんな場面で使う
アーキテクチャ調査書が確定済みで、次にポータル基盤情報ページを揃えたい時 ／ 調査書記載値と `package.json` 等の実測値の乖離を早期に検出したい時
対応 OSWindows WSL2 環境
検証状況エンジンスクリプト（validate-page-data.sh / build-detail-page.sh）はフィクスチャでの動作確認済み。スキル単体の実機検証は未実施
依存jq ／ shared/scripts/detail-pages/validate-page-data.sh・build-detail-page.sh（スキルフォルダ相対 ../../../shared/scripts/detail-pages/）／ shared/scripts/build-portal.sh（任意・カード反映時）
関連資料SKILL.md, ../../../shared/scripts/detail-pages/page-data-schema.md

§1概要

リバースエンジニアリング設計のポータルには、将来ページの受け口（output_dir 直下に該当 HTML が実在するときだけカード化される仕組み）がある。本スキルはそのうち技術スタック（T3）専任の生成係として、アーキテクチャ調査書 §2 の記載値を対象リポジトリの定義ファイル実測値と突合し、技術スタック.html を機械検証付きで作成する。

本スキルは判定・評価をしない。技術選定の良否は問わず、調査書と実測値が一致した項目のみを転記する。乖離を検出した項目があれば、生成せずに乖離内容を報告して停止する。

── 処理の全体像 ──────────────────────────────

  アーキテクチャ調査書.md（§2 技術スタック表）
       │  Phase 1  調査書の実在確認 + 定義ファイル列挙
       ▼
  対象リポジトリの定義ファイル（package.json 等）
       │  Phase 2  §2 記載値と実測値の突合（乖離なら生成せず停止）
       ▼
  page-data.json（pageKind: techstack）
       │  Phase 3  validate-page-data.sh --target-repo で機械検証
       ▼
  検証済み page-data
       │  Phase 4  build-detail-page.sh --page techstack
       ▼
  <output_dir>/技術スタック.html（+ 任意で build-portal.sh 再実行）

画面一覧・テーブル一覧のようなユニット抽出とは異なり、本スキルは「既に確定した 2 つの一次情報（調査書と定義ファイル）を突合するだけ」の係である。新規の検出戦略宣言は行わない。

§2出力

スキルを実行すると、以下の 2 点が生まれる。

技術スタック.html（<output_dir> 直下）
言語・ランタイム／フレームワーク／パッケージマネージャ／ルーティングライブラリの 4 項目を、要約タイルと明細表の 2 段で示すページ。明細表の各行には出所（突合に使った定義ファイルの実パス）が必ず付き、読んだ人は記載値がどのファイルの実測から来たかをその場で追跡できる。
埋め込み page-data JSON
一覧 HTML の中に、生成の元になった page-data.json がそのまま埋め込まれる。読んだ人は「いつ・どの調査書 §2 記載値から生成されたか」を HTML 単体で追跡でき、機械処理（後続スキルからの参照）にも使える。

調査書と定義ファイルに乖離がある場合、本スキルは生成を行わず乖離内容の報告のみで終わる。この場合、技術スタック.html は生まれない。

§3用語

page-data.json
detail-pages 系 5 スキルが共有する入力 JSON。pageKind ごとに型別スロットを持つ。本スキルは pageKind: "techstack" を固定で使う。スキーマの全量は page-data-schema.md を参照。
tiles / rows
techstack 型のスロット。tiles[] は領域別代表 4 枠以内の要約タイル、rows[] は {item, value, sourceRef} 形式の明細表 1 行 = 1 要素。
突合
アーキテクチャ調査書 §2 の記載値と、対象リポジトリの定義ファイルの実測値を項目ごとに比較する Phase 2 の処理。一致のみを転記し、乖離は捏造せず停止する。
整合検証
validate-page-data.sh --target-repo による page-data.json の機械検査。sourceRef が対象リポジトリ上に実在するかを検査する。

§4動作フロー

起動引数は target_repo_path・output_dir（必須）と portal_output_dir（任意）の 3 つ。種別は techstack に固定されているため、種別を選ぶ引数はない。

Phase内容完了条件
Phase 1
データ源読込		 アーキテクチャ調査書の実在確認（不在ならハード停止し surveying-architecture-for-reverse-docs の先行実行を案内）、対象リポジトリの定義ファイル列挙 調査書の実在確認済み、または不在を報告して停止
Phase 2
抽出・突合		 調査書 §2 の記載値と定義ファイル実測値を項目ごとに突合。全項目一致で page-data.json を組み立て、乖離があれば生成せず停止報告 page-data.json 保存済み、または乖離を報告して停止
Phase 3
整合検証（機械実行）		 validate-page-data.sh <page-data.json> --target-repo <target_repo_path> を実行。FAIL は修正して再実行（3 回失敗で Phase 2 へ差し戻し） 全項目 PASS（exit 0）
Phase 4
技術スタック.html 生成		 build-detail-page.sh <page-data.json> <output_dir> --page techstack を実行。portal_output_dir 指定時は build-portal.sh を再実行しカードへ反映 HTML 生成済み。指定時は build-portal.sh 再実行済み

エンジンスクリプトはスキルフォルダ相対 ../../../shared/scripts/detail-pages/ で参照する。

返却（呼び出し元との契約）

キー内容
statusDONE | STOPPED（調査書不在・乖離検出）| ERROR
artifacts生成した技術スタック.html のパス（STOPPED/ERROR 時は空）
page_kindtechstack（固定値）
portal_rebuiltbuild-portal.sh 再実行の有無
hint停止理由、または次工程への申し送り

§5制約と禁止事項

手作業でのプレースホルダ置換は禁止。 技術スタック.html は必ず build-detail-page.sh 経由の決定的処理で生成する。
乖離はハード停止。 調査書記載値と定義ファイル実測値が食い違う項目があれば、生成せずに乖離内容を報告して停止する。AskUserQuestion で手動値を聞き出して即興確定しない。
判定・評価は対象外。 技術選定の良否・妥当性には踏み込まない。本スキルの仕事は一致事実の転記のみ。

抽出時の落とし穴

落とし穴対処
調査書の「実在しない（理由: …）」項目を転記してしまう突合対象外とし、rows[] にも含めない
sourceRef を文書参照形式にしてしまう突合に使った定義ファイルの実パスを使う（文書参照形式は実在検査の対象外になる）
jq 未インストール環境での検証・生成失敗validate/build 両スクリプトが jq に依存するため事前に導入する
