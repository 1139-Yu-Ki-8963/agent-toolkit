画面単位リバース検証バッチスキル
running-reverse-screen-batch

画面単位のリバース設計検証を claude CLI のヘッドレスモード（claude -p、対話画面を介さず1回の呼び出しで完結する実行方式）による無人バッチループで実行するスキルの確定仕様。

こんな人向け
対象画面が4件以上あり、通常セッションでの逐次処理では時間がかかりすぎる人 ／ orchestrating-reverse-docs-flow の画面バッチ Phase から機械的に呼び出す立場の人
こんな場面で使う
画面開通・共通採録が完了し、残り画面をまとめて facts 封印〜往復検証まで通したい時 ／ 夜間などの無人運転で残ゼロまで自動継続させたい時
対応 OSWindows WSL2 環境
検証状況統括スキルの画面バッチ Phase から起動される無人バッチ実行。対話ゼロ契約
依存claude CLI（claude -p ヘッドレスモード）、orchestrating-reverse-docs-flow（起動元）

§1概要

画面単位のリバース設計検証（facts 封印〜設計書著述〜コード再生成〜往復検証）は、通常セッションでは1画面ずつの逐次処理になる。対象画面数が多いプロジェクトでは、この逐次処理が全体の完了を大きく遅らせる。本スキルは claude -p の無人バッチループにより、1画面=1呼び出しの単位で全対象画面を機械的に処理し続ける係である。

マーカー冪等性（同じ画面を再実行しても安全に再開できる性質）・limit 耐性（API 利用上限に達しても待機して再開する性質）・残ゼロまで継続、の3要件を満たす設計になっている。orchestrating-reverse-docs-flow から args 全量指定で起動され、AskUserQuestion を一切発行しない対話ゼロ契約を前提とする。

── 処理の全体像 ──────────────────────────────

  target_repo_path / docs_root / screen_ids 等の args
       │  Phase 1  引数検証 + 対象画面一覧生成
       ▼
  batch-targets.txt（1行1画面ID）
       │  Phase 2  単発ドライラン（1画面で前半・後半の完走確認）
       ▼
  ドライラン合格
       │  Phase 3  無人ループ起動（nohup + disown でバックグラウンド化）
       ▼
  周回ループ（claude -p を画面ごとに前半・後半の計2回呼び出す）
       │  Phase 4  監視と完了確認（残件カウント）
       ▼
  全画面が検証完了 / failed退避 / conflict-skip退避のいずれかで確定
       │  Phase 5  完了報告
       ▼
  verification_dir 配下の検証結果 + バッチ運転記録 + progress.jsonl

盲検分離（原本コードを読む工程と設計書のみで判定する工程を別プロセスに分離する要件）を満たすため、1画面につき claude -p を前半（著述）・後半（ファイル単位盲検検証・往復検証）の2回に分けて呼び出す。

§2出力

スキルを実行すると、以下の3点が生まれる。

全画面の検証結果（verification_dir 配下）
各画面の facts 封印・設計書著述・コード再生成・往復検証の結果が画面ごとに蓄積される。前半完了で画面レジストリの status が authored、後半完了で baseline-established になり、どこまで検証が進んだ画面かをレジストリから一目で追跡できる。
バッチ運転記録（<verification_dir>/バッチ運転記録/ 配下）
batch-log.txt（全画面の実行ログ）・failed-screens.txt（連続失敗で退避した画面のリスト）・conflict-skip-screens.txt（他プロセスとの競合でスキップした画面のリスト）の3ファイルが生まれ、読んだ人はどの画面が正常完了しどの画面に人手の追跡が必要かをこの3点だけで判断できる。
progress.jsonl（<verification_dir>/progress.jsonl）
各画面の工程開始・完了・失敗を {"ts":"...","screen_id":"...","phase":"...","status":"started|completed|failed"} 形式で1行ずつ追記したログ。呼び出し元セッションや人間はこのファイルを監視するだけで、バックグラウンドで動くバッチの現在工程をリアルタイムに把握できる。

周回ループに上限はない（意図的な設計判断）。停止するのは全画面確定（収束）または画面ごとの連続失敗上限到達（発散検知）のいずれかであり、対象画面数が有限である限り必ず有限回で終了する。

§3用語

マーカー冪等性
同じ画面を再実行しても安全に再開できる性質。成否判定は画面レジストリの status（authored / baseline-established）の grep で行い、終了コードは判定に使わない。
limit耐性
API 利用上限（limit）に達しても処理を打ち切らず、既定の待機秒数（wait_seconds）だけ sleep してから同じ画面を再試行する性質。
盲検分離
原本コードを読む工程（前半: 著述）と設計書のみで判定する工程（後半: ファイル単位盲検検証・往復検証）を別プロセスに分離する要件。前半で読んだ原本コードの情報が後半のコンテキストに混入しないよう、1画面につき claude -p を2回に分けて呼び出す。
レーン
lane_id で識別される並列実行単位。横展開（400画面超）で複数レーンを並列起動する際に用い、ログ・failed リスト・conflict-skip リストをレーン別に分離する。
conflict-skip
他プロセスが同一画面の作業コピー・dev サーバー等を使用中で開通できなかった場合の退避区分。失敗（failed）とは別区分で扱われ、連続失敗カウント（fail_limit_k）には加算しない。

§4実行フロー

起動引数は target_repo_path・docs_root・screen_ids・template_root・common_docs_root・survey_doc_path が必須。model・wait_seconds・fail_limit_k・log_path・lane_id・deadline は任意。

Phase内容完了条件
Phase 1
引数検証と対象一覧生成		 全必須引数の存在と参照先パスの実在を確認。screen_ids が "all" の場合は画面一覧HTMLのマニフェストから全画面IDを抽出し、batch-targets.txt に書き出す 対象画面一覧ファイルが生成され、総数と既検証数が確認済み
Phase 2
単発ドライラン		 対象一覧の先頭1画面だけをフォアグラウンド実行し、前半（著述、status=authored）・後半（ファイル単位盲検検証・往復検証、status=baseline-established）の両工程を1画面分だけ完走させて確認する 前半・後半の両工程完走とマーカー付与を確認済み。合格前は Phase 3 に進むことを禁止
Phase 3
無人ループ起動		 references/loop-design.md の雛形にパラメータを反映し、nohup bash -c '...' >> ログ 2>&1 & disown 構造でバックグラウンド起動する PID の生存確認が取れ、監視コマンド（tail -f <ログ>）が報告済み
Phase 4
監視と完了確認		 マーカー未付与かつ failed リスト・conflict-skip リスト外の画面数（残件）を計算する 全画面が検証完了、または failed / conflict-skip リストへの退避で確定している
Phase 5
完了報告		 最終の残件数・failed件数・conflict-skip件数・実周回数を再カウントし報告する。failed・conflict-skip の画面ID一覧を添える 完了報告提示済み

Phase 3 で使うループ本体はディスク保存せず、Phase 実行のたびに雛形からインライン展開する。これにより生成されたループはスキルディレクトリの存在に依存せず、呼び出し元セッション終了後もバックグラウンドプロセス単体で完走できる。

§5注意事項

--worktree フラグは存在しない。 claude CLI 2.1.206 時点で未実装のため、worktree が必要な場合は呼び出し側が事前に git worktree add を実行してから claude -p の cwd をそこに向ける。
バックグラウンドシェルは寿命が短い。 メインセッションが最終結果を返した直後にシェル自体が終了する仕様があるため、nohup + disown でプロセスをシェルの生死から切り離し、起動10秒後に kill -0 で生存確認する。
サンドボックスの外向き接続制限に必ず当たる。 claude -p は Anthropic API への外向き接続が必要なため、起動コマンドは dangerouslyDisableSandbox: true を常時使用する。「Not logged in」と表示されても実際は認証済みで、単に接続がサンドボックスに拒否されているだけのケースがある。
終了コードは信用しない。 claude -p が正常終了しても対象画面の検証が未完了なケースがあるため、成否判定は必ず画面レジストリの status チェックで行う。

その他の落とし穴

落とし穴対処
画面ID一覧との文字列比較・レジストリ検索が原因不明に失敗する（CRLF・エンコーディング混入）Phase 1 の一覧生成で tr -d '\r'・空行除去による正規化を省略しない
~/.claude/projects/ が肥大化し Claude Code の起動が遅くなる--no-session-persistence を必ず付ける
1周回あたりの所要時間が長い（画面数 × 実測60分程度）想定時間を事前に計算し、夜間運転で余裕を持たせる。Mac のスリープ無効を確認する
macOS 標準 bash（3.2系）で declare -A が invalid option で落ちる連想配列を使わず、awk によるファイルベースのカウンタで失敗回数を管理する
周回ループに上限がなく暴走に見える意図的な設計判断。暴走防止は画面単位の FAIL_LIMIT_K のみで担保し、対象数が有限であるため必ず有限回で終了する
