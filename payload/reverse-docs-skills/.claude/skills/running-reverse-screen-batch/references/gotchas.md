# 予想を裏切る挙動集

`running-reverse-screen-batch` の改修後の再検証にも使う、実機で確認済みの落とし穴。各項目は「事象・原因・対策」の3行構成。

## `--worktree` フラグの不存在

- **事象**: `claude --help` に `--worktree` という項目がない
- **原因**: claude CLI 2.1.206 時点でこのフラグは実装されていない
- **対策**: worktree が必要な場合は、呼び出し側が `git worktree add` を事前に実行してから `claude -p` の cwd をそこに向ける

## バックグラウンドシェルの寿命

- **事象**: `run_in_background` で起動したシェルが、メインセッションの応答が返った直後に終了してしまう
- **原因**: Claude Code のバックグラウンドシェルは「メインセッションが最終結果を返してから数秒でシェル自体が終了する」仕様がある
- **対策**: `nohup ... & disown` でシェルの生死からプロセスを切り離し、起動10秒後に `kill -0 $PID` で生存確認する

## Bash サンドボックスの外向き接続制限

- **事象**: `claude -p` の実行がタイムアウトまたは接続エラーで失敗する。`Not logged in` と表示されるケースがあるが、実際は認証済みで API 接続がサンドボックスに拒否されているだけ
- **原因**: `claude -p` は Anthropic API への外向き接続が必要だが、Bash ツールのサンドボックスが許可ドメイン外への接続を拒否する
- **対策**: 本スキルでは dangerouslyDisableSandbox: true を常時使用する設計。ドライラン（Phase 2）で接続確認する

## 終了コードは信用しない

- **事象**: `claude -p` が正常終了（exit 0）したのに、対象画面の検証が実際には未完了だった
- **原因**: ヘッドレス呼び出しは、途中で作業を打ち切っても正常終了コードを返すケースがある
- **対策**: 成否判定は必ず画面レジストリの status チェックで行う。終了コードは判定に使わない

## CRLF・エンコーディングの混入

- **事象**: 画面ID一覧との文字列比較・レジストリ検索が原因不明に失敗する
- **原因**: 画面一覧HTMLからIDを抽出した際にCRLFや不揃いなエンコーディングが混入しうる
- **対策**: Phase 1 の一覧生成で正規化（`tr -d '\r'`・空行除去）を省略しない

## `~/.claude/projects/` の肥大化

- **事象**: 大量呼び出し後にディスク使用量が増え、Claude Code の起動が遅くなる
- **原因**: `claude -p` はデフォルトでセッション記録を書き込み続ける
- **対策**: `--no-session-persistence` を必ず付ける

## 1画面あたりの処理時間

- **事象**: 1周回あたりの所要時間が長い（画面数 × 5〜15分）
- **原因**: 1画面で6工程（開通→事実→基本設計→詳細設計→基準確立→往復検証）を通すため、1呼び出しの実行時間が長い
- **対策**: 想定時間を事前に計算し（画面数 × 10分 × 想定周回数）、夜間運転で余裕を持たせる。Mac のスリープ無効を確認する

## bash の連想配列非対応（macOS標準 bash 3.2）

- **事象**: `declare -A` を使ったスクリプトが `invalid option` で落ちる
- **原因**: macOS 標準の `/bin/bash` はバージョン3.2系で連想配列に非対応
- **対策**: `awk` によるファイルベースのカウンタで失敗回数を管理する。改修時も連想配列を持ち込まない

## 周回ループに上限を設けない設計判断

- **経緯**: 要件「残ゼロまで継続」に対し周回数上限は矛盾する
- **修正**: 周回ループの上限を設けない。暴走防止は画面単位の `FAIL_LIMIT_K` のみで担保
- **なぜ安全か**: 全画面が「検証完了」または「failedリスト行き」で確定すれば remaining=0 になりループが自然終了する。画面数が有限であるためループも必ず有限回で終わる
