ユニット事実抽出スキル
extracting-unit-facts-from-code

対象ユニットの原本コードから宣言的契約factsを抽出し、独立再計数・封印・再現性検証を通して「執筆工程が原本を読まずに済む」品質のfacts.ymlを確定する。仕事はfacts.yml等3ファイルの確定のみで、対象リポジトリへの書き込み・変更は一切行わない。

こんな人向け
対象ユニット（画面）の原本コードから、後続の詳細設計執筆が参照する宣言的契約factsを抽出したい人。
こんな場面で使う
アーキテクチャ調査書が確定した後、画面ユニットの事実表を新規に抽出したいとき。判定がNG帰着(b)（facts欠落）と判定し、抽出プロファイルを改訂した上で再抽出したいとき。
動作範囲対象リポジトリは読み取り専用。出力は screen_dir 配下の 検証記録/facts/<run_id>/ の3ファイルのみ
対応プロファイルscreen のみ（Stage 3範囲）。他プロファイルは status=中断
対話AskUserQuestion不使用。args全量指定・対話ゼロで完走する
依存find / grep / awk 等の決定的コマンド、機械ゲート実行に scripts/recount-facts.sh・shared/scripts/seal-facts.sh を使用
関連資料SKILL.md, references/profile-screen.md, shared/references/facts-schema.md

§1概要

対象ユニット（画面）の原本コードを決定的手順で読解し、宣言的契約 facts.yml として確定する。抽出者と検証者を分離した独立再計数、確定後の封印、同一argsでの再現性検証を経て初めて「執筆工程が原本を読まずに済む」品質に達したとみなす。

単独動作: 他スキルへの依存を持たない。起動引数を渡せば単独で動く。工程全体は orchestrating-reverse-docs-flow が案内する。後続工程（詳細設計執筆）はこのfactsを前提として動く。
読み取り専用: 対象リポジトリへの書き込み・変更は一切行わない。出力は screen_dir 配下の1ディレクトリ（検証記録/facts/<run_id>/）のみ。

§2用語

facts.yml
9分類（①〜⑨）で構成される宣言的契約事実の正本。各項目は意味語キー・値・原本の file:line 根拠を持つ（⑨は値を持たない）。
独立再計数
recount-facts.shによる検証。facts.ymlを読まずにまずコードから分類別件数を独立算出し、その後facts.ymlの記載件数と突合する。
封印
seal-facts.shによる正規化ハッシュの記録（facts.lock）。以降の改変を1コマンドで検知できる状態にする。
孤児参照
facts.yml内の項目のevidence（file:line）のファイル部分が、target_file_pathsの集合に含まれない状態。対象ユニットの宣言範囲外を根拠にした事実を機械検知する。

§3使い方（起動引数）

全量指定・対話ゼロ。単独起動時は下表の args をユーザーから直接取得する。

引数必須内容
target_repo_path必須対象リポジトリの絶対パス
target_file_paths必須対象ユニットの対象ファイル（画面本体＋直接の子コンポーネント）の相対パス配列
screen_dir必須出力先の画面ディレクトリ絶対パス。facts は <screen_dir>/検証記録/facts/<run_id>/ に出力する
profile必須screen のみ実装。他値は status=中断で未対応と返す
survey_doc_path必須アーキテクチャ調査書のパス（方式→プロファイル選択の根拠。実在確認のみ）
run_id任意（既定 extract-1）抽出実行の識別子。出力ディレクトリ名・facts.yml内のrun_idフィールドに使う

§4設計原則

原則内容
読み取り専用対象リポジトリへの書き込み・変更は一切行わない。出力は screen_dir 配下のみ
原本事実主義推測・要約での補完を禁止する。コードに存在しない事実を書かない。全項目に file:line 根拠が必要
独立再計数による検証抽出者（Phase 2）と検証者（recount-facts.sh）を分離する。検証者はfacts.ymlを読まずにまずコードから件数を独立算出する
封印による改ざん検知確定したfacts.ymlは正規化ハッシュで封印し、以降の改変を機械検知できる状態にする
再現性の担保同一argsでの抽出結果は（run_idを除き）決定的に一致することをdiffで確認する
合格判定はスクリプトのexit codeのみ自然文の自己申告での合格判定を行わない

§5動作フロー（Phase 1〜6）

Phase 1: 前提確認
  target_repo_path・screen_dir・survey_doc_pathの実在を確認する(test -d/-f)
  target_file_paths全件の実在を確認する
  profile=screenを確認する(screen以外はstatus=中断)
  run_id確定+facts出力ディレクトリ作成
  完了条件: 全args解決済み・target_file_paths全件実在確認済み・
   facts出力ディレクトリ作成済み

Phase 2: 抽出
  profile-screen.mdの分類別抽出手順に従いfacts.ymlを作成する
  全項目に原本のfile:line根拠を付ける。推測・要約での補完を禁止する
  該当なしの分類はitems:[]+reasonで埋める
  完了条件: facts.ymlが9分類(sections配下9キー)の節を持つ

Phase 3: 独立再計数ゲート
  recount-facts.sh   
  乖離率5%以内・空欄率30%以内・孤児参照0件の3検査
  FAILしたらPhase 2に戻る(上限3回・同一NG理由2連続で発散中断)
  完了条件: recount-facts.shがexit 0、recount-report.txt保存済み

Phase 4: 封印
  seal-facts.sh seal  でfacts.lockを生成
  seal-facts.sh verify  で直後の整合性を確認
  完了条件: seal-facts.sh verifyがexit 0

Phase 5: 再現性検証
  同じargsで抽出をもう1度、一時ディレクトリに実行する
  両facts.ymlをseal-facts.sh normalizeで一時ファイルへ書き出しdiff比較
  diff≠0なら非決定箇所をhintに記録しstatus=中断(Phase 2へは戻らない)
  完了条件: 2回の正規化出力のdiffが空

Phase 6: 返却
  返却ブロックを出力する
  status=封印済み: artifacts/facts_ref/pending_measurementsを記す
  status=中断: hintに未解消の検査項目・非決定箇所・非対応プロファイル等を記す
  完了条件: statusが確定している

§6facts.ymlの9分類

スキーマの正本は shared/references/facts-schema.md。丸数字①〜⑨は分類記号であり連番IDではない。

記号セクションキーラベル備考
①importimportnamed/default/type/namespace全シンボル。副作用importも含む
②export_typeexport・型export名、interface/typeの全フィールド
③const定数トップレベル定数。オブジェクト/enumはフィールドごとに分解
④state状態変数useState/useReducer/useRef/store参照
⑤handlerイベントハンドラハンドラ名・発火要素・処理要約
⑥jsxJSX構造コンポーネントのネストのみ（属性値は書かない）
⑦styleスタイル実測値styled定数のプロパティ・数値・色
⑧apiAPI呼出BL名・契機・リクエスト/レスポンス形
⑨measurement_pending実測系（実測委譲）key・evidenceのみ。再計数対象外・転記対象外

§7機械ゲート（recount-facts.sh・seal-facts.sh）

recount-facts.shは合否を exit code のみで判定する（自然文の自己申告は用いない）。

検査内容
乖離検査①〜⑧の8分類をコードから独立再計数し、facts.ymlの記載件数との乖離率が5%を超える分類が1つでもあれば違反
空欄検査全9分類の各項目についてkey・evidence（file:line形式）の空欄率が30%を超えれば違反
孤児参照検査evidenceのファイル部分がtarget_file_pathsの集合に含まれない項目が1件でもあれば違反

seal-facts.shは3つのサブコマンドを持つ。

サブコマンド内容
sealnormalize済みfacts.ymlのsha256を facts_dir/facts.lock へ記録する
verifyfacts.lockの記録ハッシュと現在のfacts.ymlの再計算ハッシュを照合する。不一致ならexit 1
normalizerun_id行・行末空白・空行を除去した正規形をstdoutへ出す

§8返却ブロック

orchestrating-reverse-docs-flow の契約（返却ブロック共通サブセット: status/scope/artifacts/hint）に準拠する。

キー値
status封印済み または 中断
scopescreen_dir のbasename
artifactsfacts.yml・facts.lock・recount-report.txtの絶対パス（配列）
hint次工程への申し送り、または中断理由
facts_ref（拡張）facts ディレクトリの絶対パス＋facts.lockのsha256
pending_measurements（拡張）⑨実測委譲キーの一覧

§9ループ設計

要素内容
反復対象Phase 3（独立再計数ゲート）exit 1 → Phase 2（抽出）へ戻る
上限回数3回
収束条件recount-facts.sh が exit 0
発散条件同一のNG理由（同一検査の同一違反）が2回連続で再発した場合、発散として即中断する
上限到達時の報告status=中断とし、hintに未解消の検査項目・違反内容・試行回数を記録する
検証役の分離合否判定はスクリプトのexit codeのみ。Phase 5の再現性検証失敗はPhase 2へ戻らず終端条件として中断する

§10Gotchas・重要な注意事項

  • mktemp は必ず mktemp -d "${TMPDIR:-/tmp}/XXXXXX" 形式で使う。macOS の引数なし mktemp -d はサンドボックス環境で失敗する実害がある
  • 再現性検証のため facts.yml に現在時刻を書かない。run_id のみを持たせ、normalize が run_id 行を除去することで2回の抽出結果を比較可能にする
  • 独立再計数の乖離許容5%は正規表現ベースの近似計数と手動キー付けの粒度差を吸収するための閾値。profile-screen.md の分類別パターンに沿ってキー付け粒度を揃えないと乖離超過になりやすい
  • 空欄率検査は key・evidence の2フィールドのみを対象とする。value の欠落はこのメトリクスに現れない
  • 孤児参照は evidence のファイル部分が target_file_paths の集合と完全一致するかで判定する
  • grep実装（ugrep/BSD grep/GNU grep等）は同一パターンでもパイプ経由のstdinと通常ファイルとで正規表現マッチングの挙動が異なる場合がある。recount-facts.sh の分類別カウント関数は必ず実ファイルを引数に渡す
  • シェルスクリプト内で $変数 の直後に全角記号を空白なしで続けると、環境によって変数名の終端誤認識が起きる場合がある。${変数} のブレース記法を必須にする
  • Phase 5 の diff はプロセス置換 <(...) ではなく一時ファイル書き出し経由の比較に固定する（サンドボックス環境で /dev/fd への書き込みが権限拒否されるケースがある）
  • 対象リポジトリに対しては読み取り専用。書き込み・変更は一切行わない
  • 出力は screen_dir 配下のみ（検証記録/facts/<run_id>/ の3ファイル）。対象リポジトリ側には何も生成しない
  • 推測・要約での補完を禁止する。コードに存在しない事実を書かない
  • profile=screen 以外は未対応。指定された場合は抽出を行わず status=中断を返す
  • AskUserQuestionを使わない。args全量指定・対話ゼロで完走する
  • SKILL.md本文にプロジェクト固有値（リポジトリ名・画面名・絶対パス・ユーザー名）を一切書かない。固有値はすべて起動argsで受ける
