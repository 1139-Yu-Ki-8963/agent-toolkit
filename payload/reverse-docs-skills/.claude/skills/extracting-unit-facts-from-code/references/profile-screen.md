# profile-screen: screen プロファイルの抽出手順と再計数パターン

`extracting-unit-facts-from-code` の `profile=screen` 専用リファレンス。Phase 2（抽出）の分類別手順と、Phase 3（独立再計数ゲート）で `scripts/recount-facts.sh` が用いる決定的パターンを1つの表に集約する。両者は同じ対象（①〜⑧の8分類。⑨は再計数の対象外）を独立に数えることで、抽出漏れを機械検知する。

スキーマ本体（YAML構造・必須フィールド・孤児参照定義・normalize規則）は `shared/references/facts-schema.md` を正本とする。本ファイルは「どう抽出するか」「どう独立再計数するか」の手順のみを持つ。

## 分類別抽出手順と再計数パターン

| 分類 | Phase 2 抽出手順（何を拾い、どうキーを付けるか） | Phase 3 独立再計数パターン（`recount-facts.sh` が使う正規表現・数え方） |
|---|---|---|
| ①import | 対象ファイル群の `import` 文を全て読み、named/default/type/namespace の各シンボルを1事実ずつキー化する（`import-<モジュール>-<シンボル>`）。副作用 import（`import './x.css'`）も1件として記録する | 行頭 `^import[ \t]` に一致する行を対象に、`from` 以降を除去し `type`/`{`/`}`/`* as` を正規化した残りをカンマ分割し、空でないトークン数を数える。`from` を含まない行（副作用 import）は1件と数える |
| ②export・型 | `export` 宣言と、`interface`/`type` ブロック内の全フィールドをそれぞれ1事実ずつキー化する（`export-<名前>`・`type-<型名>-<フィールド名>`） | 行頭 `^export[ \t]` に一致する行を1件、かつ `interface`/`type` の開始行から閉じ `}` までの間で `^[ \t]*[A-Za-z_][A-Za-z0-9_]*\??:[ \t]*\S` に一致する行を1件として合算する |
| ③定数 | ファイル先頭レベル（インデントなし）の `const` 宣言を1事実ずつキー化する。値がオブジェクトリテラル（`{...}`）または `as const` オブジェクトの場合は、宣言行そのものを1件とはせず、直下の各フィールド（最上位1階層のみ。フィールド値がさらにオブジェクト/配列でもその内部までは再帰的に分解しない）を1事実ずつキー化する（`const-<定数名>-<フィールド名>`。value にはフィールドの実測値、evidence には当該フィールド行の file:line を記録する。宣言行の file:line ではない）。値がスカラー（文字列・数値・真偽値・単純な as const 配列等）の場合は宣言全体を1事実とする（`const-<定数名>-<値>`）。styled 定数（⑦）・状態変数（④）は除外する | 行頭 `^(export[ \t]+)?const[ \t]+[A-Za-z_][A-Za-z0-9_]*` に一致し、かつ同一行に `styled.`・`useState(`・`useReducer(`・`useRef(` を含まない宣言について、値がオブジェクトリテラルの場合はブレース深度1（最上位階層）で `key:` パターンに一致するフィールド行数を数える。値がスカラーの場合は宣言行を1件として数える |
| ④状態変数 | `useState`/`useReducer`/`useRef`/store参照の変数宣言を1事実ずつキー化する（`state-<変数名>-<初期値要約>`）。カスタムフック（`use<大文字>...`形の store参照フック等）への分割代入は分割代入された識別子ごとに1事実とする。`useParams`/`useLocation` のようなルーター系フックの戻り値も同じ扱いとする | `useState`/`useReducer`/`useRef` の直接呼出しは1呼出し=1件（正規表現: `(useState\|useReducer\|useRef)(<[^>]*>)?\(`）。それ以外の `use<大文字>...(` 形カスタムフックへの `const {...} = useFoo(...)` / `const [...] = useFoo(...)` 分割代入（複数行可）は、分割代入された識別子の数を数える。単一変数代入（`const x = useFoo(...)`）は1件。フック呼出しに `useParams<{ id: string }>()` のようなジェネリック型注釈が付いていても、`(` に隣接する `<...>` を判定前に除去してから数えるため、分割代入・単一変数代入いずれの形でも通常どおり検知する |
| ⑤イベントハンドラ | ハンドラ名・発火要素・処理1行要約を1事実ずつキー化する（`handler-<ハンドラ名>-<要約>`） | JSX の `on<大文字始まり>=\{` パターン（例 `onClick={`）の出現数を数える |
| ⑥JSX構造 | コンポーネントのネスト構造のみを1事実ずつキー化する（属性値・実測レイアウトは書かない） | `<` に続く大文字始まり識別子で、直後が空白・`/`・`>`・行末のいずれかであるもの（JSX開始タグ相当。属性を複数行に折り返し行末がタグ名直後に来るケースを含む）をユニーク抽出し、ユニーク件数を数える |
| ⑦スタイル実測値 | styled 定数のプロパティ・数値・色を1事実ずつキー化する（`style-<定数名>-<プロパティ>-<値>`） | `= styled.` を含む行数を数える |
| ⑧API呼出 | BL呼び出し名・契機・リクエスト/レスポンス形を1事実ずつキー化する（`api-<BL名>-req`/`-res`）。await・Promiseチェーンいずれも伴わない直接呼出し（呼出し形式: レシーバ経由のドット連結 / レシーバを介さない裸の関数呼出し × 代入形式: 単純識別子への代入 / 波括弧の分割代入の2軸4パターン）も対象に含める | `await <識別子>(` パターンの出現数と `.then(` パターンの出現数を合算する（Promiseチェーン形式 `foo(...).then().catch().finally()` は起点の `.then(` 1件のみを数え、`.catch`/`.finally` は継続部として数えない）。さらに直接呼出し4パターン（呼出し形式×代入形式）を加算する |
| ⑨実測系（再計数対象外） | 初期表示値・DOM配置順・要素位置等を断定せず一覧化のみ行う（`初期表示-件数` 等）。key・evidence のみ記録し value は書かない | 独立再計数の対象外。空欄率検査（key・evidence）と孤児参照検査のみ本セクションにも適用する |

## ⑥JSX構造の採録規律（属性値・文言粒度）

⑥JSX構造の抽出粒度は「ネスト構造のみ」から以下の粒度へ拡張する。上表の Phase 2 抽出手順に追加で適用する。往復検証（rebuilding-code-from-docs との往復）で発生した NG（Header の `backTo` 値・装飾テキストの具体文言・見出し区画の欠落・props/className/アイコン種別の未採録）を踏まえた強化である。

- JSX 内の**全文字列リテラル**（表示文言・装飾テキスト・`aria-label`・`placeholder` 等）は、value そのまま（一字一句）を必須採録する
- コンポーネントへ渡す**全 props の具体値**（`backTo`・`to`・`state`・`icon`/`name`・`size`・`position`・`tone`・`rotate` 等、リテラルで書かれているものすべて）を必須採録する
- 見出し要素（`h1`〜`h6`）は「レベル＋文言＋その区画の存在」を1事実として採録する
- `className` はトークン列挙で採録する（DESIGN.md 転記の原資）
- 記号1文字（末尾の「›」等）も省略しない
- 要約・代表例に丸めない。再実装者が同一 DOM を再現できる粒度が合格基準

## ⑥JSX構造の採録規律（条件分岐の全レンダリングパス）

コンポーネントが早期return・三項演算子・`&&`短絡評価によって複数のレンダリング結果を生成する場合、各分岐（パス）が生成する最外殻（ルート）要素を、上記「採録規律」と同じ粒度（全文字列リテラル・全propsの具体値・className等）で**パスごとに独立した事実として**採録する。

- 対象となる分岐: `if (...) { return (...) }` 形の早期return、`cond ? (...) : (...)` 形の三項演算子、`cond && (...)` 形の短絡評価レンダリングのいずれかによって、コンポーネントが2種類以上のJSX木を返しうる場合
- 各パスのルート要素（そのパスが返すJSX木の最も外側のタグ）を、他のパスと要素名が重複していても1事実として個別にキー化する（`jsx-path-<パス識別子>-<ルート要素名>`。パス識別子は分岐条件を要約した意味語とする。例: `jsx-path-loading-div`・`jsx-path-default-Card`）
- ルート要素のみを記録して終わらせず、そのパス固有の外殻ラッパー（他パスと異なる className・style・sx 適用を含む）を本節冒頭の採録規律に従いすべて採録する
- 同一のタグ名が複数パスに登場すること（例: 全パスとも最外殻が `<div>`）を理由に1件へ丸めてはならない。パスの数だけ独立に記録する
- ネイティブHTML要素（`<div>`・`<span>`・`<input>` 等、小文字始まりのタグ）も PascalCase コンポーネントと同様に採録対象である（属性値・文言粒度の規律を含め除外しない）

## メタ節（meta）の採録手順

9分類の抽出とは別に、`facts.yml` の `meta` 節（`shared/references/facts-schema.md` 準拠）を以下の手順で記録する。

- `meta.source_repo`: `target_repo_path` の値をそのまま転記する
- `meta.source_ref`: `git -C <target_repo_path> rev-parse HEAD` を実行し、その標準出力（コミットSHA）を転記する
- `meta.route`: `target_file_paths` に含めたルーター定義ファイルを grep で検索し、当該画面のルートパスを特定して `value` に記録する。根拠は特定に使った行を `file:line` 形式で `evidence` に記録する

## 再計数パターンの既知の限界

- 全パターンは正規表現ベースの近似計数であり、真の構文解析（AST）ではない。TypeScript のジェネリクス表記が JSX 開始タグと字面上区別しにくいケースがあるため、⑥JSX構造のパターンは「識別子の直後が空白・`/`・`>`・行末のいずれか」に限定して汎用型引数（例 `useState<ReportRow[]>`）を除外している。それでも union 型・ネストしたジェネリクスなど字面上の偶然一致が起こりうる。特に `<CapitalName` の形が行末に単独で来る多行ジェネリクス（例: `useFoo<Bar\n>(...)`）は理論上誤検知しうるが、通常の整形（prettier等）では開き `<` と型名が別行になる書式が一般的なため実害は限定的
- ①importの複数行 named import（開き `{` を含み `}` を含まない行から、`from` を含む終端行まで）は継続行として連結して追跡し、連結後の内容をシンボル単位で計数する（対応済み）。②export・型のカンマ分割は複数行にまたがる interface 宣言（開き括弧と閉じ括弧が別行）を想定していない点は限界として残る。対象ファイルは interface 宣言を1行に収める整形（prettier等）が前提
- ④状態変数のカスタムフック分割代入検知は、分割代入内のデフォルト値・リネームにカンマを含む式（例 `{ x = calc(1, 2) }`）を想定していない。単純なカンマ区切りの識別子列（リネーム `a: b`・デフォルト値 `a = 1` は識別子部分のみ抽出）を前提とする
- 乖離許容5%は近似計数と手動キー付けの粒度差を吸収するための閾値であり、0%一致を保証する設計ではない。乖離が閾値を超える場合は Phase 2 のキー付け粒度（特にオブジェクト/enum型の分解単位）を見直す
- **乖離の原因が「粒度差」ではなく「対象コードに実在する構文パターンをrecount-facts.shの正規表現が構造的に検知できないこと」である場合**（例: 記載1件に対し再計数が常に0件のまま増減しない）、それはPhase 2実行者の抽出粒度の問題ではなく本パターン自体の欠陥である。この場合は items を reason へ逃がして回避してはならない（facts-schema.md の「該当なし」は実在しない事実にのみ使う）。本ファイルの表とパターンを実在の構文に合わせて修正し、`scripts/recount-facts.sh` の対応する count_* 関数を拡張してから再実行する。新たな構造的盲点を発見した場合は本節へ追記する
- `count_jsx`の`return_jsx_count`はファイル全体（複数コンポーネントを連結した場合は複数関数分）を対象にした近似であり、関数境界を認識しない。1ファイル1コンポーネントを前提とする
- 三項演算子検出の`?`は、TypeScriptの省略可能プロパティ注記（`prop?: string`）やオプショナルチェイニング（`?.`）との字面上の区別が完全ではない。直前文字が`.`でないことのみを排除条件にしている
- 小文字タグ対応（`count_jsx`の対象文字クラス拡張）により、稀な比較式の字面一致等が理論上誤検知しうるが、既存のPascalCase系の限界と同種の許容範囲内とする
- ⑧API呼出の`await <識別子>(`/`.then(`は、await・Promiseチェーンいずれも伴わない直接呼出し（代入形 `const x = <レシーバ>.<メソッド>(...)` / 文頭ステートメント形 `<レシーバ>.<メソッド>(...)`）を構造的に検知できなかった。この直接呼出し形を追加で検知し、`await`/`.then`該当行は重複計上防止のため直接呼出し側の対象から除外する。組込みオブジェクト（console・Math・JSON等）へのレシーバ、配列・文字列の汎用メソッド（map・filter・replace等）への呼出しは業務ロジック呼出しではないため除外リスト方式で計上対象から外す。この除外リストは一般的な語彙のみを列挙する近似であり、除外リストに含まれない汎用メソッド名や、JSX式内に埋め込まれた呼出し（行頭アンカーの対象外）は引き続き検知できない残存限界がある
- 直接呼出しはさらに「呼出し形式（レシーバ経由のドット連結 / レシーバを介さない裸の関数呼出し）」×「代入形式（単純識別子への代入 / 波括弧の分割代入）」の2軸4パターンとして検知する。裸の関数呼出しは英字接頭辞+数字のモジュール名規約（例: `bl1DoSomething`・`api2FetchX`）を持つ名前のみを業務ロジック呼出しとみなし、数字を含まない一般語彙（ユーティリティ関数・制御構文・フック等）は誤検知防止のため対象外とする。フック呼出し（`use`大文字…）とTypedArrayコンストラクタは明示的に除外する
