プロジェクト共通採録スキル
compiling-project-common-docs

アーキテクチャ調査書を前提に、対象リポジトリのコードから層化サンプリングでプロジェクト共通7文書のv0を採録する。実装済みコードに現に存在する規約・共通仕様の事実だけを記録し、理想論は書かない。

こんな人向け
アーキテクチャ調査書が確定した直後、対象リポジトリに実在する規約・共通仕様の事実だけを記録したv0を採録したい人。
こんな場面で使う
後続工程（facts抽出・詳細設計執筆）の前提として、実例・頻度・例外率の三点セットで裏付けられたプロジェクト共通文書が必要なとき。判定でNG帰着(c)（共通文書欠落）が検出され、該当文書へ追記したいとき（mode=append）。
動作範囲対象リポジトリは読み取り専用。出力は docs_root 側のプロジェクト共通8ファイルのみ
対話AskUserQuestion不使用。args全量指定・対話ゼロで完走する
依存find / sort / head / grep 等の決定的コマンド、機械ゲート実行に scripts/check-common-docs.sh を使用
関連資料SKILL.md, scripts/check-common-docs.sh, references/sampling-rules.md

§1概要

対象リポジトリを層化サンプリングで走査し、その結果をプロジェクト共通8ファイル（規約4種＋共通設計書.md＋メッセージ定義書.md＋DESIGN.md＋サンプル記録.md）として確定する。

単独動作: 他スキルへの依存を持たない。起動引数を渡せば単独で動く。工程全体は orchestrating-reverse-docs-flow が案内する。後続工程（facts抽出・詳細設計執筆）はこのプロジェクト共通文書を前提として動く。
読み取り専用: 対象リポジトリへの書き込み・変更は一切行わない。出力は docs_root 配下の8ファイルのみ。
v0で止める: 完成を狙わない。v0確定後の追記は mode=append の別起動でのみ行う。

§2用語

層化サンプリング
アーキテクチャ調査書のディレクトリ責務マップの行を層とし、層ごとに find / sort / head の決定的コマンドでサンプルファイル集合を確定する手法。乱数・目視選定は禁止する。共有ファイル型の層（責務マップの行が「共有ファイル」で始まる場合）はN=1・k=1固定として扱う。詳細は references/sampling-rules.md。
三点セット
各規則行に必須の3要素。①実例（対象リポジトリ内の相対パス3件以上）②頻度（N/M形式）③例外率（%表記）。この3要素が揃わない規則は書かない。
機械ゲート
check-common-docs.sh による決定的スクリプト検証。合否判定は exit code のみで行い、自然文の自己申告では判定しない。
mode=append
NG帰着(c)（共通文書欠落）の追記モード。差し戻し元が指摘した欠落挙動・欠落文書（append_findings）を受け取り、該当文書のみ追記して全ゲートを再実行する。

§3使い方（起動引数）

全量指定・対話ゼロ。単独起動時は下表の args をユーザーから直接取得する。

引数必須内容
target_repo_path必須対象リポジトリの絶対パス
docs_root必須出力先ルート。7文書は <docs_root>/プロジェクト共通/ 配下に出力する
template_root必須テンプレ一式のルート。<template_root>/プロジェクト共通/ の既存テンプレ7文書を雛形に使う
survey_doc_path必須アーキテクチャ調査書のパス（ディレクトリ責務マップを層化サンプリングの層定義に使う）
mode任意（既定 v0）v0（新規）／append（NG帰着(c)の追記。append_findings を受け取り該当文書へ追記して全ゲートを再実行）
append_findingsmode=append 時のみ必須差し戻し元が指摘した欠落挙動・欠落文書の一覧

§4設計原則

原則内容
読み取り専用対象リポジトリへの書き込み・変更は一切行わない。出力は docs_root 配下のみ
実装事実主義サンプルに現れない規則を発明しない。理想論（あるべき姿）を書かず、実装済みコードに現に存在する事実だけを記録する
三点セット必須各規則行は実例（3件以上）・頻度・例外率を必ず添える
決定的サンプリング層化サンプリングの選定は find / sort / head の決定的コマンドに固定する。乱数・目視選定を禁止する
v0で止める完成を狙わない。v0確定後の追記は mode=append の別起動でのみ行う
合格判定はスクリプトのexit codeのみ自然文の自己申告での合格判定を行わない

§5動作フロー（Phase 1〜6）

Phase 1: 前提確認
  target_repo_path・template_root・survey_doc_pathの実在を確認する
  調査書内「ユニット種別判定」節と「ディレクトリ責務マップ」節の
  実在をgrepで確認する(調査書ゲートの再実行はしない)
  docs_root/プロジェクト共通/規約/を作成しテンプレ7文書を複製する
  完了条件: テンプレ7文書複製済み・調査書の2節実在確認済み
   (mode=append時は既存8文書と指摘文書の特定済み)

Phase 2: 層化サンプリング
  調査書のディレクトリ責務マップの各層ごとに
  find <層> -type f | sort | head -n  で決定的に確定する
  (kは層あたり3〜10。全層合計20ファイル以上)
  選定コマンドと結果一覧をサンプル記録.mdに記録する
  完了条件: サンプル記録.mdに全層のコマンドと選定ファイル一覧が
   記録済み(合計20ファイル以上)

Phase 3: 規約4種採録
  コーディング規約・命名規約・ディレクトリ構成規約・
  コンポーネント設計規約の各規則を採録する
  各規則行に①実例(相対パス3件以上) ②頻度(N/M形式)
  ③例外率(%表記)を記載する。サンプルに無い規則は発明しない
  完了条件: 全規則行が実例3件以上+頻度+例外率付き

Phase 4: 共通文書採録
  共通設計書.md(アーキテクチャ・共通処理・状態管理・通信)
  メッセージ定義書.md(実メッセージ文字列と定義箇所)
  DESIGN.md(実スタイル値)を同じ実例主義で採録する
  完了条件: 3文書のプレースホルダ残存ゼロ

Phase 5: 機械ゲート
  check-common-docs.sh /プロジェクト共通
    を実行
  FAILしたらPhase 3〜4に戻り、指摘された実例不足・頻度欠落・
  例外率欠落・未実在パス・テンプレ残存・理想論表現を修正して
  再実行する(上限5回)
  完了条件: ゲート exit 0

Phase 6: 返却
  返却ブロックを出力する
  mode=v0: status=採録v0確定。artifacts/scope/
   common_docs_root/sample_manifest_pathを記す
  mode=append: 指摘文書のみ追記し全ゲート再実行後、
   status=追記完了を返す
  上限到達で未収束: status=中断。hintに未解消の検査項目を記録
  完了条件: statusが確定している

§6機械ゲート5検査

check-common-docs.shは合否を exit code のみで判定する（自然文の自己申告は用いない）。

検査内容
実在検査プロジェクト共通8ファイル（規約4種＋共通設計書.md＋メッセージ定義書.md＋DESIGN.md＋サンプル記録.md）すべてが実在するかを確認する
規則行完備性検査規約4文書内の各テーブル行のうち、backtick囲みの相対パス（「/」を含む）トークンを1件以上含む行を「規則行」とみなし、その行に実例パス3件以上・頻度（[0-9]+/[0-9]+形式）・例外率（[0-9.]+%形式）が揃っているかを確認する
パス実在検査規約4文書内のbacktick囲み相対パス全件が target_repo_path 配下に実在するかを確認する。URL・glob・プレースホルダ・絶対パス・空白や正規表現記号を含むトークンは対象外
テンプレ残存検査プレースホルダ（<実測・<FILL・TBD・TODO）の残存ゼロを8ファイルすべてで確認する
理想論表現検査規約4文書に「すべきである」「望ましい」「べきだ」「理想的には」「今後は」等の理想論表現が無いかを確認する（実装事実の記録に限る）

§7返却ブロック

orchestrating-reverse-docs-flow の契約（返却ブロック共通サブセット: status/scope/artifacts/hint）に準拠する。

キー値
status採録v0確定 ／ 追記完了 ／ 中断
scopetarget_repo_path のbasename
artifacts8文書の絶対パス（配列）
hint次工程への申し送り、または中断理由
common_docs_root（拡張）プロジェクト共通/ の絶対パス
sample_manifest_path（拡張）サンプル記録.mdの絶対パス

§8ループ設計

要素内容
反復対象Phase 5（機械ゲート）exit 1 → Phase 3〜4（規約4種採録・共通文書採録）へ戻る
上限回数5回
収束条件check-common-docs.sh が exit 0
発散条件同一のNG理由（同一検査項目の同一違反）が2回連続で再発した場合、発散として即中断する（上限回数消化前でも中断してよい）
上限到達時の報告status=中断とし、hintに未解消の検査項目・違反内容・試行回数を記録する
検証役の分離合否判定は check-common-docs.sh の exit code のみで行う。自然文の自己申告での合格判定は行わない

§9Gotchas・重要な注意事項

  • 機械ゲートの規則行完備性検査（検査2）は、規約4文書の各テーブル行のうちbacktick囲みの相対パス（「/」を含む）トークンを1件以上含む行だけを「規則行」とみなす。見出し行・区切り行はbacktickトークンを含まないため自動的に対象外になるが、意図的に規則行だけへ実例・頻度・例外率を書くこと
  • 頻度は N/M 形式（例: 18/20）、例外率は % を含む数値表記（例: 10.0%）で、backtickの外（平文）に書く。backtick内に閉じ込めると相対パスとしても誤認識されうる
  • 記載パスの実在チェック（機械ゲート検査3）は、backtickで囲んだ「/」を含む相対パスのみを対象とする。URL（://）・glob（*/?）・プレースホルダ（</>）・絶対パス（先頭/）・空白や正規表現記号を含むトークン（コマンド例・grepパターン）は対象外
  • 理想論表現検査（検査5）は規約4文書のみを走査する。共通設計書.md・メッセージ定義書.md・DESIGN.mdは検査対象外だが、実装事実主義の原則自体はすべての文書に適用する
  • テンプレ残存検査（検査4）は8ファイルすべてを走査する。テンプレの固定文にも検出正規表現に一致する語を書いてはならない
  • mode=append は指摘文書のみ追記すればよいが、機械ゲートは全項目を再実行する。部分ゲートは存在しない
  • 発散判定（同一NG理由2連続）は上限5回を消化する前でも即中断する
  • 層あたりのk値は層内ファイル数の平方根以上・3以上10以下に丸める。全層合計20ファイル未満だとPhase 2の完了条件を満たさない。詳細は references/sampling-rules.md
  • 対象リポジトリに対しては読み取り専用。書き込み・変更は一切行わない
  • サンプルに現れない規則を発明しない。実例3件以上・頻度・例外率が揃わない規則は書かない
  • AskUserQuestionを使わない。args全量指定・対話ゼロで完走する
  • SKILL.md本文にプロジェクト固有値（リポジトリ名・画面名・絶対パス・ユーザー名）を一切書かない。固有値はすべて起動argsで受ける
