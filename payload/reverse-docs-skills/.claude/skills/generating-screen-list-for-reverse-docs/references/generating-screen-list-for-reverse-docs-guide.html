画面一覧生成スキル
generating-screen-list-for-reverse-docs

既存コードベースを調査して画面単位にファイルをグルーピングし、画面詳細設計書の単位を正確に分けるための正本「画面一覧.html」を生成するスキルの確定仕様。画面種別（unit_kind=screen）専用。

対応 OSWindows WSL2 環境
検証状況統合スキル generating-unit-list-for-reverse-docs の画面（unit_kind=screen）経路として実プロジェクトで動作実績あり。種別分割後の単独起動は分割時点では未検証
依存jq / bash（shared/scripts/unit-list/ 配下の validate-manifest.sh・build-unit-list.sh・detect-screens.sh が使用）
関連資料SKILL.md, references/screen-detection.md, ../../../shared/scripts/unit-list/ 配下の各スクリプト

§1概要

リバース設計で最初に決めるべきは「設計書を何冊に分けるか」＝画面の境界である。本スキルは既存コードベースを、スタック調査 → 検出戦略の宣言 → 戦略に基づく抽出 → 整合検証 の順で調査し、画面単位にファイルをグルーピングした「画面一覧.html」を生成する。この HTML が画面詳細設計書の単位を分けるための正本になる。

仕事は画面一覧.html の作成のみで、設計書の雛形展開・生成・記入は一切行わない。工程全体の案内は orchestrating-reverse-docs-flow が担い、本スキルは単独でも起動できる。

── リバース設計工程における位置づけ ─────────────────────────

  既存コードベース
        │
        ▼
┌───────────────────────────┐
│ 本スキル                   │  Phase 1  スタック・画面規約の特定
│ generating-screen-list-   │  Phase 2  戦略に基づく抽出
│ for-reverse-docs          │  Phase 3  整合検証（機械実行）
│                           │  Phase 4  画面一覧.html 生成
└─────────────┬─────────────┘
              ▼
      画面一覧/画面一覧.html ＝ 設計書の単位を分ける正本
              │
              ▼
      画面詳細設計書の執筆工程（別スキルが担当）
設計原則: 固定と可変の分離。 マニフェストの形式・整合検証・HTML 生成は決定的スクリプトに固定し、プロジェクトごとに異なる「画面境界の検出」だけを可変にする。どの経路で抽出しても、同じ検証（validate-manifest.sh）を通らなければ HTML は生成できない。

§2出力

スキルを完走すると、以下の 2 点が生まれる。

画面一覧.html（<output_dir>/画面一覧/画面一覧.html）
対象コードベースに存在する画面の全数を、画面キー・推定画面名・入口ファイル・構成ファイル群・検出根拠とともに一覧化した文書。読んだ人は「このシステムには画面が何枚あり、それぞれどのファイル群でできているか」を根拠付きで把握でき、画面詳細設計書を何冊書くべきかを迷わず決められる。共有部品・埋め込みビュー・未解決候補・診断警告も可視化される。
検証済みマニフェスト（HTML 内埋め込み JSON）
画面一覧の元データとなる機械可読の JSON。7 項目の整合検証を通過したものだけが HTML に埋め込まれるため、後続工程（設計書執筆・往復検証）はこの JSON を信頼して機械処理できる。配列キーは screens（後方互換）。

§3用語

画面（screen）
本スキルが数える単位。ルーティング定義に対応する route 行と、View 切替で表示される embedded-view 行のみを画面として数える。共有クラスタで参照されるだけの部品ファイルは含めない。
マニフェスト
抽出結果を保持する JSON。画面ごとに screenKey（意味キー）・entryFile・files・confidence 等を持つ。スキーマの全体は references/screen-detection.md を参照。
検出戦略宣言
Phase 1 の調査結果を JSON（unitKind/extractionMethod/screenIdRegex/viewSwitchPattern/excludePatterns 等）にまとめ、ユーザー承認（approvedByUser: true）を得た宣言。抽出と検証の両方がこの宣言を参照する。
組み込み検出器
detect-screens.sh。Next.js App/Pages Router・React Router（useRoutes 含む）・慣習ディレクトリ（pages/screens/views）を機械的に検出する高速パス。
カスタム抽出パス
組み込み検出器が適合しないプロジェクトで、Claude 自身が戦略宣言に沿って専用の抽出手順を設計・実行し、スキーマ準拠のマニフェストを出力する経路。
埋め込みビュー（embedded-view）
ルーティングを持たず View 切替関数（モーダル・タブ等）で表示される画面。Phase 1 で view-switch-pattern を宣言した場合のみ検出する。

§4使い方（起動引数と返却）

起動引数（2 つのみ）

引数内容
source_dir探索対象のソースコードディレクトリ。対象プロジェクトの実コードルート（例: frontend/src）を指定する。モノレポでアプリが複数ある場合は対象アプリのディレクトリのみを渡す（自動判別なし）
output_dir画面一覧.html の書き出し先。この直下に 画面一覧/ フォルダを作成する

旧統合スキルにあった unit_kind 引数は存在しない。本スキルは screen 固定であり、他種別（API・テーブル・バッチ・帳票・外部連携）は対応する種別別一覧スキルを使う。

返却（orchestrating-reverse-docs-flow の契約準拠）

返却キー内容
statusDONE または ERROR
artifacts生成した画面一覧.html のパス。artifacts[0] を汎用名 unit_list_html としても返す
screen_list_html従来互換のエイリアス（unit_list_html と同値）
embedded_json_refHTML 内に埋め込んだマニフェスト JSON への参照
unit_kind固定値 screen

§5動作フロー（Phase 1〜4）

開始時に TaskCreate で Phase 1〜4 のタスクを登録し、Phase の開始・完了ごとに TaskUpdate で進捗を更新する。

Phaseやること完了条件
Phase 1
スタック・画面規約の特定		 package.json・lock ファイルからフレームワーク・ルーターライブラリを確定 → ルーティング定義の所在を追跡 → 画面 ID 命名パターン・View 切替関数を調査 → 除外パターンを確定 → 検出戦略宣言を作成しユーザー承認を取る 戦略 JSON（approvedByUser: true を含む）が保存済み
Phase 2
戦略に基づく抽出		 組み込みパスかカスタム抽出パスかを分岐判定 → 選んだ経路でマニフェスト JSON を生成 → diagnostics（entryFile 集中警告等）を確認 スキーマ準拠のマニフェストが 1 件以上確定。0 件検出はハード停止（画面を捏造しない）
Phase 3
整合検証（機械実行）		 validate-manifest.sh <manifest.json> --unit-kind screen を実行し、7 項目検証を全 PASS させる。FAIL 時は修正して再実行（3 回失敗で Phase 2 の抽出方式再検討へ差し戻し） exit 0（全項目 PASS）
Phase 4
画面一覧.html 生成		 build-unit-list.sh <manifest.json> <output_dir>/画面一覧/画面一覧.html --unit-kind screen を実行する（内部で build-screen-list.sh に委譲）。build 側が validate を再実行するため、未検証マニフェストからは生成できない HTML 生成済み・埋め込み JSON がマニフェストと一致

エンジンスクリプトはいずれもスキルフォルダ相対の ../../../shared/scripts/unit-list/ 配下にある。正本リポジトリと公開先はフォルダ配置が同一のため、どちらの環境でも同じ相対参照で動く。

§6検出の 2 経路

経路使える条件担い手
組み込み検出器（高速パス） Next.js App/Pages Router（next.config.* の実在必須）・React Router（useRoutes 含む）・慣習ディレクトリのいずれかが Phase 1 の調査結果と適合する場合 detect-screens.sh（決定的スクリプト）
カスタム抽出パス 組み込み検出器が適合しない場合（カスタムルート配列・element 属性解決・動的ルート構築・深い階層の View 切替等） Claude 自身が戦略宣言に沿って Bash/Grep/Read で抽出し、スキーマ準拠 JSON を Write する
どちらの経路でも検証は同一基準。 validate-manifest.sh は抽出者非依存で動き、Phase 1 で承認した戦略宣言の存在確認も担う。カスタム抽出だからといって検証を省略する経路は存在しない。

組み込み検出器の既知の限界（カスタム抽出パスへの切替の目安）

  • 動的に構築されるルート文字列（変数結合等）は検出できない。静的リテラルの path のみが対象
  • import Foo, { Bar } from ...（default+named 混合 import）の解決は不完全
  • import 追跡は 2 段階まで。埋め込みビューのスキャンは 1 階層のみで、子コンポーネント内のさらなる View 切替は検出できない
  • diagnostics で entryFile 集中警告が出たら、組み込みパスの継続ではなくカスタム抽出パスへの切替を検討する

§7禁止事項と注意点

禁止・注意理由
手作業でのプレースホルダ置換の禁止。HTML 生成は必ず build-unit-list.sh 経由で行う過去に手動置換で entryFile=None が 10 件混入した実害がある。決定的スクリプトに固定して手作業経路を根絶する
0 件検出時のハード停止。AskUserQuestion で手動リストを聞き出さない誤った画面境界を即興で確定させないため。0 件は戦略の不適合を示す信号であり、Phase 1 へ戻って調査し直す
設計書の雛形展開・生成・記入はスコープ外本スキルの成果物は画面一覧.html のみ。設計書は後続工程の別スキルが担当する
Phase 1 の調査を省略して汎用の screen-id-regex を当てない命名規約・ナビゲーション方式はプロジェクトごとに異なり、汎用正規表現は境界の取り違えを生む
カスタム抽出時はコメントアウトされた定義を除去してから抽出するコメント内のルート定義・import 文を実在として誤検出した実害がある
jq 未インストール環境では事前に導入する検証・生成スクリプトが jq に依存する
