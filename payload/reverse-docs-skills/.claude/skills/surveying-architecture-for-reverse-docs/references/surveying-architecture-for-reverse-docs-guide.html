アーキテクチャ調査スキル
surveying-architecture-for-reverse-docs

既存リポジトリを走査し、アーキテクチャ調査書を機械検証付きで確定する。仕事はアーキテクチャ調査書.mdの確定のみで、対象リポジトリへの書き込み・変更は一切行わない。

こんな人向け
リバース設計に着手する前に、対象リポジトリの技術スタック・ディレクトリ責務・ユニット種別の実在有無を確定したい人。
こんな場面で使う
後続工程（共通文書採録・facts抽出・詳細設計執筆）の前提として、決定的な根拠に基づくアーキテクチャ調査書が必要なとき。既存の調査書に検出手がかりの欠落が見つかり、該当節のみ改訂したいとき（mode=revise）。
対応 OSWindows WSL2 環境
検証状況サンプルプロジェクトで動作確認済み
依存find / grep / ls 等の決定的コマンド、機械ゲート実行に scripts/check-architecture-survey.sh を使用
関連資料SKILL.md, scripts/check-architecture-survey.sh

§1概要

既存リポジトリを決定的コマンドで走査し、その結果をアーキテクチャ調査書.mdとして確定する。

単独動作: 他スキルへの依存を持たない。起動引数を渡せば単独で動く。工程全体は orchestrating-reverse-docs-flow が案内する。後続工程（共通採録・facts抽出・詳細設計執筆）はこの調査書を前提として動く。
読み取り専用: 対象リポジトリへの書き込み・変更は一切行わない。出力は docs_root 配下の1ファイル（アーキテクチャ調査書.md）のみ。

§2出力

リバース設計の全工程が前提として参照する、プロジェクトの技術的な地図が 1 本生まれる。

アーキテクチャ調査書
対象アプリケーションの技術スタック・ディレクトリ構成・ルーティング方式・状態管理パターン・API 通信手法・認証方式・ビルド設定を調査し、体系的にまとめた文書。後続の共通文書採録・facts 抽出・設計書執筆の全工程が、この調査書の記述を前提として作業する。

§3用語

ユニット6種別
画面・API・テーブル・バッチ・帳票・外部連携の6区分。各種別について実在有無を決定的コマンドで判定する。
機械ゲート
check-architecture-survey.shによる決定的スクリプト検証。合否判定は exit code のみで行い、自然文の自己申告では判定しない。
mode=revise
差し戻し改訂モード。差し戻し元が指摘した欠落節・欠落理由（revise_findings）を受け取り、該当節のみ改訂して全ゲートを再実行する。

§4使い方（起動引数）

全量指定・対話ゼロ。単独起動時は下表の args をユーザーから直接取得する。

引数必須内容
target_repo_path必須調査対象リポジトリの絶対パス
docs_root必須出力先ルート。調査書は <docs_root>/プロジェクト共通/アーキテクチャ調査書.md に出力する
template_root必須テンプレ一式のルート。この配下の プロジェクト共通/アーキテクチャ調査書.md を雛形に使う
target_branch任意調査時点の対象ブランチ。調査メタ節に記録する
source_ref任意調査時点のコミット参照。調査メタ節に記録する
mode任意（既定 survey）survey（新規）／revise（差し戻し改訂。revise_findings を受け取り該当節のみ改訂して全ゲートを再実行）
revise_findingsmode=revise 時のみ必須差し戻し元が指摘した欠落節・欠落理由の一覧

§5設計原則

原則内容
読み取り専用対象リポジトリへの書き込み・変更は一切行わない。出力は docs_root 配下のみ
決定的根拠のみ全記述は find / grep / ls 等の決定的コマンドの実出力を根拠とする。推測での埋め合わせを禁止する
空欄禁止・理由必須判定不能・実在しない項目は空欄にせず「実在しない（理由: …）」で必ず埋める
固定と可変の分離ゲート（check-architecture-survey.sh）は決定的スクリプトに固定する。調査そのもの（何をどう調べるか）はプロジェクトごとに可変であり、Phase 2〜3 で都度 Bash/Grep/Glob を用いて実行する
合格判定はスクリプトのexit codeのみ自然文の自己申告での合格判定を行わない

§6動作フロー（Phase 1〜6）

Phase 1: 前提確認
  target_repo_path・template_rootの実在を確認する(test -d)
  docs_rootに「プロジェクト共通/」が無ければ作成する
  mode=revise: 既存調査書の実在確認+指摘節の洗い出し
  mode=survey: テンプレを調査書の出力先へ複製する
  完了条件: 全args解決済み・実在確認済み・ディレクトリ作成済み
   (mode=revise時は既存調査書と指摘節の特定済み)

Phase 2: 決定的走査
  find/grep/lsで対象リポジトリを走査する
  ビルド設定(実在するもののみ)・起動コマンド・エントリポイント・
  ルーティング定義・依存関係を決定的出力のみを根拠に列挙する
  実行した調査コマンドは全て調査メタ節に記録する(再現可能性の担保)
  併せてgrep -rl等で複数ディレクトリから参照される共有ファイル
  (共通スタイル定数・テーマ定義ファイル等)の有無を確認し、
  該当があれば§4ディレクトリ責務マップへ「共有ファイル」行として追記する
  (該当が無ければ追記しない)
  完了条件: ディレクトリ責務マップとエントリポイント一覧が
   根拠パス付きで揃っている(共有ファイルが実在する場合はその行も追記済み)

Phase 3: ユニット6種別判定
  画面・API・テーブル・バッチ・帳票・外部連携の6種別それぞれについて
  決定的コマンドで実在を判定する
  各種別は「実在する」または「実在しない(理由: …)」のいずれかで記録
  検出に使った再現可能なgrep/findパターンを検出手がかり列に記録する
  完了条件: 6種別すべてに判定行がある(種別名と判定語が同一行)

Phase 4: 調査書執筆
  テンプレの全節(調査メタ・技術スタック・ビルドと起動・
  ディレクトリ責務マップ・エントリポイントとルーティング・
  ユニット種別判定・判定不能と実在しない項目の理由・
  後続工程への申し送り)を実測値の転記で埋める
  mode=revise: revise_findingsが指摘した節のみ改訂すればよいが、
   次のPhase 5ゲートは全項目を再実行する(部分ゲートは無い)
  完了条件: プレースホルダの残存ゼロ

Phase 5: 機械ゲート
  check-architecture-survey.sh <調査書>  を実行
  FAILしたらPhase 4に戻り、指摘された未実在パス・未判定種別・
  推測語・テンプレ残存を修正して再実行する(上限5回)
  完了条件: ゲート exit 0

Phase 6: 返却
  返却ブロックを出力する
  status=調査確定: artifacts/scope/survey_doc_path/
   unit_kinds_presentを記す。hintに後続工程への申し送りを記す
  status=中断: hintにPhase 5で未収束のまま残った検査項目・理由を記す
  完了条件: statusが確定している

§7機械ゲート5検査

check-architecture-survey.shは合否を exit code のみで判定する（自然文の自己申告は用いない）。

検査内容
パス実在検査調査書に記載された相対パス（backtick囲みで「/」を含むもの）の実在を確認する。URL・glob・プレースホルダ・絶対パス・空白や正規表現記号を含むトークンは対象外
種別網羅検査ユニット6種別すべてに判定行（種別名と判定語「実在する」または「実在しない（」が同一行）が存在するかを確認する
推測語検査調査書ファイル全体（テンプレ由来の固定文・HTMLコメントを含む）を走査し、「おそらく」「と思われ」「かもしれ」「推測」「たぶん」「恐らく」「でしょう」「のはず」等の推測語の不在を確認する
テンプレ残存検査プレースホルダ（<実測: …>等）の残存ゼロを確認する
ディレクトリ網羅検査§1のfindコマンドから走査範囲を決定的に再現し、直接ファイルを持つディレクトリがすべて§4（責務マップ本体・対象外ディレクトリ）またはその子ディレクトリとして網羅されていることを確認する。§4の.（プロジェクトルート）行が網羅済みとするのはルート直下の直接ファイルのみで、子ディレクトリへは波及しない

§8返却ブロック

orchestrating-reverse-docs-flow の契約（返却ブロック共通サブセット: status/scope/artifacts/hint）に準拠する。

キー値
status調査確定 または 中断
scopetarget_repo_path のbasename
artifacts調査書の絶対パス（配列）
hint次工程への申し送り、または中断理由
survey_doc_path（拡張）調査書の絶対パス（artifacts[0] と同値）
unit_kinds_present（拡張）実在判定が「実在する」だった種別の一覧

§9ループ設計

要素内容
反復対象Phase 5（機械ゲート）exit 1 → Phase 4（調査書執筆）へ戻る
上限回数5回
収束条件check-architecture-survey.sh が exit 0
発散条件同一のNG理由（同一検査項目の同一違反）が2回連続で再発した場合、発散として即中断する（上限回数消化前でも中断してよい）
上限到達時の報告status=中断とし、hintに未解消の検査項目・違反内容・試行回数を記録する
検証役の分離合否判定は check-architecture-survey.sh の exit code のみで行う。自然文の自己申告での合格判定は行わない

§10Gotchas・重要な注意事項

  • 記載パスの実在チェック（機械ゲート検査1）は、backtickで囲んだ「/」を含む相対パスのみを対象とする。URL（://）・glob（*/?）・プレースホルダ（</>）・絶対パス（先頭/）・空白や正規表現記号を含むトークン（コマンド例・grepパターン）は対象外
  • 6種別の判定行は、種別名と判定語（「実在する」または「実在しない（」）が同一行に存在しないと機械検知されない。表のセルを分けて書くと検知漏れになる
  • mode=revise は指摘節のみ改訂すればよいが、機械ゲートは全項目を再実行する。部分ゲートは存在しない
  • 発散判定（同一NG理由2連続）は上限5回を消化する前でも即中断する
  • 調査書のfrontmatter updated はテンプレのプレースホルダ（<実測: YYYY-MM-DD>）のまま残してはならない。確定日を実測で記入する
  • ビルド設定・エントリポイントの列挙は「実在するもののみ」を書く。対象スタックに存在しないファイル種別を推測で書かない
  • 機械ゲートの推測語検査（検査3）は調査書ファイル全体（テンプレ由来の固定文・HTMLコメントを含む）を走査する。テンプレの固定文にも検出正規表現に一致する語を書いてはならない（テンプレ改訂時の注意）
  • 対象リポジトリに対しては読み取り専用。書き込み・変更は一切行わない
  • 出力は docs_root 側のみ（アーキテクチャ調査書.md 1本）。対象リポジトリ側には何も生成しない
  • 推測禁止。全記述は決定的コマンド（find/grep/ls）の実出力を根拠とする
  • 6種別は実在しない場合も「実在しない（理由: …）」で必ず埋める。空欄・省略は禁止する
  • AskUserQuestionを使わない。args全量指定・対話ゼロで完走する
  • SKILL.md本文にプロジェクト固有値（リポジトリ名・画面名・絶対パス・ユーザー名）を一切書かない。固有値はすべて起動argsで受ける
  • ディレクトリ網羅性検査（機械ゲート検査5）は、§1「実行した調査コマンド一覧」に記録されたfindコマンドのパスと-maxdepthを再利用して走査範囲を決定する。§1にfindコマンドが未記載だと検査5は即FAILになる
  • ディレクトリ網羅性検査は「直接ファイルを持つディレクトリ」のみを対象とする。ファイルを持たない中間ディレクトリ（例: src/ 配下に直接ファイルが無く子ディレクトリのみ）は検査対象外
  • §4に「共有ファイル」型で記載されたファイルパス（例: src/styles/shared-theme.ts）は、その親ディレクトリ（src/styles）を網羅済みとみなす
  • §4の層の子ディレクトリは親層に包含されるため個別に列挙する必要はない（例: §4に src/app があれば src/app/api は自動的に網羅済み）
  • 意図的にサンプリング対象外とするディレクトリは§4内の「対象外ディレクトリ」サブセクションに除外理由付きで記録する。対象外ディレクトリが無い場合はサブセクションごと削除する（空テーブルやプレースホルダを残すと検査4に抵触する）
  • §4の.（プロジェクトルート）行が網羅済みとするのはルート直下の直接ファイルのみ。子ディレクトリは.行では網羅されず、個別行・層の親行・対象外ディレクトリ表のいずれかへの記載が必要
