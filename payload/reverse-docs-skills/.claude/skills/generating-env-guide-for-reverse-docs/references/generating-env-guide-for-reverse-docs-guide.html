環境実行手順ページ生成スキル
generating-env-guide-for-reverse-docs

アーキテクチャ調査書 §3 と任意の env-config.json から、対象リポジトリの起動手順・前提ツール・割当を機械検証付きで「環境実行手順.html」として生成するスキルの確定仕様。

こんな人向け
ポータルの将来ページ受け口に環境実行手順カードを反映したい人 ／ 調査書に書かれた起動手順の根拠パスを機械的に裏取りしたい人
こんな場面で使う
アーキテクチャ調査書が確定済みで、次にポータル基盤情報ページを揃えたい時 ／ env-config.json（surveying-local-environment の出力）が既にあり、前提ツールの実測結果も併記したい時
対応 OSWindows WSL2 環境
検証状況エンジンスクリプト（validate-page-data.sh / build-detail-page.sh）はフィクスチャでの動作確認済み。スキル単体の実機検証は未実施
依存jq ／ shared/scripts/detail-pages/validate-page-data.sh・build-detail-page.sh（スキルフォルダ相対 ../../../shared/scripts/detail-pages/）／ shared/scripts/build-portal.sh（任意・カード反映時）／ surveying-local-environment（任意・env-config.json 生成時）

§1概要

リバースエンジニアリング設計のポータルには、将来ページの受け口（docs_root 直下に該当 HTML が実在するときだけカード化される仕組み）がある。本スキルはそのうち環境実行手順（T5）専任の生成係として、アーキテクチャ調査書 §3 ビルドと起動の記載値を主データ源に、任意で env-config.json を突き合わせ、環境実行手順.html を機械検証付きで作成する。

本スキルは判定・評価をしない。起動手順の良否や簡潔さは問わず、調査書と env-config.json の記載事実のみを前提ツール表・実行手順表・割当表の 3 段に整理する。

── 処理の全体像 ──────────────────────────────

  アーキテクチャ調査書.md（§3 ビルドと起動）
       │  Phase 1  調査書の実在確認 + env-config.json の任意確認
       ▼
  env-config.json（任意。不在でも継続）
       │  Phase 2  prerequisites/steps/allocations の抽出
       ▼
  page-data.json（pageKind: env）
       │  Phase 3  validate-page-data.sh --target-repo で機械検証
       ▼
  検証済み page-data
       │  Phase 4  build-detail-page.sh --page env
       ▼
  <docs_root>/環境実行手順.html（+ 任意で build-portal.sh 再実行）

技術スタックページと同じく、本スキルは「既に確定した一次情報（調査書と env-config.json）を整理するだけ」の係である。新規の検出戦略宣言は行わない。

§2出力

スキルを実行すると、以下の 2 点が生まれる。

環境実行手順.html（<docs_root> 直下）
前提ツール／実行手順（ビルド→開発起動→本番起動の順）／割当（ポート等）の 3 表を持つページ。実行手順の各行には調査書の根拠パスが備考として付き、割当の各行には検証済みの出所（sourceRef）が付く。読んだ人はどの記載がどのファイルの実測から来たかをその場で追跡できる。
埋め込み page-data JSON
ページの中に、生成の元になった page-data.json がそのまま埋め込まれる。読んだ人は「いつ・どの調査書 §3 記載値から生成されたか」を HTML 単体で追跡でき、機械処理（後続スキルからの参照）にも使える。

調査書 §3 が不在の場合、本スキルは生成を行わず不在報告のみで終わる。この場合、環境実行手順.html は生まれない。

§3用語

page-data.json
detail-pages 系 5 スキルが共有する入力 JSON。pageKind ごとに型別スロットを持つ。本スキルは pageKind: "env" を固定で使う。スキーマの全量は page-data-schema.md を参照。
prerequisites / steps / allocations
env 型のスロット。prerequisites[] は {name, note} 形式の前提ツール、steps[] は {order, command, note} 形式の実行手順（order 昇順 = 実行順）、allocations[] は {target, value, sourceRef} 形式のポート等の割当。
env-config.json
surveying-local-environment スキルの出力。OS・パッケージ管理ツール・開発ツール（cloc/node/python3/jq/git）の有無を持つ。本スキルにとっては任意入力であり、不在でも生成を続行する。
整合検証
validate-page-data.sh --target-repo による page-data.json の機械検査。env 型では allocations[].sourceRef のみが実在検査の対象になる。

§4動作フロー

起動引数は target_repo_path・docs_root（必須）と env_config_path・portal_output_dir（任意）の 4 つ。種別は env に固定されているため、種別を選ぶ引数はない。

Phase内容完了条件
Phase 1
データ源読込		 アーキテクチャ調査書の実在確認（不在ならハード停止し surveying-architecture-for-reverse-docs の先行実行を案内）、env-config.json（既定: <docs_root>/env-config.json）の任意確認 調査書の実在確認済み、または不在を報告して停止。env-config.json の有無が確定済み
Phase 2
抽出		 prerequisites[]（env-config.json の tools 実測。不在時は空配列）、steps[]（§3 のビルド／開発起動／本番起動を order 昇順で転記）、allocations[]（§3 の環境変数定義ファイルから割当を抽出。見つからなければ空配列）を組み立て page-data.json を保存 prerequisites[]/steps[]/allocations[] を確定し page-data.json 保存済み
Phase 3
整合検証（機械実行）		 validate-page-data.sh <page-data.json> --target-repo <target_repo_path> を実行。FAIL は修正して再実行（3 回失敗で Phase 2 の allocations 抽出へ差し戻し） 全項目 PASS（exit 0）
Phase 4
環境実行手順.html 生成		 build-detail-page.sh <page-data.json> <docs_root> --page env を実行。portal_output_dir 指定時は build-portal.sh を再実行しカードへ反映 HTML 生成済み。指定時は build-portal.sh 再実行済み

エンジンスクリプトはスキルフォルダ相対 ../../../shared/scripts/detail-pages/ で参照する。

返却（呼び出し元との契約）

キー内容
statusDONE | STOPPED（調査書不在）| ERROR
artifacts生成した環境実行手順.html のパス（STOPPED/ERROR 時は空）
page_kindenv（固定値）
portal_rebuiltbuild-portal.sh 再実行の有無
hint停止理由、env-config.json 有無の注記、または次工程への申し送り

§5制約と禁止事項

手作業でのプレースホルダ置換は禁止。 環境実行手順.html は必ず build-detail-page.sh 経由の決定的処理で生成する。
env-config.json 不在はハード停止しない。 主データ源は調査書 §3 であり、env-config.json は前提ツール表を補強する二次情報にとどまる。不在時は前提ツール表のみ空欄で進める。
割当は捏造しない。 環境変数定義ファイルに割当の記載が見つからない場合、allocations[] は空配列のまま進める。AskUserQuestion で手動値を聞き出さない。

抽出時の落とし穴

落とし穴対処
steps[]/prerequisites[] に sourceRef フィールドを追加してしまうpage-data-schema.md の T5 節に両フィールドの sourceRef は存在しない。根拠パスは note へテキストとして埋め込む
env-config.json の既定パスを固定と誤解するsurveying-local-environment の出力先は呼び出し時の output_dir 次第で変わる。既定パスに無ければ env_config_path を明示する
調査書の「実在しない（理由: …）」行を転記してしまうsteps[] から除外する（存在しない手順を転記しない）
jq 未インストール環境での検証・生成失敗validate/build 両スクリプトが jq に依存するため事前に導入する
