§1概要

画面ごとの操作（ボタン押下・フォーム送信等）を選ぶと、画面→API→テーブル の呼び出し順序を表示するシーケンス図を生成する。本スキルは pageKind 体系（用語辞書・技術スタック・画面遷移図・ER図・環境構築手順・状態遷移図）には属さない。1 pageKind = 1 固定ファイル名という pageKind 契約に対し、シーケンス図は画面ごとに複数生成される画面別ページであり、出力先も 画面/screen-<ID>/ 配下に画面ごとに存在するため、shared/scripts/detail-pages/ の共通エンジン（validate-page-data.sh / build-detail-page.sh）は使わない。

判定・評価はしない。呼び出し順序や処理内容の良否には踏み込まず、facts（call_order）または手書き page-data に記録された事実のみを転記する。

§2出力

本スキルを実行すると、対象画面 1 つにつき以下の 1 点が生まれる。

シーケンス図.html
<docs_root>/画面/screen-<ID>/ 直下に固定生成される、当該画面の操作単位（ボタン押下・フォーム送信等）を選択式で切り替えられるシーケンス図ページ。固定 3 本のレーン（screen=画面 / api=API / table=テーブル）上に、facts の call_order または手書き page-data から機械変換した呼び出し順序を、根拠パス（sourceRef）付きで表示する。画面基本設計書・詳細設計書と同じ体裁の doc-nav を持ち、3 種の設計書間をタブで行き来できる。

§3使用タイミング

区分内容
前提対象画面の facts.yml（extracting-unit-facts-from-code が確定済み）または手書きの page-data がある画面について、シーケンス図を生成・更新したいとき
起動引数docs_root（画面/screen-<ID>/ の所在）・対象画面 ID（複数可）
出力先各画面ディレクトリ直下の <docs_root>/画面/screen-<ID>/シーケンス図.html に固定（build-portal.sh の DOC_NAV 判定と同値。基本設計・詳細設計フォルダの 1 階層上）

§4用語・page-data の形状

各画面の page-data（<docs_root>/画面/screen-<ID>/シーケンス図-data.json）は以下の形状を持つ。

── シーケンス図-data.json の形状 ─────────────────────────
{
  "screenId": "screen-order-list",
  "screenLabel": "注文一覧",
  "generatedAt": "ISO8601",
  "operations": [
    {
      "key": "save-click",
      "label": "保存ボタン押下",
      "handler": "handler-onSave-保存",
      "steps": [
        {"seq": 1, "from": "screen", "to": "api", "label": "POST /orders", "sourceRef": "src/screens/Order.tsx:42"},
        {"seq": 2, "from": "api", "to": "table", "label": "INSERT orders", "sourceRef": "server/orders.ts:10"},
        {"seq": 3, "from": "api", "to": "screen", "label": "201 Created → 一覧再取得", "kind": "return"}
      ]
    }
  ]
}
用語内容
レーン（ライフライン）固定 3 本: screen=画面 / api=API / table=テーブル。steps[].from/to はこの 3 値のみ
kind: "return"破線で描画されるステップ。省略時は実線
sourceRef任意。無いステップはテンプレート側で根拠列を空表示する
call_orderfacts.yml の⑤handler item が持つ任意フィールド。形式 "<連番>:<api分類のkey>@<file:line>; ..."。shared/references/facts-schema.md が定義元

§5Phase 手順

スキル開始時に TaskCreate で Phase 1〜3 のタスクを登録する。各 Phase 開始時に該当タスクを in_progress に、完了時に completed へ TaskUpdate で更新する。

Phase内容完了条件
1. page-data の確保Step 1: シーケンス図-data.json の実在確認。Step 2: 不在画面は facts.yml の call_order を持つ handler のみ機械変換（call_order が 1 件もない画面はスキップし報告）。Step 3: jq で必須キー・from/to の 3 値・seq 連番を検証対象画面ごとに実在確認済み・jq 検証通過済み、または変換不能を報告済み
2. DOC_NAV の組み立て戻るリンク・基本設計タブ・詳細設計タブ・シーケンス図タブ（自ページ）を、実在するタブのみで組み立てる（build-portal.sh セクション 3.5 と同一判定）対象画面ごとに doc_nav 文字列が確定済み
3. HTML 生成render-template.sh の render_template 関数を Bash から source して シーケンス図.html を生成。portal_output_dir 指定時は build-portal.sh を再実行対象画面すべてで シーケンス図.html が生成済み

手作業でのプレースホルダ置換（sed・perl 直書き等）は禁止する。render_template は最短前方一致で置換するため、値の中に他プレースホルダ文字列が偶然含まれても誤爆しない。新規 .sh スクリプトファイルは作らない。

§6重要な注意事項・予想を裏切る挙動

判定・評価はしない。呼び出し順序や処理内容の良否には踏み込まず、facts（call_order）または手書き page-data に記録された事実のみを転記する。call_order を持つ handler が 0 件の画面で、AskUserQuestion を使って手動でステップを聞き出さない。shared/scripts/detail-pages/ 配下・extracting-unit-facts-from-code 配下・seal-facts.sh は変更しない（別スキルの管轄）。
予想を裏切る挙動内容
出力先の階層テンプレート名（シーケンス図.html）が pageKind の FUTURE_FILES と同名でも、docs_root 直下ではなく画面ごとのフォルダ（画面/screen-<ID>/）直下になる。pageKind 体系の「1 pageKind = 1 固定ファイル名」契約はここでは適用されない
DOC_NAV の担当分担build-portal.sh は画面設計書（基本設計・詳細設計）側の DOC_NAV にのみシーケンス図タブを追加する。シーケンス図.html 自体の DOC_NAV は本スキルの Phase 2/3 が組み立てる（build-portal.sh の担当範囲外）
