ER図生成スキル
generating-er-diagram-for-reverse-docs

テーブル一覧.html に埋め込まれた manifest をエンティティの正としつつ、対象リポジトリのマイグレーション／ORM モデルから外部キーを抽出し、「ER図.html」を機械検証付きで生成するスキルの確定仕様。

こんな人向け
ポータルの将来ページ受け口に ER 図カードを反映したい人 ／ テーブル一覧に載ったテーブル同士の関連を、コードの外部キー定義から機械的に裏取りしたい人
こんな場面で使う
テーブル一覧.html が確定済みで、次にポータル基盤情報ページを揃えたい時 ／ マイグレーション・ORM モデルに散らばる外部キー定義を 1 枚の図として俯瞰したい時
対応 OSWindows WSL2 環境
検証状況エンジンスクリプト（validate-page-data.sh / build-detail-page.sh）とテーブル一覧の埋め込み manifest 契約（unit-manifest）はフィクスチャでの動作確認済み。スキル単体の実機検証は未実施
依存jq ／ shared/scripts/detail-pages/validate-page-data.sh・build-detail-page.sh（スキルフォルダ相対 ../../../shared/scripts/detail-pages/）／ shared/scripts/build-portal.sh（任意・カード反映時）

§1概要

リバースエンジニアリング設計のポータルには、将来ページの受け口（docs_root 直下に該当 HTML が実在するときだけカード化される仕組み）がある。本スキルはそのうち ER図（T4）専任の生成係として、テーブル一覧 manifest のエンティティと対象リポジトリの外部キー定義を突き合わせ、ER図.html を機械検証付きで作成する。

本スキルは判定・評価をしない。テーブル設計の良否・正規化の妥当性は問わず、外部キーとして検出できた関連のみを転記する。参照先テーブルが manifest に存在しない外部キーは捏造せず未解決として分離し、外部キーを 1 件も検出できなければ生成せず停止報告する。

── 処理の全体像 ──────────────────────────────

  テーブル一覧.html（script#unit-manifest）
       │  Phase 1  前提確認 + ORM/migration種別判別 + 検出戦略の承認
       ▼
  対象リポジトリのマイグレーション/ORMモデル
       │  Phase 2  manifestからentities抽出 + FK走査でrelations/unresolved振り分け
       ▼
  page-data.json（pageKind: er）
       │  Phase 3  孤児relation0件の確認 + validate-page-data.sh --target-repo
       ▼
  検証済み page-data
       │  Phase 4  build-detail-page.sh --page er
       ▼
  <docs_root>/ER図.html（+ 任意で build-portal.sh 再実行）

画面遷移図と同様、本スキルは「確定済みの一覧（テーブル一覧）」と「コード新規抽出（外部キー走査）」の 2 段構成を持つ。エンティティ自体は新規抽出せず、テーブル一覧 manifest をそのまま正とする。

§2出力

スキルを実行すると、以下の 2 点が生まれる。

ER図.html（<docs_root> 直下）
テーブル一覧の各テーブルをノードとし、外部キーで検出した関連を線で結んだ図と、その下に関連の詳細（参照元／参照先／cardinality／出所）を並べた補足表を持つページ。参照先テーブルが manifest に存在しない外部キーは、図には現れず「未解決」節に理由付きで一覧表示される。
埋め込み page-data JSON
ER図.html の中に、生成の元になった page-data.json がそのまま埋め込まれる。読んだ人は「いつ・どのテーブル一覧 manifest とどの外部キー定義から生成されたか」を HTML 単体で追跡でき、機械処理（後続スキルからの参照）にも使える。

外部キーが 1 件も検出できない場合、本スキルは生成を行わず 0 件検出の報告のみで終わる。この場合、ER図.html は生まれない。

§3用語

page-data.json
detail-pages 系 5 スキルが共有する入力 JSON。pageKind ごとに型別スロットを持つ。本スキルは pageKind: "er" を固定で使う。スキーマの全量は page-data-schema.md を参照。
entities / relations
er 型のスロット。entities[] は {key, label} 形式のテーブルノード、relations[] は {from, to, cardinality, sourceRef} 形式の外部キー関連。from/to は entities[].key を参照する。
unresolved
参照先テーブルが manifest に存在しない外部キー、または cardinality を導出できない関連を分離する配列。{label, reason, sourceRef} 形式。
孤児関連
relations[].from/to が entities[].key のいずれにも一致しない関連。Phase 2 の振り分けが正しく機能していれば発生しないが、Phase 3 で必ず再確認する（validate-page-data.sh 自体はこの整合性を検証しないため）。
整合検証
validate-page-data.sh --target-repo による page-data.json の機械検査。sourceRef が対象リポジトリ上に実在するかを検査する。

§4動作フロー

起動引数は target_repo_path・docs_root（必須）と portal_output_dir（任意）の 3 つ。種別は er に固定されているため、種別を選ぶ引数はない。

Phase内容完了条件
Phase 1
前提確認 + 検出戦略宣言		 テーブル一覧.html（script#unit-manifest）の実在確認（不在ならハード停止し generating-table-list-for-reverse-docs の先行実行を案内）、ORM/migration 種別の判別、検出戦略の AskUserQuestion 承認 manifest の実在確認済み・検出戦略が承認済み、または不在を報告して停止
Phase 2
抽出		 manifest の units[] から entities[] を組み立て、FK 走査で relations[]/unresolved[] へ振り分け。0 件なら生成せず停止報告 振り分け完了で page-data.json 保存済み、または 0 件を報告して停止
Phase 3
整合検証		 孤児関連 0 件の手動確認 + validate-page-data.sh <page-data.json> --target-repo <target_repo_path> を実行。FAIL は修正して再実行（3 回失敗で Phase 2 へ差し戻し） 孤児関連 0 件・全項目 PASS（exit 0）
Phase 4
ER図.html 生成		 build-detail-page.sh <page-data.json> <docs_root> --page er を実行。portal_output_dir 指定時は build-portal.sh を再実行しカードへ反映 HTML 生成済み。指定時は build-portal.sh 再実行済み

エンジンスクリプトはスキルフォルダ相対 ../../../shared/scripts/detail-pages/ で参照する。ORM 別 FK 検出パターンと cardinality 導出規則は references/er-detection.md を参照する。

§5制約と禁止事項

手作業でのプレースホルダ置換は禁止。 ER図.html は必ず build-detail-page.sh 経由の決定的処理で生成する。
manifest 外参照は捏造しない。 参照先テーブルがテーブル一覧 manifest に存在しない外部キーは、関連として転記せず unresolved[] へ分離する。
FK 0 件はハード停止。 検出戦略に沿って走査しても外部キーが 1 件も見つからない場合、生成せずに 0 件検出をユーザーへ報告して停止する。AskUserQuestion で手動関連を聞き出して即興確定しない。
判定・評価は対象外。 テーブル設計の良否・正規化の妥当性には踏み込まない。本スキルの仕事は検出できた外部キー事実の転記のみ。

抽出時の落とし穴

落とし穴対処
マイグレーションと ORM モデルの両方を無差別に走査し同一関連を二重検出するテーブル一覧生成時に確定した正本と同じ側からのみ FK を抽出する
孤児関連（from/to が entities[].key に存在しない関連）が紛れ込むPhase 2 で振り分け、Phase 3 で 0 件を再確認する（validate-page-data.sh はこの整合性を検証しない）
jq 未インストール環境での検証・生成失敗validate/build 両スクリプトが jq に依存するため事前に導入する
