## 一覧生成の種別ループ（Phase 3 / Step 9）

一覧生成工程は、アーキテクチャ調査書の `unit_kinds_present` にある全種別について、種別別の一覧生成スキルを起動する。種別とスキルの対応は下表のとおり。

| unit_kind | スキル名 | ラベル | 出力 | 配列キー |
|---|---|---|---|---|
| screen | generating-screen-list-for-reverse-docs | 画面 | 画面一覧/画面一覧.html | screens（後方互換） |
| api | generating-api-list-for-reverse-docs | API | API一覧/API一覧.html | units |
| table | generating-table-list-for-reverse-docs | テーブル | テーブル一覧/テーブル一覧.html | units |
| batch | generating-batch-list-for-reverse-docs | バッチ | バッチ一覧/バッチ一覧.html | units |
| report | generating-report-list-for-reverse-docs | 帳票 | 帳票一覧/帳票一覧.html | units |
| external | generating-external-list-for-reverse-docs | 外部連携 | 外部連携一覧/外部連携一覧.html | units |

facts抽出以降の工程（ユニット反復〜往復検証）は screen のみ実装済み。screen 以外の種別は、一覧確立をもって「後続未対応」の終端状態として記録する。

管理者（orchestrating-reverse-docs-flow）の最終報告には、全6種別それぞれについて次の3値のいずれかを必ず含める。

| 値 | 意味 |
|---|---|
| 生成済み | 実在し一覧化済み（screen のみ後続工程へ進む） |
| 対象外 | excluded-kinds.json に記録された、対象リポジトリに実在しない種別 |
| 後続未対応 | 実在し一覧化済みだが facts 抽出に進めない種別 |

## NG帰着3系統

判定（Phase 7 / Step 30）でFAILした場合、原因を以下の3系統いずれかに帰着させ、該当スキルの改訂へ差し戻す。

| 系統 | 原因 | 差し戻し先 |
|---|---|---|
| (a) 執筆規律不足 | 詳細設計書の執筆規律・転記精度に起因する不一致 | generating-reverse-detailed-design の執筆規律 reference を改訂 |
| (b) facts欠落 | 事実抽出プロファイルが対象コードの挙動を捕捉できていない | extracting-unit-facts-from-code の抽出プロファイルを改訂 |
| (c) 共通文書欠落 | 共通設計書・規約4種等のプロジェクト共通文書に該当挙動の記載が無い | generating-reverse-common-docs を mode=append で起動 |

## 責務確定

工程をまたぐ責務のうち、解釈が分かれていた5点を工程設計の観点から次のとおり確定した。キーは連番禁止。内容を要約した意味語で付ける（番号からは情報を得られないため）。

| 確定事項 | 結論 |
|---|---|
| scaffold正本 | scaffold-screen.sh の正本は `shared/scripts/scaffold-screen.sh` の1本のみ。generating-reverse-detailed-design と rebuilding-screen-unit-from-docs は起動引数 `scaffold_script_path`（管理者が解決して渡す。`audit_script_path` と同型）で受け取って実行する |
| 単体テスト正本 | 単体テストコードの生産者は rebuilding-screen-unit-from-docs のみ。rebuilding-code-from-docs が `saved_test_paths` 不在時に観点表から自作するテストは正規の代替経路だが検証用の一時テストであり、テストコード正本ディレクトリ（`<画面ディレクトリ>/テスト項目書/テストコード/単体/`）への保存を禁止する |
| e2e作成実行分担 | E2E（RT-/SM-/IT-/CMP- 系）テストの作成と基準実測は rebuilding-code-from-docs（mode=implement）が担い、実行は syncing-reverse-env の dynamic 検査が担う（`compare_request.scenarios_ready` / syncing の `scenarios` 引数に整合）。未解決の設計課題ではない |
| 比較結果の解釈権 | compare_result.status について、syncing-reverse-env は計測事実（static_diff / dynamic / env_check）の報告者であり、往復検証の PASS/FAIL の意味解釈は judge（rebuilding-code-from-docs mode=judge）が単独で担う |
| 開通の直接起動 | unlocking-reverse-target-screens が syncing-reverse-env（mode=registry）を直接起動する構成は、完全仲介方式の例外ではなく、基準タグ確立まで単独完走するという設計要件に基づく意図した正式仕様である |

## 段階計画（Cycle 0〜4）

種別横断の拡張は以下の Cycle で進める。Cycle 番号は章・節と同じ構造上の位置番号である。各 Cycle の合格条件は既存原則を踏襲する: 異種プロジェクトでスキル無改造成立・決定的出力のみで検収。

| Cycle | 状態 | 内容 |
|---|---|---|
| Cycle 0 | 完了 | 一覧スキル6分割・契約明文化・責務確定・README/全体ガイド整備 |
| Cycle 1 | 未着手 | api 縦貫。extracting-unit-facts-from-code の profile=api 追加・facts-schema 拡張 → generating-reverse-detailed-design の api 章マップ → 画面レンダリング比較に代わる検証方式（スキーマ差分・HTTP応答突合）の設計 |
| Cycle 2 | 未着手 | table / batch。table はスキーマ静的比較、batch は実行契約の facts |
| Cycle 3 | 未着手 | report / external。帳票レイアウト・外部連携契約 |
| Cycle 4 | 未着手 | 上位抽象化スキル。基本設計・要件定義文書群（`納品物フォルダ体系.md` の未実装担当分） |

基盤設計.md・UI共通設計.md・データ設計.mdの3文書は、当初 Cycle 4（上位抽象化スキル）の担当分として計画していたが、generating-reverse-common-docs の Phase 4（共通文書採録）へ責務を移管し実装済みとした（`納品物フォルダ体系.md` 参照）。Cycle 4 の残担当は機能要件一覧・帳票要件・バッチ要件・外部連携要件・ビジネス概要の5文書、およびバッチ/帳票/外部連携の各基本設計書である。

## バージョニングの一般方針

テンプレート・スキーマ変更時の既存成果物との互換性方針を以下のとおり定める。

- 既存生成済み設計書は再生成時に新テンプレートを適用する。手動追記部分（§16 要確認事項一覧・レビューコメント等、著述工程の機械転記対象外の記述）は差分マージで保持し、機械転記の上書きで消さない
- テンプレート・スキーマのフィールド追加・削除・リネーム時は、既存成果物の移行手順を本文書の「改訂記録」に記録する
