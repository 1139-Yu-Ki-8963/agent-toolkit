# 新規スキル実装計画（1スキルずつ・汎用性検証を積み上げる）

- 状態: 計画確定・Stage 1 から着手
- 作成日: 2026-07-07
- 関連文書: 同フォルダ リバース工程設計.md（Phase/Step×スキル対応の正本）・納品物フォルダ体系.md（成果物の置き場の正本）
- 完了条件（機械定義）: 作成した設計書だけから judge status=PASS（env_check全通過 ∧ 全scenario render-ready ∧ L2'内容一致 ∧ L2(ARIA)一致 ∧ L4 console一致 ∧ L3画素差分閾値内0.1% ∧ 該当時L5操作一致）の再生成コードが得られること。これが「まったく同程度の画面・機能が使える精度の設計書」の定義であり、主観判定を排除する。

## 設計の核: 鶏卵問題の解

共通基盤（アーキテクチャ調査・共通文書）を先に分析しないと各画面のリバースが破綻する。一方、共通基盤に何を書くべきかは各画面のリバースをしてみて初めて分かる。この鶏卵問題は「共通基盤は先に v0 を作り、画面ごとの垂直ループの NG 帰着で追記して育てる」二拍子で解く。

```
Stage 1        Stage 2          Stage 3        Stage 4（本丸・完了条件の検証）
アーキ調査 → 共通採録(v0) → 事実抽出 → 詳細設計執筆
              ▲                ▲              │
              │(c)共通文書の   │(b)factsの     ▼
              │ 欠落を追記     │ 欠落を        既存検証スキルで判定
              │                │ プロファイル   rebuilding-screen-unit(6計測)
              │                │ 改訂          → rebuilding-code + syncing
              │                │              → judge PASS/FAIL
              └────────────────┴──────────────┘
              NG分類で原因を(a)執筆規律/(b)facts/(c)共通文書
              の3系統に帰着して還元
```

## 全Stage共通の検証原則

- 1スキルずつ実装し、検証PASSまで次Stageへ進まない
- 汎用性の検証 = ①SKILL.md本体に固有文字列grepゼロ ②異種プロジェクトで手直しなしに成立 ③sonnetサブエージェントがSKILL.mdだけで単独完走（メインが救済したらスキル不備として改訂）
- レビューは決定的出力のみ（ファイル実在・exit code・diff値。自然文の自己申告での合格判定禁止）

| 項目 | 内容 |

## Stage 6: 横展開系（後回しを明示的に確定）

目録スキルのユニット汎用化改修・上位文書抽象化スキル・画面以外の抽出プロファイル5種は、完了条件の達成に不要なため実装しない。着手判断はStage 4の汎用性合格後。理由: パイロットでは画面一覧が既存で目録改修は不要、上位文書は詳細設計の量が揃ってから、他ユニット種別は往復検証（採点者）が画面専用のため精度検証ができない。

## 要点

1. 採点者（検証スキル群）が既に存在するため、Stage 3から先は毎回機械採点付きで実装できる
2. 鶏卵問題は「共通基盤v0→画面ループのNG帰着(c)で追記」の設計で吸収し、共通文書の完成をブロッカーにしない
3. 完了条件はjudge PASSという機械的定義に固定し、スコアの主観判定を排除する
