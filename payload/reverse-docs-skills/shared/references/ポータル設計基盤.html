PROJECT PORTAL

プロジェクトポータル 設計基盤

プロジェクトに恒久的に残る参照サイトとして、各画面の操作期待を明文化し実装の全体像を確立する

1

コンセプト

プロジェクトの設計情報を一元参照する恒久サイト。リバース工程は初回生成のきっかけにすぎない。Markdown では実現できない検索・フィルタ・ソート・図解・折りたたみをHTML で提供する。

2

読者（4種）

新規参画の開発者 レビュアー・アーキテクト QA・運用担当 AI エージェント

開発者は構成把握とパスのコピー、レビュアーは網羅性と権限の監査、QA は影響範囲の特定、AI は埋め込み JSON からの機械的読み取り。

3

技術制約

単一ファイル自己完結 外部リソースなし ダークモード対応 決定的生成

ビルドスクリプトがマニフェスト JSON からテンプレートに注入する決定的生成。手編集は次回再生成で上書きされる。

マトリクス・対応表設計

フラット表では答えられない質問にマトリクスで答える

一覧は「1行 = 1エンティティ」。2つのエンティティの関係を問う質問には「行 × 列 = 関係」の専用ページが必要

権限×画面マトリクス

画面 × ロール = 閲覧可否

答える質問: 「このロールはどの画面を見られるか」「権限設定に漏れはないか」

ログイン○○○

ロール数対応: 縦書きヘッダ + 1列目固定 + ロール選択チップで20ロール規模まで実用

レビュアー: 権限漏れの監査 QA: ロール別テスト計画

権限×機能マトリクス

機能 × ロール = C/R/U/D

答える質問: 「画面は見えるが、編集・削除までできるのは誰か」。閲覧可否より1 段深い操作レベルの権限を示す

答える質問: 「このテーブルに書き込むのはどの機能か」「テーブル変更の影響範囲はどこまでか」。設計書の古典的成果物

追跡可能性

画面 → API → テーブルの連鎖

答える質問: 「この画面を変えたら何に波及するか」「このテーブルはどの画面から使われているか」。順方向・逆方向の両方向で追跡できること

画面APIテーブル
ログイン画面POST /api/auth/loginusers, sessions
投稿エディタPOST /api/postsposts, tags

逆引き例: users ← [POST /api/auth/login] ← [ログイン画面]。テーブル起点の逆方向ビューも同一データから表示する

QA: テスト範囲の確定 AI: 変更影響の機械的読み取り

配置の設計判断: マトリクス・対応表は一覧ページの列に押し込まず、画面遷移図・ER図と同格の独立ページとする。前提としてマニフェスト JSON への権限情報・API-テーブル対応の追加（スキーマ拡張）が必要。
