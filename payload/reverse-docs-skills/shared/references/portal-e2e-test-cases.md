# ポータル E2E テストケースカタログ

## 目的と実行方法

ポータル全画面（トップ・一覧 7 種・交差ビュー 4 種・AI設定資産・画面遷移図・ER図・技術スタック・環境構築手順・用語辞書・プロジェクト共通 3 種の計 21 ページ）の操作品質を回帰検証する。ページ追加・テンプレート改修のたびに本カタログのケースを通し、導線切れ・操作機能の退行を検出する。

実行方法は 2 種を併用する。

| 実行方法 | 内容 | 対象ケース |
|---|---|---|
| 静的 | `shared/scripts/test-e2e-portal.sh` を実行し、生成済み HTML への grep / jq 検査で DOM 構造・リンク・CSS 契約を機械検証する | 検証方法 = 静的 |
| ブラウザ | 本カタログのケースを Playwright 等で手動・機械実行し、クリック・入力・遷移の実挙動を検証する | 検証方法 = ブラウザ |

静的テストのレイアウト対応: 一覧ページは「`<root>/一覧/<種別一覧>/`」（サンプル配置）と「`<root>/<種別一覧>/`」（統括フロー生成配置）の両レイアウトを探索し、検出結果をヘッダ行で出力する。ポータル index の正本は `<root>/index.html` のみで、正本外配置は FAIL のままとする（正本準拠の強制がテストの役割）。AI設定資産・交差ビューは存在する場合のみ検査し、不在は SKIP 行を出す。

## 設計判断

### test-e2e-portal.sh

- 配置: shared/scripts/test-e2e-portal.sh
- 必要性: ポータル全 21 ページの回帰検査は、リンク解決・埋め込み JSON 妥当性・CSS 契約（縦書き・行高）・DOM 構造の 20 超の検査を 20 ページへ掛け合わせる作業であり、毎回手動 grep では再現不能。本カタログの E2E ケースのうち静的検証部（検証方法 = 静的）を機械化し、同一入力から同一判定を返す回帰エンジンとして固定する
- 代替案を採用しなかった理由:
  - Bash ツール直叩き: 20 超の検査 × 20 ページの組み立てを毎回行うのは非現実的で、検査条件が実行ごとにぶれて回帰検証が成立しない
  - 既存 Makefile 拡張 / package.json scripts 追加: 本リポジトリにビルド設定が存在せず、新規導入は本検査専用の依存を増やすだけになる
- 保守責任者: 人手（ユーザー）。ページ・機能を追加した場合は本カタログのテストケース表・カバレッジ表と本スクリプトの検査を同時に追従させる
- 廃棄条件: ポータル生成が別基盤に移行した時
