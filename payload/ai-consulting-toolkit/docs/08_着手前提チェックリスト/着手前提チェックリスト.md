# 着手前提チェックリスト

AI化タスクに着手できる状態を定義する。アクセス権限・データの実物・現状手順・実行環境・体制と時間の5分類すべてが「揃った」または「不要」になるまで、そのタスクの導入作業に着手しない。フェーズ4（レベル判定合意後）からフェーズ5（WBS展開）にかけて使用する。

## 前提5分類の定義

| 分類キー | 定義 | 確認内容の例 |
|---|---|---|
| アクセス権限 | 対象システム・データ・保存場所への、作業者（AIを含む）の権限 | 対象システムの利用者登録、データフォルダの閲覧・書込権限、AIサービスから接続する際の認証情報の発行可否 |
| データの実物 | 見本や説明ではない実データの入手（`../06_データ取り扱いチェック/データ取り扱いチェック.md` の判定に従い、必要なら加工版） | 直近の実ファイル一式、実際の入力メール・帳票、過去の成果物の実物 |
| 現状手順 | 現状のタスク実施方法の手順の記録 | 手順書の実物、担当者による実演の記録、例外時の対応方法 |
| 実行環境 | AIを動かす環境の準備 | 利用するAIサービスの契約・設定、動かす端末、社内ネットワークからの接続可否、必要ツールの導入 |
| 体制と時間 | 人の時間と体制の確保 | 担当者の作業時間の確保、成果物の確認者、不明点の問い合わせ先、休暇・繁忙期の考慮 |

## 記入表

| タスクキー | 分類 | 必要なもの（具体） | 状態 | 用意する人 | 期日 | 備考 |
|---|---|---|---|---|---|---|
| | | | | | | |

状態は「揃った」「未」「不要」の3値。1タスクの全分類が「揃った」または「不要」になったら着手可能である。
タスクキーは棚卸し表（`../03_業務棚卸しテンプレート/業務棚卸しテンプレート.md`）と同じ意味語キーを使う（連番禁止）。
「必要なもの」は推測で書かず、担当者に確認した事実を書く。

## 記入例

請求書-発行タスクについて、5分類を埋めた状態を示す。

| タスクキー | 分類 | 必要なもの（具体） | 状態 | 用意する人 | 期日 | 備考 |
|---|---|---|---|---|---|---|
| 請求書-発行 | アクセス権限 | 販売管理システムの参照権限、請求書テンプレート格納フォルダの閲覧権限 | 揃った | - | - | 既存の営業事務アカウントで対応済み |
| 請求書-発行 | データの実物 | 直近3か月分の請求書発行データ一式（伏せ字化版） | 揃った | - | - | `../06_データ取り扱いチェック/データ取り扱いチェック.md` の判定済み |
| 請求書-発行 | 現状手順 | 請求書作成手順書、金額突合の確認観点 | 未 | 営業事務担当 | 2026-07-15 | 手順書は口頭説明のみで文書化されていない |
| 請求書-発行 | 実行環境 | 利用するAIサービスの契約、営業事務端末への導入 | 未 | 情報システム部 | 2026-07-20 | 契約手続き中 |
| 請求書-発行 | 体制と時間 | 導入検証の作業時間確保、成果物の確認者の指名 | 揃った | - | - | 営業事務リーダーが確認者に決定済み |

## 運用

consult-hearingスキルがヒアリング後に本チェックリストを出力する。
consult-wbsスキルはWBS展開前に充足状況を確認し、「未」が残るタスクは前提整備タスク（`../04_レベル別タスクテンプレート/レベル別タスクテンプレート.md` の共通フェーズ）から着手する。
状態の更新は事実（受領・発行・合意の完了）をもって行い、見込みや依頼済みだけでは「揃った」にしない。
