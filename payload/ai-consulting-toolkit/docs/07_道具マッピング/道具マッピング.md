# 道具マッピング

道具箱が参照する外部ツール資産の役割と使い分けをまとめる。

## 外部資産の一覧

| 資産 | 場所 | 役割 |
|---|---|---|
| wbs-hearing | `~/cloned-repos/wbs-skills/wbs-hearing/` | 選択式の対話でプロジェクト要件を深掘りするヒアリングスキル |
| wbs-generator | `~/cloned-repos/wbs-skills/wbs-generator/` | ヒアリング結果からExcelのWBS（Work Breakdown Structure。作業を階層分解した一覧表）を生成する。Excel・Excel for the web・Googleスプレッドシートの3環境で編集可能 |
| wbs-edit | `~/cloned-repos/wbs-skills/wbs-edit/` | SharePoint / Googleスプレッドシート上の共同編集WBSをAIが編集し本番反映するスキル |
| single-file-wbs | `~/cloned-repos/single-file-wbs/` | 単一HTMLのWBSビューア。データはwbs.json 1枚で、ガントチャート・進捗表示を自動描画する。AIがJSONを直接編集して保守する設計 |

## 使い分けの原則

| 用途 | 使う道具 |
|---|---|
| 顧客と共有する管理表 | Excel系（wbs-generator + wbs-edit） |
| コンサル自身の案件・AI稼働管理 | single-file-wbs |

顧客側はExcel文化が大半のため、顧客と共有する管理表はExcel系（wbs-generator + wbs-edit）に寄せる。コンサル自身が使う案件・AI稼働管理はsingle-file-wbsに寄せる。AIに日々の更新を任せられ、タスクごとに`_ai`という自由記述欄を設けてAIの処理量・稼働時間の実績を記録できる。いずれも入口は内製スキル（consult-hearing / consult-wbs）であり、外部資産は生成エンジン・ビューアとして呼び出される。

## 業務フェーズとの対応

| フェーズ | 使う道具 | 補足 |
|---|---|---|
| 課題受付・事例照合 | consult-intake スキル + consult-request-list スキル + consult-level-draft スキル | 既知課題は事例と導入スキルで即応、未知はヒアリングへ。受付直後に全課題のレベル軌道下書きを生成する |
| 現場ヒアリング | consult-hearing スキル | 課題深掘り・4判定軸評価・レベル判定を選択式対話で行う |
| 業務の棚卸し | `../03_業務棚卸しテンプレート/業務棚卸しテンプレート.md` | consult-hearing が対話の中で記入する |
| AI化レベル判定 | `../02_AI化レベル基準/AI化レベル基準.md` + consult-level-agree スキル | レベル1〜5の判定基準。consult-hearing が判定まで行い、consult-level-agree で顧客と合意する |
| 施策設計・WBS化 | consult-wbs スキル + `../04_レベル別タスクテンプレート/レベル別タスクテンプレート.md` | 判定レベルごとの施策をWBSに落とし込む |
| 実行・稼働管理 | wbs-edit（顧客共有）+ single-file-wbs（内部） | 顧客向けと内部向けを併用する |
| 効果測定 | `../05_効果測定テンプレート/効果測定テンプレート.md` + consult-measure スキル | 稼働実績と効果を突き合わせ、昇格判定を行う |

## 内製スキル

| スキル | 役割 | 元にした外部資産または位置付け |
|---|---|---|
| consult-intake | 課題の受付と振り分け。自由文の課題を案件フォルダに記録して解決事例集と照合し、既知は事例と導入スキルで即対応、未知はconsult-hearingへ引き継ぐ（フェーズ1を担当） | 道具箱の入口。外部資産の土台なし |
| consult-request-list | 課題一覧から現場への確認依頼リストを生成。実物・事実・状況説明のみを依頼し、決定タスク・人的協力依頼は含めない（フェーズ1を担当） | 道具箱の入口の補助。外部資産の土台なし |
| consult-level-draft | 受付直後の全課題にL1〜L5軌道の下書きを生成する（上限は保留、第1段階を担当） | なし（本リポジトリ独自） |
| consult-hearing | 現場ヒアリング。課題深掘り・4判定軸評価・データ取り扱い事前確認を含み、判定レベル付き業務棚卸し表を出力する（フェーズ2〜4を担当） | wbs-hearing |
| consult-level-agree | 課題別にL1〜L5のレベル軌道を可視化し現在地・目標を合意するHTML生成。フェーズ4の完了条件 | なし（本リポジトリ独自） |
| consult-slide | 与えたトピックを図解入り横1枚（16:9）の解説HTMLにする（第4段階の合意後に任意生成） | なし（本リポジトリ独自） |
| consult-wbs | 棚卸し表からdocs/04のタスク群を展開し、顧客共有用Excel（wbs-generator連携）と内部管理用wbs.json（single-file-wbs形式）を出力する（フェーズ5を担当） | wbs-generator・single-file-wbs |
| consult-case | 検証済みの課題と解決策を説明HTMLと実物資材のセットで docs/09_解決事例集/ に登録し、蓄積簿を更新する | 事例集の蓄積窓口。外部資産の土台なし |
| consult-lint-setup | 顧客プロジェクトへ文章品質の機械強制一式（textlint・置き換え辞書・文体規約・コミット検査hook）を導入し実機検証まで行う実行スキル | 解決事例「prd長文-学術調」の導入手段。外部資産の土台なし |
| consult-repo-rules | 顧客リポジトリの暗黙の様式を規約として明文化し、分割規律で配置した上でレビュー・テストの仕組みを設置する実行スキル | 解決事例「コード癖-規約不在」「長文ルール-無視」の導入手段。外部資産の土台なし |
| consult-analyze | 顧客リポジトリの技術構成・規約状況・テスト分布を解析し、解析レポートHTMLを生成する | 裏付け資料の生成窓口。外部資産の土台なし |
| consult-diagnose | Claude Code設定（規約・スキル・hook）の整備状況を診断し、HTMLレポートで報告する | 設定診断の窓口。外部資産の土台なし |
| consult-measure | 効果測定と昇格判定。実測と昇格条件を突合し棚卸し表の現在レベルを更新する（第7段階を担当） | なし（本リポジトリ独自） |

wbs-hearingを土台に、consult-hearingへ次の変更を実装済みである。

- 課題深掘りの段階に4判定軸（誤り影響度・判断定型度・入力構造化度・検証容易性）の質問を追加した
- 業務の現状把握で「その作業の入力データは何か・どこにあるか」を必ず聞く（入力構造化度の判定材料にする）
- 出力文書に業務棚卸し表（`../03_業務棚卸しテンプレート/業務棚卸しテンプレート.md` の列構成）を含める
- 施策の単位を「AI化するタスク＋判定レベル」で区切る
- データ取り扱いの制約（`../06_データ取り扱いチェック/データ取り扱いチェック.md` の観点）を初期段階で確認する質問を足した

## 将来の拡張

consult-wbsがExcelとwbs.jsonの両形式を直接出力するため、ExcelからWBS（Work Breakdown Structure。作業を階層分解した一覧表）のwbs.jsonへの変換手段は不要になった。残る拡張候補は、稼働中WBSの実績（Excel側の進捗）をwbs.jsonへ同期する仕組みである。
