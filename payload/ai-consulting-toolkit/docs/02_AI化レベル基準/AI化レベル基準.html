§1AI化レベルとは

業務のどこまでをAIに任せるかを、レベル1からレベル5までの5段階で定義したもの。段階が上がるほどAIの自律度が増し、人の関与は薄くなる。導入は一度に1レベルずつ進めるのが原則で、いきなり高いレベルから始めることはしない。

本基準は次の3つの問いに、推測を挟まずに答えるための道具である。

使い方問い使う道具
上限の判定このタスクはどのレベルまで任せてよいか4つの判定軸 → 軸別上限表（§7・§8）
現在地の診断このタスクは今どのレベルで運用されているか診断の質問 → 各レベルの判定条件（§5・§6）
レベル軌道の記述このタスクの各レベルは具体的にどう動く姿か6つの観測点 → 課題別のL1〜L5説明（§4・§11）

課題を1つ与えられたとき、本基準だけを根拠に「この課題のL1〜L5はこういう姿で、現在地はここ、上限はここ」と出力できることがゴールとなる。

§2レベル定義表

レベル呼び方人とAIの役割分担例導入の前提条件運用時の監視
レベル1（人手のみ）人手のみAIを使わない。人がすべて実行する対面交渉・物理作業なし不要
レベル2（AI補助）AI補助人が主体で作業し、AIは下書き・要約・調査を提供する。人が毎回全部を確認する文書下書き・議事録要約出力を毎回全件確認する運用を組む都度の全件確認
レベル3（人が承認するAI実行）人が承認するAI実行AIが主体で実行し、人が成果物を承認してから次工程へ進める帳票作成→人が承認→送付承認手順・差戻し手順を整備する承認時の全件確認
レベル4（例外のみ人介入）例外のみ人介入AIが自動実行する。定型外・エラー時だけ人に引き継ぐ定型問い合わせの自動応答例外検知の仕組み・引き継ぎ先を整備する例外発生時のみ、抜き取り確認を併用
レベル5（完全自動）完全自動AIが自動実行し、人は関与しないログ集計・定期報告の配信監査体制・異常検知の仕組みを整備する定期監査のみ

レベル3〜5の「人の確認・承認」は、機械的検証（自動検査の仕組み）・別のAIによる検証で代替できる。最終的な採用・出荷の判断に人が残る運用はレベル3、人が結果を見ずに流れる運用はレベル4以上として扱う。

§3レベルの視覚化

左が人手、右に行くほどAIの自律度が上がる。

L1 人手のみ

AIを使わず人がすべて実行する。

L2 AI補助

人が毎回指示文を書き、AIの出力を人が編集して完成させる。

L3 人が承認するAI実行

固定指示でAIが完成品を作り、人は手を入れず合否だけ判定する。

L4 例外のみ人介入

自動起動でAIが実行し、列挙済みの例外だけ人に引き継ぐ。

L5 完全自動

日常運用に人が登場しない。定期監査のみ行う。

§4レベルを分ける6つの観測点

レベルの違いは「AIの賢さ」の違いではなく、業務の回り方の違いである。次の6点を観測すれば、どのレベルで運用されているかは機械的に決まる。課題別のレベル説明を書くときの型でもある（§11参照）。

観測点L1 人手のみL2 AI補助L3 人が承認するAI実行L4 例外のみ人介入L5 完全自動
仕事の起動人が始める人が始める人がAIを起動する受信・時刻・他システムの通知で自動起動自動起動
AIへの指示なし人が毎回考えて書く。1つの成果物に何往復も対話する文書化された固定指示を1回渡すだけ通常運用ではゼロゼロ
手順知識の所在人の頭と手順書人の頭（指示は毎回組み立てる）指示文集・出力の型として文書化機械が実行できる自動化設定自動化設定に加え、異常検知・自動停止の仕組み
成果物を完成させる者人人（AIの出力は素材）AI（人は手を入れない）AIAI
人の確認自分の作業として行う全件を読んで手直しする完成品を全件、合否だけ判定する例外時のみ対応し、正常分は抜き取り確認定期監査のみ
誤りの発見と回復人が自分で気づいて直す手直しの過程で人が直す承認時に人が見つけ、差し戻してAIに作り直させる機械が例外を検知して人へ引き継ぐ機械が検知・停止し、定期監査で検証する

§5各レベルの詳細定義

レベル1（人手のみ）

AIをその業務で使わない。対面交渉・物理作業のようにAI化の対象にならないものと、単にまだAIを使っていないものの両方が含まれる。どちらであるかは軸別上限表（§8）による上限の判定で区別する。

判定条件: 対象業務でのAI利用実績がない。

レベル2（AI補助）

具体像: 担当者が対話型AIの画面を開き、その都度指示文を書いて下書き・要約・調査結果を得る。1つの成果物のために何往復もやり取りし、得られた出力を自分で編集して完成させる。指示文をうまく書けるかどうかが担当者の腕前に依存し、品質が人によってばらつく。

判定条件（すべて満たすこと）

  • 人が起動し、人が毎回指示文を考えて書いている
  • AIの出力は素材であり、人が手を入れて完成させている
  • 完成前に人が全件を確認している

このレベルでないもの: AIの出力を無修正のまま成果物として使っているなら、実態はレベル3の運用である。承認手順・差戻し基準がないまま無修正利用している場合は「仕組みのないレベル3」であり、レベル3の前提条件（承認手順の整備）を満たすよう是正する。

レベル3（人が承認するAI実行）

具体像: 指示文は文書化・固定化されており、人が新しく文章を考えることはない。人の仕事は「起動」と「承認」の2つに絞られる。AIは完成形の成果物を出し、人は内容に手を入れず、合格なら次工程へ進め、不合格なら差し戻してAIに作り直させる。品質は指示文と差戻し基準の出来で決まり、担当者の腕前に依存しなくなる。

判定条件（すべて満たすこと）

  • 固定の指示文（指示文集・テンプレート）と出力の型が文書化されている
  • 人はAIの出力に手を入れない。修正は差戻しでAIに行わせる
  • 承認手順と差戻し基準が文書化され、人が全件を承認している（承認は機械的検証・別のAIによる検証で代替できる。ただし採用の最終判断は人に残る）

このレベルでないもの: 人が出力に手を入れて仕上げているならレベル2。人が毎回指示文を考えているならレベル2。起動が自動（人が始めない）ならレベル4。

レベル4（例外のみ人介入）

具体像: 業務はメールの受信・決まった時刻・他システムの通知をきっかけに自動で始まり、完成まで人は登場しない。あらかじめ列挙した例外条件に該当した場合だけ機械が人へ通知し、人がその件だけを処理する。正常に流れた分は抜き取りで品質を確認する。

判定条件（すべて満たすこと）

  • 人が起動しなくても業務が始まる（自動起動）
  • 例外と判定する条件が文書で列挙されている（「なんとなく変なら」は不可）
  • 例外発生時の通知先と対応手順が決まっている
  • 処理件数・失敗率が記録されている
  • 正常分を人は都度確認しない（抜き取り確認のみ）

このレベルでないもの: 人が毎回起動しているならレベル3。例外条件が列挙されておらず、人が心配で全件を見ているならレベル3。機械的検証・別のAIが全件を検証し、人が最終判断だけ行っているならレベル3。

レベル5（完全自動）

具体像: 例外対応を含めて、日常運用に人が登場しない。異常があれば機械が検知して自動停止し、担当者ではなく監査者が月次などの定期監査でだけ関わる。

判定条件（すべて満たすこと）

  • 異常検知と自動停止の仕組みが動作している
  • 監査手順と監査記録の保管先が決まっている
  • 日常運用における人の作業がゼロである
  • 監査・異常検知は機械的検証で構成してよい

このレベルでないもの: 例外の引き継ぎ先として人が待機しているならレベル4。

§6現在地レベルの診断

現在地は自己申告ではなく観測できる証拠で判定する。証拠は担当者の実演・指示文の実物・自動化設定の画面・監査記録のいずれかとする（棚卸し表の「推測記入の禁止」と同じ原則）。

質問キー質問回答とレベルの対応
AI利用有無この業務でAIを使っていますかいいえ → レベル1
指示文の固定化AIへの指示文は毎回考えて書きますか、固定の指示文を使い回しますか毎回書く → レベル2の証拠 / 固定 → レベル3以上の証拠
出力の扱いAIの出力に手を入れて完成させますか、手を入れず合否だけ判定しますか手を入れる → レベル2 / 合否のみ → レベル3以上
起動のきっかけこの業務は何をきっかけに始まりますか。人が始めなくても動きますか人が起動 → レベル3以下 / 自動起動 → レベル4以上
確認の頻度AIの結果を人がいつ・どれだけ見ますか全件確認 → レベル3以下 / 例外と抜き取りのみ → レベル4 / 定期監査のみ → レベル5
検証主体AIの出力の合否は誰が（何が）確認していますか人が確認 → レベル2〜3の証拠 / 機械・別のAIが検証し人が最終判断 → レベル3の証拠 / 機械・別のAIが検証し人は見ない → レベル4以上の証拠

§5の各レベルの判定条件に照らし、条件をすべて満たす最も高いレベルを現在地とする。条件を一部しか満たさない場合は1つ下のレベルに倒す（例: 指示文は固定だが人が出力に手を入れている → レベル2）。

質問キーは連番禁止。内容を要約した意味語で付ける（番号からは情報を得られないため）。

§74つの判定軸

タスクごとに以下4軸を「高・中・低」の3段階で評価する。

誤り影響度

AIが誤った成果物を出した場合に、どこまで影響が及ぶかを表す軸。

評価基準
高顧客・金銭・法令に影響が及ぶ
中社内の手戻りで収まる
低すぐ修正でき影響が残らない

判断定型度

タスク内の判断が、あらかじめルール化されているかを表す軸。

評価基準
高ルール化済みで手順書がある
中おおむね定型だが例外がある
低都度判断・暗黙知に依存する

入力構造化度

タスクへの入力データがどこまで整った形式かを表す軸。

評価基準
高データが整備済み
中形式が不揃い（半構造）
低紙・口頭・暗黙知

検証容易性

AIの成果物が正しいかどうかを、人がどれだけ手早く確認できるかを表す軸。

評価基準
高数分で正誤を確認できる
中抜き取り確認で足りる
低全件の精査が必要

§8判定マトリクス（軸別上限表）

4軸の評価結果から、到達可能な上限レベルを機械的に判定する。各軸の評価値ごとに上限が決まり、最も低い上限がそのタスクの判定レベル（上限）となる。

軸評価上限理由
誤り影響度低レベル5影響が残らない誤りは定期監査で拾えば足りる
中レベル4社内の手戻りは例外運用で収まるが、完全無人にはしない
高レベル3顧客・金銭・法令に届く誤りは、人の承認を最後の防壁として残す
判断定型度高レベル5ルール化済みの判断は機械に完全移管できる
中レベル4例外がある限り、引き継ぎ先の人が必要
低レベル2ルール化されていない判断では、AIが完成品を安定して作れない
入力構造化度高レベル5整備済みデータは無人でも取り込める
中レベル4形式が不揃いだと取り込みの失敗があり得るため無人化しない
低レベル2紙・口頭・暗黙知の入力は、人が対話で補いながら渡すしかない
検証容易性高レベル5正誤の確認が速い成果物は監査に載せやすい
中レベル4抜き取り確認で足りるなら例外運用に移せる
低レベル3全件の精査が必要なものは抜き取り確認へ移行できない

追加規則: 誤り影響度=高 かつ 検証容易性=低 の組み合わせは、上限をレベル2に引き下げる（重大な誤りを見つけにくい業務では、AIに完成品を作らせない）。

判定手順: 4軸それぞれの上限を表から引き、追加規則を適用したうえで、最小値を判定レベル（上限）とする。

適用例を次に示す（4軸の並びは誤り影響度・判断定型度・入力構造化度・検証容易性）。

タスク例4軸の評価軸別上限判定
請求書発行高・高・高・高レベル3・5・5・5レベル3
問い合わせ一次応答中・高・高・中レベル4・5・5・4レベル4
契約書の確認誤り影響度=高、検証容易性=低追加規則に該当レベル2

§9昇格条件

段階導入では、下のレベルでの運用実績が次のレベルの前提になる。以下の条件がすべて満たされたときに1レベル上げる。件数・期間・比率は初期の目安であり、案件ごとに顧客と合意して確定する（AI・コンサルタントの推測で確定しない）。

昇格満たすべき条件（すべて）
レベル1→2データ取り扱いチェック（docs/06）の判定を通過している / 指示文の初版がある / 出力を毎回全件確認する運用が決まっている
レベル2→3指示文が固定化され、直近の運用で内容の変更が不要になっている / 直近20件で、AI出力への内容の手直しが2割未満（誤字調整など形式の手直しは数えない） / 出力の型・承認手順・差戻し基準が文書化されている
レベル3→4直近50件または1か月の差戻し率が5%未満 / 発生した差戻し理由が、すべて例外条件として列挙済みである / 自動起動と例外通知の仕組みが動いている / 処理件数・失敗率が記録されている
レベル4→5直近3か月の例外率が1%未満で、例外の見逃しがない / 異常検知と自動停止の動作を確認済みである / 監査手順と監査記録の保管先が決まっている

昇格条件は「AIをどこまで使いこなせているか」の判定軸でもある。指示文を固定化できているか（レベル2→3）、指示なしで動く仕組みを組めているか（レベル3→4）、無人でも異常に気づける仕組みがあるか（レベル4→5）が、それぞれの節目となる。

§10判定の流れ

§11課題からレベル軌道を書く手順

課題（タスク）を与えられたとき、その課題のL1〜L5の具体像を書く手順。

  1. 課題を「入力 → 処理 → 出力 → 確認」の4要素に分解する（何を受け取り、何をして、何を出し、誰が確かめるか）
  2. 6つの観測点（§4）の各レベルの値を、その課題の固有名詞に置き換える
  3. レベルごとに1〜2文で書く。「何をきっかけに・誰が・AIが何をして・人が何を確認するか」を必ず含める
  4. 判定レベル（上限）を超えるレベルには「上限超過」と超過理由を明記する（参考として姿を書いてもよい）

記入例: 請求書発行（4軸=高・高・高・高、上限レベル3）

レベルこの課題での姿
レベル1（人手のみ）担当者が受注一覧と単価表を突き合わせ、請求書を手作業で作成して発行する
レベル2（AI補助）担当者が受注データを対話型AIに渡して請求書の下書きを作らせ、金額と宛先を自分で修正して仕上げる。指示文は毎回担当者が書く
レベル3（人が承認するAI実行）担当者が固定指示でAIを起動すると、AIが受注データから請求書を完成形で作成する。担当者は金額突合で合否だけを判定し、合格分を発行する。不合格分は差し戻してAIに作り直させる
レベル4（例外のみ人介入）上限超過（誤り影響度=高のため、人の承認を外せない）。参考: 月次締めをきっかけにAIが自動作成・発行し、単価表にない品目など列挙済みの例外だけ人が処理する姿
レベル5（完全自動）上限超過（同上）。参考: 発行から入金確認まで無人で流れ、人は月次監査のみ行う姿

§12境界事例

判断に迷いやすい実例を示す。

請求書作成

誤り影響度は高いが、金額突合を数分で確認できるため検証容易性は高い。軸別上限表では誤り影響度=高の上限レベル3が最小となり、レベル3（人が承認するAI実行）が上限となる。

問い合わせ一次応答

判断定型度・入力構造化度が高く、誤り影響度は中にとどまる。軸別上限は誤り影響度=中と検証容易性=中のレベル4が最小となる。段階導入の原則に従いレベル3で開始し、昇格条件（差戻し率5%未満・例外条件の列挙）を満たしてからレベル4へ上げる。

契約書の確認

誤り影響度が高く、検証容易性は低い（条項の妥当性を全件精査する必要がある）。追加規則（誤り影響度=高 かつ 検証容易性=低）に該当し、上限はレベル2。AIには論点の洗い出しまでを任せ、最終判断は人が行う。
