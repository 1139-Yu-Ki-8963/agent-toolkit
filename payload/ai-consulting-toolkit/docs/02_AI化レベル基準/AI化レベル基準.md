# AI化レベル基準

業務のどこまでをAIに任せるかを判定するための共通尺度を定義する。棚卸しの段階で洗い出したタスクに対し、レベル判定の段階で本基準を適用してAI化レベルを決める。顧客との合意形成にもこの基準をそのまま使う。

本基準は次の3つの問いに、推測を挟まずに答えられる状態を目指す。

| 使い方 | 問い | 使う道具 |
|---|---|---|
| 上限の判定 | このタスクはどのレベルまで任せてよいか | 4つの判定軸 → 軸別上限表 |
| 現在地の診断 | このタスクは今どのレベルで運用されているか | 診断の質問 → 各レベルの判定条件 |
| レベル軌道の記述 | このタスクの各レベルは具体的にどう動く姿か | 6つの観測点 → 課題別のL1〜L5説明 |

課題を1つ与えられたとき、本基準だけを根拠に「この課題のL1〜L5はこういう姿で、現在地はここ、上限はここ」と出力できることが本基準のゴールである。

## AI化レベルの定義

| レベル | 呼び方 | 人とAIの役割分担 | 例 | 導入の前提条件 | 運用時の監視 |
|---|---|---|---|---|---|
| レベル1（人手のみ） | 人手のみ | AIを使わない。人がすべて実行する | 対面交渉・物理作業 | なし | 不要 |
| レベル2（AI補助） | AI補助 | 人が主体で作業し、AIは下書き・要約・調査を提供する。人が毎回全部を確認する | 文書下書き・議事録要約 | 出力を毎回全件確認する運用を組む | 都度の全件確認 |
| レベル3（人が承認するAI実行） | 人が承認するAI実行 | AIが主体で実行し、人が成果物を承認してから次工程へ進める | 帳票作成→人が承認→送付 | 承認手順・差戻し手順を整備する | 承認時の全件確認 |
| レベル4（例外のみ人介入） | 例外のみ人介入 | AIが自動実行する。定型外・エラー時だけ人に引き継ぐ | 定型問い合わせの自動応答 | 例外検知の仕組み・引き継ぎ先を整備する | 例外発生時のみ、抜き取り確認を併用 |
| レベル5（完全自動） | 完全自動 | AIが自動実行し、人は関与しない | ログ集計・定期報告の配信 | 監査体制・異常検知の仕組みを整備する | 定期監査のみ |

## レベルを分ける6つの観測点

レベルの違いは「AIの賢さ」の違いではなく、**業務の回り方**の違いである。次の6点を観測すれば、どのレベルで運用されているかは機械的に決まる。課題別のレベル説明を書くときの型でもある（「課題からレベル軌道を書く手順」を参照）。

| 観測点 | レベル1（人手のみ） | レベル2（AI補助） | レベル3（人が承認するAI実行） | レベル4（例外のみ人介入） | レベル5（完全自動） |
|---|---|---|---|---|---|
| 仕事の起動 | 人が始める | 人が始める | 人がAIを起動する | 受信・時刻・他システムの通知で自動起動 | 自動起動 |
| AIへの指示 | なし | 人が毎回考えて書く。1つの成果物に何往復も対話する | 文書化された固定指示を1回渡すだけ | 通常運用ではゼロ | ゼロ |
| 手順知識の所在 | 人の頭と手順書 | 人の頭（指示は毎回組み立てる） | 指示文集・出力の型として文書化 | 機械が実行できる自動化設定 | 自動化設定に加え、異常検知・自動停止の仕組み |
| 成果物を完成させる者 | 人 | 人（AIの出力は素材） | AI（人は手を入れない） | AI | AI |
| 人の確認 | 自分の作業として行う | 全件を読んで手直しする | 完成品を全件、合否だけ判定する | 例外時のみ対応し、正常分は抜き取り確認 | 定期監査のみ |
| 誤りの発見と回復 | 人が自分で気づいて直す | 手直しの過程で人が直す | 承認時に人が見つけ、差し戻してAIに作り直させる | 機械が例外を検知して人へ引き継ぐ | 機械が検知・停止し、定期監査で検証する |

## 各レベルの詳細定義

### レベル1（人手のみ）

AIをその業務で使わない。対面交渉・物理作業のようにAI化の対象にならないものと、単にまだAIを使っていないものの両方が含まれる。どちらであるかは軸別上限表による上限の判定で区別する。

**判定条件**: 対象業務でのAI利用実績がない。

### レベル2（AI補助）

**具体像**: 担当者が対話型AIの画面を開き、その都度指示文を書いて下書き・要約・調査結果を得る。1つの成果物のために何往復もやり取りし、得られた出力を自分で編集して完成させる。指示文をうまく書けるかどうかが担当者の腕前に依存し、品質が人によってばらつく。

**判定条件（すべて満たすこと）**

- 人が起動し、人が毎回指示文を考えて書いている
- AIの出力は素材であり、人が手を入れて完成させている
- 完成前に人が全件を確認している

**このレベルでないもの**:

- AIの出力を無修正のまま成果物として使っているなら、実態はレベル3の運用である。承認手順・差戻し基準がないまま無修正利用している場合は「仕組みのないレベル3」であり、レベル3の前提条件（承認手順の整備）を満たすよう是正する

### レベル3（人が承認するAI実行）

**具体像**: 指示文は文書化・固定化されており、人が新しく文章を考えることはない。人の仕事は「起動」と「承認」の2つに絞られる。AIは完成形の成果物を出し、人は内容に手を入れず、合格なら次工程へ進め、不合格なら差し戻してAIに作り直させる。品質は指示文と差戻し基準の出来で決まり、担当者の腕前に依存しなくなる。

**判定条件（すべて満たすこと）**

- 固定の指示文（指示文集・テンプレート）と出力の型が文書化されている
- 人はAIの出力に手を入れない。修正は差戻しでAIに行わせる
- 承認手順と差戻し基準が文書化され、人が全件を承認している

**このレベルでないもの**:

- 人が出力に手を入れて仕上げているならレベル2
- 人が毎回指示文を考えているならレベル2
- 起動が自動（人が始めない）ならレベル4

### レベル4（例外のみ人介入）

**具体像**: 業務はメールの受信・決まった時刻・他システムの通知をきっかけに自動で始まり、完成まで人は登場しない。あらかじめ列挙した例外条件に該当した場合だけ機械が人へ通知し、人がその件だけを処理する。正常に流れた分は抜き取りで品質を確認する。

**判定条件（すべて満たすこと）**

- 人が起動しなくても業務が始まる（自動起動）
- 例外と判定する条件が文書で列挙されている（「なんとなく変なら」は不可）
- 例外発生時の通知先と対応手順が決まっている
- 処理件数・失敗率が記録されている
- 正常分を人は都度確認しない（抜き取り確認のみ）

**このレベルでないもの**:

- 人が毎回起動しているならレベル3
- 例外条件が列挙されておらず、人が心配で全件を見ているならレベル3

### レベル5（完全自動）

**具体像**: 例外対応を含めて、日常運用に人が登場しない。異常があれば機械が検知して自動停止し、担当者ではなく監査者が月次などの定期監査でだけ関わる。

**判定条件（すべて満たすこと）**

- 異常検知と自動停止の仕組みが動作している
- 監査手順と監査記録の保管先が決まっている
- 日常運用における人の作業がゼロである

**このレベルでないもの**:

- 例外の引き継ぎ先として人が待機しているならレベル4

## 現在地レベルの診断

現在地は自己申告ではなく**観測できる証拠**で判定する。証拠は担当者の実演・指示文の実物・自動化設定の画面・監査記録のいずれかとする（棚卸し表の「推測記入の禁止」と同じ原則）。

### 診断の質問

質問キーは連番禁止。内容を要約した意味語で付ける（番号からは情報を得られないため）。

| 質問キー | 質問 | 回答とレベルの対応 |
|---|---|---|
| AI利用有無 | この業務でAIを使っていますか | いいえ → レベル1 |
| 指示文の固定化 | AIへの指示文は毎回考えて書きますか、固定の指示文を使い回しますか | 毎回書く → レベル2の証拠 / 固定 → レベル3以上の証拠 |
| 出力の扱い | AIの出力に手を入れて完成させますか、手を入れず合否だけ判定しますか | 手を入れる → レベル2 / 合否のみ → レベル3以上 |
| 起動のきっかけ | この業務は何をきっかけに始まりますか。人が始めなくても動きますか | 人が起動 → レベル3以下 / 自動起動 → レベル4以上 |
| 確認の頻度 | AIの結果を人がいつ・どれだけ見ますか | 全件確認 → レベル3以下 / 例外と抜き取りのみ → レベル4 / 定期監査のみ → レベル5 |

### 診断の判定

各レベルの「判定条件」（前節）に照らし、**条件をすべて満たす最も高いレベル**を現在地とする。条件を一部しか満たさない場合は1つ下のレベルに倒す（例: 指示文は固定だが人が出力に手を入れている → レベル2）。

## 4つの判定軸

タスクごとに以下4軸を「高・中・低」の3段階で評価する。評価結果は `../03_業務棚卸しテンプレート/業務棚卸しテンプレート.md` の記入表に記録する。

### 誤り影響度

AIが誤った成果物を出した場合に、どこまで影響が及ぶかを表す軸。

| 評価 | 基準 |
|---|---|
| 高 | 顧客・金銭・法令に影響が及ぶ |
| 中 | 社内の手戻りで収まる |
| 低 | すぐ修正でき影響が残らない |

### 判断定型度

タスク内の判断が、あらかじめルール化されているかを表す軸。

| 評価 | 基準 |
|---|---|
| 高 | ルール化済みで手順書がある |
| 中 | おおむね定型だが例外がある |
| 低 | 都度判断・暗黙知に依存する |

### 入力構造化度

タスクへの入力データがどこまで整った形式かを表す軸。

| 評価 | 基準 |
|---|---|
| 高 | データが整備済み |
| 中 | 形式が不揃い（半構造） |
| 低 | 紙・口頭・暗黙知 |

### 検証容易性

AIの成果物が正しいかどうかを、人がどれだけ手早く確認できるかを表す軸。

| 評価 | 基準 |
|---|---|
| 高 | 数分で正誤を確認できる |
| 中 | 抜き取り確認で足りる |
| 低 | 全件の精査が必要 |

## 判定マトリクス（軸別上限表）

4軸の評価結果から、到達可能な上限レベルを機械的に判定する。各軸の評価値ごとに上限が決まり、**最も低い上限がそのタスクの判定レベル（上限）**となる。

| 軸 | 評価 | 上限 | 理由 |
|---|---|---|---|
| 誤り影響度 | 低 | レベル5 | 影響が残らない誤りは定期監査で拾えば足りる |
| 誤り影響度 | 中 | レベル4 | 社内の手戻りは例外運用で収まるが、完全無人にはしない |
| 誤り影響度 | 高 | レベル3 | 顧客・金銭・法令に届く誤りは、人の承認を最後の防壁として残す |
| 判断定型度 | 高 | レベル5 | ルール化済みの判断は機械に完全移管できる |
| 判断定型度 | 中 | レベル4 | 例外がある限り、引き継ぎ先の人が必要 |
| 判断定型度 | 低 | レベル2 | ルール化されていない判断では、AIが完成品を安定して作れない |
| 入力構造化度 | 高 | レベル5 | 整備済みデータは無人でも取り込める |
| 入力構造化度 | 中 | レベル4 | 形式が不揃いだと取り込みの失敗があり得るため無人化しない |
| 入力構造化度 | 低 | レベル2 | 紙・口頭・暗黙知の入力は、人が対話で補いながら渡すしかない |
| 検証容易性 | 高 | レベル5 | 正誤の確認が速い成果物は監査に載せやすい |
| 検証容易性 | 中 | レベル4 | 抜き取り確認で足りるなら例外運用に移せる |
| 検証容易性 | 低 | レベル3 | 全件の精査が必要なものは抜き取り確認へ移行できない |

**追加規則**: 誤り影響度=高 かつ 検証容易性=低 の組み合わせは、上限をレベル2に引き下げる（重大な誤りを見つけにくい業務では、AIに完成品を作らせない）。

**判定手順**: 4軸それぞれの上限を表から引き、追加規則を適用したうえで、最小値を判定レベル（上限）とする。

適用例を次に示す（4軸の並びは誤り影響度・判断定型度・入力構造化度・検証容易性）。

| タスク例 | 4軸の評価 | 軸別上限 | 判定 |
|---|---|---|---|
| 請求書発行 | 高・高・高・高 | レベル3・5・5・5 | レベル3 |
| 問い合わせ一次応答 | 中・高・高・中 | レベル4・5・5・4 | レベル4 |
| 契約書の確認 | 誤り影響度=高、検証容易性=低 | 追加規則に該当 | レベル2 |

## 昇格条件

段階導入では、下のレベルでの運用実績が次のレベルの前提になる。以下の条件がすべて満たされたときに1レベル上げる。件数・期間・比率は初期の目安であり、案件ごとに顧客と合意して確定する（AI・コンサルタントの推測で確定しない）。

| 昇格 | 満たすべき条件（すべて） |
|---|---|
| レベル1→2 | `../06_データ取り扱いチェック/データ取り扱いチェック.md` の判定を通過している / 指示文の初版がある / 出力を毎回全件確認する運用が決まっている |
| レベル2→3 | 指示文が固定化され、直近の運用で内容の変更が不要になっている / 直近20件で、AI出力への内容の手直しが2割未満（誤字調整など形式の手直しは数えない） / 出力の型・承認手順・差戻し基準が文書化されている |
| レベル3→4 | 直近50件または1か月の差戻し率が5%未満 / 発生した差戻し理由が、すべて例外条件として列挙済みである / 自動起動と例外通知の仕組みが動いている / 処理件数・失敗率が記録されている |
| レベル4→5 | 直近3か月の例外率が1%未満で、例外の見逃しがない / 異常検知と自動停止の動作を確認済みである / 監査手順と監査記録の保管先が決まっている |

昇格条件は「AIをどこまで使いこなせているか」の判定軸でもある。指示文を固定化できているか（レベル2→3）、指示なしで動く仕組みを組めているか（レベル3→4）、無人でも異常に気づける仕組みがあるか（レベル4→5）が、それぞれの節目となる。

## 判定の手順

1. `../03_業務棚卸しテンプレート/業務棚卸しテンプレート.md` の記入表で、対象タスクの4軸評価を確認する。評価は担当者への質問で得た回答の記録であること（AI・コンサルタントの推測評価は禁止。未確認軸が残るタスクは判定を保留する）
2. 軸別上限表と追加規則を適用し、最小値を判定レベル（上限）とする
3. 「現在地レベルの診断」の質問と判定条件で現在地を確定する
4. 段階導入の原則に従い、目標レベルは現在地の1つ上を基本とする。判定レベル（上限）を超える目標を設定しない
5. 判定結果と導入計画を顧客に提示し、合意を得る（consult-level-agree）
6. 運用開始後は昇格条件の実績（手直し率・差戻し率・例外率）を記録し、条件を満たしたら次の1レベルへの引き上げを検討する

## 課題からレベル軌道を書く手順

課題（タスク）を与えられたとき、その課題のL1〜L5の具体像を書く手順。consult-level-agree のレベル軌道記述もこの手順に従う。

1. 課題を「入力 → 処理 → 出力 → 確認」の4要素に分解する（何を受け取り、何をして、何を出し、誰が確かめるか）
2. 「レベルを分ける6つの観測点」の各レベルの値を、その課題の固有名詞に置き換える
3. レベルごとに1〜2文で書く。**「何をきっかけに・誰が・AIが何をして・人が何を確認するか」**を必ず含める
4. 判定レベル（上限）を超えるレベルには「上限超過」と超過理由を明記する（参考として姿を書いてもよい）

### 記入例: 請求書発行（4軸=高・高・高・高、上限レベル3）

| レベル | この課題での姿 |
|---|---|
| レベル1（人手のみ） | 担当者が受注一覧と単価表を突き合わせ、請求書を手作業で作成して発行する |
| レベル2（AI補助） | 担当者が受注データを対話型AIに渡して請求書の下書きを作らせ、金額と宛先を自分で修正して仕上げる。指示文は毎回担当者が書く |
| レベル3（人が承認するAI実行） | 担当者が固定指示でAIを起動すると、AIが受注データから請求書を完成形で作成する。担当者は金額突合で合否だけを判定し、合格分を発行する。不合格分は差し戻してAIに作り直させる |
| レベル4（例外のみ人介入） | 上限超過（誤り影響度=高のため、人の承認を外せない）。参考: 月次締めをきっかけにAIが自動作成・発行し、単価表にない品目など列挙済みの例外だけ人が処理する姿 |
| レベル5（完全自動） | 上限超過（同上）。参考: 発行から入金確認まで無人で流れ、人は月次監査のみ行う姿 |

## 境界事例

判断に迷いやすい実例を示す。

### 請求書作成

誤り影響度は高いが、金額突合を数分で確認できるため検証容易性は高い。軸別上限表では誤り影響度=高の上限レベル3が最小となり、レベル3（人が承認するAI実行）が上限となる。

### 問い合わせ一次応答

判断定型度・入力構造化度が高く、誤り影響度は中にとどまる。軸別上限は誤り影響度=中と検証容易性=中のレベル4が最小となる。段階導入の原則に従いレベル3で開始し、昇格条件（差戻し率5%未満・例外条件の列挙）を満たしてからレベル4へ上げる。

### 契約書の確認
