# コンサル業務フロー

本ドキュメントは道具箱全体の入口となる業務フロー定義である。AI推進コンサルタントが1件の案件を進める際、7つのフェーズをこの順に通過する。各フェーズの目的・入力・出力・使う道具・完了条件を定義し、案件を進める際の判断基準とする。

```
+------------+   +------------+   +------------+   +------------+   +------------+   +------------+   +------------+
| 1.課題受付 |-->| 2.現場     |-->| 3.業務の   |-->| 4.AI化     |-->| 5.施策設計 |-->| 6.実行・   |-->| 7.効果測定 |
| ・事例照合 |   | ヒアリング |   | 棚卸し     |   | レベル判定 |   | ・WBS化    |   | 稼働管理   |   | ・定着化   |
| 振り分け   |   | 課題の特定 |   | 作業の分解 |   | 対応範囲判定|   | タスク展開 |   | 進捗管理   |   | 効果検証   |
+------------+   +------------+   +------------+   +------------+   +------------+   +------------+   +------------+
```

## 各フェーズの定義

### フェーズ1 課題の受付と事例照合

- 目的: 課題を自由文のまま受け付け、既知課題を即処方する
- 入力: 顧客からの課題（自由文）
- 出力: 課題受付簿と振り分け結果・現場への確認依頼リスト（ヒアリング行き課題の判断材料を依頼する文書）
- 使う道具: consult-intake スキル + [../09_解決事例集/解決事例集.md](../09_解決事例集/解決事例集.md) + consult-request-list スキル（確認依頼リストの生成）
- 完了条件: 全課題が「処方済み」または「ヒアリング行き」に振り分けられている

### フェーズ2 現場ヒアリング

- 目的: 現場の担当者が何に困っているか、どの業務に時間を取られているかを特定する
- 入力: 顧客企業から提供される業務説明・担当者へのアクセス・確認依頼で現場から届いた実物・事実
- 出力: 課題サマリーと成功の定義（導入前後の実測で比較できる数値目標）
- 使う道具: consult-hearing スキル（.claude/skills/consult-hearing/。選択式ヒアリングで課題と4判定軸を収集する） + consult-analyze スキル（リポジトリの技術構成を解析し裏付けに使う）
- 完了条件: 課題サマリーが顧客担当者との間で合意されている

### フェーズ3 業務の棚卸し

- 目的: フェーズ2で特定した業務を、事実として記録できる単位（1タスク=1つの完結した作業）に分解する。時間の定量は導入前実測（フェーズ7の効果測定）で扱う
- 入力: フェーズ2の課題サマリー
- 出力: 業務棚卸し表
- 使う道具: [03_業務棚卸しテンプレート/業務棚卸しテンプレート.md](../03_業務棚卸しテンプレート/業務棚卸しテンプレート.md)（consult-hearing が対話の中で記入まで行う）
- 完了条件: 棚卸し表の全タスクに担当者と根拠・出所（誰から・いつ・何で確認したか）が記録されている

### フェーズ4 AI化レベル判定

- 目的: 棚卸し表の各タスクについて、どこまでAIに任せられるかを判定する
- 入力: フェーズ3の業務棚卸し表
- 出力: 判定済み棚卸し表（各タスクにレベル1〜5のいずれかが付与された表）
- 使う道具: [02_AI化レベル基準/AI化レベル基準.md](../02_AI化レベル基準/AI化レベル基準.md)（consult-hearing が判定まで行う） + consult-level-agree スキル（レベル合意HTMLの生成と合意）。顧客説明に `docs/02_AI化レベル基準/AI化レベル基準.html` を使う
- 完了条件: 棚卸し表の全タスクにレベルが付与され、判定根拠が記録されている。かつ、consult-level-agree でタスクごとの現在レベル・目標レベルが顧客と合意されている

### フェーズ5 施策設計・WBS化

- 目的: 判定レベルごとに定番化されたタスク一式をWBS（Work Breakdown Structure。作業を階層分解した一覧表）として展開する
- 入力: フェーズ4の判定済み棚卸し表（現在レベル・目標レベル列が合意済み） + 着手前提チェックリスト（[08_着手前提チェックリスト/着手前提チェックリスト.md](../08_着手前提チェックリスト/着手前提チェックリスト.md)）
- 出力: WBS
- 使う道具: [04_レベル別タスクテンプレート/レベル別タスクテンプレート.md](../04_レベル別タスクテンプレート/レベル別タスクテンプレート.md) + consult-wbs スキル（Excel と wbs.json の両形式を出力）
- 完了条件: WBSの全タスクに担当者・期限が設定され、顧客と共有できる状態になっている。着手前提5分類が未充足のタスクは前提整備タスクとして計画されている。WBSが clients/ に出力されている（顧客共有Excel または内部管理wbs.json）

### フェーズ6 実行・稼働管理

- 目的: WBSに基づき実行を進め、進捗と課題を管理する
- 入力: フェーズ5のWBS
- 出力: 更新された進捗状況・課題一覧
- 使う道具: wbs-edit（共同編集WBSの編集・本番反映。顧客と共有する管理表向き）/ single-file-wbs（単一HTMLのWBSビューア。コンサル自身の案件・AI稼働管理向き）
- 完了条件: WBSの全タスクが完了または引き継ぎ状態になっている

### フェーズ7 効果測定・定着化

- 目的: AI化前後の削減時間・品質を検証し、現場に定着させる
- 入力: フェーズ6で完了したタスクの実績データ
- 出力: 効果測定結果（before/after比較）
- 使う道具: [05_効果測定テンプレート/効果測定テンプレート.md](../05_効果測定テンプレート/効果測定テンプレート.md)。測定対象はフェーズ2の成功の定義と一致させる
- 完了条件: before/after比較が完了し、顧客担当者に報告済みである

## フェーズ横断の確認事項

フェーズ2からフェーズ5のいずれの段階であっても、顧客データをAIに渡す前には必ず [06_データ取り扱いチェック/データ取り扱いチェック.md](../06_データ取り扱いチェック/データ取り扱いチェック.md) のチェックを通す。ヒアリング内容・棚卸し表・WBSのいずれにも顧客固有の情報が含まれうるため、AIへの入力前の確認を省略しない。
