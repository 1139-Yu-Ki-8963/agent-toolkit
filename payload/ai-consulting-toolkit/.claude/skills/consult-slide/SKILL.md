# 解説スライド生成（consult-slide）

与えたトピックを決まった手順で横1枚（16:9）のスライド風解説HTMLにする実行スキル。図は inline SVG で自作し、観点レビュー表による検査に合格したものだけを `docs/10_解説スライド集/` に登録する。他のモデル・別セッションでも同水準の解説HTMLを再現できるよう、デザインはテンプレートの部品カタログに、品質判定は観点レビュー表に固定してある。

## Phase 1: 要件ヒアリング

このスキルで最も重要な工程。5カテゴリのヒアリング項目を埋め、充足ゴールを満たしてから次の Phase に進む。ヒアリング不足のまま生成した図は「正しく見えるが仕組みと矛盾する」欠陥（因果矛盾・数値矛盾）の温床になる。

完了条件: 5カテゴリすべてにユーザー発話由来の回答（または仮置きの明示）が記録され、充足ゴール3判定が可、スライドキーが確定していること。

consult-level-agree から起動された場合は、レベル合意HTML・業務棚卸し表・軌道記述（合意済みのL1〜L5説明文）を Read する。これらを5カテゴリの初期値とする。説明対象は合意済みのレベル軌道、読み手は顧客の現場と経営層とする。結論は現在地から目標レベルへの移行像、提示文脈はレベル合意の説明、制約は1課題1枚とする。この場合、Step 1-1・1-2 でのユーザーへの確認は初期値の修正の要否のみに短縮できる。

### Step 1-1: ヒアリング5カテゴリの整理と初回質問

以下の5カテゴリについて、まずユーザーの依頼文から埋まる項目を埋める。consult-level-agree からの引き継ぎ時は上記の初期値を使う。埋まらないカテゴリだけを AskUserQuestion（1回最大4問）で確認する。

| カテゴリ | 確認する内容 | 図への反映先 |
|---|---|---|
| 説明対象 | 仕組みの事実。因果関係・数値条件・例外（例: 枠は開始から5時間で回復する） | 例示数値の検算・イベント線の位置 |
| 読み手 | 誰に見せるか（経営層/現場/技術者）・前提知識 | 用語レベル・補足の量 |
| 結論 | 読み手に残したい1文・読後に取ってほしい行動 | タイトル・サブタイトル・図の型 |
| 提示文脈 | どの課題への提示か・利用場面 | 蓄積簿の「提示する課題の例」列 |
| 制約 | ページ数（既定1）・含める/除く内容・業務文脈の範囲（営業日のみ等） | 軸の範囲・パネル構成 |

説明対象の仕組み詳細（数値・条件・因果）は選択肢で聞ける形になりにくいため、AskUserQuestion ではなく通常の対話で自由記述の確認をしてよい。

**入力**: ユーザーの依頼文（またはconsult-level-agreeからの引き継ぎ情報）
**完了**: 5カテゴリの充足状況（埋まった値とその出所）を整理したこと

### Step 1-2: 充足判定と追加質問

充足ゴールは「Phase 2 の構造設計を推測なしで開始できる状態」。次の3判定がすべて可になるまで、不足カテゴリを AskUserQuestion で追加確認する。

1. 図の型が選べるか（結論と説明対象の性質が分かっている）
2. 例示数値の検算ができるか（仕組みの因果・数値条件が引き出せている）
3. 読み手に合う語彙レベルが決められるか

ヒアリングは最大3ラウンド。3ラウンドで埋まらない項目は推測で埋めず「仮置き」と明示して先へ進み、Phase 6 の報告に仮置き一覧を含める。

**入力**: Step 1-1 の充足状況
**完了**: 充足ゴール3判定がすべて可、または仮置きを明示したこと

### Step 1-3: スライドキーの決定

`docs/10_解説スライド集/解説スライド集.md` を Read し、既存キーとの重複・類似を照合する。キーは連番禁止・内容を要約した意味語（例: `claude-cursor使い分け-リミット観点`）。類似スライドが既にある場合は新規作成か既存更新かを AskUserQuestion で確認する。

**入力**: 蓄積簿 `解説スライド集.md`、Step 1-1 の説明対象
**完了**: 重複のないスライドキーが確定したこと

## Phase 2: 構造設計

結論から図の型を選び、軸・パネル分割・読み順・例示数値を決める。HTMLを書き始める前に構造を固定する。

完了条件: 図の型・軸の割当・パネル分割・ステップ番号・例示数値が言語化されていること。

### Step 2-1: 図の型選択

結論の性質から図の型を1つ選ぶ。時間経過を含む説明（切り替え・サイクル・推移）→ 時系列。複数対象の優劣・違い → 対比。手順・分岐 → フロー。成熟度・段階 → 段階図。型に迷う場合は「読み手が結論を確かめるために目で追う動き」を想像し、その動線に合う型を選ぶ。

**入力**: Step 1-1 の結論1文
**完了**: 図の型を1つ選び、選定理由を1文で言語化したこと

### Step 2-2: 軸とパネル分割の決定

時間の話なら横軸は必ず時間にする（対象の並置に使わない）。異なる時間スケール（時間単位と週単位等）が混在する場合は1本の軸に混ぜず、スケールごとにパネルを分ける。時間軸の範囲は業務文脈に合わせる（営業日の説明なら土日を除く、日中の説明なら業務時間帯）。

**入力**: Step 2-1 の図の型
**完了**: 各パネルの横軸・範囲・分割数が確定したこと

### Step 2-3: 例示数値の検算

図に入れる具体値（時刻・回数・割合）を、説明したい仕組みの因果と突き合わせて検算する。例: 「開始から5時間でリセットされる枠」を描くなら、使い切り時刻はリセット時刻より前でなければならない。矛盾する値は見た目が自然でも採用しない。

**入力**: Step 2-2 の軸設計、説明対象の仕組み
**完了**: 全ての例示数値が仕組みと矛盾しないことを検算したこと

### Step 2-4: 読み順番号の設計

読み手が図を追う順序を1〜3個程度のステップ番号に分解する。番号は図中のチップと下部の要約文で同じ番号・同じ色を使う前提で設計する。

**入力**: Step 2-1〜2-3 の構造
**完了**: ステップ番号と各番号の意味（利用・切替・復帰等）が確定したこと

## Phase 3: HTML生成

テンプレートの部品カタログとデザイントークンでHTMLを組む。

完了条件: 自己完結の単一HTMLが生成されていること。

### Step 3-1: テンプレートの読み込み

`.claude/skills/consult-slide/references/slide-template.html` を Read する。デザイントークン・部品カタログ・使用規則（冒頭コメント）を確認する。

**入力**: `references/slide-template.html`
**完了**: トークンと部品の一覧、使用規則を確認したこと

### Step 3-2: 部品組み立てと本文執筆

テンプレートを骨格として Phase 2 の構造をHTMLに起こし、Write で書き出す。厳守事項:

- 先頭に `<meta charset="utf-8">` を置く
- `:root` のデザイントークンは変更しない。色は必ずトークン経由で使い、hex を本文に手書きしない（ツール識別色の追加のみ、トークン定義への追記として許可）
- 図はすべて inline SVG。外部リソース（CDN・Webフォント・外部画像・fetch）は一切参照しない
- 実在ツール・製品は実物ロゴを想起させる簡略SVGマーク+名称で描く。抽象図形の発明は禁止
- 変化の引き金（上限到達・締切・承認等）はイベント線+バッジの構造部品で描く
- 「尽きた・満杯・ゼロ」等の状態はメーター100%+警告色で極端に描く
- 複数ページの場合はテンプレートのページ送り部品を使う（1ページなら削除）

**入力**: Step 3-1 のテンプレート、Phase 2 の構造設計
**完了**: 単一HTMLファイルが書き出されていること

## Phase 4: レンダリング検証

実際に描画して観点レビュー表の全行を照合する。コードレビューだけでの合格判定を禁止する（文字切れ・重なりは実描画でしか発見できない）。

完了条件: 観点レビュー表の全行が PASS になっていること。

### Step 4-1: 実描画とスクリーンショット

一時HTTPサーバ（ポートは使用中でない番号をその都度選ぶ）で対象HTMLを配信し、Playwright 等のブラウザツールで 1280px 幅のスクリーンショットを取得する。ブラウザツールが使えない環境ではその旨を明示し、取得できる代替手段（別のレンダラ等）で必ず実描画を確認する。検証後はサーバを停止し、一時スクリーンショットを削除する。

**入力**: Phase 3 で生成したHTML
**完了**: 1280px 幅の実描画スクリーンショットを取得したこと

### Step 4-2: 観点レビュー表の全行照合

`.claude/skills/consult-slide/references/slide-review-checklist.md` を Read する。全観点キーについて PASS / NG を記録する。表示検査の観点はスクリーンショットを見て判定する。特に「文字切れ-全数目視」は全テキスト要素を1つずつ確認する。

出力前に `.claude/skills/shared/references/customer-output-checklist.md` の自問チェックを実施し、全項目「該当なし」であることを確認する（案件固有スライドは顧客に渡る成果物のため必須。汎用スライドにも同チェックを適用する）。

**入力**: Step 4-1 のスクリーンショット、観点レビュー表
**完了**: 全観点キーの PASS / NG が記録されていること

### Step 4-3: NG修正と再検証

NG があれば原因を直し、Step 4-1 から再実行する。ループ設計（後述）の上限・停止条件に従う。チェック結果の記録なしに「完了」と報告することを禁止する。

**入力**: Step 4-2 の NG 一覧
**完了**: 全行 PASS、またはループ停止条件に到達し状況を報告したこと

## Phase 5: 保存・蓄積簿更新

完了条件: 汎用スライドはdocs/10_解説スライド集/への保存と蓄積簿登録が完了していること。案件固有スライドはclients/<案件名>/への保存が完了していること（蓄積簿登録は対象外）。

### Step 5-1: スライドの保存

顧客名・案件情報を含む案件固有スライドは `clients/<案件名>/レベル目標_<課題キー>_<YYYYMMDD>.html` として Write で保存する。含まない汎用スライドは従来どおり `docs/10_解説スライド集/<スライドキー>/解説スライド.html` として Write（または Bash の cp）で保存する。ファイル名はそれぞれの規定形式に固定する。汎用スライドはこのPC固有の絶対パス・顧客固有の情報が含まれていないことを確認する。

**入力**: Phase 4 で PASS したHTML、案件固有か汎用かの判別
**完了**: clients/またはdocs/10_解説スライド集/の規定パスに保存されていること

### Step 5-2: 蓄積簿への登録

汎用スライドのみ `docs/10_解説スライド集/解説スライド集.md` のスライド一覧表に Edit で1行追記する（スライドキー / 説明対象 / 提示する課題の例 / 相対リンク / 登録日）。リンク先の実在を Bash の ls で確認する。案件固有スライドは顧客固有情報をdocs/10に置かないため蓄積簿登録をしない。

**入力**: Step 5-1 の保存パス、Phase 1 の入力情報
**完了**: 汎用スライドは蓄積簿に1行追記されリンク先が実在すること。案件固有スライドは蓄積簿登録をしないこと

### Step 5-3: 検査記録の保存

汎用スライドは `docs/10_解説スライド集/<スライドキー>/検査記録.md` に保存する。案件固有スライドは `clients/<案件名>/検査記録_レベル目標_<課題キー>_<YYYYMMDD>.md` に保存する。内容: 観点レビュー表の照合結果（全行の判定）・反復回数と各回の修正内容・**タスク限定の観点**（このスライド固有の検査事項）を記録する。例: リミットの因果条件・営業日のみの範囲制約・ツール識別マークの定義。タスク限定の観点は次回このスライドを更新する時の再検査項目になる。

**入力**: Phase 4 の照合記録
**完了**: 検査記録.mdがclients/またはdocs/10_解説スライド集/の規定パスに保存されていること

## Phase 6: 完了報告と観点の育成

完了条件: 検査結果一覧を添えた報告を提示し、新規指摘の昇格判定を終えていること。

### Step 6-1: 検査結果付きの報告

保存パス・蓄積簿の登録内容（案件固有スライドは登録対象外）・観点レビュー表の照合結果（全行の PASS 一覧）・ヒアリングの仮置き一覧（あれば）を添えて報告する。

**入力**: Phase 4 の照合記録、Phase 5 の保存結果
**完了**: 報告をユーザーに提示したこと

### Step 6-2: 観点の共通化判定（レビュー表を育てる）

生成〜報告で受けたレビュー指摘・修正で言語化された原則を列挙し、昇格基準3つを満たすものを `references/slide-review-checklist.md` に Edit で追記する。

1. 別トピックのスライドでも再発しうる（タスク非依存）
2. 意味語キー + 検査内容 + NG例の形式で書ける
3. 実描画またはファイル検査（grep 等）で機械的・目視的に検査可能

満たさない指摘は Step 5-3 の検査記録.md のタスク限定観点として保持する（捨てない）。追記・保持の判定結果を報告に含める。レビュー表の既存観点の削除は禁止（検査の弱体化になるため。表現の具体化・NG例の追加は可）。

**入力**: 本セッションのレビュー指摘一覧、`references/slide-review-checklist.md`
**完了**: 新規指摘ごとの昇格/保持の判定結果を報告に含めたこと

## ループ設計

| 要素 | 内容 |
|---|---|
| 反復条件 | Phase 1: 充足ゴール3判定に不可があれば追加質問する ／ Phase 4: NG 観点が1つでもあれば修正して Step 4-1 から再検証する |
| 上限回数 | Phase 1: 最大3ラウンド ／ Phase 4: 最大5回 |
| 停止条件 | 収束停止: 充足ゴール3判定可（Phase 1）・全行 PASS（Phase 4）／ リソース上限: 各上限到達（Phase 1 は仮置き明示で続行）／ 発散検知: 同一観点の NG が2回連続 |

上限・発散で停止した場合は残 NG と原因をユーザーに報告し、判断を仰ぐ。

## Gotchas

- ヒアリングを省略して生成に入らない。原型スライドでは仕組みの因果（開始から5時間でリセット）の確認が浅く、「5時間使ってから切り替える」という矛盾した図を一度作った
- レンダリング検証を省略しない。原型スライドでも、コード上は正しく見えて実描画で「Cursor」の文字が帯からはみ出す欠陥が見つかった
- 例示数値の検算を省略しない。「5時間リミットの説明なのに5時間使ってから切り替える」図は、見た目が自然なまま仕組みと矛盾する
- 図の型より先に色やレイアウトを選ばない。構造設計（Phase 2）を飛ばすと「左から右に変わる」以上の情報がない図になる
- ダークモード対応は不要。顧客提示前提のビジネス配色（白面+ネイビー系）に固定する
