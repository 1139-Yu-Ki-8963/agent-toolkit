# リポジトリ規約整備 詳細手順（repo-analysis-review-procedure）

`consult-repo-rules/SKILL.md` の各PhaseからStep単位の詳細（本文・入力・完了条件）を分離したもの。SKILL.md本体の骨子とあわせて参照する。内容はSKILL.mdから移動したものであり、削除・要約はしていない。

## Phase 1: リポジトリ解析

### Step 1-1: 言語・枠組みの確認

Bash で依存管理ファイル（`package.json` 等）の有無を確認し、Read で内容を読み、主要言語・フレームワークを把握する。

**入力**: `<repo>` のルートパス
**完了**: 主要言語・依存管理ファイル・フレームワークが把握されている

### Step 1-2: ディレクトリ構成の確認

Bash で `find` / `ls` を実行し、ソース・テストの配置単位を把握する。

**入力**: `<repo>` のルートパス
**完了**: ソース・テストの配置単位が把握されている

### Step 1-3: 代表コードの読み込み

Read で代表的な既存コードを 3〜5 ファイル読み込み、命名・ファイル分割の単位・エラー処理の書き方・テストの書き方・コメントの流儀を言語化する。

**入力**: Step 1-2 で把握したディレクトリ構成から選んだ代表ファイル
**完了**: 暗黙の様式が言語化されている

### Step 1-4: 既存CLAUDE.md/rulesの確認

Glob で `.claude/rules/` の有無を確認し、Read で CLAUDE.md・既存 rule.md の内容を読む。既存があれば重複を避け、不足分のみ補う方針にする。

**入力**: `<repo>/CLAUDE.md`、`<repo>/.claude/rules/`
**完了**: 既存規約の有無と内容、不足分のみ補う方針が把握されている

## Phase 2: 規約の起草（分割規律つき）

### Step 2-1: 規約テンプレートの読み込み

Read で `references/code-rule-template.md` を読み込む。

**入力**: `references/code-rule-template.md`
**完了**: テンプレートの骨組みを把握している

### Step 2-2: 抽出様式の規約起草

Write で Phase 1 で抽出した様式をテンプレートの骨組みに沿って `<repo>/.claude/rules/<topic>-rules/rule.md` として起草する。1規約1トピックで分ける。

**入力**: Phase 1 で言語化した暗黙の様式、Step 2-1 で読み込んだテンプレート
**完了**: トピック単位で分割された規約案が起草されている

### Step 2-3: 行数の確認

Bash で `wc -l` を実行し、各規約本文が50行以内の目安に収まっているか確認する。

**入力**: Step 2-2 で起草した規約ファイル
**完了**: 各規約本文が50行以内の目安に収まっていることを確認している

## Phase 3: 顧客との合意

### Step 3-1: 守る様式/直す癖の確認

AskUserQuestion で抽出様式を「守りたい様式」と「直したい癖」に分けて顧客に確認する。

**入力**: Phase 2 で起草した規約案
**完了**: 「守りたい様式」「直したい癖」の区分について顧客から回答を得ている

### Step 3-2: 合意内容の反映

Edit で顧客が「直したい癖」と答えた項目を「現状」でなく「あるべき」の形に書き換える。

**入力**: Step 3-1 で得た顧客の回答
**完了**: 「守りたい様式」「直したい癖」の区分について顧客の合意が得られている

## Phase 4: レビュー・テストの仕組み設置

### Step 4-1: 検証スキルテンプレートの読み込み

Read で `references/review-test-skill-template.md` を読み込む。

**入力**: `references/review-test-skill-template.md`
**完了**: テンプレートの骨組みを把握している

### Step 4-2: 観点表の生成

Read で Phase 2 の規約案を再読し、規約の各項目に対応するレビュー観点表を作成する。

**入力**: Phase 2 で起草した規約案
**完了**: 規約の各項目に対応する観点表が作成されている

### Step 4-3: 検証スキルの配置

Write で読み込んだテンプレートを `<repo>/.claude/skills/review-and-test/SKILL.md` として配置する。

**入力**: Step 4-1 のテンプレート
**完了**: `<repo>/.claude/skills/review-and-test/SKILL.md` が配置されている

### Step 4-4: プレースホルダの置換

Edit で配置したテンプレート内のプレースホルダを Step 4-2 で生成した観点表の実値に置換する。

**入力**: Step 4-2 で生成した観点表
**完了**: 観点表を実値に置換した review-and-test スキルが配置されている

## Phase 5: 実機検証

### Step 5-1: 違反サンプル作成

Write で顧客リポジトリの「直したい癖」を意図的に再現した違反サンプルを一時ファイルとして作成する。

**入力**: Phase 3 で合意した「直したい癖」
**完了**: 違反サンプルの一時ファイルが作成されている

### Step 5-2: 検証スキル実行

Skill(review-and-test) を実行し、Phase 4 で設置したレビュー・スキルの観点表でその違反が検出されることを確認する。

**入力**: Step 5-1 の違反サンプル、Phase 4 で配置した review-and-test スキル
**完了**: 違反サンプルの検出を確認している

### Step 5-3: 規約行数の確認

Bash で `wc -l` を実行し、Phase 2 で配置した規約本文それぞれの行数が目安（50行以内）に収まっているか確認する。

**入力**: Phase 2 で配置した規約ファイル
**完了**: 規約本文の行数上限を確認している

### Step 5-4: 後片付け

Bash で検証に使った一時ファイルを削除する。

**入力**: Step 5-1 で作成した一時ファイル
**完了**: 一時ファイルを削除している

## Phase 6: 運用の引き継ぎ

### Step 6-1: 運用手順の説明

AskUserQuestion で規約の育て方（癖・違反再発時の追記ループ、50行超過時の分割タイミング）を顧客に説明し、合意を得る。

**入力**: Phase 2〜5 で確立した規約・検証スキルの構成
**完了**: 規約の育て方（追記ループ・分割タイミング）を顧客に案内している
