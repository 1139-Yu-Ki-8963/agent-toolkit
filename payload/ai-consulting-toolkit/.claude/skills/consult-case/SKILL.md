# 解決事例の登録（consult-case）

案件で解決した汎用課題を、次の案件・顧客説明に再利用できる形で蓄積する。事例は「1課題=1フォルダ」で、説明HTMLと実物資材をセットにし、フォルダごと顧客に渡せる状態を保つ。

## Phase 1: 事実確認

登録対象の課題・原因・解決策が検証済み（実環境で動作確認済み・実測値あり）かどうかを AskUserQuestion で確認する。未検証・推測混じりと判明した場合は登録を拒否し、検証を先に行うよう案内する。

完了条件: 検証済みか否かを AskUserQuestion で確認し、未検証なら登録を停止していること。

### Step 1-1: 検証済みかの確認
AskUserQuestion（header: 検証状況 / question: 登録対象の課題・原因・解決策は実環境で検証済みですか / options: 検証済み（実測値あり）, 未検証）で確認する。未検証・推測混じりと判明した場合は登録を拒否し、検証を先に行うよう案内する。
**入力**: なし（対話開始時点）
**完了**: 検証済みか否かが確認され、未検証なら登録を停止している

## Phase 2: 課題キーの決定

内容を要約した意味語キー（連番禁止）を決める。`docs/09_解決事例集/解決事例集.md` の事例一覧表を Read し、既存キーとの重複・類似を AskUserQuestion で確認する。類似事例があれば新規登録でなく既存事例の更新を提案する。

完了条件: 新規キーの重複・類似有無を AskUserQuestion で確認し、対応方針（新規登録/既存更新）が決まっていること。

### Step 2-1: 蓄積簿の読み込み
Read `docs/09_解決事例集/解決事例集.md` で既存の事例一覧表を確認する。
**入力**: なし（ドキュメント参照）
**完了**: 既存キー一覧を把握している

### Step 2-2: 課題キーの重複確認
内容を要約した意味語キー（連番禁止）の候補を決め、Step 2-1 の一覧表と重複・類似がないか照合する。
**入力**: Step 2-1 の既存キー一覧、登録対象の課題内容
**完了**: 重複・類似の有無が判明している

### Step 2-3: キーの合意
AskUserQuestion（header: 課題キー / question: このキーで新規登録してよいですか。類似事例がある場合は既存更新にしますか / options: 新規登録する, 既存事例を更新する）で対応方針を確定する。
**入力**: Step 2-2 の重複・類似判定結果
**完了**: 新規キーと対応方針（新規登録/既存更新）が決まっている

## Phase 3: 導入手段の確認

解決策に環境構築（設定配置・ツール導入・hook設置）が伴う場合、導入を実行するスキルが道具箱にあるか確認する。無ければ事例登録の前に導入スキルを実装する（読み物の手順書で代替しない）。事例フォルダに資材を置くのは、顧客に渡す前提で作った一般資材（案件で新規作成した図・雛形等）がある場合のみ。`docs/09_解決事例集/<課題キー>/assets/` を作るのはその場合に限る。

完了条件: 導入手段の有無を確認し、無い場合は導入スキルの実装を先に済ませていること。

### Step 3-1: assets ディレクトリ作成
解決策に環境構築が伴い、かつ顧客に渡す前提の一般資材（案件で新規作成した図・雛形等）がある場合のみ、Bash で `mkdir -p docs/09_解決事例集/<課題キー>/assets` を実行する。
**入力**: Step 2-3 で確定した課題キー
**完了**: 資材がある場合に assets ディレクトリが作成されている（資材がなければこのステップはスキップ）

### Step 3-2: 実物資材のコピー
Bash で汎用化済みの資材ファイルを `docs/09_解決事例集/<課題キー>/assets/` に `cp` する。コンサル環境の設定・辞書の実物はコピーしない。
**入力**: Step 3-1 で作成した assets ディレクトリ、汎用化済みの資材ファイル
**完了**: 資材がある場合に assets/ へのコピーが完了している（資材がなければこのステップはスキップ）

## Phase 4: 説明HTMLの生成

`.claude/skills/consult-case/references/case-template.html` を `docs/09_解決事例集/<課題キー>/解決事例.html` にコピーする。`{{...}}` プレースホルダを全て実データで置換する。§1〜§10の構成・`<style>`・ズーム script は変更しない。図は inline SVG のみ。§7（実現するAI化レベル）には到達レベル（L1〜L5）と判断根拠を必ず記入する。レベルは `docs/02_AI化レベル基準/AI化レベル基準.md` の基準で判定する。§10（導入手段）には、何が導入され、導入後に誰が何をする運用になるかを顧客の言葉で書く。内部スキル名（consult- で始まる名称）は書かず、導入スキル名は蓄積簿（docs/09_解決事例集/解決事例集.md）の「導入スキル（内部）」列に記録する。

完了条件: プレースホルダを実データに置換した解決事例.html が生成されていること（§7の実現するAI化レベルを含む）。

### Step 4-1: テンプレートの読み込み
Read `.claude/skills/consult-case/references/case-template.html` でプレースホルダ構成・§1〜§10の構成を確認する。
**入力**: なし（テンプレート参照）
**完了**: プレースホルダ一覧と§構成を把握している

### Step 4-2: プレースホルダの置換
Write で `docs/09_解決事例集/<課題キー>/解決事例.html` に、Step 4-1 のテンプレートをコピーしたうえで `{{...}}` プレースホルダを全て実データで置換する。§1〜§10の構成・`<style>`・ズーム script は変更しない。§7（実現するAI化レベル）には、`docs/02_AI化レベル基準/AI化レベル基準.md` の基準で判定した到達レベル（L1〜L5）を記入する。そのレベルと判断した根拠（誰が・何を確認する運用か）もあわせて必ず記入する。§10（導入手段）には、何が導入され、導入後に誰が何をする運用になるかを顧客の言葉で書く。内部スキル名（consult- で始まる名称）は書かず、導入スキル名は蓄積簿（docs/09_解決事例集/解決事例集.md）の「導入スキル（内部）」列に記録する。
**入力**: Step 4-1 のテンプレート、Phase 1〜3 で確定した内容
**完了**: プレースホルダを実データに置換した解決事例.html が生成されている（§7の実現するAI化レベルを含む）

## Phase 5: 出力前チェック

Phase 4 で生成した解決事例HTMLについて、`.claude/skills/shared/references/customer-output-checklist.md` の共通チェックを全項目実施する。1つでも該当したら Phase 4 に戻って修正し、再チェックする。

完了条件: 共通チェック全項目が「該当なし」で確認済みであること。

完了条件: 事例一覧表に1行追記されていること。

### Step 6-1: 蓄積簿の読み込み

**入力**: なし（ドキュメント参照）

### Step 6-2: 1行追記

## 予想を裏切る挙動

- コンサル環境の設定・辞書・利用者情報などの実物を資材（assets/）に置かない。汎用化した資材のみ格納する
- `case-template.html` の構成・`<style>`・ズーム script は変更しない。プレースホルダ置換のみ行う
- §7（実現するAI化レベル）の記入漏れは許容しない。到達レベルは `docs/02_AI化レベル基準/AI化レベル基準.md` の基準で判定し、推測で記入しない

## 完了報告

`.claude/skills/shared/references/completion-report-format.md` の作業報告型骨格に従う。固有の検証行として、登録した事例キー・フォルダパス・蓄積簿（解決事例集.md）の更新有無の3点を追加報告する。

## 重要ルール

- 検証済みの設定・数値のみ記載する。推測の効果・数値の記載は禁止する
- このPC固有の絶対パスをHTMLに書かない
- 事例フォルダは単体で顧客に渡せる状態を保つ
- テンプレートのレイアウト（構成・style）を崩さない
- コンサル環境の設定・辞書の実物をコピーしない。環境構築の再現は導入スキルが担う
