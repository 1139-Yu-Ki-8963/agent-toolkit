Claude Code プロジェクト健康診断オーケストレーター
consult-ai-config-doctor

対象プロジェクトの .claude/ 配下（CLAUDE.md・Rules・Skills・Hooks・Subagents）と自動化・運用衛生を含む計 6 ドメインを読み取り専用で並列診断し、D〜S の 5 段階（星 1〜5、ソシャゲのレアリティ表示風）で採点したうえで AI 駆動レベル 1〜5 と連動させ、承認済み処方箋（fix）の適用まで一気通貫で担うオーケストレーター。Claude Code 設定の運用実態に特化した診断であり、汎用のコード品質診断（テストカバレッジ・複雑度・脆弱性スキャン等）は対象外。

§1概要

本スキルは Claude Code 設定の運用実態に特化した診断ツールである。対象プロジェクトの CLAUDE.md・Rules・Skills・Hooks・Subagents に加え、それらを支える自動化・運用衛生を合わせた計 6 ドメインを対象に、次の 3 段階を一気通貫で行う。

── 全体の流れ ─────────────────────────────────────────

  ① 診断    Phase 1〜3: 6 ドメインを読み取り専用で並列診断し、
            D〜S の 5 段階でグレード付けする
  ② 承認    Phase 4: 検出された処方箋（fix）を一覧提示し、
            ユーザーが適用対象を選ぶ
  ③ 適用    Phase 5〜6: 承認済み処方箋のみ適用し、
            変更ドメインを再診断して diff を提示する
スキップ条件:「診断」という単語だけでは発火しない。単一のスキル・ルール・フックのレビューや読み取り専用の dry-run 診断は managing-agent-configs の担当であり、本スキルはプロジェクト横断の 6 ドメイン診断のみを扱う。

§2出力

診断が完了すると、対象プロジェクトの .claude/diagnosis/ 配下に次の成果物が生成され、承認済みの処方箋があれば Phase 6 で再診断・diff 提示まで完了する。

判定票 Markdown
6 ドメイン（CLAUDE.md・Rules・Skills・Hooks・Subagents・自動化・運用衛生）それぞれの D〜S グレード・星数と、findings（CRITICAL/WARN/INFO 件数・該当箇所・充足率）を references/report-format.md のフォーマットでまとめた文書。
ダッシュボード HTML
assets/dashboard-template.html を土台に scripts/render-dashboard.mjs が生成する、外部 CDN 依存のない自己完結の一覧画面。総合グレード・AI 駆動レベル・ドメイン別星数を視覚的に確認できる。
診断 JSON（<timestamp>.json / latest.json）
集計結果を機械可読な形で保存したもの。次回診断時にこのファイルの有無で diff モードが自動判定され、前回との改善・悪化・変化なしの比較が Phase 3・Phase 6 で提示される。
総合グレード・AI 駆動レベル・処方箋適用結果
全 Phase 完了時点で確定する最終値。総合グレード（S〜D）・対応する AI 駆動レベル（1〜5）に加え、Phase 4 で承認された処方箋の適用件数（承認件数 / 適用成功件数 / 失敗件数）が完了報告に記載される。

§3Iron rules（読み取り専用契約）

SKILL.md 冒頭に定義される最上位契約。本文の他の記述より優先する。

条項内容
診断は読み取り専用Phase 1〜3（診断フェーズ）は読み取り専用。Write は報告物と診断 JSON の保存のみに使い、設定ファイル本体・アセットの変更は一切行わない
fix は承認制fix（変更）は Phase 4 でユーザーが承認した処方箋のみを対象とし、Phase 5 で実行する。未承認の処方箋を先回りして適用しない
迂回禁止permission ガード・hook が発火したら迂回せず停止して報告する。block を回避する目的でのコマンド言い換え・sandbox 解除は行わない
シークレット非引用.env の中身・API キー・トークン等はパスと存在の言及にとどめ、値そのものを報告物に転記しない

この契約により、「診断のつもりが気づかず設定を書き換えていた」という事故を構造的に防ぐ。診断（読む）と是正（書く）は Phase 4 の承認を境に明確に分離されている。

§46 ドメインと診断対象

診断対象は次の 6 ドメイン。各ドメインは Phase 1 で独立した investigator が並列に担当し、findings JSON（CRITICAL / WARN / INFO 件数・充足率）を返す。

ドメイン診断対象
CLAUDE.mdプロジェクト固有の CLAUDE.md の記述内容・規約準拠（個別チェック項目の正本は references/claude-md-checks.md）
Rules.claude/rules/ 配下のルール定義・機械強制の整合性
Skillsskills/ 配下のスキル定義・frontmatter・段階的開示の遵守状況
Hookshook スクリプトの配置・出力スキーマ・exit code 規約準拠
Subagentsサブエージェント定義の単一責任・禁止ツール制御
自動化・運用衛生上記 5 ドメインを横断する自動化の冪等性・品質ゲートの有無

各ドメインの共通禁止ブロック（読み取り専用契約・許可コマンド・シークレット非引用・除外パス）とドメイン別の診断定義は references/domain-briefs.md が正本。対象プロジェクトに .claude/ が存在しない場合、全ドメインが D 判定になるのは仕様どおりの正常動作であり、検査不能をバグとして扱わない。

§5グレード判定式（D〜S・星1〜5）

各ドメインの findings（CRITICAL / WARN 件数）と充足率（推奨項目のうち満たした割合）から、優先順位付きで D〜S の 5 段階を判定する。星表示はソシャゲのレアリティ表示に倣い、D=★1（最低）から S=★5（最高）まで直感的に序列が伝わる形にしてある。

優先順グレード星条件
1D★☆☆☆☆ドメイン不在・検査不能（例: CLAUDE.md が存在しない）
2C★★☆☆☆CRITICAL ≧ 1
3S★★★★★CRITICAL = 0 かつ WARN = 0 かつ充足率 ≧ 90%
4A★★★★☆CRITICAL = 0 かつ WARN ≦ 3
5B★★★☆☆CRITICAL = 0 かつ WARN ≧ 4
強制降格: いずれかのドメインで CRITICAL ≧ 1 件を検出した場合、そのドメインのグレードは WARN 件数・充足率に関わらず C 以下に強制される。CRITICAL を「軽微」として A・B・S に押し上げる判定は禁止する。

総合判定の算出

  • 総合判定は 6 ドメインの最低グレードを基準とする
  • 最低グレード以外の 5 ドメインの平均が最低グレードより 2 段階以上高い場合に限り、総合判定を 1 段階のみ加点する（例: 最低が D で他平均が B なら D→C の 1 段階のみ）
  • 段階順序（低い→高い）: D < C < B < A < S

充足率・境界値の詳細な算出方法は references/grading-rules.md を正本とする。本節はその要約であり、齟齬が生じた場合は同ファイルを優先する。

§6AI 駆動レベル 1〜5 との連動

総合判定から、推奨される AI 駆動レベルの上限を導出する。出典は ai-consulting-toolkit の AI 化レベル基準（誤り影響度・判断定型度・入力構造化度・検証容易性の 4 判定軸）。

総合判定AI 駆動レベルレベル名称
Sレベル5完全自動
Aレベル4例外のみ人介入
Bレベル3人が承認する AI 実行
Cレベル2AI 補助
Dレベル1人手のみ

4 判定軸と設定層整備の対応関係

設定層の整備状況が上がるほど、対応する判定軸の評価が実態として引き上がる。整備が伴わないままレベルだけを引き上げると、軸の実態と乖離した過大評価になる。

4 判定軸対応する設定層整備整備が効く理由
検証容易性Hooks 整備（JSON 出力スキーマ・exit code 規約準拠）機械強制による即時 block/advisory があれば、成果物の正誤を全件人手精査なしで確認できる
判断定型度Rules 整備（違反手順網羅・scope 適合性）判断基準が rule.md に文書化・機械参照可能な形で定型化されるほど、都度判断への依存が下がる
入力構造化度Skills 整備（frontmatter 必須項目・Phase/Step 構造）TRIGGER/SKIP・完了条件が構造化されるほど、ユーザー発話から実行手順への変換が整備済みに近づく
誤り影響度Subagents 整備（単一責任・禁止ツール制御）+ 自動化衛生（冪等性・品質ゲート）責務分離と冪等性・品質ゲートが揃うほど、誤った成果物の影響範囲が局所化される

§7Phase 構成（診断フロー）

共通の前段（進捗の可視化・スコープ確認・判定基準ロード）の後、Phase 1〜6 を順に進める。Phase 番号は 1 始まりの正整数のみを使う。

Phase内容完了条件
1並列診断: investigator 6 体をドメインごとに並列起動し、.claude/diagnosis/raw/ に findings JSON を保存6 ドメイン分の findings JSON が回収されている
2集計判定: scripts/aggregate-findings.mjs で normalize + グレード判定 + AI 駆動レベル確定集計 JSON が生成され、スクリプトが exit 0 で終了している
3レポート出力: 判定票 Markdown + ダッシュボード HTML。diff モード時は前回比較も表示。診断 JSON を .claude/diagnosis/<timestamp>.json と latest.json に保存判定票 Markdown・ダッシュボード HTML・診断 JSON の 3 成果物が出力先に存在する
4fix 承認: 処方箋一覧（risk・所要時間・期待効果を列挙）を提示し、適用対象を確定。処方箋 0 件なら Phase 6 へ直行適用対象の承認集合が確定している
5fix 適用: worker-sonnet が safe → careful → surgery の順に処方箋を適用する。surgery 分類は 1 件ごとに個別確認承認済み処方箋がすべて適用済み、または失敗が記録済み
6再診断・完了報告: 変更のあったドメインのみ再診断し、diff を表示してから完了報告を出すGoal: 総合グレードと AI 駆動レベルが確定し、diff モード時は前回比較が提示されている

対象プロジェクトの .claude/diagnosis/latest.json が存在すれば diff モードが有効化され、前回診断との比較が Phase 3・Phase 6 で提示される。診断 JSON は本スキル側ではなく 対象プロジェクト側 の .claude/diagnosis/ に保存されるため、複数プロジェクトを診断しても本スキルのディレクトリは汚れない。

§8処方箋（fix）と diff モード

各 finding は CRITICAL / WARN のみ fix（risk・所要時間・期待効果・適用プロンプト）を必須で持つ。INFO は fix: null でよい。

risk 分類適用順序・確認方法
safe最初に一括適用。影響範囲が明確で副作用がない変更
carefulsafe の次に適用。既存動作への影響を確認しながら進める
surgery1 件ごとに個別確認を取ってから適用する。分類を飛ばして一括適用しない

risk 分類・適用手順の詳細は references/fix-playbook.md が正本。

diff モード

初回診断（latest.json 不在）では prev_date: null / changes: [] / overall_from: null として扱う。2 回目以降の診断では前回診断と比較し、ドメイン別の改善・悪化・変化なしと総合判定の推移（overall_from → overall_to）を Phase 3・Phase 6 で提示する。

§9サブエージェント委任とループ設計

呼び出し箇所subagent_type期待返却値
Phase 1（ドメイン診断）investigator ×6（並列）ドメインごとの findings JSON（CRITICAL/WARN/INFO 件数・該当箇所・充足率）
Phase 5（fix 適用）worker-sonnet適用結果（成功/失敗）と検証出力（適用前後の差分要約）
Phase 6（再診断）investigator変更ドメイン分の findings JSON

ループ設計（Phase 5 → Phase 6）

適用・再診断サイクルは最大 2 周まで反復する。反復条件は「Phase 6 の再診断でグレード未改善または新規 CRITICAL 検出時」。停止条件は次の 3 パターンのうち 2 つ以上で成立する。

  • 収束停止: 全ドメインで CRITICAL 0 到達
  • 発散検知: 再診断でグレードが改善しない（同一グレードのまま停滞）
  • リソース上限: 2 周に到達

検証役（Phase 6 の investigator）は適用役（Phase 5 の worker-sonnet）と分離する。同一エージェントに適用と検証の両方をやらせると、自己評価バイアスで見落としが生じるため。
