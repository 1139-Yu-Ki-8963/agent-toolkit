# 処方箋適用手順書（fix-playbook）

Phase 4（fix 承認）・Phase 5（fix 適用）で worker-sonnet が処方箋を適用する際の手順。承認されていない処方箋の先回り適用は禁止（SKILL.md Iron rules 2）。

## 1. リスク 3 分類

処方箋ごとに `fix.risk` を以下 3 区分のいずれかに分類する。分類は findings JSON 生成時点（Phase 1・Phase 6 の investigator）で確定させ、Phase 5 では分類を信頼して適用順序のみ制御する。

| 区分 | 定義 | 例 |
|---|---|---|
| safe | 設定ファイルへの追記・文言修正等、既存挙動に影響しない可逆変更 | hook の `timeout` フィールド追加、CLAUDE.md への陳腐化記述削除 |
| careful | 既存の値・構造を変更し、変更前後の差分レビューを要する | rule.md の観点表修正、SKILL.md の Phase 構造変更 |
| surgery | 削除・移動・権限変更を伴う、複数ファイル・複数ドメインにまたがる構造変更 | hook script の配置移動、ファイル削除、rules ディレクトリの再編 |

## 2. 適用順序

1. **safe → careful → surgery** の順に適用する。逆順・混在適用は禁止
2. safe 分類はバッチ適用してよい（Phase 4 で一括承認された場合）
3. careful 分類は各処方箋の適用前後で diff を提示し、想定外の副作用がないか確認する
4. surgery 分類は **1 件ごとに `AskUserQuestion` で個別確認** を取る。バッチ承認・一括適用は禁止する

## 3. 削除の扱い（隔離方式）

処方箋が「削除」を伴う場合、ファイルシステムから即時削除しない。

1. 対象プロジェクトの `.claude/diagnosis/quarantine/<timestamp>/` へ `git mv` または `mv` で移動する
2. 同ディレクトリの `MANIFEST.tsv` に隔離記録を追記する（列: 元パス・隔離日時・処方箋キー・隔離理由）
3. `MANIFEST.tsv` が存在しない場合はヘッダ行（`元パス\t隔離日時\t処方箋キー\t隔離理由`）付きで新規作成する
4. 隔離後、対象プロジェクトが正常動作することを確認してから、隔離ファイルの完全削除はユーザー判断に委ねる（本スキルは隔離までを担当し、完全削除は行わない）

## 4. ガード・hook 発火時の対応

Phase 5 の適用中に permission ガード・hook（PreToolUse block 等）が発火した場合:

1. **迂回しない**: sandbox 解除・コマンド言い換え・別経路での同等操作は行わない
2. 適用を停止し、発火した hook 名・注入タグ・block 理由を記録する
3. 当該処方箋を「適用失敗」として記録し、失敗理由（hook 名・タグ）を集計 JSON の該当 `prescriptions[]` エントリに残す
4. 残りの承認済み処方箋（発火と無関係なもの）は継続して適用してよい

## 5. 適用後の検証

各処方箋の適用完了後、当該ドメインのみを対象に再診断する（Phase 6 と同じ investigator 委任）。

1. 適用直後に該当ドメインの findings を再取得する
2. 適用前後で該当 CRITICAL / WARN が解消されているか確認する
3. 新規 CRITICAL が発生していないか確認する（副作用の検出）
4. 検証結果は集計 JSON の `diff.changes[]` に反映する

## 6. 適用不能時の扱い

処方箋の `fix.prompt` を適用しても対象ファイルが存在しない・構造が想定と異なる等で適用できない場合、その場で仕様を推測して代替修正を行わない。適用失敗として記録し、原因（ファイル不在・構造不一致・権限不足等）を明記して報告する。
