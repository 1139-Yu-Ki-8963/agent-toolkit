# WBS展開（consult-wbs）

コンサル業務フロー（docs/01_コンサル業務フロー/コンサル業務フロー.md）のフェーズ5（施策設計・WBS化。WBSはWork Breakdown Structure、作業を階層分解した一覧表）を担う。判定レベル付き業務棚卸し表を入力に、タスクテンプレートを展開してWBSを生成する。

## Phase 1: 入力確認

`clients/` 配下を Glob で `業務棚卸し表_*.md` を探す。複数見つかった場合は AskUserQuestion で対象案件を選ばせる。見つけた棚卸し表を Read し、判定レベル列が全行埋まっているか確認する。未判定行が残っている場合は、consult-hearing での判定完了を促して処理を停止する。

`clients/` 配下の `着手前提チェックリスト_*.md` も探して Read する。存在しない場合、または状態「未」の分類が残るタスクがある場合は、その旨を AskUserQuestion で提示し、前提整備タスクを含めて展開するか確認する。

完了条件: 棚卸し表の判定レベル列が全行埋まっていること、着手前提チェックリストの確認状況を確認済みであること。

### Step 1-1: 棚卸し表の検索

Glob で `clients/` 配下の `業務棚卸し表_*.md` を探す。複数見つかった場合は AskUserQuestion で対象案件を選ばせる。

**入力**: `clients/` 配下のディレクトリ構成
**完了**: 対象案件の業務棚卸し表のパスが1件に定まっていること

### Step 1-2: 棚卸し表の読み込み

Step 1-1 で特定した棚卸し表を Read する。判定レベル列が全行埋まっているか確認する。未判定行が残っている場合は、consult-hearing での判定完了を促して処理を停止する。

**入力**: Step 1-1 で特定した業務棚卸し表のパス
**完了**: 棚卸し表を読み込み、判定レベル列の充足状況を確認したこと

### Step 1-3: 判定レベルの充足確認

Step 1-2 の結果を確定し、未判定行が無いことを確認する。未判定行があれば処理を停止する。

**入力**: Step 1-2 の判定レベル列確認結果
**完了**: 判定レベル列が全行埋まっていることを確定したこと

### Step 1-4: 着手前提チェックリストの読み込み

Glob で `clients/` 配下の `着手前提チェックリスト_*.md` を探し、見つかれば Read する。

**入力**: `clients/` 配下のディレクトリ構成
**完了**: 着手前提チェックリストの有無と内容を確認したこと

### Step 1-5: 未充足タスクの確認

チェックリストが存在しない場合、または状態「未」の分類が残るタスクがある場合は、その旨を AskUserQuestion で提示し、前提整備タスクを含めて展開するか確認する。

**入力**: Step 1-4 の着手前提チェックリストの内容
**完了**: 着手前提チェックリストの確認状況を確認済みであること

## Phase 2: タスク展開

`docs/04_レベル別タスクテンプレート/レベル別タスクテンプレート.md` を Read する。棚卸し表の各タスクについて、判定レベルがレベル1以上のものを対象に、共通フェーズ（必須）に該当レベル節のタスク群を加えて適用する。展開時は次を守る。

- テンプレ中の「○○」はすべて実業務名に置換する
- L2（親タスク）は名詞（成果物名）で書く
- L3（実行タスク）は動詞で終える。5営業日以内の粒度を守り、超える場合は分割する
- 各フェーズの末尾に予備日タスクを1件入れる
- レビューと修正のタスクを省かない
- 着手前提チェックリストで「未」が残るタスクは、`docs/04_レベル別タスクテンプレート/レベル別タスクテンプレート.md` の共通フェーズ（着手前提の整備）を必ず先頭に置き、「未」の分類に対応する整備タスクに用意する人を担当者として割り当てる

完了条件: 対象タスク全件にテンプレート項目が展開され、予備日タスクとレビュー・修正タスクが漏れていないこと。

### Step 2-1: タスクテンプレートの読み込み

`docs/04_レベル別タスクテンプレート/レベル別タスクテンプレート.md` を Read する。

**入力**: `docs/04_レベル別タスクテンプレート/レベル別タスクテンプレート.md`
**完了**: 共通フェーズとレベル別タスク群の構成を確認したこと

### Step 2-2: 共通フェーズの適用

棚卸し表の各タスクについて、判定レベルがレベル1以上のものを対象に、共通フェーズ（必須）を適用する。着手前提チェックリストで「未」が残るタスクは、共通フェーズ（着手前提の整備）を先頭に置き、「未」の分類に対応する整備タスクに用意する人を担当者として割り当てる。

**入力**: Step 2-1 のテンプレート、Phase 1 の棚卸し表・着手前提チェックリスト
**完了**: 対象タスクに共通フェーズが適用され、着手前提整備タスクが先頭に配置されていること

### Step 2-3: レベル別タスクの適用

各タスクの判定レベルに該当するレベル節のタスク群を共通フェーズに加える。L2（親タスク）は名詞（成果物名）、L3（実行タスク）は動詞で終え、5営業日以内の粒度を守り超える場合は分割する。各フェーズの末尾に予備日タスクを1件入れ、レビューと修正のタスクを省かない。

**入力**: Step 2-2 適用後のタスク一覧
**完了**: 対象タスク全件にレベル別タスクが展開され、予備日タスクとレビュー・修正タスクが漏れていないこと

### Step 2-4: 業務名への置換

**入力**: Step 2-3 で展開したタスク一覧
**完了**: 抽象表現「○○業務の〜」が残っていないこと

### Step 4-2a: Excel出力（顧客共有Excel）

wbs-generator の入力 JSON を組み立てて一時ファイルに保存し、Bash で次を実行する。

```
~/cloned-repos/wbs-skills/wbs-generator/.venv/bin/python ~/cloned-repos/wbs-skills/wbs-generator/scripts/generate_wbs.py --input <json> --output clients/<案件名>/WBS_<案件名>_<YYYYMMDD>.xlsx
```

入力 JSON の構造:

```json
{
  "project_name": "",
  "assignees": [],
  "phases": [],
  "tasks": [
    {
      "l1_phase": "",
      "l2_parent": "",
      "l3_task": "",
      "assignee": "",
      "status": "未着手",
      "start_date": "",
      "end_date": "",
      "actual_end_date": null,
      "predecessor": null,
      "remarks": ""
    }
  ]
}
```

`.venv` が存在しないエラーが出た場合は、`python3 -m venv ~/cloned-repos/wbs-skills/wbs-generator/.venv` で仮想環境を作成し、`~/cloned-repos/wbs-skills/wbs-generator/.venv/bin/pip install openpyxl` を実行してから再実行する（macOS のシステム Python は外部管理のため直接 pip 導入できない）。

**入力**: Step 4-1 で選択した出力形式（Excel を含む場合）、Phase 2・3 で確定したタスク一覧・体制・日程
**完了**: `clients/<案件名>/WBS_<案件名>_<YYYYMMDD>.xlsx` が生成されていること

### Step 4-2b: wbs.json出力（内部管理wbs.json）

Write で single-file-wbs のデータ形式で `clients/<案件名>/wbs.json` を生成する。構造:

```json
{
  "holidays": [],
  "projects": [
    {
      "name": "<案件名>",
      "milestones": [],
      "tasks": []
    }
  ]
}
```

`tasks` はL1（第1階層）・L2（第2階層）・L3（リーフ）の3階層で構成する。リーフタスクの形式:

```json
{
  "id": "1.1.1",
  "name": "",
  "qty": 1,
  "hours": 0,
  "assignee": "",
  "plan": { "start": "", "end": "" },
  "actual": { "start": null, "end": null },
  "note": "",
  "_level": "レベルN"
}
```

`hours` は目安日数×8を入れる。`id` は "1"・"1.1"・"1.1.1" 形式（single-file-wbs の仕様が定める階層の構造番号）。`_` 始まりのキーはビューアが無視するカスタムキーで、`_level` に判定レベルを記録する。

完了報告には、`~/cloned-repos/single-file-wbs/wbs_viewer.html` を Chrome で開き「ファイルを開く」で wbs.json を読み込む閲覧方法を含める。

**入力**: Step 4-1 で選択した出力形式（wbs.json を含む場合）、Phase 2・3 で確定したタスク一覧・体制・日程
**完了**: `clients/<案件名>/wbs.json` が生成されていること

完了条件: 選択した出力形式（Excel/wbs.json/両方）のファイルが対象パスに生成されていること。

## Gotchas

- `wbs-generator` の `.venv` が無いエラーが出た場合は、Phase 4 記載の bootstrap 手順（`python3 -m venv` での作成 → `pip install openpyxl`）で作ってから再実行する。手順を省略しない
- 目安日数はテンプレートの参考値。AI が推測で確定せず、必ず AskUserQuestion で顧客と確定する

## 重要ルール

- テンプレの抽象表現（「○○業務の〜」）をそのまま出力しない。必ず実業務名に置換する
- タスク展開の正本は `docs/04_レベル別タスクテンプレート/レベル別タスクテンプレート.md`、判定基準の正本は `docs/02_AI化レベル基準/AI化レベル基準.md`。両者の内容を本スキルに転記しない
- 実績管理で single-file-wbs を使う場合、AIの稼働実績はリーフタスクの `_ai` カスタムキーに追記できる
- 着手前提5分類が「揃った」または「不要」になっていないタスクの導入作業を、前提整備タスクより前に配置しない
