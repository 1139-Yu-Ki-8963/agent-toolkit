# Reviewing 参照資料

brain が Reviewing モードで使う検証観点と判定基準。

## 検証の5観点

### 1. 完全性
- 対象ファイルが全て処理されたか
- 変更パターンが全箇所に適用されたか
- 漏れがないか（grep で未変換パターンを検索）

### 2. 正確性
- 変更内容が指示通りか
- 意図しない副作用がないか
- 既存機能が壊れていないか

### 3. 一貫性
- 全ファイルで同じパターンが適用されているか
- 命名規則・フォーマットが統一されているか

### 4. 構造健全性
- ファイル構造が壊れていないか（構文エラー等）
- 参照先が全て存在するか（リンク切れ等）

### 5. 可逆性
- 変更が git で追跡可能か
- 問題発覚時に revert 可能か

## 判定基準

| 判定 | 条件 |
|---|---|
| PASS | 全5観点で問題なし |
| FAIL（軽微） | 一貫性・構造の軽微な問題 → worker-haiku で修正可能 |
| FAIL（要判断） | 正確性・完全性の問題 → worker-sonnet で修正 |
| FAIL（設計見直し） | 根本的な方針の問題 → brain で再計画 |

## プロンプト品質の検証（旧 prompt-master の知識）

プロンプトやエージェント定義をレビューする際の8観点:
1. 明確な指示（曖昧性の排除）
2. 役割の明確化（専門性・立場）
3. 具体的な情報提供（前提・制約）
4. 回答例の提示（良い例・悪い例）
5. 出力形式の指定（スキーマ）
6. 区切り記号（構造の可読性）
7. 文字数の指定（詳細度の制御）
8. 条件の指定（分岐・例外処理）
