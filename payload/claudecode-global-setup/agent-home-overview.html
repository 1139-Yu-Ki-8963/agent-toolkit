1設計思想

agent-home は「AI エージェントの行動品質を、人間の注意力に頼らず構造で担保する」ための設定基盤。3つの原則で設計されている。

作りっぱなしを許さない

スキルや hook を作ったら review → test まで自動連鎖する。テスト完了マーカーがないと commit できない。

機械強制

規約は rule.md に書くだけでは守られない。PreToolUse / PostToolUse / Stop の各タイミングで hook が発火し、違反を block または advisory で通知する。

段階的開示

常時注入する情報は最小限。スキルの詳細手順、レビュー観点、テスト手順は必要になった時点で references/ からロードする。

2ファイル構成

設定資産は ~/agent-home/ と ~/.claude/ の2箇所に配置される。後者から前者への symlink で統合される。

├── rules/ ← 行動規範（always/ + scoped/）
│ ├── always/ ← 常時適用
│ └── scoped/ ← パス条件付き適用
├── skills/ ← ワークフロー定義

├── tools/

33層構造

agent-home の設定体系は「何を守るか」「どう実行するか」「どう強制するか」の3層で構成される。

Rules
行動規範
「何を守るか」を定義する。 rule.md に規約本文、naming-values.txt / prh.yml に値を分離。always（常時注入）と scoped（パス条件付き注入）の2分類。規約違反の検知手順と修正手順も rule.md 内に記載する。
▼
Skills
ワークフロー
「どう実行するか」を定義する。 SKILL.md に Phase / Step の手順、references/ に詳細規約・テンプレート、scripts/ に自動化スクリプトを配置。/skill-name で起動し、create → review → test の連鎖で品質を担保する。
▼
Hooks
機械強制
「どう強制するか」を実装する。 settings.json の hooks セクションに登録された shell スクリプト群。Claude Code の各イベント（PreToolUse / PostToolUse / Stop 等）で発火し、規約違反を exit 2 で block するか、advisory メッセージで通知する。

Rules が「法律」、Skills が「業務マニュアル」、Hooks が「検問所」に相当する。法律だけでは守られず、検問所だけでは何を守るかわからない。3層が揃って初めて機能する。

4Rules 体系

Rules は ~/agent-home/rules/ に配置され、~/.claude/rules/ への symlink を通じて Claude Code に注入される。

always（常時注入）

全プロジェクト・全セッションで適用される普遍的な規約。

カテゴリ内容代表的な hook
agent/コーディング原則、サブエージェント委任規約check-main-agent-direct-work.sh
naming/コミット・ブランチ・識別子の命名規約dispatch-pre-bash-checks.sh
response/応答言語・文体、先送り禁止、依頼文禁止check-no-deferral-stop.sh
review-checklist/textlint 辞書、略称説明、意味キー命名check-term-explanation.sh
placement/ファイル配置ガード、ディレクトリ構成ガードcheck-playwright-filename.sh
gate/Phase/Step タスク登録ゲートcheck-phase-entry-tasks.sh
session/スキル発火ログ、セルフ設定自動削除skill-log-recorder.sh

scoped（パス条件付き注入）

特定のファイルパスに一致する場合のみ注入される規約。

カテゴリ内容トリガー条件
agent-config/hook 設計規約、CLAUDE.md ガード、配置判定*.sh / CLAUDE.md 編集時
dev-flow/実装フローゲート、worktree 規約~/Projects/ 配下のコード編集時
review-checklist/コード・文書・レポートのレビュー観点*.html / *.md / *.tsx 等の編集時
tooling/シェルスクリプト設計判断記載、curl 外部通信制御*.sh 作成時 / curl 実行時

rule.md の構造

各 rule.md は統一構造を持つ: 規約本文 → 機械強制テーブル（timing / script / tag / 挙動） → 違反検知時の手順 → プロジェクト上書き可否 → 関連リンク。値の分離が必要な場合は同ディレクトリの *-values.txt（非注入サイドカー）に置く。

6Hooks 体系

Hooks は settings.json の hooks セクションに登録された shell スクリプト群。Claude Code の各イベントで自動発火する。

イベントタイプと用途

イベントタイミング主な用途
PreToolUseツール実行前命名チェック、commit gate、ファイル配置ガード、サブエージェント選択制御
PostToolUseツール実行後先送り表現検出、識別子命名チェック、managed ファイル編集検知
UserPromptSubmitユーザー発話時サブエージェント委任ヒント、スキル作成ルール注入
Stop応答完了時依頼文検出、先送り検出、略称無断使用検出
SessionStartセッション開始時古いマーカー掃除、ネスト設定ディレクトリ削除
SessionEndセッション終了時セッションマーカーのクリーンアップ

block と advisory

exit 2 — block

ツール実行を中止させる。英語コミットメッセージ、テスト未完了の commit、対話必須コマンドの発行など、「やってはいけない操作」を物理的に止める。

exit 0 — advisory

実行は許可しつつ注意喚起する。命名リマインド、公開可否レビュー促進、ブランチ命名ガイドなど、「知っておくべき情報」を additionalContext で注入する。

rules-bash-runner（集約ランナー）

PreToolUse(Bash) の hook 10本を1プロセスに集約し、パターンマッチで該当スクリプトだけを起動する。非該当コマンドでは fork ゼロで即座に抜ける。ファイル不在時は安全にスキップする（[ -f "$1" ] || return 0 ガード）。

7サブエージェント体系

全サブエージェントは4つの役割に分類される。合否（PASS / FAIL）を宣言できるのは判定系のみ。

計画系
brain
タスク分解・複数案比較・作業指示の組み立て。計画のみを返し、実行や合否判定は行わない。
合否宣言: 不可
実行系
worker-sonnet / worker-haiku
確定済み方針のファイル反映（sonnet）、コマンド実行と結果報告（haiku）。方針の決定はメインセッションが行う。
合否宣言: 不可
調査系
investigator / researcher / plan-comprehension-prober
ローカルの事実収集（investigator）、外部情報検索（researcher）、計画の初見読解（prober）。証拠付きの事実を返す。
合否宣言: 不可
判定系
code-reviewer / document-reviewer / business-content-reviewer / report-reviewer
成果物を rules に照合し、PASS / FAIL を宣言する唯一の分類。コード・文書・ビジネス資料・調査報告をそれぞれ専門的に検証する。
合否宣言: 可

委任の基本原則: 判断と反映の分離

「内容を決める」のはメインセッション（大規模な分解は brain）、「決めた内容を反映する」のは実行系、「確かめる」のは判定系。worker への委任プロンプトには確定済みの実体をベタ書きで埋め込む。「考えて直して」と丸投げしない。

8マーカー機構

managed ファイル（skills / rules / hooks の設定資産）の変更を管理し、レビュー・テスト未完了の commit を防止する仕組み。

ファイル編集
→
needed マーカー
PostToolUse で自動作成
→
review + test
managing-agent-configs
→
report + test-passed
ハッシュ記録
→
commit gate
ハッシュ照合

commit gate の検証項目

検証内容
test-passed 実在テスト完了マーカーが存在し、空でないこと
report 実在レビュー・テスト結果のレポートが存在し、REVIEW-TEST-VERDICT: PASS 行を含むこと
鮮度検証report の mtime が needed マーカーより新しいこと（編集後に再レビューが必要）
report ハッシュ照合report の SHA256 が test-passed の REPORT_SHA256= 行と一致すること
staged ハッシュ照合staged の各 managed ファイルの内容ハッシュが test-passed に記録された値と一致すること

マーカーはセッション固有のディレクトリに配置され、SessionEnd hook で自動クリーンアップされる。

設置の動作

ソース設置先方式
agent-home/**~/agent-home/**copy（上書き）
claude-config/agents/**~/.claude/agents/**copy（上書き）
claude-config/CLAUDE.md~/.claude/CLAUDE.mdskip-if-exists
claude-config/settings-hooks.json~/.claude/settings.jsonmerge（hooks + outputStyle + permissions）
claude-config/statusline.py~/.claude/statusline.pycopy
—~/.claude/rules → ~/agent-home/rulessymlink 作成
—~/.claude/skills → ~/agent-home/skillssymlink 作成

settings.json のマージは hooks セクションの重複判定（コマンド文字列の正規化比較）、outputStyle の補完、permissions の配列マージ（allow / deny / ask）、additionalDirectories の __HOME__ → 実パス解決を行う。
