# コードレビュー共通基準（CODE-REVIEW-STANDARDS）

プロジェクトに依存しないコード変更の合格基準。コードの作成時に守るべき規約であると同時に、レビュー時の照合観点表そのものである。レビュー実行時は code-reviewer サブエージェントが本ファイルを Read し、基準ごとに判定する。

## 重要度の定義

| 重要度 | 基準 |
|---|---|
| 重大な問題 | マージ不可。セキュリティ・データ整合性の破壊 |
| 警告 | マージ可だが修正必須。コード品質・規約違反・アーキテクチャ逸脱 |
| 提案 | 改善推奨。動作に影響しないパフォーマンス・可読性の改善 |

重大な問題の指摘には「何が起きるか」「なぜ起きるか（該当 file:line を最低 1 件）」「対応案」の 3 点を必ず添える。欠ける場合は警告に格下げする。

## 基準（レビュー時はこの全カテゴリを確認する）

### セキュリティ（重大な問題）

- SQL インジェクション: 文字列結合によるクエリ組み立てがない
- クロスサイトスクリプティング: 未エスケープのユーザー入力を HTML へ出力していない
- 認証・認可バイパス: 権限チェックの欠落・迂回経路がない
- 認証情報のハードコード: API キー・パスワード・トークンがコードに埋め込まれていない

### データ整合性（重大な問題）

- 型不整合: 呼び出し側と定義側の型が一致している
- インターフェース破壊: 既存の公開関数・API の引数・戻り値を互換性なく変更していない

### コード品質（警告）

- 関数サイズ: 1 関数が過大でない（目安 60 行超は分割を検討）
- ネスト深度: 条件分岐・ループの入れ子が深すぎない（目安 4 段以上は構造を見直す）
- デバッグコード残留: console.log・print・コメントアウトされた試行コードが残っていない
- 可読性: 6 ヶ月後の読み手が前提知識なしで意図を追える。説明のない暗黙の前提・トリッキーな実装がない

### エッジケース（警告）

- null・undefined・空文字・空配列・0 件データの扱いが定義されている
- 境界値: 上限・下限・ちょうど境界の入力で正しく動く
- 並行処理: 競合状態・二重実行・非同期の順序依存がない

### 汎用規約（警告）

- 命名: 変数・関数名が役割を表している。誤解を招く名前・意味のない省略がない
- 型定義: any 型・型注釈の省略で型検査を骨抜きにしていない
- import 方向: レイヤー間の依存方向に反する import がない

### アーキテクチャ（警告）

- レイヤー分離: UI・ビジネスロジック・データアクセスの責務が混在していない
- 単一責務: 1 モジュール・1 クラスが複数の理由で変更される構造になっていない

### 公開インターフェース（警告）

- 命名一貫性: 同一 API 群でエンドポイント・関数・引数の命名規則が揃っている（同義語の発明・単複混在がない）
- エラー応答: 失敗時の応答形式・例外の型が API 群内で統一されている
- 公開範囲: 内部専用のはずの関数・エンドポイントを不用意に公開していない

### 依存追加（警告）

- 必要性: 新規依存（パッケージ・ライブラリ）の追加に、既存依存・標準機能で代替できない理由がある
- 規模適合: 1 機能のために巨大な依存を入れていない
- ライセンス: 新規依存のライセンスがプロジェクトの配布形態と矛盾しない

### データベース変更（警告）

- 破壊的変更の明示: カラム削除・型変更・制約追加など後方互換を壊す migration は、影響と移行手順が説明されている
- ロールバック可能性: migration に戻し手順（down）があるか、不可逆な場合はその旨が明示されている
- 詳細なスキーマ規約（命名・インデックス方針等）はプロジェクト受け口に定義する

### パフォーマンス（提案）

- N+1 クエリ: ループ内での逐次クエリ発行がない
- 非効率アルゴリズム: データ量に対して明らかに不適切な計算量の処理がない
- 不要な再レンダリング: React 等で props・state の設計不備による無駄な再描画がない（レンダーごとの新規オブジェクト生成・巨大コンポーネントの一枚岩化など）

### 過剰設計（警告）

- YAGNI 違反: 仮想的な将来要件のための抽象化・設定項目・拡張ポイントを作っていない
- 起こり得ないシナリオへのエラー処理・フォールバック・バリデーションを足していない（検証はシステム境界のみ）
- 一回きりの処理に不要なヘルパー・レイヤーを作っていない

### 推奨事項（提案）

- マジックナンバー: 意味のある定数に名前が付いている
- API ドキュメント: 公開インターフェースの変更に説明が追従している

### テキスト品質（警告）

- 誤字・脱字: JSX・HTML テンプレート内のテキストノードに誤字がない
- 類似文字の混入: 見た目が似た別漢字・JIS 外文字が混入していない（例: 「激魪」→「激闘」）
- 変更範囲外のテキスト変更: 依頼範囲外で変更されたテキストを列挙し、意図的か確認する

### JSX 構文（警告）

- 空 JSX 式: `{ }` はスペースを出力しない。スペースが必要な箇所は `{' '}` を使う

## レビュー結果の要件

- 全カテゴリを確認し、指摘 0 件の場合も「全カテゴリを確認し問題なし」と明示する
- 各指摘に file:line・重要度・修正案を添える。根拠を示さない指摘は無効とする
- 重大な問題が 1 件でも存在する場合、承認（LGTM）を出さない

## 機械強制

現時点では hook による機械強制なし。code-reviewer サブエージェント（レビュー委任時に本ファイルを照合基準として Read する）と、コード作成時の scoped 注入が防衛線となる。

## 違反検知時の手順

コード作成中・レビュー中に本基準への不適合を発見した場合:

1. 該当カテゴリと重要度を特定する
2. 重大な問題は即座に修正する（マージ・push へ進まない）
3. 警告は当該作業内で修正する。提案は採否を判断し、見送る場合は理由を記録する

## 設計判断

設計判断・経緯の記録は同ディレクトリの `design-notes.txt` を参照（非注入サイドカー）。

## プロジェクト上書き

- 上書き可否: 委譲可（値のみ・追加のみ）
- 受け口: `<repo>/.claude/rules/scoped/review-checklist/code/common/rule.md`
- 優先順位: 受け口が存在すれば、レビュー時に本ファイルと受け口の基準を合成して照合する。プロジェクト固有の基準（型規約・ドキュメントカバレッジ・フレームワーク固有の禁止事項等）は受け口に置く。グローバル基準の無効化・緩和は不可。枠組み（重要度定義・結果要件・違反時手順）はグローバルが常に正
