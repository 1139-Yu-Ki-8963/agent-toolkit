# code/test 設計判断・経緯（非注入サイドカー）

## 出典・経緯（2026-07-11）

レビュー基盤のギャップ棚卸し（ユーザー指摘「上流設計から考えて基本的なルール漏れてないか」）で、
テスト設計・テスト品質の観点が完全に欠落していることが判明した。subagent-selection 規約に
「アサーション削除やテスト弱体化で通すこと禁止」という行動規範はあるが、テストコード自体を
レビューで照合する観点表がどこにも存在しなかった。上流工程（要件→設計→実装→テスト）のうち
テストだけ観点ゼロだったため、code ドメインの観点として新設した。

## 設計判断（ADR）

- 必要性: テストコードは「通った」という結果だけでは品質を判定できない（弱体化・写経・恒真で
  いくらでも合格を偽装できる）。検出力・網羅・独立性という観点での照合が必要
- 代替案を採用しなかった理由:
  - code/common への追記: テストコードは paths が本体コードと異なり（*.test.* 等）、
    「テストファイルを触った時だけ注入」できる独立 rule の方が scoped の性質に合う
  - hook 化: 弱体化・写経の判定は文脈理解を要し、grep では誤検知だらけになる。
    専門家（code-reviewer）の照合が適切
- 保守責任者: 人手（ユーザー）
- 廃棄条件: テスト品質の照合が別の仕組み（カバレッジ計測基盤等）に完全移管された時

## 既知の限界

- paths は ts/js 系の *.test.*・*.spec.* と Python の tests//test_*/*_test のみ。
  他言語・他規約のテスト配置を扱う場合は paths へ追加する
- カバレッジ数値の検証はしない（数値はプロジェクト受け口・CI の担当）
