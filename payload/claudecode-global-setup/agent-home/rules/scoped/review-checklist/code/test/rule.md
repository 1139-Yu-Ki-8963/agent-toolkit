# テストコード基準（CODE-TEST-STANDARDS）

テストコードの合格基準。作成時に守るべき規約であると同時に、レビュー時の照合観点表そのものである。レビューでは code-reviewer が本ファイルを Read して照合する（フォルダ `review-checklist/code/` 配下 = code-reviewer 担当）。

テストは「スコアボード」ではなく「バグ検出器」である。通すことではなく、壊れた時に落ちることに価値がある。

## 検出力（重大な問題）

- アサーション弱体化の禁止: テストを通すためにアサーションを削除・緩和・スキップ（skip / only / xit の残置）していない
- 実装の写経禁止: 実装と同じロジックを繰り返して期待値を導出していない（実装のバグごと合格するテストは無価値）
- 恒真アサーションの禁止: 常に成立する検証（自明な比較・存在確認のみ等）でテストを水増ししていない

## 網羅（警告）

- 境界値: 上限・下限・ちょうど境界・空（null / undefined / 空配列 / 0 件）のケースがある
- 異常系: 正常系だけでなく、失敗入力・例外経路のテストがある
- 変更対象との対応: 変更したロジックに対応するテストが追加・更新されている（テスト無しのロジック変更は指摘対象）

## 独立性・安定性（警告）

- テスト間の依存禁止: 実行順序・共有状態（グローバル変数・DB の残留データ）に依存しない
- 非決定性の排除: 現在時刻・乱数・ネットワークに素で依存しない（固定・注入・モック化されている）
- 後始末: テストが作った一時状態（ファイル・レコード）を残さない

## 可読性（提案）

- テスト名が「何を検証するか」を表している（連番・番号だけのテスト名を禁止する。意味キー規約と同旨）
- 1 テスト 1 検証意図。無関係な複数の検証を 1 テストに詰め込まない

## レビュー結果の要件

- 全カテゴリを確認し、指摘 0 件の場合も「全カテゴリを確認し問題なし」と明示する
- 各指摘に file:line・重要度・修正案を添える
- 検出力カテゴリの重大な問題が 1 件でもある場合、承認を出さない

## 機械強制

現時点では hook による機械強制なし。code-reviewer の照合と、subagent-selection 規約のループ規律（「直すのはコードであり、スコアボードではない」）が防衛線となる。

## 違反検知時の手順

1. 該当カテゴリと重要度を特定する
2. 検出力の問題（弱体化・写経・恒真）は、テストではなく実装側を直すべきかを必ず確認する
3. 修正後、全カテゴリを再確認してから完成として扱う

## 設計判断

設計判断・経緯の記録は同ディレクトリの `design-notes.txt` を参照（非注入サイドカー）。

## プロジェクト上書き

- 上書き可否: 委譲可（値のみ・追加のみ）
- 受け口: `<repo>/.claude/rules/scoped/review-checklist/code/test/rule.md`
- 優先順位: 受け口が存在すれば合成して照合する。プロジェクト固有基準（カバレッジ閾値・テストランナー固有の規約等）は受け口に置く。グローバル基準の無効化・緩和は不可
