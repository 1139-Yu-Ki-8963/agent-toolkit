# single-html-output 設計判断・経緯（非注入サイドカー）

## 出典

2026-07-10、ユーザーが撮影画像9枚（別マシンの VS Code プレビューに表示された SKILL.md）から
文字起こしした「HTML作成時の観点コンテキスト」を rule 化した。
元文書はスキル（SKILL.md）として書かれていたが、ユーザーの判断により
「スキルではなく、HTML を作成する際の観点コンテキスト」として保存する方針となった。

## 元文書から削除した内容（ユーザー承認済み・2026-07-10）

1. スキル実行フロー由来の部分（観点ではなく手順のため削除）:
   - 入力コンテキスト3種（チャット文脈 / MDファイル / その他）の定義と決定権・ヒアリング条件
   - モード3種（厳格 / 通常 / 説明）の定義・決定権・再確認義務
   - 入力コンテキスト×モードの9分類
   - コンテキスト固定の手続き（単位一覧の通し番号管理、追加分の区別、図解候補一覧の記録項目）
   - 合格基準のうちモード適合基準（厳格 / 通常 / 説明ごとの判定）とモード参照行
   ※ ただし「欠落判定可能な最小の意味まとまり」という単位定義は被覆基準の判定に
     必要なため、合格基準・被覆の中に定義として残した
2. 参照先が存在しない識別子（GEN-2 / GEN-5 / LAZ-4）
3. 連番識別子（D1〜D11 / FIX-n / COV-n / FIG-n / BAN-n / STR-n / VIS-n / SELF-n / MODE-n）:
   semantic-key 規約（~/.claude/rules/always/review-checklist/meaningful-key-naming/rule.md）が禁止する
   「記号 + 通し番号」の行識別子に該当するため、意味見出し・通常の箇条書きへ変換した

## 設計判断（ADR）

- 必要性: 単一HTML成果物の生成が「入力をテンプレートへ流し込むだけの汎用HTML化」に
  落ちる事故を防ぐ観点の正本が存在しなかった。被覆率100%・図解候補の全件判定・
  自己完結という下限仕様を、作成時に自動で読まれる場所に置く必要がある
- 代替案を採用しなかった理由:
  - skill 化: スキルは呼び出された時しか読まれず、HTML を作る全場面での参照が
    保証されない。観点は呼び出しの有無に関わらず常に守るべき規約
  - docs 配置: 受動的な保管場所であり、作成時に読む保証が無い
  - always 配置（常時注入）: 本規約は長文であり、HTML と無関係なセッションにも
    毎回注入されると context を圧迫する
  - scoped 配置（paths: **/*.html）: HTML ファイルを扱う時だけ注入され、
    作成時参照の保証と context 節約を両立する。これを採用
- 保守責任者: 人手（ユーザー）。観点の追加・削除はユーザー判断で行う
- 廃棄条件: 単一HTML成果物の生成専用スキルまたはワークフローが確立し、
  本観点がそちらに正本化された時。または単一HTML成果物を作る運用自体が廃止された時
