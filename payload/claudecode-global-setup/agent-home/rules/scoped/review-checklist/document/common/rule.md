# ドキュメント共通観点（DOCUMENT-COMMON-STANDARDS）

設計書・仕様書・README などプロジェクトドキュメントの合格基準。作成時に守るべき規約であると同時に、レビュー時の照合観点表そのものである。レビューでは document-reviewer が本ファイルを Read して照合する（フォルダ `review-checklist/document/` 配下 = document-reviewer 担当）。

初版は基本観点のみ。観点を増やす場合は `## 見出し` として本ファイルに追記すること。

## 構成

- 見出し階層に飛び・不整合がない（h2 の次に h4 が来る等の乱れを禁止する）
- 目次・見出し一覧がある場合、本文の実見出しと一致している
- 1 ドキュメント 1 主題。無関係な複数トピックを 1 ファイルに詰め込まない

## 読み手第一

- 初出の略語・専門用語に正式名称または 1 行説明が添えられている
- 読み手が持たない前提知識（プロジェクト固有の概念・経緯）を説明なしで使わない
- 手順書では、実行者がそのまま実行できる具体性がある（プレースホルダには値の例を添える）

## 参照実在

- 文中で言及するファイルパス・リンク・コマンドが実在する（プレースホルダは除く）
- 相互参照（「〜を参照」）の参照先に該当の記述が実在する

## 鮮度整合

- 記載内容が現在の実装・構成と矛盾しない（改名済みのファイル名・廃止済みの手順が残っていない）
- 日付・バージョン等の時点情報がある場合、更新時に追従されている

## 用語・誤字

- 誤字・脱字・変換ミスがない
- 用語の表記ゆれがない（同一概念に複数の呼び名を使わない）
- 語彙の辞書照合（prh 辞書）・連番ID・略称は横断観点として document-reviewer が機械照合する（定義: `~/.claude/rules/always/review-checklist/` 配下の各 rule.md）

## 日本語文章の追加観点

技術的な原稿（設計書・仕様書・解説文）で確認する追加観点。textlint では検出できない文体・構造・語感のルール。

### 整形・視点

- 一文ごとに改行し、段落の区切りは空行で示す
- 日本語の地の文でダッシュ（`—` `―` `——`）・中黒（並列用途）を使わない。同格・補足の挿入は括弧、言い換え・敷衍は句点で二文に分けるか読点でつなぐ
- 用語と定義を並べる箇条書きは全角コロンで「**用語**：説明」と書く
- 結果の羅列や受動態（「特定され、判明した」）ではなく、行為者を主語にした動作の連なりで書く

### 段落と論証

- 1 段落 1 トピック。場面の進行が複数混ざった長い段落は分割する
- 論証は一方向に進める。結論を出してから反論を処理し、結論を言い直す構成にしない
- 推量・可能性を表す文（「かもしれない」「だろう」）を根拠なく断定へ変えない。事実未確認の可能性・反実仮想を表す場合は不確実性を保つ
- 複数の要因がある事象を単一の原因に還元しない

### LLM っぽい表現の禁止

次の言い回しは論点を増やさず「ちゃんと書いている感」だけを付ける。使わない。

| 分類 | NG 表現例 |
|---|---|
| 予告と総括 | 「重要なのは〜である」「本章では〜を扱う／探求する」「要するに」（直前の言い換えだけのとき） |
| 正面から系 | 「正面から扱う」「正面から回収する」 |
| 空虚な形容 | 「不可欠」「核心的」「鍵となる」「多角的」「包括的」 |
| 空虚な動詞 | 「掘り下げる」「深掘りする」「言語化する」 |
| 弱い緩和 | 「〜と言えるだろう」（根拠なく主張を弱める場合のみ。推量・仮定・読者の疑念なら残す） |

### 冗長の排除

- 同じ主張を言い換えて繰り返さない
- 場面を描写した直後に、その内容を要約し直さない
- 数文にわたる議論を一文に圧縮できるなら、圧縮した一文だけを残す

### 曖昧表現

指示文・仕様書で「誰が」「何を」が文脈から一意に復元できない表現を検出する。textlint では検出できない文脈依存の判断を要する項目のみ。

- 動詞（参照する・確認する・対応する・管理する・処理する等）に主語がなく、直前の行・段落からも主語が特定できない場合は主語を補う。同一スコープ内で主語が自明な場合は指摘しない
- 受動態（〜される・〜れる）で動作主が不明な場合は能動態に書き換える。動作主が文脈から自明な受動態は指摘しない。注: 「整形・視点」節は一般的に能動態を好む文体規約であり、本項目は動作主が文脈から特定不能な受動態のみを対象とする
- 箇条書きの項目が名詞・動詞のみで述語・目的語・文脈がない場合は、完結した文にするか説明を補う
- 動詞（チェックする・レビューする・更新する等）に目的語がなく、何を対象とする操作かが文脈から特定できない場合は目的語を補う

## 機械強制

現時点では hook による機械強制なし。commit 時の textlint（`docs/**/*.md` の追加行）が用語面のみを部分的に担う。本規約の構成・参照実在・鮮度の観点は document-reviewer の照合が防衛線となる。

## 違反検知時の手順

ドキュメント作成中・レビュー中に本基準への不適合を発見した場合:

1. 該当観点（構成 / 読み手第一 / 参照実在 / 鮮度整合 / 用語・誤字）を特定する
2. 当該ドキュメント内で修正する。参照切れは「参照先を直す」か「参照元を実在する定義へ差し替える」かを判断する
3. 修正後、全観点を再確認してから完成として扱う

## 設計判断

設計判断・経緯の記録は同ディレクトリの `design-notes.txt` を参照（非注入サイドカー）。

## プロジェクト上書き

- 上書き可否: 委譲可（値のみ・追加のみ）
- 受け口: `<repo>/.claude/rules/scoped/review-checklist/document/common/rule.md`
- 優先順位: 受け口が存在すれば、レビュー時に本ファイルと受け口の基準を合成して照合する。プロジェクト固有のドキュメント基準（章立てテンプレート・必須セクション等）は受け口に置く。グローバル基準の無効化・緩和は不可
