# ビジネス提示資料の内容品質基準（BUSINESS-CONTENT-STANDARDS）

顧客・社外へ提示する資料（スライドHTML・提案書・レポート）の内容品質の合格基準。デザイン・表示・構成の形式面は document ドメインが担当し、本ドメインは主張・論理・数値・読み手適合という中身を照合する。作成時に守るべき規約であると同時に、レビュー時の照合観点表そのものである。レビューでは business-content-reviewer が本ファイルを Read して照合する（フォルダ `review-checklist/business-content/` 配下 = business-content-reviewer 担当）。

初版は 6 カテゴリ 23 観点。その後の追加を経て現在は 26 観点（直近: 2026-07-17 に「説明-具体実値」「効果数値-未検証禁止」の 2 観点を追加）。観点を増やす場合は「意味語キー + 検査内容 + NG例」の形式で該当カテゴリの表に追記する（キーは連番禁止）。既存観点の削除は禁止（検査の弱体化になるため。表現の具体化・NG例の追加は可）。

## 適用ゲート

照合の前に、対象が「顧客・社外へ提示する目的の資料」かを判定する。

- 該当する例: 顧客向けスライドHTML・提案書・報告書・見積り説明資料・社外公開する解説資料
- 該当しない例: 内部設計書・README・SKILL.md・rule.md 等の設定資産・開発者向けガイド・内部ポータルHTML
- 該当しない場合は全観点を「対象外」とし、即 PASS として報告する
- 提示目的が判定できない場合は「判定不能・提示目的の明示を推奨」と記録したうえで、内容の照合は実施する

## メッセージ設計

| 観点キー | 検査内容 | NG例 |
|---|---|---|
| タイトル-主張文 | タイトルがテーマ名ではなく主張（結論の1文）になっているか | 「AI活用について」というラベル型タイトル |
| 結論-単一性 | 1枚（1章）に主張が1つに絞られているか | コスト削減と品質向上と組織変革が併記され本題が読み取れない |
| SoWhat-明示 | 図・データの後に「だから何か」が文章で明示されているか | グラフを置いて解釈を読み手に委ねている |
| 行動-接続 | 読後に取ってほしい行動（承認・意思決定・次工程着手）へ接続しているか | 説明で終わり次に何を判断してほしいのか不明 |
| 説明-具体実値 | 仕組み・提案の説明に、読み手が検証・追試できる実値（固有名・数値・形式名・実例）が伴っているか。実値は実在・検証済みのものに限り、具体性のための創作値は実値の不在より重い違反として扱う | 「役割を 4 分類に分ける」とだけ述べ、分類ごとの担当・権限・禁止事項の実例が一切ない |

## 論理構造

| 観点キー | 検査内容 | NG例 |
|---|---|---|
| 主張-根拠対応 | 主張のそれぞれに根拠が対応し、根拠なき断言がないか | 「効果は3倍」に根拠の記載がない |
| 分類-漏れ重複 | 列挙・分類に漏れ・重複がないか | 課題3分類の2番目と3番目が実質同じ |
| 因果-相関混同 | 相関・前後関係を因果として断言していないか | 「導入企業は売上増」を「導入すれば売上増」と表現 |
| 前提-明示 | 結論が依存する前提条件が図中または注記で明示されているか | 営業部門の実測値であることを伏せて全社に一般化 |
| 比較-条件揃え | 比較の測定条件・期間・母数が揃っているか | before は繁忙期・after は閑散期の数値で「50%削減」 |

## 数値・事実の信頼性

| 観点キー | 検査内容 | NG例 |
|---|---|---|
| 数値-出所明示 | 図中の数値に出所（実測・聞き取り・公開統計・仮定）の注記があるか | 「70%が失敗」と書いてあるが何の調査か不明 |
| 推定-事実区別 | 推定・試算に「試算」「想定」の明示があるか | 皮算用の削減額が実績と同じ体裁で並ぶ |
| 数値-桁単位整合 | 単位と桁が一貫し計算が合うか | 内訳合計が総計と不一致・時間と人日の無断混在 |
| 情報-鮮度明示 | 変化の速い情報（ツール仕様・価格）に時点が付いているか | 半年前のAIツール仕様を現在形で断言 |
| 効果数値-未検証禁止 | 効果・工数・時間の数値（短縮時間・削減率・見積もり）に検証記録・実測・公開仕様のいずれかの裏付けがあるか。裏付けを示せない効果数値は削除する（緩和表現への書き換えや「例：」付記では不可） | 検証なしに「60 分の作業が 5 分に短縮」と記載 |

## 読み手適合

| 観点キー | 検査内容 | NG例 |
|---|---|---|
| 語彙-読み手整合 | 想定読み手に対し説明なしの専門用語がないか | 経営層向けに「RAG」「トークン」が注釈なし |
| 関心軸-読み手整合 | 訴求点が読み手の意思決定軸（経営層=投資対効果・リスク、現場=負荷・手順の変化）に合っているか | 経営層向けに技術的な仕組みの精緻さを訴求 |
| 反論-先回り | 読み手が最初に抱く疑問・懸念（費用・セキュリティ・失敗時）に図中か注記で答えているか | 導入提案でセキュリティ懸念に一切触れない |

## 提案の質

| 観点キー | 検査内容 | NG例 |
|---|---|---|
| 提案-顧客文脈接続 | 一般論ではなく当該顧客の状況・課題に接続した記述か | どの会社にも出せる汎用の体で顧客課題との対応が不明 |
| 効果-リスク両面 | 効果だけでなく前提が崩れる条件・リスク・移行コストが開示されているか | 良いことしか書いていない |
| 実行-主体明示 | 実行主体（顧客側/支援側）と最初の一歩が判別できるか | 「〜すべき」の羅列で誰がいつ始めるのか不明 |
| 選択肢-推奨明示 | 複数案を並べる場合、比較基準と推奨案が明示されているか | 3案を対等に並べ「ご検討ください」で終わる |

## 表現の品位

| 観点キー | 検査内容 | NG例 |
|---|---|---|
| 誇大-表現禁止 | 裏付けのない強調語（劇的・圧倒的・必ず）がないか | 「生産性が劇的に向上」（数値裏付けなし） |
| 断定-保留使い分け | 検証済み事実は断定、推測は「〜と見込む」と書き分けているか | 確実な話と願望の区別がつかない |
| 顧客-敬意表現 | 顧客の現状を貶める表現がないか | 現状分析で顧客業務を「旧態依然」と形容 |
| 指示語-自己完結 | 見出し・カード見出し・表セル・箇条書き先頭で文外参照の指示語（この/その/あの＋名詞、これ/それ/あれ＋助詞、上記/下記/前述/先述/後述）を使わず、参照先の固有名（レベル番号・課題名・基準名等）を差し込んでいるか。除外: その他・それぞれ・そのまま・どの/どれ/どこ・引用ブロック内・コードブロック内・直後にリストが続く「以下の」。本文中の検出は警告 | パネル見出しに「この基準の説明」と書かれ何の基準か文面だけでは不明 |

## 照合に必要な入力

「読み手適合」「提案の質」のカテゴリは、想定読み手・顧客文脈（要件確認の記録・課題定義等）が委任時に添えられている場合に照合する。添えられていない場合は当該観点を「判定不能（入力未提供）」と記録する。

## 機械強制

現時点では hook による機械強制なし。reviewing-against-rules 経由の business-content-reviewer 照合が防衛線となる。

## 違反検知時の手順

作成中・レビュー中に本基準への不適合を発見した場合:

1. 該当カテゴリ（メッセージ設計 / 論理構造 / 数値・事実の信頼性 / 読み手適合 / 提案の質 / 表現の品位）を特定する
2. 当該資料内で修正する。数値の出所不明は「出所を注記する」か「裏取りできない数値を削除する」かを判断する
3. 修正後、全観点を再照合してから完成として扱う

## 設計判断

設計判断・経緯の記録は同ディレクトリの `design-notes.txt` を参照（非注入サイドカー）。
