# サブエージェント選択規約（SUBAGENT-SELECTION）設計ノート（非注入サイドカー）
# rule.md から分離した保守者向け記録。規約の挙動には影響しない。

## 設計判断

**必要性**: Claude はデフォルトで全作業を自分で実行しようとし、`~/.claude/agents/` に 4 役割を定義しても積極的に活用しない。rules で委任判定フローを常時注入し、hook でキーワード検出時に促すことで、サブエージェント活用率を構造的に引き上げる。

**代替案を採用しなかった理由**:
- CLAUDE.md に追記: CLAUDE.md は機械強制が必要なものは hook/rules に閉じる方針。行動規範を CLAUDE.md に書いても遵守率がゼロに張り付くことが過去セッションで確認済み
- Skill のみ: Skill はトリガー駆動で必要時のみロードされるため、常時の行動規範には不適
- hook のみ: hook は注入タグで促すだけで、判定フロー・具体例・並列指針の常時参照ができない

**保守責任者**: 人手（ユーザー）。新しいサブエージェント追加時に委任先テーブルを更新する。

**廃棄条件**: Claude Code 本体がサブエージェント自動委任を組み込んだ時、または 4 役割体制を廃止した時。

### check-main-agent-direct-work.sh

**必要性**: メインエージェントが直接 Write/Edit/git commit/push 等を実行するのを PreToolUse で block し、サブエージェントへの委任を構造的に強制する。規約（rule.md）と提案（suggest-subagent.sh）だけでは遵守率が不十分で、実セッションでメインが直接 push・ファイル更新を行う事例が観測された。ツール名判定・ファイルパス例外・Bash コマンドパターン例外・再帰防止カウンタの複合分岐があり、settings.json のインラインには収まらない。

**代替案を採用しなかった理由**:
- suggest-subagent.sh（提案のみ）: block しないため、メインが提案を無視して直接実行する経路が残る
- permissions.deny: ツール名単位の deny はサブエージェントまで巻き込んで全面停止する。agent_id による例外判定ができない
- Bash ツール直叩き: PreToolUse イベントにバインドできない

**保守責任者**: 人手（ユーザー）。例外パスの追加・サブエージェント体制変更時に更新する。

**廃棄条件**: Claude Code 本体がメイン/サブのロール分離を組み込んだ時、または全セッションで orchestrating-dev が起動され oradora 側の role-boundary hook が代替する時。

### worker-haiku-bash-guard.sh

**必要性**: worker-haiku を「実行専用（tools: Bash, Read）」に再設計しても、Bash 経由のファイル作成・編集（`echo > file`・`touch`・`mkdir` 等）はツール制限では防げない。定義ファイルの禁止事項を 2 段階で強化しても、haiku がファイル作成依頼に従ってしまうことが実機検証で 2 回確認された（2026-07-05、memo.md 作成の再現テスト）。プロンプト遵守に頼らず、hook 入力の `agent_type` フィールド（公式 hooks 仕様）で worker-haiku を識別し、ファイル変更コマンドを PreToolUse で機械的に block する。リダイレクト検査（fd 複製・/dev/null 除外）・セグメント分割による git 例外・変更系コマンドのパターン検査という複合分岐があり、settings.json のインラインには収まらない。

**代替案を採用しなかった理由**:
- 定義ファイルの禁止事項強化のみ: 実機検証で 2 回破られた。haiku は文中の禁止条項より依頼文に引っ張られる
- tools から Bash を剥奪: テスト・ビルド・git 定型操作という haiku の存在意義が消える
- permissions.deny: エージェント別の deny 指定ができず、メイン・他エージェントまで巻き込む
- 委任側ルール（rule.md の Q6）のみ: 委任プロンプトの書き方に依存し、構造的保証にならない

**保守責任者**: 人手（ユーザー）。ブロック対象コマンドパターンの追加・git 例外の見直し時に更新する。

**廃棄条件**: Claude Code 本体がエージェント別のコマンド実行制限（tools の粒度細分化）を提供した時、または worker-haiku を廃止した時。

### worker-haiku-bash-guard.test.sh

**必要性**: worker-haiku-bash-guard.test は素通り 9 ケース + block 10 ケースの計 19 ケース（メイン素通り・git 例外・fd 複製と /dev/null リダイレクトの許可・変更系コマンド検出）を検証する単体テスト。リダイレクト scrub とセグメント分割は正規表現の相互作用が多く、都度 Bash 直叩きで JSON を手書きすると回帰確認が再現できない。

**代替案を採用しなかった理由**:
- Bash ツール直叩き: 19 ケース × JSON 生成をセッション内で都度書くとトークン消費が大きく再現性がない
- 既存 Makefile / package.json scripts: `~/.claude/` 配下にはどちらも存在しない

**保守責任者**: 人手（ユーザー）。guard 本体のパターン変更時に同時更新する。

**廃棄条件**: worker-haiku-bash-guard.sh 廃止時。

### check-evidence-checklist.sh（旧 check-investigation-checklist.sh。2026-07-07 改名）

**必要性**: 調査・分析・レビューを委任する際にチェックリストなしで指示を出すと、調査項目の漏れ・裏取り不足・推測の断言が発生する。2026-06-27 のプロジェクト構成レビューで 5 件の事実誤認が観測された（Playwright 設定の配置場所誤認・logs/ の Git 追跡状態誤認・Render/Vercel の配置要件誤認）。全て「確認せずに推測を断言した」ことが原因で、チェックリストによる調査項目の事前定義と hook による強制で構造的に防止する。

**代替案を採用しなかった理由**:
- rule.md に「チェックリストを作れ」と書く: 長時間セッションで遵守率がゼロに張り付くことが確認済み
- Stop hook で出力を検査: 出力段階では既に誤った報告がユーザーに伝わっている。入力段階（Agent 委任時）で block するのが適切
- brain に検証を任せる: brain 自身がチェックリストなしで判断すると同じ問題が再発する

**保守責任者**: 人手（ユーザー）。調査タスク種別の追加時にチェックリストテンプレートを更新する。

**廃棄条件**: Claude Code 本体が調査委任時のチェックリスト自動生成を組み込んだ時、またはサブエージェント体制を廃止した時。

### check-subagent-choice.sh（2026-07-07 新設）

**必要性**: `general-purpose`/`claude`（model 非固定の汎用サブエージェント種別）が委任判定フローに該当するタスク（調査・レビュー・外部検索・計画・一括作業等）に使われると、名前付きサブエージェント（model 固定済み）を経由すべき責務分離が崩れる。実セッションで PR レビュー 5 件を general-purpose・model 未指定のまま並列起動し、Fable 5 の利用上限に到達して 4 件が失敗する事故が発生した（2026-07-07）。prompt 文字列からカテゴリを判定し、該当する場合は名前付きサブエージェントへの変更を、該当しない残余タスクでも model 未指定なら明示指定を要求する分岐は、複数の正規表現パターンマッチ・再帰防止カウンタ（カテゴリ判定系統と model 指定系統を独立カウント）・JSON 入出力整形を伴い、settings.json のインライン記述には収まらない。

**代替案を採用しなかった理由**:
- Bash ツール直叩き: PreToolUse(Agent) イベントに自動バインドできず、Agent 呼び出しのたびに手動チェックが必要になり、hook 導入前の状態（規約遵守を目視レビューに頼る）と変わらない
- rule.md への記載のみ（hook なし）: 過去の実測（CLAUDE.md Maintenance Notes 記載の通り）で、skills/hooks の介入がない行動規範は遵守率がゼロに張り付く傾向が確認されているため、機械強制が必要
- 既存 `check-evidence-checklist.sh` への統合: 同スクリプトは調査系タスクのチェックリスト有無のみを検査する責務であり、model 未指定・カテゴリ分類全般の検査を混在させると単一責任が崩れ、既存の再帰防止カウンタ・例外リストとの干渉リスクが生じる

**保守責任者**: 人手（ユーザー）。委任判定カテゴリ（investigator/researcher/brain/worker 系）や model 指定要件が変わった場合に同時更新する。

**廃棄条件**: Claude Code 本体が Agent 呼び出し時に model 固定サブエージェントの選択を構造的に強制する機能を持った時、または本規約（サブエージェント選択規約）が撤回された時。
