# コーディング原則（CODING-PRINCIPLES）

コード変更時の行動姿勢を規定する 6 原則。

## 原則

1. **シンプルさ優先**: YAGNI・KISS・DRY。変更は可能な限りシンプルに、影響範囲を最小に
2. **複雑度抑制**: サイクロマティック複雑度を増やさない
3. **後方互換の抑制**: 追加コストなしに実現できる場合を除き、シムやフォールバックは導入しない
4. **根本原因主義**: 一時的な修正は禁止。最小差分で根本から直す。安易な回避策に逃げず、必ず根本原因を調べる
5. **自律的なバグ修正**: バグ報告・失敗 CI・エラーログを見たら、指示を待たずに調査と修正へ進む。ユーザーにコンテキストを切り替えさせない
6. **マーカーファイル禁止**: hook の状態追跡にマーカーファイル（カウンタ・フラグ・状態ファイル）を新規作成する設計を禁止する。代替: transcript 走査（block タグの出現回数）・skill-log JSONL 参照・advisory のみ。詳細は `scoped/agent-config/hooks/rule.md`「マーカーファイル禁止」節

## プロジェクト上書き

- 上書き可否: 一律適用
- 理由: コーディング姿勢はプロジェクトに依存しない普遍的な行動規範であり、受け口を設けない
