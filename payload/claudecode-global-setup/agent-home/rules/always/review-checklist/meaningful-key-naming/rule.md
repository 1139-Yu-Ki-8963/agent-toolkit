# 意味キー規約（MEANINGFUL-KEY-NAMING）

ドキュメント内の管理簿・観点表・決定記録などで行を識別するキーに、連番 ID（記号 + 通し番号）を使うことを禁止する。識別子は内容を要約した意味語キーとする。連番は AI にとって不透明なトークンであり、番号から何の情報も得られない。

## 禁止対象

1. **記号 + 連番の行識別子** — `UT-01` `IT-03` `M-1` `D-01` `Q-1` のような自システム採番の通し番号
2. **管理簿用途の「ID」列** — 表ヘッダに `| ID |` を置いて連番を振ること
3. **連番による相互参照** — 「IT-03 を参照」のような番号ベースの参照
4. **意味キーへの枝番サフィックス** — `金額-境界-2` のような連番での重複回避（重複は意味の具体化で解消する）

## 必須形式

- 行識別子は **内容を要約した意味語キー** とする
  - 観点キー: `<対象>-<観点要約>` 形式（例: `金額-下限境界`・`api失敗-error表示`・`参照権限-保存不可`）
  - 決定キー: 議題の要約（例: `繰越表示-方式`・`ページング-方式`）
- キーだけを縦に読んだとき、表の目次として機能する状態を正とする
- 相互参照・テストコードからのトレース（テスト名・コメント）にも同じ意味キーを使う

## 対象外（番号を使ってよいもの）

- 章・節の構造番号（`§1`・`9.1`・`Phase 6` 等）— 位置を示す構造の番号であり、行の管理 ID ではない
- 外部システムが発番する ID — GitHub issue 番号・PR 番号・commit hash・HTTP ステータス等
- 日付・バージョン番号・ポート番号など、値そのものに意味がある数値

## 機械強制

現時点では hook による機械強制なし。以下が現場の防衛線となる。

- `~/agent-home/templates/` の各テンプレート表に記入規則行（「キーは連番禁止。内容を要約した意味語で付ける」）を埋め込み済み
- 手動レビュー時の検出補助: `grep -rnE '\b[A-Z]{1,4}-[0-9]+\b'` と `grep -rnE '^\| *ID *\|'`（`UTF-8` `SHA-256` 等の誤検知があるため機械 block には使わない）

## 違反検知時の手順

連番 ID・管理用「ID」列を発見した場合:

1. 該当表の「ID」列を「キー」列に改名する
2. 連番値を、行の内容を要約した意味語キーに置換する
3. その表の直近の注記に記入規則行を追加する: 「キーは連番禁止。内容を要約した意味語で付ける（番号からは情報を得られないため）」
4. 相互参照・テスト名など、旧番号を参照している箇所を新キーへ追従させる

設計判断・経緯の記録は同ディレクトリの `design-notes.txt` を参照（非注入サイドカー）。

## プロジェクト上書き

- 上書き可否: 一律適用
- 理由: 意味キーの可読性・並べ替え耐性・トレース性はプロジェクトに依存しない普遍的な性質であり、プロジェクト側で連番 ID を許可する正当な事由が存在しないため、受け口を設けない
