# 意味キー規約（MEANINGFUL-KEY-NAMING）設計ノート（非注入サイドカー）
# rule.md から分離した保守者向け記録。規約の挙動には影響しない。

## 設計判断

**必要性**: 2026-07-02 のテンプレート設計セッションで、観点表・意思決定表に UT-01 / IT-01 / D-01 等の連番 ID が混入し、ユーザー指摘で 4 ファイル 32 箇所を全面置換した。AI は管理簿を設計する際に連番 ID をデフォルトで採用する傾向があり、常時注入の規約で構造的に抑止する。意味キーはキー自体が要約として読め、行の並べ替え・挿入に耐え、テスト名への埋め込みで自然にトレースできる。

**代替案を採用しなかった理由**:
- hook（PostToolUse で表を grep）: 連番パターンは `UTF-8`・`SHA-256`・issue 番号等の正当な番号と機械的に区別しづらく誤検知が多い。まず eager rule + テンプレート内記入規則で運用し、違反が再発する場合に hook 化を再検討する
- テンプレート内の記入規則行のみ: テンプレートを経由しない新規の表（調査レポート・メモ等）に効かない
- naming-conventions スキルへの追記: スキルはトリガー駆動（commit 前・ファイル作成時）でロードされるため、表を書く瞬間の常時抑止にならない

**保守責任者**: 人手（ユーザー）

**廃棄条件**: Claude Code 本体または hook 機構が表識別子の意味キー強制を組み込んだ時。
