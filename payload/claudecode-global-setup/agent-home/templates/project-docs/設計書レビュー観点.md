# 設計書レビュー観点

記入済み設計書および観点表の合否基準の正本。体系の設計意図の正本を兼ね、フローの review gate と人間レビューの両方がこの文書を参照する。

---

## §1 体系の設計原則

| キー | 原則 | なぜ |
|---|---|---|
| コード外情報のみ | コードから再生できる情報を書かない | コードの写しは乖離した瞬間に嘘をつく負の資産になる |
| 正の一意性 | 同じ事実は 1 箇所にだけ書く | 2 箇所に書くと必ず片方が腐り、どちらが正か分からなくなる |
| 規範と参照の分離 | 守らせるものは rules、知らせるものは docs | docs に書いた規範は読まれた時しか効かない |
| 意味キー必須 | 行識別子は内容を要約した意味語 | 連番は AI にとって不透明トークンで情報がゼロ |
| 検証との対 | 規則には機械検出手段を対にする | 検証できない規則は長時間セッションで破られる |
| AI 事故からの逆算 | 未決の可視化・該当なし規則・粒度ガードレール | AI は未決を勝手に確定し、空欄を水増しし、写しを増殖させる |
| 使い捨てと恒久の区別 | 生成契約はマージ後に畳む | 恒久保守の対象を最小に保つ |
| 通り道で育てる | 地図・意図・観点は変更のたびに 1 行ずつ | 一括整備は完成時点で陳腐化が始まる |

## §2 章別の設計意図

### 画面基本設計書

| 章 | 設計意図 |
|---|---|
| 書かないもの表（冒頭） | 冒頭の憲法。コード写しの混入をレビュー可能にする |
| §1 目的と業務文脈 | 画面の存在理由はコードから読めない唯一の前提 |
| §2 機能と領域 | 結合キー（領域名）の発生源。定義権を 1 箇所にして DESIGN.md・観点表と機械照合可能にする |
| §2.3 配置図 | 領域粒度限定。ウィジェットを描いた瞬間モックの写しになる |
| §3 業務ルール | コードは「である」しか語らない。「あるべき」の正 |
| §4 画面状態の要件 | AI が empty・error の見せ方を勝手に決めて実装する事故の防止 |
| §5 入力と検査の要求 | 制約の「理由」だけが非コード情報（型・正規表現はコードとテストが正） |
| §6 遷移の業務条件 | 遷移の業務条件はルーター実装から読めない |
| §7 非機能の画面固有制約 | 目標・合意はコードに存在しない |
| §8 逸脱と例外 | 「知っていて外した」記録がないと後続が誤って共通に戻す |
| §9 意思決定 | 再生不能な合意の記録 + 未決の可視化 + 捨てた代替案の保存（ADR） |

### DESIGN.md の分離

数値・見た目は業務ルールと変更頻度・変更理由が異なる。混ぜると片方の変更がもう片方を腐らせる。分離するとバリデータで機械検証できる単位になる。

### 共通 3 ファイル（_共通/）

各画面に差分だけを書かせるための上位の正。画面 §8 逸脱の比較先。

### 機能基本設計書

§3・§4 は複数画面にまたがるドメインルールとデータ所有の正。棲み分け表（画面 §3 / 機能 §3 / フロー §4）で二重の正を防ぐ。

### 操作フロー設計書

画面横断の行程は単一画面の文書のどこにも置けない。large 観点の由来元。

### 観点表（単体・結合・E2E）

観点 = 非コード情報、具体項目 = テストコードが正（手動観点のみ例外）。受け入れ◯は TDD 二重ループの外側。

### プロジェクト地図

航法（どこを読むか）であって内容の複写ではない。

### 境界規約（rules 側）

AI は目に入ったコードを模倣するため「真似るな」の常時注入が要る。

### reverse-docs 16 章版

全体像網羅は顧客提出物として保守する前提のため、章構成を固定して納品 UI に直結させる。保守方式 3 区分（手・生成・混在）はコードから再生できるかで章を仕分け、納品時に生成章をリバース再生成することで乖離を構造的に防ぐ。章番号固定は画面横断レビュー（同番号章の比較）を可能にするため。

## §3 レビュー観点表

山括弧プレースホルダ `<...>` のみの行は判定対象外（未記入の骨格行）

各観点の適用段階: scaffold 検収時は機械検出系（数値混入・意味キー・章網羅・結合キー・frontmatter）のみを適用し、目視系と実体前提の観点（プレースホルダ残存・リンク切れの実値リンク・意思決定系）は実体化後に適用する

### A 責務境界

| キー | 観点 | 検出方法 | 意図の参照 |
|---|---|---|---|
| 数値混入-設計書 | 設計書・共通仕様に色指定などの数値が混入していない | `grep -E "#[0-9a-fA-F]{6}"` がヒット 0 | §1 コード外情報のみ |
| 領域名-一致 | §2.2 の領域名と DESIGN.md コンポーネント構成のキーが一致する | §2.2 の領域名リストと DESIGN.md の突合（目視） | §1 正の一意性 |
| 画面状態-一致 | §4 の状態名と DESIGN.md 状態別視覚仕様のキーが一致する | §4 の状態名リストと DESIGN.md の突合（目視） | §1 正の一意性 |
| target_screen-同値 | 同一画面ディレクトリ内の 4 ファイルで同値（_共通 は対象外） | 同一ディレクトリ内 `grep -r "target_screen"` で同値を確認 | §1 正の一意性 |

### B 正の一意性

| キー | 観点 | 検出方法 | 意図の参照 |
|---|---|---|---|
| ルール棲み分け | 同一業務ルールが画面 §3・機能 §3・フロー §4 に重複記載されていない | 目視（三者を並べて比較） | §2 機能基本設計書 |
| コード写し混入 | 設計書に実装コードの写しが存在しない | `grep -E "useState\|import "` がヒット 0 | §1 コード外情報のみ |
| スキーマ写し | 機能 §4 が業務的意味のみを持ち、型定義の写しでない | 目視（業務語で記述されているか確認） | §1 コード外情報のみ |

### C 構造規則

| キー | 観点 | 検出方法 | 意図の参照 |
|---|---|---|---|
| 章網羅 | 必須章がすべて存在する | `for i in $(seq 1 N); do grep "^## §$i "; done` でヒット確認（画面 §1〜9・機能 §1〜8・フロー §1〜6・本書 §1〜4） | §1 AI 事故からの逆算 |
| 該当なし規則 | 空の条件付き章に「該当なし（理由）」形式の記述がある | 目視（条件付き章を全量確認） | §1 AI 事故からの逆算 |
| 意味キー | 行識別子に連番 ID が使われていない | `grep -E '\b[A-Z]{1,4}-[0-9]+\b'` ヒットを目視除外して 0 | §1 意味キー必須 |
| プレースホルダ残存 | 実体化後にプレースホルダが残留していない | `grep -o "<[^>]+>"` がヒット 0 | §1 AI 事故からの逆算 |

### D リンク・検証

| キー | 観点 | 検出方法 | 意図の参照 |
|---|---|---|---|
| リンク切れ | 相対リンクが有効である | lychee（設定は `.config/lychee.toml`（Phase 6 で配置）。山括弧を含むプレースホルダリンクは scaffold 段階では除外） | §1 検証との対 |
| 結合キー-ディレクトリ一致 | 機能名・フロー名がディレクトリ名と一致する | `ls` と機能名リストの突合 | §1 正の一意性 |
| frontmatter 必須キー | 種別ごとの必須キーが存在する（設計書・観点表: doc_id/type/status/updated・DESIGN.md 系: doc_id/type/status/target_screen/updated・プロジェクト地図: doc_id/type/updated・本書: doc_id/type/updated） | 種別に応じて `grep -E "^(doc_id\|type\|status\|updated\|target_screen):"` でキーを確認 | §1 検証との対 |
| textlint | 追加・変更行が textlint を通過する | commit hook・Phase D で自動実行（単体: `npx textlint --config ~/agent-home/tools/linter/.textlintrc.json <対象>`） | §1 検証との対 |
| DESIGN 構造 | DESIGN.md が規定の構造を満たす | `validate-design-md.sh PASS`（単体: `bash ~/agent-home/tools/design/validate-design-md.sh <対象>`） | §2 DESIGN.md の分離 |

### E 意思決定

| キー | 観点 | 検出方法 | 意図の参照 |
|---|---|---|---|
| 承認-未決矛盾 | status: approved の時点で §9 未決定表が 0 件である | frontmatter の status と §9.2 未決定表のデータ行数を突合（approved なら 0 行） | §2 画面基本設計書 §9 |
| 選定理由-欠落 | 決定済み行の選定理由が空でない | 目視（意思決定記録を全行確認） | §2 画面基本設計書 §9 |
| 自明決定-羅列 | 代替案が実在しない決定が並んでいない | 目視（代替案欄の内容が実在する案であるか確認） | §1 AI 事故からの逆算 |

### F 観点表の質

| キー | 観点 | 検出方法 | 意図の参照 |
|---|---|---|---|
| 由来-実在 | 観点の由来 § が設計書に実在する | 由来 § の番号を設計書の章見出しと突合（目視） | §2 観点表 |
| 受け入れ-妥当 | 受け入れ◯が業務語で「リリース可能」と言い切れる観点に限る | 目視（受け入れ◯の行を全量確認） | §2 観点表 |
| traced-整合 | status: traced の観点表は全行にテスト参照が存在する | `grep -c "^status: traced"` の件数と参照数を突合 | §2 観点表 |

---

## §4 使い方

- **実行タイミング**: Phase 4 の承認前（設計書・DESIGN.md）と Phase 8 の push 前（観点表含む全量）
- 検出方法が grep・スクリプトの行は機械実行し、「目視」の行は根拠を 1 行添えて判定する
- フローの review_gates（flow-values.yml）からこの観点を参照させる配線は、スキル接続改修の範囲に含まれる
