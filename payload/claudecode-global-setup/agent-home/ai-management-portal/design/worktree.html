§1git worktree とは

git の標準機能。1 つのリポジトリから複数の作業ディレクトリを切り出す仕組み。
通常の git clone がリポジトリ全体を複製するのに対し、worktree は .git の実体を共有し、 ソースコードの作業コピーだけを別ディレクトリに展開する。
リポジトリ（実体） .git/ ← 全 worktree で共有される唯一の実体 .git/worktrees/fix-login/ ← worktree ごとのメタデータ（HEAD・index） .git/worktrees/add-api/ src/ package.json ... ← メインツリーの作業コピー worktree A: ~/Projects/worktrees/fix/login-bug/ .git ← ポインタファイル（1 行: "gitdir: /.../.git/worktrees/fix-login"） src/ package.json ... ← 独立した作業コピー worktree B: ~/Projects/worktrees/feature/add-api/ .git ← ポインタファイル src/ package.json ...

clone との違い

項目git clonegit worktree
.git の扱い完全に複製（独立）共有（ポインタで参照）
ブランチ・reflog・stash独立（push/pull で同期）全 worktree で共通
ディスク消費.git + ソース（全体の 2 倍）ソース分のみ（.git の重複なし）
同一ブランチの checkout可能不可（1 ブランチ 1 worktree）

ディスク消費の実測値

-（React + Supabase）での実測。node_modules を除いた作業コピーのサイズ。

対象サイズ備考
メインツリー（.git 含む）数百 MB.git 実体 + node_modules
worktree 1 本39〜52 MBソースコードのみ（node_modules 除外時）
worktree 1 本（npm install 後）数百 MBnode_modules が支配的
メモリへの影響: worktree 自体はディスクのみ消費し、メモリは使わない。 ただし worktree 内で開発サーバー（vite / uvicorn 等）を起動すると、サーバーごとにメモリを消費する。 worktree 削除時にサーバープロセスを kill し忘れると、メモリとポートが残骸として蓄積する。

§2なぜ worktree を使うか

orchestrating-dev-flow（13 Phase の開発オーケストレーター）は、 main ブランチ上で複数の Claude Code セッションが並行稼働する環境を前提とする。
worktree なしではファイル競合・進捗混在・セッション跨ぎの破綻が発生する。

worktree がない場合の問題

問題具体例worktree による解決
ファイル競合 セッション A と B が同じ src/api/user.ts を同時編集 → 後勝ちで上書き 各 worktree が独立したファイルシステムを持つため干渉しない
進捗ファイル混在 .flow-progress.json が 1 つだけ → セッション A の Phase 8 完了をセッション B が上書き worktree ごとに独立した進捗ファイルを持つ
セッション跨ぎ セッション再開時に「どこまで完了したか」を判別できない worktree 内の .flow-progress.json から中断 Phase を復元
ブランチ汚染 main に直接コミットが混在し、レビューなしで本番に到達する feature ブランチ + PR フローを強制

設計判断（ADR）

必要性: 並行セッション環境でのファイル分離と進捗管理の整合性を保証するために worktree を必須化した。
代替案不採用の理由
進捗ファイルを /tmp に配置セッション跨ぎの進捗引き継ぎができない。/tmp はセッション固有だが、worktree は複数セッションで共有する
main ブランチに進捗ファイルを配置複数セッションの競合が発生する。main を汚染する
git clone で独立リポジトリを作成.git 全体を複製するためディスク消費が大きい。ブランチ・stash が共有されず運用が煩雑
git stash で作業を切り替え同時並行ができない。1 セッションずつの直列作業になる
廃棄条件: Claude Code 本体がセッション分離と進捗管理を組み込んだ時、または orchestrating-dev-flow が廃止された時。

§3運用方式

worktree は作業ごとに名前付きで作成する。固定スロット（wt1/wt2/wt3）ではない。
ブランチ名に基づく命名で、作業内容が名前から読み取れる。

作成の流れ

orchestrating-dev-flow Phase 2（ブランチ準備） ↓ Step 1: worktree 判定 ├ .git がファイル → 既に worktree 内（スキップ可） └ .git がディレクトリ → メインツリー（作成必須） ↓ Step 2: Skill(parallel-dev-worktree) 呼び出し └ git worktree add -b "<branch>" "<path>" "origin/main" ↓ Step 3: 環境確認（git status, 依存パッケージ） ↓ Step 4: 進捗ファイル初期化（.flow-progress.json）

命名規則

ブランチ名worktree パス
fix/login-bug~/Projects/worktrees/fix/login-bug/
feature/add-api~/Projects/worktrees/feature/add-api/
docs/update-readme~/Projects/worktrees/docs/update-readme/
一部のプロジェクト（-）では .claude/worktrees/<name> にも配置される。 配置パスはプロジェクトごとに異なりうる。

main 直接編集が許されるケース

  • 調査・質問応答 — 実装を伴わない読み取りのみの作業
  • グローバル設定の変更 — ~/.claude/ 配下の rules / settings / agents
  • agent-home リポジトリ自体の管理 — ポータル更新・スキル追加等
check-worktree-required.sh（PreToolUse hook）がメインツリーでの Edit/Write を block する。 上記の例外パス（~/.claude/*、~/agent-home/*）は hook 内で除外されている。

§4進捗ファイルの配置

進捗管理は 2 層構造。セッション内の追跡と、セッション跨ぎの保持を分離する。
注意: この 2 層構造は本タスク（2026-06-26）で新規設計したもの。実装はまだ行われていない。
ファイル配置先用途ライフサイクル
flow-status.json <worktree>/.claude/markers/${session}/ セッション内の Phase 進捗追跡 セッション終了時に cleanup-session-markers.sh が自動削除
.flow-progress.json <worktree>/（ルート直下） セッション跨ぎの進捗保持 worktree 削除時に自動消滅。.gitignore で commit 対象外
セッション A で Phase 6 まで完了 → セッション終了 flow-status.json → 削除（cleanup-session-markers.sh） .flow-progress.json → 保持（worktree ルートに残る） セッション B で同じ worktree を再開 .flow-progress.json を読み込み → Phase 7 から再開 新しい flow-status.json を作成
.flow-progress.json は .gitignore に登録済み。commit に混入することはない。 worktree を削除すればファイルも自動消滅するため、追加のクリーンアップ処理は不要。

§5クリーンアップの仕組み

worktree の後片付けは複数の hook・スキルが連携して行う。
過去に残骸プロセスが 16 件蓄積した実績があり、自動化の仕組みを整備した。
タイミング仕組み対象
セッション終了 cleanup-session-markers.sh（SessionEnd hook） .claude/markers/${session}/ と /tmp/claude-hooks/${session}/
フロー完了（Phase 12） phase-12-post-merge-cleanup.md .flow-progress.json、worktree 本体、開発サーバープロセス
Stop hook remind-worktree-cleanup.sh マージ済みなのに残っている worktree を検出して通知
セッション開始 cleanup-stale-hook-sessions.sh（SessionStart hook） /tmp/claude-hooks/ の 7 日超 orphan ディレクトリ

残骸蓄積の過去事例

2026-06-26 のポート調査で、削除済み worktree の残骸プロセスが 16 件蓄積していた。 owner docs サーバーが 5 重起動、wt2 docs サーバーが 4 重起動。 ポート番号がランダムだったため重複検知ができず放置されていた。

対策として port-management-rules でポート番号を計算式ベースの一意割当に変更し、 remind-worktree-cleanup.sh でマージ済み worktree の残留を自動検出する仕組みを導入した。

掃除し忘れのリスク
  • worktree を手動で rm -rf すると .git/worktrees/ にメタデータが残る。git worktree remove を使うこと
  • 開発サーバーを起動したまま worktree を削除すると、プロセスとポートが残骸になる
  • remind-worktree-cleanup.sh はマージ済みの worktree のみ検出する。マージ前の放置 worktree は検出しない
  • PR レビュー中の worktree は意図的に保持される。レビュー対応が長期化すると蓄積する

§6設計判断（ADR）

worktree 運用で発生していた矛盾と未決事項を 2026-06-26 に解決した。
以下はその設計判断の記録。

ADR-1: worktree の同時数上限

項目内容
決定全プロジェクト合計で 5 本まで
根拠Docker を併用する環境でのメモリ消費を考慮。各 worktree で開発サーバーを起動すると 1 本あたり数百 MB のメモリを消費する。Docker コンテナ（Supabase 等）との共存を前提とすると、5 本が実用上の限界
強制方法.port-slot ファイルによるスロット管理。空きスロット（1〜5）がない場合は worktree 作成を拒否する
廃棄条件マシンのメモリが大幅に増設された時、または Docker を使わなくなった時に見直す

ADR-2: ポート管理と worktree 命名の統一

項目内容
旧状態port-management-rules はスロット制（wt1/wt2/wt3）前提、実際の worktree は名前ベース。対応付けが存在しなかった
決定worktree の命名は名前ベース（ブランチ名準拠）を維持する。ポート計算に必要なスロット番号（1〜5）は作成時に動的に割り当てる
仕組みworktree 作成時に空きスロットの最小番号を <worktree>/.port-slot に書き込む。ポート番号はこのスロット番号から計算する
代替案worktree 自体をスロット制（wt1〜wt5）にする案を不採用。作業内容がスロット名から読み取れず、運用上わかりにくい
worktree 作成時のスロット割当フロー 1. git worktree list で全 worktree を列挙 2. 各 worktree の .port-slot を読み、使用中スロットを特定 3. 空きスロット（1〜5）の最小番号を選択 4. 新 worktree の .port-slot に書き込み 5. ポート = プロジェクトベース + (スロット × 10) + サービスオフセット 例: fix/login-bug に slot 2 が割り当たった場合 backend: 8020, frontend: 8021, docs: 8022

ADR-3: ポート監視

項目内容
旧状態ポート計算式は定義されていたが、監視・強制する hook がなかった
決定現時点では hook による機械強制は行わない。.port-slot ファイルによるスロット割当で、ポート番号の一意性は保証される。開発サーバー起動スクリプトが .port-slot を読んでポート番号を決定する設計とする
根拠サーバー起動コマンドのパターンが多様（uvicorn / vite / next dev / python -m http.server 等）で、hook による網羅的検出が困難
廃棄条件起動スクリプトが統一され、hook でパターンマッチ可能になった時

ADR-4: 復旧スキルの対応

項目内容
旧状態recovering-slot-worktrees がスロット制（wt1-wt3）前提で、名前ベースの worktree を復旧できなかった
決定復旧スキルを名前ベース + .port-slot 方式に対応させる（未実装。改修が必要）
