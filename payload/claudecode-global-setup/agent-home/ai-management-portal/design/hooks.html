§1基本原則

hook script は 「誰の enforcement arm か」 が一目で分かる場所に置く。flat な hooks/ バケットに全 hook を投げ込む構造は禁止する。

原則は次の 1 行に要約できる:

hook script は、それが強制する規約 (rule) または延長する skill と同じフォルダに置く。 1 規約 = 1 フォルダ、1 skill = 1 フォルダ、その中に scripts/ または rule 直下のスクリプトを同居させる。

この原則の効用:

  • フォルダ名そのものが「この hook は何を守るためにあるか」を表すので、INDEX や口伝が不要
  • 規約ドキュメントと enforcement script が同居するので、片方だけ更新して陳腐化する事故が減る
  • hook を削除する時、関連ドキュメント・テストごとフォルダを rm -rf できる

§2禁止される配置

以下のパスに hook script を 新規作成すること を禁止する。既存ファイルの編集は legacy として許可する（移行は §8 参照）。

次のパスは hook ではないので本ルールの対象外。enforcement hook はこれらを判定除外する。

§3配置決定の 2 軸

新規 hook を作る前に次の 2 軸で自分の hook を分類する。

軸 1: ownership（誰の延長か）

分類判定基準例
skill 延長 特定 skill の前提や挙動を強制する。skill が無ければこの hook は存在意義を失う parallel-dev-worktree skill の前提を満たすため worktree を強要する hook
独立規約 単一 skill に紐付かない system 全体のメタ規約を強制する commit author を白リストに縛る規約・先送り表現禁止・依頼形禁止
その他 単発の使い捨て・調査用 該当なし。作らずに Bash で直接実行する

軸 2: scope（どこまで効かせるか）

分類判定基準
global全プロジェクトで効かせたい
project単一プロジェクトのみで効かせたい

§44 象限の置き場

2 軸の組合せで配置は一意に決まる。

skill × global

~/agent-home/skills/<skill>/scripts/<hook>.sh

例: parallel-dev-worktree skill の延長 → ~/agent-home/skills/parallel-dev-worktree/scripts/check-worktree-required.sh

skill × project

<repo>/.claude/skills/<skill>/scripts/<hook>.sh

例: プロジェクト固有 skill の延長で、他プロジェクトに持ち込みたくないもの

独立規約 × global

~/.claude/rules/<rule>-rules/<hook>.sh

例: 先送り禁止 → ~/.claude/rules/no-deferral-rules/check-no-deferral-pre-bash.sh

独立規約 × project

<repo>/.claude/rules/<rule>-rules/<hook>.sh

例: プロジェクト固有のコミット規約・E2E ゲート

フォルダ構造（1 規約 = 1 フォルダ）

~/.claude/rules/ ├── no-deferral-rules/ │ ├── rule.md ← 規約の正本（auto-load） │ ├── lib/no-deferral-detect.sh ← 共通検査本体 │ ├── check-no-deferral-pre-bash.sh ← PreToolUse(Bash) hook │ ├── check-no-deferral-post-write.sh ← PostToolUse(Write|Edit) hook │ ├── check-no-deferral-stop.sh ← Stop hook │ └── check-no-deferral-stop.test.sh ← テスト（任意） ├── always/response/guard/ │ └── ... └── hooks-architecture-rules/ ├── rule.md └── check-hooks-architecture.sh

§6hook の管理（一覧化）

hook は分散配置されるが、全 hook の網羅一覧は単一の HTML ドキュメント hooks.html で管理する。検索性と「漏れなく把握できる」状態を担保するため。

hooks.html に必ず登録する

  • 新規 hook を作成したら、同じターン内で hooks.html の HOOKS 配列に登録する
  • 登録項目: file（フルパス）/ group（発火イベント）/ matcher / role（挙動 1 行）
  • group は既存の GROUPS（PreToolUse · Bash / Write/Edit / Stop / SessionEnd …）から選ぶ。新区分が必要なら GROUPS にも追記する

hooks.html の役割範囲

hooks.html は「発火イベント別の網羅表」であり、規約の正本ではない。規約の正本は各 rule.md（または rule.html）にある。hooks.html はそこへのナビゲーションを担う。

将来的に hooks.html を「ファイルパスから自動収集して動的レンダリング」する仕組みにすれば手動登録は不要になる。当面は手動登録で運用する。

§7機械強制

本ガイドの違反を防ぐため、enforcement hook を用意する。

timing強制ポイント注入タグ挙動

check-hooks-architecture.sh は独立規約 × global の象限なので ~/.claude/rules/scoped/agent-config/hooks/ に置く。

~/.claude/rules/scoped/agent-config/hooks/ ├── rule.md ← 本ガイドへのリンクと要点 └── check-hooks-architecture.sh ← PreToolUse enforcement

§8既存 legacy の扱いと移行

本ガイドの導入時点で既に存在する legacy 配置:

  • ~/agent-home/tools/hooks/*.sh — 31 ファイル
  • ~/Projects/<project>/.claude/hooks/*.sh — 19 ファイル
  • ~/.claude/plugins/marketplaces/*/plugins/*/hooks/ — plugin 内

移行方針

  • 本ガイドは 新規流入の遮断 までを担う。既存 31+19 ファイルの編集は legacy として許可する
  • 移行は別タスクで 大規模に行う:
    1. 各 hook を 2 軸分類で象限を決定
    2. 新フォルダを ~/.claude/rules/<rule>-rules/ または ~/agent-home/skills/<skill>/scripts/ に作る
    3. git mv でファイル移動
    4. settings.json の hook command パスを更新
    5. hooks.html の file 列を更新
    6. テスト発火
  • plugin 配下の hooks は plugin maintainer の管轄。Claude は触らない

§9命名と ADR

ファイル命名

  • ケバブケース固定。例: check-worktree-required.sh
  • 役割を語尾に含める: -check.sh（検査） / -enforce.sh（強制） / -detect.sh（検出のみ） / -recorder.sh（記録）
  • 発火イベントを名前に含めると分散配置時に検索しやすい: -pre-bash.sh / -post-write.sh / -stop.sh

設計判断（必須）

新規 .sh 作成は security-rules/rule.md の規約に従い設計判断の記録が必須。配置先の正本ドキュメント内に ## 設計判断 セクションとして記載する。

  • global hook → 配置先の ~/.claude/rules/<rule>-rules/rule.md 内
  • project hook → 配置先の <repo>/.claude/rules/<rule>-rules/rule.md または skill の SKILL.md 内

設計判断が書けない（必要性を説明できない）hook はそもそも作らない。
