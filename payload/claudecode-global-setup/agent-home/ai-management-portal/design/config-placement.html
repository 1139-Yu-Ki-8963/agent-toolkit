§1このガイドの立ち位置

このガイドは「何を書くか」ではなく 「どこに書くか」 を決める横断的な判定ガイド。
既存の各設計ガイド（CLAUDE.md / Rules / Skills / Hooks）は「その層内での詳細設計」を扱うが、 本ガイドは どの層を選ぶか を Q1 から順に降りる決定木で示す。
CLAUDE.md に「毎回 X したら必ず Y せよ」と書くなら Hook へ。「絶対するな」と書くなら Hook + permissions.deny へ。 30 行の手順を書くなら Skills へ。src/api/ だけに効くルールなら paths 付きの Rules へ。
各層の詳細設計（命名・フォルダ構造・frontmatter）は、§7 にリンクする各設計ガイドを参照。 本ガイドは「選択肢を絞る入口」に特化している。

§27 層の一覧

層配置場所ロードタイミング向いている内容
Hooks rules/<name>-rules/*.sh
skills/<name>/scripts/*.sh		 ライフサイクルイベント（決定論的） 「必ず Y せよ」「絶対するな」の強制、linter 自動実行、block
Output styles .claude/output-styles/ セッション開始（system prompt を置き換え） ペルソナ・出力形式の全セッション制御
--append 起動フラグ その起動のみ（system prompt に追記） 単発起動時のトーン・形式の補足
Skills skills/<name>/SKILL.md スラッシュ / 自動マッチ（遅延ロード） 多段手順・フロー・条件付き知識（30 行超の手順）
Subagents .claude/agents/ Agent ツール起動時（独立 context） 独立実行・中間結果を散らかしたくない作業
Rules lazy rules/*.md（paths 付き） マッチファイル Read 時のみ（遅延） 特定パス・ファイル種別だけに効かせる制約
Rules eager rules/*.md（paths 無し） セッション開始 + compaction（常時） 機械強制 pair の規約、[TAG] 対応手順、カテゴリ別長文ルール
CLAUDE.md CLAUDE.md（ルート or サブ） セッション開始 + compaction（常時） 言語・文体・対話・コーディング原則（機械強制不可の根本姿勢）
コスト高い層は上に集中する。 CLAUDE.md と Rules eager は毎 compaction で再注入される。200 行を超えると attention が低下して個別ルールが効かなくなる。 可能な限り遅延ロード（Skills / Rules lazy / Subagents）を選ぶ。

§3決定木（Q1–Q8）

トップダウンで読み、最初にヒットした層を使う。ヒットしなかった質問はスキップ。

Q1
ライフサイクルイベントで確実に発火させたいか？
OR「毎回 X したら必ず Y せよ」「絶対にするな」を技術的に保証したいか？
OR 特定キーワードを含む prompt のときだけ context 注入したいか？
Hooks
Q2
全セッションで出力形式・ペルソナ全体をシステムレベルで制御したいか？
デフォルトの役割（ソフトウェアエンジニア）を置き換えてよいか？
Output styles
Q3
単発起動時のみのトーン・出力形式の補足か？
その起動だけで有効であり、永続させる必要がないか？
--append
Q4
特定のトリガー（スラッシュ / 自動マッチ）でのみ要る手順 or 知識か？
OR 30 行を超える手順・フロー・デプロイチェックリストか？
Skills
Q5
独立したコンテキストで完結させたい作業か？
中間結果でメイン会話を汚染したくない（ログ解析・依存関係監査など）
Subagents
Q6
特定のパス・ファイル種別に限定した制約か？
例: src/api/** のみ、**/*.sh のみ
Rules lazy
Q7
機械強制と対になる制約・規約か？ OR 全タスクで違反しうる制約か？
hook が注入する [TAG] への対応手順 / カテゴリ別長文ルール
Rules eager
Q8
ユーザーが明示 AND 機械強制不可 AND 全タスク共通 AND デフォルトと異なるか？
4 条件すべて YES → CLAUDE.md（200 行以内）。いずれか NO → どこにも書かない
CLAUDE.md
決定木の使い方
  • Q1 から順に読む。YES になった最初の層を使う
  • 複数の Q に引っかかるなら、番号が小さい Q を優先（Hook > Skill > Rules > CLAUDE.md）
  • Q8 まで全部 NO なら「どこにも書かない」（dead code を生産しない）

§4各 Q の解説

Q1 — Hooks

プロンプト指示は長時間セッション・プロンプトインジェクションで破られうる。 「絶対しない」「必ず実行する」を保証したいなら Hook 一択。 「特定キーワードを含む prompt 時だけ context を注入」は UserPromptSubmit hook で実装する。

用途hook タイミング
特定ファイル編集を blockPreToolUse(Edit|Write)
保存後 linter 実行PostToolUse(Write)
完了時 Slack 通知Stop
キーワード条件付き context 注入UserPromptSubmit
特定 Bash コマンドを blockPreToolUse(Bash)

Q2 — Output styles

システムプロンプトを置き換えるため、Claude の「ソフトウェアエンジニア」役割などデフォルト挙動が消える。 全セッション共通でペルソナを変えたい場合のみ使う。慎重に。

Q3 — --append-system-prompt

役割を置き換えず、追記のみ。その起動時のみ有効。 指示が多いほど従い度が下がる「逓減」があるため、短くシンプルに保つ。

Q4 — Skills

起動時は name と description だけ読まれ、本体は呼び出し時に遅延ロードされる。 CLAUDE.md / Rules eager に対してコスト的に優位。 手順型（デプロイ）/ 条件付き知識型（ライブラリ参照）/ 強制型（品質チェック）の 3 型がある。

Q5 — Subagents

独立したコンテキストウィンドウで動作し、最終結果だけメイン会話に返す。 中間の試行錯誤・長大な調査ログがメインの context を圧迫しない。 会話の文脈・インタラクティブな承認が必要な作業には向かない。

Q6 — Rules lazy（paths 付き）

paths frontmatter を付けると、マッチするファイルを Read したときだけ高 attention で注入される。 compaction で消失し、次のマッチ Read で再注入。「絶対に守らせたい」制約は lazy にしない。

Q7 — Rules eager（paths 無し）

CLAUDE.md と同列でセッション開始 + compaction のたびに再注入。 hook と pair で使うのが基本形。hook が注入する [TAG] への対応手順もここに書く。

Q8 — CLAUDE.md

4 条件すべてを満たさないと書けない（§5 で詳述）。 書けるものは「言語」「文体」「対話原則」「コーディング原則」程度に絞られる。

§5CLAUDE.md の厳しい条件

Q8 に到達するには Q1〜Q7 すべてに NO と答える必要がある。実際に書けるものはほとんどない。
Q8 の 4 条件（すべて YES で初めて書ける） ① ユーザーが「CLAUDE.md に書け」と明示したか？ NO → 書かない（自律追加禁止） ② 機械強制（hook / permissions / CI）で技術的に強制できないか？ できる → rules/ + hook へ誘導 ③ 全タスク共通の根本姿勢か？（トリガー駆動でない） トリガー駆動 → Skills へ ④ Claude のデフォルトと異なるか？ 同じ → dead code になるので書かない

実際に CLAUDE.md に残る内容

カテゴリ例理由
言語規約「思考は英語で、やり取りは日本語で」hook で判定不可能
文体の根本姿勢「受動態を能動表現に変える」hook で機械強制不可能
対話の根本姿勢「不足があれば 1 問ずつ確認」常時必要・強制不可能
コーディング原則YAGNI / KISS / DRY / 根本原因主義全タスク共通・強制不可能
正本ポインタ「禁止事項は rules/*.md を参照」ショートカット（実体は rules）
セッション実測で「CLAUDE.md に書いても skills/hooks の介入なしには遵守率がゼロに張り付く」ことが確認されている。 機械強制可能なものは必ず hooks/rules に移す。

§6頻出アンチパターン

✗ CLAUDE.md に「毎回 X したら必ず Y せよ」
確実な発火が必要なら Hook へ。プロンプト指示は破られる。
✓ Hook（PostToolUse / PreToolUse）
✗ CLAUDE.md に「絶対にするな」
プロンプトインジェクションで破られうる。決定論的な block が必要。
✓ Hook + permissions.deny
✗ CLAUDE.md に 30 行超の手順
特定トリガーでのみ必要な手順は毎セッションでトークンを消費する。
✓ Skills（遅延ロード）
✗ CLAUDE.md と rules 両方に同じ制約
片方が腐ったとき矛盾する。正本は 1 つに決める。
✓ rules を正本、CLAUDE.md はポインタのみ
✗ 全タスク共通の制約を paths 付き rules に
特定パスを Read しないセッションでは注入されず違反する。
✓ paths 無し rules（eager）
✗ src/api/ だけのルールを eager rules に
不要なタスクでもトークンを消費し続ける。
✓ paths 付き rules（lazy）
✗ スラッシュコマンド化したい手順を rules に
rules は「常時注入される制約」であり、手順書ではない。
✓ Skills
✗ 会話の文脈が必要な作業を Subagents に
Subagents は独立 context で動くため、親の会話履歴を参照できない。
✓ Skills（または直接実行）
