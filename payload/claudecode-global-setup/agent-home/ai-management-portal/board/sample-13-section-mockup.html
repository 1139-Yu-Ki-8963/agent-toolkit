issue #241 タスクボードにタスクの手動再実行ボタンを追加する

対象機能: agent-home タスクボード（ai-management-portal/board/task-board.html / tools/harness/scripts/task-board.mjs / state/task-board/）。本ファイルは creating-mock 規約（13 セクション・実装同等忠実度）の見本として作成した説明用 HTML であり、記載する issue 番号・タスク例は本規約を実演するための架空値。

このモックの読み方

  • 承認判断の方: 提案 → 受け入れテスト観点 → リスク の 3 つを読めば判断できます（約 3 分）
  • 実装される方: 配置 → DB設計 → ドメインロジック → API契約

背景

task-board.mjs には add / claim / complete / fail / list / next の 6 サブコマンドがあり、 failed になったタスクを再度実行可能な状態へ戻す手段が存在しない。

現状の回避策は 2 つしかない。1 つは add で同じ内容のタスクを新規 id で再登録する方法だが、 失敗の経緯（元の receipt・何回目の失敗か）が新タスクに引き継がれず、履歴が分断される。 もう 1 つは state/task-board/board.json を直接テキストエディタで編集して status を pending に書き換える方法だが、README.md が「正本は必ず board.json」としつつも 手編集は read-modify-write の整合性保証（一時ファイル経由の rename）を経ないため、書き込み中の他プロセスとの競合や JSON 構文崩れのリスクを負う。

本 issue では、failed タスクを安全に pending へ戻す retry サブコマンドと、 タスクボード画面から failed 行に対して再実行コマンドを提示する導線を追加し、この 2 つの回避策を不要にする。

提案

board/task-board.html の一覧テーブルに「操作」列を追加し、status が failed の行にのみ「再実行」ボタンを表示する。クリックすると行の下に retry コマンドのコピー用パネルが開く（board.json への書き込みは CLI 経由のみのため、 画面から直接 API を叩く実装にはしない）。以下は実際の ai-management-portal/style.css の デザイントークン・クラス構成をそのまま用いた実装同等のマークアップである。

画面の変化を把握しやすいよう、下記は「現状」「変更後」の 2 タブで比較できるようにしている。

※「再実行」ボタンは board.json を直接書き換えず、ターミナルで実行する retry コマンドを提示する（コピー動作は本ファイル内で完結し動作する）。 task-board.html は静的ミラー（board.js）を読むだけの画面であり、画面からのライブ書き込み API は持たない設計を踏襲する。

受け入れ条件

  • status: failed の行にのみ「再実行」列にボタンが表示され、それ以外の status では表示されない
  • ボタンを押すと、当該タスクの id を埋め込んだ task-board.mjs retry --id <id> コマンドがその場に展開される
  • retry コマンドは failed 以外のタスクに対しては拒否され、board.json は変更されない
  • retry_count が 3 に達したタスクは retry が拒否され、上限到達がエラーメッセージで分かる
  • retry 成功後は board.js ミラーが自動再生成され、画面再読み込みで pending 表示に反映される

リスク

重大度リスク影響緩和策
中 retry_count の上限（3 回）に達したタスクが再実行不能になったまま放置される 失敗の原因が解消されても再実行手段がなく、手動で新規タスクを再登録する二度手間が生じる 一覧の receipt 列に retry_count を明記し、acceptance-tests のケースで上限到達時の表示を検証する。上限到達後は新規 id での再登録を README の使用例に明記する
高 board.json の直接編集と retry コマンドの競合 手動編集後に retry を実行すると retry_count の起点がずれ、上限判定が不正確になる README.md に「retry_count はコマンド経由でのみ更新し手編集しない」と明記し、既存の read-modify-write（一時ファイル + rename）方式をそのまま踏襲する
低 原因未解消のまま再実行され、同じ理由で再度 failed になる retry_count を無駄に消費し、実質的な上限到達を早める --reason 引数で再実行理由を receipt に残す運用とし、次回失敗時に前回理由と比較できるようにする

ドメインロジック

業務ルール: 再実行の上限は 3 回とする（retry_count >= 3 のタスクは retry できない）。再実行が成立すると status は pending に戻り、next コマンドの取得対象へ復帰する。

状態遷移条件遷移先
pendingclaim 実行in_progress
in_progresscomplete 実行done
pending / in_progressfail 実行failed
failed 新規 retry 実行かつ retry_count < 3 pending（retry_count +1・session_id を null に戻す）
項目条件エラーメッセージ
--id該当タスクが存在することerror: task not found: <id>
statusfailed であることerror: cannot retry task in status "<status>" (must be failed): <id>
retry_count3 未満であることerror: retry_count exceeded max (3): <id>

API契約

本機能に HTTP エンドポイントは存在しない。task-board.mjs は外部依存のない Node.js CLI であり、画面（task-board.html）はこの CLI が書き出す board.js ミラーを読むだけの静的表示である。そのため本節は「呼び出すインターフェース」として retry サブコマンドの CLI 契約を定義する。
