計画暗黙知の炙り出し 仕組み解説

計画の書き手が無自覚に省略した前提（暗黙知）を、コンテキストゼロの弱モデルに初見読解させて検出し、計画へ明文化する仕組み。 plan mode の 2 地点（Write・ExitPlanMode）で hook が機械的に強制する。 ← ポータル TOP へ戻る

§1解決する問題

計画の書き手（上位モデル・人間）は、自分が持つコンテキストを前提に記述を省略する。 書き手自身が読み直しても脳内で自動補完してしまうため、省略には気づけない。

この仕組みが導入される前は、計画の品質チェックは Claude 自身の「発話マッチングによる自発的なスキル呼び出し」に依存していた。 しかし Claude は呼び出しを忘れる・省略するため、ユーザーから「仕組みとして成立させろ」との指摘を受けて hook による機械強制へ格上げされた。

§2全体像（4 構成要素）

要素種別役割
plan-comprehension-prober サブエージェント（Haiku 固定） 計画文のみを受け取り、5 項目形式（目的の言い換え・手順の再説明・推測で補った前提・判断できなかった点・最初の一手）で理解を言語化する。暗黙知の検出器
eliciting-plan-tacit-knowledge スキル（audit 型） 暗黙知の炙り出し委任 → ギャップ 4 分類 → 計画への再統合 → 収束確認（最大 2 周）を制御するオーケストレーター
check-plan-draft-write-gate.sh PreToolUse(Write) hook 計画ファイル（~/.claude/plans/*.md）への最初の永続化をスキル発火前に block する
check-plan-tacit-knowledge-gate.sh PreToolUse(ExitPlanMode) hook 計画の承認提出をスキル再発火前に block する。Write 版と消費カウンタを共有し、1 計画サイクルで 2 回の検証を強制する

§3機械強制フロー

ユーザー「やって」
  ↓
Claude が計画を組み立てる（plan mode）
  ↓
計画ファイルへ Write
  ↓
❶ check-plan-draft-write-gate（未発火なら block）
  ↓ block
Claude が Skill(eliciting-plan-tacit-knowledge) を実行
  → prober が会話テキスト上の計画を初見読解
  → ギャップ検出 → 計画を修正
  ↓
Write 通過（消費カウンタ更新: 発火 1 回消費）
  ↓
Edit で計画を修正…
  ↓
ExitPlanMode（ユーザーに承認を求める）
  ↓
❷ check-plan-tacit-knowledge-gate（再発火なしなら block）
  ↓ block
Claude が再度 Skill を実行
  → prober が修正後の計画ファイルを初見読解
  → 残ギャップ検出 → さらに修正
  ↓
ExitPlanMode 通過（消費カウンタ更新: 発火 2 回消費）
  → ユーザーに承認 UI が表示される

1 計画サイクルで最低 2 回の検証が構造的に強制される。 下書き前（会話テキスト経路）と承認前（計画ファイル経路）の 2 段。

§4なぜ弱モデルに読ませるか

コンテキストゼロの弱い読み手に計画だけを読ませると、行間を埋められず省略箇所で理解が壊れる。 壊れ方（誤読・推測・判断不能）が暗黙知の地図になる。

成立条件実現手段破るとどうなるか
読み手がコンテキストゼロ新規 Agent 委任（計画文のみ埋め込み）意図を知っている読み手は省略に気づけない
読み手が弱いモデルHaiku 固定強いモデルは行間を補完してしまい、暗黙知が顕在化しない
「わからない」を言わせる推測・判断不能の明示列挙を 5 項目形式で強制曖昧な要約だけが返り、ズレの位置を特定できない

§5ギャップの 4 分類

prober の報告と本来の意図を突合し、ズレを以下の 4 つに分類する。

分類定義対応
誤読書いてあるのに別の意味に取られた誤解を誘導した記述を書き換える
欠落書かれておらず、推測で埋められた・埋められなかった前提明文化して追記する（暗黙知の本体）
曖昧複数解釈が可能と申告された一意になるよう書き換える
偶然の的中推測したが偶然意図と一致していたそれも明文化する（次の読み手が同じ推測をする保証はない）

§6消費カウンタの仕組み

両 hook はタイムスタンプではなく発火行数の単調増加で判定する。 スキル発火ログ（~/agent-home/sessions/.skill-log/<session>.jsonl）の該当行数と、 共有マーカー（plan-tacit-knowledge-gate.consumed-count）に記録された消費済み行数を比較する。
状態判定
発火行数 > 消費済み通過。消費カウンタを現在の発火行数に更新（発火を「消費」）。自ゲートの block 連続カウンタをリセット
発火行数 ≦ 消費済みblock（exit 2）。block 連続カウンタを +1
block 連続 3 回到達自動解除（exit 0）。正規通過でカウンタはリセットされるため、散発的な block の累計ではなく「連続」で判定される
なぜタイムスタンプ比較ではないか: 初版ではスキル発火 ts の最大値と前回通過時刻を比較する方式だったが、 実機検証でログに未来日時（2099 年等）の ts が 1 件混入すると消費機構が恒久無効化されるバグが実証された。 発火行数方式はクロック非依存のため、この問題が発生しない。

§7実運用での検出実績

設置後 2 セッションで実際に暗黙知を検出し、計画を改善した実績がある。

セッション計画内容検出例
dce46c0b DESIGN.md 生成ステップの復旧計画 CSS トークン抽出の優先順位が未定義 / 出力先の衝突時動作が不明 / format-spec.md の実在が未確認 → 計画に「前提・制約」節として追記
3d710db3 スキル改名・スコープ縮小計画 新 SKILL.md 全文が「省略」で中身不明 / 置換指示に old/new 具体文言なし → 全文明記とベタ書きペア追加

両セッションとも推測 5〜6 件・判断不能 7〜10 件を検出し、主要な指摘が計画ファイルへの Edit として反映された。 「仕組みが作動した」だけでなく「作動した結果、書かれていなかった前提が明文化された」ことがログの diff で確認されている。

§8保証と限界

項目状態
計画ファイルの初回永続化と承認提出の 2 地点で暗黙知チェックを強制保証する
ユーザーの目に触れる経路をすべて塞ぐ保証しない（スキル実行自体が Agent 呼び出しとしてトランスクリプトに現れる）
plan mode を経由しない計画立案（brain 直接呼び出し等）を捕捉保証しない（ExitPlanMode / Write のみが観測点）
内部マーカーの意図的な改竄による迂回を防ぐ保証しない（通常フローでの機械強制が目的であり、暗号学的な実行証明は過剰設計として不採用）
