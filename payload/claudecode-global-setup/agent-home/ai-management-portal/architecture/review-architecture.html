レビュー基盤 仕組み解説

「観点はどこにあるか」「誰が照合するか」を PC 全体で 1 本化した仕組み。 観点の正本は rules、実行は専門家サブエージェント、スキルは入口を呼ぶだけに統一する。 ← ポータル TOP へ戻る

§1全体像（3 層構造）

以前は「コードレビュー」「HTML レビュー」「設定資産レビュー」がそれぞれ別のスキルに観点を抱え込み、 観点表への参照が壊れていたり（後述 §6）、レビュー主体がスキルごとにバラバラだった。 本基盤は観点・入口・実行主体の 3 層に整理し、ドメインが増えても仕組みを増やさず対応する。

設計原則は 「観点は 1 箇所（rule.md）」「判定は専門家、生成役と分離」「スキルは入口だけを呼ぶ」「フォルダ構造そのものが担当宣言」 の 4 点。 rule.md がそのまま観点表であり、作成時（scoped 注入）とレビュー時（専門家が Read）で同じ内容を参照するため、 観点表とは別に「レビュー用チェックリスト」を複製して持たない。担当専門家はフォルダ名 review-checklist/<domain>/ から <domain>-reviewer として機械導出する。 拡張子からの推測（rule 側の適用範囲が広がった時に取りこぼす）も、frontmatter での宣言 （フォルダと宣言の二重管理で乖離する）も採らない。構造が唯一の正である。

§2観点の正本（rules）

レビュー観点は ~/.claude/rules/scoped/review-checklist/ 配下に物理集約されている。 第 1 階層のフォルダ名が担当専門家（<domain>-reviewer）を、 各 rule.md の paths frontmatter が適用対象ファイルを宣言する。

rulepaths（適用対象）担当（フォルダ導出）
code/common（コーディング共通基準） **/*.ts *.tsx *.js *.jsx *.mjs *.py code-reviewer
code/ui（UI実装基準・アイコン等） **/*.tsx *.jsx *.vue *.html *.css *.scss code-reviewer
code/test（テストコード基準） **/*.test.* *.spec.* tests/**/*.py test_*.py 等 code-reviewer
document/html-output（単一HTML成果物） **/*.html document-reviewer
document/common（ドキュメント共通観点） docs/**/*.md README.md document-reviewer
report/common（調査報告共通観点） **/.investigation-checklist.md（報告文自体はファイル外のため直接委任が主経路） report-reviewer

1 ファイルに複数 rule・複数専門家が該当する場合は両方を並列で委任する（§3）。 例: .tsx は code/common と code/ui の両方、.html は code/ui（アイコン）と document/html-output の両方が該当しうる。

プロジェクト固有の観点（ドキュメントカバレッジ判定・案件特性など）はグローバル rule に書かず、 各リポジトリの委譲受け口 <repo>/.claude/rules/scoped/review-checklist/code/common/rule.md に追加する。 専門家はグローバルとプロジェクト受け口の両方を Read し、基準を合成して照合する。

§3単一入口（reviewing-against-rules）

レビューをしたい他スキル・作業は、観点を自分で読み書きせず本スキルを呼ぶ。 観点の要約をプロンプトに書き写す行為（乖離の温床）を禁止している。

設定資産（skills / rules / hooks / routines / subagents 自体）のレビューは対象外で、 managing-agent-configs が担当する。両者は「何をレビューするか」で棲み分ける。

§4専門家サブエージェント

粒度は「rule と 1 対 1」ではなく「ドメイン（中カテゴリ）単位」。HTML 用 rule が 3 つに増えても 専門家は document-reviewer の 1 体のまま。専門性はエージェント定義ではなく、委任時に渡す rule.md の中身で決まる。

専門家担当ドメインtools観点の持ち方
code-reviewer コード（ts/tsx/js/jsx/mjs/py） Read, Grep, Glob, Bash 定義に観点を書かない。委任された rule.md を Read してカテゴリごとに判定
document-reviewer HTML・文書成果物 Read, Grep, Glob, Bash 同上。適用範囲外（アプリの一部ページ等）の判定も rule の適用範囲定義に従って行う
report-reviewer 調査報告（会話内の報告文） Read, Bash, Grep, Glob 同上。対象がファイルでないため resolve を経由せず、report/common/rule.md を自ら Read する直接委任が主経路
生成役（worker-sonnet で作成・修正）と検証役（専門家で判定）を分離する。 これは managing-agent-configs の test モードが既に持つ原則（新規サブエージェントでの実機検証・セルフ再読禁止）を コードレビュー領域にも一般化したもの。

§5横断規約（用語辞書・連番ID）の扱い

always/ 配下の規約（文章置き換え辞書・意味キー等）は全セッションに常時注入されるため、 専門家も内容自体は認識している。ただし「知っている」ことと「レビューの都度、機械的に照合する」ことは別問題で、 当初はこの照合が専門家の判定表に明示されておらず、しかも数える先すらなかった （always/ は scoped/review-checklist/ と別階層にあるため）。

これらは常時注入（always）の性質を保ったまま、always/review-checklist/<観点>/ へ 物理集約する（always 配下ならどの topic でも常時注入されるため移動可能）。scoped 側と対称の 「フォルダ位置 = 登録」であり、登録リストの txt は持たない。 専門家の動作原則には各観点への実パス参照（散文だけでなく ~/.claude/rules/always/review-checklist/... という文字列そのもの）を埋め込み、 audit-review-coverage.sh が「always/review-checklist/ 配下の各 rule → 少なくとも 1 つの専門家定義にその文字列が実在するか」を grep で機械照合する。

横断規約照合方法
文章置き換え辞書（prh 辞書） sed 's/<[^>]*>//g' でタグ除去後、textlint（~/agent-home/tools/linter/.textlintrc.json）へ標準入力で照合
意味キー規約（連番ID禁止） grep -nE '\b[A-Z]{1,4}-[0-9]+\b' を検出補助にし、UTF-8・SHA-256 等の誤検知は目視で除外
略称の無断使用禁止 文書成果物・UI 文言の初出略語に正式名称・説明が添えられているかを目視照合

文章置き換え辞書（prh 辞書）と textlint の組み合わせで、docs・PR/issue 本文の語彙・文章品質を継続的に維持している。 既存エントリでの違反修正は上記の textlint 照合と ~/.claude/rules/always/review-checklist/text-dictionary/rule.md の違反時手順が完結して行い、 スキル呼び出しは発生しない。辞書に該当エントリが無い新規語彙を追加する場合のみ adding-textlint-dictionary-terms スキル（重複チェック → prh.yml 追記 → 既存使用箇所の一括置換）を使う。

全 rule の分類正本（rule-classification.txt）

レビュー観点は専門家別 4 rule と横断 3 rule だけではない。グローバル rules は 全 28 件あり、review-checklist 配下（場所が分類）以外の全てについて「なぜレビュー照合しないか」を scoped/agent-config/review-checklist/rule-classification.txt で分類する。 audit-review-coverage.sh はディスク上の全 rule.md がこの分類表か review-checklist 配下の どちらかに存在することを検証し、未分類の rule が 1 件でもあれば FAIL する。 新しい rule を作ると監査が失敗するため、分類の判断（照合対象か・hook 強制で足りるか）を 先送りできない構造になっている。

分類意味件数
専門家別観点（scoped/review-checklist 配下）フォルダ導出された <domain>-reviewer が rule を Read して成果物を照合5
横断観点（always/review-checklist 配下）常時注入かつ全レビュー専門家が内容を機械照合（辞書・連番ID・略称）3
config-asset（台帳で分類）設定資産の規約。managing-agent-configs（ゲート一体型）が担当5
hook-enforced（台帳で分類）hook・permissions が行為の瞬間に強制。レビューで二重照合しない15
behavior（台帳で分類）エージェント行動規範。成果物の内容観点ではない1

設定資産（skills / rules / hooks 等）の観点が managing-agent-configs の references にあるのは意図的な設計判断。 そちらは観点照合だけでなく create → review → test → マーカー → commit gate という機械強制付き ライフサイクルと観点が一体であり、専門家ディスパッチ方式に載せ替えるとゲート機構ごと作り直しになる。 「成果物は reviewing-against-rules、設定資産は managing-agent-configs」の 2 系統で棲み分ける。

§6導入前のギャップと解消内容

調査で判明した主な欠陥と、本基盤での解消内容。

ギャップ解消内容
PR レビューの観点参照先が実在しなかった（別リポジトリの存在しないパスを必須参照していた） グローバル観点を review-checklist/code/common/rule.md に正本化し、reviewing-against-rules 経由の委任へ差し替え
「commit 直前に観点ゲートが存在確認する」という記述が、実在しないスクリプトを指していた 実在する push / PR 作成時ゲート（check-review-gate.sh）を指すよう記述を修正
設定資産レビューでメインが自己レビューする余地が残っていた managing-agent-configs の review モードに「生成役と検証役の分離」を明記
観点表の参照切れを検知する仕組みがなかった rules・skills 双方のレビュー観点に「参照実在チェック」を新設（バッククォート内 ~/ パスの実在検証）
コード観点にエッジケース・過剰設計・可読性・不要な再レンダリングが欠落していた 9 カテゴリ→14 カテゴリに段階的拡充（エッジケース・過剰設計・可読性・再レンダリング、のち公開インターフェース・依存追加・DB変更）
上流設計の観点（テスト品質・出来上がったディレクトリ構成の適合）がどこにも無かった テストコード基準（code/test。検出力・網羅・独立性）を新設。構成の適合は入口スキルの手順「構成の照合」で directory-structure 許可リストと突合（mkdir hook は advisory 止まりのため、レビューが最後の防衛線）
rule のトピック分類（code-review/ ui/ 等）と専門家のドメイン判定（拡張子ベース）が別軸で、icon-usage が適用範囲に持つ tsx/jsx/vue/css/scss のうち html 以外が専門家に一度も渡らなかった rule を scoped/review-checklist/<domain>/ へ物理集約し、フォルダ名 = 専門家名の 1 対 1 とした。入口スキルは拡張子推測をやめ、フォルダ導出された専門家でグルーピングして委任する方式に変更
用語辞書・意味キー規約がレビュー観点フォルダの外（always/）にあり、専門家への参照も散文のみで、そもそも数える先が存在しなかった always/review-checklist/<観点>/ へ物理集約（常時注入の性質は維持）し、専門家定義には実パス参照を追加。audit-review-coverage.sh がドメイン⇄専門家の 1 対 1・観点数・分類・横断観点の参照有無を機械的に数える
