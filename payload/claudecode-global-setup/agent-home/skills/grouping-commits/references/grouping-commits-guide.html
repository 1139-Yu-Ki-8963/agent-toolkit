grouping-commits

変更を自動分析してグループ化し、単一目的コミットを作成するスキル。

こんな人向け
複数ファイルにまたがる変更を「ドキュメント」「テスト」「UI」などの目的別に分けてコミットしたいエンジニア。コミットメッセージの命名規則を守りながら効率よく git 履歴を整理したい場面に向く。
こんな場面で使う
コミット作成が必要な場面、または複数の変更をまとめてコミットする場面。「コミットして」「まとめてコミット」等の指示が TRIGGER になる。コミット操作を伴わない確認・編集のみの場面（SKIP 条件）は対象外。

§1概要

grouping-commits は、git status で取得した変更ファイルの一覧を自動分析し、 ファイルパスやファイル種別に基づいて「テスト」「ドキュメント」「UI」などの目的別グループに分類する。 各グループに対して単一目的コミットを順番に作成し、履歴を論理的に整理する。

スキルの位置づけ

項目値
スキル名grouping-commits
Category業務自動化
Type手順型（action）
TRIGGERコミット作成、複数変更をまとめてコミット
SKIPコミット操作なし（確認・編集のみ）

実行方針（コスト最適化）

委譲ルール グループ分けの確定後、コミット実行（git add / git commit）は worker-haiku へ Agent 委譲する。 メインエージェントが直接実行するのはグループ分けの判断と最終確認のみ。
ただし対象が 1 グループ・数ファイルの単純ケースは委譲コストが上回るため直接実行してよい。

§2出力

実行が完了すると、以下の 2 つが確立される。

単一目的コミット群
変更ファイルをテスト・ドキュメント・設定・UI などの目的別グループに分類し、グループごとに 1 コミットとして作成したもの。コミットメッセージは日本語 prefix（【テスト】・【ドキュメント】 等）と 25 字以内の subject からなり、命名規約（rules: always/naming/commit-branch）に準拠する。ブランチ名が issue-<番号> の場合は 1 issue 範囲モードが適用され、managing-github-issues（verify モード）で範囲外ファイルを除外した上で全変更を 1 コミットにまとめる。
完了レポート
作成されたコミット数・各コミットのメッセージ・未コミットのファイルが残っている場合はその理由・次に推奨する操作（リモートへの push、PR 作成等）をまとめた報告。VAL-001〜VAL-004（コミット作成・機密情報の非包含・単一目的コミット・未コミットファイル）の確認結果を含む。

§3用語と前提条件

用語

用語意味
グループファイルパス・種別で分類した「同一コミットにまとめる変更の集合」
単一目的コミット独立して意味を持ち、1 つの目的（テスト追加・ドキュメント更新等）のみを含むコミット
issue-N ブランチ名前が issue-<番号> の形式のブランチ。専用モードが適用される
機密ファイル.env, credentials.json, .secret, .key 等。コミット禁止
[PUBLISH-AUTHOR] / [PUBLISH-SAFETY]reviewing-public-readiness スキルが発行するタグ。CRITICAL 検出時はコミットを中断する

前提条件

キー条件確認コマンド
PRE-001 Git リポジトリ内であること git rev-parse --is-inside-work-tree
PRE-002 変更ファイルが存在すること git status --short
PRE-003 機密ファイル（.env 等）がステージングされていないこと 変更ファイル一覧から拡張子・ファイル名を目視確認
PRE-004 git author が許可リストのアカウントであること git var GIT_AUTHOR_IDENT
PRE-004 補足 許可リスト外の author を検出した場合、PreToolUse hook が [GIT-AUTHOR-BLOCK] を exit 2 で block する。 ~/.gitconfig の [user] セクションを name = 1139-Yu-Ki-8963 / email = 63326271+1139-Yu-Ki-8963@users.noreply.github.com に修正する（local config・環境変数での上書きは禁止）。

§4動作モード

本スキルはブランチ名によって動作が切り替わる。

モード適用条件概要
通常モード ブランチ名が issue-<番号> でない場合 変更をファイル種別・パスで複数グループに分類し、グループごとにコミットを作成する
1 issue 範囲モード ブランチ名が issue-<番号> の場合（ルーティンから呼ばれた場合も含む） グルーピングを無効化し、全変更を 1 コミットにまとめる。managing-github-issues（verify モード）を事前に実行して範囲外ファイルを除外する

1 issue 範囲モードの判定例

# ブランチ名チェック
branch=$(git rev-parse --abbrev-ref HEAD)
if [[ "$branch" =~ ^issue-([0-9]+)$ ]]; then
  # 1 issue 範囲モード
  # → managing-github-issues（verify モード）を呼び出し
  # → 🔴 NG があれば git restore --staged で除外
  # → 残りを 1 コミットにまとめる
fi

1 issue 範囲モードの手順

  1. managing-github-issues（verify モード）を git add の前に実行し、staged ファイルと対象 issue の範囲を突合する
  2. 🔴 NG（ブラックリストまたは範囲外）と判定されたパスは git restore --staged <PATH> で除外する
  3. 残りのファイルを 1 コミットとしてまとめる

§6動作フロー

Phase 3 詳細: グループごとの処理

# グループごとに以下を繰り返す

# 1. 機密ファイル最終チェック
git status --short | grep -E '\.(env|credentials|secret|key)' || echo "機密ファイルなし"

# 2. ステージング
git add [ファイルパス群]

# 必ず差分を確認してからコミット（Gotcha: 確認なしは無関係な変更の混入リスク）
git diff --staged

# 3. コミット作成（25 字以内の日本語メッセージ）
git commit -m "【prefix】変更内容の説明"

# 4. コミット確認
git log --oneline -1

§7コミットメッセージ規則

本スキルは命名規約（rules: always/naming/commit-branch）で定義された命名規則を参照して git commit -m を実行する。

ルール内容
言語日本語必須。ローマ字禁止
文字数subject は 25 字以内
形式1 行で完結。末尾ピリオドなし
prefix【テスト】 / 【ドキュメント】 / 【設定変更】 / 【機能追加】 / 【リファクタ】 / 【バグ修正】 / 【スタイル】
英語 type 禁止feat: / fix: 等の英語 type は [NAMING-BLOCK] で exit 2 block される
内容変更の「なぜ」「何を」を簡潔に記述
機密ファイル絶対禁止 .env, credentials.json などは絶対にコミットしない。 reviewing-public-readiness スキルが PreToolUse から並行発火し、CRITICAL 検出時は本スキルのコミット作成を中断してユーザーに対処を促す。

§8エラーハンドリング

状況対処
コミットが失敗した場合 git status でエラー内容を確認し、失敗理由を説明する。その後、次のグループの処理に進む
[GIT-AUTHOR-BLOCK] が発火した場合 コミットを中断し、~/.gitconfig の [user] を正規アカウントに修正する
[NAMING-BLOCK] が発火した場合 英語 type のコミットメッセージを日本語 prefix に変更して再試行する
reviewing-public-readiness が CRITICAL を検出した場合 コミット作成を中断し、ユーザーに対処（機密ファイルの除外等）を促す
managing-github-issues（verify モード）が 🔴 NG を返した場合（1 issue 範囲モード） git restore --staged <PATH> で対象パスを除外してからコミットする
Gotcha git diff --staged で差分を確認せずに git commit を実行すると、 無関係な変更が混入する可能性がある。必ずステージング後に差分を確認してからコミットすること。

§9検証項目

Phase 4 の完了レポートに以下の確認結果を含める。

キー確認内容成功条件確認方法
VAL-001 コミット作成の確認 想定されたグループ数のコミットが作成されている git log --oneline -5
VAL-002 機密情報の非包含確認 .env・credentials 等がコミットに含まれていない コミット履歴と diff の確認
VAL-003 単一目的コミットの確認 各コミットが論理的にグループ化されている 各コミットの内容確認
VAL-004 未コミットファイルの確認 意図的に残されたファイル以外はすべてコミット済み。残存する場合はその理由が完了レポートに明記されている git status

完了レポートの出力内容

  • 作成されたコミット数
  • 各コミットのメッセージ
  • 未コミットのファイルが残っている場合はその理由
  • 次の推奨操作（リモートへの push、PR 作成 等）

§10関連スキル・規約

スキル / 規約役割分担
命名規約（rules: always/naming/commit-branch） コミットメッセージの正規ルール（type / subject / 25 文字制限）の定義元。本スキルはルールを参照して git commit -m を実行する側
reviewing-public-readiness コミット作成直前に PreToolUse から並行発火（[PUBLISH-AUTHOR] [PUBLISH-SAFETY] [PUBLISH-SAFETY-FULL]）し、機密ファイル名・author 情報・公開リスクを警告。CRITICAL 検出時は本スキルのコミット作成を中断する
managing-github-issues（verify モード） 1 issue 範囲モード時に git add の前に呼び出し、staged ファイルが対象 issue の範囲内かを gh issue view の body と突合する
pre-bash-dispatch-rules（hook） git commit 時に [NAMING-BLOCK]（英語 type 検出）、[TEXTLINT-BLOCK]（docs 追加行の textlint）、[PUBLISH-AUTHOR] / [PUBLISH-SAFETY]（公開可否）を発火させるディスパッチャ
subagent-delegation-rules（規約） 機械的な git 操作（git add / git commit）は worker-haiku へ委譲するという委任方針の根拠
