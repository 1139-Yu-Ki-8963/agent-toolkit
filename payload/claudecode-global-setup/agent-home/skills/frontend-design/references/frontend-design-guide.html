§1概要

frontend-design は、新規 UI・画面をゼロから構築する際に呼び出すスキルガイドである。 Claude はこのスキルをロードすると「小規模スタジオのデザインリード」として振る舞い、 ブリーフを読み解き、独自のビジュアルアイデンティティを持つ UI を設計・実装する。

汎用テンプレートや AI デザインのデフォルト表現を避け、ブリーフ固有の選択を行うことが このスキルの中核的な姿勢である。同じブリーフで別のデザイナーが到達しそうな結果と 異なる方向を意図的に探索する。

項目内容
スキル名frontend-design
タイプreference
呼び出し方ユーザーが「画面を作って」「LP を作って」「コンポーネントを実装して」等の指示を出した時に適用
主な成果物HTML / CSS / JS によるページまたはコンポーネント
スキップ条件DESIGN.md の単体生成のみが目的の場合（orchestrating-dev-flow の Phase 4 Step 4-1b へ誘導）

§2出力

frontend-design を適用すると、ブリーフ固有の視覚言語に基づいた次の 2 つの成果物が確立される。

デザインプラン（トークン体系）
色（4〜6 個の名前付き hex 値）・タイポグラフィ（表示用書体と本文用書体など役割別のペア）・レイアウト構想（一文の説明文と ASCII ワイヤーフレーム）・シグネチャー（このページが記憶される唯一の要素）をひとまとめにした設計案。ブリーフと照らして「他の似た依頼でも同じ答えに行き着かないか」を一度レビューし、汎用的な選択になっている箇所を修正してから確定する。
UI実装（HTML / CSS / JS）
確定したデザインプランに忠実に従って構築したページまたはコンポーネント。モバイル表示への対応・キーボードフォーカスの可視化・reduced motion への配慮という品質基準を満たし、実装中の自己批評（可能な環境ではスクリーンショットでの確認）を経て仕上げる。

§3主要概念と用語

トークンシステム

カラー・タイポグラフィ・レイアウト・シグネチャの 4 軸で構成するデザインの「語彙」。 ブリーフ分析後、実装前に必ず策定する。

トークン軸定義記述形式
カラー パレット全体を 4〜6 色の名前付き Hex 値で定義 #F4E8D1 / primary-warm のような named hex
タイポグラフィ 2 役以上のフォントペアリング。ディスプレイ・本文・ユーティリティ フォントファミリー名・ウェイト・スケール
レイアウト ページ構造の概念。1 文散文 + ASCII ワイヤーフレームで表現 ASCII アート / prose
シグネチャ このページを記憶させる「唯一の要素」。ブリーフを体現する 1 つの特徴 自由記述（1〜2 文）

ヒーロー / Thesis

Web デザインにおけるファーストビュー（ヒーロー）は「命題（Thesis）」として機能する。 そのページで最も特徴的なものを、最も効果的な形式で開幕に置く。 大きな数字・統計・グラデーションアクセントという汎用解ではなく、ブリーフ固有の選択をする。

シグネチャ要素

デザイン全体の「勝負どころ」。大胆な選択を 1 か所に集中させ、それ以外は抑制する。 装飾と機能を混同せず、シグネチャ以外の要素は静かに保つ。

AI デザインの三大デフォルト（避けるべきパターン）

SKILL.md が明示する回避対象
  • ウォームクリーム背景（#F4F1EA 近辺）+ 高コントラストセリフ + テラコッタアクセント
  • ほぼ黒の背景 + 単一の蛍光アクセント（アシッドグリーン / バーミリオン）
  • ヘアラインルール・ゼロ border-radius・新聞コラム風ブロードシートレイアウト

ブリーフがこれらの方向を明示している場合は指示に従う。ブリーフが自由を与えている軸では、これらをデフォルトとして使わない。

§4発動条件とスキップ判定

条件判定備考
画面作成の依頼 TRIGGER 新規 UI・ページ・画面の設計・実装
コンポーネント実装 TRIGGER 再利用可能な UI パーツの新規作成
LP（ランディングページ）制作 TRIGGER マーケティング・プロモーションページ
デザイン刷新 TRIGGER 既存画面の全面的なビジュアル変更
DESIGN.md の単体生成が目的 SKIP orchestrating-dev-flow の Phase 4 Step 4-1b で処理する

TRIGGER / SKIP は SKILL.md の description フィールドに基づく。 「ゼロから構築するデザインガイド」という定義のとおり、 既存コードの小規模修正（バグ修正・軽微なスタイル調整）はこのスキルの対象外である（SKILL.md に規定なし）。

§5デザインプロセス

SKILL.md が定める「ブレインストーム → 探索 → 計画 → 批評 → 構築 → 再批評」の 2 パスプロセス。

Pass 1: デザイン計画の策定

# Pass 1 の手順
1. ブリーフを読む
   - 製品・主題が明示されていない場合は自分で設定する
   - 「具体的な主題 / 対象ユーザー / ページの唯一の仕事」を 1 文で宣言する
   - 記憶にあるユーザーの好み・過去のデザインもヒントとして活用する

2. トークンシステムを策定する（コンパクトに）
   - カラー: 4〜6 色の named hex
   - タイポグラフィ: 2 役以上のフォントペアリング + スケール
   - レイアウト: 1 文 prose + ASCII ワイヤーフレーム
   - シグネチャ: このページを記憶させる唯一の要素（1〜2 文）

Pass 2: 批評と検証

# Pass 2 の手順
3. 計画をブリーフと照合する
   - 「同じブリーフで別のプロンプトを渡したら到達しそうか？」を問う
   - 汎用デフォルトに見える部分があれば改訂し、変更内容と理由を記述する
   - ブリーフ固有の選択を確認できたら Pass 3 に進む

4. 確認済みの計画に従ってコードを書く
   - 計画から逸脱しない
   - カラー・タイポグラフィはすべてトークンシステムから導出する
   - CSS セレクタの詳細度（specificity）に注意する（キャンセル合戦を避ける）
   - padding / margin のセクション間衝突に特に注意する

自己批評（構築中）

コードを書きながら自分の成果物を批評する。スクリーンショット機能がある環境では 積極的にスクショを撮って確認する（1 枚の画像は 1000 トークン分の情報を持つ）。 「ミラーを見て 1 つアクセサリーを外す」—— 余分な装飾はシグネチャ要素の インパクトを希薄にする。

計画フェーズと思考の扱い ブレインストームと反復は可能な限り思考の中で行い、ユーザーに見せるのは 「これが気に入ってもらえる」という確信が高まったアイデアに限定する。 過程のすべてをユーザーに垂れ流さない。

§6デザイン原則

ヒーローは命題である

ファーストビューにはその主題世界で最も特徴的なものを置く。 形式は問わない（見出し・画像・アニメーション・ライブデモ・インタラクティブ瞬間）。 「大きな数字 + 小さなラベル + 統計 + グラデーションアクセント」はテンプレート解答であり、 それが本当に最善の場合にのみ選ぶ。

タイポグラフィは個性を運ぶ

ディスプレイと本文のフォントを意図的にペアリングする。 「どのブリーフでも選びがちなフォント」は使わない。 タイポグラフィの処理自体がデザインの記憶に残る部分になるよう設計する。

構造は情報である

ナンバリング・眉毛テキスト・仕切り・ラベルはコンテンツの真実を符号化するもの。 装飾目的では使わない。 「01 / 02 / 03」の番号付きマーカーは、コンテンツが実際に順序を持つ場合にのみ適切。 汎用パターンとして採用する前に「順序に意味があるか」を必ず確認する。

モーションは意図的に使う

アニメーションが主題に奉仕できる場所を考える（ページロードシーケンス・ スクロールトリガーリビール・ホバーマイクロインタラクション・環境的アトモスフィア）。 散漫なエフェクトより、オーケストレーションされた一瞬の方がインパクトが大きい。 ただし「アニメーションが多いほど AI 生成っぽく見える」ことも事実。少ない方が良い場合もある。

複雑さはビジョンに合わせる

マキシマリスト方向は精緻な実行が必要。ミニマル方向は余白・タイポ・ディテールの精度が命。 エレガンスとは「選んだビジョンをうまく実行すること」である。

品質フロアは黙って達成する

宣言せずに以下を確保する。

  • モバイルまでレスポンシブ
  • キーボードフォーカスが視覚的に見える
  • prefers-reduced-motion を尊重

大胆さは 1 か所に集中させる

シグネチャ要素を唯一の「記憶に残るもの」にする。 それ以外は静かに規律を保ち、シグネチャを引き立てない装飾は切り捨てる。 「リスクを取らない」こと自体がリスクになりうる。

§7文章・コピーライティング

デザインブリーフにリアルなコピーがない場合、Claude 自身がコピーを書く必要がある。 コピーはデザインと同様にテンプレート的になりうるため、以下の原則に従う。

原則内容
目的を先に問う 「このデザインが言うべきことは何か、それを最もよく言う方法は何か」を問い、書き始める
ユーザーの側から書く ユーザーが管理・認識するものを名前にする。システム内部の構造名は使わない
能動態をデフォルトに コントロールは「それを使うと何が起きるか」を正確に言う（「保存する」でなく「変更を保存」）
一貫した語彙 「公開」ボタンを押したら「公開しました」トースト。フロー全体で同じ言葉を使う
失敗と空状態 謝罪しない。何が起きたか・どう直すかを明確に。空画面は行動への招待
トーンの整合 ブランドと対象ユーザーに合わせた平易な語調。フィラー排除、各要素に 1 つの仕事
汎用コピーの罠 コピーはデザインと同じくらいテンプレート的になりうる。 「革新的なソリューション」「シームレスなエクスペリエンス」のような汎用フレーズは 特定のブリーフのコピーではなく、どのプロダクトにも当てはまる文句である。 ブリーフの具体的な主題・用語・世界観から語彙を引き出すこと。

§8制約と注意点

CSS 詳細度（Specificity）

セレクタの詳細度衝突 型ベースセレクタ（例: .section）と要素ベースセレクタ（例: .cta）が padding / margin で相殺し合うことが多い。 特に複数のセクション間のスペーシングでこの問題が起きやすい。 CSS を書く際はセレクタ階層と適用順序を意識する。

Tailwind バージョン

Tailwind v4 と v3 の違い Tailwind v4 は設定ファイル（tailwind.config.js）が不要になった。 v3 のルール（tailwind.config.js の theme.extend 等）を v4 プロジェクトに適用しないよう注意する。 プロジェクトの Tailwind バージョンを確認してから設定を書く。

ブリーフ優先の原則

ブリーフの言葉は常に勝つ。ブリーフが明確にビジュアル方向を指定している場合は それに正確に従う。「AI デザインのデフォルトを避ける」という原則は、 ブリーフが自由を与えている軸に限り適用する。

モーションとアクセシビリティ

アニメーションを追加する場合は prefers-reduced-motion: reduce メディアクエリで 動きを抑制するフォールバックを必ず用意する。
