§1概要

creating-new-project は ~/Projects/ 直下に Next.js プロジェクトを生成し、 開発着手可能な状態まで一括で整える orchestration スキル。 Phase 1（ヒアリング）から Phase 10（検証）まで順次実行し、 Phase 5（ポータル SPA）のみサブエージェント並列委任で高速化する。

§5Phase 2〜4: スキャフォールドと基盤構築

Phase 2: スキャフォールド

# Next.js のみ / Next.js + Supabase の場合
cd ~/Projects
npx create-next-app@latest <project-name> \
  --typescript --tailwind --eslint --app \
  --src-dir --import-alias "@/*" \
  --no-turbopack --use-npm

# React + Vite + FastAPI の場合（frontend/backend 分離構成）
cd ~/Projects && mkdir <project-name> && cd <project-name>
mkdir frontend backend
cd frontend && npm create vite@latest . -- --template react-ts
cd ../backend && python3 -m venv .venv && pip install fastapi uvicorn

§8サブエージェント委任仕様

Phase 5（ポータル SPA）は生成ファイル数が多いため、3 体のサブエージェントを並列で委任する。 担当領域が重ならないよう、prompt でファイルパスを明示する（重複すると書き込みが競合する）。 Phase 1〜4・6〜10 はメインエージェントが順次実行する。

担当領域subagent_typeprompt で渡す情報期待返却値
Phase 5: index.html + style.css + src/ worker-sonnet プロジェクト名・機能/画面リスト・portal-template/ を参照元として渡す 生成ファイルパス一覧
Phase 5: data/ 全体 worker-sonnet Phase 1 ヒアリング値（機能・画面・スタック）を渡す 生成ファイルパス一覧
Phase 5: 定型ファイル worker-haiku 定型テンプレートどおり作成する旨と対象パスを渡す 作成ファイルパス一覧
ファイルパス明示の必須化
3 体の担当領域（src/・data/・定型ファイル）が重ならないよう、 各 prompt に具体的なファイルパスを記載する。曖昧な指示は書き込み競合を招く。

§9エラーハンドリングと予想を裏切る挙動

エラーハンドリング

Phaseエラー対応
1プロジェクト名が重複（~/Projects/<name>/ が既存）AskUserQuestion で再入力を求める
2create-next-app 失敗 エラーログを表示し、AskUserQuestion で削除対象パスを提示・承認を得てから作成途中を rm -rf する
3設定ファイル Write 失敗ディスク容量・パーミッションを報告する
5ポータル SPA のファイル数が多いPhase 5 をサブエージェント並列で実行する（§7 参照）
8ポート範囲が枯渇AskUserQuestion でカスタムポート入力を求める
9gh repo create 失敗gh auth status で認証状態を確認し、認証切れなら報告する
10dev server 起動失敗lsof -i :<port> でポート競合を診断する

予想を裏切る挙動

Phase 2 の rm -rf は承認必須
create-next-app 失敗時の復旧で rm -rf する前に、必ず AskUserQuestion で削除対象パスをユーザーに提示し承認を得る。無承認の rm -rf は禁止。
Phase 8 直前のポート重複再確認
base_port は Phase 1 で動的検出するが、並行して他セッションが新規プロジェクトを作成中だと同じ値を提案しうる。Phase 8 で Edit する直前に再度重複を確認する。
プリフライト no-go を無視しない
Phase 3 末尾のプリフライトチェックが no-go を返した場合、Phase 4 に進んでも orchestrating-dev-flow が後続 Phase で動作しない。no-go を無視して先に進んではならない。
always/placement/directory-structure/rule.md のスタック整合
スタック（FE のみ / フルスタック）で許可ディレクトリが変わる。スタック選択と生成した許可リストの不一致は、後続の mkdir 時に advisory ノイズを生む。
Phase 5 のサブエージェント担当重複
3 体の担当領域（src/・data/・定型ファイル）を prompt でファイルパス単位に明示する。重複するとファイル書き込みが競合する。
Phase 10 のバックグラウンドジョブ競合
dev server / ポータル起動確認は & + kill %1 に依存する。同一シェルセッションに他のバックグラウンドジョブが %1 を専有していると誤ったプロセスを kill する。
