# ルーティン作成手順（creating）

`managing-agent-configs`（種別: routines） の **create モード** が参照する手順書。`conventions.md` を前段で読んだ前提で、実際の新規ルーティン作成の流れと作成後チェックリストを定義する。

このファイルを読み終えたら、ルーティンの設計書・実行プロンプトを作成し、作成後チェックリストを実行する。完了したら **自動的に review モード → test モード** へ連鎖する（ハブ SKILL.md の指示に従う）。

## 作成フロー

### Step 1: 要件の確定

**手順:**
1. ルーティンの目的・入出力をユーザーから聞き取る
2. `conventions.md` の実行環境判定基準でクラウド / ローカルを決定
3. `conventions.md` §6 の日次 / 週次判断基準で頻度を判定する:
   - 1 日遅延で被害拡大 → 日次
   - 蓄積型の改善で毎日の差分が小さい → 週次
4. profile.md の「有効なルーティン」表から現在のスロット使用状況を確認する
5. `conventions.md` §6 の提案手順に従い、`AskUserQuestion` でスケジュールをユーザーに提示する:
   - 日次 / 週次の判定結果と理由
   - 各スロット（A〜E）の現在の使用状況
   - 推奨スロット
   - 週次の場合は曜日の選択肢も提示
6. 不足情報は `AskUserQuestion` で 1 問ずつ確認

**完了条件:** 設計書の全必須フィールド（目的・実行環境・スケジュール・入力・出力・依存関係）が確定し、スケジュールが 5h グリッドのスロットに配置されていること

### Step 2: ディレクトリ作成

**手順:**
1. `routines/<project>/routines/<日本語名>/` を作成（例: `カバレッジ自動改善`）
2. **フォルダ名は日本語必須**。英語（kebab-case）は禁止
3. 既存ルーティンと名前が重複しないか確認

```bash
ls routines/<project>/routines/
```

**完了条件:** ディレクトリが日本語名で存在し、既存と重複していないこと

### Step 3: ルーティン設計書.md を作成

**手順:**
1. `conventions.md` の「5. ルーティン設計書.md の必須フィールド」に従い全フィールドを記入
2. 冪等性設計を必ず含める（再実行で issue/PR が重複しない仕組み）
3. 失敗時リカバリを必ず含める（通知先・リトライ戦略・手動介入手順）

**テンプレート:**

```markdown
# <ルーティン名>

## 概要

| 項目 | 値 |
|---|---|
| ルーティン名 | `<日本語名>`（例: `カバレッジ自動改善`）。フォルダ名と一致させる |
| 目的 | <1〜2 文で解決する問題を記述> |
| 実行環境 | クラウド Routines / ローカル CronCreate |
| スケジュール | `<cron 式>` (<タイムゾーン>) = <人間可読な説明> |

## 入出力

| 方向 | 内容 |
|---|---|
| 入力 | <何を読むか> |
| 出力（ログ） | `logs/<日本語ルーティン名>/<timestamp>.json`（必須） |
| 出力（アクション） | <アクション起因の issue/PR がある場合のみ記載。結果報告のみの issue/PR は禁止> |
| PR 影響度 | `影響なし`（マージまで自動）/ `影響あり`（PR 作成まで、E2E 必須）/ `PR なし` |

## 依存関係

- <ツール名>: <用途>（インストール方法: `<command>`）
- <サービス名>: <用途>（コネクタ要否: あり/なし）

## 処理フロー

Phase 1: <タイトル>
  └─ <概要>
Phase 2: <タイトル>
  └─ <概要>
...
Phase N: ログ出力
  └─ logs/<日本語ルーティン名>/<timestamp>.json に共通フォーマットで出力

## 冪等性設計

<再実行時に副作用が重複しない仕組みの説明>
例: issue 作成前に同一タイトルの open issue を検索し、存在すればコメント追記に切り替える

## 失敗時リカバリ

| 失敗パターン | 対応 |
|---|---|
| Phase N でエラー | <ログ記録して続行 / 中断> |
| 外部サービス不達 | <リトライ N 回 / スキップ> |
| 致命的エラー | <通知先: Slack #channel / issue 作成> |

## コネクタ要件（クラウドの場合）

- [ ] GitHub（issue/PR 操作）
- [ ] Slack（通知）— 不要なら削除
```

**完了条件:** 全必須フィールドが記入済み、冪等性設計と失敗時リカバリが空でないこと

### Step 4: 実行プロンプト.md を作成

**手順:**
1. `conventions.md` の「4. 実行プロンプト.md の必須構造」に従う
2. 事前準備セクションに以下の 2 ファイルの Read を含める（クラウドの場合は必須）:
   - `routines/<project>/共通ルール.md` — 12 観点・ルーティン一覧・prompt 登録形式
   - `routines/<project>/クラウド共通プロンプト.md` — git author・コミット規約・lint・PR テンプレ等
   - `routines/_shared/common-log-rules.md` — JSONL ログ出力ルール
3. 全 Phase に `### 完了条件` を付与
4. 自律実行ガードを記載
5. ツール・スクリプトのパスをリポジトリルート相対で記述
6. クラウド実行の場合、Phase 1 に依存ツールのインストール手順を含める

**完了条件:** 必須要素チェックリスト（conventions.md）を全て満たすこと。事前準備に `クラウド共通プロンプト.md` と `common-log-rules.md` の Read が含まれていること

### Step 5: ツール依存の確認（クラウドの場合）

**手順:**
1. `実行プロンプト.md` 内で参照している全ツール・スクリプトのパスを抽出
2. 各パスがリポジトリ内に存在するか `ls` / `find` で確認
3. 外部ツール（pip / npm 等）のインストール手順が Phase 1 に含まれているか確認
4. 環境変数の要否を確認し、必要なら設計書のコネクタ要件に追記

```bash
# 実行プロンプト内のパス参照を抽出
grep -oE '[a-zA-Z0-9_/./-]+\.(sh|py|js|ts|md|yml|yaml|json)' routines/<project>/routines/<name>/実行プロンプト.md
```

**完了条件:** 全参照パスの存在を確認済み、外部ツールのインストール手順が Phase 1 に含まれていること

### Step 6: ルーティン登録

登録・確認の操作手順は `references/cloud-operations.md` を参照。

**クラウド Routines の場合:**

1. `cloud-operations.md` の「新規登録」に従い `/schedule` でユーザーに登録してもらう
2. 登録後、`cloud-operations.md` の「Web UI 設定確認チェックリスト（7 項目）」で設定を確認する

**ローカル CronCreate の場合:**

`cloud-operations.md` の「ローカル CronCreate の場合」に従い CronCreate を実行する。

**完了条件:** クラウドの場合は `cloud-operations.md` の 7 項目チェックリストを全て確認済み。ローカルの場合は CronCreate が成功していること

## 作成前チェックリスト

- [ ] 既存ルーティンと目的が重複していないか（`ls routines/<project>/routines/` で確認）
- [ ] 実行環境を判定基準で正しく選択したか
- [ ] cron 式が `:00` / `:30` を避けているか
- [ ] 5h グリッドのスロット（05/10/15/20/00 JST）に配置しているか

## 作成後チェックリスト

- [ ] ルーティン設計書.md の全必須フィールドが記入済み
- [ ] 実行プロンプト.md の必須要素チェックリストを全て満たす
- [ ] ツール依存の確認が完了（クラウドの場合）
- [ ] ルーティン登録が完了または指示を提示済み
- [ ] git commit 済み（クラウドの場合は fresh clone に含まれる必要がある）
- [ ] **review モードへ自動連鎖**（managing-agent-configs（種別: routines） ハブが制御）
- [ ] **test モードへ自動連鎖**（review 完了後）

## 予想を裏切る挙動

- 作成しただけで終わらせない: ハブが自動で review → test へ連鎖する。連鎖を止めるのはユーザー明示時のみ
- 設計書と実行プロンプトは別物: 設計書は人間向けの仕様、実行プロンプトはエージェント向けの指示
- クラウドの `/schedule` はユーザーが手動実行する必要がある。Claude からは呼べない
- 実行プロンプト.md を git commit せずに `/schedule` しても、fresh clone に含まれず動作しない
