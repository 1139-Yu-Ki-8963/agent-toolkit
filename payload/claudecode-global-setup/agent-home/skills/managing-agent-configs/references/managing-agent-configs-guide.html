managing-agent-configs

スキル・hooks・rules・ルーティン・サブエージェントという 5 種のアセットについて、作成 → 観点ベース静的レビュー → 白紙実行者による実機検証を 1 つの動線で担うオーケストレーターのガイド。

§1概要

このスキルの核は 3 つある。① 作りっぱなしを許さない — 作成したアセットは自動的に静的レビューへ、レビューが済んだら実機検証へ連鎖する。② 6 種のアセットを 1 つの動線で扱う — フロー構造は共通化し、種別ごとの規約・観点・手順は references/<type>/ に分離して必要時のみロードする。③ 検証は白紙の実行者が行う — 書き手は自分のプロンプトの品質を客観視できないため、毎回新規のサブエージェントに実際に実行させて評価する。

設計思想: ① 作りっぱなしを許さない（create したら review → test まで自動連鎖） ② 骨格（モード判定・連鎖制御・完了報告）はハブに集約し、型別の詳細は references/ に分離 ③ 段階的開示 — ハブ本体は「種別判定 → モード判定 → 該当 references のロード指示」のみを行う ④ 6 種の統合であって画一化ではない — 共通化するのは「フロー構造」であり「内容」ではない

§2出力

create → review → test の連鎖が完了すると、以下の状態が確立される。

配置済みアセット
規約（references/<type>/conventions.md）に沿って新規配置、または既存の内容が規約準拠の形へ修正されたアセット本体（SKILL.md・hook script・rule.md・ルーティン設計書・subagent 定義等）。配置先は種別ごとの正しいディレクトリに確定している。
レビュー結果
観点別（種別ごとの reviewing.md が定義する観点）に検出した CRITICAL / WARN / INFO の件数と、修正が必要な箇所への対応履歴。CRITICAL が 0 件、または修正承認済みの状態になっている。
テスト完了マーカー
新規サブエージェントによる実機検証が全項目 PASS した場合にのみ書き出される、report ファイル（REVIEW-TEST-VERDICT: PASS 行を含む）とハッシュ照合マーカーの 2 点セット。git commit 時の commit gate は、この 2 点の実在・鮮度・ハッシュ一致を検証してから commit を許可する。
完了報告
対象種別・アセット名・通過モード、review 結果（CRITICAL/WARN/INFO 件数と修正件数）、test 結果（発火 PASS/FAIL 件数と停止理由）、健全性判定を 1 つの報告にまとめたもの。

§3対象アセット 6 種

ユーザー発話・対象ファイルパスから種別（asset_type）を 1 つ判定する。複数候補がある場合は質問で確認する。誤判定のまま進めると誤った references をロードしてしまうため、判定はフローの最初に置く。

種別判定キーワード・対象ファイル配置先（例）自動連鎖
skillsスキル・SKILL.md~/.claude/skills/<name>/SKILL.mdcreate → review → test
hooksフック・settings.json の hooks配置判定は references/hooks/conventions.mdcreate → review → test
rulesルール・rules ディレクトリ<category>-rules/rule.mdcreate → review → test
routinesルーティン・cron 登録<project>/routines/<name>/create → review → test
subagentsサブエージェント・agents ディレクトリ~/.claude/agents/<name>/<name>.mdcreate → review → test

§4モードと自動連鎖

種別が決まったら、動詞・キーワードからモードを判定する。判定が曖昧な場合や「全部やって」の場合は create フル連鎖に倒す。

動詞・キーワードモードロードする references連鎖
作る・追加・設計・改善・新規createconventions.md + creating.md→ review → test
レビュー・監査・点検・診断reviewconventions.md + reviewing.md（hooks / rules は check-items.md も）→ test
テスト・発火検証・実機検証testconventions.md + testing.md（終端）

共通の前段（どのモードでも最初に実行）

  • 進捗の可視化 — 主要 Phase をタスクとして登録し、開始・完了で状態を更新する
  • 種別判定 — §2 の表で asset_type を確定する
  • 規約のロード — references/<asset_type>/conventions.md を読む。これが型別の正本で、読まずに作成・点検・検証を始めない
  • 外部正本の参照 — 種別によっては設計ガイド（hooks / rules / subagents の design ドキュメント）も併読する

連鎖の中断制御

各モード境界で自動連鎖する直前に、次のいずれかに該当する場合のみ質問を挟む。それ以外は連鎖を継続するのが既定動作 — 「作って → 検証なし」を許す既定はバグの温床になる。

  • 直前モードで CRITICAL 級の問題が検出され、かつ修正が未完了
  • 直前モードがユーザーの判断待ち（自動修正の可否・配置移動の可否など）で止まった
  • ユーザーが事前に「作るだけ」「レビューまで」「dry-run のみ」を明示している

§5create モード

規約（conventions.md）と手順（creating.md）をロードしてから新規アセットを書く。skills 種別の要点は次のとおり。

規律内容
description 規律説明文は 50 字以内。TRIGGER when: と SKIP: の固定キーワードを必ず含める（発動条件の解析対象）
文字数予算全スキル合計の description を 2,000 字以内に維持する。超過すると起動時に後方のスキルが読み込まれなくなる
Type 決定木orchestration / gateway / gate / audit / verification / reactive / reference / transform / action の 9 種から 1 つに絞る
フロー系の追加必須セクションType が orchestration / gateway、または Phase 見出し 3 つ以上のスキルは「完了条件（Goal 行つき）」「サブエージェント委任仕様」「ループ設計（反復条件・上限・停止条件）」を持つ
本体サイズSKILL.md は 500 行以下。詳細は references/ に分離（公式ドキュメントのコピーは置かない）
作成後チェックスキル一覧（README）への登載・アンチパターン照合・review モードへの自動連鎖

hooks 種別は「配置の 2 軸判定（ownership × scope）」、rules 種別は「eager / lazy 判定と ADR 必須項目」、routines 種別は「実行環境判定とプロンプト形式」がそれぞれの creating.md の中心になる。

§6review モード

既存アセットを観点ベースで静的レビューし、検出した問題をユーザーの選択（全件 / 個別 / CRITICAL のみ / スキップ）に従って修正する。skills 種別の観点は 7 カテゴリ・32 項目。

カテゴリ見るもの代表例
A. frontmatter（9）name / description / invocation / type / allowed-tools の規約適合TRIGGER・SKIP の欠落は CRITICAL
B. description 品質（3）発火条件の具体性・誤発火リスク・境界スキルへの誘導「〜用」だけの抽象説明は WARN
C. サイズ・段階的開示（6）500 行制限・references 分離・予想を裏切る挙動の有無500 行超は CRITICAL
D. 単一責務（2）1 スキル 1 機能・共通手順のコピペ検出複数責務の詰め込みは分割提案
E. 副作用安全性（3）push / merge / 削除系のオート発火可否破壊的操作の自動発火は CRITICAL
F. ツール活用（13・フロー系のみ）Phase 分割・完了条件・ループ設計・委任仕様Phase の完了条件欠落は CRITICAL
G. 登録・整合性（3）配置パス・一覧への登載・見出しの統一絶対パス直書きは CRITICAL
健全性の目安: CRITICAL 0 件・WARN 3 件以下・INFO 制限なし・test 連鎖の発火検証が全 PASS（誤発火 0 件）。すべて満たした場合のみ「健全」と報告する。

hooks / rules の review には full（修正あり・既定）と dry-run（読み取り専用・連鎖なし）の 2 系統がある。「診断」「読み取りのみ」と言われたら dry-run。

§7test モード（白紙実行者ループ）

プロンプトの書き手はその品質を正確に判断できない。書き手が明確だと思うほど、別の実行者はつまずきやすくなる。test モードの核心は、バイアスのない実行者に指示を実際に実行させ、双方向で評価し、改善が横ばいになるまで繰り返すことにある。

評価軸

軸取得方法意味
成否[critical] 項目がすべて 〇 のときのみ成功最低基準
精度要件チェックリストの達成率（〇=満点・×=0・部分=0.5）部分的成功の度合い
ステップ数・所要時間実行メタデータ指示の無駄・認知負荷の代理指標
不明瞭な点・独自補填実行者のセルフレポート定性的な改善材料（こちらを主とする）

レッドフラグ（合理化に注意）

  • 「自分で再読しても同じ効果がある」— 自分のテキストは客観視できない。必ず新しい実行者を使う
  • 「シナリオ 1 つで十分」— 1 つだと過学習する。最低 2 つ、理想は 3 つ
  • 「複数の不明瞭な点を一度に解消しよう」— 何が効いたか分からなくなる。1 回 1 テーマ
  • 「同じ実行者を再利用しよう」— 前の改善を学習してしまっている

§8hook 連携（commit gate）

hooks を併用する環境では、レビュー・テストの完遂をコミット時に機械強制できる。連携は 3 つのマーカーで動く。

マーカーいつ書かれるか役割
needed マーカー管理対象ファイル（SKILL.md・rule.md・hook script 等）の編集を検知した時「このセッションでレビューが必要」の印。同時に以前のテスト完了マーカーを無効化する（編集後は再テストが必要なため）
test-passed マーカーtest モードの全項目 PASS 時git commit の許可条件。commit gate（PreToolUse hook）がこのマーカーの有無で判定する
自動清掃セッション終了時マーカーは揮発領域（${TMPDIR}/claude-hooks/<session>/）に置かれ、セッション終了 hook が削除する

この機構により「レビューせずにコミットする」経路が構造的に塞がれる。hooks を使わない環境ではこの節は適用されず、連鎖の既定 ON が防衛線になる。

§9references の地図（段階的開示）

ハブ本体（SKILL.md）は骨格だけを持ち、型別の知識は 36 ファイルの references に分離されている。ロードは「必要になってから」— これがコンテキスト消費を抑え、6 種を 1 スキルに統合しても破綻しない理由になる。

種別ファイル役割
共通の型
（各種別にある）		 conventions.md型別規約の正本。全モードで最初に読む
creating.md作成手順とチェックリスト
reviewing.md観点と Phase の定義
testing.md実機検証の手順と失敗パターン台帳
check-items.md観点別の機械検出式（skills / hooks / rules のみ）
skills 固有 folder-structure.mdフォルダ構成の推奨手順
description-examples.mddescription の書き方詳細例
anti-patterns.mdアンチパターン集
advanced-techniques.md高度な設計テクニック
hooks 固有 event-recipes.mdイベント別の入出力レシピ
examples.md実在フックの実例カタログ
output-schema.mdhook 出力 JSON スキーマの完全リファレンス
routines 固有cloud-operations.mdクラウド実行ルーティンの登録・確認・変更手順
workflowsworkflow-documentation.mdワークフロー文書の作成手順・テンプレート・記法（この種別はこの 1 ファイルで完結）

このほか assets/ に型別テンプレート 3 種（手順型・条件付き知識型・強制型）を同梱しており、create モードの雛形として使う。

§10完了報告

最終モードまで完了したら、次の項目を能動文の 1 報告にまとめる。作業ログの羅列ではなく「何がどういう状態で完了したか」を答える。

── 完了報告に含める項目 ─────────────────────────────────────

  対象:      種別（skills / hooks / rules / routines /
             subagents / workflows）・アセット名・通過モード
  create:    作成したファイルのパス
  review:    CRITICAL / WARN / INFO の件数、修正件数、未対応件数
  test:      発火 PASS / FAIL 件数、
             停止理由（収束 / 発散 / リソース上限のどれか）
  健全性:    reviewing.md の目安（CRITICAL 0・WARN 3 以下・誤発火 0）
             に照らした判定
