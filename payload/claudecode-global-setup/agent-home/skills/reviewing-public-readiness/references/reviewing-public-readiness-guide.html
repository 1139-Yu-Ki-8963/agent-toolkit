公開可否レビュー
reviewing-public-readiness

リポジトリ公開前に機密混入・履歴漏洩をスナップショット全体で走査するスキル。

こんな人向け
リポジトリを OSS 化・public 切り替えする前に、機密情報の混入や author 情報の露出を検査したい開発者。または [PUBLISH-*] タグ注入を受けてコミット時に自動チェックを走らせたい場合。
こんな場面で使う
OSS 化判断、private → public 切り替え前、公開前の最終チェック、[PUBLISH-AUTHOR] / [PUBLISH-SAFETY] / [PUBLISH-SAFETY-FULL] が PreToolUse フックから注入されたとき。PR レビュー・差分セキュリティレビュー・単一ファイルチェックには使わない（SKIP 対象）。
対応 OSmacOS
検証状況SKILL.md を正本として整理（個別の実機検証記録なし）
依存allowed-tools: Agent, Bash, Read, Grep, Glob。外部コマンド: git, git grep, find
関連資料SKILL.md、references/detection-patterns.md

§1概要

reviewing-public-readiness は、リポジトリを公開する前に機密ファイル・履歴漏洩・author 情報・内部限定情報・高エントロピー文字列など 12 観点でリポジトリ全体を走査する監査スキル。

最大の特徴は 差分ではなくリポジトリ全体のスナップショット を見ること。PR レビューや差分セキュリティレビューとは対象が異なる。

§2出力

走査が完了すると、以下の内容が確立される。

判定サマリー
対象リポジトリ・可視性（public/private）・ブランチを明示したうえで、CRITICAL / WARN / INFO の件数と現在の author 情報（過去の author 数を含む）から導いた結論ラベル（公開不可 / 条件付き公開可 / 公開可）を提示する完了報告。質問・確認は書かず、判定のみを示す。
詳細レポート（要求時のみ）
ユーザーから「詳細」「CRITICAL の中身」等の追加要求があった場合に限り出力する、該当カテゴリごとの検出箇所と対処コマンド例。CRITICAL 項目であっても自動修正は行わず、指摘に留める。

§3発火条件と SKIP

TRIGGER（発火する場面）

シーン例
OSS 化の判断時「このリポジトリ、OSS 化しても大丈夫？」
public 切り替え前「private → public にするけどリスクある？」
コミット時の自動発火PreToolUse フックから [PUBLISH-AUTHOR] [PUBLISH-SAFETY] [PUBLISH-SAFETY-FULL] 注入時（~/.claude/settings.json 経由）
公開前の最終チェック「公開前チェックして」

SKIP（発火しない場面）

SKIP 対象：PR レビュー・差分セキュリティレビュー・単一ファイルチェック。
これらは security-review スキルまたは個別 Read/Grep で対応する。

hook 連携による自動発火

~/.claude/settings.json に登録された PreToolUse フックがコミット時に以下のタグを注入する。タグ別の意味は次のとおり。

注入タグ主な確認観点
[PUBLISH-AUTHOR] author 情報（個人実名・会社メール）の妥当性確認（カテゴリ 9 相当）
[PUBLISH-SAFETY] 機密ファイル名を列挙した CRITICAL 観点の優先確認（カテゴリ 1 相当）
[PUBLISH-SAFETY-FULL] 12 カテゴリ全体をスナップショットで走査するフルレビュー

タグが注入されていない場合でも、ユーザーが明示的に公開前チェックを依頼すれば同等の走査を行う。

§4基本ワークフロー（6 ステップ）

# Step 1: 対象確定
git rev-parse --show-toplevel          # リポジトリルートを取得
git remote -v                          # public / private を判別

# Step 2: author 情報取得
git config user.name
git config user.email
git log --all --format='%aN <%aE>' | sort -u   # 全 author 一覧

# Step 3: 走査（12 カテゴリ、Bash + git grep のみ）
git ls-files | grep -E '<機密ファイルパターン>'
git grep -nE '<高エントロピー正規表現>'
git log --all --full-history -- '*.env'
find . -size +10M

# Step 4: 分類（自動）
#   [CRITICAL] → 公開即被害確定
#   [WARN]     → 公開可だが対処推奨
#   [INFO]     → 整備推奨

# Step 5: 完了報告（質問・確認は一切書かない）
#   「## 公開可否レビュー」のフォーマットで出力

# Step 6: 追加要求への対応
#   ユーザーから「詳細」「CRITICAL の中身」等を聞かれた時のみ出力
制約：スクリプトファイル（.sh .py .js 等）の作成は禁止。すべて Read / Grep / Glob / 許可済み Bash コマンドで完結させる。
完了報告では質問・確認を一切書かない。詳細はユーザーが自発的に問い合わせた時のみ出力する。

例示ファイルの実値混入：ファイル名が *example* *sample* でも、内容を grep して高エントロピー文字列が含まれれば CRITICAL に格上げ。

§6重大度と結論ラベル

ラベル意味結論
CRITICAL 公開即被害確定（実鍵・履歴中の鍵） 公開不可
WARN 公開可だが対処強く推奨 条件付き公開可
INFO 整備推奨 公開可（INFO 項目は整備推奨）

絵文字は使わない。テキストラベルで統一（SKILL.md 制約）。
CRITICAL が 1 件でも存在すれば結論は「公開不可」。CRITICAL 0 件で WARN のみなら「条件付き公開可」。

CRITICAL の自動修正禁止

制約：CRITICAL 指摘はレポートのみ。自動修正はしない。修正は手動または managing-agent-configs（種別: skills、review モード）で対応する。

§8誤検知除外ルール

出典: references/detection-patterns.md — 誤検知除外ルール・ファイル名一致時の内容確認ルール

除外候補としてマークする箇所

除外対象理由
README.md docs/** 内の例示ドキュメントの説明用文字列
tests/fixtures/** __fixtures__/** **/*.fixture.*テストデータ
**/*.example.* **/*.sample.*例示ファイル（ただし内容に高エントロピー文字列があれば CRITICAL 格上げ）
コードコメント内 # example: ... // example: ...例示コメント
.gitignore 自身に書かれたパターン除外対象を列挙しているだけ
第三者 vendored ディレクトリ（vendor/** third_party/** plugins/marketplaces/** node_modules/** 等）自リポジトリの管理外
スキャナー本体・SKILL.md・references/detection-patterns.md 内のパターン定義テキスト自己言及ヒット

ファイル名一致時の 2 段判定（カテゴリ 1 拡張）

カテゴリ 1 でファイル名がパターンに合致しても、ファイル名だけで CRITICAL 確定としない。内容を grep して秘密値が含まれるか確認する。

ファイル内容判定基準（CRITICAL 維持の条件）
.env*[A-Z_]= の形式で値が設定されている（空でない値）
.npmrc_authToken= _password= _auth= のいずれかが含まれる（公開レジストリ URL のみは INFO 降格）
.pypircpassword = username = __token__ 等の認証情報を含む（リポジトリ URL のみは INFO 降格）
credentials.jsonclient_secret private_key token 等のキーが含まれる
*.pem *.key-----BEGIN [A-Z ]*PRIVATE KEY----- を含む（公開鍵のみは INFO 降格）
service-account*.jsonprivate_key キーを含む

内容判定で秘密値がなければ INFO 降格 + 除外候補マーク。第三者 vendored ディレクトリ配下のものはさらに除外候補として確定する。

走査前の必須手順：走査開始前に references/detection-patterns.md の誤検知除外ルールとファイル名一致時の内容確認ルールを必ず確認する。ファイル名一致だけで CRITICAL 確定はしない。

§9レポートフォーマット

完了報告のみ出力する。質問・確認は一切書かない。詳細はユーザーが自発的に問い合わせた時のみ出力する。

## 公開可否レビュー

**対象**: <repo> | **可視性**: <public/private> | **ブランチ**: <branch>

**[<結論ラベル>]** CRITICAL <N> / WARN <N> / INFO <N>
Author: `<name> <email>`（過去 <N> 名）

結論ラベルの選択基準

状況結論ラベル
CRITICAL が 1 件以上[公開不可]
CRITICAL 0 件・WARN が 1 件以上[条件付き公開可]
CRITICAL 0 件・WARN 0 件（INFO のみ）[公開可]

詳細要求への対応

ユーザーから「詳細」「CRITICAL の中身」等を聞かれた時のみ、該当カテゴリの検出箇所と対処コマンド例を出力する。完了報告に含めない。 

# 詳細要求時の出力例（カテゴリ 1 CRITICAL の場合）
## カテゴリ 1: 機密ファイル混入 [CRITICAL]

検出ファイル:
  .env.production  （内容確認: KEY=<実値あり>）

対処コマンド例:
  git rm --cached .env.production
  echo '.env.production' >> .gitignore
  # 履歴から削除が必要な場合:
  git filter-branch --force --index-filter \
    'git rm --cached --ignore-unmatch .env.production' \
    --prune-empty --tag-name-filter cat -- --all

§10関連スキルと制約

他スキルとの役割分担

スキル役割分担
security-review PR・作業ブランチの差分・コード脆弱性レビュー。本スキルは差分ではなくリポジトリ全体のスナップショットを対象とする
reviewing-single-pr-with-inline-comments PR の取得・コメント投稿。本スキルは PR を扱わない
grouping-commits コミット作成。本スキルが PreToolUse で並行発火し、機密混入が検出された場合は grouping-commits を中断する
managing-agent-configs（種別: hooks） フックの作成・レビュー・実機検証。本スキルを発火させる [PUBLISH-AUTHOR] [PUBLISH-SAFETY] [PUBLISH-SAFETY-FULL] フックの実体は settings.json

スキル制約一覧

スクリプトファイル禁止：.sh .py .js 等を作成しない。すべて Read / Grep / Glob / 許可済み Bash で完結。
絶対パス記述禁止：SKILL.md 内では ~/agent-home/skills/... 形式を使う（ハードコードしない）。
質問しない：完了報告のみ。判断はユーザーに委ねる。詳細はユーザーが問い合わせた時のみ出力する。
再帰防止：直前ターンで同一リポジトリのレビューを起動済みなら、初回結果を再利用する。
CRITICAL 自動修正禁止：CRITICAL 指摘はレポートのみ。修正は手動または managing-agent-configs（種別: skills、review モード）で対応する。
