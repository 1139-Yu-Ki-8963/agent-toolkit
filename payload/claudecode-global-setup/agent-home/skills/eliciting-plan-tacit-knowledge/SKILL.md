# 計画暗黙知の抽出

計画書をコンテキストゼロの弱モデル（plan-comprehension-prober / Haiku）に初見読解させ、その理解と本来の意図とのズレから、計画が省略している前提（暗黙知）を抽出して計画へ明文化するフロー。

## 原理（なぜ弱モデルに読ませるか）

計画の書き手（上位モデル・人間）は、自分が持つコンテキストを前提に記述を省略する。書き手自身が読み直しても脳内で自動補完されるため、省略には気づけない。コンテキストゼロの弱い読み手に計画だけを読ませると、行間を埋められず省略箇所で理解が壊れる。壊れ方（誤読・推測・判断不能）が暗黙知の地図になる。

成立条件は 3 つ。1 つでも欠くと検出力が落ちる。

| 条件 | 実現手段 | 破るとどうなるか |
|---|---|---|
| 読み手がコンテキストゼロ | 新規サブエージェント委任（プロンプトに計画文のみ埋め込み） | メインセッションが初見読者を演じても意図を知っているため検出不能 |
| 読み手が弱いモデル | plan-comprehension-prober（model: haiku 固定） | 強いモデルは行間を補完してしまい、暗黙知への依存が顕在化しない |
| 「わからない」を言わせる形式 | 推測・判断不能の明示列挙を強制する 5 項目出力 | 曖昧な要約だけが返り、ズレの位置を特定できない |

## 使用タイミング

- brain の計画立案後・ユーザー承認前（推奨実行位置）
- 実装計画・移行計画・作業指示書など、別の実行者（人・エージェント）が読む文書の完成前
- 「計画の暗黙知を炙り出して」「前提を明文化して」と言われた時

## 機械強制

`ExitPlanMode`（計画をユーザー承認へ出す実ツール）の実行前に、本スキルの発火が `scripts/check-plan-tacit-knowledge-gate.sh`（PreToolUse(ExitPlanMode) hook）により機械検査される。セッションのスキル発火ログに本スキルの発火記録が無ければ `[PLAN-TACIT-KNOWLEDGE-GATE-BLOCK]` で exit 2 block される。発話マッチングによる自発的な起動判断には依存しない。

適用範囲の限界: 本ゲートが保証できるのは plan mode（`ExitPlanMode`）を経由する計画のみ。brain サブエージェント呼び出し等、plan mode を経由しない計画立案経路は対象外（Claude Code のネイティブツールとして存在する唯一の汎用的な「計画→承認」ポイントが `ExitPlanMode` であるため、これを起点に採用した）。判定は「スキルが発火したこと」のみを見る。Phase 5 の収束完了までは見ない（起動さえすれば内部ループは本スキル定義側が担保する）。

計画ファイル（`~/.claude/plans/*.md`）への最初の永続化（disk上に未存在の状態でのWrite）についても、`scripts/check-plan-draft-write-gate.sh`（PreToolUse(Write) hook）が同様に本スキルの発火有無を検査する。メインセッションのみ対象（`agent_id` が非空＝サブエージェントの計画下書きは対象外）。この新ゲートと既存の `check-plan-tacit-knowledge-gate.sh` は併存し、同一の `plan-tacit-knowledge-gate.consumed-count` マーカー（消費カウンタ）を共有する。どちらのゲートも「セッションのスキル発火ログの該当行数が消費カウンタを超えていること」を要求し、通過時に消費カウンタを現在の発火行数へ更新して発火を消費する。タイムスタンプ比較ではなく発火行数の単調増加で判定するため、ログに未来日時の ts が混入しても消費機構は無効化されない。この設計により、1 計画サイクルで「下書きの永続化前に 1 回（会話テキストに対するプローブ）・`ExitPlanMode` 承認前に 1 回（書き上がった計画ファイルに対するプローブ）」の計 2 回の検証が構造的に強制され、Write 後の Edit 変更が未検証のまま承認に到達する経路と、同一セッション内の 2 つ目以降の計画が過去の発火で素通りする経路の両方が塞がれる。

副作用として、Write 版ゲート通過後にその Write 自体が別の hook で block された場合、消費済みの発火は戻らず、再試行にはスキルの再実行が必要になる（3 回連続 block の自動解除が最終的な救済弁）。

**この拡張でも保証できないこと**: `eliciting-plan-tacit-knowledge` の実行自体がAgentツール呼び出し（計画全文をpromptに埋め込む）としてトランスクリプトに現れるため、その時点で計画内容は既に露出しうる。保証できるのは「計画ファイルへの最初の永続化は、最低1回の暗黙知チェックを経た内容になる」という限定的な性質のみ。

## Phase 1: 対象と意図ソースの確定

## Step 1-1: 対象計画と意図ソースの特定

1. 対象計画（ファイルパス、または会話中の計画テキスト）を特定し、ファイルの場合は Read で全文を取得する
   計画ファイルがまだ disk 上に存在しない場合（plan mode中の初回下書き作成前に `check-plan-draft-write-gate.sh` がblockした場合等）は、ファイルパスではなく会話中の計画テキストを直接使う。
2. 意図の正解ソース（ユーザーの元指示・issue・設計書・会話での決定事項）を特定し Read する。所在が不明な場合は Grep で該当箇所を探す

**完了**: 計画全文と意図ソースが手元に揃っている

## Phase 2: 理解プローブ

## Step 2-1: plan-comprehension-prober への委任

Agent ツールで plan-comprehension-prober へ委任する。prompt の構成:

- 冒頭に `[CHECKLIST-EXEMPT]`（外部事実の調査ではなく、埋め込まれた計画文の理解言語化のため）
- 計画全文をベタ書きで埋め込む
- 5 項目出力形式（エージェント定義の出力フォーマット）の遵守を指示する
- 意図・背景・補足情報を prompt に混ぜない。混ぜた分だけ検出力が落ちる

**完了**: 5 項目形式のプローブ報告を受領している

## Phase 3: ギャップ判定（呼び出し元が「決める」）

## Step 3-1: プローブ報告と意図の突合・4 分類

プローブ報告と意図ソースを突合し、ズレを 4 分類する。

| 分類 | 定義 | 対応 |
|---|---|---|
| 誤読 | 書いてあるのに別の意味に取られた | 誤解を誘導した記述を書き換える |
| 欠落 | 書かれておらず、推測で埋められた・埋められなかった前提 | 明文化して追記する（暗黙知の本体） |
| 曖昧 | 複数解釈が可能と申告された | 一意になるよう書き換える |
| 偶然の的中 | 推測したが偶然意図と一致していた | それも明文化する（次の読み手が同じ推測をする保証はない） |

**完了**: プローブ報告の全指摘（推測・判断不能・誤読）が 4 分類のいずれかに割り当てられている

## Step 3-2: 修正文の起草・確定

各ギャップへの追記・修正文をこの Step で起草し確定する（判断と反映の分離の「決める」工程）。

**完了**: 全ギャップに対応する修正文が確定している

## Phase 4: 計画への再統合

## Step 4-1: worker-sonnet への反映委任

Agent ツールで worker-sonnet へ委任し、確定した修正文を計画ファイルへ反映させる。

- 欠落・偶然の的中の追記は「## 前提・制約（暗黙知の明文化）」節へ集約し、誤読・曖昧は該当記述を直接書き換える
- 計画がファイルでなく会話中テキストの場合は、改訂版計画を本文で提示する（委任不要。Step 4-2 も省略）

**完了**: worker-sonnet から反映完了の報告を受領している

## Step 4-2: 反映の確認

Bash / Grep で「前提・制約（暗黙知の明文化）」節の存在と、確定した修正文が計画に反映されていることを確認する。

**完了**: 改訂済み計画に全修正が反映されている

## Phase 5: 収束確認

## Step 5-1: 再プローブ

改訂後の計画全文で Step 2-1 を再実行する（Agent ツールで毎回新規の prober 呼び出し。前回の対話を継続しない）。

**完了**: 改訂後計画に対する 5 項目形式のプローブ報告を受領している

## Step 5-2: 収束判定と報告

1. 新規の「欠落」「曖昧」が 0 件なら収束
2. 残があれば Phase 3 〜 5 を繰り返す（ループ設計の上限まで）

**完了**: 収束、または上限到達時に残ギャップをユーザーへ報告済み

## 完了条件

| Phase | 完了条件 |
|---|---|
| Phase 1 | 計画全文と意図ソースが手元に揃っている |
| Phase 2 | 5 項目形式のプローブ報告を受領している |
| Phase 3 | 全ギャップが 4 分類され修正文が確定している |
| Phase 4 | 改訂済み計画が存在する |
| Phase 5 | 収束（新規の欠落 + 曖昧 0 件）または残ギャップ報告済み |
| **Goal** | 抽出された暗黙知がすべて計画に明文化され、初見の読み手が意図どおりに理解できる計画になっている |

## サブエージェント委任仕様

| 呼び出し箇所 | subagent_type | prompt 骨格 | 期待返却値 |
|---|---|---|---|
| Phase 2 / Phase 5 | plan-comprehension-prober | `[CHECKLIST-EXEMPT]` + 計画全文ベタ書き + 5 項目出力形式の遵守指示。意図・背景は渡さない | ①目的の言い換え ②手順の再説明 ③推測で補った前提 ④判断できなかった点 ⑤最初の一手 |
| Phase 4 | worker-sonnet | 対象ファイルパス + 確定済み修正文（変更前後の対）+ スコープ外変更の禁止 + 完了条件（grep で確認可能な基準） | 反映結果と変更箇所の報告 |

## ループ設計

| 要素 | 内容 |
|---|---|
| 反復条件 | Phase 5 で新規の欠落・曖昧が検出されたら Phase 3 → 4 → 5 を再実行する |
| 上限回数 | 2 周 |
| 停止条件 | 収束停止（新規の欠落 + 曖昧 0 件）/ リソース上限（2 周到達）/ 発散検知（同一ギャップが 2 回連続で解消されない） |

検証役（plan-comprehension-prober）と生成役（呼び出し元セッション + worker-sonnet）は分離されている。

## 重要な注意事項

- prober に意図・背景・補足を渡さない（コンテキスト混入は検出力の低下に直結する）
- prober の誤読を「読み手の間違い」として扱わない。誤読は計画側の記述欠陥のシグナル
- 合否（PASS/FAIL）は宣言しない。収束判定は件数基準の機械的判定
- 同種の暗黙知が複数の計画で繰り返し抽出される場合、rules・テンプレート化の候補としてユーザーへ報告する

## 予想を裏切る挙動

- 1 周目で「全部合っている」報告が返った場合、成功と即断せず prompt への補足情報混入を疑う（意図まで書いてあれば当然一致する）
- 「偶然当たった推測」も修正対象になる。合っていたから放置、は次の読み手で再発する
- 再プローブは毎回新規呼び出し。同一 prober へ「修正版です、再確認して」と続けると前回文脈で補完され検出力を失う

## 完了報告

`~/agent-home/skills/managing-agent-configs/references/skills/completion-report-format.md` の共通骨格（作業報告型）に従う。固有の検証行: 抽出ギャップ件数（4 分類別）・収束までの周回数・計画への追記/書き換え件数。

## 設計判断

**必要性**: 「brain の計画立案後・ユーザー承認前に本スキルを通す」という運用が、Claude 自身の発話マッチングによる自発的判断に依存していたため、機械強制への格上げをユーザーから明示的に指示された（2026-07-12「スキルの呼び出しはしません。仕組みとして成立をさせろ」）。`ExitPlanMode` は plan mode の計画をユーザー承認へ出す唯一の汎用ネイティブツールであり、この直前に本スキルの発火有無を検査することで、発話マッチングに頼らない構造的な強制が実現できる。

**実機検証で判明した既知の限界（意図的な改竄は対象外）**: カウンタファイル・スキル発火ログはいずれも内部マーカーであり、これらを手動で書き換える（偽の発火ログを追記する、カウンタに負数を書く等）ことでゲートを回避することは技術的に可能。本ゲートは「発話マッチングへの依存を排し、通常の使用フローで機械的に強制する」ことが目的であり、内部マーカーの意図的な改竄という別の脅威モデルへの対策（暗号学的な実行証明等）は過剰設計として対象外とした。カウンタが非数値・負数など不正な値を持っていた場合に算術エラーでクラッシュしfail-openになる問題のみ、通常運用でも発生しうる堅牢性の欠陥として修正済み（数値検証を追加し、不正値は0として扱う）。

**適用範囲拡張（Write時点のゲート追加）の経緯**: ユーザーから「`ExitPlanMode`のみのゲートでは、計画ファイルが Phase 4 で既に永続化された後にしか発火せず、ユーザーが最初に見る計画は無検証のままではないか」との指摘を受けた。実際に plan mode を1サイクル観察し、計画ファイルへの Write が `ExitPlanMode` より先に行われることを実地で確認した。これを受けて `check-plan-draft-write-gate.sh`（PreToolUse(Write)）を追加し、計画ファイルへの最初の永続化時点でも本スキルの発火を要求するようにした。ただしこの拡張でも「ユーザーの目に触れる経路をすべて塞ぐ」ことは達成できない（本スキルの実行自体がAgent呼び出しとしてトランスクリプトに現れるため）。過大な保証を主張しないことを明記した上で採用した。

**厳格化改訂（2026-07-12・同日レビューによる）**: 初版の Write 版ゲートは「セッション内でスキルが 1 回でも発火したか」の存在チェックだったが、レビューで「Write 前の 1 回の発火が両ゲートを同時に満たし、初回サイクルでは Write 後の Edit 変更が再検証されないまま `ExitPlanMode` 承認に到達できる」穴が確認された（既存ゲートの last-pass マーカーが初回は初期値のため、同じ発火ログエントリで通過する）。ユーザー承認のうえ、Write 版も last-pass 比較 + 通過時消費へ変更し、両ゲートでマーカーを共有する設計に統一した。コストは 1 計画サイクルあたりプローブ実行が最低 2 回になること。

なお厳格化直後の実機検証で、当初のタイムスタンプ比較（発火 ts の最大値 > 前回通過時刻）は、ログに未来日時の ts が 1 件混入すると消費機構が恒久的に無効化されるバグを持つことが実証された。判定を発火行数の単調増加カウンタ（発火行数 > 消費済みカウンタ）へ即日変更し、クロック非依存・未来日時耐性を持たせた。

**代替案を採用しなかった理由**:
- advisory のみ（block なし）: Claude が無視・省略できる経路が残り、「仕組みとして成立」の要求を満たさない
- PostToolUse(Agent) で subagent_type=brain を検知して advisory: brain を経由しない計画立案経路もあり、かつ block ではなく advisory では同じ問題が残る
- プロンプト埋め込み型の例外チャネル（`[CHECKLIST-EXEMPT]` 相当）: `ExitPlanMode` の tool_input には計画本文が含まれず（allowedPrompts のみ）、この方式が使えない
- `/tmp/.allow-*` 形式のワンショット例外フラグ: `managing-agent-configs` の hooks レビュー観点 G1 で CRITICAL 相当のアンチパターンとされているため不採用。カウンタ方式の自動解除（3 回連続 block で解除）のみを例外機構とした

**保守責任者**: 人手（ユーザー）。スキル名・タグ名を変更する場合は hook 本体・settings.json の登録・本節を同時に更新する。

**廃棄条件**: `eliciting-plan-tacit-knowledge` スキル自体が廃止された時、または Claude Code 本体が plan mode 突入前のスキル実行強制を標準機能として提供した時。
