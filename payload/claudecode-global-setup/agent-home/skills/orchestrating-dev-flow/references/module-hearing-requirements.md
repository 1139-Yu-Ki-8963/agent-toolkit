# hearing-requirements（orchestrating-dev-flow 内部モジュール）

計画のあらゆる側面について、共通理解に到達するまで徹底的にヒアリングする。設計ツリーの各分岐を1つずつ辿り、判断の依存関係を順番に解消していく。

## 基本ワークフロー

1. ユーザーが計画・設計を提示する
2. 設計ツリーの全分岐を洗い出す
   - 対象画面の特定（既存画面の変更か新規画面か）
   - UI 変更の有無（screen 型モック / spec 型モックの判定材料）
   - UI 変更ありの場合: レイアウト・コンポーネント配置の方針
3. 各分岐について 1 問ずつ質問する（推奨回答を添える）
4. ユーザーの回答を受けてから次の質問に進む
5. 共通理解に到達するまで繰り返す

## 重要な注意事項

- **1 問ずつ質問する**: 複数の質問を一度に出さない。一度に複数の質問を浴びせると混乱を招く
- **推奨回答を添える**: 各質問には自分の推奨回答を提示する。ユーザーが判断しやすくなる
- **コードベースで答えられる質問は自分で調べる**: ユーザーに聞く前にコードを探索して解決できるなら、そちらを優先する
- **依存関係を意識する**: 判断の依存関係を 1 つずつ解消していく。前提が決まっていない質問を先に出さない
- **対象画面がある場合は画面基本設計書を Read する**: `docs/個別設計/画面/<画面名>/画面基本設計書.md` が存在すれば Read して既存構造を把握してから質問する。ユーザーに聞く前にコードと設計書から分かることは自分で調べるルールの延長

## 予想を裏切る挙動

- 質問を一度にまとめて出すとユーザーが圧倒される。必ず 1 問ずつ、回答を待ってから次へ
