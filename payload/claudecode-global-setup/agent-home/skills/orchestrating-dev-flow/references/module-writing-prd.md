# writing-prd（orchestrating-dev-flow 内部モジュール）

現在の会話コンテキストとコードベースの理解をもとに PRD を生成する。ユーザーへの追加ヒアリングはしない。既に議論した内容を整理・構造化するだけ。

## 基本ワークフロー

1. コードベースを探索し、現状を把握する（未実施の場合）。プロジェクトのドメイン用語を PRD 全体で使い、関連する ADR（アーキテクチャ決定記録）を尊重する
2. 機能をテストする継ぎ目（seam）をスケッチする。既存の seam を新規より優先し、可能な限り高い位置の seam を使う。seam は少ないほどよい（理想は 1 つ）。ユーザーに seam が期待と合っているか確認する
3. 以下の PRD テンプレートに従って PRD を作成し、ユーザーの承認を得てから issue tracker に公開する

## PRD テンプレート

### 課題（Problem Statement）

ユーザー視点で直面している課題を記述する。

### 解決策（Solution）

ユーザー視点での解決策を記述する。

### ユーザーストーリー

網羅的なユーザーストーリー一覧。各ストーリーは以下の形式:

```
1. <役割>として、<機能>がほしい。なぜなら<利点>だから。
```

この一覧は機能のあらゆる側面を網羅する。

### 実装判断

議論で決まった実装に関する判断の一覧:

- 構築・修正するモジュール
- 修正するインターフェース
- 技術的な明確化事項
- アーキテクチャ判断
- スキーマ変更
- API 契約
- 特定のインタラクション

具体的なファイルパスやコードスニペットは原則書かない（すぐ陳腐化するため）。例外: プロトタイプで得たスニペットが散文より正確に判断を表現する場合（ステートマシン・リデューサー・スキーマ・型定義）は、判断に必要な部分のみ抜粋して含める。

### テスト判断

テストに関する判断の一覧:

- 良いテストの基準（実装詳細ではなく外部振る舞いをテストする）
- テスト対象モジュール
- 既存テストの先行例

### スコープ外

この PRD の対象外となる事項。

### 補足

機能に関する追加の補足事項。

## 重要な注意事項

- **ヒアリングしない**: このモジュールは議論済みの内容を整理するだけ。追加質問が必要なら `module-hearing-requirements.md` の手順を先に使う
- **ドメイン用語を使う**: コードベースの用語に合わせる。独自の用語を持ち込まない
- **ファイルパスを書かない**: 実装判断にファイルパスやコードを含めない（陳腐化する）。判断の本質を散文で表現する

## 予想を裏切る挙動

- seam のスケッチを省略するとテスト方針が曖昧になる。必ずユーザーに確認してから PRD を書く
