# formatting-pr（orchestrating-dev-flow 内部モジュール）

`gh pr create` の `--body` に流し込む本文を、リポジトリの PR テンプレートとガイドに厳密準拠する形で組み立てる。最後にセルフチェックして登録する。

**flow-values.yml（プロジェクトルートに配置）の参照キー**:

```yaml
pr:
  template: ".github/pull_request_template.md"   # 既定値
  guide: ".github/pull_request_guide.md"          # 存在する場合のみ読む（省略可）
  required_sections:                              # 省略時は以下 7 件を使用
    - "## 概要"
    - "## なぜこの実装か"
    - "## 影響範囲"
    - "## 確認方法"
    - "## テスト"
    - "## 生成プロンプト"
    - "## 実害検証"
  critical_globs:                                 # 実害検証を必須とするパターン（省略時は src/**・app/**・migrations/**）
    - "src/**"
    - "app/**"
    - "migrations/**"
  skip_globs:                                     # 実害検証を自動 skip とするパターン
    - "docs/**"
    - ".github/**"
```

---

## Phase 1: 前提確認

以下の条件をすべて満たしていない場合、処理を中断してユーザーに報告する:

1. 現在ブランチが `main` ではない
2. 直前に `git push` 済み（`git status` で `nothing to commit, working tree clean`、`git log @{u}..` が空）
3. `flow-values.yml の pr.template` で指定されたテンプレートファイルが存在する

条件をすべて満たしていれば、ユーザーへの確認なしに Phase 2 へ進む。

---

## Phase 2: テンプレート読込

以下のファイルを必ず Read する:

- `flow-values.yml の pr.template` で指定されたファイル（テンプレート本体）
- `flow-values.yml の pr.guide` で指定されたファイル（各セクションの書き方ガイド、存在する場合のみ）

このモジュールは、ガイドの内容を確認してから本文の組み立てを開始する。

---

## Phase 3: 本文の組み立て

会話コンテキスト（実装内容・テスト結果・コミット履歴）をもとに、以下の構造で本文を組み立てる。

```
<Closes #N>          ← issue 連携あり: 1 行目に Closes/Fixes/Resolves #N
                     ← 連携なし:       1 行目に「該当なし」

## 概要
変更の目的と背景を 1〜2 文。技術用語より「何が変わるか／なぜ変えるか」を優先。

## 変更フロー
mermaid 図で示す。省略する場合は <!-- 省略: 理由 --> をコメントで残す。
省略可能条件: 変更ファイルが 3 件以下、かつ新規 API がなく、かつ既存フローを変えない場合。

## なぜこの実装か
採用根拠を「〜なので〜にした」の構文で書く。

## 検討した代替案
不採用にした案とその理由。検討がない場合は「なし」。

## 影響範囲
- **変更ファイル**: 具体的パスで列挙（省略禁止）
- **影響する機能**: ユーザー操作の視点で書く
- **破壊的変更**: あり / なし

## 確認方法
レビュアーが動作確認する手順を番号付きで記載。

## テスト

### 実施済み
- [x] <レイヤー名>: <コマンド>（<件数> passed）
- [ ] E2E: <未実施なら理由>

### 未実施・残課題
- 「できなかった」だけでなく「なぜできなかったか」を書く

## 生成プロンプト

<以下の 3 形態から 1 つを選んで記載>

形態 A（人間実装）:
  手動実装

形態 B（インタラクティブセッション）:
  Claude Code インタラクティブセッション
  使用モデル: <model>
  主なプロンプト: <要約>

形態 C（ルーティン）:
  ルーティン: <ルーティン名>
  実行日時: <YYYY-MM-DD HH:MM JST>
  リポジトリ: https://github.com/<owner>/<repo>

## 実害検証

<必須対象なら 3 行テンプレ・自動 skip 対象なら「該当なし（<種別>）」を Phase 3.5 の規則に従って挿入>
```

---

## Phase 3.5: 実害検証セクションの自動挿入

PR 本文生成時にスキルが自動でテンプレ草稿を挿入する。**既存の `## 実害検証` セクションがある場合は上書きしない差分マージ方式**。

### Step 1: PR 種別判定

`git diff --name-only origin/main...HEAD` で変更ファイル一覧を取得し、`flow-values.yml の pr.critical_globs` と `pr.skip_globs` に照合して判定する。

| 判定 | 条件 | 自動挿入文言 |
|------|------|------|
| **必須対象** | `pr.critical_globs` のいずれかに 1 件でもマッチ | 3 行テンプレ（観察方法・期待挙動・観察結果） |
| **skip: docs のみ** | `docs/` のみ | `該当なし（docs のみ）` |
| **skip: CI 設定のみ** | `.github/` のみ | `該当なし（CI 設定のみ）` |
| **skip: メタ設定のみ** | `pr.skip_globs` のみ、かつ必須対象を含まない | `該当なし（メタ設定のみ）` |

判定優先順位は必須対象 > skip 各種。必須対象を 1 件でも含むなら必ず 3 行テンプレを挿入する。

### Step 2: 既存セクションの保存

`## 実害検証` 見出し直下に非空記述または「該当なし」記述が含まれる場合は**そのまま温存する**（上書き禁止）。

### Step 3: 自動挿入

必須対象なら以下を末尾に追加する:

```
## 実害検証

- 観察方法: <UI 操作 / API 呼び出し / E2E smoke / 赤チーム検証 のうち、本 PR で行ったもの>
- 期待挙動: <PR の意図する挙動>
- 観察結果: <ログ抜粋・スクショ・curl の status と body・PASS 件数>
```

自動 skip 対象なら以下を末尾に追加する:

```
## 実害検証

該当なし（<判定済みの種別>）
```

挿入後、Phase 4 のセルフチェックで `## 実害検証` セクション存在・本文非空の両方を再検証してから登録に進む。

---

## Phase 4: セルフチェック（必須）

組み立てた body 文字列に対し、以下の点を機械的に検証する。1 つでも欠けていれば本文を修正してから Phase 5 へ進む:

1. **1 行目**: `^(Closes|Fixes|Resolves)[[:space:]]+#\d+` または `該当なし` で始まる
2. **必須セクション**: `flow-values.yml の pr.required_sections` で指定されたすべての見出しが含まれる（既定 7 件）
3. **生成プロンプト直下**: `ルーティン:|手動実装|Claude Code インタラクティブセッション` のいずれかが含まれる
4. **タイトル prefix**: `【機能追加】|【バグ修正】|【ドキュメント】|【設定変更】|【リファクタ】|【テスト】|【スタイル】|【CI】|【パフォーマンス】|【取り消し】` のいずれかで始まる（scope を含める場合はコロン区切り 例: 【設定変更:routine】）
5. **実害検証セクション本文**: 必須対象 PR は `- 観察方法:` `- 期待挙動:` `- 観察結果:` の 3 行が空でない。自動 skip 対象 PR は「該当なし（<種別>）」が含まれる

---

## Phase 5: 登録

```bash
gh pr create \
  --base main \
  --head "<現在ブランチ名>" \
  --title "<prefix>: <短いタイトル>" \
  --body "$(cat <<'EOF'
<組み立てた本文>
EOF
)"
```

登録後、PR URL を表示する。CI 待ちは行わない。

**単独呼び出し時の責務**: `orchestrating-dev-flow` の Phase 9（PR 作成・マージ）経由でない場合は、続けて `references/module-fixing-review-findings.md` を Read して手順に従い、警告／critical の自動修正と自動マージまで完遂する。省略する例外はない。

---

## 予想を裏切る挙動

- 単独呼び出しでは登録後に必ず `references/module-fixing-review-findings.md` を Read して手順に従い、自動マージまで完遂する義務がある。プロジェクト固有の Stop hook がこの義務を機械強制している場合がある
- `## 実害検証` セクションはプロジェクト固有の hook が機械検査する場合がある。必須対象 PR では 3 行テンプレ（観察方法・期待挙動・観察結果）をすべて埋めること
- セクション見出しを 1 文字でも変えるとプロジェクト hook の正規表現マッチが外れてブロックされる場合がある。`flow-values.yml の pr.required_sections` と一致させること
- `pr.guide` が存在しない場合は Phase 2 のガイド読込をスキップし、テンプレートのみを参照する
- ルーティン経由で呼ばれた場合は形態 C（ルーティン）を選び、実行日時とリポジトリ URL を必ず埋める
- `pr.template` が `flow-values.yml` に未定義のプロジェクトでは既定値（`.github/pull_request_template.md`）を使う
