§1概要

orchestrating-dev-flow は開発タスクを受け取って 5 ルートのいずれかに分類し、 Phase 1〜11 の定型フローを通じて「計画 → 実装 → レビュー → マージ」を一貫して制御するオーケストレーター。 各 Phase の詳細手順は references/phase-N-*.md に分離されており、着手時に Read して実行する（事前一括読み込み禁止）。

フロー全体は .claude/rules/always/project-context/flow-values.yml から注入されるプロジェクト固有コンテキスト （classify 閾値・ドメイン用語・設計規約・テスト規約・review gate マッピング）に基づいて動作する。 このファイルが存在しないプロジェクトでは creating-new-project スキルによるセットアップが先に必要。

スキップ条件
調査・質問のみ ／ agent-home 内スキル・ルール・フック管理（→ managing-agent-configs）／ 単発コミット（→ grouping-commits）には本スキルを使用しない。

§2出力

フローが完走すると、選択したルートの全 Phase が完了条件を満たし、以下の状態が確立される。

main へのマージ済み変更
5 ルートのいずれかを通過し、review gate・テスト・完了チェックを経て main ブランチにマージされたコード変更。config-with-review-and-verify ルートではドキュメント・skills・rules・hooks 等の設定変更が対象になる。
完了報告
managing-agent-configs/references/skills/completion-report-format.md の共通骨格（作業報告型）に従い、通過したルート識別子と完了 Phase 数を明記した報告。
後片付け済みの作業環境
Phase 10 でマージ後の worktree・feature ブランチ・使用ポートが削除され残留プロセスがない状態。Phase 11 で main ブランチが最新化されている。

§35 ルート一覧

Phase 1 でタスク内容と flow-values.yml の classify セクションを照合し、以下のいずれか 1 つを選択する。

識別子日本語名用途目安時間
feature-with-full-planning 機能実装（フル計画） 新機能追加・挙動変更を伴うバグ修正 60 分
feature-with-quick-delivery 機能修正（クイック） 変更ファイル ≤ 2 / migration なし / UI なし / API 契約なし / DB スキーマなし 20 分
config-with-review-and-verify 設定・ドキュメント編集 アプリコードを変えない docs・skills・rules・hooks・agents 変更 15 分
refactor-with-safety-guarantee リファクタ（挙動保証） 挙動を変えない lint・deps 更新・純粋リファクタ 45 分
incident-with-emergency-path 本番障害復旧 P0 障害の緊急復旧（本番ダウン・データ消失等） 最速

§5起動フローと進捗管理

起動直後の固定手順

# 1. プリフライト
references/module-preflight-check.md を Read → go/no-go 判定

# 2. Phase 1: ルート確定
references/phase-1-route-determination.md を Read
→ 「ルート判定: 本タスクは [ルート名]」とユーザーに提示

# 3. TaskCreate 登録（これ以前の実装ファイル編集は禁止）
該当ルートの全 Phase を TaskCreate で登録

# 4. Phase ファイルを着手時に Read して順次実行
Phase 2 完了後 → ScheduleWakeup(600s) で定期ヘルスチェック開始
Phase 10 完了後 → ヘルスチェックループ停止
TaskCreate 前の実装ファイル編集禁止
全 Step の TaskCreate 登録が完了するまで実装ファイルへの Write/Edit を開始してはならない。enforce-flow-feature-taskcreate.sh を設定しているプロジェクトでは hook が機械的にブロックする。

§6プリフライトチェック

Phase 1 より前に references/module-preflight-check.md の手順が毎回実行される。go を返したらフローを継続する。

状況判定対処
flow-values.yml が存在しない no-go Skill(creating-new-project) でセットアップを案内
flow-values.yml の YAML パースエラー no-go 構文エラーをユーザーに報告
CRITICAL ツールに最終 FAIL あり no-go 該当ツールのインストール方法を案内（自動修復を試みてから判定）
OPTIONAL ツールに FAIL あり WARN → go 警告のみ。フローは継続する
全 CRITICAL ツール PASS go 次ステップへ進む

CRITICAL ツール: Node.js（v18+）、npm、Python（3.10+）、git、gh CLI（認証済み）。OPTIONAL ツール: textlint 設定、lychee、gitleaks、Playwright MCP。詳細は references/module-preflight-check.md を参照。
incident-with-emergency-path では最小モード（CRITICAL のみ確認、その他の Step をスキップ）で実行する。

§7ルート昇格・停止点・中断

ルート昇格

以下の条件を満たした場合、即座に feature-with-full-planning に昇格し Phase 1 から再開する。昇格は 1 回限り。昇格後は元のルートに戻らない。

元ルート昇格トリガーチェックタイミング
feature-with-quick-delivery classify 条件の違反（migration 追加・UI 変更・API 契約変更・DB スキーマ変更） Phase 6 各 commit 直前
refactor-with-safety-guarantee 挙動変更の発見 Phase 7（完了チェック）の commit 直前

停止点（ユーザー承認待ち）

フローを止めてユーザー承認を待つ箇所は 2 か所のみ。それ以外の Phase ではルート判定・計画策定の宣言のみ行い、待機せず次 Phase に進む。

ルート停止箇所承認内容
feature-with-full-planningPhase 4（1 回のみ）仕様確認 + 画面 UI モック承認
feature-with-quick-deliveryなし（全自走）—
config-with-review-and-verifyなし（全自走）—
refactor-with-safety-guaranteeなし（全自走）—
incident-with-emergency-pathI4（本番操作承認）不可逆操作のため例外的に停止

中断手順

「やめたい」「中断して」と言われた場合、AskUserQuestion で変更破棄を確認してからクリーンアップを実行する。

進行状況クリーンアップ対象
Phase 1 までなし。即終了
Phase 2 以降worktree 削除 + ブランチ削除 + ポート kill
Phase 6 以降上記 + 未 push コミットは worktree 内に残したまま削除（復元不可を警告）
Phase 9 以降上記 + PR をクローズ

§8hook 連携（FLOW-GATE）

~/.claude/rules/scoped/dev-flow/gate/rule.md で規定される FLOW-GATE hook が、 前提 Phase 未完了の状態で ~/Projects/ 配下のコードファイルへの書き込みをブロックする。サブエージェントも対象。

hookタイミングタグ挙動
check-dev-flow-phase-gate.sh PreToolUse(Write|Edit|MultiEdit|NotebookEdit) [DEV-FLOW-PHASE-GATE-BLOCK] 前提 Phase 未完了なら exit 2 でブロック。.flow-progress.json の直接編集もブロック
check-worktree-required.sh PreToolUse(Write|Edit|MultiEdit|NotebookEdit) [WORKTREE-REQUIRED] メインツリーでの実装ファイル編集を decision:block でブロック

ルート別コード書き込み前提条件

ルートphases_completed に必要な Phase
feature-with-full-planning1, 2, 3, 4, 5
feature-with-quick-delivery1, 2, 5
refactor-with-safety-guarantee1, 2, 5
config-with-review-and-verify通過（コードゲート不要）
incident-with-emergency-path通過（緊急復旧はゲートで止めない）
[DEV-FLOW-PHASE-GATE-BLOCK] 受信時の対処
フロー未設定の場合 → Skill(orchestrating-dev-flow) を起動する。
前提 Phase 未完了の場合 → block メッセージの不足 Phase を順番に実行してから書き込みを再試行する。
.flow-progress.json の更新は update-flow-status.sh 経由のみ許可。
