# 概要タブの記載ルール

orchestration.js の概要タブ（section: "overview" / "routes"）に表示されるコンテンツの記載ルール。

## 対比カード（n: 1-2）

### セクションタイトル
- 「チャット実装依頼との違い」とする
- 「通常チャットとの違い」は不可（チャット自体ではなく、チャットでの実装依頼が比較対象）

### カードタイトル
- 何を対比しているか明確にする
- 良い例: 「× 通常チャットでの実装依頼」「✓ 実装フローでの実装依頼」
- 悪い例: 「通常のチャット」「実装フロー」（対比対象が不明確）

### 記号
- data（orchestration.js）の featureTitle に × / ✓ を 1 回だけ付ける
- JS（flow-detail.js）側で二重付与しない

### 内容
- 「→」で操作の流れを箇条書き風に繋ぐ
- 通常チャット側: 中立的な事実を書く。ネガティブすぎる表現（「品質は運任せ」等）は避ける
- 実装フロー側: 具体的な流れを書く

## 特徴カード（n: 3-5）

### タイトル
- 「特徴 N: 結論。補足説明」の形式
- 結論は平易な日本語で。技術用語をそのまま使わない

### detail
- 1 文ずつ「。」で区切り、箇条書きとして表示される
- 専門用語には括弧で補足を入れる（例: worktree（独立した作業ディレクトリ））
- 保証のない具体的な数値（時間・パーセント等）を書かない
- 「N 段階のレビュー」等を書く場合は内訳を明記する
- ステータスバーの例は detail にインラインで書かない（専用ブロックを使う）

### 禁止表現
- 「TDD 垂直スライス」→「テストを先に書いてから実装する TDD サイクル」
- 「20 分のクイック〜60 分のフル計画」→ 時間表記は削除
- 「品質は運任せ」→ 中立的な表現に
- 「PRD」単独使用 → PRD は廃止。「説明用 YAML（core.yaml）」または「仕様（説明用 HTML）」を使う
- 「決定木」→「条件分岐」（prh.yml で禁止登録済み）
- 「画面モック」単独使用 →「画面 UI モック」（UI を付けて区別する）
- 「説明ページ」「説明 HTML」→「説明用 HTML」で統一
- 「ADR」単独使用 → 使わない。「アーキテクチャ決定記録」で表記する
- 「flow-values.yml」単独使用 →「プロジェクト固有設定ファイル（flow-values.yml）」（初出時）または「プロジェクト固有設定ファイル」（2 回目以降）で表記する
- 「ASCII art」を画面 UI モックの入力として参照しない → 既存コード（.tsx / .vue）の Read + 画面設計書（コーディングレベルの詳細）を入力とする
- 内部ファイル名（.flow-progress.json / flow-status.json / core.yaml / view.yaml 等）を説明なしで使わない → 「セッション跨ぎ進捗ファイル」「ステータスライン進捗ファイル」「説明用 YAML」等の人間向け表現を使い、初出時のみ括弧でファイル名を添える
- 存在しない機能・スキル・Hook・API を記述しない。実装されていない機能を「できる」「自動で行う」と書くのは虚偽。未実装の場合は記載しないか「未実装」と明記する
- detail 内で手順を実行する記述は日本語の説明にする（例: 「references/module-hearing-requirements.md の手順を実行する」→「要件ヒアリングの手順を実行する」）。refs の pill text には手順名をそのまま残す
- 「セキュリティレビュー」等、実態と異なる名称でレビューゲートを呼ばない。3 段階レビューは「仕様適合・実装品質・プッシュ前」
- 「コンフリクト自動解消」等、手動対応が必要な処理を「自動」と書かない
- 画面設計書には 7 項目（レイアウト・コンポーネント一覧・状態定義・データバインディング・ユーザー操作・エラー表示・アニメーション）が必要

## 事前準備タブ

- プリフライトチェック（references/module-preflight-check.md）の動作解説（何をチェックし、go/no-go で何が起きるか）
- creating-new-project の動作解説（初回時に何が生成されるか）
- 準備される成果物一覧（flow-values.yml / layers.yml / project-portal 等の役割）

## サンプル画像

- 特徴 1 カードの直後に画面 UI モックと説明用 HTML のサンプル画像を表示する
- orchestration.js の sampleImages フィールドで src / alt / caption を定義
- 画像は ai-management-portal/assets/ に配置
- **サンプルはわかりやすいダミー画面を新規作成する**。実際の mock-archive のスクリーンショットは文字が小さく伝わらない
- 画面 UI モック: Before/After 形式でアプリ画面の変化が一目でわかるもの
- 説明用 HTML: 仕様書ページ風の構成（背景・提案・変更ファイル・受け入れ条件）が見えるもの

## ステータスバー表示例ブロック

- 特徴 2 カードの内部（箇条書きの下）に配置
- ターミナル風の CSS で表現（ダーク背景 + シアン色）
- 例は新 Phase 構成に合わせる（Phase 1「調査 + ルート判定」等）
- 各パーツ（⚙ / Phase 番号 / Phase 名 / 進捗バー / カウント / Step 名）を span で分離

## ルート一覧テーブル（section: "routes"）

### 列構成
- 識別子 / 日本語名 / 用途 / 承認
- **目安時間列は設けない**（保証がないため）

### duration フィールド
- orchestration.js に duration フィールドは残してよいが、テーブルには表示しない
- incident ルートの duration: "最速" は許容（相対表現のため）

## flowSummary の目的バッジ

- flowSummary の「」で囲まれた目的は、Phase 色のバッジとして視覚的に表示される
- 目的がないと手順の羅列に見えるため、必ず先頭に付ける
- バッジの CSS: `.odf-phase-goal`（Phase 色背景 + 白文字）

## 完了前の確認義務

- UI に影響する変更（orchestration.js / flow-detail.js）を push した場合、**ブラウザで実際に表示を確認してからタスクを完了にする**
- 仕組みだけ実装して実体ファイル（画像等）が未配置の場合は完了にしない
- サブエージェントの報告を鵜呑みにせず、自分で確認する

## ターゲット層

- Claude Code ユーザー（開発者）
- 実装フローを初めて見る人が、概要タブだけで「何ができるか」「どう違うか」を理解できる粒度で書く
- Phase / Step / worktree 等の用語は初出時に補足する
