user 直近指示「mock がクリックして表示・skill で統一レイアウト・複数観点で test」への 本質対応。Layer 1 skill Layer 2 PostToolUse Layer 3 PreApproval の 3 層で mock 承認サイクル全停止 blocker を解消する。

本セッションで露呈した障害連鎖:

  • 各 planner が HTML mock を bespoke レイアウト で作成（統一テンプレなし）
  • portal click → mock 表示 の動作確認なし → ユーザーが mock を物理的に開けない
  • mock_awaiting 遷移時に上記をブロックする hook 不在
  • 結果: state.json 内 mock_awaiting 5 件（#1556 / #1561 / #1549 / #1560 / #1533）が滞留・承認進まず
関連既存資産との重複回避
  • #1561: check-mock-skill-invoked.sh は「skill 起動強制」の 1 本のみ。本 issue は skill 本体 + 品質検証を担当（重複しない）
  • #1587: mock-viewer は 表示側（archive 配信）。本 issue は 作成 + 検証側（責務分担明示）
  • preparing-manual-mockup: マニュアル画像加工専用。汎用 HTML mock の正規テンプレが本 issue で新設

6. リスク・緩和策

リスク影響緩和策
skill 未起動で hook が誤検知 既存 60+ mock が一斉 block marker 不在は警告のみ・新規作成のみ block の段階導入
layout 厳格すぎて planner 創意阻害 多様な mock が描けない 必須 section は 7 種に絞り、その下の自由度は確保
3 連発火 auto-release で永続バイパス 機構が機能しない 1 セッション内のみ有効・新セッションで再評価
reachability check のネットワーク負荷 UI 遅延 HEAD only + Promise.all 並列 + 30 秒 cache
archive-integrity が #1587 実装に依存 本 issue 単独で完結しない #1587 完了後に Layer 3 観点 4 を有効化する段階導入

実害検証手順（5 ケース）

  1. skill 不使用 → block: Write を skill 経由せず実行 → [MOCK-QUALITY-BLOCK] 確認
  2. layout 違反 → block: 必須 section を欠いた mock を Write → block 確認
  3. 全 mock 200 → pass: 既存 5 件を template 準拠で改修 → /api/approve POST が pass
  4. archive 不整合 → block: ~/.claude/mock-archive/issue-<N>/ を削除 → block
  5. 3 連発火 auto-release: 同一 task で 3 回 block → 4 回目は warning に降格
