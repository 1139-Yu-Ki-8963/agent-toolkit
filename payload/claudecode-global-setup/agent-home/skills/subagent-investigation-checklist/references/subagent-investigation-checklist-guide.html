調査チェックリスト作成スキル
subagent-investigation-checklist

サブエージェントへの調査・分析・レビュー委任前に、漏れのない調査チェックリストを作成するスキル。調査指示の品質を構造的に保証し、調査エージェントの報告品質とレビュアーの検証基準を一体で担保する。

こんな人向け
調査・分析・レビューをサブエージェントに委任するメインエージェント ／ 調査結果の事実性・証拠充足度を担保したい開発者 ／ サブエージェント委任のたびに「何を調べるか」を一から考えたくない人
こんな場面で使う
プロジェクト構成レビュー・コード調査・仕様調査・ログ分析など事実確認を伴う調査をサブエージェントに委任する前 ／ check-evidence-checklist.sh フックが [CHECKLIST-MISSING] を注入した時 ／ 調査結果をレビュアーに渡す前に検証基準を明確にしたい時
対応 OSmacOS
検証状況managing-agent-configs の create→review→test 連鎖で管理（作成・変更時に静的レビューと発火検証を実行する運用。日付付きの実行記録は次回 test 実行時から本欄に記載）
使用ツールClaude Code 標準ツール（組み込み・追加インストール不要）: Read（ファイル読み取り）、Write（ファイル新規作成）、Bash（シェルコマンド実行）

§1概要

調査指示の品質を構造的に保証するスキル。「何を調べるか」を事前に網羅することで、調査エージェントの報告品質を担保し、レビュアーの検証基準を明確にする。

サブエージェントに調査を委任するとき、指示が曖昧だと以下の問題が発生する。

  • 確認せずに推測を事実として断言する（「〜のはず」が finding になる）
  • 数量の主張（「N 件」）の裏取りを省略する
  • 存在しないファイル・ディレクトリに言及する
  • 外部ツールの配置要件を公式ドキュメントなしで断定する

本スキルはチェックリストを事前に作成することで、これらの誤りを構造的に防ぐ。

位置づけ

担当責務
本スキル
subagent-investigation-checklist		 タスク種別を判定し、必須共通項目・種別固有項目・タスク固有項目からなるチェックリストを生成して worktree 内（または TMPDIR）に配置する。調査エージェントの prompt に埋め込む形式も提供する
調査エージェント（opus） チェックリストの全項目を実行し、各項目に証拠（コマンド出力）を添付して報告する
レビュアー チェックリストと調査結果の両方を照合し、証拠なし断言・推測の事実化・未確認項目の有無を検証して PASS / FAIL を判定する

§2出力

本スキルの実行が完了すると、以下の成果物が確立される。

調査チェックリスト（.investigation-checklist.md）
必須共通項目・種別固有項目・タスク固有項目の 3 層を含む Markdown ファイル。worktree 内にいれば worktree ルート直下、worktree 外（メインツリー）なら ${TMPDIR:-/tmp}/claude-hooks/${session}/ 配下に配置される。commit はせず、worktree 削除やセッション終了時に自動消滅する一時成果物。
調査エージェント向け埋め込みテキスト
チェックリスト全文を「全項目を実行し証拠を添付すること」という遵守指示とともに調査エージェントの prompt に埋め込んだテキスト。調査エージェントが何を確認し、何を証拠として提示すべきかを一意に規定する。
レビュアー向け引き渡しテキスト
チェックリスト全文と調査エージェントの報告全文を並べ、証拠充足度・推測と事実の分離を検証させるためのレビュー指示付きテキスト。レビュアーの PASS / FAIL 判定基準となる。

§3用語

調査チェックリスト
調査委任前に生成する Markdown ファイル。必須共通項目・種別固有項目・タスク固有項目の 3 層からなり、調査エージェントが全項目を実行して証拠を添付する義務を負う。
finding
調査エージェントが報告する個々の発見事項。証拠（コマンド出力・URL 引用）なしのものは finding として認めない。
タスク種別
委任タスクの性質に応じた分類（プロジェクト構成レビュー / コード調査 / 仕様調査 / ログ・データ分析 / 混合）。種別ごとに異なる固有チェック項目が追加される。
証拠
finding を裏付けるコマンド出力・公式ドキュメントの URL と該当セクション引用。推測・記憶・前回確認は証拠として認めない。
.investigation-checklist.md
本スキルが出力するチェックリストファイル。worktree 内のルートに配置し、セッション終了や worktree 削除時に自動消滅する（永続 commit はしない）。

§4トリガーとスキップ条件

発火するタイミング（TRIGGER）

SKILL.md の記述: 調査・分析・レビュー・構成確認など事実確認を伴う委任前

状況具体例
プロジェクト構成レビュー委任前ディレクトリ構成・設定ファイル配置・依存関係の評価を opus に委任する前
コード調査委任前影響範囲分析・呼び出し元追跡・dead code 検出を worker-sonnet に委任する前
仕様調査委任前外部ツールの公式仕様・API ドキュメントの確認を researcher に委任する前
ログ・データ分析委任前ログ解析・transcript 読解・実態把握を opus に委任する前
フック検出時check-evidence-checklist.sh が [CHECKLIST-MISSING] を注入した時

スキップ条件（SKIP）

SKILL.md の記述: 機械的作業（一括リネーム等）・worker-haiku/routine-worker への定型委任時

状況理由
worker-haiku への機械的作業委任変数名の一括リネーム・テスト実行など、判断を伴わない定型作業はチェックリスト不要
routine-worker への定型実行定型ルーティンはチェックリストで保護する「事実確認」を行わない
単一ファイルの grep / Read1 コマンドで完結する簡易調査はオーバーヘッドが不要
prompt に [CHECKLIST-EXEMPT] を明示した場合呼び出し元が意図的に除外を宣言している
注意: スキップ判断はメインエージェントが行う。迷った場合は本スキルを起動することを推奨する。チェックリスト生成のコストは、調査品質の担保コストより常に低い。

§5タスク種別の分類

Step 1 でタスクを以下の種別に分類する。種別は単一でも混合でも構わない。

種別例固有項目の特徴
プロジェクト構成レビュー ディレクトリ構成・設定ファイル配置・依存関係の評価 Git 追跡状態の確認・外部ツールの公式配置要件の裏取り
コード調査 影響範囲分析・呼び出し元追跡・dead code 検出 grep による全参照箇所の網羅・import パスの実解決確認
仕様調査 外部ツールの公式仕様・API ドキュメントの確認 公式ドキュメント URL の記録・該当セクションの引用義務
ログ・データ分析 ログ解析・transcript 読解・実態把握 行数・サイズの実測・サンプル行の実際の確認・集計コマンド出力の添付
混合 上記複数を含むタスク 該当する全種別の固有項目を合算して追加する

§6動作フロー（Step 1〜7）

── 全体フロー ───────────────────────────────────────────────

  Step 1  タスク種別の判定
          委任タスクを §4 の 5 種別に分類する

  Step 2  必須共通項目の生成（全種別共通）
          ・全ファイル・ディレクトリの実在確認（ls / find で検証）
          ・数量の主張の裏取りコマンド実行と出力添付
          ・推測は「未確認」と明記し事実として断言しない
          ・各 finding に確認コマンドと出力を証拠として添付

  Step 3  種別固有項目の追加（§6 参照）
          タスク種別に応じた追加チェック項目をリストに加える

  Step 4  タスク固有項目の追加
          ユーザーの依頼内容から動的に追加項目を生成する

  Step 5  チェックリストの配置（§7 参照）
          worktree 内または TMPDIR に .investigation-checklist.md を書き出す

  Step 6  調査エージェントへの埋め込み
          チェックリスト全文を調査エージェントの prompt に埋め込む

  Step 7  レビュアーへの引き渡し
          調査完了後、チェックリストと調査結果の両方をレビュアーに渡す

── 完了条件 ──────────────────────────────────────────────────

  ・.investigation-checklist.md が worktree 内（または TMPDIR）に配置済み
  ・チェックリストに必須共通項目 + 種別固有項目 + タスク固有項目が含まれている

§7チェックリストの構造と必須項目

必須共通項目（全種別）

## 調査チェックリスト

### 必須確認項目（全種別共通）
- [ ] 言及する全ファイル・ディレクトリの実在確認（ls / find で検証）
- [ ] 数量の主張（「N 件」「N 個」）の裏取りコマンド実行と出力添付
- [ ] 推測は「未確認」と明記し、事実として断言しない
- [ ] 各 finding に確認コマンドと出力を証拠として添付

### 出力要件
- 証拠なしの finding は finding として認めない
- 「〜の仕様上必須」等の外部仕様主張には公式ドキュメントの URL を添付
- 未確認の項目は「未確認」セクションに分離する

種別固有項目

種別固有チェック項目
プロジェクト構成レビュー git ls-files <dir> | wc -l で各ディレクトリの Git 追跡状態を確認
ls -la でルート直下ファイルの正確な一覧を取得
設定ファイルの所属先確認（中身を読んで専用先を判定）
外部ツール（Vercel, Render 等）の設定ファイル配置要件を公式ドキュメントで確認
コード調査 対象シンボルの全参照箇所を grep で網羅（--include で拡張子を限定）
参照元ファイルの実在確認
import / require の実パス解決確認
仕様調査 公式ドキュメントの URL を記録
ドキュメントの該当セクションを引用
「〜はルート固定」等の配置要件は公式ソースで裏取り
ログ・データ分析 ログファイルの行数・サイズを確認（wc -l, du -sh）
サンプル行を実際に読んで内容を確認
集計値は集計コマンドの出力を添付

タスク固有項目

ユーザーの依頼内容を読み、上記の共通・種別固有では網羅できない調査項目を動的に追加する。

### タスク固有項目
- [ ] （タスク内容から動的に生成）
出力品質の原則: 証拠なしの finding は finding として認めない。 推測・記憶・「前回確認した」は証拠として認めない。 未確認の項目は「未確認」セクションに分離する。 外部仕様の主張には必ず公式ドキュメントの URL を添付する。

§8チェックリストの配置先

Step 5 で生成したチェックリストを以下の規則で配置する。

状況配置先
worktree 内にいる場合 .investigation-checklist.md（worktree ルート直下）
worktree 外の場合（メインツリー） ${TMPDIR:-/tmp}/claude-hooks/${session}/investigation-checklist.md 
── 配置コマンド例 ───────────────────────────────────────────

  # worktree 内にいる場合
  echo "<チェックリスト内容>" > .investigation-checklist.md

  # worktree 外の場合（メインツリー）
  echo "<チェックリスト内容>" > ${TMPDIR:-/tmp}/claude-hooks/${session}/investigation-checklist.md
ライフサイクル: .investigation-checklist.md は commit しない。worktree 削除時に自動消滅し、セッション終了時に cleanup-session-markers.sh が TMPDIR 内のファイルを清掃する。 永続保存が必要な調査結果は別途 docs/ 等の正規パスに書き出す。

§9調査・レビューエージェントへの引き渡し

調査エージェント（Step 6）への prompt 埋め込み形式

## 調査チェックリスト（必須遵守）

<チェックリスト全文>

上記チェックリストの全項目を実行し、各項目に証拠（コマンド出力）を添付して報告すること。
未実行の項目は「未確認」と明記すること。

レビュアー（Step 7）への引き渡し形式

## レビュー対象

### 調査チェックリスト
<チェックリスト全文>

### 調査結果
<調査エージェントの報告全文>

## レビュー指示
1. チェックリストの各項目が実行されているか確認
2. 各 finding に証拠（コマンド出力）が添付されているか確認
3. 証拠なしの断言がないか確認
4. 推測が事実として記述されていないか確認
5. PASS / FAIL を判定し、FAIL の場合は不足項目を列挙

FAIL 時のパイプライン

レビュアーが FAIL を返した場合、不足項目を補完した上で調査エージェントを再起動する。最大 2 回まで再試行する（subagent-delegation-rules の調査チェックリストパイプライン仕様）。

判定次のアクション
PASS 調査結果をユーザーに報告する（事実と推測を分離して提示）
FAIL 不足項目を確認し、調査エージェントを再起動する（最大 2 回）
2 回 FAIL 残った不足項目をユーザーに報告して中断する
