# 調査チェックリスト作成

サブエージェントへの調査委任前に、漏れのないチェックリストを作成する。

## 目的

調査指示の品質を構造的に保証する。「何を調べるか」を事前に網羅することで、調査エージェントの報告品質を担保し、レビュアーの検証基準を明確にする。

## 手順

### Step 1: タスク種別の判定

委任するタスクを以下のいずれかに分類する:

| 種別 | 例 |
|---|---|
| プロジェクト構成レビュー | ディレクトリ構成・設定ファイル配置・依存関係の評価 |
| コード調査 | 影響範囲分析・呼び出し元追跡・dead code 検出 |
| 仕様調査 | 外部ツールの公式仕様・API ドキュメントの確認 |
| ログ・データ分析 | ログ解析・transcript 読解・実態把握 |
| 混合 | 上記の複数を含む |

### Step 2: 必須項目の生成

全種別共通の必須項目:

```markdown
## 調査チェックリスト

### 必須確認項目（全種別共通）
- [ ] 言及する全ファイル・ディレクトリの実在確認（ls / find で検証）
- [ ] 数量の主張（「N 件」「N 個」）の裏取りコマンド実行と出力添付
- [ ] 推測は「未確認」と明記し、事実として断言しない
- [ ] 各 finding に確認コマンドと出力を証拠として添付

### 出力要件
- 証拠なしの finding は finding として認めない
- 「〜の仕様上必須」等の外部仕様主張には公式ドキュメントの URL を添付
- 未確認の項目は「未確認」セクションに分離する
```

### Step 3: 種別固有項目の追加

#### プロジェクト構成レビュー
```markdown
### 種別固有: プロジェクト構成レビュー
- [ ] 各ディレクトリの Git 追跡状態（git ls-files <dir> | wc -l）
- [ ] ルート直下ファイルの正確な一覧（ls -la でパス確認）
- [ ] 設定ファイルの所属先確認（中身を読んで、どのサブプロジェクト専用か判定）
- [ ] 外部ツール（Vercel, Render 等）の設定ファイル配置要件を公式ドキュメントで確認
```

#### コード調査
```markdown
### 種別固有: コード調査
- [ ] 対象シンボルの全参照箇所を grep で網羅（--include で拡張子を限定）
- [ ] 参照元ファイルの実在確認
- [ ] import / require の実パス解決確認
```

#### 仕様調査
```markdown
### 種別固有: 仕様調査
- [ ] 公式ドキュメントの URL を記録
- [ ] ドキュメントの該当セクションを引用
- [ ] 「〜はルート固定」等の配置要件は公式ソースで裏取り
```

#### ログ・データ分析
```markdown
### 種別固有: ログ・データ分析
- [ ] ログファイルの行数・サイズを確認（wc -l, du -sh）
- [ ] サンプル行を実際に読んで内容を確認
- [ ] 集計値は集計コマンドの出力を添付
```

### Step 4: タスク固有項目の追加

ユーザーの依頼内容に応じて、追加の調査項目を生成する。

```markdown
### タスク固有項目
- [ ] （タスク内容から動的に生成）
```

### Step 5: チェックリストの配置

生成したチェックリストを worktree 内に配置する:

```bash
# worktree 内にいる場合
echo "<チェックリスト内容>" > .investigation-checklist.md

# worktree 外の場合（メインツリー）
echo "<チェックリスト内容>" > ${TMPDIR:-/tmp}/claude-hooks/${session}/investigation-checklist.md
```

### Step 6: 調査エージェントへの埋め込み

チェックリストの全文を調査エージェントの prompt に埋め込む:

```
## 調査チェックリスト（必須遵守）

<チェックリスト全文>

上記チェックリストの全項目を実行し、各項目に証拠（コマンド出力）を添付して報告すること。
未実行の項目は「未確認」と明記すること。
```

### Step 7: レビュアーへの引き渡し

調査完了後、チェックリストと調査結果の両方をレビュアーに渡す:

```
## レビュー対象

### 調査チェックリスト
<チェックリスト全文>

### 調査結果
<調査エージェントの報告全文>

## レビュー指示
1. チェックリストの各項目が実行されているか確認
2. 各 finding に証拠（コマンド出力）が添付されているか確認
3. 証拠なしの断言がないか確認
4. 推測が事実として記述されていないか確認
5. PASS / FAIL を判定し、FAIL の場合は不足項目を列挙
```

## 予想を裏切る挙動

- チェックリストの配置先は worktree 内とメインツリーで異なる: worktree 内なら `.investigation-checklist.md`、メインツリーなら TMPDIR 配下に置く

## 完了条件

- `.investigation-checklist.md` が worktree 内（または TMPDIR）に配置されている
- チェックリストに必須共通項目 + 種別固有項目 + タスク固有項目が含まれている

## 完了報告

`managing-agent-configs/references/skills/completion-report-format.md` の共通骨格（調査報告型）に従う。
固有の検証行: レビュー PASS / FAIL 判定
