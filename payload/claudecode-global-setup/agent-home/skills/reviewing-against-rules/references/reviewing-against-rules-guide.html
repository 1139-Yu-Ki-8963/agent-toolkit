rules 照合レビューの単一入口
reviewing-against-rules

対象ファイルから適用ルールを機械的に解決し、ドメイン別の専門家サブエージェントに照合を委任するスキル。

こんな人向け
成果物・差分のレビューを依頼された、または他スキルのレビュー工程（PR レビューの中の 1 段階・開発フローのレビュー工程）から呼び出された Claude。レビュー観点をその場で自作せず、既存の規約（ルール）に基づいて判定したい場合に使う。
こんな場面で使う
「レビューして」と成果物・差分を指定されたとき、他スキルのレビュー工程（PR レビューの一段階・開発フローのレビューゲート）から呼ばれたとき、HTML・コードの完成判定時。設定資産（skills / rules / hooks / routines / subagents）のレビューは対象外（managing-agent-configs が担当）。調査報告の事実性検証は report-reviewer への直接委任。公開可否チェックは reviewing-public-readiness。

§1概要

レビュー観点の正本は rules（~/.claude/rules/scoped/ 配下の各 rule.md）にある。reviewing-against-rules は「対象ファイル → 適用ルールの解決 → ドメイン別専門家への委任 → 判定回収」という一連の流れを定型化し、スキルやセッションごとに観点の参照がばらつく事故（参照切れ・観点の省略・呼び出し元自身によるレビュー）を防ぐ。

本スキル自体は判定を行わない。判定は必ず document-reviewer / code-reviewer 等のドメイン別専門家に委ね、本スキルは「どのルールを、誰に、どう照合させるか」の配線だけを担当する。

§2出力

本スキルの実行が完了すると、以下が確立される。

適用ルールの解決結果
レビュー対象ファイルそれぞれについて、該当する rule.md の実パス一覧、またはレビュー対象外（適用ルールなし）であることが明示された状態。
専門家による証拠付き判定
適用ルールがあるファイルについて、ドメイン別専門家（document-reviewer または code-reviewer）が rule.md の基準ごとに PASS / FAIL（または対象外）と証拠を返した基準別判定表、および総合判定（合格 / 不合格・承認可 / 修正必須 / マージ不可）。
収束済みの修正ループ
不合格だった場合に worker-sonnet へ委任した修正と、その修正を踏まえた再判定の結果。合格またはユーザーへの差し戻しのいずれかで終端している状態。

§3原則

原則内容防ぐ事故
観点表を直接読まない レビュー観点は専門家サブエージェントが rule.md から読む。呼び出し側（メインセッション・他スキル）が観点を要約してプロンプトに書き写すことを禁止する 要約の過程で観点が欠落・乖離する
メインの自己レビュー禁止 判定は必ず専門家サブエージェントが行う。生成役と検証役を分離する 自分で書いたものを自分で甘く採点する
適用 rule は機械的に解決する scoped rule の paths: frontmatter が適用対象の宣言。手動でルールを選ばない 該当するはずのルールをうっかり見落とす、または恣意的に選ぶ

§4手順（5 ステップ）

# Step 1: 対象の確定
#   レビュー対象ファイル一覧を確定する
#   PR・ブランチ差分なら git diff --name-only で取得する

# Step 2: 適用 rule と担当専門家の解決
scripts/resolve-applicable-rules.sh <file...>
#   グローバル scoped rules と、カレントリポジトリの受け口
#   （<repo>/.claude/rules/scoped/）の両方から、paths にマッチする
#   <file, rule.md実パス, specialist> の組を全件得る（3列TSV）
#   専門家の選択はファイル拡張子から推測しない。
#   各 rule.md の frontmatter が宣言する specialist: フィールドを使う

# Step 3: specialist ごとにグルーピング
#   specialist 列の値ごとに <file, rule> の組をグルーピングする
#   specialist が (unassigned) → 専門家未整備として警告扱い
#   rule が (none)            → レビュー対象外と報告して終了
#   設定資産（SKILL.md・rule.md・hook 等） → 本スキル対象外。managing-agent-configs へ
#   複数 specialist が必要な場合 → 並列起動（1 メッセージで複数 Agent 呼び出し）

# Step 4: 委任
#   各専門家に次を埋め込んで委任する（文脈依存語「直前の」「例の」は禁止）
#     - 対象ファイルの実パス一覧
#     - Step 2 で解決した rule.md の実パス一覧
#     - 「## 調査チェックリスト」見出し配下に
#       「rule の全基準を確認し、基準ごとに PASS / FAIL / 対象外 + 証拠を返す」旨
#     - 出力形式（専門家定義の出力形式に従う）と行数上限

# Step 5: 判定回収と後続
#   不合格・マージ不可 → worker-sonnet へ修正委任（専門家は修正しない）
#                        → 修正後 Step 4 を再実行（最大 5 回、同一指摘の
#                          2 回連続再発でユーザーへ差し戻し）
#   合格             → 判定・照合 rule 一覧・指摘の要約をユーザー（または
#                        呼び出し元スキル）へ返す

§5rule と specialist の対応

rulepaths（適用対象）specialist
review-checklist/code *.ts *.tsx *.js *.jsx *.mjs *.py code-reviewer
review-checklist/document/html-output *.html document-reviewer
review-checklist/code/ui *.tsx *.jsx *.vue *.html *.css *.scss document-reviewer
設定資産（SKILL.md・rule.md・hook 等） - 本スキル対象外。managing-agent-configs に委譲
適用 rule なし - レビュー対象外と報告して終了（観点なしの裁量レビューをしない）
1 ファイル複数 specialist：icon-usage は html-output と同じ specialist（document-reviewer）だが適用範囲が html 以外にも及ぶため、.tsx は code-reviewer（review-checklist/code）と document-reviewer（icon-usage）の両方が該当しうる。specialist ごとに並列起動する（1 メッセージで複数 Agent 呼び出し。順次実行は無駄な待ち時間を生む）。rule 数・specialist 実在は scripts/audit-review-coverage.sh が機械的に数える。

§6サブエージェント委任仕様

呼び出し箇所subagent_type期待返却値
手順 4（HTML・文書照合） document-reviewer 基準別判定表 + 総合判定（合格 / 不合格）
手順 4（コード照合） code-reviewer 基準別判定表 + 総合判定（承認可 / 修正必須 / マージ不可）
手順 5（不合格時の修正） worker-sonnet 修正ファイル一覧と修正内容の要約

専門家に渡す rule.md の実パスは、必ず Step 2 の resolve-applicable-rules.sh の解決結果をそのまま使う。呼び出し側が観点を要約して渡すことは §3 の原則で禁止されている。

§7完了条件

適用ルールの解決が全件完了している
対象ファイル全件について、適用 rule の解決結果（該当 rule またはレビュー対象外）が示されている。
専門家判定が回収済みである
適用 rule があるファイルは専門家の総合判定（証拠付き）が回収されている。
修正ループが終端している
不合格 → 修正 → 再判定のループが収束（合格）、またはユーザー差し戻しで終端している。

§8予想を裏切る挙動

専門家は観点を持たない：専門家の定義自体には観点が書かれていない。rule.md のパスを渡し忘れると「照合 rule なし」で判定不能になる。これは仕様であり、観点なしレビューを許さないための設計。
resolve-applicable-rules.sh は scoped rule のみ解決する：always rule（常時注入）は全作業で既に効いているため、レビュー委任時に改めて渡す必要はない。
新しいドメインの観点 rule を作った場合：例えば文章品質のような新しい観点 rule を作った場合、対応する専門家が未整備なら worker-sonnet で代替せず、専門家の新設を managing-agent-configs（種別 subagents）で行う。ドメイン単位 1 専門家の原則を守る（rule と 1 対 1 の専門家を増やさない）。

§10よくある誤用と正しい使い方

誤用正しい使い方
設定資産（SKILL.md・rule.md・hook）を本スキルでレビューしようとする managing-agent-configs の該当種別（skills / rules / hooks / routines / subagents）に委任する
調査報告の事実性（数値・再現性）を検証したい report-reviewer サブエージェントに直接委任する（本スキルはドメイン別専門家への配線が役割であり、事実性検証は担当しない）
公開前の機密混入・履歴漏洩チェックをしたい reviewing-public-readiness を使う（本スキルはコード・HTML の規約照合が対象であり、公開可否は別軸）
呼び出し側がレビュー観点を要約して専門家に渡す rule.md の実パスをそのまま渡す。要約は §3 の原則「観点表を直接読まない」に違反する
