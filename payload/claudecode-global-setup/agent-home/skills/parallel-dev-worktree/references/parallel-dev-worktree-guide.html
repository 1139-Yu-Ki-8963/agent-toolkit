並列開発ワークフロー（git worktree）
parallel-dev-worktree

git worktree で実装用ブランチを作成・管理・復旧する。orchestrating-dev-flow の Phase 2 から呼び出されるほか、単独でも発動する。

こんな人向け
実装・修正・issue 解決タスクを担うエージェント。main ブランチを汚さず複数作業を並走させたい場面、または worktree の復旧が必要な場面で参照する。

§1概要とライフサイクル

このスキルは git worktree を使い、実装用ブランチをメイン作業ツリーと分離して管理する。スキルの型は gateway で、リポジトリ非依存に動作する（メインツリーのパスもデフォルトブランチ名も動的に取得する）。

worktree のライフサイクル

必ず実施する Phase / 条件付き Phase

Phase必ず実施 / 条件付き条件
Phase 1必ず—
Phase 2条件付きプロジェクト側 CLAUDE.md に「コンフリクトしたファイル」見出しがある場合のみ。それ以外は OPEN PR 存在チェックのみ
Phase 3必ず—
Phase 4必ず—
Phase 5必ず（GitHub 以外は push まで）—
Phase 6必ず—
Phase 7必要時のみレビュー指摘の修正タスク発生時
適用方針: 「実装」「修正」「直して」「対応」「issue 解決」と指示された時は、規模を問わず必ず Phase 1 から実行する。SKILL.md の 1 行修正・typo fix 等の軽微変更も対象。例外はユーザーが明示的に「main で直接」「worktree 不要」と指定した時、既存 worktree 内での追加コミット時、読み取りのみの調査タスク、~/.claude/ または ~/agent-home/ のグローバル設定リポジトリ対象の時のみ。

§2出力

本スキルの実行が完了すると、以下の 2 つの状態が確立される。

push 済みブランチと PR（または push のみ）
実装内容が origin 上のブランチへ push 済みであり、GitHub リモートの場合はレビュー可能な状態の PR が作成されている。GitHub 以外のリモートでは push 完了までで停止し、リモートの web UI での PR 作成をユーザーに促す。ローカルブランチは残すのが既定であり、レビュー指摘が来た場合は Phase 7 で再 checkout して再利用できる。
後片付け済みのメイン作業ツリー
worktree 専用に確保したポート・コンテナ等の後始末を終えたうえで git worktree remove が実行され、メイン作業ツリーはデフォルトブランチに戻っている。worktree 自体は畳まれるが、push 済みのブランチと変更内容は失われず、並行作業の分離状態が正常に解消されている。

§3配置・命名規約

項目ルール
worktree パス ~/Projects/worktrees/<repo>/<branch-name>/
<repo> 階層でリポジトリ間のブランチ名衝突を防ぐ。スラッシュはディレクトリ階層として展開される。
ブランチ名 命名規約（rules: always/naming/commit-branch）の「ブランチ名」に従う。prefix: feature/ fix/ docs/ chore/ refactor/ release/ のいずれか + <slug>（ケバブケース・50 文字以内）。
WORKTREE_ROOT 上書き 環境変数 PARALLEL_DEV_WORKTREE_ROOT に設定すると、デフォルトの ~/Projects/worktrees/$REPO_NAME を上書きできる（detailed-procedures.md 参照）。
slot worktree（撤去済み） slot 運用（<repo>/.claude/worktrees/wtN + .slot-pool.json）は 2026-06-28 に撤去済み（歴史的記述）。全リポジトリの worktree は ~/Projects/worktrees/<repo>/<branch>/ の中央規約に一本化されている。
注意: worktree パスのスラッシュはディレクトリ階層として解釈される。例えばブランチ名が feature/profile-cache-headers の場合、worktree パスは ~/Projects/worktrees/<repo>/feature/profile-cache-headers/ となる。

§4共通変数の取得

各 Phase で使う変数は最初に決定する。以下は SKILL.md 記載の基本形と detailed-procedures.md 記載の完全版をまとめたもの。

# 基本形（SKILL.md）
REPO=$(git rev-parse --show-toplevel)
DEFAULT_BRANCH=$(git -C "$REPO" symbolic-ref refs/remotes/origin/HEAD --short 2>/dev/null | sed 's@^origin/@@')
[ -z "$DEFAULT_BRANCH" ] && DEFAULT_BRANCH=main
REPO_NAME=$(basename "$REPO")
WORKTREE_ROOT="$HOME/Projects/worktrees/$REPO_NAME"
HAS_GH=$(gh repo view --json url 2>/dev/null && echo yes || echo no)

# 完全版（detailed-procedures.md）: origin/HEAD 未設定時のフォールバック込み
REPO=$(git rev-parse --show-toplevel) || { echo "ここは git リポジトリではない"; exit 1; }

DEFAULT_BRANCH=$(git -C "$REPO" symbolic-ref refs/remotes/origin/HEAD --short 2>/dev/null | sed 's@^origin/@@')
if [ -z "$DEFAULT_BRANCH" ]; then
  git -C "$REPO" remote set-head origin --auto >/dev/null 2>&1
  DEFAULT_BRANCH=$(git -C "$REPO" symbolic-ref refs/remotes/origin/HEAD --short 2>/dev/null | sed 's@^origin/@@')
fi
[ -z "$DEFAULT_BRANCH" ] && DEFAULT_BRANCH=main

REPO_NAME=$(basename "$REPO")
WORKTREE_ROOT="${PARALLEL_DEV_WORKTREE_ROOT:-$HOME/Projects/worktrees/$REPO_NAME}"
mkdir -p "$WORKTREE_ROOT"

if gh repo view --json url >/dev/null 2>&1; then
  HAS_GH=yes
else
  HAS_GH=no
fi
注意: origin/HEAD が未設定の場合は git remote set-head origin --auto で自動設定を試みる。それでも取れない場合は main をフォールバックとして使い、ユーザーに通知する（detailed-procedures.md 参照）。

§5通常フロー（Phase 1–7）

Phase 1: 着手前チェック（必ず実施）

メイン作業ツリーで以下を並列実行する。

# 1) 作業ツリーの汚れ確認（汚れがあればユーザー確認。勝手に stash しない）
git -C "$REPO" status --short

# 2) 現在ブランチ確認（$DEFAULT_BRANCH 以外にいる場合は理由を確認）
git -C "$REPO" branch --show-current

# 3) OPEN PR 一覧取得（GitHub のみ）
gh pr list --state open --json number,title,headRefName,files --limit 30

# 4) origin の最新化（ローカルのデフォルトブランチには checkout・pull しない）
git -C "$REPO" fetch origin "$DEFAULT_BRANCH" --prune

Phase 2: コンフリクトしたファイル衝突予測（条件付き）

# プロジェクト側 CLAUDE.md に「コンフリクトしたファイル」見出しがあるか確認
grep -nE '^#{1,3}\s*コンフリクトしたファイル' "$REPO/CLAUDE.md" 2>/dev/null
# ヒットした場合: 触る予定のファイルと OPEN PR のファイル一覧を突合する
# 衝突がある場合: ユーザーに「先方 PR のマージ待ち / 並走 / 統合」のどれを取るか確認

Phase 3: worktree 作成（必ず実施）

# 命名規約（rules: always/naming/commit-branch）に従いブランチ名を決定する
BRANCH="feature/profile-cache-headers"
WT="$WORKTREE_ROOT/$BRANCH"

# 同名ブランチ・同名パスの事前チェック（存在した場合はユーザーに提示して停止）
git -C "$REPO" show-ref --verify --quiet "refs/heads/$BRANCH" && echo "[STOP] 既に存在"
[ -d "$WT" ] && echo "[STOP] パスが既に存在"

# worktree 作成（-b で新規ブランチ生成と worktree 作成を同時に行う）
git -C "$REPO" worktree add -b "$BRANCH" "$WT" "origin/$DEFAULT_BRANCH"
Phase 3 の禁止事項: 既に同名ブランチや同名パスが存在する場合、強制削除してはならない。エラーをユーザーに提示して判断を仰ぐ。復旧手順は references/detailed-procedures.md を参照する。

Phase 4: 作業（必ず実施）

# 以後の作業は worktree 内に閉じる
cd "$WT"
  • プロジェクト側 CLAUDE.md に「単機能フロー」「バルクフロー」「メンテナンスフロー」の定義がある場合はそのフローに従う。定義がない場合は TDD の 3 ステップ（テスト先書き→最小実装→リファクタリング）で進める。
  • pre-commit / pre-push フックは worktree でも自動的に効く（.git がリンク参照されるため）。
  • コミットは grouping-commits スキルを Skill ツールで起動して行う。
  • rebase 競合が発生した場合、プロジェクト側 CLAUDE.md に「コンフリクト解消プロトコル」見出しがあればそちらを参照する。

Phase 5: PR 作成（必ず実施）

実行前に必ずユーザー承認を取ること: push はリモートリポジトリに変更を反映する副作用を伴う。AskUserQuestion ツールで「git push -u origin <branch> を実行してよいか？」を確認してから実行する。 
# ユーザー承認後に push
cd "$WT"
git push -u origin "$BRANCH"

# GitHub リモートあり: formatting-pr スキルで本文を整形し PR 作成
# GitHub 以外: ユーザーに「push のみ完了」と通知して停止

Phase 6: 後片付け（必ず実施・PR 作成完了時点）

cd "$REPO"
git -C "$REPO" worktree remove "$WT"
git -C "$REPO" worktree prune
git -C "$REPO" branch --show-current   # → $DEFAULT_BRANCH であることを確認
失敗メッセージ原因対処
contains modified or untracked files未コミット変更ありgit status で確認 → ユーザーに「コミット・stash・破棄」を選んでもらう。勝手に破棄しない
is locked意図的にロックされているロックの理由をユーザーに確認してから git worktree unlock を提案する
nested worktree foundworktree 内にさらに worktree がある内側の worktree を先に削除してから外側を畳む

Phase 7: レビュー指摘で再作業が必要になった時（必要時のみ）

# 既に push 済みのブランチを再 checkout する。-b は付けない
BRANCH="feature/profile-cache-headers"
WT="$WORKTREE_ROOT/$BRANCH"
git -C "$REPO" fetch origin "$BRANCH"
git -C "$REPO" worktree add "$WT" "$BRANCH"
cd "$WT"
# 修正 → push → Phase 6 を再実行
Phase 7 の注意: 再 checkout では -b を付けない。-b は新規ブランチ作成フラグであり、既存ブランチに付けるとエラーになる。

非 GitHub リモートのフォールバック

状況挙動
GitLab / Bitbucket / 自前 git サーバーPhase 1–3 の gh pr list を skip。Phase 5 は git push で停止し、ユーザーに「リモートの web UI で PR を作成してください」と通知する
リモート設定なし（ローカルのみ）git push が失敗するため、ユーザーに「git remote add origin <URL> を実行してください」と通知する

§6スロット運用とポート管理

ポート管理規約（~/.claude/rules/always/local-environment/port-management/rule.md）と連携し、worktree ごとに開発サーバーのポート番号を一意に決定する。

スロット割当方式

worktree は作業ごとに名前ベース（ブランチ名準拠）で作成する。ポート計算に必要なスロット番号（1〜5）は作成時に動的に割り当てる。

手順操作
1git worktree list で現在の worktree 一覧を取得する
2各 worktree の .port-slot ファイルで割り当て済みスロット番号を確認する
3空きスロット（1〜5）のうち最小のものを割り当てる
4<worktree>/.port-slot にスロット番号を書き込む（1 行、数字のみ）
5空きスロットがない場合はエラー（全プロジェクト合計 5 本が上限）

ポート計算式

# ポート = プロジェクトベース + (worktree スロット × 10) + サービスオフセット
# 例: kakeibo frontend, slot 2 → 8200 + (2 × 10) + 1 = 8221
サービスオフセットサービス
+0backend (API)
+1frontend (dev server)
+2docs / portal
+3DB ツール（Prisma Studio 等）
制約: ランダムポートでのサーバー起動は禁止。全プロジェクト合計で worktree は 5 本まで（Docker を併用する環境でのメモリ消費を考慮した設計）。worktree 削除時に該当スロットのポート範囲を一括 kill する。詳細は ~/.claude/rules/always/local-environment/port-management/rule.md を参照。

slot worktree（撤去済み・歴史的記述）

<repo>/.claude/worktrees/wtN + .slot-pool.json による slot 運用（旧プロジェクト固有の例外）は 2026-06-28 に撤去済み。全リポジトリの worktree は ~/Projects/worktrees/<repo>/<branch>/ の中央規約に一本化されている。詳細は §6 を参照。

§7復旧モード（撤去済み・歴史的記述）

<repo>/.claude/worktrees/wtN + .slot-pool.json による slot 運用（旧プロジェクト固有の例外）は 2026-06-28 に撤去済み。全リポジトリの worktree は ~/Projects/worktrees/<repo>/<branch>/ の中央規約（本ガイド §2・§4）に一本化されており、本節に記載していた wt1〜wt3 台帳復旧手順（Phase R1〜R4）は現存する基盤に対応しない。異常発生時は §4 の Phase 6・references/detailed-procedures.md の通常復旧手順に従う。

§8hook による強制（worktree-required-check）

scripts/check-worktree-required.sh が PreToolUse(Write|Edit|MultiEdit|NotebookEdit) に登録されており、メイン作業ツリーへの直接編集を検出して decision:block で止める。

block される条件

判定動作
編集対象ファイルパスが取得できない通過（exit 0）
/tmp/.allow-main-edit が存在する通過 + ファイルを消費（ワンショット）
対象が ~/.claude/ 配下通過（グローバル設定は例外）
対象が ~/agent-home/ 配下通過（agent-home は例外）
対象ディレクトリが存在しない通過
git リポジトリ外通過
repo/.git がファイル（worktree 内）通過
上記以外（メインツリーへの直接編集）BLOCK — [WORKTREE-REQUIRED] を注入

注入タグへの対応

[WORKTREE-REQUIRED] 受信時
作業を中断し、parallel-dev-worktree スキルを発動して worktree を作成する。その後 worktree 内で作業を再開する。例外マーカー（/tmp/.allow-main-edit）はワンショット消費で 1 回のみ block を解除する。人間がターミナルから手動で touch する専用であり、エージェントによる touch は permissions.deny で自動拒否される。

remind-worktree-cleanup.sh

Stop hook として動作する。実装フロー完了時に worktree の後片付けが未実施の場合（セッションの cwd がマージ済み・クリーンな worktree のまま）に WORKTREE-CLEANUP を検出し、decision:block で書き戻す。counter が 2 に達すると自動解除される（livelock 防止）。

§9よくある誤操作と防御

誤操作防御・対処
メイン作業ツリーで git pull origin main Phase 1 では git fetch のみ。pull は使わない。ローカルの $DEFAULT_BRANCH ブランチ作業ツリーには checkout・pull しない。
既存 worktree がある状態で同名 worktree add Phase 3 の存在チェックで停止。強制削除せずユーザーに判断を仰ぐ。
PR 作成前に worktree を畳む Phase 6 の発動条件を「PR 作成完了時点」に固定。
Phase 7 で -b を付けて再 checkout Phase 7 の手順に「-b は付けない」と明記。付けるとエラーになる。
worktree 内の未コミット変更ごと remove Phase 6 の失敗時は --force で強制削除しない。ユーザーに提示して指示を仰ぐ。
Phase 1 で作業ツリーが汚れていたら自動 stash ユーザー確認なしに stash しない。汚れの内容を提示して指示を仰ぐ。
孤児 worktree の放置 git worktree prune --dry-run で確認後 git worktree prune を実行する。

デバッグコマンド

# 現存 worktree 一覧
git -C "$REPO" worktree list

# 孤児 worktree のクリーンアップ（実体ディレクトリが消えているもの）
git -C "$REPO" worktree prune --dry-run
git -C "$REPO" worktree prune

# .git/worktrees/ の生データ（緊急時のみ）
ls -la "$REPO/.git/worktrees/"

§10他スキルとの連携

スキル役割・連携ポイント
命名規約（rules: always/naming/commit-branch） ブランチ名の prefix・slug 規約。Phase 3 でブランチ名を決定する際に参照する。本スキルでは再列挙しない。
grouping-commits Phase 4 のコミット段階で Skill ツールから呼び出す。
orchestrating-dev-flow 統合開発フロー。Phase 2 が本スキルを呼び出す。PR 整形は同スキルの references/module-formatting-pr.md、レビュー指摘修正は references/module-fixing-review-findings.md に従う。
reviewing-single-pr-with-inline-comments /tmp/pr-<N>-... で PR レビュー用 worktree を切る別目的スキル。本スキルとは用途が異なり、同時に動く場面はない。
managing-github-issues（verify モード） issue ベースのブランチ（issue-<N> 形式）作業時に併用される。本スキルの <prefix>/<slug> 形式とは別系統。
launching-battle-base-dev-servers / test-e2e Phase 4 でアプリ起動・E2E テストを行うタスク（UI 変更・画面追加・E2E テスト追加）の時のみ呼び出す。CWD を $WT に向ける。
auto-ship Phase 5 の代替として使用可能。内部で grouping-commits / formatting-pr / fixing-review-findings を Phase 1〜5 で順に呼び、PR 作成からマージまで自走する。「PR 作成後に手動レビューを挟みたい」かどうかで本スキルの通常フローと使い分ける。

実装着手前チェックリスト

  • $REPO $DEFAULT_BRANCH $WORKTREE_ROOT $HAS_GH を取得した
  • Phase 1 の 4 項目を並列実行した
  • ブランチ名は命名規約（rules: always/naming/commit-branch）の prefix（feature/ 等）に従った
  • git worktree add -b "$BRANCH" "$WT" "origin/$DEFAULT_BRANCH" で -b を付けた（Phase 7 のみ -b なし）
  • PR 作成後、git worktree remove で worktree を畳んだ
  • メイン作業ツリーが $DEFAULT_BRANCH に戻っていることを確認した
