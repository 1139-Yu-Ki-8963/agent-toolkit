# create モード — issue 作成（対話型・整形型）

managing-github-issues の create モード本文（旧 creating-issue スキル。Type 性質: action）。

ユーザーへの質問を1問ずつ重ね、GitHub issue テンプレートを埋めて起票する。
内容が把握済みの場合（「issueにして」「issue に起こして」など）は対話をスキップして直接整形する。

---

## 基本ワークフロー

### 事前判定: 把握済みかどうか

Claude が同一セッションで既にバグ内容・機能要件を把握している場合は **Phase 1-1 をスキップして Phase 1-2 へ**進む。
把握済みかどうか不明な場合や内容を1から決めたい場合は Phase 1-1 から始める。

残タスクを「続けるか・issue に積むか」をユーザーに選択させる起点として使う場合も同様に Phase 1-2 から始める。

---

### Phase 1-1: 続行 or 登録の選択（コンテキスト把握済みの場合のみ）

完了条件: ユーザーが「そのまま対応」「issue として登録」のいずれかを選択済み

Claude が会話コンテキストから未対応の作業を特定している場合は、AskUserQuestion でユーザーに選択させる:

- **question**: 「[作業内容の1行要約] について、どうしますか？」
- **options**:
  - 「このセッションでそのまま対応する」
  - 「issue として登録する（あとで対応）」

「そのまま対応する」→ スキルを終了し、Claude がそのまま作業を続行する。
「issue として登録する」→ Phase 1-2 へ進む。

---

### Phase 1-2: 種別判定

完了条件: バグ報告・機能追加のいずれかに種別が確定している

把握済みの場合は会話コンテキストからバグ報告か機能追加かを自動判定する。
判定できない場合、または内容を1から決める場合は AskUserQuestion で種別を選択させる:

- **question**: 「どちらのissueを作成しますか？」
- **options**:
  - 「バグ報告 — 既存機能が正しく動かない」
  - 「機能追加・改善 — 新しい機能や変更の提案」

---

### Phase 2: 情報収集

完了条件: 必要情報が収集済み、またはスキップ扱いで空欄化されている

把握済みの場合はこの Phase をスキップして Phase 3 へ進む（**ユーザーへの質問は一切しない**。情報が足りなければ `なし` / `不明` と記入する）。

把握済みでない場合は各質問を AskUserQuestion で **1問ずつ** 聞く。
「なければスキップ可」の質問でユーザーが「スキップ」と答えた場合は、そのセクションを空欄のままにする。

#### バグ報告の場合（`bug-report.md` テンプレートに対応）

1. 「何が起きていますか？エラーメッセージがあれば貼ってください。」
2. 「正しく動いた場合、どうなるはずですか？」
3. 「再現手順を教えてください。（わからなければスキップ可）」
4. 「影響するファイルのパスはわかりますか？（わからなければスキップ可）」
5. 「修正方針のイメージはありますか？Before / After 形式で書けると理想的です。（なければスキップ可）」

#### 機能追加・改善の場合（`feature-request.md` テンプレートに対応）

1. 「何を実現したいですか？1〜2文で教えてください。」
2. 「この機能が『完成』と言える条件を教えてください。箇条書きでOKです。」
3. 「変更が必要なファイルはわかりますか？（わからなければスキップ可）」

---

### Phase 3: テンプレート読込・整形・生成確認

完了条件: 本文全体がユーザーに提示され「このまま登録する」が選択されている

`.github/ISSUE_TEMPLATE/bug-report.md` または `feature-request.md` を Read し、収集した情報（または把握済みの内容）でテンプレートを埋める。本文全体をユーザーに見せる。

テンプレートファイルが存在しないリポジトリでは、Phase 2 の質問項目（バグ報告: 現象・期待動作・再現手順・影響ファイル・修正方針 / 機能追加: 実現内容・完了条件・変更ファイル）をそのままセクション構成として代替する。対象リポジトリがカレントと異なる可能性がある場合は、起票先リポジトリをユーザーに確認する。

把握済みの場合の埋め方の方針:
- 情報がある → 具体的に記述する
- 情報がない → `なし` または `不明` と記入する。セクションは削除しない。

AskUserQuestion で確認する:

- **question**: 「この内容でissueを登録しますか？」
- **options**:
  - 「このまま登録する」
  - 「修正してから登録する」

「修正する」を選んだ場合は、修正したいセクション名と修正内容をユーザーにチャットで入力してもらい、該当セクションを修正して Phase 3 の冒頭に戻る。

---

### Phase 4: `gh issue create` で登録

完了条件: `gh issue create` が成功し issue URL が表示されている

```bash
gh issue create \
  --title "【バグ】<タイトル>" \
  --label bug \
  --body "$(cat <<'EOF'
<生成した本文>
EOF
)"
```

機能追加・改善の場合は `--title` を `【機能】<タイトル>`、`--label` を `enhancement` に変える。

登録後に issue URL を表示する。

## ループ設計

| 要素 | 内容 |
|---|---|
| 反復条件 | ユーザーが修正を要求した場合 |
| 上限回数 | 3 回 |
| 収束停止 | 「このまま登録する」選択時 |
| 発散停止 | 同一修正依頼が 2 回連続で再発 |

## 完了条件

各 Phase の完了条件は各見出し直後の `完了条件:` 行を参照。

**Goal**: `gh issue create` が成功し、テンプレートのセクション構造を保ったまま issue が登録されている

## テンプレート参照先

- バグ報告: `.github/ISSUE_TEMPLATE/bug-report.md`
- 機能追加: `.github/ISSUE_TEMPLATE/feature-request.md`

テンプレートを Read してから、プレースホルダー（`<!-- ... -->` コメント）を収集した情報で置き換える。

---

## 重要な注意事項

- 対話モードでは質問を必ず1問ずつ聞く。まとめて複数を投げない。
- 「スキップ可」の質問でスキップされた場合は、テンプレートの該当セクションを `なし` または空欄のままにする。削除はしない。
- 把握済みモードでは Phase 3 でユーザーに質問しない。情報が足りなければ `なし` / `不明` と書く。
- Phase 1-1 の選択肢で「そのまま対応する」が選ばれたらスキルを即終了する。後続段階を実行しない。
- テンプレートのセクション構造（見出し・チェックリスト）は変更しない。
- ユーザーが途中でキャンセルを求めた場合はすぐに中断する。

## 完了報告

`managing-agent-configs/references/skills/completion-report-format.md` の共通骨格（作業報告型）に従う。
固有の検証行: issue URL の取得確認

## 予想を裏切る挙動

- `gh issue create` は対話入力を求めるため `--body` フラグで本文を渡す — 省略するとターミナルが止まる
- issue の内容を 1 から対話的に決めたいか・把握済みで整形のみかは、会話コンテキストから自動判定する
