§1概要

managing-github-issues は GitHub issue の起票・選択着手・スコープ検証を 1 つの動線で担うモードハブ。 旧 creating-issue / picking-issues / verifying-issue-scope の 3 スキルを統合し、 ユーザー発話・状況からモードを判定した上で、モードごとの references/ を必要時のみロードする。

本体ハブだけでは作業できない。モード判定後、必ず該当 references を Read してから着手する。 モード間の自動連鎖はない — 各モードは独立に完結する。

§2出力

モードごとに、実行が完了した時点で以下の成果物が確立される。

create
登録された GitHub issue。テンプレート構造（見出し・チェックリスト）を保った本文と、gh issue create 実行結果として表示される issue URL。
pick
選択 issue に対する実装フロー起動。issue の flow:* ラベルに応じたスキル（flow-feature / flow-maintenance）が Skill ツールで起動された状態。ルーティン経由の場合は処理/スキップ一覧も出力される。
verify
staged ファイルごとの OK / 要確認 / NG 判定表。NG が 1 件以上あれば git restore --staged の具体コマンドも併記される。

§3モード判定

ユーザー発話・状況から 1 モードを判定し、該当 references のみを Read する。

ユーザー発話・状況モードロードする references
起票・issue作成・issueにして・バグ報告を作成createreferences/creating.md
issueを選んで実装・一覧から着手・どのissueをやるかpickreferences/picking.md
スコープ検証・issue範囲チェック・issue-N ブランチで commit 前verifyreferences/verifying.md

判定が曖昧な場合は AskUserQuestion で確認する。モード間の自動連鎖はない（各モードは独立に完結する）。

§4起動トリガーとスキップ条件

種別内容
TRIGGER（起動する） 「issue作成」「起票」「issueにして」「issueを選んで実装」「issue一覧から着手」「スコープ検証」「issue範囲チェック」、issue-N ブランチで commit 前
SKIP（起動しない） PR レビュー（→ reviewing-single-pr-with-inline-comments）、issue 番号指定済みのブランチ作成（→ parallel-dev-worktree）、コミット分割（→ grouping-commits）

§6完了条件

verify全 staged ファイルの判定（OK / 要確認 / NG）が報告されている
Goal判定したモードの references に定義された完了条件をすべて満たしている

§7ループ設計

反復は create モードのみ。pick / verify は反復構造を持たない単一パス。

要素内容
反復条件（create）ユーザーが修正を要求した場合
上限回数（create）3 回
収束停止（create）「このまま登録する」選択時
発散停止（create）同一修正依頼が 2 回連続で再発
pick / verify反復構造を持たない単一パス
