claude CLI のヘッドレスモード（`claude -p`、対話画面を介さず1回の呼び出しで完結する実行方式）で、大量の対象（数十件以上）に対して1件=1呼び出しの無人バッチループを構築・起動する。マーカー冪等性（同じ対象を再実行しても安全に再開できる性質）・limit耐性（API利用上限に達しても待機して再開する性質）・残ゼロまで継続する3要件を満たし、朝まで無人運転する。

## 使用タイミング

- TRIGGER: 「大量の対象を1件ずつAI処理」「朝まで無人で回して」「ヘッドレスでバッチ」と言われた時、数十件以上で1件ごとにLLM判断が必要な時
- SKIP: 対象が数件（目安10件未満。その場でサブエージェント委任の方が速い）、定期実行（→スケジュール実行の仕組み）、LLM不要の一括置換（→Bash/sedで直接処理）

## 前提（実機確認済み CLI 仕様）

claude CLI 2.1.206 で実機確認済みの仕様。

- `-p/--print`・`--allowedTools`・`--permission-mode`（acceptEdits/auto/bypassPermissions/manual/dontAsk/plan）・`--output-format`（text/json/stream-json）・`--no-session-persistence` が存在する
- `--worktree` フラグは存在しない。worktree（作業ツリーの分離複製）が必要なら呼び出し側で `git worktree` を事前に用意する
- ヘッドレス実行中の対話承認は不可能（承認プロンプト自体が出ない）。無人化には `--allowedTools` と `--permission-mode` が必須。既定は `--permission-mode dontAsk` + `--allowedTools` 明示。ファイル編集主体のタスクは `acceptEdits` を使う
- 大量呼び出し時は `--no-session-persistence` で `~/.claude/projects/` の肥大化を防ぐ
- `claude -p` はローカル実行。Mac 本体のスリープ・シャットダウンで停止する。実行前に本体をスリープさせない設定を確認する

## Phase 1: パラメータ確定

## Step 1-1: 対象定義とマーカー仕様をヒアリングする

AskUserQuestion ツールで、対象の定義方法（ファイル一覧かリストか）とマーカー文字列・付与位置、編集してよい範囲を確定する。

**完了**: 対象定義・マーカー文字列・付与位置・編集許可範囲がユーザー承認済み。

## Step 1-2: 実行パラメータの既定値を確認する

AskUserQuestion ツールで、使用モデル（既定: 安価モデル）・limit 待機間隔（既定60分）・ログ出力先・同一対象の連続失敗上限 K（既定3）を確認する。

**完了**: 全パラメータ表（対象定義・マーカー・編集範囲・モデル・待機間隔・ログ先・失敗上限K）が確定しユーザー承認済み。

## Phase 2: 対象一覧の生成と正規化

## Step 2-1: 対象一覧を生成する

対象がファイル群の場合は Glob ツールで対象パターンに合致するファイル一覧を取得する。対象がリスト由来の場合はユーザー提示のリストをそのまま使う。取得した一覧は Write ツールで1行1対象のテキストファイルに書き出す。

**完了**: 対象一覧ファイルが生成されている。

## Step 2-2: 正規化して件数を確認する

Bash ツールで CRLF 除去（`tr -d '\r'`）・エンコーディング正規化・空行除去を行う。Grep ツールで一覧内の既存マーカー付与件数（既処理件数）をカウントする。

**完了**: 正規化済み一覧と総数・既処理数がユーザーに提示済み。

## Phase 3: 単発ドライラン

## Step 3-1: ドライラン実行環境の承認を得る

`claude -p` は Anthropic API への外向き接続が必要なため、Bash ツールのサンドボックス（コマンド実行環境の権限制限機構）が外向き接続を制限する環境では、AskUserQuestion ツールでサンドボックス無効化の承認をここで取得する。

**完了**: サンドボックス無効化の要否判断とユーザー承認が完了している（不要な環境では本 Step は「不要」と記録して次に進む）。

## Step 3-2: 1件実行し成否判定を検証する

Bash ツールで対象1件だけフォアグラウンド実行する。Read ツールで出力を確認し、Grep ツールでマーカーが対象に実際に付与されたか判定する。**合格するまで Phase 4 に進むことを禁止する。**

**完了**: ドライラン1件でマーカー実付与と成否判定コマンドの正当性を確認済み。

## Phase 4: 無人ループ起動

## Step 4-1: ループスクリプトの雛形にパラメータを反映する

Read ツールで `references/loop-design.md` の雛形を読み込み、Phase 1 で確定した値（TARGETS_FILE・MARKER・LOG・WAIT_SECONDS・FAIL_LIMIT_K・MODEL・ALLOWED_TOOLS・PER_ITEM_PROMPT）を埋める。埋めた内容はディスクに保存せず、次 Step の Bash コマンド文字列にそのまま展開する。

**完了**: 全プレースホルダが実値に置換され、実行可能なコマンド文字列が組み上がっている。

## Step 4-2: バックグラウンド起動しPID（プロセスID。起動したプロセスを識別する番号）生存を確認する

Bash ツールで `nohup bash -c '...' >> ログ 2>&1 & disown` 構造で起動する。Claude Code のバックグラウンドシェルが「メインセッション応答後まもなく終了する」仕様の影響を避けるため、シェルから切り離す。起動直後に PID を取得し、10秒後に `kill -0 $PID` で生存確認する。

**完了**: PID の生存確認が取れ、監視コマンド（`tail -f <ログ>`）をユーザーに提示済み。

## Phase 5: 監視と完了確認

## Step 5-1: 残件カウントを確認する

Bash ツールで残件カウントコマンド（マーカー未付与かつ failed リスト外の件数）を実行し、Grep ツールでマーカー未検出件数を数える。

**完了**: 残件数が確認され、直近の周回サマリと突き合わせ済み。

## Step 5-2: ログを確認し完了判定する

Read ツールでログファイルの末尾（周回サマリ部分）を確認する。残ゼロ、または failed リストへの退避で全件確定していれば完了とする。

**完了**: 残ゼロ、または failed リスト退避で全対象が確定している。

## Phase 6: 完了報告

## Step 6-1: 完了報告を提示する

Bash ツールで最終の残件数・failed 件数・実周回数を再カウントしてから、`## 完了報告` 節の形式で報告する。

**完了**: 完了報告の形式（後述の `## 完了報告` 節）に従い、ログの場所・マーカー文字列・対象総数・実周回数・失敗件数を含む報告を提示済み。

## ループ設計

| 層 | 反復条件 | 上限・停止条件 |
|---|---|---|
| 周回ループ | 未処理残（未処理かつfailedリスト外）> 0 | 周回数自体には上限を設けない（意図的。要件「残ゼロまで継続」のため）。停止条件は次の2パターン: (1) 収束停止 — 全対象が成功しマーカー付与済みでremaining=0 (2) 発散検知（対象単位） — 個々の対象がFAIL_LIMIT_K回連続失敗するとfailedリストへ退避され、以後remainingの計算対象から除外される。この2パターンの組み合わせにより、対象数が有限である限りremainingは必ず0に収束し周回ループも必ず有限回で終了する |
| 対象ループ | 対象一覧を先頭から1件ずつ処理 | マーカー既存ならスキップ。`claude -p` 実行後、**マーカー実在で成否判定する（終了コードは使わない）**。未付与なら失敗カウント加算し、同一対象K回連続失敗でfailedリストへ退避し以後スキップする |
| limit待機 | 出力にlimit文言パターンを検知 | 検知したら待機間隔だけsleepしてから処理を再開する。当該対象は未完のまま残り、次周回で再試行される。パターン正本は `references/loop-design.md` |

## サブエージェント委任仕様

ここでの委任先は Agent ツールではなく `claude -p` の子プロセス（ヘッドレス呼び出し）。

| 項目 | 内容 |
|---|---|
| 呼び出し箇所 | Phase 3（ドライラン1件）・Phase 4（ループ内の各対象） |
| 実行主体 | `claude -p` プロセス。subagent_type は該当なし。モデルはPhase 1で確定した安価モデル |
| プロンプト骨格（3要素契約・必須） | (1) 編集してよい範囲の明示 (2) 完了時に必ず付けるマーカー文字列 (3) 「マーカーが既にあれば何もせず即終了」。1呼び出し=1対象に収まる小ささに分割する |
| 期待返却値 | 対象へのマーカー実書き込み。テキスト出力は成否判定に使わない |

## Gotchas

- `--worktree` フラグは存在しない。worktree が必要なら事前に `git worktree` を用意する
- バックグラウンドシェルの寿命が短いため `nohup` + `disown` + 起動10秒後の生存確認が必須
- サンドボックスの外向きネットワーク制限に当たりうる。ドライラン（Phase 3）で承認を取る
- 終了コードは信用しない。成否はマーカー実在の grep で判定する
- 周回ループに上限はない（意図的な設計判断。`references/gotchas.md` 参照）
- 詳細は `references/gotchas.md` を参照

## 設計判断

`scripts/` ディレクトリを置かない。ループ本体をディスク保存しない方針（Phase 4）と整合させるため、limit検知パターン等は `references/loop-design.md` からその都度インライン展開する。これにより生成されたループがスキルディレクトリの存在に依存しない自己完結ワンライナーになり、スキル呼び出し元のセッションが終了した後もバックグラウンドプロセス単体で完走できる。

周回ループには意図的に上限を設けない。本スキルの目的（冒頭参照）は「残ゼロまで継続」であり、これは全対象が処理されるまで無制限に周回を繰り返すという意味であるため、周回数の上限は要件と矛盾する。暴走防止は対象単位の `FAIL_LIMIT_K` のみで担保する。各対象は最大 `FAIL_LIMIT_K` 回失敗すると failed リストへ退避し以後の周回から除外されるため、全対象が「成功」または「failedリスト行き」のいずれかで確定すれば `remaining` は自動的に0になりループが自然終了する。対象数が有限である限りループも必ず有限回で終わる。
