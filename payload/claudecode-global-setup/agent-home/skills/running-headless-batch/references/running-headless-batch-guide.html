ヘッドレス無人バッチループ
running-headless-batch

claude CLI のヘッドレスモード（claude -p）で、大量の対象に対して1件=1呼び出しの無人バッチループを構築・起動する。マーカー冪等性・limit耐性・残ゼロまで継続の3要件を満たし、朝まで無人運転する。

こんな人向け
数十件以上の対象に、1件ごとに LLM（大規模言語モデル）の判断を要する処理を、無人・長時間で回したいエージェント・利用者。
こんな場面で使う
「大量の対象を1件ずつAI処理」「朝まで無人で回して」「ヘッドレスでバッチ」と言われた時。対象が数件（10件未満）ならその場でサブエージェント委任、定期実行ならスケジュール実行の仕組み、LLM 不要の一括置換なら Bash/sed を使い、本スキルは使わない。
対応 OSmacOS（claude CLI 2.1.206 で実機確認済み）
検証状況SKILL.md を正本として整理。実機検証手順は references/testing-playbook.md（Test A〜D）
依存claude CLI（-p/--allowedTools/--permission-mode/--output-format/--no-session-persistence）、Bash・Read・Write・Grep・Glob・AskUserQuestion（allowed-tools）
関連資料SKILL.md、references/loop-design.md、references/gotchas.md、references/testing-playbook.md

§1概要と適用範囲

このスキルは claude -p（対話画面を介さず1回の呼び出しで完結するヘッドレス実行モード）を使い、大量の対象それぞれに独立した LLM 判断を適用する無人バッチループを構築する。スキルの型は orchestration で、Phase 1〜6 を順に実行する。

状況本スキルの要否
対象が数十件以上、1件ごとにLLM判断が必要TRIGGER 本スキルを使う
対象が数件（目安10件未満）SKIP その場でサブエージェント委任の方が速い
定期実行したいだけSKIP スケジュール実行の仕組みを使う
LLM判断が不要な一括置換SKIP Bash/sed で直接処理する

§2出力

本スキルの実行が完了すると、以下が確立される。

起動済みの無人バッチループ（バックグラウンドプロセス）
nohup + disown でメインセッションから切り離され、PID（プロセスID）の生存確認が取れた状態で稼働し続ける。
進捗ログファイル
周回（lap）ごとの開始・成功・未完了・limit検知・待機・failed退避の記録が時系列で追記され続ける。tail -f で監視できる。
failedリスト
同一対象がK回連続で失敗した際に退避される一覧。以後の周回でスキップされ、無限周回を防ぐ。
完了報告
ログの場所・マーカー文字列・対象総数・実周回数・失敗件数を含む、共通骨格（作業報告型）に従った最終報告。

§3実機確認済み CLI 仕様

claude CLI 2.1.206 で実機確認済みの仕様。

フラグ役割
-p / --printヘッドレスモード。対話画面を介さず1回で完結する
--allowedToolsヘッドレス実行中は対話承認が一切出ないため、許可するツールをここで明示する
--permission-modeacceptEdits/auto/bypassPermissions/manual/dontAsk/plan の6種。既定は dontAsk
--output-formattext/json/stream-json の3種
--no-session-persistence~/.claude/projects/ へのセッション記録を無効化する。大量呼び出し時の既定
存在しないフラグ: --worktree は claude CLI に存在しない。worktree（作業ツリーの分離複製）が必要な場合は呼び出し側が git worktree を事前に用意する。

§6ループ設計の3層構造

層反復条件上限・停止条件
周回ループ未処理残 > 0上限なし（意図的。要件「残ゼロまで継続」に基づく）。停止条件は2パターン: (1)収束停止＝残ゼロ (2)発散検知（対象単位）＝FAIL_LIMIT_K連続失敗退避
対象ループ対象一覧を先頭から1件ずつ処理マーカー既存ならスキップ。K回連続失敗でfailed退避
limit待機limit文言検知時待機間隔だけsleepしてから処理を再開。当該対象は次周回で再試行

§7サブエージェント委任仕様

ここでの委任先は Agent ツールではなく claude -p の子プロセス（ヘッドレス呼び出し）である点に注意する。

項目内容
呼び出し箇所Phase 3（ドライラン1件）・Phase 4（ループ内の各対象）
実行主体claude -p プロセス。モデルはPhase 1で確定した安価モデル
期待返却値対象へのマーカー実書き込み。テキスト出力は成否判定に使わない

§81件分プロンプトの3要素契約

(1) 編集してよい範囲の明示
対象自身のみ、あるいはPhase 1で確定した範囲に限定する。範囲外のファイルには触れさせない。
(2) 完了時に必ず付けるマーカー文字列
Phase 1で確定したマーカー文字列と付与位置を明記する。
(3) マーカーが既にあれば何もせず即終了
再実行時の冪等性を保証する。1呼び出し=1対象に収まる小ささに分割する。

§9Gotchas

落とし穴対策
--worktreeフラグの不存在呼び出し側で事前にgit worktree addを実行する
バックグラウンドシェルの短命化nohup + disown + 起動10秒後の生存確認を必須にする
サンドボックスの外向き通信制限Phase 3のドライランで承認を取る
終了コードの非信頼性マーカー実在のgrepで成否判定する
bashの連想配列非対応（macOS標準bash 3.2）awkベースのファイルカウンタで失敗回数を管理する

詳細は references/gotchas.md（事象・原因・対策の3行構成）を参照。

§10実機検証手順（Test A〜D）

Test目的合格基準の要点
Test AE2E＋マーカー冪等性3件全てにマーカー付与・既付与分はスキップ・再実行で呼び出し0回
Test B成否判定がマーカー実在で行われること書き込み不可対象でK回失敗後にfailedリストへ退避
Test Climit耐性（モック）1回目limit検知→待機→2回目で完走
Test Dバックグラウンド生存起動10秒後もPID生存・ログが成長

実行コマンド例・詳細な合格基準は references/testing-playbook.md を参照。
