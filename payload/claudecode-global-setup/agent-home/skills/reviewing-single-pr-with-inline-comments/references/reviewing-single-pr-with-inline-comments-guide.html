PR レビュー・インラインコメント投稿
reviewing-single-pr-with-inline-comments

GitHub PR の差分を読んでコードレビューし、行指定のインラインコメント・総括コメント・LGTM/request-changes を GitHub へ投稿するスキル。単一 PR だけでなく複数 PR の並列一括処理にも対応する。

こんな人向け
GitHub PR のレビュー作業を自動化したい開発者。差分確認・コードレビュー・コメント投稿・マージ判定を一連の流れで実施したい場面全般。
こんな場面で使う
「PR をレビューしてほしい」「コメントを確認して」「差分を確認して LGTM か判定して」「routine PR をまとめてレビュー」と伝えたとき。PR URL か PR 番号が指定されれば即開始し、未指定の場合は一覧表を提示してユーザーに選択させる。PR なしのローカル編集・コミットのみのタスクには使わない。
対応 OSmacOS
検証状況SKILL.md を正本として整理（個別の実機検証記録なし）
依存gh CLI（GitHub CLI、認証済み）、Agent / Bash / Read / Grep（allowed-tools）、check-no-rebase-on-pr-branch.sh（scripts/）
関連資料SKILL.md、references/finding-formats.md、scripts/check-no-rebase-on-pr-branch.sh

§1概要

reviewing-single-pr-with-inline-comments は type orchestration のスキル。 GitHub CLI (gh) を通じて PR 差分を取得し、AI によるコードレビューとテスト実行を経て インラインコメント・総括コメント・レビュー判定（approve / request-changes）を GitHub に投稿する。

項目値
スキル名reviewing-single-pr-with-inline-comments
種別orchestration（オーケストレーション型）
カテゴリ品質・レビュー
タイプ強制型
TRIGGER「PR レビュー」「コメント」「差分確認」「LGTM」「まとめてレビュー」「routine PR」
SKIP 条件PR なしのローカル編集・コミットのみ
allowed-toolsAgent / Bash / Read / Grep
外部影響: 本フローは GitHub へコメント投稿・レビュー判定（approve / request-changes）を行う。 Phase 8 の判定ロジックを満たした時のみ自動投稿する。判定を満たさない不確実なケースでは投稿せずユーザーに報告する。

§2出力

レビューが完了すると、GitHub 上に次の投稿物が生成され、判定結果に応じて PR の状態が変化する。

インラインコメント
重大な問題・警告のうち file と line が特定できた指摘について、該当コード行に直接投稿されるコメント。修正案コードと「詳細は総括コメントを参照」の文言を含む。投稿後に取得した html_url は総括コメント本文へのリンクとして埋め込まれる。
総括コメント
重大な問題・警告・提案の指摘一覧、lint・ユニットテスト・E2E テストの OK/FAIL 結果、ドキュメントカバレッジ未更新等の補足を 1 本にまとめたコメント。各指摘は該当インラインコメントへのリンクを伴う。
レビュー判定（approve / request-changes）
重大な問題が 0 件かつ全テスト（lint・FE/BE ユニット・E2E）が通過していれば approve（LGTM）、重大な問題またはテスト失敗が 1 件でもあれば request-changes として GitHub の Review API に投稿される。警告が残る場合は approve を出さない。
routine PR の後処理（routine ラベル付き PR のみ）
同じルーティンの後続 PR が存在する場合は古い PR のクローズ、レビュー安全かつテスト PASS の場合はユーザー承認を得たうえでのマージ（ブランチ削除込み）。

§3PR 情報の取得と差分読み取り

PR URL のパース

PR URL https://github.com/OWNER/REPO/pull/NUMBER から以下を抽出する。

要素用途
OWNERリポジトリオーナー
REPOリポジトリ名
NUMBERPR 番号

PR 基本情報の取得

# PR メタ情報取得（Phase 2）
gh pr view NUMBER --repo OWNER/REPO \
  --json title,body,headRefName,headRefOid,baseRefName,author,files

取得する主要フィールドの用途は次のとおり。

フィールド用途
headRefNameworktree に checkout するブランチ名
headRefOidインラインコメント投稿に必要な commit SHA（Phase 8 で流用）
files変更ファイル一覧（レビュー優先度判定・ドキュメントカバレッジチェックに使用）

差分取得（行番号付き）

インラインコメントの行指定に使う L数字 / R数字 プレフィックスを付けた形式で差分を取得する。

# 行番号付き差分（Phase 2 / Phase 5 AIレビュー用）
gh pr diff NUMBER --repo OWNER/REPO | awk '
/^@@/ {
  match($0, /-([0-9]+)/, old)
  match($0, /\+([0-9]+)/, new)
  old_line = old[1]
  new_line = new[1]
  print $0
  next
}
/^-/ { printf "L%-4d     | %s\n", old_line++, $0; next }
/^\+/ { printf "     R%-4d| %s\n", new_line++, $0; next }
/^ / { printf "L%-4d R%-4d| %s\n", old_line++, new_line++, $0; next }
{ print }
'

L数字: LEFT（base）側の行番号 → インラインコメントで side=LEFT に使用。
R数字: RIGHT（head）側の行番号 → インラインコメントで side=RIGHT に使用。

既存コメントの取得

# Issue Comments（PR全体へのコメント）
gh api repos/OWNER/REPO/issues/NUMBER/comments \
  --jq '.[] | {id, user: .user.login, created_at, body}'

# Review Comments（コード行へのコメント）
gh api repos/OWNER/REPO/pulls/NUMBER/comments \
  --jq '.[] | {id, user: .user.login, path, line, created_at, body, in_reply_to_id}'

§4自動フルレビューフロー（Phase 1〜9）

Phase 1: PR 選択（番号未指定時のみ）

# オープンな PR を最大 50 件取得して表形式で提示
gh pr list \
  --state open \
  --limit 50 \
  --json number,title,author,createdAt,headRefName,reviewDecision

| 2   | #10   | ダッシュボード実装    | bob    | feat/dashboard  | 修正依頼      | 2026-05-03 |

Phase 2: PR 情報取得

Phase 3: git worktree セットアップ

# フロントエンド依存関係

# バックエンド仮想環境

[ -d .venv ] || python3 -m venv .venv
.venv/bin/pip install -r requirements.txt -r requirements-dev.txt -q

§5AI コードレビュー（Phase 5 詳細）

レビュー観点の正本は ~/.claude/rules/scoped/review-checklist/code/common/rule.md（プロジェクト受け口があれば合成）。 判定は reviewing-against-rules スキル経由で code-reviewer サブエージェントに委任し、メインの自己レビューを禁止する。

優先度の定義

重要度基準
重大な問題 マージ不可。セキュリティ（S-*）・データ整合性（D-*）・案件特性の誤認
警告 マージ可だが修正必須。コード品質（Q-*）・規約違反（C-*/P-*）・アーキテクチャ（A-*）
提案 改善推奨。パフォーマンス・推奨手順（B-*）のうち動作に影響しないもの

チェックカテゴリ（全て確認する）

カテゴリ確認内容
案件特性特-01〜03 の設計方針を把握し、誤検知を避ける
セキュリティ S-*SQLi・XSS・認証バイパス・ハードコード認証情報
データ整合性 D-*型不整合・インターフェース破壊
コード品質 Q-*関数サイズ・ネスト深度・デバッグコード残留
汎用規約 C-*命名・型定義・import 方向
プロジェクト固有規約 P-01〜P-09Input/Output 型禁止・null 返却禁止・フォールバック禁止など
アーキテクチャ A-*レイヤー分離・単一責務
パフォーマンスN+1・非効率アルゴリズム
推奨手順 B-*マジックナンバー・命名・API ドキュメント
テキスト品質 TQ-*（警告扱い）JSX/HTML テンプレ内の誤字・脱字・似た別漢字・変更範囲外テキストの列挙
JSX 構文 JX-*（警告扱い）{ } がスペースを出力しない空 JSX 式になっていないか確認

ドキュメントカバレッジチェック

ドキュメントカバレッジの判定基準はプロジェクト固有観点であり、正本はプロジェクト受け口 <repo>/.claude/rules/scoped/review-checklist/code/common/rule.md に定義する。 受け口に基準が無いリポジトリではスキップし「カバレッジ観点は未定義」と明記する。 テスト・CI・lint・コメント・変数名のみの変更はプロジェクトを問わずスキップしてよい。

ドキュメント未更新を検出した場合は警告として指摘し、Phase 8 のコメント本文に含める。

判定ツリー（Phase 5 → Phase 8）

§6指摘の記録フォーマット

Phase 5 でのレビュー結果は次の YAML-like 形式で記録する。 この一次データを Phase 8 でインラインコメントと総括コメントの両方に変換する。 詳細テンプレートは references/finding-formats.md を参照すること。

A. 内部記録形式（指摘リスト）

# 1 指摘 = 以下のフィールドを持つ YAML-like レコード
- id: <ラベル例: D-CACHE-01>
  severity: critical | warning | suggestion
  title: <短い 1 行タイトル>
  file: <相対パス>
  line: <行番号>           # 特定できない場合は null
  side: RIGHT | LEFT        # 既定 RIGHT
  what_happens: |
    <ユーザー視点の現象。「いつ / どの操作で / どうなる」を含む 2〜5 行>
  why: |
    <コード視点の原因。該当箇所 file:line を bullet で並べる>
  fix: |
    <対応案。選択肢が複数なら「案 A / 案 B」で分け 3 ステップ以内>
  inline_summary: |
    <該当行用の 1〜2 行要約。インライン本文に流用>
  related_locations:        # 同一指摘が複数行に跨る場合のみ
    - <file:line>

重大な問題（severity: critical）の必須要件

3 フィールドすべてを埋めること: what_happens（ユーザー視点の現象）・ why（file:line 最低 1 件）・ fix（実行可能な選択肢）。 1 つでも欠ける場合は severity を warning に下げるか、file:line を特定し直す。

B. 総括コメント用「重大な問題」セクション

# severity: critical の各指摘に適用する固定構造
### 重大な問題 N: <title>

#### 何が起きるか（ユーザー視点）
<what_happens の本文>

#### なぜ起きるのか（コード視点）
<why の本文。関連箇所が複数なら表で網羅>

#### 対応案
<fix の本文>

該当行: [インラインコメント](<インライン投稿の HTML URL>)

C. 総括コメント用「警告」セクション

# 警告は 1 件 1 行で要約
- **[<id>] <title>**（<file:line>）<inline_summary 1 行> — [インライン](<URL>)

D. インラインコメント本文（コード行直結）

**[<id>] <title>**

<inline_summary 本文 1〜2 行>

修正案:
```<lang>
<提案コード 1〜5 行>
```

詳細: 総括コメント「重大な問題 N: <title>」または「警告: <title>」を参照

§7インライン・総括コメントの投稿（Phase 8）

ステップ 1: インラインコメント投稿

severity: critical と severity: warning の各指摘について、 file と line が特定できているものを 1 件ずつインライン投稿する。 file:line が特定できない指摘はスキップし、総括のみで扱う。 

# head commit SHA は Phase 2 の headRefOid を流用
HEAD_SHA=$(jq -r '.headRefOid' <<< "$PR_INFO")

# 単一行インライン投稿
gh api repos/OWNER/REPO/pulls/NUMBER/comments \
  --method POST \
  -f body="$(cat path/to/inline-body.md)" \
  -f commit_id="$HEAD_SHA" \
  -f path="<相対パス>" \
  -F line=<行番号> \
  -f side=RIGHT \
  --jq '.html_url'

# 複数行に跨る指摘（10〜15 行目の例）
gh api repos/OWNER/REPO/pulls/NUMBER/comments \
  --method POST \
  -f body="..." \
  -f commit_id="$HEAD_SHA" \
  -f path="src/example.py" \
  -F line=15 \
  -f side=RIGHT \
  -F start_line=10 \
  -f start_side=RIGHT

返り値の .html_url を各指摘の inline_url として保持する。 総括コメント本文の「該当行: [インラインコメント](inline_url)」リンクに埋め込む。 複数箇所に跨る指摘は代表 1 行のみインライン投稿し、他箇所は総括の表に列挙する。

注意: -F（大文字）は数値パラメータ（line・start_line）に使用する。 -f（小文字）だと文字列になり API エラーになる。 side は RIGHT（追加行）または LEFT（削除行）のみ有効。

ステップ 2: 総括コメント投稿

全インライン投稿後、--body-file で総括コメントを投稿する（HEREDOC はメモリ上の理由で避ける）。

# 総括コメント投稿
gh pr comment NUMBER --repo OWNER/REPO --body-file path/to/summary-body.md

総括コメント本文の構造:

## レビュー結果: 修正依頼

### 重大な問題
<severity: critical 各 1 件を §5 B テンプレートで展開。インライン URL を末尾に貼る>

### 警告
<severity: warning 各 1 件を §5 C の 1 行要約で列挙。インライン URL を末尾に貼る>

### 提案
<severity: suggestion の指摘を 1 件 1 行で列挙（インラインなし）>

### テスト結果
- [OK/FAIL] FE lint (biome)
- [OK/FAIL] BE lint (ruff) + 型チェック (mypy)
- [OK/FAIL] FE ユニットテスト (vitest)
- [OK/FAIL] BE ユニットテスト (pytest)
- [OK/FAIL] E2E テスト (playwright)

### 補足
<docs カバレッジ未更新等の追加指摘>

ステップ 3: GitHub Review API による request-changes

重大な問題がある場合のみ実行する。

# request-changes（重大な問題がある場合のみ）
gh pr review NUMBER --repo OWNER/REPO \
  --request-changes \
  --body "詳細は総括コメントを参照: <総括コメントの URL>"

コメントへの返信

# 既存コメントへの返信（COMMENT_ID はコメント取得の id フィールド）
gh api repos/OWNER/REPO/pulls/NUMBER/comments/COMMENT_ID/replies \
  --method POST \
  -f body="返信内容"

§8LGTM 判定条件とマージ

LGTM 投稿条件（approve）

LGTM 条件（すべてを満たすこと）:
  • 重大な問題（severity: critical）の指摘数 = 0
  • FE lint (biome) エラーなし
  • FE ユニットテスト (vitest) 全通過
  • BE ユニットテスト (pytest) 全通過
  • E2E テスト (playwright) 全通過
警告（severity: warning）が残る場合は LGTM を出さない。 
# LGTM（approve）の場合（インライン投稿はスキップ）
gh pr review NUMBER --repo OWNER/REPO --approve --body-file path/to/lgtm-body.md

LGTM 本文の構造:

## LGTM

コードレビューおよびテストが全て通過しました。

### テスト結果
- [OK] FE lint (biome)
- [OK] BE lint (ruff) + 型チェック (mypy)
- [OK] FE ユニットテスト (vitest)
- [OK] BE ユニットテスト (pytest)
- [OK] E2E テスト (playwright)

### AI レビュー所見
<提案 指摘一覧、または「指摘なし」>

routine PR の追加処理

routine ラベル付き PR を対象にしている場合は次を追加で実施する。

条件処理
同じルーティンの後続 PR が存在する 古い PR をクローズ候補として gh pr close <number> --comment "後続 PR で上書き済み" を実行
レビュー安全 + テスト PASS gh pr merge <number> --merge --delete-branch を実行

§9複数 PR モード

「PR を全部レビュー」「まとめてレビュー」「routine PR を処理」などの指示では複数 PR モードで動く。

Step 1: PR 一覧取得・選択

# routine ラベル付きを対象にする場合は --label "routine" を追加
gh pr list \
  --state open \
  --limit 50 \
  --json number,title,author,createdAt,headRefName,reviewDecision

LGTM 済み PR を判定して視覚的に区別し、「LGTM 済みも含めてすべてレビューする」の選択肢を含めてユーザーに選択させる。

Step 2: Reviewer サブエージェントを並列起動

並列起動必須: 独立した PR は必ず並列で起動する。直列処理は禁止。 
# 各 PR に対して同時にサブエージェントを起動する例
Task(general-purpose, "reviewing-single-pr-with-inline-comments スキルを使って PR #N をレビューせよ。
  Phase 1（PR選択）はスキップし、PR番号 N を直接使うこと。
  Phase 9（worktree削除）は実行しないこと——Fixer が引き継ぐ。
  最終的に以下の JSON 形式で結果を返せ:
  {pr_number, head_sha, worktree_path, p0_list, p1_list, p2_list, tests_passed}")

Step 3: 結果集約・Fixer 起動判定

条件処理
p0_list に 1 件以上 Fixer を起動しない。request-changes を投稿してスキップ
p0_list が空・p1_list に 1 件以上 fixing-review-findings スキルを使う Fixer サブエージェントを並列起動
p0_list も p1_list も空 approve / LGTM コメントを即時投稿

Step 4: routine PR の追加処理

routine ラベル付き PR を対象にしている場合のみ実施する（§7 と同様の処理）。

Step 5: 完了報告

全 PR の処理結果を表形式で報告する。

§10制約・関連スキル・予想を裏切る挙動

制約まとめ

SKIP 条件（SKILL.md TRIGGER より）: PR なしのローカル編集・コミットのみのタスクには本スキルを使わない。
GitHub 外部投稿の条件: コメント投稿・approve / request-changes は Phase 8 の判定ロジックを満たした時のみ実行する。 不確実なケースでは投稿せずユーザーに報告する。
インライン投稿優先順: ステップ 1 でインラインを先に投稿して .html_url を取得し、 総括コメント本文へ埋め込む順序を守ること。逆順にすると URL が空になる。
並列 PR は必ず並列起動: 独立した PR を順次処理することは禁止する（予想を裏切る挙動より）。
routine ラベル外の PR はルーティンモード対象外: routine ラベルがない PR には routine 固有の追加処理（古い PR のクローズ・マージ）を適用しない。

check-no-rebase-on-pr-branch.sh（PreToolUse hook）

PR worktree 内での git rebase を block し、git merge を使うよう誘導する hook。 rebase はコミット SHA を書き換えるため force push が必要になるが、force push は deny ルールで禁止されている。

判定条件動作
cwd が *-pr-[0-9]* パターンまたは */.claude/worktrees/* 配下 REBASE-BLOCKED を注入して exit 2
git rebase --abort / --continue / --skip 進行中の操作のためスキップ（block しない）
PR worktree 以外での git rebase スキップ（block しない）

main の最新を取り込む場合は git merge origin/<BASE_BRANCH> --no-edit を使うこと （fixing-review-findings スキルの Phase 2 を参照）。

スキル名関係
fixing-review-findings 本スキルがレビューした結果（p1_list）の修正を担う。警告指摘がある場合の worktree 削除も担当
launching-<project>-dev-servers Phase 7 で開発スタックが停止していた場合にフルスタックを起動するスキル
parallel-dev-worktree worktree の作成・管理の共通エンジン

予想を裏切る挙動

事象原因・対処
-F と -f の混同 数値パラメータ（line・start_line）は必ず -F（大文字）を使う。-f だと文字列として送られて API エラーになる
インライン URL が空 インライン投稿前に総括コメントを作成すると URL を埋め込めない。ステップ 1（インライン）→ ステップ 2（総括）の順序を厳守する
E2E テストがローカルでしか動かない E2E テストはローカルでのみ実行可能（SKILL.md の予想を裏切る挙動より）。CI 上では実行しない
rebase 後に push できない PR worktree で rebase を使うと force push が必要になるが deny ルールで禁止。check-no-rebase-on-pr-branch.sh が事前に block する
