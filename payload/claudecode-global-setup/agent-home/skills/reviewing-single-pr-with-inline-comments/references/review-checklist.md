# Phase 5 レビューカテゴリ詳細

reviewing-single-pr-with-inline-comments の Phase 5（AIコードレビュー）で使う観点詳細・記録フォーマット・ドキュメントカバレッジ判定をまとめる。

レビュー観点の正本は rules にある。**観点をこのファイルや委任プロンプトに書き写さず、`reviewing-against-rules` スキルの手順に従い code-reviewer サブエージェントに照合を委任する。** code-reviewer はグローバル基準 `~/.claude/rules/scoped/review-checklist/code/common/rule.md` と、存在すればプロジェクト受け口 `<repo>/.claude/rules/scoped/review-checklist/code/common/rule.md` を Read して判定する。以下の重要度定義・記録フォーマットは本スキル側（投稿処理）の仕様として残す。

## 優先度の定義

| 重要度 | 基準 |
|--------|------|
| **重大な問題** | マージ不可。セキュリティ（S-*）・データ整合性（D-*）・案件特性の誤認 |
| **警告** | マージ可だが修正必須。コード品質（Q-*）・規約違反（C-*/P-*）・アーキテクチャ（A-*） |
| **提案** | 改善推奨。パフォーマンス・推奨手順（B-*）のうち動作に影響しないもの |

## レビューの進め方

チェックリストのカテゴリを**全て**確認する:

1. **案件特性** — 特-01〜03 の設計方針を把握し、誤検知を避ける
2. **重大な問題 セキュリティ** `S-*` — SQLi・XSS・認証バイパス・ハードコード認証情報
3. **重大な問題 データ整合性** `D-*` — 型不整合・インターフェース破壊
4. **警告 コード品質** `Q-*` — 関数サイズ・ネスト深度・デバッグコード残留
5. **警告 汎用規約** `C-*` — 命名・型定義・import 方向
6. **警告 プロジェクト固有規約** `P-01〜P-09` — Input/Output 型禁止・null 返却禁止・フォールバック禁止など
7. **警告 アーキテクチャ** `A-*` — レイヤー分離・単一責務
8. **提案 パフォーマンス** — N+1・非効率アルゴリズム
9. **提案 推奨手順** `B-*` — マジックナンバー・命名・APIドキュメント
10. **テキスト品質** `TQ-*` — JSX・HTML テンプレート内のテキストを確認する（**警告**扱い）
    - TQ-01: テキストノードに誤字・脱字がないか確認
    - TQ-02: 見た目が似た別漢字・JIS 外文字の混入を確認（例: 「激魪」→「激闘」、「紅蒮」→「紅蓮」）
    - TQ-03: PR の変更範囲外で変更されたテキストを全て列挙し、意図的か確認する（意図的でなければ **警告**）
11. **JSX 構文** `JX-*`
    - JX-01: `{ }` はスペースを出力しない空 JSX 式。スペースが必要な箇所は `{' '}` を使うこと（**警告**）

## 指摘の記録フォーマット

各指摘は次の構造化形式で記録する。インライン投稿と総括コメントで使い分けるため、両形式を作る。

記録形式（A）・総括テンプレート（B/C）・インライン本文（D）・重大な問題の必須要件・模範パターンの詳細は `references/finding-formats.md` を参照する。要点のみ次に示す。

- **A. 内部記録形式**: 1 指摘 = `id / severity / title / file / line / side / what_happens / why / fix / inline_summary / related_locations` の YAML-like レコード。Phase 8 の一次データとなる
- **B. 総括「重大な問題」**: `severity: critical` を「何が起きるか → なぜ起きるか → 対応案」の固定構造で展開し、該当行のインライン URL を貼る
- **C. 総括「警告」**: 1 件 1 段落の要約 + インラインへのリンク
- **D. インライン本文**: タイトル + 1〜2 行要約 + 修正案コード
- **重大な問題の必須要件**: `what_happens` / `why`（file:line 最低 1 件）/ `fix` の 3 つを必ず埋める。欠けたら severity を warning に下げる

## レビュー後の処理（Phase 8 へのハンドオフ）

- 内部記録形式（A）の指摘リストを Phase 8 に引き渡す
- Phase 8 が「インライン投稿 → 総括投稿」の 2 ステップを実施する
- **重大な問題が 1 件でも存在 → Phase 8 で LGTM を出さない**
- 指摘が 0 件の場合も「全チェック項目を確認し問題なし」と明示する

## ドキュメントカバレッジチェック

ドキュメントカバレッジ（変更ファイルに対応する設計書・仕様書が更新されているか）の判定基準はプロジェクト固有観点であり、正本はプロジェクト受け口 `<repo>/.claude/rules/scoped/review-checklist/code/common/rule.md` に定義する。code-reviewer がグローバル基準と合成して照合する。

受け口にカバレッジ基準が定義されていないリポジトリでは本チェックをスキップし、レビュー結果に「ドキュメントカバレッジ観点は未定義」と明記する。テスト・CI・lint・コメント・変数名のみの変更はプロジェクトを問わずスキップしてよい（docs 更新不要）。

ドキュメント未更新を検出した場合は警告として指摘し、Phase 8 のコメント本文に含める。
