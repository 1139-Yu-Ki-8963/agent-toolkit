# 解説スライド生成（generating-explanation-html-slides）

与えたトピックを決まった手順で横1枚（16:9）のスライド風解説HTMLにする実行スキル。図は inline SVG で自作し、観点レビュー表による検査に合格したものだけを出力先ディレクトリに登録する。他のモデル・別セッションでも同水準の解説HTMLを再現できるよう、デザインはテンプレートの部品カタログに、品質判定は観点レビュー表に固定してある。

## Phase 1: 要件ヒアリング

このスキルで最も重要な工程。5カテゴリのヒアリング項目を埋め、充足ゴールを満たしてから次の Phase に進む。ヒアリング不足のまま生成した図は「正しく見えるが仕組みと矛盾する」欠陥（因果矛盾・数値矛盾）の温床になる。

完了条件: 5カテゴリすべてにユーザー発話由来の回答（または仮置きの明示）が記録され、充足ゴール3判定が可、スライドキーが確定していること。

外部スキルから引き継ぎ情報（業務棚卸し表・軌道記述等）が渡された場合は、それらを5カテゴリの初期値として使用する。この場合、Step 1-2・1-3 でのユーザーへの確認は初期値の修正の要否のみに短縮できる。

### Step 1-1: スライド型の選択

ユーザーの依頼文からスライド型を判定する。ユーザーが型を明示している場合はその型で進める。明示がない場合は以下の判定ロジックで自動選択し、結果を AskUserQuestion で確認する。

**型自動判定ロジック**:
| テーマの性質 | 判定結果 |
|---|---|
| 時間経過・プロセス・サイクル・推移を含む | 図解 |
| 複数対象の比較・優劣 | 図解 |
| 手順・分岐 | 図解 |
| 成熟度・段階 | 図解 |
| 課題の整理と対策の提示 | 論証テーブル |
| 論点を根拠付きで並列整理 | 論証テーブル |
| 上記に明確に当てはまらない | AskUserQuestion で確認 |

AskUserQuestion の選択肢:
(A) 図解スライド — 仕組み・プロセス・比較を inline SVG で説明
(B) 論証テーブル — 課題→現状→打ち手を表形式で整理
(C) AI が判定した型で進める（判定結果と理由を提示）

**使用ツール**: AskUserQuestion
**入力**: ユーザーの依頼文
**完了**: スライド型（図解 or 論証テーブル）が確定していること

### Step 1-2: ヒアリング5カテゴリの整理と初回質問

以下の5カテゴリについて、まずユーザーの依頼文から埋まる項目を埋める。外部スキルからの引き継ぎ時は上記の初期値を使う。埋まらないカテゴリだけを AskUserQuestion（1回最大4問）で確認する。

| カテゴリ | 確認する内容 | 図への反映先 |
|---|---|---|
| 説明対象 | 仕組みの事実。因果関係・数値条件・例外（例: 枠は開始から5時間で回復する） | 例示数値の検算・イベント線の位置 |
| 読み手 | 誰に見せるか（経営層/現場/技術者）・前提知識 | 用語レベル・補足の量 |
| 結論 | 読み手に残したい1文・読後に取ってほしい行動 | タイトル・サブタイトル・図の型 |
| 提示文脈 | どの課題への提示か・利用場面 | 蓄積簿の「提示する課題の例」列 |
| 制約 | ページ数（既定1）・含める/除く内容・業務文脈の範囲（営業日のみ等） | 軸の範囲・パネル構成 |

説明対象の仕組み詳細（数値・条件・因果）は選択肢で聞ける形になりにくいため、AskUserQuestion ではなく通常の対話で自由記述の確認をしてよい。

**入力**: ユーザーの依頼文（または外部スキルからの引き継ぎ情報）
**完了**: 5カテゴリの充足状況（埋まった値とその出所）を整理したこと

### Step 1-3: 充足判定と追加質問

充足ゴールは「Phase 2 の構造設計を推測なしで開始できる状態」。次の3判定がすべて可になるまで、不足カテゴリを AskUserQuestion で追加確認する。

1. 図の型が選べるか（結論と説明対象の性質が分かっている）
2. 例示数値の検算ができるか（仕組みの因果・数値条件が引き出せている）
3. 読み手に合う語彙レベルが決められるか

ヒアリングは最大3ラウンド。3ラウンドで埋まらない項目は推測で埋めず「仮置き」と明示して先へ進み、Phase 6 の報告に仮置き一覧を含める。

**入力**: Step 1-2 の充足状況
**完了**: 充足ゴール3判定がすべて可、または仮置きを明示したこと

### Step 1-4: スライドキーの決定

`references/meaningful-key-naming.md` を Read し、スライドキー命名の原則（連番禁止・意味語キー）を確認する。出力先ディレクトリの蓄積簿（プロジェクトに応じたパスを使用。例: `docs/解説スライド集/解説スライド集.md`）を Read し、既存キーとの重複・類似を照合する。キーは連番禁止・内容を要約した意味語（例: `claude-cursor使い分け-リミット観点`）。類似スライドが既にある場合は新規作成か既存更新かを AskUserQuestion で確認する。

**入力**: 出力先ディレクトリの蓄積簿、Step 1-2 の説明対象
**完了**: 重複のないスライドキーが確定したこと

## Phase 2: 構造設計

結論から図の型を選び、軸・パネル分割・読み順・例示数値を決める。HTMLを書き始める前に構造を固定する。本 Phase はツールを使わない設計推論工程。

完了条件: 図の型・軸の割当・パネル分割・ステップ番号・例示数値が言語化されていること。

### Step 2-1: 図の型選択 / 列構成の確定

**図解の場合**: 結論の性質から図の型を1つ選ぶ。時間経過を含む説明（切り替え・サイクル・推移）→ 時系列。複数対象の優劣・違い → 対比。手順・分岐 → フロー。成熟度・段階 → 段階図。型に迷う場合は「読み手が結論を確かめるために目で追う動き」を想像し、その動線に合う型を選ぶ。

**コンテンツ均衡のヒント**: 段階比較型（stages-grid）は全カードが同程度の情報密度であることを前提とする。テーマの性質から段階ごとのコンテンツ量に大きな偏りが予想される場合（例: 到達段階だけ要件が多い、初期段階は一言で済む）は、layer-stack型（全幅帯の縦積み）を優先的に検討する。最終的なバランス検査は Step 2-4 で行う。

**論証テーブルの場合**: 列構成の確定。2 つの既定構成から選択する。
- 課題分析型（既定 3 列）: 課題 / 現状 / 打ち手。`references/argument-table-template.html` を使用
- 現状vs理想 対比型（5 列）: 順序 / カテゴリ / 現状 / 理想 / ギャップ。`references/gap-analysis-table-template.html` を使用
テーマに応じてカスタム列を追加してもよい。

**入力**: Step 1-2 の結論1文
**完了**: 図解の場合は図の型を1つ選び選定理由を1文で言語化したこと。論証テーブルの場合は列構成（既定3列 or カスタム列追加）が確定していること

### Step 2-2: 軸とパネル分割の決定 / 行の洗い出し

**図解の場合**: 時間の話なら横軸は必ず時間にする（対象の並置に使わない）。異なる時間スケール（時間単位と週単位等）が混在する場合は1本の軸に混ぜず、スケールごとにパネルを分ける。時間軸の範囲は業務文脈に合わせる（営業日の説明なら土日を除く、日中の説明なら業務時間帯）。

**論証テーブルの場合**: 行の洗い出し。3〜5行に収める。5行を超える場合は詰め込まず、テーマを2枚に分割する案を提示する。

**入力**: Step 2-1 の図の型（または列構成）
**完了**: 図解の場合は各パネルの横軸・範囲・分割数が確定していること。論証テーブルの場合は3〜5行の行内容（超過時は分割案）が確定していること

### Step 2-3: 例示数値の検算 / 各行の論理検算

**図解の場合**: 図に入れる具体値（時刻・回数・割合）を、説明したい仕組みの因果と突き合わせて検算する。例: 「開始から5時間でリセットされる枠」を描くなら、使い切り時刻はリセット時刻より前でなければならない。矛盾する値は見た目が自然でも採用しない。

**論証テーブルの場合**: 各行の論理検算。左から右に読むだけで「問題→事実→対策」の論理が完結しているか確認する。課題列に対策を書いていないか、現状列に意見を書いていないか（列の役割を混ぜない）。数字がない箇所は「※要確認」と明記する。

**入力**: Step 2-2 の軸設計（または行内容）、説明対象の仕組み
**完了**: 図解の場合は全ての例示数値が仕組みと矛盾しないことを検算したこと。論証テーブルの場合は全行が「問題→事実→対策」で論理完結し、列の役割混在がないことを検算したこと

### Step 2-4: カード間コンテンツバランスの検査

**図解の場合（stages-grid / compare-row を使用時のみ）**: 各カードに配置する予定のコンテンツ量を以下の単位で数える。

計測単位: desc行数（40字/行で換算）+ status-row数 + example-chip数 + tree-line数 + その他ブロック要素数

最大カードと最小カードの比を算出する。比率が概ね2倍を超える場合（既存36枚の分析で空白問題の境界値として実測）、以下の対処法を順に検討する。

1. **均衡化**: 短いカードにコンテンツを追加できるか検討する（具体例の追加・status-rowの追加・desc文の補足）
2. **充填パターン適用**: 短いカードに `.tree`(テンプレートで flex:1 を持つ唯一のブロック要素)を配置し、余剰高さを構造的に吸収させる。`.layer-examples` は flex:1 を持たないため充填には使えない
3. **型変更**: 上記で解消できない場合、layer-stack型（全幅帯の縦積み）への変更を検討する

**論証テーブルの場合**: 本検査は不要（テーブル行は自然高で空白問題が発生しない）

**入力**: Step 2-1〜2-3 の構造設計
**完了**: カード型レイアウト使用時に、カード間のコンテンツ量比が2倍以内であること。または、比率超過への対処法を選択し記録したこと

### Step 2-5: 読み順番号の設計 / リード帯とヘッドメッセージの起草

**図解の場合**: 読み手が図を追う順序を1〜3個程度のステップ番号に分解する。番号は図中のチップと下部の要約文で同じ番号・同じ色を使う前提で設計する。

**論証テーブルの場合**: リード帯とヘッドメッセージの起草。リード帯は主張を1文・30字前後で言い切る。ヘッドメッセージは■付きで2〜3行、「〜と考えられる」「〜が必要」と文で断定する。リード帯とヘッドメッセージだけ読めば表を見なくても結論が分かる状態にする。

**入力**: Step 2-1〜2-4 の構造
**完了**: 図解の場合はステップ番号と各番号の意味（利用・切替・復帰等）が確定していること。論証テーブルの場合はリード帯（30字前後の断定文）とヘッドメッセージ（■付き2〜3行）が確定していること

## Phase 3: HTML生成

テンプレートの部品カタログとデザイントークンでHTMLを組む。

完了条件: 自己完結の単一HTMLが生成されていること。

### Step 3-1: テンプレートの読み込み

`references/slide-template.html` を Read する。デザイントークン・部品カタログ・使用規則（冒頭コメント）を確認する。テンプレートは型別3ページ構成（段階比較・時系列・層構造）のため、Step 2-1 で選んだ図の型に対応するページの部品を優先して使う。論証テーブルの場合は Step 2-1 で選択した列構成に対応するテンプレートを Read する（課題分析型: `references/argument-table-template.html`、現状vs理想 対比型: `references/gap-analysis-table-template.html`）。`references/html-output-standards.md` を Read し、単一 HTML 成果物の合格基準（基本原則・図解・自己完結等）を確認する。

**入力**: `references/slide-template.html`
**完了**: トークンと部品の一覧、使用規則を確認したこと

### Step 3-2: 部品組み立てと本文執筆

テンプレートを骨格として Phase 2 の構造をHTMLに起こし、Write で書き出す。厳守事項:

- 先頭に `<meta charset="utf-8">` を置く
- `:root` のデザイントークンは変更しない。色は必ずトークン経由で使い、hex を本文に手書きしない（ツール識別色の追加のみ、トークン定義への追記として許可）
- 図はすべて inline SVG。外部リソース（CDN・Webフォント・外部画像・fetch）は一切参照しない
- 実在ツール・製品は実物ロゴを想起させる簡略SVGマーク+名称で描く。抽象図形の発明は禁止
- 変化の引き金（上限到達・締切・承認等）はイベント線+バッジの構造部品で描く
- 「尽きた・満杯・ゼロ」等の状態はメーター100%+警告色で極端に描く
- 複数ページの場合はテンプレートのページ送り部品を使う（1ページなら削除）
- スライド寸法は 1280×720px 固定。`.slide` 内に `clamp()` / `vw` / `vh` を使わない（`transform: scale()` が全体スケーリングを担う）
- スケーリング用 `<script>` は常時必須。1ページ構成でも削除しない（削除可はページ送りスクリプトのみ）
- メタデータフッター（`.slide-meta`）は区切り線と下端余白のみを提供する。内容は入れない
- 段階・成熟度・対比は stage-card 部品（上辺色: 警告=critical / 中間=partial / 推奨=good）で描き、利点・欠点は status-row（✓/✕）で列挙する
- ディレクトリ・ファイル構成は tree 部品（等幅フォント+tree-tag）で描く。パス文字列の箇条書きで代替しない
- 層・積層の説明は layer-band 部品（全幅帯+番号チップ）で描く
- 見出し・帯テキスト・キャプションの文節境界に `<wbr>` を挿入する。基準は `references/wbr-insertion-rules.md` を Read して適用する

**論証テーブルの場合の追加厳守事項**:
- リード帯はネイビー地（`--ink`）に白文字で、主張を1文で言い切る
- ヘッドメッセージは■マーク付き2〜3行で、事実→必要性を断定する
- テーブルヘッダーはネイビー地（`--ink`）に白文字
- すべての打ち手に実行主体（誰がやるか）を明記する
- 出典行をテーブル下部に1行で記載する（社内データなら「出典：◯◯部調べ」の形）
- 数字がない箇所は赤字（`--critical`）で「※要確認」と明記する

**入力**: Step 3-1 のテンプレート、Phase 2 の構造設計

**生成直後の語彙チェック**（Phase 4 到達前でも語彙確認を求められた場合の正規手順）:
- `references/business-content-standards.md` の「語彙-読み手整合」: 想定読み手に対し説明なしの専門用語がないか
- `references/customer-output-checklist.md`: 内部スキル名（`Skill()`）・内部パス（`.claude/rules/`）・内部管理語彙が表示テキストに出ていないか
- `references/prh.yml`: 辞書登録済み語の置き換え漏れがないか
この 3 本を照合する。prh 辞書のみの走査で完了扱いにしない。

**完了**: 単一HTMLファイルが書き出されていること

## Phase 4: レンダリング検証

実際に描画して観点レビュー表の全行を照合する。コードレビューだけでの合格判定を禁止する（文字切れ・重なりは実描画でしか発見できない）。

完了条件: document-reviewer による独立判定で観点レビュー表の全行が PASS になっていること。検証用スクリーンショットが `evidence/検証用スクリーンショット.png` に保持されていること。

### Step 4-1: 実描画とスクリーンショット

- `scripts/check-wbr-quality.sh` で `<wbr>` の品質を機械検証する（FAIL なら修正して再実行）

1280×720px のフルサイズスクリーンショットを取得し、`<スライドキー>/evidence/検証用スクリーンショット.png` に保持する（削除禁止。commit 時に hook が存在・寸法を検査する）。

取得手段の優先順:
1. `node scripts/build-thumbs.mjs --evidence`（ヘッドレス Chromium 直接起動。MCP 非依存で最も安定）
2. Playwright MCP で navigate → resize(1280,720) → screenshot（MCP が利用可能な場合）
3. 一時 HTTP サーバ + 別のブラウザツール

Playwright MCP のブラウザロック等でフルサイズ取得に失敗した場合は、サムネイル（640×360）での代替を禁止する。手段 1 または 3 にフォールバックし、いずれでもフルサイズ取得できない場合は NG として報告して停止する（「取得できなかったが PASS」は構造的に不可能）。

**使用ツール**: Bash（一時HTTPサーバの起動・停止）、Playwright 等のブラウザツール（スクリーンショット取得）
**入力**: Phase 3 で生成したHTML
**完了**: 1280×720px（Playwright ビューポート幅 1280px = scale 1）の実描画スクリーンショットを取得したこと

### Step 4-2: 観点レビュー表の全行照合

Phase 4 の PASS/NG 判定は生成者自身の自己採点を禁止する。`Skill("reviewing-against-rules")` を起動し、`document-reviewer` に証跡スクリーンショット（`evidence/検証用スクリーンショット.png`）を渡して独立判定させる。document-reviewer は Read でマルチモーダルに PNG を読めるため、Playwright は不要。

`references/slide-review-checklist.md` と `references/business-content-standards.md` を reviewing-against-rules の委任プロンプトに含める。出力前に `references/customer-output-checklist.md` の自問チェックも実施する。

検査記録の各「目視必須」観点の備考欄に、判定根拠のスクリーンショットファイルパスを引用する（パス引用なき PASS は無効）。

**入力**: Step 4-1 の証跡スクリーンショット（evidence/ に保持済み）、観点レビュー表
**完了**: document-reviewer による独立判定で全観点キーの PASS / NG が記録されていること

### Step 4-3: NG修正と再検証

NG があれば原因を直し、Step 4-1 から再実行する。ループ設計（後述）の上限・停止条件に従う。チェック結果の記録なしに「完了」と報告することを禁止する。

**使用ツール**: Edit（NG 原因箇所の修正）
**入力**: Step 4-2 の NG 一覧
**完了**: 全行 PASS、またはループ停止条件に到達し状況を報告したこと

## Phase 5: 保存・蓄積簿更新

完了条件: 汎用スライドは出力先ディレクトリへの保存と蓄積簿登録が完了していること。案件固有スライドは案件用ディレクトリへの保存が完了していること（蓄積簿登録は対象外）。

### Step 5-1: スライドの保存

顧客名・案件情報を含む案件固有スライドは案件用ディレクトリ配下（例: `clients/<案件名>/レベル目標_<課題キー>_<YYYYMMDD>.html`）として Write で保存する。含まない汎用スライドは出力先ディレクトリ内の `<スライドキー>/解説スライド.html` として Write（または Bash の cp）で保存する。ファイル名はそれぞれの規定形式に固定する。汎用スライドはこのPC固有の絶対パス・顧客固有の情報が含まれていないことを確認する。

**入力**: Phase 4 で PASS したHTML、案件固有か汎用かの判別
**完了**: 案件用ディレクトリまたは出力先ディレクトリの規定パスに保存されていること

### Step 5-2: 蓄積簿への登録

汎用スライドのみ出力先ディレクトリの蓄積簿にスライド一覧表として Edit で1行追記する（スライドキー / 説明対象 / 提示する課題の例 / 相対リンク / 登録日）。リンク先の実在を Bash の ls で確認する。案件固有スライドは顧客固有情報を出力先ディレクトリに置かないため蓄積簿登録をしない。

`references/text-dictionary.md` を Read し、語彙の設計原則を確認する。textlint 環境があれば自動検査を実行し、なければ以下の 3 本を参照して手動で語彙を確認する（prh 辞書のみの走査で完了扱いにしない）:
- `references/prh.yml`: 辞書登録済み語の置き換え漏れ
- `references/business-content-standards.md` の「語彙-読み手整合」: 想定読み手に対し説明なしの専門用語
- `references/customer-output-checklist.md`: 内部管理語彙の混入

**入力**: Step 5-1 の保存パス、Phase 1 の入力情報
**完了**: 汎用スライドは蓄積簿に1行追記されリンク先が実在すること。案件固有スライドは蓄積簿登録をしないこと

### Step 5-3: 検査記録の保存

汎用スライドは出力先ディレクトリ内の `<スライドキー>/検査記録.md` に保存する。案件固有スライドは案件用ディレクトリ内の `検査記録_レベル目標_<課題キー>_<YYYYMMDD>.md` に保存する。内容: 観点レビュー表の照合結果（全行の判定）・反復回数と各回の修正内容・**タスク限定の観点**（このスライド固有の検査事項）を記録する。例: リミットの因果条件・営業日のみの範囲制約・ツール識別マークの定義。タスク限定の観点は次回このスライドを更新する時の再検査項目になる。

**使用ツール**: Write（検査記録の書き出し）
**入力**: Phase 4 の照合記録
**完了**: 検査記録.mdが案件用ディレクトリまたは出力先ディレクトリの規定パスに保存されていること

## Phase 6: 完了報告と観点の育成

完了条件: 検査結果一覧を添えた報告を提示し、新規指摘の昇格判定を終えていること。

### Step 6-1: 検査結果付きの報告

保存パス・蓄積簿の登録内容（案件固有スライドは登録対象外）・観点レビュー表の照合結果（全行の PASS 一覧）・ヒアリングの仮置き一覧（あれば）を添えて報告する。（ツール不使用・テキスト報告のみ）

**入力**: Phase 4 の照合記録、Phase 5 の保存結果
**完了**: 報告をユーザーに提示したこと

### Step 6-2: 観点の共通化判定（レビュー表を育てる）

生成〜報告で受けたレビュー指摘・修正で言語化された原則を列挙し、昇格基準3つを満たすものを `references/slide-review-checklist.md` に Edit で追記する。

1. 別トピックのスライドでも再発しうる（タスク非依存）
2. 意味語キー + 検査内容 + NG例の形式で書ける
3. 実描画またはファイル検査（grep 等）で機械的・目視的に検査可能

満たさない指摘は Step 5-3 の検査記録.md のタスク限定観点として保持する（捨てない）。追記・保持の判定結果を報告に含める。レビュー表の既存観点の削除は禁止（検査の弱体化になるため。表現の具体化・NG例の追加は可）。

**入力**: 本セッションのレビュー指摘一覧、`references/slide-review-checklist.md`
**完了**: 新規指摘ごとの昇格/保持の判定結果を報告に含めたこと

## ループ設計

| 要素 | 内容 |
|---|---|
| 反復条件 | Phase 1: 充足ゴール3判定に不可があれば追加質問する ／ Phase 4: NG 観点が1つでもあれば修正して Step 4-1 から再検証する |
| 上限回数 | Phase 1: 最大3ラウンド ／ Phase 4: 最大5回 |
| 停止条件 | 収束停止: 充足ゴール3判定可（Phase 1）・全行 PASS（Phase 4）／ リソース上限: 各上限到達（Phase 1 は仮置き明示で続行）／ 発散検知: 同一観点の NG が2回連続 |

上限・発散で停止した場合は残 NG と原因をユーザーに報告し、判断を仰ぐ。

## 予想を裏切る挙動

- ヒアリングを省略して生成に入らない。原型スライドでは仕組みの因果（開始から5時間でリセット）の確認が浅く、「5時間使ってから切り替える」という矛盾した図を一度作った
- レンダリング検証を省略しない。原型スライドでも、コード上は正しく見えて実描画で「Cursor」の文字が帯からはみ出す欠陥が見つかった
- 例示数値の検算を省略しない。「5時間リミットの説明なのに5時間使ってから切り替える」図は、見た目が自然なまま仕組みと矛盾する
- 図の型より先に色やレイアウトを選ばない。構造設計（Phase 2）を飛ばすと「左から右に変わる」以上の情報がない図になる
- ダークモード対応は不要。顧客提示前提のビジネス配色（白面+ネイビー系）に固定する

## 機械強制

| timing | スクリプト | 注入タグ | 挙動 |
|---|---|---|---|
| PreToolUse(Bash) | `scripts/check-slide-evidence.sh` | `[SLIDE-EVIDENCE-BLOCK]` | staged に `解説スライド.html` がある `git commit` で (a) `evidence/検証用スクリーンショット.png` 不在 (b) 寸法が 1280×720 でない (c) 同フォルダの `検査記録.md` が未 staged、または `evidence/検証用スクリーンショット` のパス引用か `document-reviewer` の判定記録を欠く のいずれかなら exit 2 で block。同セッション 3 回連続 block で自動解除 |
| Phase 4 直接実行 | `scripts/check-wbr-quality.sh <file>` | `[WBR-QUALITY-BLOCK]` | Phase 4 Step 4-1 で引数にファイルパスを渡して直接実行。禁則違反があれば exit 2。hook 登録はしない（スキル単体配布時の settings.json 依存を避けるため） |

## 違反検知時の手順

### `[SLIDE-EVIDENCE-BLOCK]` 受信

1. 証跡不在の場合: `node scripts/build-thumbs.mjs --evidence` で 1280×720 の検証用スクリーンショットを生成する
2. 寸法不一致の場合: サムネイル（640×360）を誤って配置していないか確認し、フルサイズで再生成する
3. 独立レビュー未実施の場合: `Skill("reviewing-against-rules")` を起動し、document-reviewer に証跡スクリーンショットを渡して独立判定させる
4. 上記を完了してから再度 `git commit` する

### `[WBR-QUALITY-BLOCK]` 受信

1. stderr に出力された検証キーと詳細を確認する
2. 該当ファイルの `<wbr>` を `references/wbr-insertion-rules.md` の基準に沿って修正する
3. `git add` で修正をステージし直してから再度 `git commit` する
