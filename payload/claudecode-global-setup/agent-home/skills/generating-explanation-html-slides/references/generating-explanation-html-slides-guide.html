§1概要

generating-explanation-html-slides は、与えたトピックを横1枚（16:9）のスライド風解説HTMLに変換する実行スキル。 図はすべて inline SVG で自作し、外部リソース（CDN・Webフォント・外部画像・fetch）は一切参照しない自己完結型の単一HTMLとして出力する。 観点レビュー表による検査に全行 PASS したものだけを蓄積簿 docs/10_解説スライド集/ に登録する。

スキルの位置づけ

項目値
スキル名generating-explanation-html-slides
Typetransform（入力トピックを成果物HTMLへ変換）
TRIGGER「解説スライドを作って」「説明用HTMLにして」「スライドHTML」「図解して1枚にまとめて」
SKIP解決事例の登録（→consult-case）、リポジトリ解析レポート（→consult-analyze）、レベル合意HTMLそのものの生成（→consult-level-agree）
consult-level-agree からの引き継ぎ consult-level-agree から起動された場合、レベル合意HTML・業務棚卸し表・軌道記述（合意済みのL1〜L5説明文）を Read し、 ヒアリング5カテゴリの初期値として使う。Phase 1 のユーザー確認は初期値の修正要否のみに短縮できる。

§2出力

実行が完了すると、以下が確立される。

解説スライドHTML
横1枚（16:9）の自己完結型HTML。汎用スライドは docs/10_解説スライド集/<スライドキー>/解説スライド.html、案件固有スライドは clients/<案件名>/レベル目標_<課題キー>_<YYYYMMDD>.html に保存される。観点レビュー表の全行 PASS が保存の前提条件。
蓄積簿への登録行（汎用スライドのみ）
docs/10_解説スライド集/解説スライド集.md のスライド一覧表に、スライドキー・説明対象・提示する課題の例・相対リンク・登録日が1行追記される。案件固有スライドは顧客固有情報を含むため登録対象外。
検査記録
観点レビュー表の照合結果（全行の判定）・反復回数と各回の修正内容・タスク限定の観点（このスライド固有の検査事項）を記載したMarkdown。汎用スライドは docs/10_解説スライド集/<スライドキー>/検査記録.md、案件固有スライドは clients/<案件名>/検査記録_レベル目標_<課題キー>_<YYYYMMDD>.md に保存される。

§4ヒアリング5カテゴリ

Phase 1 で最も重要な工程。5カテゴリすべてに回答（または仮置きの明示）が揃うまで次の Phase に進まない。ヒアリング不足のまま生成すると「正しく見えるが仕組みと矛盾する」欠陥（因果矛盾・数値矛盾）の温床になる。

カテゴリ確認する内容図への反映先
説明対象仕組みの事実。因果関係・数値条件・例外（例: 枠は開始から5時間で回復する）例示数値の検算・イベント線の位置
読み手誰に見せるか（経営層/現場/技術者）・前提知識用語レベル・補足の量
結論読み手に残したい1文・読後に取ってほしい行動タイトル・サブタイトル・図の型
提示文脈どの課題への提示か・利用場面蓄積簿の「提示する課題の例」列
制約ページ数（既定1）・含める/除く内容・業務文脈の範囲（営業日のみ等）軸の範囲・パネル構成

充足ゴール3判定（Step 1-2）

  1. 図の型が選べるか（結論と説明対象の性質が分かっている）
  2. 例示数値の検算ができるか（仕組みの因果・数値条件が引き出せている）
  3. 読み手に合う語彙レベルが決められるか

ヒアリングは最大3ラウンド。埋まらない項目は推測で埋めず「仮置き」と明示して先へ進み、Phase 6 の報告に仮置き一覧を含める。

§5Phase構成と動作フロー

Phase別の完了条件

Phase完了条件
Phase 1ヒアリング5カテゴリが充足（または仮置き明示）し、充足ゴール3判定が可、スライドキーが確定していること
Phase 2図の型・軸・パネル分割・番号・例示数値が言語化されていること
Phase 3自己完結の単一HTMLが生成されていること
Phase 4観点レビュー表の全行が PASS になっていること
Phase 5汎用スライドはスライド・蓄積簿・検査記録の3点、案件固有スライドはスライド・検査記録の2点が保存されていること
Phase 6検査結果付き報告を提示し、新規指摘の昇格/保持判定を終えていること

§6HTML生成の厳守事項

Phase 3（Step 3-2）でテンプレートを骨格に本文を執筆する際の厳守事項。

項目規則
文字コード先頭に <meta charset="utf-8"> を置く
デザイントークン:root は変更しない。色は必ずトークン経由（ツール識別色の追加のみ、トークン定義への追記として許可）
図の描画すべて inline SVG。外部リソース（CDN・Webフォント・外部画像・fetch）は一切参照しない
実在ツール・製品実物ロゴを想起させる簡略SVGマーク+名称で描く。抽象図形の発明は禁止
変化の引き金上限到達・締切・承認等はイベント線+バッジの構造部品で描く
極端な状態「尽きた・満杯・ゼロ」等はメーター100%+警告色で描く
複数ページテンプレートのページ送り部品を使う（1ページなら削除）
Gotcha コードレビューだけでの合格判定を禁止する。文字切れ・重なりは実描画（Phase 4）でしか発見できない。原型スライドでも、コード上は正しく見えて実描画で文字が帯からはみ出す欠陥が見つかった実例がある。

§7観点レビュー表と検証

Phase 4 で references/slide-review-checklist.md を Read し、全観点キーについて PASS / NG を記録する。表示検査の観点はスクリーンショットを見て判定し、特に「文字切れ-全数目視」は全テキスト要素を1つずつ確認する。

Step使用ツール内容
4-1 実描画とスクリーンショットBash（一時HTTPサーバの起動・停止）、Playwright 等のブラウザツール（スクリーンショット取得）1280px 幅で実描画を確認する
4-2 観点レビュー表の全行照合Read全観点キーの PASS / NG を記録。出力前に顧客向け成果物チェックも実施
4-3 NG修正と再検証Edit原因を直し Step 4-1 から再実行。ループ設計の上限・停止条件に従う

観点の育成（Step 6-2）

生成〜報告で受けたレビュー指摘・修正で言語化された原則のうち、次の昇格基準3つを満たすものを観点レビュー表に Edit で追記する。

  1. 別トピックのスライドでも再発しうる（タスク非依存）
  2. 意味語キー + 検査内容 + NG例の形式で書ける
  3. 実描画またはファイル検査（grep 等）で機械的・目視的に検査可能

満たさない指摘は検査記録.mdのタスク限定観点として保持する（捨てない）。観点レビュー表の既存観点の削除は禁止（検査の弱体化になるため）。

§8ループ設計

要素内容
反復条件Phase 1: 充足ゴール3判定に不可があれば追加質問する ／ Phase 4: NG 観点が1つでもあれば修正して Step 4-1 から再検証する
上限回数Phase 1: 最大3ラウンド ／ Phase 4: 最大5回
停止条件収束停止: 充足ゴール3判定可（Phase 1）・全行 PASS（Phase 4）／ リソース上限: 各上限到達（Phase 1 は仮置き明示で続行）／ 発散検知: 同一観点の NG が2回連続

上限・発散で停止した場合は残 NG と原因をユーザーに報告し、判断を仰ぐ。

スキル / 規約役割分担

意味キー規約（review-checklist/meaningful-key-naming）スライドキー・観点キーの連番禁止・意味語キー命名の根拠
