§1概要

generating-explanation-html-slides は、与えたトピックを横1枚（16:9）のスライド風解説HTMLに変換する実行スキル。 図解型と論証テーブル型の2型があり、Step 1-1 でどちらを使うか選択する。 図はすべて inline SVG で自作し、外部リソース（CDN・Webフォント・外部画像・fetch）は一切参照しない自己完結型の単一HTMLとして出力する。 観点レビュー表による検査に全行 PASS したものだけを、プロジェクトに応じた出力先ディレクトリの蓄積簿（例: docs/解説スライド集/）に登録する。

スキルの位置づけ

項目値
スキル名generating-explanation-html-slides
Typetransform（入力トピックを成果物HTMLへ変換）
TRIGGER「解説スライドを作って」「説明用HTMLにして」「スライドHTML」「図解して1枚にまとめて」「論証テーブルで整理して」
SKIPリポジトリ解析レポート、データ分析レポート等の非スライド成果物
外部スキルからの引き継ぎ 外部スキルから引き継ぎ情報（業務棚卸し表・軌道記述等）が渡された場合、それらをヒアリング5カテゴリの初期値として使う。 この場合、Phase 1 のユーザー確認は初期値の修正要否のみに短縮できる。

実行が完了すると、以下が確立される。

解説スライドHTML
横1枚（16:9）の自己完結型HTML。汎用スライドは出力先ディレクトリ内 <スライドキー>/解説スライド.html（例: docs/解説スライド集/<スライドキー>/解説スライド.html）、案件固有スライドは clients/<案件名>/レベル目標_<課題キー>_<YYYYMMDD>.html に保存される。観点レビュー表の全行 PASS が保存の前提条件。
蓄積簿への登録行（汎用スライドのみ）
出力先ディレクトリの蓄積簿（例: docs/解説スライド集/解説スライド集.md）のスライド一覧表に、スライドキー・説明対象・提示する課題の例・相対リンク・登録日が1行追記される。案件固有スライドは顧客固有情報を含むため登録対象外。

§4スライド型とヒアリング5カテゴリ

Phase 1 で最も重要な工程。5カテゴリすべてに回答（または仮置きの明示）が揃うまで次の Phase に進まない。ヒアリング不足のまま生成すると「正しく見えるが仕組みと矛盾する」欠陥（因果矛盾・数値矛盾）の温床になる。

Step 1-1: スライド型の選択

ユーザーの依頼文からスライド型を判定する。ユーザーが型を明示している場合はその型で進める。明示がない場合は次の判定ロジックで自動選択し、結果を AskUserQuestion で確認する。

テーマの性質判定結果
時間経過・プロセス・サイクル・推移を含む図解
複数対象の比較・優劣図解
手順・分岐図解
成熟度・段階図解
課題の整理と対策の提示論証テーブル
論点を根拠付きで並列整理論証テーブル
上記に明確に当てはまらないAskUserQuestion で確認

AskUserQuestion の選択肢は (A) 図解スライド／(B) 論証テーブル／(C) AI が判定した型で進める、の3択。

Step 1-2: ヒアリング5カテゴリの整理

カテゴリ確認する内容図への反映先
説明対象仕組みの事実。因果関係・数値条件・例外（例: 枠は開始から5時間で回復する）例示数値の検算・イベント線の位置
読み手誰に見せるか（経営層/現場/技術者）・前提知識用語レベル・補足の量
結論読み手に残したい1文・読後に取ってほしい行動タイトル・サブタイトル・図の型
提示文脈どの課題への提示か・利用場面蓄積簿の「提示する課題の例」列
制約ページ数（既定1）・含める/除く内容・業務文脈の範囲（営業日のみ等）軸の範囲・パネル構成

充足ゴール3判定（Step 1-3）

  1. 図の型が選べるか（結論と説明対象の性質が分かっている）
  2. 例示数値の検算ができるか（仕組みの因果・数値条件が引き出せている）
  3. 読み手に合う語彙レベルが決められるか

ヒアリングは最大3ラウンド。埋まらない項目は推測で埋めず「仮置き」と明示して先へ進み、Phase 6 の報告に仮置き一覧を含める。

§6HTML生成の厳守事項

Phase 3（Step 3-2）でテンプレートを骨格に本文を執筆する際の厳守事項。

項目規則
文字コード先頭に <meta charset="utf-8"> を置く
デザイントークン:root は変更しない。色は必ずトークン経由で使い、hex を本文に手書きしない（ツール識別色の追加のみ、トークン定義への追記として許可）
図の描画すべて inline SVG。外部リソース（CDN・Webフォント・外部画像・fetch）は一切参照しない
実在ツール・製品実物ロゴを想起させる簡略SVGマーク+名称で描く。抽象図形の発明は禁止
変化の引き金上限到達・締切・承認等はイベント線+バッジの構造部品で描く
極端な状態「尽きた・満杯・ゼロ」等はメーター100%+警告色で描く
複数ページテンプレートのページ送り部品を使う（1ページなら削除）
スライド寸法1280×720px 固定。.slide 内に clamp() / vw / vh を使わない（transform: scale() が全体スケーリングを担う）
スケーリング用 script常時必須。1ページ構成でも削除しない（削除可はページ送りスクリプトのみ）
メタデータフッター.slide-meta は区切り線と下端余白のみを提供する。内容は入れない
段階・対比stage-card 部品（上辺色 critical/partial/good）+ status-row（✓/✕）で描く
ディレクトリ構成tree 部品（等幅+tree-tag）で描く。箇条書きで代替しない
層・積層layer-band 部品（全幅帯+番号チップ）で描く

論証テーブル型の追加厳守事項

項目規則
リード帯ネイビー地（--ink）に白文字で、主張を1文で言い切る
ヘッドメッセージ■マーク付き2〜3行で、事実→必要性を断定する
テーブルヘッダーネイビー地（--ink）に白文字
実行主体すべての打ち手に実行主体（誰がやるか）を明記する
出典行テーブル下部に1行で記載する（社内データなら「出典：◯◯部調べ」の形）
※要確認数字がない箇所は赤字（--critical）で「※要確認」と明記する
Gotcha コードレビューだけでの合格判定を禁止する。文字切れ・重なりは実描画（Phase 4）でしか発見できない。原型スライドでも、コード上は正しく見えて実描画で文字が帯からはみ出す欠陥が見つかった実例がある。

references/business-content-standards.md の6カテゴリを要約する。顧客・社外へ提示する目的の資料にのみ適用され（適用ゲート）、該当しない資料は全観点「対象外」として即 PASS になる。

顧客提示前チェック（第4層）の要旨

顧客に渡す・見せる出力（HTML・Excel・依頼文書・報告表）を出す前の最終自問。機械検査では拾いきれない「内部運用の痕跡」（内部スキル名・固有パス・内部管理語彙など）が表示テキストに紛れていないかを人が最後に確認する層で、1つでも該当したら生成元の Phase に戻って修正し再チェックする。

  • consult- で始まる内部スキル名・特定PC固有のパスを含めていないか
  • 根拠のない数値・頻度・時刻を断定していないか
  • 作業用の内部分類・記号見出し（①②③・判定ラベル等）をそのまま出していないか
  • 内部の記録ファイル名（拡張子付き）・内部の管理語彙（正本・転記等）が表示テキストに出ていないか
  • パネル見出し・セクション見出しに文外参照の指示語（「このレベル」等）を使っていないか

正本は references/customer-output-checklist.md。案件固有スライドは顧客に渡る成果物のため必須、汎用スライドにも同チェックを適用する。

観点の2層構造と育成サイクル（Step 6-2）

観点は2層で管理する。references/slide-review-checklist.md が全スライド共通の観点の正本（第1層）、各スライドの検査記録.md がタスク限定の観点を保持する（第2層）。

育成サイクルは次の4手順で回す。

  1. 生成したスライドを観点レビュー表で照合し、ユーザーレビューを受ける
  2. 受けた指摘・修正ループで見つけた欠陥を原則として言語化する
  3. 昇格判定: 次の3基準をすべて満たすものだけを観点レビュー表に Edit で追記する
  4. 満たさない指摘は検査記録.md にタスク限定の観点として保持する（次回このスライドを更新する時の再検査項目にする）

昇格基準3つ:

  1. 別トピックのスライドでも再発しうる（タスク非依存）
  2. 意味語キー + 検査内容 + NG例の形式で書ける
  3. 実描画またはファイル検査（grep 等）で機械的・目視的に検査可能

満たさない指摘は検査記録.mdのタスク限定観点として保持する（捨てない）。観点レビュー表の既存観点の削除は禁止（検査の弱体化になるため。表現の具体化・NG例の追加は可）。

§8ループ設計

要素内容
反復条件Phase 1: 充足ゴール3判定に不可があれば追加質問する ／ Phase 4: NG 観点が1つでもあれば修正して Step 4-1 から再検証する
上限回数Phase 1: 最大3ラウンド ／ Phase 4: 最大5回
停止条件収束停止: 充足ゴール3判定可（Phase 1）・全行 PASS（Phase 4）／ リソース上限: 各上限到達（Phase 1 は仮置き明示で続行）／ 発散検知: 同一観点の NG が2回連続

上限・発散で停止した場合は残 NG と原因をユーザーに報告し、判断を仰ぐ。
