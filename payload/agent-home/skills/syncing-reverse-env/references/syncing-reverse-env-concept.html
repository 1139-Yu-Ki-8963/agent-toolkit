§1目的と全体像

リバースエンジニアリング設計の中心課題は「設計書が完全であることの証明」にある。本システムはこれを、設計書だけを見て再構築したリバースコード環境が、オリジナルコード環境とポート番号以外で完全に同じ動作をすることで証明する。

── リバース設計検証システムの三角関係 ──────────────────────────

                リバース対象システム
                        │
        ┌───────────────┴────────────────────┐
        ▼                                    ▼
┌────────────────────┐   リバース設計   ┌──────────────────┐
│ オリジナルコード環境    │ ──────────────▶ │  設計書一式        │
│ リバース元コード        │  コード → 文書   │  (reverse-docs    │
│ ＝常に「正」           │                 │   テンプレート準拠)  │
└────────────────────┘                  └────────┬─────────┘
        ▲                                        │ 設計書「だけ」を見て
        │                                        │ コードを再構築
        │       ポート以外は完全一致か？            ▼ [リバースを行うスキル]
        │     ┌─────────────────────┐   ┌────────────────────┐
        └─────┤ 本スキルが worktree の  ├───│ リバースコード環境     │
              │ 用意・検証・整列・       │   │ 設計書由来コード       │
              │ 基準コミット確立を担う   │   └────────────────────┘
              └─────────────────────┘
   ※ リバース設計（コード → 文書）もリバースを行うスキルが担う
   ※ worktree 名は original-code-<system> / reverse-code-<system>（§4）
      どちらも本スキルが用意・命名する（ADR-4）

責務境界（本スキル vs リバースを行うスキル）

担当責務
本スキル
syncing-reverse-env		 ⓪ オリジナルコード環境 / リバースコード環境の用意と名前管理（初回のみ作成・命名規則・固定ポート割当・冪等な再発見・ユーザー依頼時のみ削除）
① オリジナルコード環境とリバースコード環境の同一性検証（静的 diff + Playwright 動的比較）
② 差分があればリバースコード環境をオリジナルコード環境へ整列
③ 一致証明後のコミット＝基準コミットの確立と記録
④ 基準コミットへの復帰（sync の reset-first オプション）
リバースを行うスキル
（設計書修正ループ・将来作成）		 設計書の不備を修正し、リバースコード環境の中身（コード）を再構築するループ全体の運転。worktree という容器の用意は本スキルに任せ、返却された判定・ヒントを修正ループの入力にする

オリジナルコード環境が常に「正」。リバースコード環境と設計書のどちらを直すかの判断はリバースを行うスキルが持ち、本スキルは環境の用意・判定・整列に徹する。

リバース以外の使い方（可能性）

本スキルはリバース検証のために設計する。ただし「ポート以外同一の 2 環境を永続 worktree で維持し、静的 diff + Playwright 4 層で突合する」という仕組みの性質上、リバース以外にも次のような使い方ができると考えられる。

応用オリジナルコード環境（正）リバースコード環境（対象）使い方
PR レビュー実機確認 main ブランチ PR ブランチ 並行起動して挙動差分・ビジュアル差分を実機で突合する。「意図した変更以外に差分がないか」を機械判定でき、PR レビュースキルの検証層として連携できる
リファクタ安全性証明 リファクタ前 リファクタ後 「挙動が変わらないこと」を L1〜L4 の全層一致で証明する。この場合は実差分ゼロではなく動的比較の全層一致が合格条件になる（コードは変わるが挙動は変わらない）
UI ビジュアルリグレッション 変更前 変更後 L3（スクリーンショット画素比較）を単独利用し、意図しない見た目の変化を検出する。デザイン変更の影響範囲確認に使う

応用時は「何を一致とみなすか」（静的一致まで求めるか、動的一致のみか）が用途ごとに変わる。合格条件のプリセットを mode とは別の軸で持たせることが、スキル本体設計時の検討事項になる。

§2登場物と用語

オリジナルコード環境
リバース対象のオリジナルコードを保持する git worktree。正式名は original-code-<system>（§4 の命名規則で導出）。本スキルが設計書 frontmatter の source_repo / source_ref から用意する読み取り専用コピーで、常に「正」。検証中にオリジナルコード環境の内容を変更することはない。
リバースコード環境
reverse-docs 準拠の設計書だけを見て再構築したコードの git worktree。正式名は reverse-code-<system>。容器（worktree・ブランチ・固定ポート）は本スキルが用意し、中身のコードはリバースを行うスキルが書く。オリジナルコード環境とは別ブランチで、コミット履歴は「設計書からの再構築の記録」として独立に保つ。
worktree のライフサイクル（永続・使い回し）
オリジナルコード環境 / リバースコード環境は初回実行時に一度だけ作成し、以後の検証ではすべて同じものを使い回す。検証のたびに作り直すことはなく、削除はユーザーが明示的に依頼した時（teardown）のみ。ポート・リバースコード環境のブランチ履歴を環境自身が保持し続ける、長寿命の資産である（ADR-5）。
基準コミット（baseline commit）
「オリジナルコード環境とリバースコード環境がポート以外で完全一致する」ことを証明できた時点のリバースコード環境側コミット。設計書の完全性が証明された座標であり、リバースコード環境はいつでもここへ git reset --hard reverse-baseline/<system> で復帰できる。コミットの実体は worktree ではなくリポジトリ本体に保存されるため、worktree を破棄しても消えない。
設計書 frontmatter
画面基本設計書（reverse-docs テンプレート）の YAML frontmatter。本スキル用に追加するのは source_repo / source_ref（リバース元の同一性＝設計書の出自情報）の 2 フィールドだけ。基準コミットの記録は設計書には書かず、基準タグ（下記）が担う（ADR-1）。
基準タグ（reverse-baseline/<system>）
基準コミットに打つ annotated git tag。基準の唯一の記録。リポジトリ本体に保存されるため worktree の破棄と無関係に永続し、指したコミットを gc からも保護する。タグメッセージに検証日と判定サマリを記録する。
固定ポート
両環境の稼働ポート。プロジェクトに依存しない全体共通の固定値で、config.yml を正とする（オリジナル=9100 番台 / リバース=9110 番台。§4 の割当表参照）。両環境は必ず異なる帯域を持ち、これが「唯一許容される差分＝ポート番号」の根拠になる。
config.yml（スキルフォルダ直下）
プロジェクトによって変えたくなる可能性がある値（ポート番号・diff 除外リスト・L3 閾値・max-loop 既定値）の置き場。defaults ＋ projects 別上書きの 2 層構造（§4・ADR-6）。
環境命名規則
オリジナルコード環境 / リバースコード環境の worktree 名・ブランチ名を設計書から機械的に導出する規則（§4）。本スキルはこの規則によって、環境を「与えられる」のではなく、再実行時に自ら同じ環境を再発見する。

§3アクターとユースケース

本スキルの利用者は「リバースを行うスキル」と「人間」の 2 種。mode は運用の 3 局面 —「初回の環境整備」「それ以降の検証サイクル」「検証終了時の片付け」— に対応する 3 つだけに絞る。

アクター意図入口と mode期待する出力
リバースを行うスキル
（初回の環境整備）		 検証システムの立ち上げ時に、書き込み先となるリバースコード環境と比較相手のオリジナルコード環境を整備したい Skill ツール・args 全量指定
setup		 両環境のパス・ブランチ名・確定ポート一覧（§4 の割当表）。比較は実行しない
リバースを行うスキル
（検証サイクル）		 設計書を直してリバースコード環境を再構築した直後、「今回の設計書で完全再現できたか」を機械判定し、一致すれば基準を確立したい Skill ツール・args 全量指定
sync（既定）		 構造化された PASS / FAIL ＋ 差分サマリ ＋ 設計書章マップへの対応ヒント（§10）。PASS 時は基準タグを更新
人間（状態確認） 今、2 つの環境がどれくらいズレているかを見たいだけ。環境は変更したくない 直接起動
sync + dry-run		 差分レポート（静的 diff 一覧 ＋ スクリーンショット証跡）。整列・タグ更新なし＝完全無副作用
人間（片付け） 検証が終わったので環境一式を片付けたい 直接起動（明示依頼のみ）
teardown		 固定ポート帯（9100 / 9110 番台）の一括 kill ＋ 両 worktree の削除報告。基準タグとブランチは残す（基準コミットは消えない）。本スキルが自発的に削除することはない

基準タグへの復帰（git reset --hard reverse-baseline/<system>）は独立モードにせず、sync のオプション reset-first として提供する。検証サイクルの冒頭でリバースコード環境を基準状態へ戻してから再構築・検証に入る使い方を想定する。

呼び出し関係の明記: SKILL.md 規約に従い、呼ばれる側である本スキルの frontmatter SKIP 行に「設計書の修正・リバースコード環境の再構築は対象外（→リバースを行うスキル）」と記載する。

§4起動契約とプリフライト

入力（起動契約）

区分引数内容
必須 design-doc対象の画面基本設計書パス。唯一の必須入力＝検証の主キー。リバース元の同一性（source_repo / source_ref）はこの frontmatter が保持する。基準コミットは基準タグが保持する（§5）
任意 modesetup（初回の環境整備のみ）/ sync（検証 → 整列 → 基準確立。既定）/ teardown（検証終了時の片付け。ユーザーの明示依頼のみ）の 3 つだけ
dry-runsync 時のみ: 整列・基準タグ更新を行わず、差分レポートだけを返す（完全無副作用）
reset-firstsync 時のみ: 開始前にリバースコード環境を基準タグへ git reset --hard で復帰させてから検証サイクルに入る
routes動的比較で巡回する画面パスのリスト。省略時は設計書 frontmatter の route から導出
max-loop整列 → 再検証の収束ループ上限。既定は config.yml の値（3）

アクター別の受け渡し

  • リバースを行うスキルから: args で全量指定する。対話ゼロで完走できることを契約とする（AskUserQuestion を発行しない）。
  • 人間から: design-doc だけ渡せば起動できる。
  • 共通: worktree のパスを引数で受け取ることはない。オリジナルコード環境 / リバースコード環境は本スキルが命名規則から導出・確保する（下記・ADR-4）。

環境確保と命名規則（本スキルの固有責務）

worktree の用意・命名・再発見はすべて本スキルが担う。作成は初回実行時の一度だけで、以後の検証は同じ worktree を再発見して使い回す。削除はユーザーが明示的に依頼した時（teardown）のみ。どの mode でも Phase 1 で必ず実行し、冪等に振る舞う。

対象命名・確保の規則
システム識別子 <system>設計書 frontmatter の doc_id（またはリポジトリ名）から機械的に導出する。再起動しても同じ名前に解決されることが冪等性の根拠
オリジナルコード環境worktree 名 original-code-<system>。source_ref を固定 checkout した読み取り専用コピー。汚れ・ズレを検出したら内容だけを source_ref へ復元する（worktree 自体・名前・ポートは保持するため、ADR-5 の永続性と矛盾しない）
リバースコード環境worktree 名 reverse-code-<system>、ブランチ feature/reverse-code-<system>（ブランチ命名規約準拠）。基準コミットと再構築履歴を持つ本体
基準タグannotated tag reverse-baseline/<system>。基準コミットの唯一の記録（ADR-1）。worktree・ブランチの寿命と独立にリポジトリ本体で永続する
ポートconfig.yml の固定ベース（オリジナル=9100 / リバース=9110）から §4 の割当表のとおり確定する。プロジェクトに依存しない
作成の実務worktree の物理作成・削除は parallel-dev-worktree を Skill ツールで呼び出して委譲する。名前とポートの決定は本スキルが行う
Gotcha: worktree は全体で 5 本までの上限があり、本スキルは 2 本を長期に占有する（永続・使い回し設計のため）。またポートは全体共通の固定値（9100 / 9110 番台）のため、リバース検証環境は同時に 1 組だけ稼働できる。

設定ファイル（config.yml）

プロジェクトによって変えたくなる可能性がある値は、スキルフォルダ直下の config.yml に集約し、スキルは実行時にそこから参照する（ADR-6）。SKILL.md 本文には値をベタ書きしない。

── config.yml の構造（defaults + プロジェクト別上書き）─────────

  defaults:
    ports:
      original_code_base: 9100  # オリジナルコード環境のベースポート
      reverse_code_base: 9110   # リバースコード環境のベースポート
    launch: "npm run dev -- --port {port}"  # 起動コマンドの既定形
    l3_threshold_percent: 0.1   # L3 視覚比較の許容差分画素率
    max_loop: 3                 # 収束ループ上限の既定値
    tag_namespace: reverse-baseline
    diff_exclude:               # §6 静的比較の除外リスト
      - .git/
      - node_modules/
      - dist/
      - .next/
      - "*.lock"
      - ".env*"                 # ポート設定を含む環境ファイル

  projects:                     # プロジェクト単位の上書き（必要時のみ）
    "<project-name>":
      ports: { original_code_base: 9120, reverse_code_base: 9130 }
      launch:                   # 起動手順は唯一のアプリ依存要素
        backend: "<backend 起動コマンド> --port {port}"
        frontend: "<frontend 起動コマンド> --port {port}"

値の階層は 3 層で役割分担する: 普遍のルール＝SKILL.md / プロジェクト単位の可変値（ポート上書き・起動手順）＝config.yml / 画面単位の可変値（routes・マスク対象セレクタ）＝設計書 frontmatter。アプリ毎に変わるのは起動手順だけで、それを config.yml に閉じ込めることで、ユーザーへの報告定型文（§10）は全アプリ共通の形になる。

ポート割当（確定値）

ポート番号はプロジェクトに依存しない全体共通の固定値とし、config.yml の ports を正とする。通常の開発サーバー帯域と衝突しない 9100 番台を割り当てる。

環境ベースサービス別ポート
オリジナルコード環境9100backend 9100 / frontend 9101 / docs 9102 / DB tools 9103（9104〜9109 は予約）
リバースコード環境9110backend 9110 / frontend 9111 / docs 9112 / DB tools 9113（9114〜9119 は予約）

全体共通の固定ポートのため、リバース検証環境は同時に 1 組だけ稼働できる。別の組を並行稼働させたい場合のみ、config.yml の projects: 上書きで別帯域（例: 9120 / 9130）を割り当てる。

プリフライト（go / no-go 判定）

本処理に入る前に以下を検証し、1 つでも満たさなければ NO-GO として停止・報告する。

#検証項目NO-GO 時の扱い
1設計書が存在し reverse-docs テンプレート準拠（type: screen-basic-design）で、source_repo / source_ref が解決できるNO-GO
2環境確保（上記の命名規則でオリジナルコード環境 / リバースコード環境を検出。無ければ作成）が完了しているworktree 総数上限（5 本）超過は NO-GO
3オリジナルコード環境が source_ref と一致し cleanズレ・汚れは内容を source_ref へ自動復元する（worktree 自体は保持）
4リバースコード環境の dirty 状態を記録sync では許容（整列で上書きされる）。dry-run 指定時は警告を提示
5固定ポート（9100 / 9110 番台）が未占有stale プロセスは kill して続行
6Playwright が利用可能、リバースコード環境側の依存インストール（node_modules 等）が完了NO-GO（ERROR 系。§9）
7reset-first 指定時のみ: 基準タグ reverse-baseline/<system> が存在NO-GO（基準未確立と報告）

§5全体フロー（初回 / 2 回目以降の分岐）

初回か再実行かの判定

基準タグ reverse-baseline/<system> の有無で判定する（git tag -l）。タグが存在する場合は参照先コミットの実在も確認する。「タグなし」「参照先不明（履歴書き換え等）」はどちらも初回扱いにフォールバックし、安全側に倒す。

── 分岐の全体図 ────────────────────────────────────────────

  Phase 0  起動契約解決（args 検証・config.yml 読込）
     │
  Phase 1  環境確保 + プリフライト（§4）── NO-GO ──▶ 停止・報告
     │ GO    worktree は初回のみ作成・以後は再発見して使い回す
     │
     ├─ mode=setup ────▶ 両 worktree のパス・ブランチ・確定ポートを
     │  （初回の整備）    返して終了（リバースを行うスキルはこの
     │                   リバースコード環境に再構築コードを書く）
     │
     ├─ mode=teardown ─▶ ポート一括 kill → worktree 削除 → 終了
     │  （検証終了時）   （ユーザーの明示依頼がある時だけ受け付ける。
     │                    基準タグとブランチは残す＝基準コミットは消えない）
     │
     └─ mode=sync（既定。2 回目以降の検証サイクル）
          │
          │  基準タグ reverse-baseline/<system> の有無を確認
          │  （git tag -l + 参照先確認。なし・参照先不明は初回扱い）
          │
          ├─ reset-first 指定 かつ タグあり
          │    → git reset --hard reverse-baseline/<system> で
          │      基準状態へ戻してから続行
          │
          ▼
        Phase 2〜7 の検証・整列サイクルへ
        （PASS 時: タグが無ければ新規に打ち、あれば -f で打ち直す。
          旧基準のハッシュは返却レポートに記録。
          dry-run 指定時: 整列・タグ更新なしの差分レポートのみ）

── 検証・整列の収束ループ（Phase 2〜7）──────────────────────

  Phase 2  静的比較（§6: diff + ポート正規化判定）
     │
  Phase 3  整列（§9。実差分あり かつ dry-run でない時のみ）
     │
  Phase 4  環境同一性チェック（§8）＋ 動的比較（§7: Playwright 4 層）
     │
  Phase 5  収束判定（dry-run 指定時は 1 周で確定・ループしない）
     │        ├─ FAIL（ループ上限内）──▶ Phase 2 へ戻る
     │        └─ FAIL（max-loop 到達）─▶ FAIL 確定・差し戻し（§10）
     │ PASS
  Phase 6  基準確立（dry-run 指定時はスキップ）
     │        grouping-commits でコミット（日本語 prefix 規約）
     │        → 基準コミットに annotated tag を打つ
     │           git tag -a reverse-baseline/<system>
     │           （メッセージに検証日と判定サマリを記録）
     │
  Phase 7  結果報告（§10 の返却フォーマットと定型文）
Gotcha: dry-run 指定時は Phase 3（整列）と Phase 6（基準確立）を実行しない。実差分が見つかった時点で FAIL としてレポートを返し、リバースコード環境にも基準タグにも一切触れない（完全無副作用）。

§6静的比較の設計

「パス除外 → 残った差分行のポート正規化判定」の 2 段構えで、全差分を 3 分類に落とす。

第 1 段: パス除外

比較対象から除外するパス（config.yml の diff_exclude ＋ 設計書 frontmatter での画面固有の追加指定）:

  • .git/
  • node_modules/・ビルド成果物（dist/, .next/ 等）
  • lock ファイル（package-lock.json 等。依存解決の非決定性によるノイズ排除。§11 ADR-3）
  • ポート設定を含む環境ファイル（.env* 等。両環境でポート番号が異なるため）
  • .flow-progress.json・ログ・キャッシュ類

第 2 段: ポート正規化判定

ポートが固定値（§4 の割当表）で決定的であることを利用する。config.yml の両ベース（9100 / 9110）から各サービスの期待ポート集合を算出し、差分行ごとに次を判定する。

── ポート差分の許容判定 ─────────────────────────────────────

  オリジナルコード環境側の行に現れたポート番号
     │ サービスオフセットへ逆引き（backend=+0, frontend=+1, ...）
     ▼
  「リバースコード環境側の同位置の値 ＝ そのサービスの
   リバース側期待値（9110 番台）」か？
     │
     ├─ YES ──▶ ポート差のみ（許容）
     └─ NO  ──▶ 実差分（FAIL 対象）
                 例: オフセット取り違え・ポートのハードコードミス

  ※ 行内のポート以外の部分は完全一致を要求する。
    「数字が違うだけなら許容」にはしない。

判定結果の 3 分類

分類意味扱い
一致差分なし—
ポート差のみ固定ポートの期待値どおりのポート番号相違許容（件数をレポートに記録）
実差分上記以外のすべての相違FAIL 対象。ファイル別・追加/削除/変更別に集計してレポートへ

§7動的比較の設計（Playwright 4 層）

オリジナルコード環境・リバースコード環境をそれぞれの固定ポート（9100 / 9110 番台）で起動し、routes の各画面を 4 層で突合する。決定的な判定を一次とし、視覚的な判定を証跡と位置づける。

層内容判定基準
L1 起動両環境のヘルスチェック / トップ応答両方の起動成功が前提条件。片方でも失敗なら以降をスキップして FAIL
L2 構造アクセシビリティスナップショット（DOM 構造）を、ポート・絶対 URL の正規化後に比較完全一致を要求（一次判定）。決定的で誤差の概念が不要
L3 視覚同一ビューポートでのスクリーンショット画素比較。タイムスタンプ等の動的領域はマスク差分画素率が閾値以下（config.yml の l3_threshold_percent、既定 0.1%）。フォントレンダリング揺らぎがあるため補助判定・証跡の位置づけ
L4 コンソールconsole error / warning の集合比較（メッセージ内のポートは正規化）リバースコード環境にのみ存在するエラーがあれば FAIL。両方に出るオリジナルコード環境由来の既存エラーは許容

総合判定: L1 ∧ L2 ∧ L4 を必須とする。L3 は閾値超過で FAIL とするが、レポートに画像ペアを添付して人間が最終裁定できるようにする。

偽陽性対策（拡張点）: DB 内容・乱数・日時などの動的データによる誤検出に備え、マスク対象セレクタを設計書 frontmatter で宣言できる拡張点を設ける。

§8環境同一性の検証（環境ヘルスチェック）

静的比較（§6）はコードの一致を、動的比較（§7）は挙動の一致を確認する。本節はその土台として、実行環境そのものが「ポート以外は同一」かつ「互いに独立」であることを機械的に確認する。実行タイミングは Phase 4 の冒頭（起動前: #1〜#2, #4〜#8, #10）と起動後（#3, #9）。

#検証項目具体的な確認方法NG 時の扱い
1ポート占有の帰属 起動前に固定ポート帯（9100〜9119）を lsof -i で走査し、LISTEN しているプロセスの cwd が自環境の worktree であるかを突合する stale プロセス（cwd 消滅）は kill して続行。無関係な稼働プロセスの占有は ERROR（環境起因）
2ポート設定の混入（転写後） リバースコード環境の全テキストを 9100 番台の出現で grep する（§6 の除外適用後）。オリジナルコード環境側も 9110 番台で逆方向に grep する 相手帯域の番号が 1 件でもあれば転写時の持ち込み・書き換え漏れ。ポート再生成（§9 手順 3）をやり直す
3実際の待受ポート 起動後、各プロセスの LISTEN ポート一覧を取得し、§4 割当表の期待値と完全一致するかを確認する 期待外ポートでの待受・期待ポートの不応答は FAIL（設定と実挙動の乖離）
4シンボリックリンクの実態 find -type l で worktree 内の symlink（node_modules 内を含む）を列挙し、realpath で実体を解決して「自 worktree 内 / 共有 store / worktree 外」に分類する 実体が相手側 worktree・元リポジトリを指すものは FAIL（独立性が崩れる）。パッケージマネージャの共有 store は「両環境が同一バージョンの実体を指す」ことを確認して許容
5依存ツリーの同一性 インストール済み依存の解決結果（npm ls --all 相当）をパス除去で正規化し、名前＋バージョンの集合を両環境で比較する 差分があれば依存再構築を 1 回リトライ → なお不一致なら FAIL（lock を diff 除外した ADR-3 の補完）
6ランタイムの同一性 両環境で実際に使われるランタイムのバージョン（node -v 等）を実測して突合する 不一致は ERROR（コードではなく実行基盤の問題）
7環境変数の同一性 起動プロセスに注入される環境変数をダンプし、ポート正規化後に比較する ポート以外の差分は FAIL（挙動差の温床）
8ファイルモード・種別 git ls-files --stage 等で実行ビットと種別（通常ファイル / symlink）を比較する（内容 diff では検出できない） モード・種別の差は実差分として扱い、整列（§9）で揃える
9プロセス独立性 片方の環境を停止しても、もう片方が応答し続けることを確認してから再起動する 連動して停止する場合は FAIL（プロセス・DB の共有を示す）
10ストレージ分離 ローカル DB・キャッシュ・アップロード先のパス設定を走査し、worktree 内に閉じているかを確認する 両環境が同一の外部パスを読み書きしていれば FAIL（独立性が崩れる）
設計の要点: #1・#4・#9・#10 は「同一であること」ではなく「独立していること」の検証。2 つの環境が実体（プロセス・モジュール・DB）を共有していたら、一致して見えるのは当然であり証明にならない。完全一致の証明は「独立した 2 環境が同じに振る舞う」ことで初めて成立する。

§9整列と基準コミット確立

sync で実差分が見つかった場合（dry-run 指定がない時）、リバースコード環境をオリジナルコード環境に揃える。ポート設定だけはリバースコード環境側の値を保全する。

── 整列の手順 ───────────────────────────────────────────────

  1. 保全    リバースコード環境のポート設定ファイル群を退避
  2. 転写    オリジナルコード環境 → リバースコード環境へミラーコピー
             （削除同期あり。除外は §6 準拠）
  3. 再生成  config.yml の固定ベース（9110）から
             リバースコード環境側のポート設定を再生成
  4. 再構築  依存インストールを再実行（package.json 変更の可能性）
  5. 再検証  Phase 2（静的）→ Phase 4（動的）へ合流し、収束ループを継続

基準コミット確立（Phase 6）

  • コミットは Skill ツールで grouping-commits を起動して行う（コミットメッセージは日本語 prefix 規約に従う）。
  • 基準コミットに annotated tag reverse-baseline/<system> を打つ。タグメッセージに検証日と判定サマリを記録し、再実行時は -f で打ち直す（旧基準のハッシュは返却レポートに残す）。設計書 frontmatter には書かない（ADR-1）。
Gotcha: 整列は git 操作（checkout / merge）ではなくファイル転写で行う。オリジナルコード環境とリバースコード環境は別ブランチであり、歴史を混ぜない（§11 ADR-2）。

§10失敗時の挙動と返却フォーマット

異常系の挙動

局面挙動
プリフライト NO-GO停止して報告（source_repo / source_ref 解決不能・固定ポート帯の占有 等）。ポート衝突時は stale プロセスの kill を試み、解消しなければ config.yml のポート変更を提案する
静的比較 FAILdry-run 指定時: 報告のみ。通常の sync: 自動整列（§9）へ進む
動的比較 FAIL（整列後も）依存再構築を 1 回だけ自動リトライ → なお FAIL なら収束ループ上限内で継続、上限到達で停止・差し戻し
収束ループ上限到達FAIL 確定。リバースコード環境は最後の整列状態のまま残し、コミットしない（基準を更新しない）
環境起因（ポート占有・Playwright 不通 等）差分判定と区別して ERROR として報告。リバースを行うスキルが「設計書の不備」と誤解釈することを防ぐ

返却フォーマット（リバースを行うスキルへの契約）

最終応答の末尾に、次のキーバリュー形式の構造化ブロックを必ず付ける。

── 返却ブロックの例 ─────────────────────────────────────────

  status:          PASS | FAIL | ERROR
  mode:            setup | sync | teardown（+ dry-run / reset-first の有無）
  original_code:   <オリジナルコード環境のパス>（ブランチ・ポート付き）
  reverse_code:    <リバースコード環境のパス>（ブランチ・ポート付き）
  baseline_tag:    reverse-baseline/<system> → <hash>  # PASS 時は新基準
  static_diff:     実差分 <N> 件（上位 <n> 件のファイル一覧）
                   ポート差（許容） <M> 件
  dynamic:         L1=PASS L2=PASS L3=PASS(0.03%) L4=PASS
                   （route 別の内訳を添付）
  env_check:       全項目 PASS（NG があれば §7-2 のキーと内容を列挙）
  artifacts:       <スクリーンショット等の保存先パス>
  hint:            # FAIL 時のみ
                   src/components/Foo.tsx の差分
                   → 設計書 §9 領域別仕様 の記載不足と推定

hint は「差分ファイル → 設計書章マップ（§1〜§16）の対応推定」であり、リバースを行うスキルの設計書修正ループの入力になる。本スキルの判定価値の中核。

ユーザー向け定型文（検証後の報告）

機械向けの返却ブロック（上記）とは別に、人間がすぐ確認・操作できる定型文で報告する。値の出どころは 3 つ — ポートと起動手順は config.yml、確認 URL のパスは設計書 frontmatter の route、判定と証跡は本スキルの実行結果。アプリ依存は起動手順（config.yml の launch）だけなので、定型文の形は全アプリ共通になる。

── ユーザーへ返す定型文（PASS の例）─────────────────────────

判定によって差し替える部分は次のとおり。それ以外のセクション（確認 URL・環境情報・再起動・片付け）は全判定共通。

§11設計判断（ADR）と関連スキル

ADR-1基準コミットの記録は git tag 単独とする（設計書には書かない）

採用: annotated tag reverse-baseline/<system> を基準コミットに打ち、これを唯一の正とする。理由: ① タグはリポジトリ本体に保存され、worktree の破棄と無関係に永続する ② タグが指すコミットは到達可能になり、ブランチを削除しても gc で消えない ③ 決定論的な名前で再発見できるため記録ファイルが不要 ④ 設計書に検証用メタデータが残り続けない（設計書が持つのは出自情報 source_repo / source_ref のみ）。

却下案: 設計書 frontmatter に記録 — 検証が終わったあとも設計書の関心事（画面仕様）と無関係なメタデータが残り続け、gc 保護もできない。/ リバースコード環境内の JSON ファイル — gitignore すると worktree 削除で消え、コミットするとオリジナルコード環境に無いファイルが増えて「完全一致」と自己矛盾する。/ 両方持ち — 二重管理の不整合リスクが利便を上回る。

ADR-2整列は git 操作ではなくファイル転写で行う

採用: オリジナルコード環境 → リバースコード環境へのミラーコピー（削除同期あり）。理由: オリジナルコード環境とリバースコード環境は別ブランチであり、checkout / merge では歴史が混ざる。リバースコード環境のコミット履歴は「設計書からの再構築の記録」として独立に保つ価値がある。

ADR-3lock ファイルは静的比較から除外する

採用: 依存解決の非決定性によるノイズを排除するため除外。トレードオフ: 「lock まで一致」を求める厳格運用も考えられる。依存バージョン差が動的比較（L2〜L4）で挙動差として現れれば検出できるため、初期設計では除外とし、運用実績を見て見直す。

ADR-4worktree を引数で受け取らず、本スキルが用意・命名する

採用: 唯一の必須入力は設計書パスとし、オリジナルコード環境 / リバースコード環境は本スキルが命名規則（§4）から導出・確保する。理由: リバース環境の整備そのものが本スキルの責務であり、用意と名前管理を一元化することで、呼び出し側ごとの命名揺れ・ポート重複を排除し、再実行時に必ず同じ環境へ解決される冪等性を保証できる。

却下案: worktree パスを引数で受け取る — 環境管理の責任が呼び出し側へ分散し、再実行時に別の worktree を渡す事故や、オリジナルコード環境の同一性（どのコミットが「正」か）の追跡不能を招く。

ADR-5worktree は初回に一度だけ作成し、ユーザーの明示依頼まで削除しない

採用: 検証のたびに worktree を作成・削除するのではなく、初回実行時に作成した同じ worktree を使い回す。削除の唯一の経路は mode: teardown（ユーザーの明示依頼専用）で、固定ポート帯の一括 kill を含めて parallel-dev-worktree の削除手順へ委譲する。理由: ① リバースコード環境のブランチ履歴（再構築の記録）が環境と共に維持される ② ポートが固定され、リバースを行うスキルも人間も「いつもの場所・いつものポート」で扱える ③ worktree 作成と依存インストールのコストを検証のたびに払わない。

却下案: 検証ごとの使い捨て worktree — 毎回の依存再構築コストが大きく、リバースコード環境の再構築履歴が散逸する。ポートの予見性も失われる。

ADR-6プロジェクト可変値は config.yml に集約する

採用: ポート番号・diff 除外リスト・L3 閾値・max-loop 既定値など、プロジェクトによって変えたくなる可能性がある値は、スキルフォルダ直下の config.yml（defaults ＋ projects 別上書き）に置き、スキル本文から分離する。理由: SKILL.md を書き換えずにプロジェクト固有の調整ができ、値の変更がスキルのレビュー・テストサイクルを毎回要求しない。値の階層は「普遍のルール＝SKILL.md / プロジェクト単位＝config.yml / 画面単位（routes・マスク対象セレクタ）＝設計書 frontmatter」の 3 層で役割分担する。

却下案: SKILL.md へのベタ書き — 値変更のたびに managed ファイル編集となりレビューゲートが走る。/ 設計書 frontmatter に全部書く — プロジェクト共通の値が画面ごとに重複し、変更漏れを招く。

ADR-7WSL2 対応と /mnt 配置禁止

採用: コマンド解決（lsof が使えなければ ss -ltnp ＋ /proc/<pid>/cwd へフォールバック）と、fs-location プリフライトで対応する。worktree 配置先が Windows 側ファイルシステム /mnt/* 配下の場合は ERROR とする。理由: /mnt/* はパーミッションが 777 に丸められ、inotify が動作せず、I/O も劣化するため、Linux ネイティブファイルシステム配下への配置を必須とする。

却下案: /mnt 配下もサポートする — パーミッション・inotify・I/O の問題を個別に回避する実装コストが、WSL2 ユーザーが素直に Linux ネイティブ FS 側に worktree を置けば得られる利益に対して見合わない（不採用）。

ADR-8env ファイル直接比較（env-file-parity）の追加

採用: .env* は静的比較から除外したまま、ポート正規化付きの専用突合を env_check の項目（env-file-parity）として追加する。理由: バンドラのみが読み込む変数（MSW フラグ等、起動プロセスの環境変数ダンプには現れない値）が、既存の env-runtime-parity の検証網にかからない穴になっていた。

却下案: diff_exclude から .env* を外して静的比較に戻す — 両環境のポート番号が異なるため常に差分として検出されてしまい、実質的に使えない（不採用）。

ADR-9npm ci 化と lock 転写（ADR-3 の部分改訂）

採用: 整列時の依存再構築を install_command（既定 npm ci）に固定し、*.lock を diff 除外から外して転写対象に変更する。理由: 従来の npm install 相当は非決定的で、依存ツリーの不一致を dep-tree-parity のリトライに頼って検出・解消する運用だった。lock を転写したうえで npm ci に固定することで、依存解決そのものを決定論的にする。

却下案: lock ファイルも引き続き diff 除外のままとする（ADR-3 を維持） — 依存解決の決定論が担保できず、dep-tree-parity のリトライ頼みの運用が残ってしまう（不採用）。

ADR-10dry-run 無副作用対象の精緻化

採用: dry-run の無副作用対象を「リバースコード環境の git 管理内容」と「基準タグ」の 2 つに限定する。node_modules・バンドラキャッシュ等の git 管理外生成物の修復（npm ci・キャッシュ削除）は dry-run でも実行を許容する。理由: Phase 5（環境同一性チェック＋動的比較）を dry-run でも実行し、静的比較が FAIL でも L1〜L4 まで計測して診断情報を最大化するという既存方針を成立させるための前提整備。node_modules が壊れたままでは L1 起動そのものが失敗し、診断ができない。

却下案: 完全無副作用を維持する（生成物の修復も dry-run では行わない） — node_modules 破損時に L1 が起動不能となり、dry-run で得られるはずの診断情報が得られなくなる（不採用）。

ADR-11Playwright 実行方式の 2 モード化（mcp / node-script）

採用: 動的比較（§7）の実行方式を playwright_exec.mode で mcp（既定）と node-script の 2 つに分岐する。mcp は従来どおり Playwright MCP ツールを呼び出す。node-script は config.yml の command_template を Bash から直接実行し、Playwright MCP が接続されていない環境でも動的比較を完走できるようにする。理由: 実プロジェクトのセッションに Playwright MCP が接続されておらず、既定の MCP 前提のままでは Phase 5 が実行不能だった。command_template は実機検証で判明した 3 件の不具合 — waitUntil="networkidle" が Vite dev server の HMR 用 WebSocket で繋がりっぱなしになり永久タイムアウトする／page.accessibility.snapshot() が Playwright 1.61 系で撤去済み／navigate 先を localhost にすると WSL2 の Chromium が名前解決待ちでタイムアウトする — を修正した検証済み版であり、waitUntil="load"・ariaSnapshot()・127.0.0.1 固定の 3 点をそのまま固定事項として保持する。さらにクリーン初回起動での再検証で 2 件の不具合を追加修正した — .catch() だけで process.exit(1) していたため失敗時に browser.close() が呼ばれず Chromium プロセスが孤児化して残存し（旧セッションの残骸が 10 個近く蓄積し CPU/メモリを消費して以降の起動を不安定にしていた）、これを try/finally で browser.close() を必ず呼ぶ形に修正した／timeout: 30000 は dev サーバー再起動直後の初回リクエスト（Vite が該当ルートの依存モジュールを遅延変換するため大規模アプリでは 40〜50 秒かかる。2 回目以降は数秒）に対して短すぎたため 60000 へ延長した（node_modules_strategy の問題ではなく純粋なタイムアウト不足）。

却下案: Playwright MCP 接続を必須要件のまま維持する — MCP が接続されない環境では本スキルそのものが運用できなくなるため不採用。

なお playwright_exec.mode の既定は後続 ADR（ADR-14）で auto 化し、MCP 可用性を実行時検出して mcp / node-script を自動選択するようにした。また timeout: 60000 は固定事項から外し、後続 ADR（ADR-15）で config.yml の nav_timeout_ms / settle_wait_ms パラメータとして外出しした。

ADR-12allow_mnt_fs による /mnt 配置の WARN 緩和（ADR-7 の部分緩和）

採用: fs-location プリフライトの /mnt/*（drvfs）判定を allow_mnt_fs（既定 false＝ERROR）で切替可能にし、true 指定時は ERROR ではなく性能 WARN として扱う。理由: 実プロジェクトの source_repo が /mnt/c/...（Windows 側 drvfs）固定で、既定 ERROR のままでは即 NO-GO になる。実測では worktree 作成（約 80 秒/本）も全量 diff も動作しており、低速であるだけで運用は成立する。

却下案: /mnt を常に ERROR のまま維持する — drvfs 上に配置された実プロジェクトを検証できなくなるため不採用。／常に許可する — ADR-7 が定めたネイティブ FS 配置の推奨を既定側から崩すことになるため不採用。allow_mnt_fs は既定 false を保ったまま projects 単位の opt-in にとどめる。

なお allow_mnt_fs の既定は後続 ADR（ADR-14）で auto 化し、source_repo が /mnt/* 配下かを実行時検出して性能 WARN 続行を自動判定するようにした。

ADR-13node_modules_strategy による symlink_main 方式の追加

採用: 依存構築の方式を node_modules_strategy（既定 npm_ci＝各環境が独立して npm ci を実行 / symlink_main＝source_repo の node_modules へ symlink）で切替可能にする。symlink_main 選択時は node_modules を「独立した 2 環境であることの証明対象」から除外し、env_check の symlink-independence・node-modules-health・bundler-cache-fresh の各項目に但し書きを設ける。理由: drvfs 配下では npm ci が極端に低速で、実プロジェクトの既存運用は source_repo の node_modules へ symlink する方式だった。

却下案: 常に npm ci に固定する（ADR-9 を無条件維持） — drvfs 上での実行時間が長すぎ実運用に耐えないため不採用。トレードオフ: symlink_main は .vite 等のバンドラキャッシュも共有するため、初回 navigate が dependency re-optimize による reload に巻き込まれる可能性がある。1 回のリトライで吸収する。

なお node_modules_strategy の既定は後続 ADR（ADR-14）で auto 化し、drvfs 検出により symlink_main / npm_ci を自動選択するようにした。ただしこの自動選択はさらに後続 ADR（ADR-16）で撤回し、auto は常に npm_ci に固定した。

ADR-14環境固有値のゼロコンフィグ自動解決（既定 auto）

採用: node_modules_strategy / allow_mnt_fs / playwright_exec.mode の既定を auto、playwright_exec.node_path の既定を null（自動解決）とし、Phase 2 プリフライトが実行時に検出して具体値へ解決する。allow_mnt_fs=auto は source_repo が /mnt/* 配下かで性能 WARN 続行を判定し、node_modules_strategy=auto は drvfs 検出で symlink_main / それ以外 npm_ci を選び、playwright_exec.mode=auto は Playwright MCP 可用性で mcp / node-script を選び、node_path=null は require.resolve / npm root で解決先を取得する。明示値は auto を上書きする強制指定として残す。理由: これらの値はすべて実行時に機械検出できる（パス prefix・fs 種別・MCP 可用性・モジュール解決）。にもかかわらず config.yml への手入力を要求すると、node_path のような npx キャッシュハッシュ付き絶対パスを人間が用意することになり非現実的で、初回セットアップが破綻する。検出できる値は検出して埋め、人間が書くのはアプリ固有の起動コマンド（services / launch）だけに限定する。

却下案: 環境値を config.yml への手入力必須とする — 人間が node_path 等を用意できず初回運用が成立しないため不採用。／既定を安全側の固定値（false / npm_ci / mcp）に置き projects で上書きさせる — 特殊環境ごとに毎回手書きが必要で、ゼロコンフィグにならないため不採用。トレードオフ: 自動検出は誤検出の可能性があるため、解決結果と根拠（drvfs 検出・MCP 可用性）を §8 の報告に必ず含めて可監査にする。強制上書きの明示値も残す。

なお node_modules_strategy=auto の drvfs 検出による symlink_main 自動選択は後続 ADR（ADR-16）で撤回し、auto は常に npm_ci に固定した。allow_mnt_fs / playwright_exec.mode / node_path の自動解決方針は本 ADR のまま維持する。

ADR-15調整可能な数値パラメータの config 外出しと projects 上書き

採用: command_template にハードコードしていたタイムアウト（page.goto の 60000・描画安定待ちの 10000）を playwright_exec.nav_timeout_ms / settle_wait_ms として config.yml に外出しし、テンプレートは {nav_timeout} / {settle_wait} プレースホルダで参照する。これらを含む数値系 defaults はすべて projects.<system> で上書きでき、有効値は defaults ← projects の深いマージ（projects 優先）で解決する。理由: プロジェクトごとに初回変換の重さ・描画の重さが異なり、単一のハードコード値では大規模アプリで初回タイムアウト、軽量アプリで無駄待ちが起きる。スキル本体（command_template）を書き換えずにプロジェクト単位で調整できる逃し口が必要。

却下案: タイムアウトをテンプレートにハードコード維持する — プロジェクト差に対応できず不採用。／値をコマンド引数で毎回渡す — 呼び出し側の負担が増え、config の値階層と二重管理になり不採用。

ADR-16独立性を犠牲にする選択は自動化しない（node_modules_strategy=auto は常に npm_ci）

採用: node_modules_strategy=auto を「常に npm_ci に解決」に変更し、symlink_main は人間が config/projects で明示指定した時だけ使う。drvfs 検出時は「symlink_main で高速化できるが node_modules の独立性が証明対象から外れる」を WARN で提示するに留め、自動切替はしない。あわせて playwright_exec.node_path の自動解決を、テスト対象アプリの多くが playwright を自依存に持たない現実に即した再現可能な具体手順（① reverse worktree での require.resolve('playwright') → ② npm root -g 配下の playwright/ → ③ npx キャッシュの実探索 find "${npm_config_cache:-$HOME/.npm}/_npx" -type d -path '*/node_modules/playwright' のヒット先の親 node_modules → ④ いずれも不在なら runtime-deps-ready を NO-GO とし人間に node_path 明示を促す）に確定した。理由: 本スキルの核心目的は「2 環境の完全独立の証明」である。独立性を犠牲にするか否かは目的の根幹に関わる判断であり、性能都合で自動判定が黙って行うと、証明の前提が利用者に気づかれぬまま崩れる。低速環境でも npm_ci は時間がかかるだけで動作・正しさは変わらないため、既定を npm_ci に倒すコストは待ち時間のみ。また ADR-14 時点の node_path 自動解決（「require.resolve → だめなら npx/npm root の解決先」）は後段が曖昧で機能しない。テスト対象アプリの多くは playwright を自依存に持たず require.resolve が失敗し、npm root（プロジェクトローカル）は npx キャッシュのハッシュ名フォルダを探さないため、再現可能な実探索手順に置き換えた。

却下案: auto が drvfs 検出時に symlink_main を自動選択し理由は事後報告に留める（ADR-14 の原案） — 独立性という核心を黙って犠牲にするため不採用。
