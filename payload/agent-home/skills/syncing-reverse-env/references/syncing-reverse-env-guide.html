リバース環境同期スキル ガイド
syncing-reverse-env

リバース元の「オリジナルコード環境」と設計書由来の「リバースコード環境」の 2 つの worktree を用意・名前管理し、ポート番号以外の完全同一を検証して、一致証明済みの「基準コミット」を確立するスキルの確定仕様。

こんな人向け
リバースエンジニアリングで設計書の完全性を証明したい人 ／ PR レビューで挙動差分を実機確認したい人 ／ リファクタ前後の安全性を機械的に証明したい人
こんな場面で使う
設計書だけからコードを再構築する検証サイクル ／ main と PR ブランチを並行起動して差分を見たい時 ／ リファクタ前後で挙動が変わっていないかを証明したい時
対応 OSmacOS / Windows（WSL2・Ubuntu）
検証状況サンプルプロジェクトで動作確認済み（E2E: setup → sync → 基準タグ確立 → reset）。node-script 方式（MCP 非接続）でも実プロジェクト（WSL2・drvfs）で L1〜L4 まで実機検証済み
依存git / Playwright（MCP または node-script 直実行のいずれか）

§1概要

リバースエンジニアリング設計の中心課題は「設計書が完全であることの証明」にある。本スキルはこれを、設計書だけを見て再構築したリバースコード環境が、オリジナルコード環境とポート番号以外で完全に同じ動作をすることで証明する。

責務境界

担当責務
本スキル
syncing-reverse-env		 ⓪ 両 worktree の用意と名前管理（初回のみ作成・固定ポート割当・冪等な再発見・ユーザー依頼時のみ削除）
① 同一性検証（静的 diff ＋ 環境ヘルスチェック ＋ Playwright 動的比較）
② 差分があればリバースコード環境をオリジナルコード環境へ整列
③ 一致証明後のコミット＝基準コミットの確立と基準タグの記録
④ 基準コミットへの復帰（sync の reset-first オプション）
rebuilding-code-from-docs スキル
（呼び出し元）		 設計書の不備を修正し、リバースコード環境の中身（コード）を再構築するループ全体の運転。worktree という容器の用意は本スキルに任せる

オリジナルコード環境が常に「正」。リバースコード環境と設計書のどちらを直すかの判断は rebuilding-code-from-docs スキルが持ち、本スキルは環境の用意・判定・整列に徹する。

§2リバース以外の使い方

本スキルはリバース検証のために設計する。ただし「ポート以外同一の 2 環境を永続 worktree で維持し、静的 diff ＋ Playwright 4 層で突合する」という仕組みの性質上、リバース以外にも次のような使い方ができると考えられる。

応用オリジナルコード環境（正）リバースコード環境（対象）使い方
PR レビュー実機確認main ブランチPR ブランチ 並行起動して挙動差分・ビジュアル差分を実機で突合する。「意図した変更以外に差分がないか」を機械判定できる
リファクタ安全性証明リファクタ前リファクタ後 「挙動が変わらないこと」を L1〜L4 の全層一致で証明する。この場合は実差分ゼロではなく動的比較の全層一致が合格条件になる
UI ビジュアルリグレッション変更前変更後 L3（スクリーンショット画素比較）を単独利用し、意図しない見た目の変化を検出する

応用時は「何を一致とみなすか」（静的一致まで求めるか、動的一致のみか）が用途ごとに変わる。合格条件のプリセットを mode とは別の軸で持たせることが、スキル本体実装時の検討事項になる。

§3用語

<system>
設計書 frontmatter の source_repo 末尾のリポジトリ名（.git 除去）から機械的に導出する識別子。
<画面ID>
設計書 frontmatter doc_id: screen-<画面ID> から screen- 接頭辞を除去して導出する識別子。
<scope>
<system>-<画面ID>。worktree・ブランチ・基準タグ・ポートスロットの管理単位。共有アプリを画面単位で最大 5 画面まで並行検証するために、この粒度で分離する。
オリジナルコード環境
リバース対象のオリジナルコードを保持する git worktree（worktree 名 original-code-<scope>）。設計書 frontmatter の source_repo / source_ref から本スキルが用意する読み取り専用コピーで、常に「正」。内容を変更することはない。
リバースコード環境
設計書だけを見て再構築したコードの git worktree（worktree 名 reverse-code-<scope>、ブランチ feature/reverse-code-<scope>）。容器（worktree・ブランチ・ポート）は本スキルが用意し、中身のコードは rebuilding-code-from-docs スキルが書く。
worktree のライフサイクル（永続・使い回し）
両 worktree は初回実行時に一度だけ作成し、以後の検証ではすべて同じものを使い回す。削除はユーザーが明示的に依頼した時（teardown）のみ。
基準コミット / 基準タグ
ポート以外の完全一致を証明できた時点のリバースコード環境側コミットが基準コミット。annotated git tag reverse-baseline/<scope>（基準タグ）がその唯一の記録で、タグメッセージに検証日と判定サマリを持つ。コミットとタグはリポジトリ本体に保存されるため、worktree を破棄しても消えない。リバースコード環境はいつでも git reset --hard reverse-baseline/<scope> で基準状態へ復帰できる。
ポートモデル
両環境の稼働ポートは <scope> ごとに動的割当てするスロット番号（1〜5）から計算式で決まる（§5）。両環境は必ず異なる帯域を持ち、これが「唯一許容される差分＝ポート番号」の根拠になる。
config.yml（スキルフォルダ直下）
プロジェクトによって変えたくなる可能性がある値（ポート計算式・サービス一覧・起動手順・diff 除外リスト・L3 閾値・max-loop 既定値）の置き場。defaults ＋ projects 別上書きの 2 層構造。

§4使い方（mode とユースケース）

mode は運用の 3 局面 —「初回の環境整備」「それ以降の検証サイクル」「検証終了時の片付け」— に対応する 3 つ。

mode局面動作
setup初回両 worktree を作成し、名前・固定ポートを割り当てて返す。比較は実行しない
sync（既定）2 回目以降検証（静的＋環境＋動的）→ 差分があれば整列 → 一致で基準タグを更新
teardown検証終了時固定ポート帯の一括 kill ＋ 両 worktree の削除。ユーザーの明示依頼がある時だけ受け付ける

sync のオプション

オプション動作
dry-run整列・基準タグ更新を行わず、差分レポートだけを返す。無副作用の対象は「リバースコード環境のgit 管理内容」と「基準タグ」の 2 つ（Phase 5 の起動・観測とオリジナル側の source_ref 復元は実行する）。node_modules・バンドラキャッシュは git 管理外の生成物であり、その修復（npm ci・キャッシュ削除）は診断を最大化するため dry-run でも実行を許容する
reset-first開始前にリバースコード環境を基準タグへ git reset --hard で復帰させてから検証サイクルに入る

アクター別ユースケース

アクター意図入口と mode期待する出力

§5環境と設定（命名・ポート・config.yml）

対応 OS とコマンド互換

本スキルは macOS と Windows（WSL2・Ubuntu）の両方に対応する。起動時に OS を判定し、以下のとおりコマンドを解決する。

項目macOSWSL2（Ubuntu）
ポート走査lsof -ilsof が使えればそれを優先。無ければ ss -ltnp ＋ /proc/<pid>/cwd で cwd を解決する
symlink 実態解決realpathrealpath（存在確認のうえ利用。無ければ導入手順を提示して NO-GO）
Playwright 依存通常のインストールのみで動作npx playwright install-deps 相当のシステムライブラリ導入を確認。不足は導入手順を提示して NO-GO
WSL2 固有の注意: ① worktree の配置先（source_repo の親ディレクトリ）が Windows 側ファイルシステム /mnt/*（drvfs）配下だと既定で ERROR。ただし config.yml / projects の allow_mnt_fs: true で性能 WARN に緩和できる（worktree 作成は約 80 秒/本、全量 diff はバックグラウンド実行が必要なほど低速だが動作はする） ② dev サーバーが localhost（または 0.0.0.0）で待ち受ければ、WSL2 の localhost フォワーディングにより Windows 側ブラウザから http://localhost:<port> でそのままアクセスできる ③ inotify の watch 数上限に注意する。最大 5 scope × 2 環境で常駐する dev サーバーは最大 10 プロセスに達しうるため、fs.inotify.max_user_watches の余裕を事前に確認する ④ band_start（既定 9100）が Windows 側の常駐プロセスと衝突する場合は、config.yml で band_start を変更する

命名規則と環境確保

worktree の用意・命名・再発見はすべて本スキルが担う。作成は初回実行時の一度だけで、以後は同じ worktree を再発見して使い回す。どの mode でも Phase 2 で必ず実行し、冪等に振る舞う。

対象規則
システム識別子 <system>設計書 frontmatter の source_repo 末尾のリポジトリ名（.git 除去）から機械的に導出する
画面ID <画面ID>設計書 frontmatter doc_id: screen-<画面ID> から screen- 接頭辞を除去して導出する
スコープ <scope><system>-<画面ID>。worktree・ブランチ・基準タグ・ポートスロットの管理単位（共有アプリを画面単位で最大 5 画面まで並行検証するため）
オリジナルコード環境worktree 名 original-code-<scope>。source_ref を固定 checkout した読み取り専用コピー。汚れ・ズレを検出したら内容だけを source_ref へ復元する（worktree 自体・名前・ポートは保持）
リバースコード環境worktree 名 reverse-code-<scope>、ブランチ feature/reverse-code-<scope>。基準コミットと再構築履歴を持つ本体
基準タグannotated tag reverse-baseline/<scope>。基準コミットの唯一の記録
ポートconfig.yml のポート計算式と、<scope> に動的割当てするスロット番号から下の計算式のとおり確定する。プロジェクトに依存しない
作成の実務worktree の物理作成・削除は git worktree add / remove を本スキルが直接実行する（オリジナルは --detach、リバースは -b。配置は source_repo の親ディレクトリ直下）。通常開発用の worktree 管理の仕組みとは命名・配置規約が異なるため委譲しない

ポートモデル（計算式）

worktree は初回に一度だけ作成し、teardown まで使い回す。ポートは <scope> ごとに動的割当てするスロット番号（1〜5）から次の式で決まる。

── ポート計算式 ─────────────────────────────────────────────

  slot k = 1..5（動的割当）
  original = band_start + slot_stride×(k-1) + サービス index
  reverse  = original + reverse_offset
  既定: band_start=9100, slot_stride=20, reverse_offset=10
  → slot k の占有帯 = [9100+20(k-1), 9119+20(k-1)]（連続 20 ポート）

サービス index は config.yml の services 配列の並び順（既定 [frontend] → index 0）。既定の slot 1・frontend では原ポートは original=9100 / reverse=9110 になる。

スロット割当手順: 兄弟 worktree（original-code-* / reverse-code-*）を走査し、各ルート直下の .reverse-port-slot（番号 1 行）を読んで使用中スロットを把握 → 空いている最小番号を両 worktree のルートに書き込む。開発用 worktree のスロットファイル .port-slot とは誤カウント防止のため別名にしている。max_slots（既定 5）を超えて空きがない場合は ERROR。

互換注記: 旧固定表（frontend = オリジナル 9101 / リバース 9111）は廃止し、既定の slot 1 では frontend = オリジナル 9100 / リバース 9110 になる（稼働中環境が存在しないため移行措置は設けない）。

config.yml

プロジェクトによって変えたくなる可能性がある値は、スキルフォルダ直下の config.yml に集約する。値の階層は 3 層で役割分担する: 普遍のルール＝SKILL.md / プロジェクト単位の可変値（サービス一覧・起動手順）＝config.yml / 画面単位の可変値（routes・マスク対象セレクタ）＝設計書 frontmatter。

── config.yml の構造 ───────────────────────────────────────

  defaults:
    ports:
      band_start: 9100      # slot 1 のオリジナル側ベースポート
      slot_stride: 20        # スロット 1 個あたりの占有帯幅
      reverse_offset: 10     # オリジナル→リバースのポート加算値
      max_slots: 5           # 同時並行できる画面スロット数の上限
    services:                # 並び順 = ポートオフセット（0..9）。未使用分類は書かない
      - frontend
    launch:
      frontend: "npm run dev -- --port {port}"  # 起動コマンドの既定形
    install_command: "npm ci"                 # 整列・修復時の依存インストール（projects で上書き可）
    env_file_globs: [".env", ".env.*"]        # env-file-parity（§7-2）の比較対象
    bundler_cache_dirs:                        # 整列後に削除するキャッシュ（§7-4 手順⑤）
      - node_modules/.vite
      - node_modules/.cache
    artifacts_root: "~/agent-home/sessions/reverse-env"  # 証跡ルート（scope/実行日時で分離。§8）
    l3_threshold_percent: 0.1   # L3 視覚比較の許容差分画素率
    max_loop: 3                 # 収束ループ上限の既定値
    tag_namespace: reverse-baseline
    diff_exclude:               # 静的比較の除外リスト（§7）
      - .git/
      - node_modules/
      - dist/
      - .next/
      - "*.lock"                 # 整列時は転写対象（§7-4）。静的比較のみ除外
      - ".env*"                 # env-file-parity（§7-2）で直接比較するため静的比較からは除外
      - .reverse-port-slot       # スロット番号記録ファイル（開発用 .port-slot とは別物）
    node_modules_strategy: npm_ci   # npm_ci（既定・各環境で独立インストール。
                                    #   §7-2 symlink-independence / node-modules-health
                                    #   の独立性検証対象） /
                                    # symlink_main（source_repo の node_modules へ
                                    #   symlink。独立性の証明対象から除外。
                                    #   drvfs 上での npm ci 低速回避に使う）
    allow_mnt_fs: false             # false（既定）: fs-location（§5 プリフライト #3）が
                                    #   /mnt/* 配下を ERROR とする /
                                    # true: ERROR ではなく性能 WARN に緩和
                                    #   （worktree 作成 約 80 秒/本・全量 diff は
                                    #    BG 実行要。drvfs 上でも動作はする）
    playwright_exec:
      mode: mcp                     # mcp（既定・Playwright MCP ツールを使用） /
                                    # node-script（command_template を Bash で直接
                                    #   実行。MCP 非接続環境向け）
      node_path: null               # node-script 時に playwright の解決先へ渡す
                                    #   NODE_PATH。null で環境依存
      # command_template: 実機検証で判明した固定事項（waitUntil="load" 固定・
      #   ariaSnapshot() 使用・navigate 先 127.0.0.1 固定）を理由ごと保持する
      #   検証済みスクリプト。全文は config.yml を正とし、本書では再掲しない

  projects:                     # プロジェクト（<system>）単位の上書き（必要時のみ）
    "<system>":
      services: [backend, frontend, docs, "db-tools"]
      launch:                   # 起動手順は唯一のアプリ依存要素
        backend: "<backend 起動コマンド> --port {port}"
        frontend: "<frontend 起動コマンド> --port {port}"
      allow_mnt_fs: true                    # 画面別に個別上書き可
      node_modules_strategy: symlink_main   # 画面別に個別上書き可
      playwright_exec:
        mode: node-script                   # 画面別に個別上書き可

具体的な絶対パスを含む実エントリ（node_path の実パス等）はデプロイ先ローカルの config.yml に置く。上記は汎用例。

制約

制約: ① 画面スロット制のため最大 5 画面まで並行検証できる（slot 1〜5、各 20 ポート帯）。6 画面目以降は空きスロット待ちとなる ② worktree は全体で 5 本までの上限のうち 2 本を <scope> ごとに長期占有する ③ teardown はユーザーの明示依頼がある時だけ受け付け、本スキルが自発的に削除することはない ④ teardown 後も基準タグとブランチは残り、基準コミットは消えない

プリフライト（go / no-go 判定）

#キー検証項目NO-GO 時の扱い
1design-doc-contract設計書が存在し reverse-docs テンプレート準拠（type: screen-detail-design）で、doc_id が screen- 接頭辞を持ち、source_repo / source_ref が解決できるNO-GO
2os-toolchainOS を判定（macOS / WSL2）し、コマンドを解決する（§5 対応 OS とコマンド互換）。WSL では Playwright 依存ライブラリの導入確認を含むコマンド解決不能・依存不足は導入手順を提示して NO-GO
3fs-locationworktree 配置先（source_repo の親ディレクトリ）が WSL の /mnt/* 配下でないことを確認/mnt/* 配下は既定 ERROR。allow_mnt_fs: true 指定時は性能 WARN に緩和（worktree 作成 約 80 秒/本・全量 diff は BG 実行要）
4env-provision環境確保（命名規則で両環境を検出。無ければ作成）とスロット割当が完了している。git worktree list --porcelain で prunable・破損を検出したら prune してから再作成する。スロット割当は親ディレクトリへの mkdir ロック（走査 → 書込 → 解放）で原子化する。setup 時も symlink・HEAD・ブランチ整合を軽量検証するworktree 総数上限（5 本）超過・スロット枯渇（max_slots）は NO-GO
5original-cleanオリジナルコード環境が source_ref と一致し cleanズレ・汚れは内容を source_ref へ自動復元する
6reverse-dirty-logリバースコード環境の dirty 状態を記録sync では許容（整列で上書きされる）。dry-run 指定時は警告を提示
7port-band-free当該 <scope> のスロット帯（計算式で確定するポート）が未占有stale プロセスは kill して続行。WSL で Windows 側常駐プロセスが占有し kill 不能な場合は band_start の変更を案内して ERROR
8runtime-deps-readyplaywright_exec.mode で分岐する。mode=mcp（既定）: Playwright MCP 接続とブラウザバイナリ導入済みを要求。mode=node-script: MCP 非接続でも NO-GO にせず、node_path で解決される playwright モジュールと chromium バイナリの存在を確認する。sync 時は node_modules の存在・.bin/* の実行可能性・lock ファイルの存在まで確認する（setup 直後の空容器では可用性確認のみ行う）NO-GO（ERROR 系。§8）
9baseline-tag-existsreset-first 指定時のみ: 基準タグが存在NO-GO（基準未確立と報告）

§7検証内容

7-1. 静的比較

「パス除外 → 残った差分行のポート正規化判定」の 2 段構えで、全差分を 3 分類に落とす。除外パスは config.yml の diff_exclude ＋ 設計書 frontmatter での画面固有の追加指定。

ポートが計算式で決定的であることを利用し、差分行ごとに「オリジナル側の値をサービス index へ逆引きし、リバース側の同位置の値が reverse = original + reverse_offset の期待値と一致する」場合のみ許容する。行内のポート以外の部分は完全一致を要求し、「数字が違うだけなら許容」にはしない。

分類意味扱い
一致差分なし—
ポート差のみ計算式どおりの期待値どおりの番号相違許容（件数をレポートに記録）
実差分上記以外のすべての相違FAIL 対象。ファイル別・追加/削除/変更別に集計

7-2. 環境同一性チェック（全項目）

実行環境そのものが「ポート以外は同一」かつ「互いに独立」であることを機械的に確認する。起動前（listen-actual・process-independence を除く全項目）と起動後（listen-actual・process-independence）に実行する。

#キー検証項目確認方法NG 時の扱い
1port-ownerポート占有の帰属os-toolchain の解決結果に従い確認する（macOS: lsof -i の cwd 突合 / WSL: ss -ltnp ＋ /proc/<pid>/cwd の cwd 突合）で当該 <scope> のスロット帯を走査し、LISTEN プロセスの cwd が自環境の worktree かを突合stale は kill して続行。無関係な稼働プロセスは ERROR
2port-config-crossポート設定の混入（転写後）リバース側をオリジナル側帯域で、オリジナル側をリバース側帯域で相互に grep相手帯域の番号が 1 件でもあれば転写ミス。ポート再生成をやり直す
3listen-actual実際の待受ポート起動後の LISTEN 一覧を §5 の計算式による期待値と突合期待外の待受・不応答は FAIL
4symlink-independenceシンボリックリンクの実態find -type l ＋ realpath で実体を解決し「自 worktree 内 / 共有 store / worktree 外」に分類相手側 worktree・元リポジトリを指すものは FAIL。共有 store は同一バージョンの実体を指すことを確認して許容。ただし node_modules_strategy: symlink_main のときは node_modules（およびその配下）が source_repo を指す symlink であることは想定内であり、独立性の証明対象から除外する（FAIL としない）
5dep-tree-parity依存ツリーの同一性npm ls --all 相当を正規化し、名前＋バージョン集合を比較依存再構築（npm ci 再実行）を 1 回リトライ → なお不一致は FAIL
6node-modules-healthnode_modules の健全性node_modules の存在・.bin/vite 等の実行可能性（--version 実行）・独立性（相手側/元リポジトリへの symlink でないこと）を確認。ただし node_modules_strategy: symlink_main のときは source_repo への symlink が正常状態であり、存在と .bin 実行可能性のみ確認するNG は npm ci で修復してから再検証
7runtime-parityランタイムの同一性node -v 等を両環境で実測して突合不一致は ERROR（実行基盤の問題）
8env-runtime-parity環境変数の同一性起動時に注入される env をダンプし、ポート正規化後に比較ポート以外の差分は FAIL
9env-file-parityenv ファイルの同一性両環境の .env* ファイル群（config.yml env_file_globs）をポート正規化付きで直接突合し、バンドラのみが読む変数の差分も捕捉するポート以外の差分は FAIL
10file-mode-parityファイルモード・種別git ls-files --stage のインデックス値のみで実行ビット・種別を比較する（実 FS の stat は使わない。/mnt 配下の 777 化対策）実差分として扱い、整列で揃える
11bundler-cache-freshバンドラキャッシュの新鮮さ整列が走った周回では config.yml の bundler_cache_dirs が削除済みであることを確認。node_modules_strategy: symlink_main のときは .vite が source_repo と共有されるため、整列後の削除ではなく初回 navigate 時の dependency re-optimize を許容し、巻き込まれ reload は 1 回リトライで吸収する検証不能なら起動時の強制再構築を案内
12process-independenceプロセス独立性片方を停止してももう片方が応答し続けることを確認して再起動連動停止は FAIL（プロセス・DB の共有）
13storage-isolationストレージ分離DB・キャッシュ等のパス設定が worktree 内に閉じているか走査同一の外部パスを読み書きしていれば FAIL

port-owner・symlink-independence・process-independence・storage-isolation は「同一」ではなく「独立」の検証。実体を共有した 2 環境は一致して見えて当然であり、完全一致の証明は「独立した 2 環境が同じに振る舞う」ことで初めて成立する。

7-3. 動的比較（Playwright 4 層）

両環境をそれぞれの固定ポートで起動し、routes の各画面を 4 層で突合する。総合判定は L1 ∧ L2 ∧ L4 を必須とし、L3 は閾値超過で FAIL（画像ペアを添付して人間が最終裁定できる）。

層内容判定基準
L1 起動両環境のヘルスチェック / トップ応答。playwright_exec.mode で実行手段が分岐する: mode=mcp は MCP ツールの navigate/snapshot、mode=node-script は config.yml の command_template を Bash で直接実行する。navigate 先 URL は 127.0.0.1 固定・waitUntil="load" 固定両方の成功が前提。片方でも失敗なら以降をスキップして FAIL
L2 構造アクセシビリティスナップショット（DOM 構造）をポート・URL 正規化後に比較。node-script モードでは body の ARIA snapshot 文字列（ariaSnapshot()）を突合対象とする完全一致を要求（一次判定）
L3 視覚同一ビューポートのスクリーンショット画素比較（動的領域はマスク）差分画素率が閾値以下（config.yml、既定 0.1%）。補助判定・証跡
L4 コンソールconsole error / warning の集合比較（ポートは正規化）リバース側にのみ存在するエラーで FAIL。両方に出る既存エラーは許容

L1 が失敗した場合の診断手順: ① dev サーバーの stdout/stderr を artifacts へ収集 → ② 待受ポートを再確認 → ③ 1 回だけ再起動してリトライ → それでも失敗すれば ERROR とし、収集した log のパスを添付する。

DB 内容・乱数・日時などの動的データによる誤検出には、マスク対象セレクタを設計書 frontmatter で宣言して対応する。

7-4. 整列

sync で実差分が見つかった場合（dry-run 指定がない時）、リバースコード環境をオリジナルコード環境に揃える。整列は git 操作（checkout / merge）ではなくファイル転写で行い、両ブランチの歴史を混ぜない。

── 整列の手順 ───────────────────────────────────────────────

  1. 保全    リバースコード環境のポート設定ファイル群を退避
  2. 転写    オリジナルコード環境 → リバースコード環境へミラーコピー
             （削除同期あり。除外は 7-1 の diff_exclude から
              *.lock を除いたもの。lock ファイルは転写対象に含める）
  3. 再生成  当該 <scope> のスロットから算出したリバース側ポートで
             リバースコード環境側のポート設定を再生成
  4. 再構築  node_modules_strategy で分岐する:
             npm_ci（既定）: install_command で各環境に独立インストール。
                            lock 不整合による失敗は ERROR
             symlink_main:   source_repo の node_modules へ symlink を張る
  5. 掃除    node_modules_strategy=npm_ci のみ実施。config.yml の
             bundler_cache_dirs を削除。symlink_main では .vite が
             source_repo と共有されるためスキップし、§7-2 の
             bundler-cache-fresh 但し書きに従う
  6. 再検証  Phase 3（静的）→ Phase 5（環境＋動的）へ合流し、
             収束ループを継続

§8報告フォーマット

異常系の挙動

局面挙動
プリフライト NO-GO停止して報告（source_repo / source_ref 解決不能・スロット帯の占有・スロット枯渇 等）
静的比較 FAILdry-run 指定時: 報告のみ。通常の sync: 自動整列へ進む
動的比較 FAIL（整列後も）依存再構築を 1 回だけ自動リトライ → なお FAIL なら収束ループ上限内で継続、上限到達で停止・差し戻し

setup / teardown / ERROR の早期終了時も返却ブロックの形を保ち、未実施フィールドは「未実施」と記す。

artifacts の保存規約: <artifacts_root>/<scope>/<実行日時>/ 配下に route 別のサブディレクトリを作成して保存する（artifacts_root は config.yml、既定 ~/agent-home/sessions/reverse-env）。

§9関連スキル・規約
