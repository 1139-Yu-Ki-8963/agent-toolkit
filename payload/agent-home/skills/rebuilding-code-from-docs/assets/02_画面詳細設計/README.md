# rebuilding-code-from-docs/assets/02_画面詳細設計 — レガシー画面の 16 章網羅版設計書

人手で書かれた既存コード（レガシー）の画面を文書化し、顧客提出物として保守する前提のテンプレート。コードから再生できない情報のみを持つ project-docs 版と異なり、全体像を網羅する 16 章構成とする。保守方式はコードから再生できるかどうかで 3 区分する。

本テンプレートは `rebuilding-code-from-docs` スキルの `assets/` 配下に配置され、往復検証（設計書からのコード再生成 → 元コードとの機械突合）の入力として使う。

## project-docs 版との違い

| 観点 | project-docs/02_画面基本設計 | rebuilding-code-from-docs/assets/02_画面詳細設計 |
|---|---|---|
| 対象 | greenfield：最初から AI 駆動で開発 | legacy：人手で書かれた既存コード |
| 章数 | 9 章（§1〜§9）コード外情報のみ | 16 章（§1〜§16）全体像を網羅 |
| 保守 | コードが写す章を持たない | 手保守章とコードから生成する章のハイブリッド |

## 章の保守方式 3 区分

| 区分 | 定義 | 対象章（例） |
|---|---|---|
| 手 | 業務ルール・非機能・共通準拠など、コードから再生できない章 | §1 §2 §4 §13 §14 §16 |
| 生成 | 納品時にコードからリバース生成する章（状態管理・実装契約など） | §5 §10 §15 |
| 混在 | 生成した骨格に業務的意味の列を手で重ねる章 | §3 §9 §11 §12 |

章ごとの詳細な保守方式は本文の章マップに従う（§7 は API 定義書が正であり本書は画面固有の使い方のみ保持）。

## 三点照合の運用

設計書を起こす前に、以下の 3 つを照合する。

| 照合対象 | 位置づけ |
|---|---|
| 旧設計書 | あるべき姿（ただし古い場合がある） |
| コード | 現在の実際の動作（である） |
| 有識者判定 | 乖離の分類と真の仕様の確定 |

乖離は「記録されなかった仕様変更 / 生きているバグ / 文書の腐敗」の 3 分類で処理する。検証済みの内容だけを本書に記載し、現行踏襲でバグを仕様に祭り上げない。

## 提出物ビルドの考え方

手の章の md とコードから生成した章を結合し、顧客フォーマット（Excel/Word）へ変換して納品する。md → 顧客フォーマット変換はスクリプト化できる。

## common_spec_version の運用

frontmatter の `common_spec_version` は、画面が準拠する共通設計書のバージョンを記録する。共通設計書の改版時にこの値を機械抽出し、追従未了の画面を洗い出す。

## 記入規則

- §1〜§16 は既定の章立て。往復検証は章マップの役割キーで解決するため、章を増減する場合も役割キー行は保つ
- 行識別子は意味キー必須・連番禁止
- 該当なしの章は見出しを残して「該当なし（理由）」と記入する
