# 修正指示書・最終報告の書式

Phase 8（修正指示書）と Phase 9（最終報告）の書式の正。配置先は `<画面ディレクトリ>/検証記録/<timestamp>/`（設計書リポジトリ内）。生証跡（スクリーンショット・テストログ・返却ブロック全文）は `~/agent-home/sessions/` 配下に置き、指示書・報告からパス参照する。

## 修正指示書（Phase 8）

ファイル名: `<画面ディレクトリ>/検証記録/<timestamp>/修正指示書.md`

```markdown
# 修正指示書

## 対象設計書パス

<設計書の相対パス>

## NG 一覧

| 失敗クラス | 帰着§ | 修正指示 | 根拠となる証跡パス |
|---|---|---|---|
| <ng-classification.md の失敗クラスキー> | §<番号> | <具体的に何をどう書き足すか> | <sessions/ 配下の証跡ファイルパス> |

NG が 0 件の場合は「NG なし」と明記し、上記表は省略する。

## テンプレート改善提案（あれば）

| 提案内容 | 対応する test-item-patterns.md エントリ |
|---|---|
| <テンプレート側に足すべき記載パターン> | <パターンキー> |

該当なしの場合は「テンプレート改善提案なし」と明記する。
```

### 記入時の注意

- 修正指示は「設計書の該当§に何を書き足せば実装が一意に定まったか」を具体的に書く。抽象的な「もっと詳しく書け」は禁止
- コード修正・設計書修正は本書には含めない（本書自体が指示書であり実行物ではない）
- 根拠となる証跡パスは必ず実在するファイルを指す（Phase 7 の返却ブロック・Playwright スクリーンショット等）
- 帰着§が対象設計書に未存在の場合、指示書には「当該§を新設して書き足す」内容として記述する（設計書自体は修正しない）

## 最終報告（Phase 9）

ファイル名: `<画面ディレクトリ>/検証記録/<timestamp>/最終報告.md`

```markdown
# 最終報告

## 判定

PASS / FAIL / 無効（凍結検証失敗時）

## 凍結コミットハッシュ

<Phase 6 で確定したハッシュ>

## 各 Phase の実行結果

| Phase | 結果 | 備考 |
|---|---|---|
| Phase 1 | 実施 | <基準タグ確認結果> |
| Phase 2 | 実施 | <内部整合性監査結果> |
| Phase 3 | 実施 | <白紙化コミットハッシュ> |
| Phase 4 | 実施 / [未実行] | <実装結果> |
| Phase 5 | 実施 / [未実行] | <検証結果> |
| Phase 6 | 実施 / [未実行] | <凍結コミットハッシュ> |
| Phase 7 | 実施 / [未実行] | <syncing-reverse-env 返却ブロック要約> |
| Phase 8 | 実施 | <修正指示書パスまたは「NG なし」> |
| Phase 9 | 実施 | 本報告自体 |

`[未実行]` が 0 件であることを明示する（Phase 2 で内部矛盾により Phase 4 以降をスキップした場合は、その旨と理由をここに明記した上で `[未実行]` として扱う）。

## syncing-reverse-env 返却ブロック要約

- status: <PASS/FAIL/ERROR>
- static_diff: <3 分類の集計>
- dynamic: <L1〜L4 の判定>
- env_check: <10 項目の通過数>
- hint: <あれば>

## 凍結検証結果

`scripts/check-freeze.sh` の実行結果（PASS/FAIL）。FAIL の場合は判定を「無効」とし、理由（HEAD 不一致 / 作業ツリー汚染）を明記する。

## 証跡パス一覧

| 証跡種別 | パス |
|---|---|
| Phase 7 返却ブロック全文 | <sessions/ 配下のパス> |
| Playwright スクリーンショット | <sessions/ 配下のパス> |
| テストログ | <sessions/ 配下のパス> |
```

### PASS 時の基準タグ更新

PASS かつ凍結検証 PASS の場合のみ、AskUserQuestion でユーザー承認を取り、`syncing-reverse-env` を `sync`（非 dry-run）で呼び出して基準タグを更新する。承認前に基準タグを更新してはならない。
