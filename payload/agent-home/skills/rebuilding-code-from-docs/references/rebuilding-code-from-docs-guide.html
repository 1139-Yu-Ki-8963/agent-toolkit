§1概要

リバースエンジニアリングで作成した画面基本設計書（templates/reverse-docs/02_画面基本設計/ 準拠、§1〜§16 構成）は納品物であり、「設計書だけを読んで忠実にコーディングできる」詳細度が求められる。本スキルはその品質を検証するため、設計書だけからコードを再生成し、元コードと機械突合して設計書の欠落を発見する往復検証を行う。

目的の分離: コードを直すことは目的ではない。フィードバック先は (a) 当該画面設計書の該当§、(b) 欠落が汎用的ならテンプレート自体の記載粒度改善の 2 つ。

環境管理・比較エンジンは syncing-reverse-env に全面委譲する。本スキルは値を保持せず、都度その返却ブロックから受け取る。

§2用語

用語意味
オリジナルコード環境worktree 名 original-code-<system>。設計書 frontmatter の source_repo / source_ref から用意する読み取り専用コピー（9100 番台）
リバースコード環境worktree 名 reverse-code-<system>、ブランチ feature/reverse-code-<system>。本スキルが Phase 4 で書き込む対象（9110 番台）
基準タグannotated tag reverse-baseline/<system>。基準コミットの唯一の記録
白紙化ゲートPhase 3 で reverse-code 環境の追跡ファイルを全削除するコミット。削除前に AskUserQuestion でユーザー承認を取る。以降カンニング防止のため original-code の Read を禁止する
凍結コミットPhase 6 で確定するコミットハッシュ。Phase 7 以降の比較対象点として固定される
失敗クラスng-classification.md が定義する、検出シグナルを分類した意味キー。設計書の該当§への帰着先を持つ

§3使い方

起動引数は 画面ディレクトリのみ。設計書・DESIGN.md・共通設計書・メッセージ定義書・テスト観点表は設計書 frontmatter を経由してロードする。

前提条件: 基準タグ reverse-baseline/<system> が確立済みであること。未確立の場合は Phase 1 で停止し、syncing-reverse-env（mode=setup → sync）の完走を指示する。
項目内容
起動引数画面ディレクトリ（例: 02_画面基本設計/screen-<画面ID>/）のみ
環境値の取得元syncing-reverse-env（mode=setup 冪等呼び出し）の機械向け返却ブロック
比較エンジン自前実装しない。syncing-reverse-env の mode=sync, dry-run のみ
1 起動の単位1 起動 = 1 往復。設計書修正後の再往復は再起動として扱う

§4動作フロー（Phase 1〜9）

Phase 1 入力ロード + preflight ── 基準タグ確認・syncing-reverse-env(setup)
Phase 2 設計書内部整合性監査 ── 内部矛盾あれば Phase 8 へ直行
Phase 3 白紙化ゲート ── ユーザー承認後に reverse-code 追跡ファイル全削除コミット
Phase 4 設計書からの TDD 実装 ── 唯一コードを書けるフェーズ
Phase 5 自己完結チェック ── 型チェック/テスト/スモーク（内側ループ上限3）
Phase 6 凍結コミット ── push禁止・ハッシュ確定
Phase 7 答え合わせ ── syncing-reverse-env(sync, dry-run)
Phase 8 NG分類 → フィードバック ── 修正指示書のみ・修正禁止
Phase 9 証跡保存 + 最終報告 ── check-freeze.sh・承認後に基準タグ更新提案

§5設計原則

  1. 正本一元化: 環境固有値の正は syncing-reverse-env/config.yml。画面固有値の正は設計書 frontmatter。本スキル専用の config.yml は新設しない
  2. 比較エンジンの非自前実装: 判定は syncing-reverse-env の mode=sync, dry-run 呼び出しのみ（args 全量指定・対話ゼロ）
  3. 1 起動 = 1 往復: 設計書修正後の再往復は本スキルの再起動として扱う
  4. 検証役と生成役の分離: Phase 4 実装エージェントの自己申告では判定しない。判定は Phase 7 の決定的出力のみ

§6ガード設計

ガード方式
修正禁止（Phase 7 以降）advisory + check-freeze.sh による事後機械検証（不一致 = 結果無効）
push 禁止SKILL.md に明記。凍結コミットはローカルブランチに留める
Phase 5 内側ループ上限 3 回・同一エラーシグネチャ 3 連続で発散確定
カンニング防止Phase 3 白紙化コミット + Phase 4〜Phase 6 での original-code パス Read 禁止の二重化
Phase 8 の制約: コード・設計書とも修正禁止。修正指示書の作成と test-item-patterns.md への台帳追記のみ許可する。

§7検証内容

Phase 2: 設計書内部整合性監査

scripts/audit-consistency.sh による機械チェック（a §2 機能一覧表 × 観点表の機能キー集合突合（両方向一致）、b 未記入プレースホルダ検出、c 連番キー検出 WARN）に加え、目視で §7 API 型 × §15.2 型定義、§5 状態変数 × §9/§15 の経路を突合する。詳細は references/phase-details.md。

Phase 5: 自己完結チェック

型チェック・単体/結合テスト・Playwright スモーク（reverse 側 9110 番台のみ）を実行する。内側ループは上限 3 回、同一エラーシグネチャ 3 連続で発散確定とし、発散エラー自体を設計書欠落候補として Phase 8 へ渡す。

Phase 7: 答え合わせ

syncing-reverse-env の mode=sync, dry-run 呼び出しで、静的 3 分類・env_check 10 項目・Playwright L1〜L4・hint を受領する。比較の自前実装は禁止。

Phase 9: 凍結検証

scripts/check-freeze.sh で Phase 6 の凍結コミットハッシュと現状（HEAD・作業ツリー）を突合する。 PASS = HEAD一致かつ作業ツリークリーン、 FAIL = 不一致（結果全体を無効宣言）。

§8報告フォーマット

書式の正は references/report-format.md。配置先は <画面ディレクトリ>/検証記録/<timestamp>/（設計書リポジトリ内）。生証跡（スクリーンショット・テストログ・返却ブロック全文）は ~/agent-home/sessions/ 配下に置く。
