# 失敗クラス台帳（追記型）

Phase 4・Phase 8 が参照・追記する台帳。過去の往復検証で発見された「設計書の記載粒度が足りず実装が食い違ったパターン」を蓄積し、再発を防ぐ。

## 運用ルール

- **Phase 4** は実装前に本台帳を参照し、既知の失敗クラスを先回りして設計書の該当章を確認してから実装する
- **Phase 8** は新しい失敗クラスを発見した時点で本台帳にエントリを追記する（`ng-classification.md` の表 B に定義がない・または既存の失敗クラスの新パターンの場合）
- テンプレ反映状況が「未」のまま同一失敗クラスが複数エントリで蓄積した場合、テンプレート改版（`assets/02_画面詳細設計/画面詳細設計書.md` の該当章記載例の拡充）の入力とする
- 行識別子（パターンキー）は連番禁止。内容を要約した意味語で付ける（番号からは情報を得られないため）

## エントリ形式

| パターンキー | 発見日 | 発見画面 doc_id | 検出シグナル | 失敗クラス | 設計書に足すべき記載例（before/after） | テンプレ反映状況 |
|---|---|---|---|---|---|---|

- **パターンキー**: 内容を要約した意味語（例: `絶対配置要素-条件レンダー誤変換`）
- **発見日**: `YYYY-MM-DD`
- **発見画面 doc_id**: 設計書 frontmatter の `doc_id`
- **検出シグナル**: `ng-classification.md` 表 A のキー（例: `L2構造差`）
- **失敗クラス**: `ng-classification.md` 表 B のキー（例: `表示制御方式差（常時マウント vs 条件レンダー）`）
- **設計書に足すべき記載例（before/after）**: 実際に画面構造章（既定 §3.1）や領域別仕様章（既定 §9）にどう書き足せば実装が一意に定まったかを before/after で示す
- **テンプレ反映状況**: 済 / 未 / 対象外（対象外 = 当該画面固有の事情で他画面には汎用化できない）
