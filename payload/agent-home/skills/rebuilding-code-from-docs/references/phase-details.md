# Phase 詳細（P2 突合式・P4 実装規律・P5 検証コマンド）

`SKILL.md` の Phase 2・4・5 で参照する詳細手順。SKILL.md 本体には要約のみを置き、具体的なチェック方法はここに集約する。

## スクリプト入力契約

### `scripts/audit-consistency.sh <画面ディレクトリ>`

- **設計書の特定**: `<画面ディレクトリ>/画面基本設計書.md` を第一候補とする。存在しなければ frontmatter に `type: screen-basic-design` を持つ `.md` を探す（観点表にも `doc_id` があるため、`doc_id` の有無だけでは設計書を一意に特定できない）。どちらも見つからなければ「設計書を特定できません」エラーで `exit 1`
- **終了コード**: 違反（§2×観点表キー不一致・未記入プレースホルダ）が 1 件以上で `exit 1`。違反 0 件なら WARN（連番キー疑い・観点表未検出）があっても `exit 0`
- **キー突合ルール**: §2 の各機能キーは単体/結合いずれかの観点表に出現すればよい。観点表側にしか無いキーは違反（機能一覧の記載漏れ）
- **検査 a の抽出範囲**: 観点表側の機能キー抽出は、観点表ファイル内の「## 観点表」見出し配下（次の `## ` 見出しの手前まで）にあるテーブルのみを対象にする。テストサイズ対応表・本書に書かないもの・観点の導出元マップ等のガイドテーブルは対象外（これらの先頭列まで機能キーとして拾うと誤検出するため）。（手動観点表・テストサイズ対応表・乖離分類の定義などの隣接テーブルは対象外）
- **プレースホルダ検出（検査 b）の対象範囲**: frontmatter を含むファイル全体。ただし HTML コメント（`<!-- ... -->`）内は除外する
- **突合方式**: §2 機能キー集合と観点表機能キー集合の**両方向一致**（キー集合突合）で判定する。件数比較（行数の合計と項目数を数だけ比べる）ではない

### `scripts/check-freeze.sh <reverse-code worktree パス> <凍結コミットハッシュ>`

- **検査内容**: (1) worktree の `HEAD` が指定コミットハッシュと一致するか、(2) `git status --porcelain` が空（作業ツリークリーン）か、の 2 点
- **終了コード**: 両方満たせば `exit 0`（PASS）。いずれか不一致なら違反内容を stderr に出力して `exit 1`（FAIL）

## Phase 2: 設計書内部整合性監査の突合式

`scripts/audit-consistency.sh` が機械チェックする 3 観点に加え、目視で以下を突合する。

### 突合 1: §2 機能一覧表 × テスト観点表

- §2.1 機能一覧表の「キー」列（意味キー）の集合と、frontmatter の `unit_test_sheet` / `integration_test_sheet` が指す観点表に出現する機能キーの集合を、両方向一致（キー集合突合）で突合する
- 機能一覧表にあって観点表にないキー → 「観点未整備」として記録する
- 観点表にあって機能一覧表にないキー → 「機能一覧の記載漏れ」として記録する

### 突合 2: §7 API 型 × §15.2 型定義

- §7.2 各 API のリクエスト/レスポンス型に登場するフィールド名と、§15.2 の型定義に登場するフィールド名を突合する
- フィールド名・型の両方が完全一致することを確認する。片方向のみの一致は不一致として扱う

### 突合 3: §5 状態変数 × §9/§15 の経路

- §5.3（メイン画面の状態変数）・§5.4（サブコンポーネントの状態変数）に列挙された変数名が、§9（領域別仕様）の項目定義・イベント挙動、または §15.1（ファイル分割）・§15.5（スコープ分類）のいずれかで参照されていることを確認する
- どこからも参照されない状態変数は「デッド変数疑い、または §9/§15 の記載漏れ」として記録する

### 突合 4: 意味キー連番検出

- `~/.claude/rules/semantic-key-rules/rule.md` 準拠。`grep -E '\b[A-Z]{1,4}-[0-9]+\b'` と `grep -E '^\| *ID *\|'` で連番 ID・「ID」列を検出する
- `UTF-8` / `SHA-256` 等の正当な表記は誤検知として除外する（WARN 扱い、exit 1 にはしない）

### 突合 5: 未記入プレースホルダ検出

- `<...>` 形式のプレースホルダが本文（HTML コメント外）に残っていないか検出する
- テンプレートの `<!-- ... -->` コメント内の説明文は対象外とする

## Phase 4: 実装規律

### READ → SPEC → WRITE → VERIFY

1. **READ**: 対象設計書の該当§のみを読む。他画面の設計書・オリジナルコード環境は読まない
2. **SPEC**: 単体/結合テスト観点表から、実装前にテストケースを書き出す（TDD の Red）
3. **WRITE**: テストを通す最小実装を書く（TDD の Green）
4. **VERIFY**: §15.1 ファイル一覧との過不足を機械チェックする

### import/export/Props 名の完全一致

- §15.1 の export 名、§15.3 の import 名、コンポーネント Props 名は設計書の記載と実装で **文字列完全一致** させる。省略形・別名は禁止
- 一致しない場合は Phase 8 の失敗クラス「export-import-型不一致」として記録する

### キー集合 EXTRA/MISSING 両方向一致

- 設計書に記載されたキー集合（型フィールド・状態変数・定数名等）と実装のキー集合を両方向で突合する
- **EXTRA**（実装にあるが設計書にない）と **MISSING**（設計書にあるが実装にない）の両方を検出する。片方向だけの確認は不十分

### 絶対配置+座標で表示制御する要素は常時マウント（条件付きレンダリング禁止）

- §3.6（スタイル適用パターン）や §9 の領域仕様で、絶対配置（`position: absolute` 等）+ 座標により表示/非表示を制御する要素は、DOM 上は常時マウントし CSS で表示制御する
- `{condition && <Component />}` のような条件付きレンダリングで対応する要素は、DOM 構造がオリジナルと一致せず Phase 7 の L2（構造スナップショット）で実差分となる。設計書に明記されたパターンに厳密に従う

## Phase 5: 検証コマンド詳細

- **型チェック**: `reverse-code-<system>` worktree 内で実行する型チェックコマンド（プロジェクトの `package.json` の型チェックスクリプトに従う）
- **単体/結合テスト**: 単体テスト観点表・結合テスト観点表に記載されたケースを実装したテストファイルを実行する
- **Playwright スモーク**: **reverse 側 9110 番台のみ**を対象とする。オリジナル側 9100 番台は Phase 7 の答え合わせでのみ起動する（Phase 5 時点でオリジナル側を見るとカンニングになる）

### 発散検知の記録

- 同一エラーシグネチャ（エラーメッセージの型・スタックの先頭フレーム等で識別）が 3 連続で再発した場合、発散確定とする
- 発散確定したエラーは「設計書に記載が足りず実装が収束しない箇所」の候補として、エラー内容そのものを Phase 8 へ引き継ぐ
