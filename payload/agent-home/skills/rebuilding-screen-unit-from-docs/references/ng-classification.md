# NG 分類（検出シグナル → 失敗クラス → 章帰着）／stage1 版

P7 で使う分類表。stage2（`rebuilding-code-from-docs`）の同名ファイルを、単体・ファイル単位・5 計測由来のシグナルに翻案したもの。表 A の由来は「P5 の 5 計測」「P2 の内部整合監査」「P5 の発散確定」の 3 系統に限定する（stage1 は Playwright L2/L3/L4 の動的比較を行わないため対象外）。表 B は stage2 の章帰着マップをそのまま流用する（章の役割キーは画面詳細設計書.md の全体構成に依存し、検証粒度〔単体/結合〕によって変わらないため）。

キーは連番禁止。内容を要約した意味語で付ける（番号からは情報を得られないため）。

## 表 A: 検出シグナル → 失敗クラス

| 検出シグナルキー | 由来 Phase | 判定内容 | 遷移先 |
|---|---|---|---|
| import diff非zero | P5（5 計測） | `import_diff_lines > 0` | 表 B の「export-import-型不一致」へ |
| style diff非zero | P5（5 計測） | `style_diff_lines > 0` | 表 B の「スタイル数値差」へ |
| 実質diff超過 | P5（5 計測） | `substantive_diff_lines > 20` | 表 B の該当失敗クラスへ（1 件が複数クラスにまたがる場合は目視で分解する） |
| 単体テスト仕様不合格 | P5（単体テスト仕様の検査） | 禁止パターン・import・コンポーネント/API 利用のいずれかが不合格 | 表 B の該当失敗クラスへ |
| P2内部矛盾 | P2（白紙化前の内部整合監査） | `audit-consistency.sh` 通常モードの違反検出（機能一覧×観点表キー不一致・未記入プレースホルダ等） | 表 B の該当失敗クラスへ。白紙化・実装を経ず直接 P7 へ |
| P5発散エラー | P5→P6→P2 の再投入ループ | 同一差分集合（import diff・style diff・実質 diff の内容シグネチャ）が 2 連続で再発し発散確定 | 差分内容から表 B の該当失敗クラスを推定 |

## 使い方

1. P5 の 5 計測出力・単体テスト仕様の検査結果、P2 の内部整合監査結果、P5→P6→P2 ループの発散確定エラーを検出シグナルとして集める
2. 表 A で検出シグナルを失敗クラスへ写像する
3. 表 B で失敗クラスから一次帰着（役割・既定§）を決め、修正指示書（`references/phase-details.md` の「修正指示書・検証記録の書式」参照）の該当欄に記入する
4. テンプレ昇格条件に合致する場合、stage2 の `~/agent-home/skills/rebuilding-code-from-docs/references/test-item-patterns.md` に新規エントリを追記し「テンプレ反映状況: 未」で記録する（stage1 専用の台帳は新設せず、stage2 の台帳を単体・結合共通の正本として使う）
5. 「正の分担違反」（スタイル数値差・文言差）は、設計書本体ではなく `DESIGN.md` / メッセージ定義書側の記載不足として指摘する（章マップの正の宣言に従う）
6. P2 内部矛盾または未収束（発散確定）の場合、「stage2 結合検証時の要注意ファイル」として検証記録に記録し、stage2 実行時に参照できるようにする
