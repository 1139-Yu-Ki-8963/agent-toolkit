# 設計書からの単体ファイル再構築スキル（stage1）

リバース設計書（`assets/02_画面詳細設計/` 準拠、章の役割キー 16 種で構成）の往復検証を、**「1 起動 = 1 ファイル・単体観点・from-zero」** で行う stage1 スキル。兄弟の `rebuilding-code-from-docs`（画面単位・結合観点・e2e の stage2）とは役割が異なる。stage1 で設計書の記載粒度をファイル単位でブートストラップし、合格した画面から stage2（画面単位の結合検証）へ引き継ぐ。

コードを直すことは目的ではない。P5 の差分は「コードの誤り」ではなく「設計書の記載不足」として扱い、フィードバック先は当該設計書の該当章（役割キー）。設計書に対象ファイルの記載自体が無い場合は、ひな形から新規作成してよい（from-zero）。

比較エンジンは自前実装する。`syncing-reverse-env` の重い環境同期・基準タグ運用には委譲しない（stage1 は 1 ファイル規模の軽量往復であり、画面単位の環境同期は過剰）。

## 使用タイミング

- リバース済み設計書のファイル単位往復検証・from-zero での設計書品質ブートストラップを行いたいとき
- 起動引数は **画面ディレクトリ + 対象ファイルパス**（実装契約章のファイル分割表〔既定 §15.1〕の 1 エントリ）
- 前提: 対象ファイルが §15.1 のファイル分割表に列挙されていること。列挙が無い場合は from-zero として扱い、P1 で新規追加を宣言してから進める
- 画面単位の結合・e2e 検証（Playwright スモーク・複数ファイル横断の突合）が必要なときは stage2（`rebuilding-code-from-docs`）を使う。本スキルは単体（small）観点のみを扱う

## 設計原則

1. **正本一元化**: 環境固有値（ポート・worktree パス）には触れない。画面固有値の正は設計書 frontmatter。本スキル専用の config.yml は新設しない
2. **比較は自前の 5 計測で行う**。`scripts/measure-file-diff.sh` の決定的出力のみを判定根拠とし、`syncing-reverse-env` の環境同期・基準タグ運用には委譲しない
3. **1 起動 = 1 ファイルの往復**。他のファイルへは進まない。画面内の別ファイルを検証する場合は本スキルの再起動として扱う
4. **カンニング防止の二層**（詳細は次節）

## カンニング防止（二層）

生成物が「正解を見て書いた」ものにならないよう、2 つの独立した層で防止する。

### 層 1: ファイル内の答え

対象ファイルを P2 で `git rm` により白紙化する。削除直前の HEAD コミットを「原本参照コミット」として記録し、以降 P5 まで対象ファイルの内容は作業ツリー上に存在しない。原本の内容は P5 でのみ、原本参照コミットから `git show <commit>:<path>` で一時ファイルへ抽出する。

### 層 2: セッション文脈の汚染

P4（コード生成）は必ず `worker-sonnet` へ隔離委任する。prompt には「対象設計書の該当章 + 共通部品（DESIGN.md・共通設計書・メッセージ定義書の関連部分）+ 単体テスト観点表の該当行 + P3 で書いたテスト」のみを渡し、**元コードのパス・内容を一切含めない**。メインエージェント自身も元コードを P5（差分比較）でのみ Read する。ループの各周でサブエージェントを新規起動し、前周の生成文脈を持ち越さない。これにより、メインが P5 で原本を見て汚染された状態になっても、次周の生成コンテキストには漏れない（新規サブエージェントは前周の会話を一切継承しない）。

## 基本ワークフロー（P1〜P7）

### P1: 入力ロード + preflight

起動引数（画面ディレクトリ + 対象ファイルパス）から設計書・`DESIGN.md`・共通設計書・メッセージ定義書・単体テスト観点表を frontmatter 経由でロードする。設計書は必須。他は frontmatter にキーがあれば必須（キーがあるのに実体が無ければ preflight エラー）、キー自体が無ければ「不在」と記録して続行する。`scripts/audit-consistency.sh --list-contract-files <画面ディレクトリ>` で実装契約章のファイル分割表（既定 §15.1）の対象ファイル一覧を取得し、起動引数の対象ファイルパスがその一覧に含まれるか確認する。含まれない場合は **from-zero**（設計書に当該ファイルの記載が無い状態）として扱い、§15.1 表・関連章への新規追加をひな形（`assets/02_画面詳細設計/画面詳細設計書.md` のテンプレ構造）に沿って宣言してから P2 へ進む。

元コード Read 可否: **可（存在確認のみ・字面読み禁止）**。`git ls-files <path>` 等でパスの実在を確認するに留め、内容を Read して転記してはならない。

完了条件: 対象ファイルが §15.1 リストに含まれることを確認済み（または from-zero 新規追加を宣言済み）・設計書一式がロード済み

### P2: 白紙化ゲート

白紙化の前に `scripts/audit-consistency.sh <画面ディレクトリ>` を通常モードで実行し、対象ファイルに関連する章の内部整合性を確認する。from-zero で新規宣言した章（P1 参照）も本監査の対象に含める。監査で矛盾が出た場合は、宣言した章の記載を補ってから再監査する（それでも解消しなければ下記の通り P7 へ離脱する）。違反（機能一覧×観点表のキー不一致・未記入プレースホルダ等）があれば白紙化・実装を経ず、直接 P7（NG 分類）へ直行する（欠陥のある設計書からの生成は往復検証として無意味なため）。この離脱は P7 で「設計書内部矛盾」クラスとして扱う。

内部整合性に問題が無ければ、対象ファイル **1 件のみ** を白紙化する。削除コミットを打つ**前に** AskUserQuestion でユーザー承認を取る。承認確認文言は「設計書検証のため対象ファイル 1 件 `<path>` のみを白紙化します（他のファイルは削除しません）。よいですか？」の形式とし、(A) 白紙化して続行 / (B) 中止 の 2 択を提示する。(B) の場合は本スキルを中止する。

**全消しフォールバックは禁止**（対象ファイルの特定に失敗しても、無関係なファイルや画面ディレクトリ全体を削除してはならない、fail-closed）。承認後、削除直前の HEAD コミットハッシュを「原本参照コミット」として記録し、対象ファイルを `git rm` で削除してコミットする（コミットメッセージ: `【リファクタ】<対象ファイル> を白紙化（単体往復検証）`）。以降 P5 まで **原本参照コミットの内容の Read を禁止**する（カンニング防止層 1）。

元コード Read 可否: **可（対象特定のため。字面読み禁止は維持）**

完了条件: 内部整合性の監査結果が記録済み（矛盾があれば P7 へ直行）・ユーザー承認取得済み・原本参照コミットハッシュ記録済み・白紙化コミット完了

### P3: 単体テスト仕様作成

単体テスト観点表から、対象ファイルが担う観点行を全て拾い、テストケースを先に書き出す（TDD の Red、テストが先）。対象ファイルに対応する観点が 1 件も無い場合は、P1 の from-zero 宣言と同様に扱い、観点表側にも最低限の行を意味キー規約（連番禁止・内容要約キー）に沿って追記してから進める。

from-zero では設計書側の記載も薄く、原本参照禁止のため P3 テストの妥当性が P5 まで判明しない。テストは「設計書に記載された宣言的契約（型・API 契約・ロジック構造・定数値〔静的に決まるもの〕）」に限定して書き、原本の挙動を推測して書かない。P5 の実質 diff に現れた差分は、観点表・設計書への追補材料として P6 で扱う。

**実測が要る断定は stage1 で確定しない**。初期表示値・DOM 配置順序・要素位置・レイアウトなど、実行時の観測（Playwright 実測等）でしか正しく決められない事実は、from-zero で目視転記も推測もせず、設計書・観点表に `[stage2で実測]` プレースホルダとして残す。stage1 が確定させるのは、静的に決定可能な宣言的契約（型・import・export・Props 名・API 契約形・ロジック構造・定数値）に限定する。理由: stage1 は Playwright を持たない軽量往復であり、実測できない構造断定を字面読み（カンニング違反）や推測で埋めると往復検証の土台が崩れるため。実測が要る断定は Playwright を持つ stage2（`rebuilding-code-from-docs`）の担当とする。

元コード Read 可否: **禁止**。白紙化済みで作業ツリー上に実体が無いことに加え、`git show` 等の間接参照も本フェーズでは禁止する。

完了条件: 対象ファイルの観点に対応するテストケース一式が書き出し済み（RED 状態）

### P4: サブエージェント生成

**唯一コードを書けるフェーズ**。`worker-sonnet` へ隔離委任する（1 エージェント = 1 ファイル）。prompt 骨格・禁止事項の詳細は `references/phase-details.md` を参照。

元コード Read 可否: **禁止**（メイン・サブエージェントとも）

完了条件: 対象ファイルが生成され、P3 のテストがローカルで実行可能な状態になっている

### P5: 差分比較（5 計測）

ここで初めて原本参照コミットの内容の Read を許可する。`git show <原本参照コミット>:<対象ファイルパス>` でスクラッチ領域へ一時ファイルとして抽出する（worktree・設計書リポジトリを汚染しないよう `$CLAUDE_JOB_DIR/tmp/` 配下に置く）。

1. **単体テスト仕様の検査**: P3 で書いたテストを P4 の生成物に対して実行する（禁止パターン・import・コンポーネント/API 利用の確認）。全項目合格が必須
2. **5 計測**: `scripts/measure-file-diff.sh <generated-file> <original-tmp-file>` を実行し、import diff / style diff / 全体 diff / 実質 diff / verdict を取得する

差分は「コードの誤り」ではなく「設計書の記載不足」として解釈する。この段階でのコードの手修正は禁止する。

`measure-file-diff.sh` の verdict は生シグナルとして扱う。差分行のうち `[stage2で実測]` プレースホルダに該当する実測事実（初期表示値・DOM 配置順・要素位置・レイアウト起因の style/JSX 差分）は「stage2 委譲」として分類し、**収束判定の対象から除外**する。収束（PASS/FAIL）は、静的に決定可能な宣言的契約由来の差分（import・型・export・Props 名・定数・ロジック）だけで判定する。除外した実測差分は P7 の stage2 申し送りに記録する。

元コード Read 可否: **可**（本フェーズのみ）

完了条件: 単体テスト仕様の検査結果 + 5 計測の結果（省略なし）+ 差分の stage2 委譲/静的契約の分類結果が記録済み

### P6: 設計書修正

P5 の差分を 4 分類に振り分け、修正先の章を決定する。

| 差分種別 | 修正先 |
|---|---|
| import | 実装契約章の依存（既定 §15.3） |
| style | `DESIGN.md`（画面構造章のスタイル適用パターン経由、既定 §3.6） |
| logic | データフロー（既定 §6）・API通信（既定 §7） |
| JSX | 画面構造（既定 §3）・領域別仕様（既定 §9） |

**実測断定に該当する差分（初期表示値・DOM 順・位置・レイアウト）は確定修正せず `[stage2で実測]` を維持する**。これらは設計書へ確定値を書き込まず、収束判定の分母からも除外し、stage2 実測での確定に委ねる。P6 で確定修正するのは静的な宣言的契約由来の差分のみ。

修正の順序は「テストが先」の原則を維持する: まず単体テスト観点表に不足していた観点（境界値・異常系等）を追記し、その後に設計書へ記載を追加する。

修正後、単体テスト仕様の検査または 5 計測のいずれかが不合格であれば **P2 へ再投入**する（原本参照コミットは変わらないため再取得不要。対象ファイルを再度白紙化し P3〜P5 をやり直す）。上限 5 回。

元コード Read 可否: **禁止**。設計書修正は P5 の差分結果（数値・パターン）のみを根拠に行い、原本コードを再度 Read しない。

完了条件: 差分 4 分類・修正内容（または P2 再投入の判断）が記録済み

### P7: NG 分類 + 報告

収束（5 計測 PASS かつ単体テスト仕様の検査 全項目合格）/ 未収束（上限 5 回到達で発散確定）を判定する。収束時は `references/ng-classification.md` の該当有無を確認し「NG なし」または軽微な修正指示書を出力する。未収束時は検出シグナルを `references/ng-classification.md` の表で失敗クラスに分類し、修正指示書を作成する。未収束の内容は「stage2 結合検証時の要注意ファイル」として検証記録に記録し、stage2（`rebuilding-code-from-docs`）実行時に参照できるようにする。

P2 の内部整合違反による早期離脱の場合は、白紙化・生成を経ていないため「設計書内部矛盾」クラスの修正指示書を作成し、収束/未収束の判定対象外（実装検証未実施）として明示する。

元コード Read 可否: **禁止**（報告作成は P5 の結果のみを参照する）

完了条件: 修正指示書または「NG なし」の明示が完了・収束/未収束の判定が完了

## 5 計測の合格判定表

| 計測 | 合格条件 |
|---|---|
| 単体テスト仕様の検査（禁止パターン・import・コンポーネント/API 利用） | 全項目合格 |
| import の diff | 0 行 |
| スタイル定数の diff | 0 行 |
| ファイル全体の diff | 行数を出力（参考値・合格判定には使わない） |
| 実質 diff（コメント・空行除外） | 20 行以下 |

**import diff だけでの合格判定は重大違反**とする。5 計測はすべて出力し、いずれか 1 つでも省略してはならない。

## ループ設計

| 要素 | 内容 |
|---|---|
| 反復条件（内側） | P5 の判定（単体テスト仕様の検査 または verdict）が FAIL の場合、P6 で設計書を修正した上で P2（白紙化）へ再投入し、P3→P4→P5 をやり直す |
| 上限回数（内側） | 5 回 |
| 停止条件（内側） | 下表「停止条件の優先順位と遷移先」を参照 |
| 反復条件（外側） | なし（1 起動 = 1 ファイルの往復。他のファイルへは進まない） |
| 上限回数（外側） | 1 |
| 検証役の分離 | P5 の判定は `scripts/measure-file-diff.sh` の決定的出力のみで行う。P4 生成エージェントの自己申告では判定しない |

### 停止条件の優先順位と遷移先

3 つの停止条件は同時に成立しうるため、優先順位を明記する。

| 優先順位 | 停止条件 | 判定基準 | 停止後の遷移先 |
|---|---|---|---|
| 1 | 収束停止 | 単体テスト仕様の検査 全項目合格 かつ verdict=PASS。収束判定は静的な宣言的契約由来の差分のみを対象とし、`[stage2で実測]` 委譲差分（初期表示値・DOM 順・位置・レイアウト由来）は分母から除外する | P7 へ（収束として報告） |
| 2 | 発散検知 | 同一差分集合（import diff・style diff・実質 diff の内容シグネチャ）が 2 連続で再発 | 上限回数（5 回）への到達を待たず、上限より優先して即座に P7 へ。P6 の設計書修正は行わず打ち切り、未収束・発散確定として分類する |
| 3 | リソース上限 | 5 回到達 | P7 へ（未収束として分類） |

## 重要な注意事項

- P4 以外でコードを書いてはならない
- P2〜P5 の間（P5 は取得時のみ）、原本参照コミットの内容の Read を禁止する（カンニング防止層 1）
- P4 のサブエージェント委任 prompt に元コードのパス・内容を絶対に含めない（カンニング防止層 2）。ループの各周で新規サブエージェントを起動し、前周の生成文脈を持ち越さない
- push は一切禁止。白紙化コミット・修正コミットはローカルブランチに留める
- 進捗は Step 単位で TaskCreate/TaskUpdate する（一括登録しない）
- P7 でコード・設計書とも修正禁止。修正指示書の作成のみ許可する（P6 での収束を目指す修正とは異なり、P7 は最終申し送り）

## Gotchas

- P6 の修正が複数周にわたる場合、追加する観点行は意味キー規約（連番禁止・内容要約キー）を守る
- 検証記録の配置先は `<画面ディレクトリ>/検証記録/単体-<対象ファイル basename>/<timestamp>/` とし、他ファイルの検証記録・stage2 の検証記録と混在しないようファイル名で分離する
- P5 の発散確定（同一差分集合 2 連続）は失敗ではなく手がかり。差分内容自体を P7 の NG 分類材料として使う
- from-zero（§15.1 に対象ファイルが存在せず設計書側にも記載が無い）場合、P1 で新規追加を宣言してから P2 以降に進む。ひな形は `assets/02_画面詳細設計/画面詳細設計書.md` のテンプレ構造に従う
- P7 で未収束の場合、「stage2 結合検証時の要注意ファイル」として検証記録に残す。stage2（`rebuilding-code-from-docs`）の Phase 1 で当該画面を検証する際に参照できるようにする
- 実測が要る構造断定（初期表示値・DOM 順・位置・レイアウト）を stage1 の from-zero で埋めようとしない。字面読みも推測も禁止で入力源が無いため、`[stage2で実測]` プレースホルダに留め、実測は stage2 に委ねる（このスコープ境界を破ると T-A-1-11 型の誤記載事故＝構造の見落としによる誤った断定を、検証の土台側に作り込むことになる）

## 設計判断

### measure-file-diff.sh

**必要性**: P5 の 5 計測のうち import diff・style diff・全体 diff・実質 diff の 4 項目は、複数のヒューリスティック抽出（import 行・スタイル定数行・コメント/空行除外）を組み合わせた決定的検査であり、毎回手書きの diff コマンド列で実行すると抽出条件がぶれ再現性が失われる。「import diff だけでの合格判定は重大違反」とする規約上、4 項目を同一スクリプトで一括算出し、省略を構造的に防ぐ必要がある。

**代替案を採用しなかった理由**:
- Bash 直叩き: 4 種の diff 抽出条件（import 行パターン・スタイル定数パターン・コメント/空行除外パターン）を都度手書きすると、一部の計測を省略しがちになり「import diff だけで合格判定」という重大違反を誘発する
- 既存 Makefile ターゲット拡張: `~/agent-home/` 配下には Makefile がない
- package.json scripts 追加: グローバルスキル用途でありプロジェクトの package.json に属さない

**保守責任者**: 人手（ユーザー）。スタイル定数抽出パターン変更時に更新する。

**廃棄条件**: 本スキル廃止時、または stage2 の `syncing-reverse-env` 比較エンジンに stage1 相当の軽量 diff モードが統合された時
