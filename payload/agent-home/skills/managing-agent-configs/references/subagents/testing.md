# サブエージェント実機検証手順

## 原則

- test モードは **必ず新規サブエージェントをディスパッチ** して実行する。セルフ再読は著者バイアスが入るため禁止
- ディスパッチできない環境では「empirical evaluation skipped: dispatch unavailable」と明示してスキップ

## 検証ワークフロー

### Step 1: 評価シナリオの設計

対象 subagent の責務に応じて 2〜3 のシナリオを設計する。

| シナリオ種別 | 目的 | 例 |
|---|---|---|
| 正常系 | 期待する専門性で応答するか | brain に「このタスクを分解して」と依頼 |
| 境界系 | SKIP すべきタスクを適切に拒否するか | worker-haiku に判断が必要なタスクを渡す |
| 出力形式 | 定義したフォーマットに沿うか | researcher の「情報源・発見事項・信頼度」構造 |

### Step 2: 要件チェックリストの作成

シナリオごとに固定のチェックリストを作成する。一度決めたら途中で変更しない。

```markdown
## 要件チェックリスト
1. [critical] <最低限満たすべき項目>
2. [critical] <最低限満たすべき項目>
3. <望ましい項目>
4. <望ましい項目>
```

### Step 3: サブエージェントのディスパッチ

Agent ツールで新規サブエージェントを起動する。

```
Agent(
  subagent_type: "<target-subagent-name>",
  prompt: "<シナリオの説明と実行指示>"
)
```

呼び出し規約:

```markdown
あなたは <対象サブエージェント名> として以下のシナリオを実行してください。

## シナリオ
<実世界のタスクを 1 段落で記述>

## タスク
1. シナリオに従って作業を実行
2. 完了後、以下のレポート構造で結果を返す

## レポート構造
- 成果物: <何を出力したか>
- 判断根拠: <なぜそう判断したか>
- 出力形式: <定義通りの構造に沿っているか>
- 境界判断: <SKIP すべきタスクをどう処理したか>
```

### Step 4: 結果の評価

サブエージェントの応答を要件チェックリストと突合する。

| 評価軸 | 合格基準 |
|---|---|
| 専門性 | 定義された責務に沿った応答をしている |
| 出力形式 | frontmatter で定義したフォーマットに沿っている |
| 境界遵守 | SKIP 対象のタスクを拒否し、代替を案内している |
| ツール使用 | `tools` に列挙されたツールのみを使用している |

### Step 5: 反復（必要な場合）

| 条件 | 対応 |
|---|---|
| [critical] 未達 | 定義ファイルを修正して再ディスパッチ |
| 出力形式の逸脱 | 本文の出力フォーマット節を具体化して再ディスパッチ |
| 境界判断の失敗 | description の SKIP を具体化して再ディスパッチ |

収束基準: 2 回連続で全 [critical] 項目が合格。

打ち切り基準: 3 回反復しても [critical] 未達が解消しない場合、定義の設計自体を見直す。

## 検証レポートのフォーマット

```markdown
## 検証結果: <subagent-name>

### シナリオ 1: <名前>
- 結果: PASS / FAIL
- [critical] 項目1: ○ / ×
- [critical] 項目2: ○ / ×
- 項目3: ○ / ×
- 備考: <特記事項>

### シナリオ 2: <名前>
...

### 総合判定
- 合格: 全シナリオの [critical] が全て ○
- 要修正: [critical] × あり → 修正箇所を明示
- 設計見直し: 3 回反復後も未達 → 根本的な再設計が必要
```

## 失敗パターン台帳

反復で発見された典型的な失敗を記録し、同じ問題の再発見を防ぐ。

| パターン | 症状 | 原因 | 修正方法 |
|---|---|---|---|
| 責務越境 | brain がコードを直接編集する | tools に Write/Edit が含まれている | tools から除去 |
| 判断放棄 | worker-haiku が「判断できない」と返す | 判断が必要なタスクを渡している | worker-sonnet に切替 |
| 出力散逸 | 定義フォーマットに沿わない自由記述 | 出力フォーマット節が曖昧 | 具体的な構造を明示 |
| MCP 不通 | researcher が「ツールが見つからない」 | headless 環境で MCP 未接続 | 環境前提を明示 |
| 検証偽装 | 実行していないコマンドを「実行して確認した」と記述する（例: tool 使用 0 回なのに「find で件数を確認」） | prompt 内の数値をそのまま反復し、検証済みと装う | 評価時に usage の tool 使用回数と報告内容を突合する。プロンプトに「実行していないコマンドを実行済みとして書かない」を明記 |
