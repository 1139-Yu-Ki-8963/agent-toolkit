レガシー画面一覧生成スキル
generating-screen-list-for-reverse-docs

レガシーコードベースを画面単位にグルーピングし、画面一覧.HTML を生成する。仕事は画面一覧.HTMLの作成のみで、設計書の雛形展開・生成は行わない。

こんな人向け
既存（レガシー）コードベースをリバースエンジニアリングで文書化したい人。画面の境界がまだ確定していない状態から着手したい人。
こんな場面で使う
設計書を作る前に、対象コードのどのファイル群が1画面を構成するかを機械的に洗い出したいとき。
対応 OSmacOS・Windows WSL（Linux）（Bash 3.2 互換・BSD/GNU 両対応で実装。Windows WSL 環境での動作を別環境で確認済み）
検証状況実プロジェクト（oradora-battle-base〔Vite + React Router〕・kakeibo〔Next.js〕等）で動作確認済み。Windows WSL 環境でも別環境で動作確認済み。managing-agent-configs の test モードによる実機検証を経て運用中
依存標準的な POSIX ツール（find / grep / sed / awk）に加え、整合検証（validate-manifest.sh）・画面一覧.HTML生成（build-screen-list.sh）で jq を使用

§1概要

レガシー（既存）コードベースを画面単位にファイルをグルーピングし、その結果を 画面一覧.HTML として出力する。

単独動作: syncing-reverse-env（2 環境同期・基準タグ確立）には依存しない。ソースディレクトリと出力先ディレクトリを直接指定して起動する。
仕事は画面一覧.HTMLの作成のみ: 設計書（画面詳細設計書.md 等）の雛形展開・生成・記入は一切行わない。他のスキル・人手による作業の入力として画面一覧.HTMLを渡すところまでが本スキルの範囲。

§2用語

画面キー（screenKey）
意味キー規約に準拠した画面の一意識別子。ルートの静的セグメントから導出し、連番サフィックスは使わない。
confidence
画面境界検出の確からしさ。high（ルーティング定義との直接一致）／medium（正規表現ヒューリスティック）／low（慣習ディレクトリからの推測）の3段階。

§3使い方（起動引数）

引数内容
ソースコードディレクトリ探索対象。モノレポの場合は対象アプリのディレクトリを直接指定する（自動判別なし）
出力先ディレクトリ画面一覧.HTMLの書き出し先

0件検出時は AskUserQuestion で結果を報告して停止する（手動リスト入力は行わない）。

§4動作フロー（Phase 1〜4）

固定（スキーマ契約・整合検証・HTML生成）と可変（抽出）を分離する。抽出はPhase 2 Step 1の分岐判定で「組み込みパス」「カスタム抽出パス」のいずれかを選ぶ。

Phase 1: スタック・画面規約の特定（TaskCreateで全Phase登録）
  Step 1: package.json / lockファイルからフレームワーク・ルーターを確定
  Step 2: ルーティング定義の所在を特定(useRoutes(router)等は定義ファイルまで追跡)
  Step 3: 画面ID命名パターン・View切替関数・メニュー定義有無を調査
  Step 4: 除外パターン(tests/stories/mocks等)をlsで確認し確定
  Step 5: 検出戦略宣言(strategy JSON)を作成しAskUserQuestionで承認取得

Phase 2: 戦略に基づく抽出
  Step 1: 抽出方式を分岐判定 ─┬─ 適合 ──→ 組み込みパス
                              └─ 不適合 →  カスタム抽出パス
  ┌─ 組み込みパス ─────────────────────────────────┐
  │ Step 2: detect-screens.sh source-dir manifest.json    │
  │   --strategy-json strategy.json                       │
  │   [--screen-id-regex re] [--view-switch-pattern re]   │
  │   [--exclude pattern]                                 │
  │   0件ならハード停止(exit 3)。捏造しない               │
  └─────────────────────────────────────────────────┘
  ┌─ カスタム抽出パス ─────────────────────────────┐
  │ Step 2: Claudeが戦略宣言に沿った専用手順を         │
  │   Bash/Grep/Readで実行し、スキーマ準拠の            │
  │   manifest JSONをWriteで出力                        │
  └─────────────────────────────────────────────────┘
  Step 3: diagnostics確認。entryFile集中警告等→カスタムパスへの切替検討(Step1へ)

Phase 3: 整合検証（機械実行・抽出方式に非依存）
  Step 1: validate-manifest.sh manifest.json を実行。全7項目PASSが完了条件
  Step 2: FAIL時は修正しStep1を再実行。3回失敗でPhase 2 Step 1へ差し戻し

Phase 4: 画面一覧.HTML 生成
  Step 1: build-screen-list.sh manifest.json output-dir/画面一覧.html
  手作業でのプレースホルダ置換は禁止。決定的スクリプト経由のみ

§5画面キー生成規則

semantic-key-rules（連番禁止）に準拠したアルゴリズム。

手順内容
1ルートの静的セグメントのみ抽出（動的パラメータ・ワイルドカードを除外）
2末尾1セグメントを仮キーとする
3衝突する画面があれば、末尾からのセグメント数を1つずつ増やして再判定
4全セグメントを使っても衝突する場合のみ、エントリディレクトリのパスで具体化する（連番サフィックスは使わない）
5ルートが / または静的セグメント無しの場合は top

§6検出戦略宣言の作り方

Phase 1でエージェントが調査した内容をもとに、画面単位の定義・screen-id-regex・view-switch-patternの3点を宣言し、AskUserQuestionで1回だけ確認を取る。汎用の正規表現をプロジェクトを問わず当てるのではなく、対象プロジェクトの実際の命名規約・ナビゲーション方式から導出する。

項目決め方例
画面単位の定義1ファイル=1画面か、複数ファイルの集合（コンポーネント分割）かをディレクトリ構成から判断する「エントリファイル＋直下components/を1画面とする」
screen-id-regexファイル名に画面IDが埋め込まれている場合、そのIDを抽出する正規表現を決める。命名パターンが無ければ「なし」と明記するT-[A-Z]-\d-\d+（T-A-1-113.tsxから画面IDを抽出）
view-switch-patternrouter.push型ではなく状態切替関数でView遷移するSPA構成の場合、その切替関数呼び出しを検出する正規表現を決めるsetEditView\(['"]([\w-]+)['"]\)

画面規約が不明瞭な場合は空の宣言（「なし」）で進めてよいが、後続Phaseの検出精度は confidence に反映される。

カスタム抽出パスへの切替

Phase 2 Step 1で組み込み検出器（Next.js App/Pages Router・React Router（useRoutes含む）・慣習ディレクトリ）がPhase 1の調査結果と適合しないと判断した場合、カスタム抽出パスを取る。

組み込みが合わない兆候内容
未知のルーティング方式useRoutes以外の独自ルート定義（設定オブジェクト配列・element属性からの動的解決等）で組み込み検出器のパターンにヒットしない
entryFile集中診断Phase 2 Step 3のdiagnosticsで特定の1〜2ファイルに大量の画面候補が集中する警告が出る（境界を取り違えている兆候）
path抽出0件組み込み検出器を実行しても静的なpathリテラルが1件も抽出できない

カスタム抽出時の手順は次の順で進める。

  1. Phase 1 Step 5で検出戦略宣言（extractionMethod: custom）をAskUserQuestionで承認する
  2. Claude自身がPhase 2 Step 2でBash/Grep/Readを用い、宣言した専用手順（element属性のviewId/pageIdから物理ファイルを組み立てる・カスタムルート配列のJSON解析等）を実行し、スキーマ準拠のmanifest JSONをWriteする
  3. Phase 3のvalidate-manifest.shを通過させる。抽出者（組み込みスクリプトかClaudeか）に関わらず同一基準で検証されるため、通過しないマニフェストのままPhase 4へは進めない

§7画面一覧.HTML フォーマット

テーブル列: 画面ID／画面キー／区分／画面名／ルート／検出方式／confidence／共有／構成ファイル数／主ファイル。<script type="application/json" id="screen-manifest"> に構造化データを埋め込み、人間可読と機械可読を両立する。外部CDN不使用の単一ファイル自己完結。

列内容
画面IDPhase 1の screen-id-regex で抽出したID（該当なしなら空欄）
区分kind（route / embedded-view / unresolved）
共有sharedWith / clusterId の有無。共有クラスタに属する画面は screenNameGuess を表示しない
正本の位置づけ: 画面一覧.HTML は後続フェーズが読む計画書ではなく、実施済み事実のレポート。画面詳細設計書.md の単位を正確に分けるための正本として機能する。

§8Gotchas・スコープ外

  • validateゲート: Phase 4の build-screen-list.sh は内部で validate-manifest.sh を再実行する。Phase 3の検証を経ないマニフェストからはHTMLを生成できない（抽出方式に依らず強制）
  • validate-manifest.sh・build-screen-list.sh はいずれも jq に依存する。未インストール環境では事前に導入する
  • import グラフ解析なし（組み込み検出器は物理的同居のみでファイル収集。カスタム抽出パスは戦略宣言に沿った収集）
  • 組み込み検出器は React Router の深いネスト親子パス合成に非対応（フラット抽出のみ）、useRoutesと2段階importまでのimport追跡には対応
  • 画面名は技術名の機械整形のみ（業務的な日本語意訳はしない）
  • 動的に構築されるルート文字列は組み込み検出器では検出不可（静的リテラルのみ）
  • モノレポ・複数アプリ構成の自動判別なし
  • 0件検出時はハード停止。AskUserQuestionでの手動リスト入力にフォールバックしない
  • 設計書（画面詳細設計書.md 等）の雛形展開・生成・記入は一切行わない（本スキルのスコープ外）
