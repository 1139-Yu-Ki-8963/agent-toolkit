レガシー画面一覧生成スキル
generating-screen-list-for-reverse-docs

レガシーコードベースを画面単位にグルーピングし、画面一覧.HTML を生成する。仕事は画面一覧.HTMLの作成のみで、設計書の雛形展開・生成は行わない。

こんな人向け
既存（レガシー）コードベースをリバースエンジニアリングで文書化したい人。画面の境界がまだ確定していない状態から着手したい人。
こんな場面で使う
設計書を作る前に、対象コードのどのファイル群が1画面を構成するかを機械的に洗い出したいとき。
対応 OSmacOS（Bash 標準ツールに依存。Linux 移植時は sed 構文の差異に注意）
検証状況managing-agent-configs の test モードによる実機検証を経て運用開始
依存標準的な POSIX ツール（find / grep / sed / awk）のみ。外部コマンド依存なし

§1概要

レガシー（既存）コードベースを画面単位にファイルをグルーピングし、その結果を 画面一覧.HTML として出力する。

単独動作: syncing-reverse-env（2 環境同期・基準タグ確立）には依存しない。ソースディレクトリと出力先ディレクトリを直接指定して起動する。
仕事は画面一覧.HTMLの作成のみ: 設計書（02_画面基本設計 等）の雛形展開・生成・記入は一切行わない。他のスキル・人手による作業の入力として画面一覧.HTMLを渡すところまでが本スキルの範囲。

§2用語

画面キー（screenKey）
意味キー規約に準拠した画面の一意識別子。ルートの静的セグメントから導出し、連番サフィックスは使わない。
confidence
画面境界検出の確からしさ。high（ルーティング定義との直接一致）／medium（正規表現ヒューリスティック）／low（慣習ディレクトリからの推測）の3段階。

§3使い方（起動引数）

引数内容
ソースコードディレクトリ探索対象。モノレポの場合は対象アプリのディレクトリを直接指定する（自動判別なし）
出力先ディレクトリ画面一覧.HTMLの書き出し先

0件検出時は AskUserQuestion で結果を報告して停止する（手動リスト入力は行わない）。

§4動作フロー（Phase 1〜2）

Phase 1: 画面境界検出
  detect-screens.sh source-dir manifest.json
  優先順位: Next.js App Router → Pages Router → React Router(flat) → 慣習ディレクトリ
  0件ならハード停止(exit 3)。捏造しない

Phase 2: 画面一覧.HTML 生成
  screen-list-template.html を土台に Write
  埋め込みJSON = Phase1の確定結果の正本コピー

§5画面キー生成規則

semantic-key-rules（連番禁止）に準拠したアルゴリズム。

手順内容
1ルートの静的セグメントのみ抽出（動的パラメータ・ワイルドカードを除外）
2末尾1セグメントを仮キーとする
3衝突する画面があれば、末尾からのセグメント数を1つずつ増やして再判定
4全セグメントを使っても衝突する場合のみ、エントリディレクトリのパスで具体化する（連番サフィックスは使わない）
5ルートが / または静的セグメント無しの場合は top

§6画面一覧.HTMLの活用方法

画面一覧.HTMLは「画面詳細設計の画面単位を正確に分けるための正本」として位置づけられる。本スキルはこの一覧を作るところまでが仕事であり、以降の設計書作成・雛形展開は対象外。

用途使い方
人間によるレビューブラウザで開き、検出方式・confidence列を見て境界の妥当性を目視確認する
後続作業への入力埋め込みJSON（#screen-manifest）をパースし、別プロセス・別スキルが設計書作成等の入力に使う

confidence が low の画面は自動判定の確度が低いため、後続作業に渡す前に人間の確認を挟むことを推奨する。

§7画面一覧.HTML フォーマット

テーブル列: 画面キー／画面名／ルート／検出方式／confidence／構成ファイル数／主ファイル。<script type="application/json" id="screen-manifest"> に構造化データを埋め込み、人間可読と機械可読を両立する。外部CDN不使用の単一ファイル自己完結。

正本の位置づけ: 画面一覧.HTML は後続フェーズが読む計画書ではなく、実施済み事実のレポート。画面詳細設計の単位を正確に分けるための正本として機能する。

§8Gotchas・スコープ外

  • import グラフ解析なし（物理的同居のみでファイル収集）
  • React Router の深いネスト親子パス合成なし（フラット抽出のみ）
  • 画面名は技術名の機械整形のみ（業務的な日本語意訳はしない）
  • 動的に構築されるルート文字列は検出不可（静的リテラルのみ）
  • モノレポ・複数アプリ構成の自動判別なし
  • 0件検出時はハード停止。AskUserQuestionでの手動リスト入力にフォールバックしない
  • 設計書（02_画面基本設計 等）の雛形展開・生成・記入は一切行わない（本スキルのスコープ外）
