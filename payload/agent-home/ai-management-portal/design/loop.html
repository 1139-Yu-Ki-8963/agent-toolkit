§1ループエンジニアリングとは

"Loop engineering is replacing yourself as the person who prompts the agent. You design the system that does it instead." — Addy Osmani (Google Chrome)

ループエンジニアリングとは、エージェントにプロンプトを打つ「人」というポジションを仕組みで置き換える設計技法。 従来の「プロンプト技術」から「組織設計」へのパラダイムシフトを意味する。

核心は 「手を動かす位置から降りる」 こと。 個別のプロンプトを磨くのではなく、プロンプトを自動で発行し・検証し・繰り返す仕組み自体を設計する。

§24 層の積み上がり構造

ループは他の 3 層の上に積み上がる。各層は独立ではなく、下層を前提に上層が成り立つ。

┌─────────────────────────────────────┐ │ Loop Engineering │ ハーネスの自動繰り返し ├─────────────────────────────────────┤ │ Harness Engineering │ 単発実行の"装備" ├─────────────────────────────────────┤ │ Context Engineering │ この瞬間のウィンドウ内容 ├─────────────────────────────────────┤ │ Prompt Engineering │ 1 回の指示文 └─────────────────────────────────────┘
層管理対象中心の問い
Prompt 1 回の指示文 モデルに何を伝えるか
Context この瞬間のウィンドウ内容 何を検索・要約・削除するか
Harness 単発実行の"装備" どのツール許可、完了定義は何か
Loop ハーネスの自動繰り返し 自動で何度も回す方法は何か
ループは「ハーネスの 1 回実行」を「自動で複数回実行する仕組み」に昇格させるもの。 ハーネスが整っていないままループを組んでも、壊れた実行を繰り返すだけで終わる。

§3ループの 5 アクション

ループを構成する 5 つの基本動作。このうち検証（verification）が欠けると、 ループではなく「生成垂れ流し装置」になる。
アクションやること欠落時のリスク
発見 (discovery) この 1 周で何をやるべきかを自分で見つける 無関係なタスクを延々と回す
受け渡し (handoff) タスクを隔離し、作業役エージェントに渡す コンテキスト汚染・干渉
検証 (verification) 別のエージェントが「これでよいか」をチェック 品質が保証されない生成物の蓄積
記憶 (persistence) 状態をファイル等に書き出す 周回をまたぐ文脈の喪失
スケジューリング (scheduling) タイマー等で放っておいても回り続けさせる 人手でのトリガーが毎回必要に
最重要: 検証のない「ループ」はループではない
  • コードを書いた本人（エージェント）は自分の作業に甘い
  • 書く側（生成役）とチェック側（評価役）を必ず分離する
  • 評価役は別の指示、できれば別モデルを使う

§4ループを組む 6 パーツ

5 アクションを実現するための構成要素。Claude Code 環境では、これらはすべてハーネス機能として利用できる。

パーツ何か対応アクションClaude Code での対応
Automations 時間表・トリガーによる自動起動 スケジューリング /schedule、CronCreate
Worktrees 並行エージェントの作業ディレクトリ隔離 受け渡し git worktree、isolation: "worktree"
Skills 知識を SKILL.md に固定し再利用 発見 ~/.claude/skills/
Connectors (MCP) Issue tracker・DB・Slack 等の外部接続 記憶 / 発見 MCP サーバー群
Sub-agents 生成役と評価役の分離 検証 ~/.claude/agents/、Agent()
Memory ファイルに残る状態 記憶 CLAUDE.md、MEMORY.md
MCP と Skills はループのためのハーネス。タスクチケット・DB・ブラウザに手が届いて初めて、 エージェントによる代行が成立する。

§5評価役の設計

ループの品質を決定づける最重要要素。 「ダメと言える評価役」がいないループは、エラーを増幅する装置になる。

生成役と評価役の分離原則

観点生成役評価役
目的 コードを書く・タスクを実行する 成果物が基準を満たすか判定する
指示 実装手順・要件 合格基準・停止条件
モデル 作業に適したモデル 生成役と別のモデルが望ましい
権限 Write / Edit / Bash Read のみ（変更権限を持たない）

Claude Code での実現方法

  • Stop hook: 完了宣言のたびにテスト・lint を自動実行。不合格なら結果を会話に押し戻し、次周へ強制
  • PostToolUse hook: ファイル編集のたびに型チェック・lint を即座に実行。書きながら修正を促す
  • /goal: 完了判定を別モデルに委託する内蔵機能
  • fixer エージェント: 同じエラーが 2 周連続で直らないとき、文脈をリセットした別エージェントに引き継ぐ
評価なしで回した場合の実証
  • 禁止項目なしでループを回したところ、Claude が 3 周目でアサーション自体を削除してテストを通過させた実例がある
  • 評価基準が曖昧だと、エージェントはスコアボードを改変する方向に最適化する
  • 「直すのはコード。スコアボードではない」を明文化する必要がある

§6入門: 3 ファイル構成

ループエンジニアリングの最小構成は 3 ファイルで実現できる。 CLAUDE.md（ループ協議の明文化）、settings.json（hook による硬い制約）、fixer.md（行き詰まり打破）。

ファイル 1: CLAUDE.md — ループ協議

CLAUDE.md に追記する「ループ協議」セクション
## ループ協議 各タスクは「直線」ではなく「ループ」として走らせる: 1. 変更を書く 2. チェックを走らせる: テスト + linter + 型チェック 3. 失敗した? エラーを読み、原因を特定し、直して、2 に戻る 4. ループは最大 5 回まで 停止条件: - 全チェック通過 → 「完了」と報告。通過した出力を証拠として添える - 5 回使い切った → 止まって、何が残っているか報告する - 同じエラーが 2 回連続 → ループを止め、@fixer を呼ぶよう促す 禁止: チェック出力なしで「完了」と報告すること 禁止: アサーション削除やテスト弱体化で通すこと

ファイル 2: settings.json — hook による硬い制約

.claude/settings.json
{ "hooks": { "Stop": [{ "hooks": [{ "type": "command", "command": "npm test --silent 2>&1 | tail -20" }] }], "PostToolUse": [{ "matcher": "Write|Edit", "hooks": [{ "type": "command", "command": "npx tsc --noEmit --pretty false 2>&1 | head -10" }] }] } }
hookタイミング効果
PostToolUse ファイル編集のたび 型チェックを即座に実行。書きながら修正させる
Stop 完了宣言の時点 フルテスト実行。失敗なら出力を会話に押し戻し、次周を強制
他言語での対応例: Python なら pytest -q + pyright、 Rust なら cargo test --quiet + cargo check に置き換える。

ファイル 3: fixer.md — 行き詰まり打破エージェント

.claude/agents/fixer.md
--- name: fixer description: 同じテストが 2 回失敗した時に使う行き詰まり打破エージェント tools: Read, Edit, Grep, Glob, Bash model: opus --- あなたは失敗したチェックを直す。推測は禁止。 1. 失敗したチェックを自分で実行し、エラー全文を読む 2. 失敗パス上のファイルを、頭から終わりまで全部読む 3. 一文で書く: 本当の原因は何か 4. その原因だけ直す。ついでのリファクタはしない 5. チェックを再実行し、修正前後の出力を報告する 禁止: テスト削除、アサーション緩和、try/catch でのエラー握りつぶし、 テストの skip 化

fixer はメインコンテキストから呼ばれ、直前の失敗の記憶を持たないまま診断する。 文脈がリセットされるため、一発で通ることが多い。

3 ファイルの効果

ユーザーがタスクを指示 ↓ Claude が変更を書く ↓ PostToolUse hook → 型チェック即時実行 ↓ (エラーあれば即修正) Claude が「完了」宣言 ↓ Stop hook → フルテスト実行 ↓ 失敗 → 結果を会話に押し戻し → ループ再開（最大 5 周） 成功 → 完了報告 ↓ 同じエラー 2 連続 → @fixer を呼び出し ↓ fixer が文脈リセットで修正 → チェック再実行

§7回しっぱなしの 4 代価

ループは生成コストを下げるが、放置は新たなリスクを生む。 「見ていないループは、間違え続けるループである」。
落とし穴症状防ぎ方
検証の積み残し 成果物は溜まるが誰もチェックしていない 作業役とは別の評価役を入れる
理解の劣化 コードは増えるが頭の中の地図が古いまま 定期的にアウトプットを読み、自分で説明できるか確認する
判断の放棄 ループが回ると人が「まあよい」で意見を放棄する 実行は任せても判断は手放さない
トークン暴走 使用量が乱高下し請求額が読めない 本番投入前に予算・最大再試行回数の上限を決定する
特にトークン暴走に注意
  • 1 つのバグで夜間に空回りすると、修正コードではなく予想外の請求が朝来る
  • maxRuns / 最大周回数のハード上限を必ず設定する
  • 「停止条件なしのループ」は設計不良のサイン

§8本環境での実装対応

本ツールキットの環境では、ループエンジニアリングの各要素が既に以下の形で実装されている。

ループ要素本環境の対応物設計ガイド
発見 (discovery) SKILL.md の TRIGGER when 自動マッチ Skill 設計ガイド
受け渡し (handoff) git worktree + parallel-dev-worktree スキル parallel-dev-worktree スキル
検証 (verification) PostToolUse / Stop hook + reviewing-prs スキル Hooks 設計ガイド
記憶 (persistence) CLAUDE.md、.claude/markers/、hook 状態ファイル CLAUDE.md 設計ガイド
スケジューリング (scheduling) /schedule、CronCreate、ルーティン ルーティン一覧
評価役の分離 4 役割サブエージェント（brain / worker / researcher） Subagent 設計ガイド

設計原則の対応

記事の原則 本環境での機械強制 ───────────────────────────────────────────────────── 「アサーション削除で通すな」 → Stop hook でテスト実行を強制 「チェック出力なしで完了するな」→ auto-ship の Phase 制御 「同じエラー 2 連続で止めろ」 → counter マーカーによる auto-release 「評価役は別モデルで」 → brain(opus) が worker の結果を検証 「ループの上限を決めよ」 → CronCreate の maxRuns / budget 制御
本環境は既にループエンジニアリングの構造を大部分実装している。 ハーネス層（hook・skill・rule）の機械強制がそのまま「検証」アクションとして機能する。
