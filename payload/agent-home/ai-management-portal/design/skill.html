§1Skill の定義

skill とは、トリガー駆動で起動する拡張であり、起動したときに限り、Claude がデフォルトでは出せない 手順・条件付き知識・強制 のいずれかを提供するもの。
  • Skill フォルダ（読み取り専用・配布物）には SKILL.md / スクリプト / テンプレート / リファレンスといった不変の部品だけを置く。
  • 可変データ（状態・ログ・設定）はフォルダの外に出す（詳細は §10）。

§2Skill にすべきかの判定

判定フロー

Q1. それは「常に」必要か？ YES → context / rules に書く（skill にしない） NO（特定のときだけ要る）→ Q2 へ Q2. その合図が来たとき、次のどれかを提供するか？ a. 多段の手順・フロー（デプロイ、調査、雛形生成 など） b. その領域を触るときだけ要る参照知識・落とし穴 c. スクリプト/フックで機械的に効かせたい強制 → どれにも当たらない → skill にしない → どれかに当たる → Q3 へ Q3. それは Claude がデフォルトでやることの言い換えにすぎないか？ YES → 作らない（無駄なコストと誤発火を足すだけ） NO（デフォルトから押し出す中身がある）→ ✅ skill にする

除外則（当たったら即・非 skill）

  • 常に効く固定ルール → rules ＋ CI/フック
  • 「スクリプトを同梱したいだけ」 → 普通のフォルダで足りる

タイプ（3 型）

タイプ 説明 例

条件付き知識型 その領域を触るときだけの参照知識 ライブラリリファレンス、落とし穴

§4起動モデル

3 状態を使い分ける。frontmatter で制御（§12）。

状態 frontmatter 用途
既定（無印） なし 自動 ＋ 明示の両方から呼べる
明示限定 disable-model-invocation: true 副作用・破壊的 skill（deploy / commit / 送信 / 削除 等）にのみ付ける。Claude が勝手に発火しないようにする
自動限定 user-invocable: false 知識型の純粋背景知識（コマンドとして叩く意味がないもの）にのみ付ける
排他制約
  • disable-model-invocation と user-invocable: false は同時に付けない（排他）。
  • 自動起動を許可するため、description には「いつ使うか」のトリガー語を 1 つ含める（§6）。

§5命名規約

最優先：名前だけで「何をするか」が一意に伝わること（解釈違いゼロ）

基本方針

  • 短さより曖昧さ排除。64 字まで使え、長くてよい。
  • 二人が別解釈しうるなら、語を足して潰す。
  • 語形：action-oriented（動詞-目的語）で確定。create- / verify- / deploy- など動詞始まり。
  • 略語・隠語を避け、平易な一般語を使う。
  • バージョン表記は入れない（version はメタデータで管理）。
  • 亜種は番号でなく差分を語で表す（例：create-skill-minimal）。
  • 同じ動作は同じ動詞で固定する。

衝突チェック（必須）

  1. 自分の skill / slash コマンドと重複しない。
  2. Anthropic 公式 skill と重複しない（参照元：anthropics/skills。例：frontend-design, pdf, docx, pptx, xlsx …）。
  3. 予約語 anthropic / claude を含めない。

硬い制約（公式）

小文字 / 数字 / ハイフン のみ使用可。最大 64 字。

§6description 規律（日本語・200K モデル前提）

項目制約
合計上限約 2,000 文字（コンテキスト窓の 1% / 日本語 ≒ 1 字 1tok の保守見積）
1 件あたり≤ 50 文字
内容「何をするか」を三人称で 1 文。自動起動のためトリガー語を 1 つ含める
形式TRIGGER when: / SKIP: の固定英語キーワードを必ず含める（多行形式）
目安個数2,000 ÷ 50 ＝ 約 40 個

40 個を超えそうなとき（優先順位）

  1. 説明をさらに削る（30 字台へ）
  2. skillListingBudgetFraction: 0.02 で枠を倍（≒ 80 個）
  3. よく使うものは自動で説明が残る（低頻度から落ちる）
  4. 名前は常時載るので /name 呼び出しは個数無制限

固定英語キーワード（TRIGGER when: / SKIP:）の仕様

自動発火の精度に直結するため、日本語の「使用時：」等は使わず必ず英語キーワードを使う。

  • パース精度 — 日本語は表記ゆれが多く、モデルが発火条件ブロックを確実に区別できない。英語固定形式は変種なく認識される。
  • 構造認識 — LLM は英語構造で学習されているため、TRIGGER when: / SKIP: で条件節を切り出す精度が高い。
  • 誤発火防止 — SKIP: を明記することで、類似トリガーを持つ別スキルとの衝突・過剰発火を防ぐ。
多行形式サンプル
description: | ブランチ作成・worktree 管理を行うスキル。 TRIGGER when: ブランチ作成時、git checkout 時、worktree 作成時。 SKIP: 既存ブランチの読み取り・確認のみの時。
監視：/doctor で溢れを確認する。

§7ツール選択

選択ロジック

Step1. skill の型を判定（= なぜ skill か） 条件付き知識型 / 手順型 / 強制型 のどれか Step2. 型ごとの「基本ツールセット」を採る（下のマッピング） Step3. 横断ツールを必要時だけ足す ・起動時にユーザーの選択が要る → AskUserQuestion ・他の skill を呼ぶ → Skill ・外部サービスに触る → MCP 系（要 MCP 接続） ・定期・遅延実行が要る → Cron 系 / ScheduleWakeup ・何かを監視し続ける → Monitor Step4. 最小権限に絞り allowed-tools に固定 ・不要な Write/Bash は渡さない（特に知識型） ・「常に・決定論的に」強制したい挙動はツールでなく hook

型 → 最適ツール マッピング

条件付き知識型
（読むのが主・副作用は最小）
基本：Read, Grep, Glob
必要なら：WebFetch / MCP 系（外部参照のとき）
原則：Write・Bash は渡さない（知識 skill に実行権は危険）
手順型
（実行と編成が主）
基本：Bash, Read, Write, Edit
よく足す：
  • Monitor … CI/ログ/エラー率を監視（CI-CD, Runbook, インフラ運用）
  • Agent … 並列調査や重い下請け（Runbook）
  • Skill … データ取得 skill を合成（CI-CD, 業務自動化）
  • Task 系 … 多段手順の進行管理
  • AskUserQuestion … セットアップ/投稿先選択（業務自動化）
  • Cron 系 … soak 期間・定期実行（インフラ運用）
強制型
（検査して介入する）
基本：Agent（第三者視点の批評）, Read, Grep, LSP, Bash（linter）
介入するなら：Edit（修正を適用）
注意：「毎回必ず弾く」はツールでなく hook の領域。skill のツールは"起動時に検査する"ところまで。

§8フォルダ構造

共通スケルトン

<skill>/ SKILL.md … 概要 + 目次(=型ごとの分割単位) + 全体ルール + Gotchas(必須) <分割ディレクトリ> … 型で単位が決まる（下記）。1 単位 = 1 ファイル scripts/ … 実行スクリプト assets/ … テンプレート

共通ルール

  • 状態・ログはセッションフォルダへ出力。skill フォルダには絶対置かない。
  • 参照は SKILL.md から 1 階層まで（深いネスト禁止）。
  • ファイル名は内容が分かる名前（doc2.md ✗ / form_validation_rules.md ○）。
  • 100 行超のファイルは冒頭に目次を置く。
  • 型が決まれば分割単位は自動で決まる（判断の余地を残さない）。

手順型 → 単位 = Phase

phases/ 01-<phase名>.md 02-<phase名>.md ...
  • Phase と Step を必ず明記。
  • Phase ごとに 1 ファイル（番号をファイル名先頭に＝順序が一意）。
  • 各ファイル冒頭：# Phase N: 名前
  • 各操作：## Step N を連番で。
  • SKILL.md に Phase 一覧（目次）＋ 進捗チェックリスト（Claude がコピーして 1 つずつ消す）。

条件付き知識型 → 単位 = ドメイン/トピック

reference/ <ドメイン or トピック>.md … 1 トピック = 1 ファイル（索引構造、フロー化しない）
  • SKILL.md は「どのトピックがどのファイルか」のナビ表。
  • 各ファイル 100 行超なら冒頭に目次。
  • Gotchas は該当トピックのファイル内に置く。

強制型 → 単位 = ルールカテゴリ

rules/ <ルールカテゴリ>.md … 1 カテゴリ = 1 ファイル（該当時だけ読む） scripts/ <検査スクリプト> … 決定論的チェックはコード化
  • 検査は feedback loop を Step 化（検査 → 修正 → 再検査 → 合格まで）。
  • 機械的に効かせるなら on-demand フックを併用（§10）。

§9SKILL.md 本文ルール

冒頭の自己宣言（AI 向け・本文先頭）

# <skill名> > Category: <9 分類のいずれか 1 つ> > Type: 手順型 / 条件付き知識型 / 強制型

必須セクション

  • Gotchas（定義は下）
  • 重要指示は冒頭付近に置く（溢れ時は先頭が残る＝ truncation 対策）

書く / 書かない

方針
書かない当たり前のこと（Claude が既に知ること）
書くデフォルトから押し出す情報・社内固有知識

自由度の設定

  • 高自由度 = 文章指示（複数の道が正解／文脈依存）
  • 低自由度 = 手順・スクリプト指定（脆い・破壊的・一意手順必須）

一貫性・鮮度

  • 用語は 1 概念 1 語で統一する。
  • 時限情報は避け、必要なら old patterns 節へ隔離する。

選択肢の提示

  • 手段を並べない。既定を 1 つ ＋ 特定条件の代替を 1 つだけ。
  • 例：pdfplumber を使う。スキャン PDF の OCR 時のみ pdf2image。

Gotchas 定義

その skill で Claude が繰り返す "直感に反する罠" を蓄積する節（最高シグナル）。

## Gotchas - <罠/直感に反する点>: <実際はこう> → <だからこうする>
Gotchas の例
  • subscriptions は追記専用。最新行は created_at 最大でなく version 最大
  • gateway は @request_id、billing は trace_id（同じ値）
  • staging は webhook 未処理でも 200 を返す → payment_events で真の状態確認
1 個から始め、失敗のたびに足して育てる。

§10セットアップ・記憶・on-demand フック

共通原則

  • 可変データ（設定・記憶）は skill フォルダに置かない（フォルダは不変）。
  • 置き場：セッションフォルダ（~/agent-home/sessions/YYYY-MM-DD/）/ 永続が要るなら ${CLAUDE_PLUGIN_DATA}（※設計上の参照変数。現時点で実装例なし）

セットアップとは

skill が「ユーザー/環境ごとに違う値」を知らないと動けない場合に、その値を 一度だけ埋めて記憶 すること。

対象かどうか例
対象 一度決めたら変わらない値 Slack 投稿先チャンネル、社内 API エンドポイント・トークン、BigQuery プロジェクト ID、デプロイ対象環境
非対象 毎回変わる入力 その時の PR 番号、対象ファイル
  • 未設定なら AskUserQuestion で聞く（自由文では聞かない）。
  • 機密（トークン等）は userConfig の sensitive（keychain）へ。
  • 非機密は ${CLAUDE_PLUGIN_DATA}/config.json に保存し、次回は聞かない。
  • skill フォルダ内 config は禁止。

記憶

  • 形式：追記ログ / JSON / SQLite
  • 置き場：セッションフォルダ（~/agent-home/sessions/YYYY-MM-DD/）/ 永続は ${CLAUDE_PLUGIN_DATA}（現時点で実装例なし）
  • 使い方：自分の履歴を読み、差分や一貫性に使う（例：standups.log）

on-demand フック（skill の hooks 設定に置く）

原則：「この skill の実行中だけ効かせたい決定論的なガード/検査」に使う。 常時・全セッションで効かせたいものは CI か常時フックへ（ここではない）。
カテゴリタイミング例
ブロック系 PreToolUse /careful = rm -rf / DROP TABLE / force-push を止める
/freeze = 指定ディレクトリ外の Edit/Write を止める
検証系 PostToolUse 編集のたびに linter / formatter / 型チェック / テストを走らせ、結果を Claude に返して直させる

§11テンプレート方針

型テンプレートの要否
手順型 定型アウトプットがあれば assets/ にテンプレートを用意（必須）
条件付き知識型 テンプレート不要
強制型 レポート/修正の定型があれば assets/ にテンプレートを用意
中身は各 skill が作る。設計ルールは「用意するか否か」まで。

§12frontmatter テンプレート

SKILL.md の先頭（frontmatter）
--- name: <動詞-目的語・小文字/数字/ハイフン・最大64字> description: | <三人称・日本語50字以内・「何をするか」> TRIGGER when: <発動条件を具体的なキーワードで列挙> SKIP: <非発動条件> invocation: <name と同値> allowed-tools: <型ごとの最小セット> # 例 知識型: Read, Grep, Glob # 以下は該当時のみ付ける（既定はどちらも付けない＝自動＋明示の両方可） disable-model-invocation: true # 副作用・破壊的skill（deploy/commit/送信/削除）のみ user-invocable: false # 知識型の純粋背景知識のみ（コマンドの意味がないもの） ---

チェックリスト

  • 既定（無印）= 自動・明示の両方から呼べる。
  • disable-model-invocation と user-invocable: false は同時に付けない（排他）。
  • invocation は name と同値で必ず記載する。

型別 allowed-tools 最小セット

型最小セット
条件付き知識型 Read, Grep, Glob
手順型 Bash, Read, Write, Edit（＋必要な横断ツール）
強制型 Agent, Read, Grep, Bash（介入するなら Edit を追加）
