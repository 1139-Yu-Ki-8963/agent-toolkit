# project-docs/02_画面基本設計 — AI 駆動開発の 1 画面分ドキュメント 4 ファイルセット

AI 駆動で開発する画面 1 枚分のドキュメント雛形。原則「コードから再生できる情報を書かない」を貫く。4 ファイルは連動キーで結ばれており、単独では使わない。

## _共通 3 ファイルの役割

プロジェクトに 1 セット置く共通の正本。画面ごとにコピーしない。

| ファイル | 役割 |
|---|---|
| _共通/DESIGN.md | 全画面のスタイル・数値の上位の正。個別画面の DESIGN.md は差分のみを持つ |
| _共通/メッセージ定義書.md | UI 文言の正。意味キーで一元管理し、各画面設計書から参照する |
| _共通/画面共通仕様.md | 全画面共通の挙動の正。各画面設計書 §8（逸脱と例外）の比較先 |

> **コピー時の注意**: `_共通/` はプロジェクトの `docs/02_画面基本設計/_共通/` に 1 セット置く。
> 画面ごとのディレクトリ（`<画面名>/`）にはコピーしない。

## コピー手順

1. `<project>/docs/02_画面基本設計/<画面名>/` を作成する
2. このディレクトリ内の 4 ファイルをコピーする
3. frontmatter のプレースホルダを置換する（`target_screen` は全ファイルで同値にする）

## 4 ファイルの役割分担

| ファイル | 正として持つもの | 補足 |
|---|---|---|
| 画面基本設計書.md | 何を・なぜ（コード外情報。§1〜§9） | §9 は ADR 形式の意思決定章 |
| DESIGN.md | どう見せるか（スタイル・数値） | validate-design-md.sh の PASS が受け入れ条件 |
| 単体テスト観点表.md | small の観点（TDD 内側ループの RED 起点） | 具体的な項目はテストコードが正 |
| 結合テスト観点表.md | 画面内結合と単画面ブラウザ検証（medium）の観点と受け入れ条件 | 手動観点のみ項目レベルを持つ |

## 連動キー

| キー | 定義権 | 参照先 |
|---|---|---|
| 領域名 | 画面基本設計書.md §2（領域マップ） | DESIGN.md の Components が同名で使用する（名前の変更不可） |
| 画面状態 | 画面基本設計書.md §4 | DESIGN.md の画面状態の視覚仕様が参照する |
| target_screen | 全ファイルで同値を維持 | 一方を変更したら他の全ファイルも追随させる |
| 由来 § | 観点表の由来 § 列 | 画面基本設計書の章番号と一字一句一致させる |

## 開発フローとの接続

    Phase 4: 画面基本設計書.md + DESIGN.md を記入 → モック生成 → 承認
             status: draft → approved（§9 未決 0 件 + モック承認が条件）
        ↓
    Phase 5: 単体テスト観点表.md + 結合テスト観点表.md を起票（status: draft）
        ↓
    Phase 6: TDD 二重ループ
             外側ループ: 結合観点表の受け入れ◯ を failing テストとして先行作成
             内側ループ: 単体観点表を RED → GREEN → REFACTOR で消化
        ↓
    Phase 7: 受け入れ◯ 全 green + 手動観点 確認済み → PR マージ
             全行のテスト参照が埋まったら status: traced に更新

## 記入規則

- 画面基本設計書.md の章は §1〜§9 固定・省略禁止（該当なしは見出しを残す）
- 行識別子は意味キー必須・連番禁止
- 数値・色は DESIGN.md のみに記入する（設計書には書かない）
- status ライフサイクル
  - 画面基本設計書.md / DESIGN.md : draft → approved → implemented
  - 単体テスト観点表.md / 結合テスト観点表.md : draft → traced
