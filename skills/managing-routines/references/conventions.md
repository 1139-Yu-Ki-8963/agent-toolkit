# ルーティン共通規約（conventions）

`managing-routines` の create / review / test 全モードが参照する規約。

## 正本の所在

ルーティン資産は `<project>/routines/` に一元管理する。プロジェクトごとにサブディレクトリを持つ。

```
<project>/routines/
├── _shared/                      ← 全プロジェクト共通
│   └── common-log-rules.md       ← JSONL エンベロープ仕様
├── <project>/                    ← プロジェクト固有
│   ├── profile.md                ← 技術スタック・パス・閾値
│   ├── 共通ルール.md              ← 12 観点定義・ルーティン一覧
│   ├── クラウド共通プロンプト.md    ← git author・コミット規約・lint
│   └── routines/                 ← 各ルーティンの設計書・実行プロンプト
│       └── <ルーティン名>/
│           ├── ルーティン設計書.md
│           ├── 実行プロンプト.md
│           └── 検証仕様.md
```

| 正本 | パス | 内容 |
|---|---|---|
| プロジェクトプロファイル | `routines/<project>/profile.md` | 技術スタック・パス・閾値・git author・有効ルーティン一覧 |
| 共通ルール | `routines/<project>/共通ルール.md` | 12 観点定義・JSONL 仕様・prompt 登録形式 |
| クラウド共通プロンプト | `routines/<project>/クラウド共通プロンプト.md` | git author・コミット規約・lint 必須・PR テンプレ等 |
| JSONL ログ仕様 | `routines/_shared/common-log-rules.md` | 共通 JSONL エンベロープ（全プロジェクト共通） |

**create / review 時は対象プロジェクトの `profile.md` → `共通ルール.md` の順に Read すること。**

## 1. フォルダ名は日本語必須

`共通ルール.md` のフォルダ構成に加え、以下を適用する:

- `<name>` は **日本語名必須**（例: `シークレット漏洩監視`、`カバレッジ自動改善`、`PR レビュー統合`）
- kebab-case（英語）のフォルダ名は禁止。既存の英語名フォルダはレビュー時に日本語名へリネームする
- ログ出力先の `<slug>` は設計書内の `slug` フィールド（kebab-case）を使う

## 2. 実行環境判定基準

`共通ルール.md` はクラウドルーティン前提で書かれている。ローカル CronCreate との判定は以下の決定木で行う:

```
Q1. localhost / ローカルサービスへのアクセスが必要か?
  YES → ローカル CronCreate
  NO  → Q2

Q2. ~/.claude/ やローカルファイルシステム固有のパスを参照するか?
  YES → ローカル CronCreate
  NO  → Q3

Q3. 実行間隔が 1 時間未満か?
  YES → ローカル CronCreate
  NO  → クラウド Routines
```

判定結果はルーティン設計書.md の「実行環境」フィールドに明記する。

## 3. 実行プロンプト.md の必須構造

以下の構造とする。事前 Read ファイルはルーティン資産側（`routines/<project>/`）とプロジェクト側の両方を含む:

```markdown
# <ルーティン名> エージェント

リポジトリのルートが作業ディレクトリです。<概要 1 行>。
途中でユーザーへ質問してはいけません。最後まで止まらずに実行してください。

## 事前準備: 共通ルールの読み込み

本ルーティンの実行前に、次のファイルを必ず Read します。

### ルーティン資産側
- `routines/<project>/共通ルール.md`
- `routines/<project>/クラウド共通プロンプト.md`
- `routines/_shared/common-log-rules.md`

### プロジェクト側（プロジェクト固有ルール。profile.md の事前 Read 一覧に従う）
- 例: `docs/docs-portal/sites/rules/08-auto/code-docs-mapping/rule.html` 等

## 事前準備: 自律実行ガード

- ユーザーへの確認発話・`AskUserQuestion` の呼び出しは全面禁止
- エラーが発生してもログに記録して続行する（致命的エラーは除く）

## Phase 1: <タイトル>

### 手順
<具体的な Bash コマンドと説明>

### 完了条件
<この Phase が成功したと判断する基準>

## Phase N: ログ出力

### 手順
共通 JSONL エンベロープで `logs/<slug>/YYYY-MM-DD.jsonl` に run_end を出力

### 完了条件
JSONL ファイルに run_end 行が追記されていること
```

### Phase / Step 記法ルール

正本: `~/.claude/skills/skill-design-spec.md` §12-13。以下はルーティン固有の要約:

| 要素 | 書式 | 必須サブセクション |
|---|---|---|
| Phase | `## Phase N: <タイトル>` | `### 手順` + `### 完了条件`（両方必須） |
| Step（Phase 内の細分化） | `### Step N: <タイトル>` | 手順の記述（完了条件は Phase 側で包含可） |

**ルール:**
- Phase は連番。飛ばさない（Phase 1 → Phase 2 → Phase 3）
- 各 Phase に `### 完了条件` を必ず付ける。「正常に動作すること」のような曖昧な条件は禁止。コマンド戻り値・ファイル存在・数値比較で判定可能な基準を書く
- Phase 1 には依存ツールのインストール手順を含める（クラウドの場合）
- 最終 Phase は「ログ出力」とし、共通 JSONL エンベロープで出力して締める

### TaskCreate / TaskUpdate の使用ルール

managing-routines の create / review / test 各モード実行時は、`TaskCreate` で Phase を登録し進捗を可視化する。正本: `~/.claude/skills/skill-design-spec.md` §13。

**具体例（create モードの場合）:**

```
# モード開始時に全 Step を登録
TaskCreate({ subject: "Step 1: 要件の確定", description: "設計書の必須フィールドを確定", activeForm: "要件を確定中" })
TaskCreate({ subject: "Step 2: ディレクトリ作成", description: "日本語名でフォルダを作成" })
TaskCreate({ subject: "Step 3: 設計書作成", description: "ルーティン設計書.md を Write" })
TaskCreate({ subject: "Step 4: 実行プロンプト作成", description: "実行プロンプト.md を Write" })
TaskCreate({ subject: "Step 5: ツール依存確認", description: "参照パスの存在確認" })
TaskCreate({ subject: "Step 6: ルーティン登録", description: "/schedule 指示提示 or CronCreate" })

# 各 Step 開始時
TaskUpdate({ taskId: "1", status: "in_progress" })

# 各 Step 完了時
TaskUpdate({ taskId: "1", status: "completed" })
```

**ルール:**
- モード開始時に全 Step / Phase を一括登録する（途中で追加しない）
- 開始時に `in_progress`、完了時に `completed` に切り替える
- `activeForm` は進行形で書く（例: 「要件を確定中」「設計書を作成中」）

### 必須要素チェックリスト

- [ ] 事前準備セクションで `共通ルール.md` を Read しているか
- [ ] 事前準備セクションで `クラウド共通プロンプト.md` を Read しているか（クラウドの場合は必須）
- [ ] 冒頭に「途中でユーザーへ質問してはいけません」の自律実行ガード
- [ ] 全 Phase に `### 完了条件` サブセクション
- [ ] 完了条件がコマンド戻り値・ファイル存在・数値比較で判定可能か（曖昧条件は禁止）
- [ ] `AskUserQuestion` 呼び出し禁止の明示
- [ ] エラー時の挙動（ログ記録 / スキップ / 中断）の明示
- [ ] ツール・スクリプトのパスがリポジトリルート相対で記述されている

## 4. ルーティン設計書.md の必須フィールド

| フィールド | 説明 |
|---|---|
| ルーティン名 | 日本語名（例: `シークレット漏洩監視`）。フォルダ名と一致させる |
| 目的 | このルーティンが解決する問題（1〜2 文） |
| 実行環境 | `クラウド Routines` または `ローカル CronCreate`（判定基準で決定） |
| スケジュール | cron 式 + タイムゾーン + 人間可読な説明 |
| 入力 | 何を読むか（リポジトリ全体 / 特定ディレクトリ / 外部 API 等） |
| 出力 | JSONL ログ（必須）+ コード変更を伴う PR（条件付き）。issue 起票・通知は全面禁止 |
| PR 影響度 | `影響なし`（マージまで自動）/ `影響あり`（PR 作成まで、E2E 必須）/ `PR なし`（ログのみ）|
| 依存関係 | 前提となるツール・サービス・他ルーティン |
| 冪等性設計 | 再実行時に副作用が重複しない仕組み |
| 失敗時リカバリ | 通知先・リトライ戦略・手動介入手順 |
| コネクタ要件 | クラウド実行時に必要な MCP コネクタ一覧（クラウドの場合のみ） |

## 5. PR のマージ判断基準

ルーティンが PR を作成する場合、**アプリへの影響度** でマージ可否を決定する。

```
Q. PR の変更はアプリの実行時動作に影響するか？

  NO（影響なし）→ ルーティンがマージまで自動実行してよい
    例: テストファイル追加、ドキュメント更新、lint 修正、
        型定義の補完、コメント修正、設定ファイル整備

  YES（影響あり）→ PR 作成まで。マージ禁止。E2E テスト通過が必須
    例: プロダクションコード変更、API 変更、DB スキーマ変更、
        依存パッケージ更新、環境変数変更、ビルド設定変更
```

影響度が混在する場合（テスト追加 + プロダクションコード修正）は「影響あり」として扱う。

## 6. スケジューリング規約（5h グリッド）

### 5 スロット（固定・全プロジェクト共通）

実行密度を最適化し、ルーティン同士の競合を防ぐため、5 時間間隔の固定スロットに配置する。日次上限は 2026 年 6 月時点で撤廃済みだが、5h グリッドは実行の分散・管理容易性のために維持する。

| スロット | JST | cron 分 |
|---|---|---|
| A | 05:00 | `3 5 * * *` |
| B | 10:00 | `3 10 * * *` |
| C | 15:00 | `3 15 * * *` |
| D | 20:00 | `3 20 * * *` |
| E | 00:00 | `3 0 * * *` |

- `:00` ちょうどは集中回避のため数分ずらす（例: `3 5` = 05:03）
- **5h グリッド外の時刻を提案してはならない**
- ローカル CronCreate: JST で解釈される
- クラウド Routines: `/schedule` コマンドは自然言語で時刻指定可（内部で変換）

### 日次 vs 週次の判断基準

| 分類 | 基準 | 例 |
|---|---|---|
| 日次 | 1 日の遅延で被害が拡大する。毎日差分が発生する | シークレット漏洩・バックアップ・PR レビュー・統計レポート |
| 週次 | 蓄積型の改善で、毎日実行しても差分が小さい | カバレッジ改善・デッドコード・linter 進化 |

### スロット割当ルール

1. **日次ルーティン** が 5 スロットを先に占有する
2. **週次ルーティン** は **木曜日に集中配置** する（同一スロットに 2 件は禁止）
3. **統計レポート** はスロット外で運用（他ルーティンの実行後に集約するため）
4. profile.md の「有効なルーティン」表で現在の割当状況を確認してから配置する

### create モードでの提案手順

新規ルーティンのスケジュールを決める際は、以下の手順で `AskUserQuestion` を使ってユーザーに提示する:

1. 日次 / 週次を判断基準で判定する
2. profile.md の有効なルーティン表から現在のスロット使用状況を取得する
3. 空きスロット・空き曜日を特定する
4. 推奨スロットを提案し、ユーザーに確認する

```
AskUserQuestion({
  questions: [{
    question: "<name> のスケジュールを提案します。判定: <日次/週次>（理由: ...）。どのスロットに配置しますか？",
    header: "スケジュール",
    options: [
      { label: "Slot A（05:00 JST）", description: "現在の使用状況: <日次 N 件 / 週次 N 件>" },
      { label: "Slot B（10:00 JST）", description: "現在の使用状況: <日次 N 件 / 週次 N 件>" },
      { label: "Slot C（15:00 JST）", description: "現在の使用状況: <日次 N 件 / 週次 N 件>" },
      { label: "Slot D（20:00 JST）", description: "現在の使用状況: <日次 N 件 / 週次 N 件>" }
    ],
    multiSelect: false
  }]
})
```

週次の場合は曜日の選択肢も提示する。

## 7. ツール依存の確認（クラウド実行向け）

クラウド Routines は fresh clone で実行されるため、以下を事前に確認する:

1. **リポジトリ内ツール**: `実行プロンプト.md` が参照するスクリプト・設定ファイルがリポジトリ内に存在するか
2. **外部ツール**: `pip install` / `npm install` 等で取得するツールは Phase 1 で明示的にインストール手順を書く
3. **環境変数**: クラウド環境の環境変数設定で事前定義が必要なものを設計書に列挙する
4. **コネクタ**: 外部サービス連携は https://claude.ai/customize/connectors で設定済みか確認する

## 設計判断

### 5h グリッドスケジューリング（旧 ADR-0052）

**必要性**: 実行密度の最適化とルーティン競合防止のため、5 時間間隔の固定スロット（05:00 / 10:00 / 15:00 / 20:00 / 00:00 JST）に配置する。日次上限は 2026 年 6 月時点で撤廃済みだが、管理容易性のためグリッドは維持する。

**代替案を採用しなかった理由**: 等間隔 4.8h は半端な時刻（09:48 / 14:36 等）で人間が把握しにくい。固定時刻なし（自由配置）はルーティン追加のたびに全体の再調整が必要で運用コストが高い。Max プラン前提（15 runs/day）はプラン変更に依存する設計になる。

**保守責任者**: managing-routines スキル + ユーザー。

**廃棄条件**: プラン体系が根本的に変わり、5h グリッドの分散が不要になった場合。

### 週次ルーティン木曜集中配置

**必要性**: 週次ルーティン（パッケージ脆弱性パッチ・linter 自己進化・ログ清掃等）を木曜日に集中させる。金曜日までに結果を確認し、週末前にマージ判断を完了できるため、週末を跨ぐ未処理 PR の滞留を防げる。ユーザー指示（2026-06-21）による決定。

**代替案を採用しなかった理由**: 曜日分散は各ルーティンの結果確認が週の各所に散らばり、注意力の配分コストが高い。月曜集中は週明けの手動タスクと競合する。金曜集中は修正が週末に持ち越される。

**保守責任者**: ユーザー。

**廃棄条件**: ルーティン数が木曜 1 日の 5 スロットに収まらなくなった場合、または曜日分散に戻す明示指示があった場合。

## Gotchas

- `実行プロンプト.md` は git commit しないと fresh clone に含まれない。未コミットのまま `/schedule` しても動かない
- ローカル CronCreate の `durable: true` は `.claude/scheduled_tasks.json` に書き出すが、クラウド Routines とは別の仕組み
- クラウド Routines は `claude/` プレフィックスのブランチにしか push できない（デフォルト設定）
- `共通ルール.md` のルーティン一覧と `共通ルール.md` 内の JSONL 仕様が正本。conventions.md でこれらを再定義しない
