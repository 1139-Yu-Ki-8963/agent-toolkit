# ルーティンレビュー手順（reviewing）

`managing-routines` の **review モード** が参照する手順書。12 観点 × 6 Phase でルーティンの品質を検証する。

12 観点の一覧定義の正本は `routines/<project>/共通ルール.md` にある。本ファイルは各観点の **詳細チェック項目・検出パターン・判定基準** を定義する。

## 12 観点の詳細

### 観点 A: クラウド実行適合性

**目的:** fresh clone 環境で正常に動作するか検証する

**チェック項目:**
- [ ] `実行プロンプト.md` がローカル専用パス（`~/`、`/Users/`、`/home/`）を参照していないか
- [ ] `~/.claude/` 配下のファイルを前提としていないか
- [ ] ローカルサービス（`localhost`、`127.0.0.1`）への接続を含まないか
- [ ] 環境変数がクラウド環境で設定可能か（設計書のコネクタ要件と突合）
- [ ] `claude/` プレフィックス以外のブランチへ push していないか
- [ ] ルーティンのロジックがクラウド環境で動作するか（例: E2E テスト・flaky テスト等、localhost 前提の機能を含んでいないか）（← D-1）
- [ ] クラウド実行に必要な `allowed_tools` 権限が設定されているか（例: `Bash(*)` が必要なのに `Bash` のみになっていないか）（← R-3）
- [ ] 事前準備セクションで `routines/<project>/クラウド共通プロンプト.md` を Read しているか（git author 設定・コミット規約・lint 必須・PR テンプレ・AskUserQuestion 禁止等をクラウド環境で担保する共通指示）

**検出パターン:**
```bash
grep -nE '(~/|/Users/|/home/|localhost|127\.0\.0\.1|\.claude/)' 実行プロンプト.md
# allowed_tools の確認（クラウド Routine の設定画面で確認）
grep -nE 'allowed.?tools' ルーティン設計書.md
# クラウド共通プロンプトの Read 有無
grep -nE '(クラウド共通プロンプト|cloud-shared-prompt)' 実行プロンプト.md || echo "CRITICAL: クラウド共通プロンプト.md の Read がない"
```

**判定:** ローカル CronCreate として設計されている場合、本観点はスキップ。クラウド向けで `クラウド共通プロンプト.md` の Read がなければ CRITICAL。allowed_tools 不足なら CRITICAL

### 観点 B: プロンプト構造・完了条件

**目的:** 各 Phase に完了条件があり、自律実行ガードが含まれるか検証する

**チェック項目:**
- [ ] 冒頭に自律実行ガード（「ユーザーへ質問してはいけません」）が存在するか
- [ ] 全 Phase に `### 完了条件` サブセクションが存在するか
- [ ] `AskUserQuestion` の呼び出し禁止が明示されているか
- [ ] エラー時の挙動（ログ記録 / スキップ / 中断）が Phase ごとに定義されているか
- [ ] 各 Phase の手順が具体的な Bash コマンドまたはツール呼び出しで記述されているか
- [ ] フォルダ名が日本語で、設計書のルーティン名と一致しているか

**検出パターン:**
```bash
# 完了条件の有無を Phase 数と突合
phase_count=$(grep -c '^## Phase' 実行プロンプト.md)
completion_count=$(grep -c '^### 完了条件' 実行プロンプト.md)
echo "Phase: $phase_count / 完了条件: $completion_count"
# フォルダ名の日本語チェック（ASCII のみなら違反）
dirname=$(basename "$(pwd)")
echo "$dirname" | grep -qP '^[a-zA-Z0-9_-]+$' && echo "WARN: フォルダ名が英語" || echo "OK: 日本語フォルダ名"
```

**判定:** Phase 数と完了条件数が一致しなければ CRITICAL。フォルダ名が英語なら WARN（日本語へのリネームを提案）

### 観点 C: ツール依存の解決可能性

**目的:** 実行プロンプトが参照するツール・スクリプト・ファイルが実行環境で利用可能か検証する

**チェック項目:**
- [ ] 参照されている全スクリプト（`.sh`、`.py` 等）がリポジトリ内に存在するか
- [ ] 外部ツール（`detect-secrets`、`trivy` 等）のインストール手順が Phase 1 に含まれているか
- [ ] MCP コネクタが必要な操作（Slack 通知等）のコネクタ設定が設計書に記載されているか
- [ ] ツールのパスがリポジトリルート相対で記述されているか（絶対パス禁止）
- [ ] Bash コマンドがクラウド環境（Ubuntu）で動作するか（`globstar`・`**/*.ts` のネスト展開・macOS 固有オプション等の移植性）（← D-2）

**検出パターン:**
```bash
# パス参照を抽出して存在確認
grep -oE '[a-zA-Z0-9_/./-]+\.(sh|py|js|ts)' 実行プロンプト.md | while read f; do
  [ -f "$f" ] && echo "OK: $f" || echo "MISSING: $f"
done
```

**判定:** MISSING が 1 件以上なら CRITICAL

### 観点 D: プロンプト管理方式

**目的:** 実行指示が `実行プロンプト.md` に一元管理され、インライン記述が排除されているか検証する

**チェック項目:**
- [ ] `実行プロンプト.md` がリポジトリ内に存在し、git 管理下にあるか
- [ ] CronCreate / クラウド UI の `prompt` が参照形式（`cloud-prompt-template.md` のテンプレート）のみか
- [ ] `実行プロンプト.md` の内容が設計書の処理フローと整合しているか
- [ ] クラウド UI の設定が `cloud-operations.md` の 7 項目チェックリストを満たしているか（リポジトリ 2 つ指定・環境・コネクタ・パーミッション等）

**検出パターン:**
```bash
git ls-files --error-unmatch 実行プロンプト.md 2>/dev/null && echo "TRACKED" || echo "UNTRACKED"
```

**判定:** UNTRACKED かつクラウド Routines 向けなら CRITICAL（fresh clone に含まれない）。クラウド UI 設定の突合は `cloud-operations.md` の 7 項目で実施する

### 観点 E: 冪等性

**目的:** ルーティンが途中失敗→再実行しても副作用が重複しないか検証する

**チェック項目:**
- [ ] issue 作成前に同一タイトルの open issue を検索しているか
- [ ] PR 作成前に同一ブランチの open PR を検索しているか
- [ ] ファイル書き込み時に既存内容との差分チェックを行っているか
- [ ] 外部通知（Slack 等）に重複防止の仕組みがあるか（またはべき等な操作か）
- [ ] 設計書の「冪等性設計」フィールドが空でないか
- [ ] ログファイル名にタイムスタンプ等を含み、同日複数回実行でも上書きされない設計か（← R-5）

**判定:** issue/PR 作成を含むルーティンで重複検索がなければ CRITICAL。ログ上書きリスクがあれば WARN

### 観点 F: スコープ逸脱検知

**目的:** ルーティンのプロンプトがリポジトリ外の操作やコネクタ未設定の外部サービスにアクセスしていないか検証する

**チェック項目:**
- [ ] GitHub API 操作を含む場合、GitHub コネクタが設定済みか
- [ ] Slack 通知を含む場合、Slack コネクタが設定済みか
- [ ] 外部 API（SaaS）呼び出しを含む場合、ネットワークポリシー（Trusted / Custom / Full）で許可されているか
- [ ] `git push` 先が `claude/` プレフィックスのブランチか（クラウドデフォルト制限）

**判定:** コネクタ未設定の外部サービス呼び出しがあれば WARN

### 観点 G: 実行コスト・時間

**目的:** 実行時間と頻度が妥当か検証する

**チェック項目:**
- [ ] 1 回の実行が 30 分以内で完了する見込みか
- [ ] 重い処理（全ファイルスキャン等）にタイムアウトが設定されているか
- [ ] 不要な頻度で実行されていないか（日次で十分なのに時間単位等）
- [ ] 5h グリッドのスロット（05/10/15/20/00 JST）に配置されているか

**削除済みチェック項目（2026-06-21）:**
- ~~全登録済みルーティンの日次実行回数合計がプラン上限内か（Pro: 5, Max: 15, Team: 25）~~ — 日次上限が撤廃されたため削除。claude.ai/code/routines/ の UI で上限表示が消えていることを確認

**判定:** 5h グリッド外の時刻に配置されていれば WARN

### 観点 H: 失敗時リカバリ設計

**目的:** ルーティン失敗時の通知・リトライ・手動介入が設計されているか検証する

**チェック項目:**
- [ ] 設計書の「失敗時リカバリ」フィールドが空でないか
- [ ] 致命的エラー時の通知先（Slack / issue / メール）が指定されているか
- [ ] リトライ戦略（回数・間隔・条件）が定義されているか
- [ ] 手動介入が必要な場合の手順が記載されているか
- [ ] 実行プロンプトのエラー時挙動と設計書のリカバリ設計が整合しているか

- [ ] 後続タスクの委任先（別ルーティン・人手等）が存在し稼働しているか確認済みか（← D-7）

**判定:** 「失敗時リカバリ」が空なら WARN。委任先が存在しないか稼働停止中なら CRITICAL

### 観点 I: エージェント制御性

**目的:** エージェントが Phase をスキップしたり、実行不可能な方式を「0 件」として素通りしたりしないか検証する

**チェック項目:**
- [ ] Phase スキップ防止の仕組みがあるか（各 Phase の冒頭に「この Phase をスキップしてはならない」等の明示、または Phase 完了条件に「実行した証跡の出力」を含む）（← R-1）
- [ ] 指示された方式が物理的に実行可能か（例: 「1 テストずつ再計測して効果を確認」が数百ファイルで現実的か）（← R-6）
- [ ] 「0 件」結果が「検査して 0 件」なのか「検査をスキップ」なのか区別できるログ設計になっているか（← R-8）
- [ ] エージェントが判断に迷う曖昧な分岐（「適宜」「必要に応じて」）がないか
- [ ] 各 Phase の手順が具体的な Bash コマンドまたはツール呼び出しで記述され、エージェントの裁量余地が最小化されているか

**検出パターン:**
```bash
# 曖昧表現の検出
grep -nE '(適宜|必要に応じて|可能であれば|できれば|状況に応じ)' 実行プロンプト.md
# Phase ごとの証跡出力の有無
grep -A5 '^## Phase' 実行プロンプト.md | grep -c 'echo\|log\|出力'
```

**判定:** Phase スキップ防止がなく品質検証 Phase を含むルーティンなら CRITICAL。実行不可能な方式が含まれていれば CRITICAL

### 観点 J: ログ・追跡性

**目的:** ルーティンの実行結果が JSONL ログに一元出力され、統計レポートで集約可能か検証する。正本: `routines/_shared/common-log-rules.md`

**チェック項目:**
- [ ] 結果が `logs/<slug>/YYYY-MM-DD.jsonl` に JSONL 形式で出力される設計か（← D-8）
- [ ] **issue 起票・Slack 通知等の報告を一切行っていないか**（脆弱性検出等も issue ではなく `run_end.status:"fail"` + `detail` に記録。報告は `ルーティン統計レポート` に一本化）
- [ ] PR 作成はコード変更を伴う場合のみに限定されているか（結果報告だけの PR は禁止）
- [ ] JSONL が共通エンベロープ（`run_start` → `phase` × N → `run_end`）に準拠しているか（← R-2）
- [ ] 実行プロンプトの事前準備セクションで `common-log-rules.md` を Read しているか
- [ ] `run_end.status` の判定ロジック（1 phase でも fail → fail、warn あり → warn）が正しいか
- [ ] 改善前後の差分が `phase.detail` に記録される設計か（← R-7）
- [ ] 検査実施の証跡と未実施の区別が可能か（`phase` イベントが存在 + `status:"ok"` vs phase イベント自体がない）（← R-8）
- [ ] `run_end` が無い場合（crash）の try-finally パターンが考慮されているか
- [ ] ログローテーションは `ログ清掃` ルーティンに委任しているか（自前でローテーションしない）

**検出パターン:**
```bash
# issue 起票の検出（全面禁止）
grep -nE '(gh issue create|gh issue comment|issue.*作成|issue.*起票)' 実行プロンプト.md && echo "CRITICAL: issue 起票は禁止"
# Slack 通知の検出（全面禁止）
grep -nE '(slack|通知|notify)' 実行プロンプト.md && echo "CRITICAL: 通知は禁止"
# JSONL 出力の有無
grep -nE '(\.jsonl|run_start|run_end|JSONL)' 実行プロンプト.md || echo "CRITICAL: JSONL 出力がない"
# common-log-rules.md の Read
grep -nE 'common-log-rules' 実行プロンプト.md || echo "WARN: common-log-rules.md の Read がない"
```

**判定:** issue 起票・通知が含まれていれば CRITICAL。JSONL 出力がなければ CRITICAL。common-log-rules.md の Read がなければ WARN

**検出パターン:**
```bash
# ログ出力の有無
grep -nE '(log|ログ|出力|記録|report)' 実行プロンプト.md
# before/after パターンの有無
grep -nE '(before|after|改善前|改善後|変更前|変更後)' 実行プロンプト.md
```

**判定:** ログ設計が設計書にない場合 WARN。改善系ルーティンで before/after が記録されない場合 CRITICAL

### 観点 K: 成果物品質ゲート

**目的:** ルーティンが生成する成果物（テスト・PR・コード変更等）に品質検証 Phase が含まれているか検証する

**チェック項目:**
- [ ] コード変更を含むルーティンで lint / 型チェックがマージ前に実行されるか（← D-9）
- [ ] テスト生成ルーティンで品質チェック（過剰テスト・トートロジー・デッドコード検出）が含まれているか（← D-5）
- [ ] 外部 API を呼ぶコードを生成する場合、モック方針が設計書に記載されているか（← D-4）
- [ ] PR 作成を含むルーティンで、PR の説明文・ラベル・レビュアー指定の基準が定義されているか
- [ ] 生成されたコードが既存テストを破壊しないことを確認する Phase があるか
- [ ] **PR のマージ判断基準が設計書に明記されているか:**
  - 設計書の「PR 影響度」フィールドが `影響なし` / `影響あり` / `PR なし` のいずれかで指定されているか
  - `影響なし`（テスト追加・ドキュメント・lint 修正等）→ マージまで自動実行する設計か
  - `影響あり`（プロダクションコード変更・API 変更・依存更新等）→ PR 作成で停止し、E2E テスト通過後に手動マージとする設計か
  - 影響度が混在する変更を「影響あり」として扱っているか
- [ ] 実行プロンプトに影響度に応じたマージ / 非マージの分岐が含まれているか

**検出パターン:**
```bash
# 品質チェック Phase の有無
grep -nE '(lint|eslint|tsc|型チェック|type.?check|品質|quality)' 実行プロンプト.md
# テスト実行の有無
grep -nE '(vitest|pytest|jest|npm test|テスト実行)' 実行プロンプト.md
# マージ判断の有無
grep -nE '(merge|マージ|E2E|影響)' 実行プロンプト.md
# 設計書の PR 影響度フィールド
grep -nE '(PR 影響度|影響なし|影響あり|PR なし)' ルーティン設計書.md
```

**判定:** コード変更を含むルーティンで lint/型チェックがなければ CRITICAL。テスト生成ルーティンで品質チェックがなければ CRITICAL。PR 作成を含むルーティンでマージ判断基準が未定義なら CRITICAL

### 観点 L: スコープ制御

**目的:** ルーティンの処理量・対象選択・目標設定が適切に制御されているか検証する

**チェック項目:**
- [ ] 1 回あたりの処理上限（ファイル数・PR 数等）が設定されているか（← D-10）
- [ ] 処理対象の選択ロジックに偏りがないか（例: FE/BE を共通ソートすると一方が永久に後回しになる）（← R-4）
- [ ] 目標値（閾値等）と到達戦略が明確か（「90% まで」の先の戦略が未定義でないか）（← D-6）
- [ ] 後続タスク（マージ・レビュー等）の責任者が明示されているか（← D-7）
- [ ] 処理対象が 0 件の場合のショートカット（早期終了）が設計されているか

**検出パターン:**
```bash
# 上限設定の有無
grep -nE '(上限|max|limit|最大|件まで|ファイルまで)' 実行プロンプト.md
# ソート・選択ロジック
grep -nE '(sort|ソート|優先|選択|フィルタ|filter)' 実行プロンプト.md
```

**判定:** 処理上限がないルーティンで PR 作成を含む場合 CRITICAL。選択ロジックに偏りがあれば WARN

## レビュー Phase

### Phase 1: 対象発見

1. `routines/<project>/routines/` 配下の全ルーティンを列挙
2. レビュー対象を特定（指定がなければ全件）
3. 各ルーティンの設計書と実行プロンプトの存在を確認
4. フォルダ名が日本語か確認。英語（kebab-case）のフォルダ名は規約違反として検出する

**検出パターン:**
```bash
# 英語フォルダ名の検出（ASCII のみで構成されたディレクトリ名を抽出）
ls routines/<project>/routines/ | grep -P '^[a-zA-Z0-9_-]+$'
```

**完了条件:** レビュー対象の一覧が確定し、ファイル存在が確認済み。英語フォルダ名がある場合は WARN として検出済み

### Phase 2: 静的検査

1. 12 観点のチェック項目を順に検査
2. 検出パターン（grep / bash）を実行
3. 結果を CRITICAL / WARN / INFO に分類

**完了条件:** 全観点の検査が完了し、結果が分類済み

### Phase 3: クロスチェック

1. 設計書と実行プロンプトの整合性を確認:
   - 設計書の Phase 数と実行プロンプトの Phase 数が一致するか
   - 設計書の依存関係と実行プロンプトのインストール手順が整合するか
   - 設計書のスケジュールと登録済みスケジュールが一致するか
2. 複数ルーティン間の衝突を確認:
   - 同一時刻に 3 件以上集中していないか
   - 同一ファイル / ブランチへの競合書き込みがないか

**完了条件:** 整合性不一致が全て検出済み

### Phase 4: レポート出力

```
## ルーティンレビューレポート

対象: <ルーティン名> (<クラウド/ローカル>)
日時: <レビュー実施日>

### 結果サマリ
| 重大度 | 件数 |
|---|---|
| CRITICAL | N |
| WARN | N |
| INFO | N |

### 検出事項
1. [CRITICAL] 観点B: Phase 3 に完了条件がない
2. [WARN] 観点H: 失敗時リカバリの通知先が未指定
3. [INFO] 観点G: 実行時間の見積もりが記載されていない
```

**完了条件:** レポートが出力され、全検出事項に重大度が付与されていること

### Phase 5: 自動修正提案

1. CRITICAL / WARN の検出事項に対して修正案を提示
2. ユーザーに修正の承認を求める
3. 承認された修正を適用

**完了条件:** CRITICAL が 0 件になるか、ユーザーが残存を明示的に承認したこと

### Phase 6: 連鎖判定

1. CRITICAL が残存していれば `AskUserQuestion` で連鎖を確認
2. CRITICAL 0 件なら test モードへ自動連鎖

**完了条件:** 連鎖先（test / 終了）が決定していること

## 健全性目安

| 状態 | 基準 |
|---|---|
| 健全 | CRITICAL 0 件、WARN 2 件以下 |
| 要改善 | CRITICAL 0 件、WARN 3 件以上 |
