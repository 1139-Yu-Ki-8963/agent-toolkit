§1概要

リバースエンジニアリング設計の中心課題は「設計書が完全であることの証明」にある。本スキルはこれを、設計書だけを見て再構築したリバースコード環境が、オリジナルコード環境とポート番号以外で完全に同じ動作をすることで証明する。

責務境界

担当責務
本スキル
syncing-reverse-env		 ⓪ 両 worktree の用意と名前管理（初回のみ作成・固定ポート割当・冪等な再発見・ユーザー依頼時のみ削除）
① 同一性検証（静的 diff ＋ 環境ヘルスチェック ＋ Playwright 動的比較）
② 差分があればリバースコード環境をオリジナルコード環境へ整列
③ 一致証明後のコミット＝基準コミットの確立と基準タグの記録
④ 基準コミットへの復帰（sync の reset-first オプション）
リバースを行うスキル
（呼び出し元・将来作成）		 設計書の不備を修正し、リバースコード環境の中身（コード）を再構築するループ全体の運転。worktree という容器の用意は本スキルに任せる

オリジナルコード環境が常に「正」。リバースコード環境と設計書のどちらを直すかの判断はリバースを行うスキルが持ち、本スキルは環境の用意・判定・整列に徹する。

§2リバース以外の使い方

本スキルはリバース検証のために設計する。ただし「ポート以外同一の 2 環境を永続 worktree で維持し、静的 diff ＋ Playwright 4 層で突合する」という仕組みの性質上、リバース以外にも次のような使い方ができると考えられる。

応用オリジナルコード環境（正）リバースコード環境（対象）使い方
PR レビュー実機確認main ブランチPR ブランチ 並行起動して挙動差分・ビジュアル差分を実機で突合する。「意図した変更以外に差分がないか」を機械判定できる
リファクタ安全性証明リファクタ前リファクタ後 「挙動が変わらないこと」を L1〜L4 の全層一致で証明する。この場合は実差分ゼロではなく動的比較の全層一致が合格条件になる
UI ビジュアルリグレッション変更前変更後 L3（スクリーンショット画素比較）を単独利用し、意図しない見た目の変化を検出する

応用時は「何を一致とみなすか」（静的一致まで求めるか、動的一致のみか）が用途ごとに変わる。合格条件のプリセットを mode とは別の軸で持たせることが、スキル本体実装時の検討事項になる。

§3用語

オリジナルコード環境
リバース対象のオリジナルコードを保持する git worktree（worktree 名 original-code-<system>）。設計書 frontmatter の source_repo / source_ref から本スキルが用意する読み取り専用コピーで、常に「正」。内容を変更することはない。
リバースコード環境
設計書だけを見て再構築したコードの git worktree（worktree 名 reverse-code-<system>、ブランチ feature/reverse-code-<system>）。容器（worktree・ブランチ・固定ポート）は本スキルが用意し、中身のコードはリバースを行うスキルが書く。
worktree のライフサイクル（永続・使い回し）
両 worktree は初回実行時に一度だけ作成し、以後の検証ではすべて同じものを使い回す。削除はユーザーが明示的に依頼した時（teardown）のみ。
基準コミット / 基準タグ
ポート以外の完全一致を証明できた時点のリバースコード環境側コミットが基準コミット。annotated git tag reverse-baseline/<system>（基準タグ）がその唯一の記録で、タグメッセージに検証日と判定サマリを持つ。コミットとタグはリポジトリ本体に保存されるため、worktree を破棄しても消えない。リバースコード環境はいつでも git reset --hard reverse-baseline/<system> で基準状態へ復帰できる。
固定ポート
両環境の稼働ポート。プロジェクトに依存しない全体共通の固定値（オリジナル=9100 番台 / リバース=9110 番台。§5）。両環境は必ず異なる帯域を持ち、これが「唯一許容される差分＝ポート番号」の根拠になる。
config.yml（スキルフォルダ直下）
プロジェクトによって変えたくなる可能性がある値（ポート番号・起動手順・diff 除外リスト・L3 閾値・max-loop 既定値）の置き場。defaults ＋ projects 別上書きの 2 層構造。

§4使い方（mode とユースケース）

mode は運用の 3 局面 —「初回の環境整備」「それ以降の検証サイクル」「検証終了時の片付け」— に対応する 3 つ。

mode局面動作
setup初回両 worktree を作成し、名前・固定ポートを割り当てて返す。比較は実行しない
sync（既定）2 回目以降検証（静的＋環境＋動的）→ 差分があれば整列 → 一致で基準タグを更新
teardown検証終了時固定ポート帯の一括 kill ＋ 両 worktree の削除。ユーザーの明示依頼がある時だけ受け付ける

sync のオプション

オプション動作
dry-run整列・基準タグ更新を行わず、差分レポートだけを返す。無副作用の対象は「リバースコード環境の内容」と「基準タグ」の 2 つ（Phase 5 の起動・観測とオリジナル側の source_ref 復元は実行する）
reset-first開始前にリバースコード環境を基準タグへ git reset --hard で復帰させてから検証サイクルに入る

アクター別ユースケース

アクター意図入口と mode期待する出力
リバースを行うスキル
（初回の環境整備）		 検証システムの立ち上げ時に、書き込み先となるリバースコード環境と比較相手のオリジナルコード環境を整備したい Skill ツール・args 全量指定
setup		 両環境のパス・ブランチ名・確定ポート一覧。比較は実行しない
リバースを行うスキル
（検証サイクル）		 設計書を直してリバースコード環境を再構築した直後、「今回の設計書で完全再現できたか」を機械判定し、一致すれば基準を確立したい Skill ツール・args 全量指定
sync		 PASS / FAIL ＋ 差分サマリ ＋ 設計書章マップへの対応ヒント（§8）。PASS 時は基準タグを更新
人間（状態確認） 今、2 つの環境がどれくらいズレているかを見たいだけ。環境は変更したくない 直接起動
sync + dry-run		 差分レポート（静的 diff 一覧 ＋ スクリーンショット証跡）。整列・タグ更新なし
人間（片付け） 検証が終わったので環境一式を片付けたい 直接起動（明示依頼のみ）
teardown		 固定ポート帯の一括 kill ＋ 両 worktree の削除報告。基準タグとブランチは残す

入力（起動契約）

区分引数内容
必須 design-doc対象の画面基本設計書パス。唯一の必須入力＝検証の主キー（全 mode 共通。teardown でも <system> 導出に使う）。リバース元の同一性（source_repo / source_ref）はこの frontmatter が保持する
任意 modesetup / sync（既定）/ teardown
dry-run / reset-firstsync のオプション（上記）
user-approvedteardown 時のユーザー依頼発話の引用。呼び出し元スキル経由の teardown では必須（無ければ削除せず ERROR で差し戻す）
routes動的比較で巡回する画面パス。省略時は設計書 frontmatter の route から導出
max-loop整列 → 再検証の収束ループ上限。既定は config.yml の値（3）
  • リバースを行うスキルから: args で全量指定する。対話ゼロで完走できることが契約（AskUserQuestion を発行しない）。
  • 人間から: design-doc だけ渡せば起動できる。
  • 共通: worktree のパスを引数で受け取ることはない。両環境は本スキルが命名規則（§5）から導出・確保する。

§5環境と設定（命名・ポート・config.yml）

命名規則と環境確保

worktree の用意・命名・再発見はすべて本スキルが担う。作成は初回実行時の一度だけで、以後は同じ worktree を再発見して使い回す。どの mode でも Phase 2 で必ず実行し、冪等に振る舞う。

対象規則
システム識別子 <system>設計書 frontmatter の source_repo 末尾のリポジトリ名から機械的に導出する（doc_id は画面粒度のため使わない）
オリジナルコード環境worktree 名 original-code-<system>。source_ref を固定 checkout した読み取り専用コピー。汚れ・ズレを検出したら内容だけを source_ref へ復元する（worktree 自体・名前・ポートは保持）
リバースコード環境worktree 名 reverse-code-<system>、ブランチ feature/reverse-code-<system>。基準コミットと再構築履歴を持つ本体
基準タグannotated tag reverse-baseline/<system>。基準コミットの唯一の記録
ポートconfig.yml の固定ベース（オリジナル=9100 / リバース=9110）から下の割当表のとおり確定する。プロジェクトに依存しない
作成の実務worktree の物理作成・削除は git worktree add / remove を本スキルが直接実行する（オリジナルは --detach、リバースは -b。配置は source_repo の親ディレクトリ直下）。通常開発用の worktree 管理の仕組みとは命名・配置規約が異なるため委譲しない

ポート割当（確定値）

環境ベースサービス別ポート
オリジナルコード環境9100backend 9100 / frontend 9101 / docs 9102 / DB tools 9103（9104〜9109 は予約）
リバースコード環境9110backend 9110 / frontend 9111 / docs 9112 / DB tools 9113（9114〜9119 は予約）

config.yml

プロジェクトによって変えたくなる可能性がある値は、スキルフォルダ直下の config.yml に集約する。値の階層は 3 層で役割分担する: 普遍のルール＝SKILL.md / プロジェクト単位の可変値（ポート上書き・起動手順）＝config.yml / 画面単位の可変値（routes・マスク対象セレクタ）＝設計書 frontmatter。

── config.yml の構造 ───────────────────────────────────────

  defaults:
    ports:
      original_code_base: 9100  # オリジナルコード環境のベースポート
      reverse_code_base: 9110   # リバースコード環境のベースポート
    launch: "npm run dev -- --port {port}"  # 起動コマンドの既定形
    l3_threshold_percent: 0.1   # L3 視覚比較の許容差分画素率
    max_loop: 3                 # 収束ループ上限の既定値
    tag_namespace: reverse-baseline
    diff_exclude:               # 静的比較の除外リスト（§7）
      - .git/
      - node_modules/
      - dist/
      - .next/
      - "*.lock"
      - ".env*"                 # ポート設定を含む環境ファイル

  projects:                     # プロジェクト単位の上書き（必要時のみ）
    "<project-name>":
      ports: { original_code_base: 9120, reverse_code_base: 9130 }
      launch:                   # 起動手順は唯一のアプリ依存要素
        backend: "<backend 起動コマンド> --port {port}"
        frontend: "<frontend 起動コマンド> --port {port}"

制約

制約: ① ポートは全体共通の固定値のため、リバース検証環境は同時に 1 組だけ稼働できる（並行稼働は config.yml の projects 上書きで別帯域を割り当てた場合のみ） ② worktree は全体で 5 本までの上限のうち 2 本を長期に占有する ③ teardown はユーザーの明示依頼がある時だけ受け付け、本スキルが自発的に削除することはない ④ teardown 後も基準タグとブランチは残り、基準コミットは消えない

プリフライト（go / no-go 判定）

#検証項目NO-GO 時の扱い
1設計書が存在し reverse-docs テンプレート準拠（type: screen-basic-design）で、source_repo / source_ref が解決できるNO-GO
2環境確保（命名規則で両環境を検出。無ければ作成）が完了しているworktree 総数上限（5 本）超過は NO-GO
3オリジナルコード環境が source_ref と一致し cleanズレ・汚れは内容を source_ref へ自動復元する
4リバースコード環境の dirty 状態を記録sync では許容（整列で上書きされる）。dry-run 指定時は警告を提示
5固定ポート（9100 / 9110 番台）が未占有stale プロセスは kill して続行
6Playwright が利用可能、リバースコード環境側の依存インストールが完了（依存確認は sync のみ適用。setup 直後の空容器では Playwright 可用性の確認だけ行う）NO-GO（ERROR 系。§8）
7reset-first 指定時のみ: 基準タグが存在NO-GO（基準未確立と報告）

§6動作フロー

初回か再実行かは基準タグ reverse-baseline/<system> の有無で判定する（git tag -l ＋ 参照先コミットの実在確認。タグなし・参照先不明はどちらも初回扱い）。

── 分岐の全体図 ────────────────────────────────────────────

  Phase 1  起動契約解決（args 検証・config.yml 読込）
     │
  Phase 2  環境確保 + プリフライト（§5）── NO-GO ──▶ 停止・報告
     │ GO    worktree は初回のみ作成・以後は再発見して使い回す
     │
     ├─ mode=setup ────▶ 両 worktree のパス・ブランチ・確定ポートを
     │  （初回の整備）    返して終了（リバースを行うスキルはこの
     │                   リバースコード環境に再構築コードを書く）
     │
     ├─ mode=teardown ─▶ 明示依頼を確認できた場合のみ
     │  （検証終了時）    ポート一括 kill → worktree 削除 → 終了
     │                   （人間なら AskUserQuestion、スキル経由なら
     │                    args の user-approved 必須。無ければ削除せず
     │                    ERROR 差し戻し。基準タグとブランチは残す）
     │
     └─ mode=sync（既定。2 回目以降の検証サイクル）
          │
          │  基準タグの有無を確認（なし・参照先不明は初回扱い）
          │
          ├─ reset-first 指定 かつ タグあり
          │    → git reset --hard reverse-baseline/<system> で
          │      基準状態へ戻してから続行
          │
          ▼
        Phase 3〜8 の検証・整列サイクルへ
        （PASS 時: タグが無ければ新規に打ち、あれば -f で打ち直す。
          dry-run 指定時: 整列・タグ更新なしの差分レポートのみ）

── 検証・整列の収束ループ（Phase 3〜8）──────────────────────

  Phase 3  静的比較（§7-1: diff + ポート正規化判定）
     │
  Phase 4  整列（§7-4。実差分あり かつ dry-run でない時のみ）
     │
  Phase 5  環境同一性チェック（§7-2）＋ 動的比較（§7-3: Playwright 4 層）
     │       ※ dry-run でも実行する（静的 FAIL でも L1〜L4 まで計測し
     │         診断情報を最大化する）
     │
  Phase 6  収束判定（dry-run 指定時は 1 周で確定・ループしない）
     │        ├─ FAIL（ループ上限内）──▶ Phase 3 へ戻る
     │        └─ FAIL（max-loop 到達）─▶ FAIL 確定・差し戻し（§8）
     │ PASS
  Phase 7  基準確立（dry-run 指定時はスキップ）
     │        git commit で直接コミット（日本語 prefix 規約）
     │        → 基準コミットに annotated tag を打つ
     │           git tag -a reverse-baseline/<system>
     │        ※ 整列が走らず新規コミット対象が無い場合はコミットを
     │          スキップし、現 HEAD にタグを打ち直す
     │
  Phase 8  結果報告（§8 の返却フォーマットと定型文）

§7検証内容

7-1. 静的比較

「パス除外 → 残った差分行のポート正規化判定」の 2 段構えで、全差分を 3 分類に落とす。除外パスは config.yml の diff_exclude ＋ 設計書 frontmatter での画面固有の追加指定。

ポートが固定値で決定的であることを利用し、差分行ごとに「オリジナル側のポートをサービスオフセットへ逆引きし、リバース側の同位置の値がリバース側期待値（9110 番台）と一致する」場合のみ許容する。行内のポート以外の部分は完全一致を要求し、「数字が違うだけなら許容」にはしない。

分類意味扱い
一致差分なし—
ポート差のみ固定ポートの期待値どおりの番号相違許容（件数をレポートに記録）
実差分上記以外のすべての相違FAIL 対象。ファイル別・追加/削除/変更別に集計

7-2. 環境同一性チェック（10 項目）

実行環境そのものが「ポート以外は同一」かつ「互いに独立」であることを機械的に確認する。起動前（#1〜#2, #4〜#8, #10）と起動後（#3, #9）に実行する。

#検証項目確認方法NG 時の扱い
1ポート占有の帰属lsof -i で固定ポート帯を走査し、LISTEN プロセスの cwd が自環境の worktree かを突合stale は kill して続行。無関係な稼働プロセスは ERROR
2ポート設定の混入（転写後）リバース側を 9100 番台で、オリジナル側を 9110 番台で相互に grep相手帯域の番号が 1 件でもあれば転写ミス。ポート再生成をやり直す
3実際の待受ポート起動後の LISTEN 一覧を §5 割当表の期待値と突合期待外の待受・不応答は FAIL
4シンボリックリンクの実態find -type l ＋ realpath で実体を解決し「自 worktree 内 / 共有 store / worktree 外」に分類相手側 worktree・元リポジトリを指すものは FAIL。共有 store は同一バージョンの実体を指すことを確認して許容
5依存ツリーの同一性npm ls --all 相当を正規化し、名前＋バージョン集合を比較依存再構築を 1 回リトライ → なお不一致は FAIL
6ランタイムの同一性node -v 等を両環境で実測して突合不一致は ERROR（実行基盤の問題）
7環境変数の同一性起動時に注入される env をダンプし、ポート正規化後に比較ポート以外の差分は FAIL
8ファイルモード・種別git ls-files --stage 等で実行ビット・種別を比較実差分として扱い、整列で揃える
9プロセス独立性片方を停止してももう片方が応答し続けることを確認して再起動連動停止は FAIL（プロセス・DB の共有）
10ストレージ分離DB・キャッシュ等のパス設定が worktree 内に閉じているか走査同一の外部パスを読み書きしていれば FAIL

#1・#4・#9・#10 は「同一」ではなく「独立」の検証。実体を共有した 2 環境は一致して見えて当然であり、完全一致の証明は「独立した 2 環境が同じに振る舞う」ことで初めて成立する。

7-3. 動的比較（Playwright 4 層）

両環境をそれぞれの固定ポートで起動し、routes の各画面を 4 層で突合する。総合判定は L1 ∧ L2 ∧ L4 を必須とし、L3 は閾値超過で FAIL（画像ペアを添付して人間が最終裁定できる）。

層内容判定基準
L1 起動両環境のヘルスチェック / トップ応答両方の成功が前提。片方でも失敗なら以降をスキップして FAIL
L2 構造アクセシビリティスナップショット（DOM 構造）をポート・URL 正規化後に比較完全一致を要求（一次判定）
L3 視覚同一ビューポートのスクリーンショット画素比較（動的領域はマスク）差分画素率が閾値以下（config.yml、既定 0.1%）。補助判定・証跡
L4 コンソールconsole error / warning の集合比較（ポートは正規化）リバース側にのみ存在するエラーで FAIL。両方に出る既存エラーは許容

DB 内容・乱数・日時などの動的データによる誤検出には、マスク対象セレクタを設計書 frontmatter で宣言して対応する。

7-4. 整列

sync で実差分が見つかった場合（dry-run 指定がない時）、リバースコード環境をオリジナルコード環境に揃える。整列は git 操作（checkout / merge）ではなくファイル転写で行い、両ブランチの歴史を混ぜない。

── 整列の手順 ───────────────────────────────────────────────

  1. 保全    リバースコード環境のポート設定ファイル群を退避
  2. 転写    オリジナルコード環境 → リバースコード環境へミラーコピー
             （削除同期あり。除外は 7-1 準拠）
  3. 再生成  config.yml の固定ベース（9110）から
             リバースコード環境側のポート設定を再生成
  4. 再構築  依存インストールを再実行
  5. 再検証  Phase 3（静的）→ Phase 5（環境＋動的）へ合流し、
             収束ループを継続

§8報告フォーマット

異常系の挙動

局面挙動
プリフライト NO-GO停止して報告（source_repo / source_ref 解決不能・固定ポート帯の占有 等）
静的比較 FAILdry-run 指定時: 報告のみ。通常の sync: 自動整列へ進む
動的比較 FAIL（整列後も）依存再構築を 1 回だけ自動リトライ → なお FAIL なら収束ループ上限内で継続、上限到達で停止・差し戻し
収束ループ上限到達FAIL 確定。リバースコード環境は最後の整列状態のまま残し、コミット・基準更新はしない
環境起因（ポート占有・Playwright 不通 等）差分判定と区別して ERROR として報告（「設計書の不備」との誤解釈を防ぐ）

機械向け返却ブロック（リバースを行うスキルへの契約）

setup / teardown / ERROR の早期終了時も返却ブロックの形を保ち、未実施フィールドは「未実施」と記す。

── 返却ブロック ─────────────────────────────────────────────

  status:          PASS | FAIL | ERROR
  mode:            setup | sync | teardown（+ dry-run / reset-first の有無）
  original_code:   <オリジナルコード環境のパス>（ブランチ・ポート付き）
  reverse_code:    <リバースコード環境のパス>（ブランチ・ポート付き）
  baseline_tag:    reverse-baseline/<system> → <hash>  # PASS 時は新基準
  static_diff:     実差分 <N> 件（上位 <n> 件のファイル一覧）
                   ポート差（許容） <M> 件
  dynamic:         L1=PASS L2=PASS L3=PASS(0.03%) L4=PASS
                   （route 別の内訳を添付）
  env_check:       10/10 PASS（NG があれば §7-2 の項目番号と内容を列挙）
  artifacts:       <スクリーンショット等の保存先パス>
  hint:            # FAIL 時のみ
                   src/components/Foo.tsx の差分
                   → 設計書 §9 領域別仕様 の記載不足と推定

hint は「差分ファイル → 設計書章マップの対応推定」であり、リバースを行うスキルの設計書修正ループの入力になる。
