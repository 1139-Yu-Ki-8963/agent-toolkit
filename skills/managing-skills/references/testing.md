# スキル実機検証手順（testing）

`managing-skills` の **test モード** が参照する手順書。`conventions.md` を前段で読んだ前提で、サブエージェントを使った経験的プロンプトチューニングを実行する。

このファイルは create / review モードからの **自動連鎖の終端** にあたる。test モード完了後は最終レポートを返す（さらなる連鎖はない）。

## 概要

プロンプトの書き手はその品質を正確に判断できない。書き手が明確だと思うほど、別のエージェントがつまずきやすくなる。このモードの核心は **バイアスのない実行者に指示を実際に実行させ、双方向で評価し、繰り返す** こと。改善が横ばいになるまでやめない。

## 使用タイミング

- スキル・スラッシュコマンド・タスクプロンプトを新規作成・大幅改訂した直後（create / review モードからの自動連鎖）
- エージェントが期待どおりに動かず、指示側の曖昧さに原因があると思われるとき
- 重要な指示を強化するとき（頻繁に使うスキル・自動化の中核プロンプト）

使用しないとき:
- 使い捨ての一時プロンプト（評価コストが見合わない）
- 成功率の改善ではなく、書き手の主観的な好みを反映させることが目的のとき

## ワークフロー

0. **イテレーション 0 — description・本文の一貫性チェック**（静的、ディスパッチ不要）
   - フロントマター `description` が主張するトリガー・ユースケースを確認する
   - 本文が実際にカバーする範囲を確認する
   - ずれがあれば、イテレーション 1 に進む前に description か本文を修正する
   - 例: description が「ナビゲーション / フォーム入力 / データ抽出」と言っているのに、本文が `npx playwright test` の CLI リファレンスだけ — そのようなずれを検出する
   - このステップを省略すると、サブエージェントが本文を description に合わせて「再解釈」し、スキルが実際には要件を満たしていなくても精度が高く出てしまう（偽陽性）

1. **ベースライン準備**: 対象プロンプトを固定し、以下の 2 つを用意する。
   - **評価シナリオ** 2〜3 種（標準 1 つ＋エッジ 1〜2 つ）。対象プロンプトが実際に適用される状況を想定した現実的なタスク。
   - **要件チェックリスト**（精度計算用）。各シナリオに対して、成果物が満たすべき項目を 3〜7 個列挙する。精度 % ＝ 満足した項目数 ÷ 全項目数。あらかじめ固定し、後から動かさない。

2. **バイアスなし読み**: 「白紙状態」の実行者に指示を読ませる。Task ツールで **新しいサブエージェントをディスパッチ** する。セルフ再読で代替しない（自分が書いたテキストを客観視するのは構造的に不可能）。複数シナリオを並行実行する場合は、1 メッセージに複数の Agent 呼び出しを並べる。ディスパッチが使えない環境での対処は「環境制約」セクションを参照。

3. **実行**: 以下に示す **サブエージェント呼び出し規約** に従ったプロンプトをサブエージェントに渡し、シナリオを実行させる。実行者は成果物を生成し、最後にセルフレポートを返す。

4. **双方向評価**: 返ってきた結果から以下を記録する。
   - **実行者セルフレポート**: 不明瞭な点 / 独自判断で補填した箇所 / テンプレート適用で詰まった箇所
   - **トレース解釈**: 不明瞭な点は、その不明瞭さが生じた段階でタグ付けする（Understanding / Planning / Execution / Formatting）。段階局所の修正はグローバルな「プロンプトが不明瞭だった」という修正より効果が高い。
   - **構造化リフレクション**: 不明瞭な点ごとに `Issue / Cause / General Fix Rule` の 3 行で返す。`General Fix Rule` はクラスレベルの抽象化であり、「失敗パターン台帳」に入力する。
   - **指示側の計測**:
     - 成否: `[critical]` タグが付いた要件が **すべて** ○のときのみ成功（○）。1 つでも × または部分的なら失敗（×）。
     - 精度（要件チェックリストの達成率 %。○ = 満点、× = 0、部分 = 0.5）
     - ステップ数（Task ツールの戻り値 usage メタの `tool_uses` フィールド）
     - 所要時間（Task ツールの usage メタの `duration_ms`）
     - リトライ数（サブエージェントのセルフレポートから抽出）
     - **失敗時は「不明瞭な点」セクションに「どの `[critical]` 項目が落ちたか」を 1 行で追記する**
   - 要件チェックリストには `[critical]` タグ付き項目が **少なくとも 1 つ** 必要。後から追加・削除しない。

5. **差分を適用**: 不明瞭な点を解消する最小限の修正をプロンプトに加える。1 イテレーション 1 テーマ。
   - **修正を適用する前に「この修正が要件チェックリストのどの項目・判定文言を満たすか」を明示する**（軸名から推測した修正は当たらないことが多い）
   - **まず失敗パターン台帳を参照する**。`General Fix Rule` が既知パターンに一致する場合、最初の問いは「なぜ既存の修正が再発を防げなかったか」とする。

6. **再評価**: 新しいサブエージェントで 2 → 5 を繰り返す（同じエージェントを再利用しない）。

7. **収束チェック**: 「新しい不明瞭な点が 0 かつメトリクス改善が閾値未満の状態が 2 連続したら停止」が大まかな目安。重要なプロンプトは 3 連続にする。

## 評価軸

| 軸 | 取得方法 | 意味 |
|---|---|---|
| 成否 | `[critical]` 全 ○ で成功（2 値） | 最低基準 |
| 精度 | 要件の何 % を成果物が満たしたか | 部分的成功の度合い |
| ステップ数 | `tool_uses` フィールド | 指示の無駄のインジケーター |
| 所要時間 | `duration_ms` | 認知負荷の代理指標 |
| リトライ数 | 同じ判断のやり直し回数 | 指示の曖昧さのシグナル |
| 不明瞭な点（セルフレポート） | 実行者が箇条書きで列挙 | 定性的な改善材料 |
| 独自補填（セルフレポート） | 指示に固定されておらず独自判断した決定 | 暗黙の仕様を浮き彫りにする |

**重み付け**: 定性的（不明瞭な点 / 独自補填）を主、定量的（時間 / ステップ数）を補助とする。時間短縮のみを追うとプロンプトが薄くなりすぎる。

### `tool_uses` の定性的解釈

精度だけ見るとスキルの問題が隠れる。`tool_uses` を **シナリオ間の相対値** として使うと構造的欠陥が見える。

- あるシナリオが他と比べて **3〜5 倍以上** であれば、そのスキルは **自己完結性が低く参照ツリー頼み** のサイン
- 典型例: 全シナリオの `tool_uses` が 1〜3 なのに 1 シナリオだけ 15 以上 → スキル内にそのシナリオのレシピがなく、参照を横断検索している
- 対策: SKILL.md の先頭に「インライン最小完全例」または「参照を読むタイミングのガイダンス」を追加すると `tool_uses` が大幅に下がる

### 修正の伝播パターン

修正 → 効果は線形ではない。事前推定は以下の 3 パターンになりえる。

- **保守的スイング**（推定 > 実際）: 複数の軸を狙った修正が 1 軸しか動かなかった
- **オーバーシュート**（推定 < 実際）: 構造的な情報 1 つが複数の軸の判定文言を同時に満たした
- **ゼロシュート**（推定 > 0、実際 = 0）: 軸名から推測した修正が判定文言に届かなかった

推定精度を安定させるには、**差分を適用する前に「この修正がどの判定文言を満たすか」をサブエージェントに言語化させる**。

## サブエージェント呼び出し規約

実行者に渡すプロンプトは以下の構造に従う。

```
あなたは <対象プロンプト名> を白紙状態で読む実行者です。

## 対象プロンプト
<対象プロンプトの本文全体を貼り付けるか、Read 用のパスを指定する>

## シナリオ
<シナリオの文脈を 1 段落で設定する>

## 要件チェックリスト（成果物が満たすべき項目）
1. [critical] <最低基準に属する項目>
2. <通常項目>
3. <通常項目>
...
（[critical] は最低 1 つ必須）

## タスク
1. 対象プロンプトに従ってシナリオを実行し、成果物を生成する。
2. 完了後、以下のレポート構造で回答する。

## レポート構造
- 成果物: <成果物または実行概要>
- 要件達成: 各項目について ○ / × / 部分（理由付き）
- **トレース**（各段階に OK / stuck / skipped をタグ付け、OK 以外は 1 行で理由を記述）:
  - Understanding（指示を読んでメンタルモデルを構築する）
  - Planning（方法・順序を決定する）
  - Execution（実際に作業する）
  - Formatting（成果物を期待される形に整える）
  - *短縮形可*: 4 段階すべてが OK のとき、1 行「トレース: すべて OK」で十分
- **不明瞭な点（構造化）**: 問題ごとに 3 行:
  - Issue: <観察できた事象>
  - Cause: <なぜ、指示レベルで診断した原因>
  - General Fix Rule: <スポット修正ではなく、このクラスのミスを防ぐクラスレベルのルール>
- 独自補填: 指示に固定されておらず独自判断で補填した箇所（箇条書き）
- リトライ: 同じ判断を何回やり直したか、理由
```

呼び出し元は、レポートのセルフレポート部分を抽出し、Agent ツールの usage メタから `tool_uses` / `duration_ms` を取得して評価軸テーブルを埋める。

## 環境制約

新しいサブエージェントをディスパッチできない環境（すでにサブエージェントとして実行中、Task ツールが無効など）では、このモードを **適用しない**。
- 代替 1: 親セッションのユーザーに別の Claude Code セッションを開いてもらい、そちらで評価を行う
- 代替 2: 評価を諦め、「empirical evaluation skipped: dispatch unavailable」とユーザーに明示的に報告する
- **NG**: セルフ再読で代替する（バイアスが入るため、評価結果を信頼してはならない）

**構造レビューモード**: 経験的評価ではなく、スキル・プロンプトの **説明文の一貫性と明確さ** だけを確認したい場合は、構造レビューモードとして明示的に切り出す。サブエージェントへのリクエストプロンプトに「今回は構造レビューモード: テキストの一貫性チェックであり、実行ではない」と明記する。構造レビューは経験的評価の補助であり、代替ではない（連続クリア判定に使えない）。

## 反復停止基準

- **収束（停止）**: 以下を **すべて** 満たす状態が 2 連続したとき:
  - 新しい不明瞭な点: 0
  - 前回比の精度改善: +3 ポイント以下
  - 前回比のステップ数変動: ±10% 以内
  - 前回比の所要時間変動: ±15% 以内
  - **過学習チェック**: 収束判定時に、これまで使っていない hold-out シナリオを 1 つ追加して評価する。直近の平均から精度が 15 ポイント以上下がったら過学習
- **発散（設計を疑う）**: 3 回以上のイテレーションを経ても新しい不明瞭な点が減らない場合 → プロンプト自体の設計方向が間違っている可能性。パッチ修正をやめて構造から書き直す
- **リソース上限**: 重要度と改善コストが釣り合わなくなったら停止（「80 点で出荷する」という判断）

## 失敗パターン台帳

イテレーション間で蓄積する失敗モードのリスト。台帳がないと各イテレーションが同じクラスのミスを再発見し続ける。

エントリ形式:

```
- **パターン名**: 短い記述ハンドル
  - 例: <あるイテレーションの代表的な Issue 文言>
  - General Fix Rule: <そのイテレーションの構造化リフレクションから得たクラスレベルのルール>
  - 発生イテレーション: N、M、...
```

ルール:
- ワークフロー ステップ 5 で修正を生成する前に台帳をスキャンする
- 的確な修正を施しても 3 回以上再発するパターンは構造的なシグナル。「発散」基準を適用しパッチ修正を打ち切って構造から書き直す
- 台帳は対象プロンプトごと。全チューニング実行を横断するグローバルなものではない

## バリアント探索（任意・停滞打破）

イテレーションが横ばいに近づいているが収束基準（2 連続クリア）を満たさない場合、局所最適を疑って 2 バリアントラウンドを実行する。

- **保守バリアント**: 現在のプロンプト + 次善の小修正
- **探索バリアント**: 現在のプロンプトに 1 つの構造変更

同じシナリオに対して新鮮なサブエージェントを並行ディスパッチする。精度が高いほうを採用し、同点なら不明瞭な点が少ないほうを選択し、さらに同点なら `tool_uses` が少ないほうを選択する。

ペアワイズ比較の注意:
- 「A と B を直接評価させる」ことをサブエージェントに頼まない（LLM の位置バイアスと自己選好バイアス）
- 客観的な軸（精度・ステップ数・不明瞭な点数・段階弱点数）だけで比較する
- 定性的な比較が本当に必要な場合はカウンターバランスをとる: (A, B) と (B, A) の両順序を実行し、両方の順序が一致した場合のみ判定を受け入れる

コスト: バリアント探索はイテレーションあたりのディスパッチ数が 2 倍。デフォルトでは使わない。

## 提示フォーマット

各イテレーションでユーザーへの提示は以下の形式で記録・報告する。

```
## イテレーション N

### 変更内容（前回からの差分）
- <修正内容を 1 行で>
- 適用パターン: <台帳のパターン名、または「（新規）」>

### 実行結果（シナリオごと）
| シナリオ | 成否 | 精度 | ステップ数 | 所要時間 | リトライ数 | 弱い段階 |
|---|---|---|---|---|---|---|
| A | ○ | 90% | 4 | 20s | 0 | — |
| B | × | 60% | 9 | 41s | 2 | Execution |

### 構造化リフレクション（今回新たに浮上したもの）
- <シナリオ B>: [critical] 項目 N が × — <低下の 1 行理由>
  - Issue: <観察できた事象>
  - Cause: <なぜ、指示レベルで>
  - General Fix Rule: <クラスレベルの抽象化>

### 独自補填（今回新たに浮上したもの）
- <シナリオ B>: <補填内容>

### 台帳更新
- 追加: <パターン名>（シナリオ B から）
- 再発: <パターン名>（元はイテレーション K）— 既存の修正が再発を防げなかった理由: <理由>

### 次の修正案
- <最小限の修正を 1 行で>

（収束チェック: X 連続クリア / 停止条件まで残り Y ラウンド）
```

## レッドフラグ（合理化に注意）

| 浮上する合理化 | 現実 |
|---|---|
| 「自分で再読しても同じ効果がある」 | 自分が書いたテキストを客観的に見ることはできない。必ず新しいサブエージェントをディスパッチする |
| 「シナリオ 1 つで十分」 | 1 つだと過学習する。最低 2 つ、理想は 3 つ |
| 「不明瞭な点がゼロになったので完了」 | 偶然かもしれない。2 連続で確定する |
| 「複数の不明瞭な点を一度に解消しよう」 | 何が効いたかわからなくなる。1 イテレーション 1 テーマ |
| 「メトリクスが良いので定性フィードバックは無視」 | 時間短縮は薄くなりすぎているサインにもなる |
| 「最初から書き直すほうが速い」 | 3 回以上のイテレーションを経ても不明瞭な点が減らない場合は正しい判断。それ以前の段階では逃げ |
| 「同じサブエージェントを再利用しよう」 | 前の改善を学習してしまっている。必ず新しいサブエージェントをディスパッチする |

## よくある失敗

- **シナリオが簡単すぎる / 難しすぎる**: どちらもシグナルが得られない
- **メトリクスしか見ない**: 時間短縮だけを追うと重要な説明が削られてもろくなる
- **1 イテレーションの変更が多すぎる**: 「あのとき何の修正が効いたか」をたどれなくなる
- **シナリオを修正に合わせて調整する**: 本末転倒

## Gotchas

- スキルの発火テストは description を読んで判定するのではなく、実際のプロンプトでサブエージェントに実行させて確認する
- test モードは create / review からの **連鎖の終端**。さらなる連鎖はなく、最終レポートで完了する
